
# 스케줄링

> Operating Systems 101 시리즈 (3/10)

<!-- a-grade-intro:begin -->

**핵심 질문**: 같은 컴퓨터에 수백 개의 프로세스가 떠 있는데, OS는 누구에게 다음 1밀리초의 CPU를 줄지를 어떻게 결정할까요?

> 스케줄러는 OS의 가장 자주 동작하는 부품 중 하나입니다. 매 ms 단위로 "지금 CPU를 누구에게 줄 것인가"를 결정하고, 그 결정이 응답성, 처리량, 공정성, 전력 소비를 모두 좌우합니다. 이 글에서는 스케줄러가 왜 필요한지, 어떤 정책들이 있는지, 그리고 컨텍스트 스위치라는 비용이 왜 자꾸 말 끝에 따라붙는지를 정리합니다.

<!-- a-grade-intro:end -->

## 이 글에서 배울 것

- 스케줄러의 목표와 그 사이의 트레이드오프
- 대표적인 스케줄링 정책 (FCFS, RR, Priority, MLFQ, CFS)
- 컨텍스트 스위치의 의미와 비용
- 사용자가 조정할 수 있는 것 — `nice`, 우선순위, CPU 어피니티

## 왜 중요한가

스케줄러는 평소에 잘 보이지 않지만, 시스템이 느려지거나 응답성이 흔들릴 때 거의 매번 의심해야 할 첫 번째 대상입니다. CPU가 한가한데 응답이 느린 이유, 같은 코드를 실행하는데 결과 시간이 들쑥날쑥한 이유, 컨테이너에 CPU 제한을 걸었더니 갑자기 굼떠진 이유 — 모두 스케줄러의 결정과 관련이 있습니다.

> 응답성, 처리량, 공정성, 전력 — 스케줄러는 이 네 가지를 동시에 만족할 수 없고, 어느 한쪽을 늘리면 어느 한쪽이 줄어듭니다.

## 개념 한눈에 보기

> 스케줄러는 실행 가능한(runnable) 작업의 큐에서 다음에 실행할 작업을 골라 CPU에 올립니다. 작업이 I/O를 기다리거나, 시간 할당량을 다 쓰거나, 더 높은 우선순위의 작업이 깨어날 때 스케줄러가 다시 호출됩니다.

```text
Runnable queue        Running              Blocked (waiting I/O)
+--------------+      +----------+         +-------------------+
| T1, T3, T5   | ---> |   T2     |         |  T4 (read)        |
+--------------+      +----------+         |  T6 (sleep)       |
       ^                   |               +-------------------+
       |                   v
       +-- preempt /  time slice expires --+
                           |
                           +--> back to runnable queue
```

## 핵심 용어 정리

| 용어 | 설명 |
| --- | --- |
| 컨텍스트 스위치 | 한 작업의 실행 상태를 저장하고 다른 작업의 상태를 복원하는 과정 |
| 시간 할당량(time slice) | 선점형 스케줄러가 한 작업에 한 번에 주는 CPU 시간 |
| 우선순위 | 어느 작업이 먼저 또는 더 자주 실행될지의 가중치 |
| `nice` 값 | 리눅스에서 사용자가 조절할 수 있는 우선순위 힌트(낮을수록 우선) |
| CFS | 리눅스의 Completely Fair Scheduler, 시간을 가상으로 공정하게 분배 |

## Before / After

**Before — "OS가 알아서 골고루 돌리겠지":**

```bash
# 백그라운드 작업 4개를 동시에 돌린다
./heavy_task & ./heavy_task & ./heavy_task & ./heavy_task &
```

같은 시간에 끝날까? 한 작업이 다른 작업을 굶길까? 응답성에 어떤 영향이 갈까? 짐작만으로는 알 수 없습니다.

**After — "각 작업의 상태와 시간을 추적할 수 있다":**

```text
T1: 실행 50ms → 시간 할당량 만료 → runnable
T2: I/O 대기 → 데이터 도착 → runnable
T3: 우선순위 높음 → T2 깨어날 때 즉시 선점
T4: nice +10 → 같은 자원 경쟁에서 적게 받음
```

스케줄러는 이런 작은 결정을 초당 수천 번 합니다.

## 실습: 단계별로 따라하기

### 1단계: 컨텍스트 스위치 횟수 측정

```bash
/usr/bin/time -v python3 -c "
import threading
def loop():
    for _ in range(10**6): pass
ts = [threading.Thread(target=loop) for _ in range(8)]
for t in ts: t.start()
for t in ts: t.join()
" 2>&1 | grep -E "context switches"
```

`Voluntary context switches`(자기가 양보)와 `Involuntary`(스케줄러가 빼앗음)를 구분해서 보여 줍니다. 자발/비자발 비율이 시스템의 부하를 짐작하게 합니다.

### 2단계: `nice`로 우선순위 조정

```bash
# 일반 우선순위로 한 번
nice -n 0  python3 -c "x=0
for _ in range(10**8): x+=1" &
# 더 낮은 우선순위로 한 번
nice -n 19 python3 -c "x=0
for _ in range(10**8): x+=1" &
wait
```

`top`이나 `htop`에서 두 프로세스의 CPU 점유율을 비교해 보면, `nice 19` 쪽이 시스템이 한가할 때만 양보받아 늘어나는 모습을 볼 수 있습니다.

### 3단계: CPU 어피니티로 특정 코어에 묶기

```bash
# CPU 0번에만 묶어서 실행
taskset -c 0 python3 my_workload.py
# 어피니티 마스크 확인
taskset -p $(pgrep -f my_workload.py)
```

CPU 어피니티는 캐시 친화성을 높이거나, 특정 코어를 다른 작업으로부터 보호하는 데 사용됩니다.

### 4단계: 스레드별 상태 모니터링

```bash
# 1초 단위로 특정 PID의 모든 스레드 상태 보기
ps -L -p <PID> -o pid,tid,stat,wchan,comm
```

`stat`의 `R`은 실행/실행 가능, `S`는 대기, `D`는 디스크 I/O로 인터럽트 불가능 대기를 뜻합니다. 응답성이 이상할 때 스레드 상태 분포를 보면 단서가 잡힙니다.

### 5단계: 실시간 우선순위로 스케줄링 정책 바꾸기

```bash
# 현재 정책 확인
chrt -p $(pgrep -f my_workload.py)
# 라운드 로빈 실시간 정책으로 변경 (root 권한)
sudo chrt -r -p 50 $(pgrep -f my_workload.py)
```

리눅스는 일반(SCHED_OTHER, CFS) 외에도 SCHED_FIFO/SCHED_RR 같은 실시간 정책을 제공합니다. 잘못 쓰면 시스템 응답성이 크게 무너지므로, 실시간성이 정말 필요한 경우에만 사용합니다.

## 이 코드에서 주목할 점

- 컨텍스트 스위치는 거저 일어나는 것처럼 보여도 비용이 있습니다(보통 수 마이크로초)
- `nice`는 절대 우선순위가 아니라 힌트입니다
- I/O 대기 상태(`S`, `D`)와 실행 가능 상태(`R`)의 구분이 디버깅에 결정적입니다
- 실시간 정책은 강력하지만 시스템 전체의 응답성을 망칠 수 있습니다

## 자주 하는 실수 5가지

| 실수 | 문제 | 해결 |
| --- | --- | --- |
| CPU 사용률만 보기 | 컨텍스트 스위치/대기 시간 놓침 | `vmstat`, `pidstat`로 종합 관찰 |
| 스레드를 늘리면 항상 빠르다 가정 | 컨텍스트 스위치 폭주로 오히려 느려짐 | 코어 수 근방에서 멈춰보고 측정 |
| `nice -n -20`이 모든 걸 빠르게 해줄 거라 기대 | 다른 작업 굶기고 시스템 불안정 | 진짜 필요한 만큼만, 결과 측정 |
| 컨테이너 CPU 제한 무시 | cgroup throttling으로 응답성 폭락 | `cpu.cfs_quota_us` 등 한계 인지 |
| `D` 상태 프로세스를 그냥 죽이려 함 | `kill`이 안 듣고 좀비처럼 남음 | 디스크/네트워크 원인부터 해결 |

## 실무에서는 이렇게 쓰입니다

- 컨테이너 운영: 쿠버네티스의 CPU request/limit가 cgroup 스케줄러로 매핑
- 데이터베이스 튜닝: I/O 대기와 CPU 사용의 비율로 병목 진단
- 게임/오디오: 낮은 지연이 필요한 스레드를 실시간 우선순위로 운영
- 백그라운드 잡: ETL 워커를 `nice`로 낮춰 사용자 응답성 보호
- 노트북 전력 관리: 스케줄러가 코어 주파수와 협력해 절전과 성능 균형

## 시니어 엔지니어는 이렇게 생각합니다

시니어 엔지니어는 "느리다"는 보고를 받으면 단순히 "CPU 100%인가"를 묻지 않습니다. CPU가 100%인 것보다 더 무서운 것은 CPU가 50%인데 컨텍스트 스위치가 초당 수십만 번 일어나는 상황입니다. 시니어는 `vmstat`, `pidstat`, `perf sched` 같은 도구로 스케줄러의 시점에서 시스템을 관찰합니다.

또한 시니어는 "스케줄러는 우리 코드의 협력자"임을 압니다. 우리 코드가 짧게 일하고 자주 양보하면 스케줄러는 부드럽게 돌아갑니다. 우리 코드가 락을 오래 잡거나 시그널을 막으면 스케줄러도 무력해집니다. 좋은 응답성은 정책 설정만큼이나 코드 패턴에서 옵니다.

## 연습 문제

1. 자신의 시스템에서 `vmstat 1`을 10초간 실행하고, `cs`(context switch) 컬럼의 평균을 계산하세요. 다음으로 무거운 빌드 작업을 백그라운드에 띄운 뒤 같은 측정을 반복해 차이를 비교합니다.

2. 같은 파이썬 스크립트를 `nice -n 0`과 `nice -n 19`로 동시에 실행하고, `top`에서 두 프로세스의 CPU 점유율 변화를 1분간 관찰하세요. 시스템이 한가할 때와 바쁠 때 어떻게 달라지는지 정리합니다.

3. 스레드 8개로 무거운 정수 연산을 하는 스크립트를 작성하고, `taskset -c 0` (단일 코어)와 어피니티 없음(전 코어) 두 가지로 실행해 시간을 비교하세요. 결과가 직관과 다르면 이유를 설명합니다.

## 정리 및 다음 단계

스케줄러는 실행 가능한 작업 중에서 다음에 CPU를 줄 대상을 고르는 부품이고, 그 결정은 응답성·처리량·공정성·전력 사이의 트레이드오프입니다. 컨텍스트 스위치는 이 결정의 실현 비용이고, `nice`나 어피니티 같은 손잡이로 사용자도 일부 영향을 줄 수 있습니다.

다음 글에서는 여러 흐름이 같은 자원을 동시에 만질 때 발생하는 문제 — race condition을 봅니다. 스케줄러가 언제 흐름을 끊는지를 알면, 이 문제가 왜 그렇게 자주 생기는지가 자연스럽게 보입니다.
