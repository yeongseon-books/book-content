## 왜 중요한가

복잡도를 생각하는 습관은 코드 품질을 판단하는 첫 기준이 됩니다. 기능이 맞는지만 보는 단계에서 벗어나, 입력이 커질 때 비용이 어떻게 늘어나는지 설명할 수 있어야 비로소 전공다운 문제 해결이 시작됩니다.

## 한눈에 보는 개념

> 자료구조는 데이터를 담는 방식이고, 알고리즘은 그 데이터를 처리하는 절차이며, 복잡도는 그 선택의 비용표입니다.

자료구조는 데이터를 어떻게 담을지에 관한 선택이고, 알고리즘은 그 데이터를 어떤 절차로 처리할지에 관한 선택입니다. 두 선택이 함께 맞아야 복잡도를 논할 수 있고, 그 복잡도 감각이 문제 해결과 면접 준비까지 이어집니다.

## 핵심 용어

- **배열(array)**: 연속된 메모리 공간을 활용하는 자료구조입니다.
- **리스트(list)**: 연결 구조를 중심으로 생각하는 자료구조입니다.
- **트리(tree)**: 계층 구조를 표현하기 좋은 자료구조입니다.
- **그래프(graph)**: 관계를 표현하는 구조입니다.
- **복잡도(complexity)**: 입력이 커질 때 비용이 늘어나는 비율입니다.

## 실습: 미니 DSA 키트

### 1단계 — 배열 합

```python
def total(xs):
    return sum(xs)
```

가장 단순한 순회 예시입니다. 입력 전체를 한 번 훑는다는 감각이 이후 복잡도 이해의 기준점이 됩니다.

### 2단계 — 선형 탐색

```python
def find(xs, t):
    return any(x == t for x in xs)
```

정렬 여부를 모를 때 가장 자연스러운 탐색입니다. 대신 최악의 경우 끝까지 다 봐야 합니다.

### 3단계 — 이진 탐색

```python
def bsearch(xs, t):
    lo, hi = 0, len(xs) - 1
    while lo <= hi:
        m = (lo + hi) // 2
        if xs[m] == t:
            return m
        if xs[m] < t:
            lo = m + 1
        else:
            hi = m - 1
    return -1
```

정렬된 데이터라는 전제가 있으면 탐색 범위를 절반씩 줄일 수 있습니다. 복잡도 사고가 왜 중요한지 가장 직관적으로 보여 주는 예시입니다.

### 4단계 — 스택

```python
stack = []
stack.append(1)
stack.pop()
```

아주 단순하지만 많은 문제의 뼈대가 되는 구조입니다. 함수 호출, 괄호 검사, 되돌리기 같은 개념도 같은 축에서 이해할 수 있습니다.

### 5단계 — 그래프 BFS

```python
from collections import deque
def bfs(g, s):
    seen, q = {s}, deque([s])
    while q:
        u = q.popleft()
        for v in g[u]:
            if v not in seen:
                seen.add(v); q.append(v)
    return seen
```

너비 우선 탐색은 그래프에서 가까운 노드부터 넓게 확장하는 방식입니다. 최단 거리와 연결성 같은 문제에서 자주 출발점이 됩니다.

## 이 코드에서 먼저 볼 점

- 선형과 로그의 차이는 입력이 커질수록 크게 벌어집니다.
- 스택과 큐는 단순하지만 아주 기본적인 도구입니다.
- BFS는 최단 경로 문제의 출발점이 됩니다.

## 자주 하는 실수 5가지

1. **복잡도를 전혀 적어 보지 않는 일입니다.**
2. **재귀 호출의 스택 비용을 잊는 일입니다.**
3. **해시를 만능 해법처럼 쓰는 일입니다.**
4. **그래프 표현 방식을 섞어 놓고 혼동하는 일입니다.**
5. **입력 크기를 가볍게 보고 테스트를 너무 작게만 하는 일입니다.**

## 정리

자료구조와 알고리즘은 특정 시험을 위한 과목이 아니라 문제 해결의 공통 언어를 익히는 과목입니다. 데이터를 어떤 모양으로 둘지, 어떤 절차로 처리할지, 비용이 얼마나 늘어날지를 함께 설명할 수 있어야 코드 품질도 함께 올라갑니다. 다음 글에서는 코드가 실제로 돌아가는 바닥인 시스템 과목으로 넘어가겠습니다.

## 자료구조 선택은 문제 해석의 언어입니다

자료구조 과목에서 가장 많이 발생하는 오해는 "정답 자료구조를 외우면 된다"는 생각입니다. 실제로는 반대입니다. 먼저 문제의 연산 패턴을 읽고, 그 연산을 가장 싸게 수행하는 구조를 고르는 과정이 핵심입니다. 예를 들어 삽입/삭제가 빈번한지, 임의 접근이 많은지, 정렬 상태가 유지되어야 하는지, 범위 질의가 많은지에 따라 선택이 달라집니다. 이 관점을 익히면 같은 문제를 보고도 구현 이전에 시간 복잡도 상한을 예측할 수 있습니다.

아래 표는 학부 수준에서 자주 쓰는 자료구조를 연산별로 비교한 것입니다.

| 자료구조 | 접근 | 탐색 | 삽입 | 삭제 | 사용 맥락 |
| --- | --- | --- | --- | --- | --- |
| 배열 | O(1) | O(n) | O(n) | O(n) | 인덱스 접근이 핵심일 때 |
| 연결 리스트 | O(n) | O(n) | O(1)
(위치 확보 시) | O(1)
(위치 확보 시) | 삽입/삭제 위치가 이미 주어질 때 |
| 해시 테이블 | - | 평균 O(1) | 평균 O(1) | 평균 O(1) | 키 기반 빠른 조회가 필요할 때 |
| 이진 탐색 트리 | O(log n) | O(log n) | O(log n) | O(log n) | 정렬 상태 유지 + 동적 갱신 |
| 힙 | - | O(n) | O(log n) | O(log n) | 우선순위 큐, top-k |
| 그래프(인접 리스트) | - | O(V+E) | O(1)~O(V) | O(1)~O(V) | 연결 관계, 경로 문제 |

표를 해석할 때는 Big O만 보지 말고 상수 비용과 구현 복잡도도 함께 보아야 합니다. 예를 들어 해시 테이블은 평균 O(1)이지만 해시 충돌 처리, 메모리 사용량, 해시 함수 선택에 따라 실제 성능이 달라질 수 있습니다.

## 알고리즘 패러다임 비교와 문제 유형 매핑

알고리즘은 개별 기법을 따로 외우기보다 패러다임 관점에서 묶어 보는 편이 유지율이 높습니다.

| 패러다임 | 핵심 질문 | 대표 문제 | 주의할 점 |
| --- | --- | --- | --- |
| 완전 탐색 | 가능한 경우를 모두 보면 되는가 | 부분집합, 순열 | 가지치기 없으면 폭발 |
| 분할 정복 | 문제를 같은 형태의 작은 문제로 나눌 수 있는가 | 병합정렬, 퀵정렬 | 병합/결합 비용 점검 |
| 그리디 | 지금 최선이 전체 최선으로 이어지는가 | 회의실 배정, 허프만 | 교환 논증 확인 |
| 동적 계획법 | 부분 문제 해를 재사용할 수 있는가 | 배낭, LIS | 상태 정의가 절반 |
| 그래프 탐색 | 관계망에서 도달/최단/연결을 묻는가 | BFS, DFS, 다익스트라 | 표현 방식 통일 |

문제를 읽고 이 표의 첫 번째 열을 먼저 고르는 습관이 붙으면 풀이 시간이 단축됩니다. 구현 속도보다 중요한 것은 "왜 이 패러다임인지"를 설명하는 능력입니다.

## Python 구현 예시를 읽는 포인트

Python으로 자료구조를 구현할 때는 언어 제공 자료형과 직접 구현을 구분해야 합니다. 예를 들어 `list`는 동적 배열이고, 큐에는 `collections.deque`, 우선순위 큐에는 `heapq`를 쓰는 것이 일반적입니다. 학습 단계에서는 두 가지를 모두 해 보는 것이 좋습니다. 직접 구현은 내부 동작을 이해하게 하고, 표준 라이브러리는 실전 생산성을 높입니다.

실습 시 최소 검증 루프는 다음과 같습니다.

- 연산별 단위 테스트: push/pop, enqueue/dequeue, insert/search
- 경계 사례 테스트: 빈 구조, 단일 원소, 중복 키
- 복잡도 추정 메모: 최선/평균/최악 케이스 기록
- 실패 원인 분류: 자료구조 선택 오류 vs 구현 버그

이 루프를 반복하면 알고리즘 과목이 면접 대비 과목이 아니라 설계 감각 과목으로 전환됩니다.

## 복잡도 학습을 실전 습관으로 바꾸기

복잡도를 진짜로 체득하려면 풀이마다 세 줄 메모를 남기는 습관이 필요합니다. 첫째, 입력 크기 상한. 둘째, 핵심 연산 횟수. 셋째, 메모리 사용 근거입니다. 이 세 줄이 있으면 같은 정답 코드라도 왜 통과했고 왜 느려질 수 있는지 설명할 수 있습니다.

문제 풀이 순서도 중요합니다. 처음 10분은 구현하지 말고 상태 정의와 자료구조 선택만 적는 시간을 두십시오. 특히 DP 문제는 점화식보다 상태 정의가 먼저이고, 그래프 문제는 알고리즘보다 표현 방식이 먼저입니다. 이 순서를 지키면 디버깅 시간이 크게 줄어듭니다.

면접 준비에서도 동일한 원칙이 유효합니다. 정답을 맞히는 것보다 선택 이유를 구조적으로 말하는 사람이 더 신뢰를 얻습니다.

## 실행 가능한 학습 운영 원칙

아래 원칙은 전공 주제가 달라도 공통으로 적용됩니다. 첫째, 매주 하나의 "관찰 가능한 결과"를 남겨야 합니다. 결과물은 코드, 표, 짧은 리포트, 테스트 로그 중 무엇이든 좋지만 타인이 재확인할 수 있어야 합니다. 둘째, 선택 이유를 한 문장으로 기록해야 합니다. 같은 결과가 나와도 선택 근거가 없으면 다음 학습에서 재사용하기 어렵습니다. 셋째, 실패를 유형화해야 합니다. 단순히 틀렸다고 적지 말고 개념 혼동, 조건 누락, 검증 부족, 범위 과대 같은 형태로 분류하면 개선 속도가 빨라집니다.

또한 전공 학습은 "입력-처리-출력-회고"의 폐루프로 운영하는 편이 안정적입니다. 입력은 강의와 자료 조사, 처리는 문제 풀이와 구현, 출력은 산출물, 회고는 개선 기록입니다. 많은 학생이 입력과 처리에만 시간을 쓰고 출력과 회고를 생략하는데, 이 경우 학습이 누적되지 않습니다. 반대로 작은 출력이라도 꾸준히 남기면 포트폴리오, 면접, 협업 문서까지 한 번에 연결됩니다.

실제 주간 운영 예시는 다음과 같습니다.

- 월~화: 개념 정리와 핵심 문제 2개 해결
- 수~목: 구현 또는 실험 1회, 측정 지표 기록
- 금: 결과 요약 10줄 작성, 다음 주 약점 1개 선택
- 주말: 약점 보강 60분 + 문서 업데이트

이 방식의 장점은 완벽주의를 줄이고 지속 가능성을 높인다는 점입니다. 전공 학습에서 중요한 것은 단기간 최대치보다 장기간 반복 가능성입니다. 특히 학기 후반에는 과제와 시험이 겹치므로, 미리 정해 둔 최소 루틴이 없으면 우선순위가 흔들립니다.

마지막으로, 스스로에게 던질 검증 질문을 고정해 두면 품질이 올라갑니다. "이번 주 결과를 타인이 10분 안에 재현할 수 있는가", "내 선택 이유를 반대 관점에서 반박해도 버틸 수 있는가", "다음 주에 같은 문제를 더 짧게 풀 수 있는가" 같은 질문입니다. 이 질문에 답할 수 있다면 학습은 단순 반복을 넘어 실제 역량으로 축적되고 있다고 볼 수 있습니다.

## 마지막 보강: 학기 운영과 검증 습관

전공 학습을 장기적으로 유지하려면 학기 중간에 한 번, 학기 말에 한 번 구조 점검을 수행하는 것이 좋습니다. 중간 점검에서는 "진도는 따라가지만 이해가 빈약한 영역"을 찾고, 학기 말 점검에서는 "다음 과목으로 넘길 수 있는 수준인지"를 확인합니다. 이때 핵심은 감정 평가가 아니라 증거 평가입니다. 예를 들어 개념을 설명한 노트, 문제 풀이 로그, 실행 가능한 코드, 실험 결과 표, 회고 문서 같은 증거가 있으면 준비 상태를 명확히 판단할 수 있습니다.

또한 학기 운영에서 가장 흔한 실패는 우선순위 충돌입니다. 과제 마감과 시험 대비, 프로젝트 통합이 동시에 몰리면 쉽게 무너집니다. 이를 줄이려면 주간 계획에 반드시 완충 구간을 넣어야 합니다. 완충 구간은 비어 있는 시간이 아니라, 예상 밖 오류를 처리하는 시간입니다. 이 습관이 있으면 긴급 상황에서 전체 루틴이 붕괴하지 않습니다.

마지막으로 협업 과목이 있는 경우 개인 학습과 팀 산출물의 연결을 의도적으로 설계해야 합니다. 개인이 정리한 개념 노트와 팀의 의사결정 문서를 분리하지 말고, PR 설명이나 회의록에 근거를 남기는 방식으로 연결하면 학습과 결과가 동시에 강화됩니다. 이런 기록은 졸업 이후에도 포트폴리오, 면접 답변, 온보딩 문서로 재활용할 수 있는 장기 자산이 됩니다.

## 처음 질문으로 돌아가기

- **왜 자료구조와 알고리즘은 거의 모든 컴퓨터학과의 핵심 과목일까요?**
  - 자료구조는 데이터를 어떻게 보관하고 효율적으로 접근할 것인가를 다루며, 알고리즘은 그 데이터에 대해 어떤 계산 순서를 거쳐 결과를 얻을 것인가를 다룹니다. 이 둘의 결합이 실전 프로그래밍의 백본입니다.
- **자료구조를 배우는 일과 알고리즘을 배우는 일은 어떻게 다르고 왜 함께 가야 할까요?**
  - 배열은 순서가 중요한 데이터에, 해시는 빠른 검색에, 트리는 계층 구조에, 그래프는 연결 관계에 각각 최적화됩니다. 문제를 보면 자동으로 어느 자료구조를 떠올려야 하는 직관을 기르는 것이 목표입니다.
- **시간 복잡도와 공간 복잡도는 실제 코드에서 무엇을 판단하게 해 줄까요?**
  - O(n log n) 정렬, O(1) 해시 조회, O(log n) 트리 탐색 같은 복잡도 표기는 외우는 것이 아니라, 왜 그렇게 되는지 구조를 이해하고, 그것이 실전에서 얼마나 중요한지 체감하는 데 있습니다.
