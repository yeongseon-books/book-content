# Computer Science Major 101 (10/10): 졸업 전 갖춰야 할 역량

이 글은 Computer Science Major 101 시리즈의 마지막 글입니다.

졸업이 가까워질수록 많은 학생이 비슷한 불안을 느낍니다. 과목은 거의 다 들었는데, 정말 현업에 들어갈 준비가 되었는지는 잘 모르겠다는 감각입니다.


![Computer Science Major 101 10장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/computer-science-major-101/10/10-01-graduation-skill-map.ko.png)
*컴퓨터학과 전공 가이드 10장 흐름 개요*
> 졸업 전 갖춰야 할 역량의 핵심은 기술적 깊이만이 아니라, 기술을 언제 어디에 써야 하고 왜 선택했는지 설명할 수 있는 능력에 있습니다.

## 먼저 던지는 질문

- 졸업장만으로는 설명되지 않는 역량은 무엇일까요?
- 기초, 시스템, 데이터, 협업, 문서화라는 다섯 축은 왜 함께 봐야 할까요?
- 학점과 자격증만으로 실력을 설명하기 어려운 이유는 무엇일까요?

## 이 글에서 배울 것

- 기초 역량
- 시스템 감각
- 데이터 감각
- 협업 도구
- 글쓰기 능력

## 왜 중요한가

첫 직장에 들어간 뒤 처음 몇 달은 생각보다 중요합니다. 이 시기의 기본기가 이후 경력의 기준선을 만들기 때문에, 졸업 직전 점검은 단순한 자기평가가 아니라 출발점을 다지는 작업에 가깝습니다.

## 한눈에 보는 개념

> 졸업 직전 점검은 과목 이름을 세는 일이 아니라, 현업에서 바로 쓰일 다섯 축이 비어 있지 않은지 확인하는 일입니다.

다섯 축은 따로 놀지 않습니다. 기초가 있어야 시스템을 이해하고, 시스템과 데이터를 알아야 서비스를 만들고, 협업과 문서화가 있어야 팀 안에서 결과를 확장할 수 있습니다.

## 핵심 용어

- **기초 역량(fundamentals)**: 바닥 체력입니다.
- **시스템 감각(systems sense)**: 런타임을 읽는 감각입니다.
- **데이터 감각(data sense)**: 데이터를 저장하고 해석하는 감각입니다.
- **협업(collab)**: 팀으로 일하는 방식입니다.
- **문서화(writing)**: 생각과 판단을 글로 남기는 능력입니다.

## 바뀌는 지점

**Before**: 학점이 곧 실력이라고 생각합니다.

**After**: 증거가 곧 실력이라는 기준으로 자신을 봅니다.

## 실습: 졸업 자기 점검표

### 1단계 — 자료구조

```python
fund = ["array", "list", "tree", "graph", "hash"]
```

기초 역량은 결국 설명 가능한 개념 목록으로 확인할 수 있습니다. 각 항목을 말로 풀 수 있는지, 장단점을 설명할 수 있는지 점검해 보아야 합니다.

### 2단계 — 시스템

```python
sys_topics = ["process", "memory", "io", "network"]
```

시스템 감각은 디버깅 실력과 직접 연결됩니다. 코드가 어디에서 어떤 비용을 쓰는지 감을 잡는 축이기 때문입니다.

### 3단계 — 데이터

```python
data_topics = ["sql", "stats", "ml_basic"]
```

데이터 저장과 해석의 기본은 거의 모든 현대 소프트웨어 팀에서 다시 등장합니다. SQL과 통계는 생각보다 훨씬 자주 마주칩니다.

### 4단계 — 협업

```python
collab = ["git", "review", "issues", "ci"]
```

협업 도구 경험은 입사 후 적응 속도에 큰 영향을 줍니다. 혼자 작성한 코드와 팀이 함께 다루는 코드는 요구 조건이 다릅니다.

### 5단계 — 문서화

```python
docs = ["readme", "design_doc", "post_mortem"]
```

문서 목록은 곧 증거 목록입니다. 코드만으로 설명하기 어려운 판단을 문서가 보완해 주기 때문에, 글쓰기는 부가 능력이 아니라 핵심 역량에 가깝습니다.

## 이 코드에서 먼저 볼 점

- 이 목록은 자기 점검표입니다.
- 영역별로 3개에서 5개 정도면 충분합니다.
- 증거는 코드와 문서 두 곳에 함께 남습니다.

## 자주 하는 실수 5가지

1. **자격증만 나열하는 일입니다.**
2. **학점만 강조하는 일입니다.**
3. **GitHub가 비어 있는 일입니다.**
4. **문서를 한 편도 남기지 않는 일입니다.**
5. **회고 없이 경험을 흘려보내는 일입니다.**

## 실무에서는 이렇게 드러납니다

채용과 온보딩에서는 다섯 축의 균형이 중요하게 보입니다. 특정 한 축이 완전히 비어 있으면 다른 강점도 함께 흔들립니다. 코드만 잘 쓰는 것, 말만 잘하는 것, 프로젝트만 많은 것 중 어느 하나만으로는 오래 버티기 어렵습니다.

## 증거 매핑 예시

졸업 전 자기 점검에서 가장 중요한 질문은 "나는 안다고 말할 수 있는가"보다 "무엇으로 보여 줄 수 있는가"입니다. 아래처럼 역량과 증거를 연결해 두면 준비 상태가 훨씬 선명해집니다.

| 축 | 최소 증거 예시 | 점검 질문 |
| --- | --- | --- |
| 기초 | 자료구조 문제 풀이 저장소, 알고리즘 노트 | 시간 복잡도와 선택 이유를 말로 설명할 수 있는가 |
| 시스템 | 운영체제/네트워크 디버깅 메모, 성능 측정 실습 | 느린 코드가 왜 느린지 한 단계 더 아래에서 설명할 수 있는가 |
| 데이터 | SQL 과제, 간단한 분석 리포트 | 저장 구조와 조회 비용을 함께 설명할 수 있는가 |
| 협업 | GitHub PR, 이슈, 코드 리뷰 기록 | 혼자 코드를 쓰는 것과 팀으로 바꾸는 과정을 설명할 수 있는가 |
| 문서화 | README, 설계 문서, 회고 글 | 판단 근거와 배운 점을 글로 남길 수 있는가 |

이 표의 핵심은 증거의 크기가 아니라 연결입니다. 거대한 프로젝트가 없어도 괜찮습니다. 대신 각 역량 축에 최소 한 가지씩은 "직접 보여 줄 수 있는 결과"를 남겨 두는 편이 훨씬 강합니다.

## 선배 엔지니어는 이렇게 봅니다

- 결국 기본기가 가장 오래 갑니다.
- 시스템 감각은 디버깅 실력으로 이어집니다.
- 데이터는 거의 모든 팀의 공통 언어입니다.
- 협업 기록은 태도를 보여 주는 증거입니다.
- 문서는 시간이 갈수록 자산이 됩니다.

## 체크리스트

- [ ] 다섯 축으로 자기 점검을 해 보았습니다.
- [ ] 각 축에 대응하는 증거를 적어 보았습니다.
- [ ] 가장 약한 영역을 표시했습니다.
- [ ] 다음 학습 계획을 한 줄이라도 세웠습니다.

## 연습 문제

2. 시스템 감각의 의미를 한 줄로 적어 보세요.
3. 문서화 능력이 왜 중요한지 한 줄로 써 보세요.

## 정리

졸업 전에는 과목 이수 여부보다 어떤 역량을 실제로 갖추었는지 점검하는 편이 더 중요합니다. 기초, 시스템, 데이터, 협업, 문서화라는 다섯 축으로 자신을 다시 보면 어디가 비어 있는지 분명해집니다. 이 시리즈는 여기서 마무리하지만, 다음 공부는 가장 약한 축을 메우는 일에서 시작하면 됩니다.

## 졸업 전 의사결정 프레임: 대학원 vs 취업

졸업 시점의 가장 큰 선택 중 하나는 대학원 진학과 취업 사이의 경로 결정입니다. 이 선택은 정답 문제가 아니라 적합도 문제이므로, 감정적 선호만으로 결정하면 후회 확률이 높습니다. 핵심은 본인의 강점, 목표 시간축, 재정 조건, 연구/실무 선호를 동시에 평가하는 것입니다.

아래 비교표는 의사결정 초기 점검에 유용한 최소 프레임입니다.

| 판단 기준 | 대학원 진학 우세 신호 | 취업 우세 신호 |
| --- | --- | --- |
| 관심 문제 성격 | 장기 연구 질문이 명확함 | 제품/서비스 문제 해결 선호 |
| 성과 선호 | 논문, 실험, 재현성 중심 | 출시, 운영, 사용자 지표 중심 |
| 시간 투자 | 2~5년의 집중 학습 의지 | 빠른 현장 경험 축적 선호 |
| 불확실성 감수 | 탐색 실패를 학습으로 수용 | 실행 결과를 빠르게 확인 선호 |
| 재정/생활 조건 | 장기 계획 가능 | 조기 수입 필요성 높음 |

표의 목적은 즉시 결론이 아니라 자기 인식입니다. 각 항목에 대해 본인의 현재 상태를 솔직히 적어 보면 경로 적합도가 선명해집니다.

## 전공 심화 로드맵(졸업 직후 12개월)

경로가 달라도 공통적으로 필요한 심화 축은 존재합니다. 아래 로드맵은 취업/진학 모두에 유효한 1년 프레임입니다.

- 1~3개월: 자료구조·시스템·SQL 핵심 복습, 약점 정리
- 4~6개월: 관심 도메인 프로젝트 1개 완성, 문서화
- 7~9개월: 성능/품질 개선 라운드 1회, 측정 기록
- 10~12개월: 결과 공개(블로그, 포트폴리오, 발표) 및 피드백 반영

중요한 것은 학습량보다 폐루프입니다. 입력(학습)-출력(산출물)-피드백(개선)이 한 번이라도 완결되어야 실력이 자산으로 남습니다.

## 최종 점검 체크리스트

졸업 직전에는 불안 때문에 새 기술을 무작정 늘리기 쉽습니다. 이 시점에는 확장보다 정리가 더 중요합니다.

- 내가 풀고 싶은 문제를 한 문장으로 말할 수 있는가.
- 최근 1년 결과물 중 다시 설명 가능한 프로젝트가 2개 이상 있는가.
- 코드 외 증거(문서, 테스트, 회고, 지표)가 함께 남아 있는가.
- 약점 축을 인정하고 다음 3개월 보강 계획을 적었는가.

이 네 가지가 채워지면 경로 선택 이후 실행력이 훨씬 높아집니다.

## 전공 역량을 다음 3개월 행동으로 전환하기

자기 점검은 목록 작성으로 끝나면 효과가 약합니다. 약한 축 하나를 고른 뒤 3개월 실행 계획으로 바꿔야 합니다. 예를 들어 시스템 축이 약하다면 월별 목표를 "1개월차: 프로세스/메모리 복습", "2개월차: 성능 측정 실습 2개", "3개월차: 개선 리포트 1편"처럼 구체화합니다.

취업과 대학원 선택 모두 결국 실행력으로 검증됩니다. 따라서 선택 직후에는 정보 탐색을 줄이고 산출물 생산으로 전환하는 것이 중요합니다. 코드, 문서, 발표 자료 중 하나라도 매달 완성본을 남기면 불안이 줄고 의사결정의 질이 올라갑니다.

## 실행 가능한 학습 운영 원칙

아래 원칙은 전공 주제가 달라도 공통으로 적용됩니다. 첫째, 매주 하나의 "관찰 가능한 결과"를 남겨야 합니다. 결과물은 코드, 표, 짧은 리포트, 테스트 로그 중 무엇이든 좋지만 타인이 재확인할 수 있어야 합니다. 둘째, 선택 이유를 한 문장으로 기록해야 합니다. 같은 결과가 나와도 선택 근거가 없으면 다음 학습에서 재사용하기 어렵습니다. 셋째, 실패를 유형화해야 합니다. 단순히 틀렸다고 적지 말고 개념 혼동, 조건 누락, 검증 부족, 범위 과대 같은 형태로 분류하면 개선 속도가 빨라집니다.

또한 전공 학습은 "입력-처리-출력-회고"의 폐루프로 운영하는 편이 안정적입니다. 입력은 강의와 자료 조사, 처리는 문제 풀이와 구현, 출력은 산출물, 회고는 개선 기록입니다. 많은 학생이 입력과 처리에만 시간을 쓰고 출력과 회고를 생략하는데, 이 경우 학습이 누적되지 않습니다. 반대로 작은 출력이라도 꾸준히 남기면 포트폴리오, 면접, 협업 문서까지 한 번에 연결됩니다.

실제 주간 운영 예시는 다음과 같습니다.

- 월~화: 개념 정리와 핵심 문제 2개 해결
- 수~목: 구현 또는 실험 1회, 측정 지표 기록
- 금: 결과 요약 10줄 작성, 다음 주 약점 1개 선택
- 주말: 약점 보강 60분 + 문서 업데이트

이 방식의 장점은 완벽주의를 줄이고 지속 가능성을 높인다는 점입니다. 전공 학습에서 중요한 것은 단기간 최대치보다 장기간 반복 가능성입니다. 특히 학기 후반에는 과제와 시험이 겹치므로, 미리 정해 둔 최소 루틴이 없으면 우선순위가 흔들립니다.

마지막으로, 스스로에게 던질 검증 질문을 고정해 두면 품질이 올라갑니다. "이번 주 결과를 타인이 10분 안에 재현할 수 있는가", "내 선택 이유를 반대 관점에서 반박해도 버틸 수 있는가", "다음 주에 같은 문제를 더 짧게 풀 수 있는가" 같은 질문입니다. 이 질문에 답할 수 있다면 학습은 단순 반복을 넘어 실제 역량으로 축적되고 있다고 볼 수 있습니다.

## 마지막 보강: 학기 운영과 검증 습관

전공 학습을 장기적으로 유지하려면 학기 중간에 한 번, 학기 말에 한 번 구조 점검을 수행하는 것이 좋습니다. 중간 점검에서는 "진도는 따라가지만 이해가 빈약한 영역"을 찾고, 학기 말 점검에서는 "다음 과목으로 넘길 수 있는 수준인지"를 확인합니다. 이때 핵심은 감정 평가가 아니라 증거 평가입니다. 예를 들어 개념을 설명한 노트, 문제 풀이 로그, 실행 가능한 코드, 실험 결과 표, 회고 문서 같은 증거가 있으면 준비 상태를 명확히 판단할 수 있습니다.

또한 학기 운영에서 가장 흔한 실패는 우선순위 충돌입니다. 과제 마감과 시험 대비, 프로젝트 통합이 동시에 몰리면 쉽게 무너집니다. 이를 줄이려면 주간 계획에 반드시 완충 구간을 넣어야 합니다. 완충 구간은 비어 있는 시간이 아니라, 예상 밖 오류를 처리하는 시간입니다. 이 습관이 있으면 긴급 상황에서 전체 루틴이 붕괴하지 않습니다.

마지막으로 협업 과목이 있는 경우 개인 학습과 팀 산출물의 연결을 의도적으로 설계해야 합니다. 개인이 정리한 개념 노트와 팀의 의사결정 문서를 분리하지 말고, PR 설명이나 회의록에 근거를 남기는 방식으로 연결하면 학습과 결과가 동시에 강화됩니다. 이런 기록은 졸업 이후에도 포트폴리오, 면접 답변, 온보딩 문서로 재활용할 수 있는 장기 자산이 됩니다.

## 짧은 실천 과제

이번 주 안에 적용할 수 있는 가장 작은 과제는 세 가지입니다. 첫째, 본문에서 다룬 핵심 개념 하나를 5문장으로 요약합니다. 둘째, 해당 개념이 실제 코드 또는 과제에서 어떻게 나타나는지 예시를 1개 적습니다. 셋째, 이해가 부족한 지점을 질문 형태로 1개 남깁니다. 이 세 단계는 분량이 작지만 학습의 방향을 잃지 않게 해 주며, 다음 학습 행동을 구체적으로 만들어 줍니다.

중요한 점은 완벽한 답안을 만드는 것이 아니라, 다음 반복에서 개선 가능한 초안을 남기는 것입니다. 반복 가능한 초안이 쌓일수록 전공 학습은 단발성 이벤트가 아니라 누적형 성장으로 전환됩니다.

졸업 직전에는 대학원 지원 일정과 취업 준비가 충돌하기 쉽습니다. 아래 타임라인을 기준으로 병행 구간을 설계하면 우선순위 충돌을 줄일 수 있습니다.

## 부록: 전공 운영용 템플릿 묶음

아래 부록은 학기 중 실제로 바로 쓸 수 있는 공용 템플릿입니다. 과목이 달라도 동일한 형식을 유지하면 비교와 개선이 쉬워집니다.

### 1) 커리큘럼 맵 작성 템플릿

| 학기 | 필수 과목 | 선택 과목 | 선행 의존성 | 산출물 |
| --- | --- | --- | --- | --- |
| 1-1 | 기초수학, 프로그래밍입문 | 교양 1 | 없음 | 요약 노트, 코드 1개 |
| 1-2 | 이산수학, 자료구조 기초 | 교양 1 | 1-1 과목 | 오답 노트, 미니 프로젝트 |
| 2-1 | 알고리즘, 컴퓨터구조 | 전공선택 1 | 이산수학, 프로그래밍 | 성능 비교표 |
| 2-2 | 운영체제, 데이터베이스 | 전공선택 1 | 자료구조, 컴퓨터구조 | 실험 리포트 |
| 3-1 | 네트워크, 소프트웨어공학 | 전공선택 2 | 운영체제, 데이터베이스 | 팀 과제 문서 |
| 3-2 | AI/데이터 과목 | 전공선택 2 | 선형대수, 확률통계 | 실험 로그 |
| 4-1 | 캡스톤 1 | 전공심화 | 3학년 핵심 과목 | 중간 데모 |
| 4-2 | 캡스톤 2 | 취업/진학 준비 과목 | 캡스톤 1 | 최종 발표, 포트폴리오 |

### 2) 과목 의존성 다이어그램 해설 템플릿

- 기초 수학 -> 알고리즘/AI: 증명과 최적화 이해의 기반입니다.
- 프로그래밍 기초 -> 시스템/데이터: 구현 속도와 디버깅 품질의 기반입니다.
- 시스템 과목 -> 성능/운영: 병목 원인 추론의 기반입니다.
- 데이터 과목 -> 서비스 신뢰성: 상태 관리와 질의 비용 통제의 기반입니다.
- 프로젝트 과목 -> 진로 연결: 앞선 지식을 통합해 보여 주는 증거입니다.

### 4) GPA 계산과 리스크 점검 예시

```text
학기 GPA(예시) = (A+ 3학점×4.5 + A0 3학점×4.0 + B+ 3학점×3.5 + B0 3학점×3.0) / 12
               = (13.5 + 12.0 + 10.5 + 9.0) / 12 = 45.0 / 12 = 3.75
```

GPA 계산의 목적은 단순 점수 확인이 아니라 학습 리스크 탐지입니다. 목표 GPA 대비 위험 과목을 조기에 찾고 시간을 재배치하는 것이 핵심입니다.

### 5) 논문 읽기 가이드(학부용)

1. 초록에서 문제 정의와 기여를 한 문장으로 요약합니다.
2. 실험 설정에서 데이터셋, 분할, 지표를 먼저 확인합니다.
3. 기준선 대비 개선폭을 표로 옮기고 과장 여부를 점검합니다.
4. 실패 사례와 한계 항목을 별도 메모로 기록합니다.
5. 내 환경에서 재현 가능한 최소 실험 하나로 축소합니다.

### 7) 검증 체크리스트

- [ ] 이번 주 결과를 타인이 10분 안에 재현할 수 있습니다.
- [ ] 선택 이유를 한 문장으로 설명할 수 있습니다.
- [ ] 실패 원인을 유형화해 다음 반복 계획에 반영했습니다.
- [ ] 코드와 문서가 같은 결론을 가리키도록 정리했습니다.
- [ ] 다음 2주 계획이 현재 약점을 직접 보강하도록 구성되었습니다.


### 추가 실전 메모

전공 학습에서 가장 중요한 것은 한 번의 고점이 아니라 반복 가능한 기준입니다. 같은 과목을 다시 배울 필요가 없도록, 매주 산출물 1개와 회고 1개를 남기는 습관을 고정하십시오. 산출물은 코드, 표, 다이어그램, 짧은 글 중 무엇이든 괜찮지만 타인이 재검증할 수 있어야 합니다.

또한 학기 중반 이후에는 새로운 자료를 계속 늘리기보다 기존 노트와 실습을 정리하는 시간이 더 큰 효과를 냅니다. 이미 배운 내용을 구조화해 두면 시험, 프로젝트, 면접 준비가 서로 분리되지 않고 하나의 자산으로 축적됩니다.

마지막으로 스터디나 팀 과제에서는 역할을 기술 스택이 아니라 책임 단위로 나누는 편이 안전합니다. 예를 들어 API 구현 담당, 데이터 검증 담당, 테스트 자동화 담당, 문서화 담당처럼 나누면 누락과 중복을 줄일 수 있습니다. 이 방식은 학부 프로젝트뿐 아니라 실무 온보딩에서도 그대로 사용됩니다.

## 처음 질문으로 돌아가기

- **졸업장만으로는 설명되지 않는 역량은 무엇일까요?**
  - 본문의 기준은 졸업 전 갖춰야 할 역량를 한 덩어리 개념으로 보지 않고 입력, 처리, 검증, 운영 신호가 만나는 경계로 나누어 확인하는 것입니다.
- **기초, 시스템, 데이터, 협업, 문서화라는 다섯 축은 왜 함께 봐야 할까요?**
  - 예제와 그림에서는 어떤 값이 들어오고, 어느 단계에서 바뀌며, 어떤 기준으로 통과 또는 실패하는지를 먼저 확인해야 합니다.
- **학점과 자격증만으로 실력을 설명하기 어려운 이유는 무엇일까요?**
  - 운영에서는 이 판단을 체크리스트, 로그, 테스트로 남겨 다음 변경에서도 같은 실패가 반복되지 않게 막아야 합니다.
