# Computer Science Major 101 (4/10): 시스템 과목 이해하기

코드를 오래 쓰다 보면 결국 같은 질문으로 돌아옵니다. 이 한 줄이 왜 이렇게 동작하는지, 왜 느린지, 왜 같은 코드인데 환경에 따라 다르게 보이는지 궁금해지는 순간이 옵니다.

이 글은 Computer Science Major 101 시리즈의 4번째 글입니다.

## 먼저 던지는 질문

- 시스템 과목은 코드 한 줄이 실제로 어떻게 동작하는지 어디까지 설명해 줄까요?
- 운영체제, 컴퓨터 구조, 컴파일러는 왜 함께 배워야 할까요?
- 프로세스, 메모리, 파일, 시간 같은 개념이 디버깅과 성능 분석에 왜 중요할까요?

이 그림에서 보이듯 시스템 과목들은 운영체제부터 컴파일러까지 계산기 위에서 코드가 어떻게 실행되는지를 아래에서 위로 차근차근 풀어봅니다. 각 계층이 위 계층을 위해 무엇을 제공하는지 이해하는 것이 중요합니다.

> 시스템 과목들의 핵심은 개념과 실행 세부사항을 함께 이해해, 추상과 현실 사이의 갭을 메우는 데 있습니다.

## 이 글에서 배울 것

- 운영체제의 의미
- 컴퓨터 구조의 기본 관점
- 컴파일러의 역할
- 시스템 프로그래밍 감각
- 더 나은 디버깅 능력

## 왜 중요한가

성능 문제와 장애 원인 분석은 거의 항상 시스템 지식 위에서 이루어집니다. CPU, 메모리, I/O, 프로세스, 시스템 호출을 모르면 코드 바깥에서 벌어지는 비용을 읽기가 어렵습니다.

## 한눈에 보는 개념

> 코드는 운영체제와 하드웨어라는 무대 위에서만 실행되므로, 시스템 과목은 그 무대를 읽는 법을 가르칩니다.

코드는 진공 속에서 실행되지 않습니다. 하드웨어 위에 구조가 있고, 운영체제가 자원을 배분하고, 컴파일러가 코드를 번역한 뒤에야 프로그램이 실제로 돌아갑니다. 시스템 과목은 이 무대를 읽는 훈련입니다.

## 핵심 용어

- **운영체제(OS)**: 자원을 관리하는 소프트웨어입니다.
- **프로세스(process)**: 실행 중인 프로그램 단위입니다.
- **스레드(thread)**: 더 가벼운 실행 단위입니다.
- **레지스터(register)**: 가장 빠른 메모리 영역입니다.
- **컴파일러(compiler)**: 코드를 번역하는 도구입니다.

## Before/After

**Before**: 코드는 그냥 돌아간다고 생각합니다.

**After**: CPU, 메모리, 운영체제가 아래에서 어떻게 관여하는지 보입니다.

## 실습: 시스템 감각 만들기

### 1단계 — 프로세스 ID

```python
import os
print(os.getpid())
```

프로그램은 실행되는 순간 운영체제 안에서 하나의 프로세스로 다뤄집니다. 고유 ID가 있다는 사실만으로도 프로그램을 추상적인 코드가 아니라 시스템 자원으로 보게 됩니다.

### 2단계 — 환경 변수

```python
print(os.environ.get("PATH", ""))
```

환경 변수는 프로세스 단위의 실행 맥락입니다. 로컬에서는 되는데 서버에서는 안 되는 문제 중 상당수가 이런 실행 환경 차이와 연결됩니다.

### 3단계 — 파일 디스크립터

```python
with open("/etc/hostname") as f:
    print(f.fileno())
```

파일도 운영체제가 관리하는 자원입니다. 숫자 하나처럼 보이는 디스크립터 뒤에 입출력 모델과 자원 한계가 숨어 있습니다.

### 4단계 — 시간 측정

```python
import time
t = time.perf_counter()
sum(range(10_000_000))
print(time.perf_counter() - t)
```

성능은 느낌으로 판단하지 않습니다. 시간을 재는 순간부터 코드의 비용을 시스템 관점에서 보기 시작할 수 있습니다.

### 5단계 — 메모리 감각

```python
import sys
print(sys.getsizeof([0] * 1000))
```

같은 코드라도 메모리를 얼마나 쓰는지 모르면 비용을 절반만 보고 있는 셈입니다. 시스템 과목은 바로 이런 감각을 붙이는 데 도움을 줍니다.

## 이 코드에서 먼저 볼 점

- 프로세스에는 고유 ID가 있습니다.
- 환경은 프로세스 단위로 붙습니다.
- 시간 측정과 자원 사용은 시스템 감각의 출발점입니다.

## 자주 하는 실수 5가지

1. **메모리 주소와 값을 같은 것으로 보는 일입니다.**
2. **프로세스와 스레드를 혼동하는 일입니다.**
3. **스택과 힙을 섞어 이해하는 일입니다.**
4. **버퍼링이 동작에 미치는 영향을 잊는 일입니다.**
5. **시스템 호출 비용을 너무 가볍게 보는 일입니다.**

## 실무에서는 이렇게 드러납니다

장애 보고서를 읽다 보면 근본 원인이 애플리케이션 코드보다 OS 자원 한계로 이어지는 경우가 많습니다. CPU 사용량, 메모리 부족, 파일 디스크립터 고갈, 느린 I/O 같은 문제는 모두 시스템 과목에서 배우는 기본 개념 위에서 읽힙니다.

## 선배 엔지니어는 이렇게 봅니다

- 코드는 결국 기계 위에서 돌아갑니다.
- 비용은 CPU와 메모리로 나타납니다.
- 동시성의 실제 조건은 운영체제가 정합니다.
- 컴파일러는 실행 형태를 바꿉니다.
- 시스템 호출은 생각보다 비쌉니다.

## 정리

시스템 과목은 코드를 더 낮은 층에서 읽게 만드는 훈련입니다. 운영체제, 컴퓨터 구조, 컴파일러, 시스템 프로그래밍 개념이 손에 잡히기 시작하면 디버깅 방식과 성능을 보는 시선도 함께 달라집니다. 다음 글에서는 저장과 전달을 맡는 데이터베이스와 네트워크로 이어가겠습니다.

## 시스템 과목을 실행 경로 관점으로 묶기

시스템 과목은 범위가 넓어 보이지만 핵심 질문은 하나입니다. "내 코드가 CPU에서 어떤 경로로 실행되고 어떤 자원을 소모하는가"입니다. 운영체제, 컴퓨터구조, 시스템 프로그래밍, 컴파일러를 따로 배우더라도 이 질문으로 묶으면 지식이 분산되지 않습니다. 예를 들어 웹 요청 하나가 들어왔을 때 프로세스 스케줄링, 컨텍스트 스위치, 메모리 할당, 시스템 콜, 네트워크 I/O, 파일 접근이 어떤 순서로 발생하는지 말로 추적할 수 있어야 합니다.

아래 표는 OS 핵심 개념을 실무 디버깅 신호와 연결해 놓은 매핑표입니다.

| OS 핵심 개념 | 수업에서 배우는 내용 | 실무에서 보이는 신호 | 점검 명령/지표 |
| --- | --- | --- | --- |
| 프로세스/스레드 | 스케줄링, 실행 단위 | CPU 급등, 응답 지연 | `top`, `ps`, 런큐 길이 |
| 메모리 가상화 | 페이지, 캐시, 스왑 | OOM, GC 지연, 페이지 폴트 | RSS, page fault rate |
| 파일 시스템 | inode, 버퍼 캐시 | 디스크 I/O 병목 | iowait, latency |
| 시스템 콜 | user/kernel 경계 | syscall 폭증, context switch 증가 | strace, perf |
| 동기화 | 락, 세마포어 | 데드락, 처리량 저하 | lock wait time |

이 표를 기준으로 보면 "느리다"라는 증상을 더 구체화할 수 있습니다. CPU-bound인지 I/O-bound인지 먼저 구분하면 해결 방향이 완전히 달라집니다.

## 시스템 콜 예시를 통해 비용 감각 만들기

고수준 언어 코드 한 줄도 내부적으로는 다수의 시스템 콜을 유발할 수 있습니다. 파일 읽기, 소켓 송수신, 프로세스 생성은 모두 사용자 공간에서 커널 공간으로 넘어가는 경계를 통과합니다. 이 경계는 기능적으로 필수이지만 비용이 있습니다. 그래서 반복 루프 안에서 불필요한 파일 열기/닫기, 잦은 작은 네트워크 송신, 과도한 프로세스 생성은 성능에 즉시 영향을 줍니다.

실습 시 추천하는 관찰 포인트는 다음과 같습니다.

- 같은 기능을 배치 I/O와 개별 I/O로 나눠 실행 시간 비교
- 문자열 처리보다 시스템 콜 빈도가 병목인지 확인
- 멀티스레드 도입 전 락 경합 가능성 먼저 점검
- 측정 없는 최적화 금지: 변경 전후 지표 기록

이 네 가지 원칙만 지켜도 시스템 과목에서 배운 개념이 성능 개선 작업으로 자연스럽게 이어집니다.

## 시스템 학습 로드맵(학부 기준)

1. 프로세스/스레드/메모리 기본 개념을 그림으로 설명할 수 있게 만듭니다.
2. 간단한 서버를 실행하고 요청 처리 경로를 추적해 봅니다.
3. I/O 병목 예제를 만들어 측정 도구 사용법을 익힙니다.
4. 컴파일/링크 과정과 바이너리 실행 흐름을 연결해 봅니다.
5. 프로젝트에서 한 번 이상 성능 이슈를 재현하고 개선 기록을 남깁니다.

여기까지 오면 시스템 과목이 추상적 암기 과목이 아니라 "문제를 좁히는 기술"로 바뀝니다.

## 시스템 관점 디버깅 체크리스트

시스템 문제가 의심될 때는 코드 수정 전에 관찰 순서를 고정해야 합니다. 1) 증상 재현 조건 확인, 2) CPU/메모리/I/O 중 병목 축 분류, 3) 로그 타임라인 정렬, 4) 시스템 콜/네트워크 경계 확인, 5) 변경 전후 측정 비교. 이 순서를 지키면 감각적 추측을 줄이고 원인을 빠르게 좁힐 수 있습니다.

학부 수준에서는 완벽한 튜닝보다 "측정 가능한 가설"을 세우는 연습이 더 중요합니다. 예를 들어 "파일을 한 줄씩 읽는 루프가 병목"이라는 가설을 세웠다면 배치 읽기 버전과 시간을 비교해 근거를 남겨야 합니다. 이런 기록이 쌓이면 시스템 과목은 암기 과목이 아니라 실험 과목으로 바뀝니다.

## 실행 가능한 학습 운영 원칙

아래 원칙은 전공 주제가 달라도 공통으로 적용됩니다. 첫째, 매주 하나의 "관찰 가능한 결과"를 남겨야 합니다. 결과물은 코드, 표, 짧은 리포트, 테스트 로그 중 무엇이든 좋지만 타인이 재확인할 수 있어야 합니다. 둘째, 선택 이유를 한 문장으로 기록해야 합니다. 같은 결과가 나와도 선택 근거가 없으면 다음 학습에서 재사용하기 어렵습니다. 셋째, 실패를 유형화해야 합니다. 단순히 틀렸다고 적지 말고 개념 혼동, 조건 누락, 검증 부족, 범위 과대 같은 형태로 분류하면 개선 속도가 빨라집니다.

또한 전공 학습은 "입력-처리-출력-회고"의 폐루프로 운영하는 편이 안정적입니다. 입력은 강의와 자료 조사, 처리는 문제 풀이와 구현, 출력은 산출물, 회고는 개선 기록입니다. 많은 학생이 입력과 처리에만 시간을 쓰고 출력과 회고를 생략하는데, 이 경우 학습이 누적되지 않습니다. 반대로 작은 출력이라도 꾸준히 남기면 포트폴리오, 면접, 협업 문서까지 한 번에 연결됩니다.

실제 주간 운영 예시는 다음과 같습니다.

- 월~화: 개념 정리와 핵심 문제 2개 해결
- 수~목: 구현 또는 실험 1회, 측정 지표 기록
- 금: 결과 요약 10줄 작성, 다음 주 약점 1개 선택
- 주말: 약점 보강 60분 + 문서 업데이트

이 방식의 장점은 완벽주의를 줄이고 지속 가능성을 높인다는 점입니다. 전공 학습에서 중요한 것은 단기간 최대치보다 장기간 반복 가능성입니다. 특히 학기 후반에는 과제와 시험이 겹치므로, 미리 정해 둔 최소 루틴이 없으면 우선순위가 흔들립니다.

마지막으로, 스스로에게 던질 검증 질문을 고정해 두면 품질이 올라갑니다. "이번 주 결과를 타인이 10분 안에 재현할 수 있는가", "내 선택 이유를 반대 관점에서 반박해도 버틸 수 있는가", "다음 주에 같은 문제를 더 짧게 풀 수 있는가" 같은 질문입니다. 이 질문에 답할 수 있다면 학습은 단순 반복을 넘어 실제 역량으로 축적되고 있다고 볼 수 있습니다.

## 마지막 보강: 학기 운영과 검증 습관

전공 학습을 장기적으로 유지하려면 학기 중간에 한 번, 학기 말에 한 번 구조 점검을 수행하는 것이 좋습니다. 중간 점검에서는 "진도는 따라가지만 이해가 빈약한 영역"을 찾고, 학기 말 점검에서는 "다음 과목으로 넘길 수 있는 수준인지"를 확인합니다. 이때 핵심은 감정 평가가 아니라 증거 평가입니다. 예를 들어 개념을 설명한 노트, 문제 풀이 로그, 실행 가능한 코드, 실험 결과 표, 회고 문서 같은 증거가 있으면 준비 상태를 명확히 판단할 수 있습니다.

또한 학기 운영에서 가장 흔한 실패는 우선순위 충돌입니다. 과제 마감과 시험 대비, 프로젝트 통합이 동시에 몰리면 쉽게 무너집니다. 이를 줄이려면 주간 계획에 반드시 완충 구간을 넣어야 합니다. 완충 구간은 비어 있는 시간이 아니라, 예상 밖 오류를 처리하는 시간입니다. 이 습관이 있으면 긴급 상황에서 전체 루틴이 붕괴하지 않습니다.

마지막으로 협업 과목이 있는 경우 개인 학습과 팀 산출물의 연결을 의도적으로 설계해야 합니다. 개인이 정리한 개념 노트와 팀의 의사결정 문서를 분리하지 말고, PR 설명이나 회의록에 근거를 남기는 방식으로 연결하면 학습과 결과가 동시에 강화됩니다. 이런 기록은 졸업 이후에도 포트폴리오, 면접 답변, 온보딩 문서로 재활용할 수 있는 장기 자산이 됩니다.

## 짧은 실천 과제

이번 주 안에 적용할 수 있는 가장 작은 과제는 세 가지입니다. 첫째, 본문에서 다룬 핵심 개념 하나를 5문장으로 요약합니다. 둘째, 해당 개념이 실제 코드 또는 과제에서 어떻게 나타나는지 예시를 1개 적습니다. 셋째, 이해가 부족한 지점을 질문 형태로 1개 남깁니다. 이 세 단계는 분량이 작지만 학습의 방향을 잃지 않게 해 주며, 다음 학습 행동을 구체적으로 만들어 줍니다.

중요한 점은 완벽한 답안을 만드는 것이 아니라, 다음 반복에서 개선 가능한 초안을 남기는 것입니다. 반복 가능한 초안이 쌓일수록 전공 학습은 단발성 이벤트가 아니라 누적형 성장으로 전환됩니다.


## 마무리 보완 메모

학습 계획은 길게 적는 것보다 실행 가능하게 적는 편이 더 효과적입니다. 이번 단원에서는 개념 이해, 예시 적용, 질문 정리를 하나의 반복 루프로 묶는 것이 핵심입니다. 특히 다음 과목으로 넘어가기 전에 최소 한 번은 본문 핵심을 자신의 언어로 다시 설명해 보고, 실제 과제나 코드에서 동일 개념을 확인해 두면 전이 학습이 쉬워집니다. 작은 반복을 꾸준히 수행하는 습관이 결국 큰 격차를 만듭니다.


간단한 확인 항목으로는 개념을 말로 설명하기, 예시를 코드로 재현하기, 그리고 측정 결과를 한 줄로 남기기가 있습니다. 이 세 가지를 완료하면 단원 이해가 기억이 아니라 실행 수준으로 전환됩니다.

## 처음 질문으로 돌아가기

- **시스템 과목은 코드 한 줄이 실제로 어떻게 동작하는지 어디까지 설명해 줄까요?**
  - 운영체제는 하드웨어와 애플리케이션 사이에서 자원을 중재하고, 컴퓨터 아키텍처는 CPU가 명령을 어떻게 실행하는지를 설명하고, 컴파일러는 고급 언어를 기계어로 어떻게 변환하는지를 보여줍니다.
- **운영체제, 컴퓨터 구조, 컴파일러는 왜 함께 배워야 할까요?**
  - 프로세스와 스레드의 차이, 메모리 계층(캐시, RAM, 디스크), 파이프라인 해저드 같은 개념들은 추상 개념이 아니라 실제 하드웨어가 어떻게 동작하는지를 설명하는 것입니다.
- **프로세스, 메모리, 파일, 시간 같은 개념이 디버깅과 성능 분석에 왜 중요할까요?**
  - 시스템 깊이를 이해하면 '왜 이 코드가 느린가', '왜 이 버그가 나는가'를 디버깅할 때 추론이 다릅니다. 이것이 초급과 경력자의 가장 큰 차이 중 하나입니다.
