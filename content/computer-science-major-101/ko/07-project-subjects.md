# 프로젝트 과목

> 컴퓨터학과 전공 학습 가이드 101 시리즈 (7/10)

## 이 글에서 다룰 문제

- 왜 프로젝트 과목은 전공 후반부의 핵심으로 여겨질까요?
- 개인 과제와 팀 프로젝트는 무엇이 다르고 무엇을 더 요구할까요?
- 문제 정의, 범위 조절, 일정 관리, 시연 준비는 왜 모두 중요할까요?
- 프로젝트 과목이 포트폴리오와 취업 준비로 이어지는 이유는 무엇일까요?

전공 후반부로 가면 학생들은 점점 같은 질문을 하기 시작합니다. 이제까지 배운 것을 어디에 써 보아야 할까요? 자료구조, 데이터베이스, 네트워크, AI를 각각 따로 공부하는 동안에는 지식이 조각처럼 느껴질 수 있습니다. 그런데 프로젝트 과목에 들어가면 그 조각들이 한꺼번에 맞물리기 시작합니다.

프로젝트 과목이 중요한 이유는 지식을 시험 답안으로만 확인하지 않기 때문입니다. **문제를 정의하고, 팀과 협업하고, 일정 안에 결과물을 만드는 과정**이 전부 평가 대상이 됩니다. 그래서 많은 학생이 이때 처음으로 작은 제품을 만드는 감각을 얻게 됩니다.

이 글에서는 프로젝트 과목이 무엇을 요구하는지, 왜 전공 후반의 중심인지, 그리고 작은 계획을 세울 때 어떤 항목을 챙겨야 하는지 살펴보겠습니다.

## 이 글에서 배울 것

- 프로젝트 과목의 목적과 역할
- 팀 구성과 역할 분담의 기본 원칙
- 범위 조절과 산출물 정리 방법
- 시연과 회고가 중요한 이유

## 왜 중요한가

많은 학생에게 첫 포트폴리오는 프로젝트 과목에서 나옵니다. 개인 공부 기록도 의미가 있지만, 문제 정의부터 구현, 테스트, 발표까지 이어지는 결과물은 훨씬 강한 증거가 됩니다. 무엇을 만들었는지뿐 아니라 어떤 판단을 했는지도 함께 보여 줄 수 있기 때문입니다.

또한 프로젝트 과목은 실무와 닮아 있습니다. 요구사항이 바뀌고, 일정이 부족하고, 팀원 사이의 속도가 다르고, 예상치 못한 버그가 계속 나옵니다. 이런 경험은 강의실 시험만으로는 얻기 어렵습니다.

## 한눈에 보는 흐름

```mermaid
flowchart LR
    P[계획] --> D[설계]
    D --> I[구현]
    I --> T[테스트]
    T --> S[시연]
```

프로젝트는 코드를 먼저 쓰는 활동이 아닙니다. 계획과 설계가 먼저 있고, 구현과 테스트가 뒤따르며, 마지막에는 결과를 보여 주는 시연과 회고가 이어집니다. 이 순서를 머릿속에 두면 프로젝트가 훨씬 덜 혼란스러워집니다.

## 핵심 용어

- **범위(scope)**: 이번 프로젝트에서 실제로 다룰 문제의 경계입니다.
- **MVP**: 가장 작은 기능 집합으로 만든 첫 제품입니다.
- **데모**: 결과물을 직접 보여 주는 시연입니다.
- **이해관계자(stakeholder)**: 결과에 관심을 갖는 사람이나 집단입니다.
- **회고(retrospective)**: 작업이 끝난 뒤 과정과 문제를 돌아보는 정리입니다.

## Before/After

**Before**: 프로젝트를 단순한 과제로 봅니다.

**After**: 작은 제품을 만드는 과정으로 봅니다.

## 프로젝트 과목에서 진짜 어려운 부분

프로젝트 과목에서 가장 어려운 일은 구현 자체가 아닐 때가 많습니다. 문제를 너무 크게 잡지 않는 일, 사용자 입장에서 핵심 기능을 먼저 고르는 일, 팀원 역할을 겹치지 않게 나누는 일이 더 어렵습니다. 코드는 밤을 새워서라도 어느 정도 나오지만, 방향이 틀리면 뒤늦게 고치기 어렵습니다.

또한 프로젝트 과목은 혼자 잘해서 끝나는 구조가 아닙니다. 팀원 한 명이 막히면 전체 일정이 밀릴 수 있고, 문서가 없으면 서로 만든 기능을 합치기 어렵습니다. 그래서 Git 규칙, 회의 기록, 이슈 정리 같은 협업 습관이 성적만큼 중요해집니다.

## 프로젝트 미니 플랜

### 1단계 — 문제 정의

```python
problem = "강의 시간표 충돌 검사기"
```

프로젝트는 문제 정의에서 시작합니다. 문제 한 줄이 분명해야 기능과 범위를 잡기 쉽습니다.

### 2단계 — 사용자

```python
users = ["student", "advisor"]
```

누가 쓰는지 모르면 기능이 흔들립니다. 사용자 목록은 요구사항을 좁히는 데 큰 도움이 됩니다.

### 3단계 — 핵심 기능

```python
features = ["upload", "detect_conflict", "notify"]
```

모든 아이디어를 다 넣기보다 핵심 기능부터 정해야 합니다. 이 목록이 사실상 MVP의 뼈대가 됩니다.

### 4단계 — 일정

```python
weeks = {"plan": 1, "build": 6, "test": 2, "demo": 1}
```

일정은 계획을 현실로 바꿉니다. 구현만 길게 잡고 테스트와 시연 준비를 빼먹으면 막판에 가장 크게 흔들립니다.

### 5단계 — 위험

```python
risks = ["scope_creep", "team_sync", "data_format"]
```

위험 요소를 미리 적어 두면 문제가 생겼을 때 당황이 줄어듭니다. 범위 확장, 팀 일정 불일치, 데이터 형식 문제는 학생 프로젝트에서 아주 흔합니다.

## 이 코드에서 주목할 점

- 문제 정의가 프로젝트의 출발점이 됩니다.
- 사용자를 알아야 핵심 기능을 고를 수 있습니다.
- 일정이 있어야 계획이 실제 작업으로 바뀝니다.

## 자주 하는 실수 5가지

1. 기획서나 요구사항 정리 없이 바로 코드를 쓰는 일입니다.
2. 팀 역할이 모호해서 책임이 겹치거나 비는 일입니다.
3. 주간 점검 회의 없이 각자만 작업하는 일입니다.
4. Git 브랜치나 커밋 규칙을 정하지 않는 일입니다.
5. 데모로 끝내고 회고를 남기지 않는 일입니다.

## 실무에서는 이렇게 쓰입니다

스타트업의 초기 MVP는 학생 프로젝트와 꽤 닮아 있습니다. 문제를 작게 자르고, 핵심 사용자에게 필요한 기능부터 만들고, 빠르게 보여 주고, 피드백을 받아 다시 고칩니다. 그래서 프로젝트 과목 경험은 단순한 학교 과제가 아니라 작은 제품 개발 경험으로 읽힐 수 있습니다.

## 선배 엔지니어는 이렇게 봅니다

- 처음부터 크게 만들지 않습니다.
- 자주 보여 주고 빨리 피드백을 받습니다.
- 문서는 구현 속도를 늦추지 않고 오히려 합치는 비용을 줄입니다.
- 회고를 남겨야 다음 프로젝트가 좋아집니다.
- 데모는 결과를 설명하는 가장 강한 순간입니다.

## 체크리스트

- [ ] 문제를 한 줄로 설명할 수 있습니다.
- [ ] 핵심 기능 목록을 적었습니다.
- [ ] 일정표를 간단히라도 만들었습니다.
- [ ] 위험 요소를 미리 정리했습니다.

## 연습 문제

1. MVP를 한 줄로 설명해 보세요.
2. 데모가 왜 중요한지 한 줄로 적어 보세요.
3. 회고가 무엇을 남기는 과정인지 한 줄로 써 보세요.

## 정리 및 다음 단계

프로젝트 과목은 전공 지식을 한데 묶어 결과물로 만드는 단계입니다. 문제 정의, 사용자 이해, 범위 조절, 협업, 테스트, 시연까지 모두 해 봐야 비로소 작은 제품을 만든 경험이 남습니다. 다음 글에서는 이렇게 많은 과목과 프로젝트를 버티기 위한 전공 공부 방법을 정리하겠습니다.
