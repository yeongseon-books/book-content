MetaData is the catalog that holds the schema your application knows about. This post shows why that container becomes the starting point for SQLAlchemy Core.

Why this matters

Why this matters
Managing schema with raw SQL has one big problem: when the column names in your INSERT/SELECT statements drift from the actual schema, you don't find out until runtime. A no such column error suddenly appears in production, and your IDE can't even autocomplete column names.

SQLAlchemy Core keeps the schema as Python objects, so you can build SQL through those objects. A typo in a column name turns into an AttributeError at import time, your IDE autocompletes users.c.name, and the same definition is reused by Alembic's autogenerate, Pandas' read_sql, FastAPI's SQL building, and more.

To handle types consistently across SQLite and other dialects, you also need a generic type system. When you write String(255), it expands to VARCHAR(255) on PostgreSQL and to VARCHAR(255) (TEXT affinity in practice) on SQLite. That's the layer that lets the same code run on SQLite locally and PostgreSQL in production.

Finally, the MetaData becomes Alembic's target_metadata, the basis for autogenerated migrations. The schema you define in this post is also the starting point for the alembic-101 series.

Mental Model

Mental model
MetaData is a catalog of your schema. It's a container that holds every Table definition your application knows about, and you hand the entire container to an Engine to create or compare the schema in one go.

MetaData is your application's schema specification. A Table is a page in that spec, and a Column is one line on a page. Without an Engine, MetaData is just an in-memory document. Without MetaData, an Engine has nothing to build.

                    MetaData (in-memory catalog)
                         │
        ┌────────────────┼────────────────┐
        ▼                ▼                ▼
    Table("users")  Table("posts")  Table("tags")
        │
   ┌────┼────┐
   ▼    ▼    ▼
  Col  Col  Col
 (id, name, email)
        │
        ▼
  metadata.create_all(engine)
        │
        ▼
   Issues CREATE TABLE against SQLite file

The Core flow is:

  1. metadata = MetaData() creates the catalog.
  2. users = Table("users", metadata, Column(...), ...) registers a page (it auto-attaches to metadata at construction time).
  3. metadata.create_all(engine) materializes the tables in the SQLite file.
  4. From then on, you reference users.c.name, users.insert(), select(users) to build SQL.

Core concepts

Core concepts
MetaData

from sqlalchemy import MetaData

metadata = MetaData()

MetaData() creates an empty catalog. Most apps keep one of these as a module-level global, typically in something like db.py. Multiple instances are useful when you want to namespace schemas explicitly (multi-tenant, plugin systems).

The naming_convention argument controls how SQLAlchemy generates names for constraints.

metadata = MetaData(
    naming_convention={
        "ix": "ix_%(table_name)s_%(column_0_name)s",
        "uq": "uq_%(table_name)s_%(column_0_name)s",
        "fk": "fk_%(table_name)s_%(column_0_name)s_%(referred_table_name)s",
        "pk": "pk_%(table_name)s",
        "ck": "ck_%(table_name)s_%(constraint_name)s",
    }
)

A naming convention is critical when Alembic generates ALTER TABLE statements. Without consistent names, DROP CONSTRAINT may not work.

Table and Column

from sqlalchemy import Table, Column, Integer, String, DateTime
from datetime import datetime, timezone

users = Table(
    "users",
    metadata,
    Column("id", Integer, primary_key=True, autoincrement=True),
    Column("name", String(100), nullable=False),
    Column("email", String(255), nullable=False, unique=True),
    Column("created_at", DateTime, nullable=False, default=lambda: datetime.now(timezone.utc)),
)

The Table constructor takes the table name, the MetaData, then a varargs sequence of Column and constraint objects. Common Column options:

primary_key=True
- Meaning: Member of the PK

nullable=False
- Meaning: NOT NULL

unique=True
- Meaning: Auto-create UNIQUE constraint

index=True
- Meaning: Auto-create single-column INDEX

default=value_or_callable
- Meaning: Python-side default

server_default=text("...")
- Meaning: DB-side DEFAULT clause

onupdate=value_or_callable
- Meaning: Auto-applied on UPDATE

default vs server_default is a frequent source of confusion. default is filled in by SQLAlchemy when Python issues the INSERT. server_default is emitted as the database's DEFAULT clause and works even when raw SQL inserts the row. You almost never want both, so pick one per column.

Type system and SQLite affinities

SQLAlchemy expands generic types into dialect-specific types. Common mappings:

Integer
- SQLite resolution: INTEGER
- Notes: 64-bit

String(n)
- SQLite resolution: VARCHAR(n) (TEXT affinity)
- Notes: SQLite does not enforce length

Text
- SQLite resolution: TEXT
- Notes: Explicitly unbounded

Boolean
- SQLite resolution: BOOLEAN (NUMERIC affinity)
- Notes: Stored as 0/1

DateTime
- SQLite resolution: DATETIME (TEXT affinity)
- Notes: ISO-8601 string

Numeric(precision, scale)
- SQLite resolution: NUMERIC
- Notes: Avoids floating point

JSON
- SQLite resolution: JSON (native since 1.x)
- Notes: Use json_extract()

LargeBinary
- SQLite resolution: BLOB
- Notes: bytes

SQLite uses type affinity, which means a column type is more of a hint than a hard contract. So String(100) won't actually reject a 200-character string; if you need length enforcement, validate it in your application or with a CHECK constraint.

DateTime is stored as an ISO-8601 string in SQLite. To handle timezone-aware datetimes safely, always store them with explicit timezone, e.g., datetime.now(timezone.utc).

create_all and drop_all

metadata.create_all(engine)   # Issues CREATE TABLE IF NOT EXISTS for each table
metadata.drop_all(engine)     # Drops in dependency order

create_all is idempotent (thanks to IF NOT EXISTS). However, it does not issue ALTER TABLE for schema changes. ALTER belongs to migrations, which we'll cover in the alembic-101 series.

A common test pattern:

@pytest.fixture
def engine():
    eng = create_engine("sqlite:///:memory:")
    metadata.create_all(eng)
    yield eng
    eng.dispose()

ForeignKey and the SQLite trap

from sqlalchemy import ForeignKey

posts = Table(
    "posts",
    metadata,
    Column("id", Integer, primary_key=True),
    Column("user_id", Integer, ForeignKey("users.id", ondelete="CASCADE"), nullable=False),
    Column("title", String(200), nullable=False),
)

Trap: SQLite does not enforce foreign key constraints by default. Even with ondelete="CASCADE", nothing will happen unless you turn on PRAGMA foreign_keys = ON (from post 1). The PRAGMA is connection-scoped, so apply it in a connect event.

from sqlalchemy import event

@event.listens_for(engine, "connect")
def _enable_fk(dbapi_conn, _):
    cur = dbapi_conn.cursor()
    cur.execute("PRAGMA foreign_keys = ON")
    cur.close()

Composite constraints and Index

Multi-column constraints are declared explicitly with PrimaryKeyConstraint, UniqueConstraint, ForeignKeyConstraint, and Index.

from sqlalchemy import UniqueConstraint, Index

memberships = Table(
    "memberships",
    metadata,
    Column("user_id", Integer, ForeignKey("users.id"), nullable=False),
    Column("group_id", Integer, ForeignKey("groups.id"), nullable=False),
    Column("role", String(20), nullable=False),
    UniqueConstraint("user_id", "group_id", name="uq_membership_user_group"),
    Index("ix_membership_role", "role"),
)

For a composite primary key, mark both columns with primary_key=True.

Reflection

When tables already exist in a SQLite file and you want SQLAlchemy to learn the schema:

from sqlalchemy import Table, MetaData, create_engine

engine = create_engine("sqlite:///legacy.db")
metadata = MetaData()
users = Table("users", metadata, autoload_with=engine)
print([c.name for c in users.columns])

To reflect everything at once, use metadata.reflect(bind=engine). This is invaluable when migrating a legacy database into SQLAlchemy gradually.

Before-After

Before: schema as raw SQL strings

DDL = """
CREATE TABLE IF NOT EXISTS users(
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    name TEXT NOT NULL,
    email TEXT NOT NULL UNIQUE,
    created_at TEXT NOT NULL
);
"""

with engine.begin() as conn:
    conn.execute(text(DDL))

# Elsewhere, an INSERT
conn.execute(text("INSERT INTO users(name, emai, created_at) VALUES (:n, :e, :c)"),
             {"n": "Alice", "e": "alice@example.com", "c": "..."})
# typo (emai) only surfaces as OperationalError at runtime.

After: MetaData + Table

from sqlalchemy import MetaData, Table, Column, Integer, String, DateTime, insert
from datetime import datetime, timezone

metadata = MetaData()

users = Table(
    "users",
    metadata,
    Column("id", Integer, primary_key=True, autoincrement=True),
    Column("name", String(100), nullable=False),
    Column("email", String(255), nullable=False, unique=True),
    Column("created_at", DateTime, nullable=False,
           default=lambda: datetime.now(timezone.utc)),
)

metadata.create_all(engine)

with engine.begin() as conn:
    conn.execute(insert(users).values(name="Alice", email="alice@example.com"))
    # users.c.emai → AttributeError raised by IDE/runtime immediately

Now column-name typos are caught at schema definition time or via IDE autocompletion. You reference columns through users.c.name, and reuse the same schema object across select/insert/update.

Step-by-step practice

Step-by-step practice
Step 1: A schema module

schema.py:

from sqlalchemy import (
    MetaData, Table, Column, Integer, String, Text, DateTime, ForeignKey,
    UniqueConstraint, Index,
)
from datetime import datetime, timezone

metadata = MetaData(
    naming_convention={
        "ix": "ix_%(table_name)s_%(column_0_name)s",
        "uq": "uq_%(table_name)s_%(column_0_name)s",
        "fk": "fk_%(table_name)s_%(column_0_name)s_%(referred_table_name)s",
        "pk": "pk_%(table_name)s",
    }
)

users = Table(
    "users", metadata,
    Column("id", Integer, primary_key=True, autoincrement=True),
    Column("name", String(100), nullable=False),
    Column("email", String(255), nullable=False, unique=True),
    Column("created_at", DateTime, nullable=False,
           default=lambda: datetime.now(timezone.utc)),
)

posts = Table(
    "posts", metadata,
    Column("id", Integer, primary_key=True, autoincrement=True),
    Column("user_id", Integer,
           ForeignKey("users.id", ondelete="CASCADE"), nullable=False),
    Column("title", String(200), nullable=False),
    Column("body", Text, nullable=False),
    Column("created_at", DateTime, nullable=False,
           default=lambda: datetime.now(timezone.utc)),
    Index("ix_posts_user_id_created", "user_id", "created_at"),
)

Step 2: Apply the schema

from sqlalchemy import create_engine, event
from schema import metadata

engine = create_engine("sqlite:///app.db", echo=True)

@event.listens_for(engine, "connect")
def _enable_fk(dbapi_conn, _):
    cur = dbapi_conn.cursor(); cur.execute("PRAGMA foreign_keys = ON"); cur.close()

metadata.create_all(engine)

With echo=True, you can watch the actual CREATE TABLE / CREATE INDEX statements that SQLAlchemy emits.

Step 3: Inspect SQLite type affinity

with engine.connect() as conn:
    rows = conn.execute(text("SELECT name, type FROM pragma_table_info('users')")).all()
    for r in rows:
        print(r.name, r.type)

String(100) shows up as VARCHAR(100), DateTime as DATETIME. But remember, SQLite's affinity model means VARCHAR(100) won't enforce the length.

Step 4: Verify ON DELETE CASCADE

from sqlalchemy import insert, delete, select

with engine.begin() as conn:
    uid = conn.execute(insert(users).values(name="Alice", email="a@example.com")).inserted_primary_key[0]
    conn.execute(insert(posts).values(user_id=uid, title="hi", body="hello"))
    conn.execute(delete(users).where(users.c.id == uid))
    remaining = conn.execute(select(posts)).all()
    assert remaining == []

Run the same code with an Engine that doesn't enable PRAGMA foreign_keys = ON and the final assert fails. That single line of difference captures the SQLite trap.

Step 5: Reflection

from sqlalchemy import MetaData, Table

reflected = MetaData()
users_r = Table("users", reflected, autoload_with=engine)
print([c.name for c in users_r.columns])
print([(c.name, c.type) for c in users_r.columns])

You can pull an existing schema and use it for SQL building immediately.

Common mistakes

1. Creating a fresh MetaData inside a function. Like the Engine, it should be a module-level singleton. Multiple catalogs scatter your table definitions and confuse reflection and migration tools.

2. Starting without naming_convention. It hurts a lot the day you adopt Alembic. The disciplined choice is to set a naming convention from day one.

3. Writing ON DELETE CASCADE on SQLite without enabling the PRAGMA. It becomes a silent bug. Always pair it with the connect event from post 1.

4. Using both default and server_default. When values come from two places, you can't easily tell which one filled in a row. Pick one per column.

5. Trusting String(n) length enforcement on SQLite. SQLite won't reject longer strings. Validate length in the application or via a CHECK constraint.

6. Leaving every Column as nullable=True. The default is nullable=True. If you don't actually want NULLs, write nullable=False explicitly.

Real-world application

In real apps, MetaData lives in a single module and Table definitions are split by domain.

app/
  db/
    __init__.py        # metadata, engine, get_conn
    users.py           # users Table
    posts.py           # posts Table
    schema.py          # imports the above so they auto-register on metadata

Once schema.py does from . import users, posts, the Table objects auto-attach to metadata at import time. Anywhere else, from app.db.schema import metadata gives you the entire schema.

Another common pattern is sharing the same metadata between Core Table and ORM declarative base. When the ORM's declarative base uses the same metadata object, a single metadata.create_all(engine) call materializes both layers. We dive into that in the ORM posts.

In production, prefer Alembic migrations over metadata.create_all. But for test fixtures, local dev DB initialization, and demos, create_all is still the fastest bootstrap.

Checklist

  • [ ] MetaData is a module-level singleton
  • [ ] You configured naming_convention from the very start
  • [ ] Every Column has an explicit nullable choice
  • [ ] You have a rule for choosing String(n) vs Text (bounded vs unbounded)
  • [ ] You use only one of default / server_default per column
  • [ ] When using ForeignKey(ondelete=...) on SQLite, you also enable PRAGMA foreign_keys = ON
  • [ ] Composite constraints are declared with explicit UniqueConstraint/Index
  • [ ] You know reflection is available via autoload_with=engine

Exercises

  1. Create a MetaData and define users(id, name, email) and posts(id, user_id FK→users.id, title, body). Materialize via metadata.create_all(engine), then print column names and types using pragma_table_info.
  2. Apply ondelete="CASCADE" on posts.user_id. Run a DELETE on the parent against two Engines: one with the foreign_keys PRAGMA enabled, one without. Compare child-row outcomes.
  3. Build the same schema with and without naming_convention. Inspect the sqlite_master table to compare the auto-generated names of indexes and unique constraints.
  4. Pre-create a legacy(id INTEGER PRIMARY KEY, payload TEXT) table via raw SQL on a SQLite file. Then use Table("legacy", metadata, autoload_with=engine) to reflect it, and use the reflected Table to INSERT and SELECT rows.
