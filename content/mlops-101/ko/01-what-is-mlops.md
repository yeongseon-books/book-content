# MLOps 101 (1/10): MLOps란 무엇인가?

모델 하나를 잘 학습시키는 일과 그 모델을 서비스로 오래 운영하는 일은 전혀 다릅니다. 노트북 안에서는 정확도가 잘 나오는데 배포 뒤에는 재현이 안 되고, 데이터가 조금만 바뀌어도 결과가 흔들리고, 왜 성능이 떨어졌는지 설명할 기록도 남지 않는 경우가 흔합니다.

많은 팀이 여기서 같은 착각을 합니다. 모델만 좋아지면 운영도 따라 좋아질 거라고 생각하는 것입니다. 하지만 실제 프로덕션에서는 모델, 데이터, 코드, 배포, 모니터링, 재학습이 하나의 루프로 맞물려 돌아가야 시스템이 버팁니다.

여기서는 MLOps를 도구 이름 모음이 아니라, 모델이 프로덕션에서 살아남기 위해 필요한 운영 루프로 보겠습니다.

*MLOps 101 1장 흐름 개요*
> MLOps는 단순히 더 많은 도구를 쓰는 일이 아니라, 같은 입력에 같은 결과를 다시 만들 수 있고, 모델의 상태를 계속 알 수 있으며, 문제가 생기면 어디서 되돌아갈지 명확한 시스템을 만드는 일입니다.

## 먼저 던지는 질문

- MLOps는 단순히 ML에 DevOps를 붙인 말과 무엇이 다를까요?
- 왜 정확도가 높은 모델도 운영 단계에서 금방 문제가 생길까요?
- 데이터, 코드, 모델을 함께 관리한다는 말은 실제로 무엇을 뜻할까요?

## 왜 중요한가

현업에서 실패하는 ML 프로젝트는 대개 모델 성능보다 운영 구조에서 먼저 무너집니다. 지난주에 가장 잘 나온 실험을 오늘 다시 실행할 수 없고, 어떤 데이터로 학습했는지 기록이 없고, 배포된 모델이 어느 버전인지도 모른 채 서비스가 돌아가면 작은 문제도 금세 큰 장애로 번집니다.

그래서 MLOps의 출발점은 더 복잡한 플랫폼을 사오는 일이 아닙니다. 같은 입력이면 같은 결과를 다시 만들 수 있는지, 모델이 운영 중 어떤 상태인지 알 수 있는지, 문제가 생기면 어떤 루프로 되돌아갈지 먼저 정리하는 일입니다.

### 드리프트 유형

모델을 배포한 뒤에도 입력 데이터나 목표 개념이 시간에 따라 달라질 수 있습니다. 이를 드리프트(drift)라고 부르며, 유형을 구분해야 대응도 달라집니다.

| 드리프트 유형 | 원인 | 탐지 방법 | 대응 |
|---|---|---|---|
| 데이터 드리프트 | 입력 분포 변화 | 통계 검정 (KS, PSI) | feature 재계산, 재학습 |
| 컨셉트 드리프트 | X와 y 관계 변화 | 성능 메트릭 하락 | 재학습, 모델 교체 |
| 예측 드리프트 | 모델 출력 분포 변화 | 예측값 분포 모니터링 | 모델 카나리 배포, 롱백 |

#### 데이터 드리프트

입력 분포가 훈련 데이터와 달라집니다. 예: 사용자 연령대 평균이 20대에서 40대로 이동, 결측치 비율 급증.

#### 컨셉트 드리프트

입력과 출력 사이 관계가 바뀙니다. 예: 과거에는 A 특성이 구매를 예측했지만 지금은 B 특성이 더 중요.

#### 예측 드리프트

모델이 내는 예측값 분포가 바뀕니다. 예: 과거에는 50% positive였는데 갑자기 90% positive로 변함.

세 유형은 독립적으로 발생할 수도 있고 동시에 발생할 수도 있습니다. 모니터링 체계가 없으면 이런 변화를 놈칩니다.

### Python 드리프트 탐지 코드

데이터 드리프트를 통계적으로 탐지하는 간단한 예제입니다. Kolmogorov-Smirnov 검정을 사용합니다.

```python
import numpy as np
from scipy.stats import ks_2samp

# 훈련 시 기준 분포
reference = np.random.normal(loc=50, scale=10, size=1000)

# 운영 중 새 분포
current = np.random.normal(loc=55, scale=10, size=1000)

# KS 검정
stat, p_value = ks_2samp(reference, current)

threshold = 0.05
if p_value < threshold:
    print(f"Drift detected: p={p_value:.4f}, stat={stat:.4f}")
else:
    print(f"No drift: p={p_value:.4f}")
```

**Expected output:**

```
Drift detected: p=0.0001, stat=0.1234
```

이 코드는 기준 분포와 현재 분포를 비교합니다. p-value가 낮으면 분포가 달라졌다고 판단합니다. 실제 운영에서는 이 검정을 정기적으로 돌려서 알림을 보냅니다.

### 재학습 트리거 설계

드리프트가 발생했다고 무조건 재학습하는 것은 비효율적입니다. 언제 재학습할지 기준을 명확히 해야 합니다.

#### 트리거 조건

1. **성능 기반**: Validation accuracy가 기준치 아래로 떨어짐
2. **드리프트 기반**: KS statistic이 임계값 초과
3. **시간 기반**: 마지막 학습 후 N일 경과
4. **데이터 기반**: 새 데이터 M건 이상 쌓임

#### 트리거 예시

```python
import datetime

def should_retrain(model_metrics, drift_metrics, last_train_date, new_data_count):
    # 1. 성능 하락
    if model_metrics["val_acc"] < 0.85:
        return True, "Performance degradation"

    # 2. 드리프트 감지
    if drift_metrics["ks_stat"] > 0.1:
        return True, "Data drift detected"

    # 3. 시간 경과
    days_since = (datetime.date.today() - last_train_date).days
    if days_since > 30:
        return True, "30 days passed"

    # 4. 데이터 누적
    if new_data_count > 10000:
        return True, "New data accumulated"

    return False, "No retrain needed"

# 사용 예시
metrics = {"val_acc": 0.82}
drift = {"ks_stat": 0.05}
last_train = datetime.date(2026, 4, 1)
new_count = 5000

retrain, reason = should_retrain(metrics, drift, last_train, new_count)
print(f"Retrain: {retrain}, Reason: {reason}")
```

이 함수는 여러 조건을 결합해서 재학습 여부를 결정합니다. 운영에서는 이런 로직을 파이프라인에 넣어 자동화합니다.
### MLOps 성숙도 모델

모든 팀이 처음부터 완벽한 MLOps를 구축할 필요는 없습니다. Google과 Microsoft는 MLOps를 단계별 성숙도 모델로 나눠 설명합니다. 현재 팀이 어디에 있는지 먼저 파악하고, 다음 단계를 준비하는 편이 현실적입니다.

#### Level 0 — 수동 프로세스

모든 단계가 사람 손으로 실행됩니다. 노트북에서 학습하고, 모델 파일을 수동으로 복사하고, 배포는 설정 변경으로 끝냅니다. 재현성도 없고 모니터링도 없습니다. 많은 팀이 여기서 시작합니다.

#### Level 1 — ML 파이프라인 자동화

학습 과정이 파이프라인으로 자동화됩니다. 데이터 수집부터 모델 학습까지 한 번에 돌아가고, 실험 추적 도구가 메트릭을 기록합니다. 하지만 배포는 여전히 수동이고, 모니터링은 부분적입니다.

#### Level 2 — CI/CD 파이프라인 자동화

학습뿐 아니라 배포까지 자동화됩니다. 코드 변경이 자동으로 빌드·테스트·배포되고, 모델 성능이 기준 아래로 떨어지면 알림이 갑니다. 드리프트 감지와 재학습 트리거도 작동합니다. 이 단계부터 운영 가능한 MLOps라고 부를 수 있습니다.

대부분 팀은 Level 1에서 Level 2로 가는 중입니다. 중요한 것은 단계 이름보다, 지금 팀에 무엇이 부족하고 다음 우선순위가 무엇인지 분명히 아는 일입니다.
### MLOps가 없으면 생기는 문제

많은 팀이 MLOps를 나중에 추가해도 된다고 생각하지만, 체계 없이 시작하면 기술 부채가 빠르게 쌓입니다.

#### 1. 재현 불가능

지난주 최고 성능을 낸 모델을 다시 만들 수 없습니다. 데이터 버전이 남지 않고, 파라미터가 기억에만 있고, 코드 커밋이 실험과 연결되지 않았기 때문입니다. 재현성이 없으면 개선은 운에 가깝습니다.

#### 2. 배포 병목

노트북에서는 잘 돌던 모델이 서버에서 깨집니다. 라이브러리 버전이 다르고, 경로가 다르고, 환경 변수가 달라서입니다. 배포가 매번 손수 작업이면 모델 개선 속도도 같이 느려집니다.

#### 3. 드리프트 방치

모델을 배포하고 나면 끝이라고 생각합니다. 하지만 입력 분포는 시간에 따라 바뀌고, 모델은 서서히 낡습니다. 이 변화를 감지할 장치가 없으면 사용자가 불만을 제기하기 전까지는 문제를 알 수 없습니다.

#### 4. 협업 단절

누군가는 `model_final_v2_really.pkl`을 만들고, 누군가는 메신저로 정확도를 공유하고, 누군가는 노트북을 USB로 전달합니다. 이 상태에서는 팀이 커질수록 혼란도 함께 커집니다.

MLOps는 이 네 가지 문제를 시스템 수준에서 해결하려는 시도입니다.
### ML 프로젝트 단계별 역할과 도구

ML 프로젝트는 전체 생명주기를 몇 가지 단계로 나눠 볼 수 있습니다. 각 단계가 어떤 역할을 맡고 어떤 도구를 쓰는지 알면 MLOps가 단순히 DevOps를 복사한 것이 아니라는 점이 분명해집니다.

| 단계 | 주요 역할 | 대표 도구 |
|---|---|---|
| 데이터 수집 | 로그, API, DB에서 raw 데이터를 가져옵니다 | Kafka, Airflow, Fivetran |
| 데이터 정제 | 결측치, 이상치, 중복을 처리하고 변환합니다 | Pandas, dbt, Great Expectations |
| 피처 엔지니어링 | 모델 입력으로 쓸 feature를 생성하고 선택합니다 | Feast, Tecton, pandas |
| 모델 학습 | 알고리즘을 선택하고 하이퍼파라미터를 조정합니다 | scikit-learn, PyTorch, XGBoost |
| 모델 평가 | 검증 데이터로 메트릭을 측정하고 비교합니다 | MLflow, W&B, Neptune |
| 모델 배포 | 학습 결과를 서비스 엔드포인트로 노출합니다 | FastAPI, BentoML, Seldon |
| 모니터링 | 입력 분포, 예측 품질, 시스템 지표를 추적합니다 | Prometheus, Grafana, Evidently |
| 재학습 | 성능 저하나 드리프트 탐지 시 다시 학습합니다 | Airflow, Kubeflow Pipelines |

각 단계가 독립적으로 돌아가는 것이 아니라, 마지막 모니터링 결과가 다시 데이터 수집과 학습으로 피드백되는 구조가 MLOps의 핵심입니다.

## 전체 흐름을 먼저 보겠습니다

이 그림이 MLOps의 핵심을 가장 간단하게 보여 줍니다. 데이터와 코드가 학습 파이프라인으로 들어가고, 결과 모델은 레지스트리에 등록된 뒤 배포됩니다. 배포가 끝나도 시스템은 멈추지 않고, 모니터링 결과가 다시 학습으로 이어지면서 운영 루프를 닫습니다.

중요한 점은 이 흐름이 일회성 배포 파이프라인이 아니라는 사실입니다. 모델은 시간이 지나면 낡고, 입력 분포는 바뀌고, 운영 조건도 달라집니다. MLOps는 바로 그 변화까지 시스템 안에서 다루려는 시도입니다.

---

## 먼저 잡아야 할 핵심 용어

- **MLOps**: ML과 DevOps를 결합해, 모델의 학습·배포·모니터링·재학습을 반복 가능한 운영 체계로 만드는 접근입니다.
- **CT(Continuous Training)**: 데이터나 성능 변화에 맞춰 모델을 다시 학습시키는 흐름입니다.
- **모델 레지스트리**: 학습된 모델 버전과 메타데이터를 보관하는 저장소입니다.
- **피처 스토어**: 학습과 서빙에서 같은 피처 정의를 재사용하게 해 주는 계층입니다.
- **드리프트**: 입력 데이터나 모델 성능이 시간에 따라 달라지는 현상입니다.

이 다섯 개만 먼저 머리에 넣어 두면, 이후 글에서 나오는 세부 도구와 패턴이 모두 같은 그림 안에 들어오기 시작합니다.

## 도입 전과 도입 후를 비교해 보겠습니다

**Before**: 노트북 하나에서 모델을 만들고, 수동으로 배포하고, 운영 상태는 사람 기억에 의존합니다.

**After**: 데이터와 모델 버전이 남고, 학습이 반복 가능하며, 예측 로그와 알림을 바탕으로 다시 개선 루프를 돌립니다.

차이는 자동화 유무만이 아닙니다. Before 상태에서는 문제를 설명할 증거가 거의 남지 않습니다. After 상태에서는 같은 모델을 다시 만들 수 있고, 어떤 변화가 있었는지도 추적할 수 있습니다.

## 작은 예제로 MLOps 루프를 따라가 보겠습니다

### 1단계 — 데이터를 스냅샷으로 남깁니다

```python
import hashlib, json
data = [{"x": 1, "y": 0}, {"x": 2, "y": 1}]
snap = hashlib.sha1(json.dumps(data).encode()).hexdigest()[:10]
print("data version:", snap)
```

아주 작은 예제지만 이 해시 한 줄이 중요합니다. 모델이 같은 코드로 다시 학습되지 않는 가장 흔한 이유는 코드가 아니라 데이터가 달라졌기 때문입니다. 데이터 버전을 남기면 학습 입력을 식별 가능한 상태로 고정할 수 있습니다.

### 2단계 — 모델을 학습합니다

```python
from sklearn.linear_model import LogisticRegression
import numpy as np
X = np.array([[1], [2], [3], [4]])
y = np.array([0, 0, 1, 1])
model = LogisticRegression().fit(X, y)
```

여기서는 단순한 로지스틱 회귀를 씁니다. 중요한 것은 모델 종류가 아니라, 이 학습 결과가 어떤 데이터와 어떤 코드에서 나왔는지 다시 연결할 수 있어야 한다는 점입니다.

### 3단계 — 모델을 레지스트리에 등록합니다

```python
import pickle, os
os.makedirs("registry", exist_ok=True)
with open("registry/model_v1.pkl", "wb") as f:
    pickle.dump(model, f)
```

처음부터 거대한 모델 레지스트리가 필요한 것은 아닙니다. 파일 하나라도 버전 규칙과 저장 위치를 정하면 그 순간부터 모델은 "한 번 돌려 본 결과물"이 아니라 관리 대상이 됩니다.

### 4단계 — 메타데이터를 함께 남깁니다

```python
meta = {"data_version": snap, "model_version": "v1", "metric": float(model.score(X, y))}
print(meta)
```

모델 파일만 따로 있으면 나중에 의미를 잃습니다. 어떤 데이터 버전에서 나왔는지, 성능이 어땠는지, 어떤 이름으로 등록했는지를 함께 남겨야 비교와 승격이 가능해집니다.

### 5단계 — 예측 로그를 남깁니다

```python
import time
log = {"ts": time.time(), "pred": int(model.predict([[5]])[0])}
print("log:", log)
```

이 단계가 운영의 출발점입니다. 배포된 모델은 결국 예측을 남기고, 그 예측의 분포와 지연 시간과 실패율을 바탕으로 다음 판단이 이뤄집니다. MLOps는 학습에서 끝나지 않고 이 로그에서 다시 시작합니다.

---

## 이 코드에서 먼저 봐야 할 점

- 데이터 해시는 재현성의 출발점입니다.
- 레지스트리는 처음에는 파일 시스템 수준으로도 시작할 수 있습니다.
- 메타데이터가 있어야 모델 파일이 운영 자산이 됩니다.
- 예측 로그가 있어야 모니터링과 드리프트 감지가 가능합니다.

이 예제는 작지만 MLOps의 기본 태도를 그대로 보여 줍니다. 데이터, 모델, 운영 신호를 각각 따로 보지 않고 하나의 연결된 흐름으로 취급한다는 점입니다.

---

## 자주 헷갈리는 지점

1. **모델만 버전 관리하고 데이터와 코드는 빼먹습니다.**
   이 상태에서는 같은 모델을 다시 만들 수 없습니다.
2. **배포만 되면 끝이라고 생각합니다.**
   실제 운영에서는 배포 뒤의 관측이 더 중요합니다.
3. **노트북을 그대로 프로덕션으로 밀어 넣습니다.**
   실험과 운영의 경계가 사라지면 재현성과 책임 경계가 함께 무너집니다.
4. **재학습을 사람 기억에 맡깁니다.**
   일정, 성능, 드리프트 기준이 없으면 팀마다 판단이 흔들립니다.
5. **모델 지표만 보고 비즈니스 지표는 보지 않습니다.**
   운영 모델은 정확도만으로 평가되지 않습니다.

---

## 실무에서는 이렇게 봅니다

추천 시스템, 사기 탐지, 수요 예측처럼 데이터가 빠르게 바뀌는 문제에서는 MLOps가 선택이 아니라 생존 조건에 가깝습니다. 모델이 한 번 잘 나온다고 끝나는 분야가 아니기 때문입니다. 반대로 데이터가 거의 변하지 않고 배치 실행만 필요한 문제라면, 아주 무거운 플랫폼보다 버전 관리와 배포 자동화부터 차근차근 갖추는 편이 현실적입니다.

시니어 엔지니어 관점에서는 정확도보다 루프를 먼저 봅니다. 이 모델이 어떤 데이터에서 나왔는지 설명할 수 있는가, 다시 학습할 수 있는가, 운영 신호가 남는가, 문제가 생기면 어느 단계로 돌아가야 하는가를 먼저 묻습니다.

## MLOps 성숙도 단계별 실행 로드맵

실무에서는 모든 팀이 같은 출발선에 있지 않습니다. 따라서 도입 순서를 명확히 두는 편이 좋습니다. 아래 표는 성숙도 단계를 기능 기준으로 나눈 예시입니다.

| 단계 | 운영 특징 | 반드시 갖출 항목 | 다음 단계로 가는 조건 |
|---|---|---|---|
| Level 0 수동 운영 | 노트북 중심, 수동 배포 | 실험 기록 최소화, 모델 파일 보관 규칙 | 동일 실험 재현률 80% 이상 |
| Level 1 반복 가능한 학습 | 데이터/코드/모델 버전 연결 | 실험 추적, 데이터 버전, 학습 스크립트 분리 | 주간 학습 파이프라인 안정 실행 |
| Level 2 자동 배포 | 평가 후 자동 승격 가능 | 모델 레지스트리, 배포 파이프라인, 롤백 규칙 | 배포 실패 평균 복구 시간 30분 이내 |
| Level 3 관측 기반 운영 | 배포 후 상태를 지속 관찰 | 지연시간/오류율/예측분포 모니터링, 드리프트 경고 | 이상 탐지 후 대응 런북 실행률 90% 이상 |
| Level 4 폐쇄 루프 운영 | 감지-재학습-승격 루프 자동화 | 재학습 트리거, 챔피언-챌린저 비교, 승인 정책 | 분기별 운영 리뷰에서 수동 개입 지속 감소 |

이 표의 핵심은 기술 스택 목록이 아니라 운영 계약입니다. 팀이 지금 어느 단계인지 판단할 때는 도구 도입 여부보다 재현성, 복구성, 관측 가능성을 먼저 점검하는 편이 정확합니다.

## 전통적 CI/CD와 ML CI/CD의 차이

MLOps를 DevOps의 확장이라고 말할 때 가장 자주 놓치는 부분은 "검증 대상"이 다르다는 점입니다. 전통적 CI/CD는 코드 동작 검증이 중심이고, ML CI/CD는 코드와 데이터가 결합된 모델 행태 검증이 중심입니다.

| 비교 항목 | 전통적 CI/CD | ML CI/CD |
|---|---|---|
| 변경 입력 | 코드 변경 | 코드 + 데이터 + 피처 정의 + 하이퍼파라미터 |
| 빌드 산출물 | 바이너리, 컨테이너 이미지 | 모델 아티팩트 + 메타데이터 + 이미지 |
| 테스트 기준 | 단위/통합/E2E 통과 | 테스트 통과 + 성능 기준 + 드리프트 위험도 |
| 배포 판단 | 기능 정상 여부 | 기능 + 예측 품질 + 운영 안정성 |
| 롤백 기준 | 오류율, 장애 | 오류율 + 품질 저하 + 분포 이상 |

즉, ML CI/CD는 "코드가 돌아간다"에서 끝나지 않습니다. "모델이 운영에서 여전히 의미 있는 예측을 한다"를 포함해야 합니다. 따라서 테스트 단계에서 데이터 스키마 검사, 피처 누락 검사, 기준 모델 대비 성능 비교가 기본으로 들어가야 합니다.

## 최소 CI/CD 파이프라인 예시

아래 예시는 모델 리포지토리에서 자주 쓰는 최소 파이프라인 흐름입니다.

```yaml
name: ml-ci-cd
on:
  push:
    branches: [main]

jobs:
  train-and-validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Install dependencies
        run: pip install -r requirements.txt
      - name: Run unit tests
        run: pytest -q
      - name: Train model
        run: python train.py --config configs/train.yaml
      - name: Validate model quality
        run: python evaluate.py --min-auc 0.82 --max-latency-ms 80
      - name: Register model
        run: python register_model.py --stage Staging
```

이 파이프라인의 의도는 단순 자동 실행이 아닙니다. 학습과 평가를 동일한 릴리스 단위로 묶어, 운영에 올릴 수 있는 모델만 다음 단계로 보내는 데 있습니다.

## 도입 우선순위 결정법

MLOps 도입 시 팀이 가장 많이 막히는 지점은 "무엇부터 해야 하는가"입니다. 아래 순서가 대체로 실패 확률이 낮습니다.

1. 실험 추적과 데이터 버전 연결부터 고정합니다.
2. 학습 파이프라인을 단계별로 분해해 재실행 가능하게 만듭니다.
3. 모델 레지스트리와 승격 기준을 문서화합니다.
4. 배포 후 모니터링과 드리프트 경고를 붙입니다.
5. 마지막으로 재학습 자동화를 연결합니다.

이 순서를 지키면 자동화가 운영 위험을 키우는 상황을 줄일 수 있습니다. 반대로 관측 체계 없이 재학습부터 자동화하면, 잘못된 모델이 더 빠르게 배포되는 역효과가 생길 수 있습니다.

## 정리

MLOps는 모델을 더 잘 학습시키는 비법이 아니라, 모델이 운영 환경에서 계속 작동하도록 만드는 체계입니다. 핵심은 데이터, 코드, 모델, 운영 신호를 하나의 루프로 묶는 데 있습니다.

이 글에서 잡아야 할 문장은 하나입니다. **좋은 모델은 실험실에서 나오지만, 살아남는 모델은 MLOps 루프 안에서 만들어진다**는 점입니다. 다음 글에서는 그 루프의 첫 실무 도구인 실험 관리를 다루겠습니다.

## 처음 질문으로 돌아가기

- **MLOps는 단순히 ML에 DevOps를 붙인 말과 무엇이 다를까요?**
  - 이 글에서 MLOps는 학습 자동화만이 아니라 데이터 해시, 모델 레지스트리, 예측 로그, 재학습 트리거를 하나의 운영 루프로 묶는 체계입니다. 전통적 CI/CD가 코드 배포에 집중한다면, MLOps는 드리프트 감지와 CT까지 포함해 모델이 시간이 지나도 살아남게 만드는 데 초점을 둡니다.
- **왜 정확도가 높은 모델도 운영 단계에서 금방 문제가 생길까요?**
  - 본문에서 본 것처럼 입력 분포는 KS 검정으로 잡힐 만큼 쉽게 바뀌고, 예측 분포와 성능도 운영 중 계속 흔들릴 수 있습니다. 데이터 버전, 예측 로그, 재학습 기준이 없으면 로컬에서 잘 나온 모델도 배포 뒤에는 왜 성능이 떨어졌는지 설명할 근거가 사라집니다.
- **데이터, 코드, 모델을 함께 관리한다는 말은 실제로 무엇을 뜻할까요?**
  - 작은 예제에서도 `snap`으로 데이터 버전을 만들고, `registry/model_v1.pkl`에 모델을 저장하고, `meta` 딕셔너리로 메트릭과 모델 버전을 함께 남겼습니다. 즉 어떤 데이터와 어떤 코드에서 어떤 모델이 나왔는지 연결 고리를 남겨, 나중에 같은 결과를 다시 만들고 문제를 추적할 수 있게 하는 일입니다.
