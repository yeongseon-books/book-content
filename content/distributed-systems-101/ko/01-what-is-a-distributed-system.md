# Distributed Systems 101 (1/10): 분산 시스템이란 무엇인가?

서비스를 둘로 나누는 순간, 같은 코드를 돌려도 더 이상 같은 세상에 살지 않습니다. 네트워크가 끼고, 한쪽만 느려지거나 죽을 수 있고, 각 노드가 보는 시간도 조금씩 다르기 때문입니다.

여기서는 그 차이를 지연, 장애, 조정이라는 세 축으로 정리해 두고, 이후 글 전체를 읽을 때 기준점이 되는 멘탈 모델을 세웁니다.

## 먼저 던지는 질문

- 분산 시스템은 정확히 무엇이며 단일 머신 프로그램과 무엇이 다를까요?
- 지연, 장애, 조정이라는 세 축은 왜 분산 시스템의 기본 언어일까요?
- 분산 컴퓨팅의 8가지 허상은 왜 지금도 반복될까요?

## 왜 중요한가

오늘 만드는 거의 모든 서비스는 사실상 분산 시스템입니다. 복제본이 하나라도 붙은 데이터베이스는 이미 분산 시스템이고, 두 개의 마이크로서비스가 서로 통신하는 순간도 분산 시스템입니다. 즉시 응답한다, 항상 성공한다, 시계는 하나다 같은 단일 머신의 직관으로 작성한 코드는 실제 트래픽을 만나면 곧바로 흔들립니다.

> 분산 시스템은 단일 머신 프로그램의 가정이 깨지는 정확한 지점에서 시작됩니다.

## 한눈에 보는 개념

이 그림의 화살표 하나하나는 지연과 부분 장애, 그리고 응답 불확실성을 품고 있습니다. 이것이 단일 함수 호출과 본질적으로 다른 점입니다.

## 핵심 용어

- **분산 시스템**: 독립적으로 동작하는 여러 노드가 메시지 전달을 통해 협력하는 시스템입니다.
- 지연: 메시지가 상대편에 도달하는 데 걸리는 시간입니다.
- 장애: 노드, 네트워크, 디스크가 일부 또는 전체로 멈추는 현상입니다.
- 조정: 여러 노드가 하나의 결정을 함께 맞추는 과정입니다.
- **부분 장애**: 어떤 노드는 살아 있고 어떤 노드는 죽어 있는 상태입니다. 분산 환경의 대표적인 조건입니다.

## 적용 전후 비교
**Before — 단일 머신 직관**

```text
calls finish instantly / always succeed / there is one clock
```

**After — 분산 환경**

```text
calls take ms to s / can fail in part / each node has its own clock
```

이 단순한 전환이 이 시리즈의 거의 모든 주제를 만듭니다. 재시도, 타임아웃, 합의는 모두 여기서 출발합니다.

## 실습: 차이를 몸으로 느껴 보기

### 1단계 — 로컬 함수 호출

```python
# 1_local.py
def add(a, b):
    return a + b

print(add(1, 2))  # 3, instantly
```

이 호출은 마이크로초 수준에서 끝납니다. 실패를 따로 걱정할 이유도 없습니다.

### 2단계 — 같은 머신의 다른 프로세스 호출(HTTP)

```python
# 2_local_http.py
# pip install fastapi uvicorn requests
from fastapi import FastAPI
app = FastAPI()
@app.get("/add")
def add(a: int, b: int): return {"r": a + b}
# run: uvicorn 2_local_http:app --port 8001
```

```python
# 2_client.py
import requests
print(requests.get("http://127.0.0.1:8001/add", params={"a":1,"b":2}, timeout=1).json())
```

같은 머신 안인데도 지연은 밀리초 단위로 뛰어오릅니다. 이것이 첫 번째 분리의 비용입니다.

### 3단계 — 서버를 죽여 보기

```bash
# after killing the server with ctrl+c
python3 2_client.py
# requests.exceptions.ConnectionError
```

호출자는 서버의 실제 상태를 알지 못합니다. 이런 종류의 오류는 단일 머신 함수 호출에서는 나타나지 않았습니다.

### 4단계 — 응답이 느리면 어떻게 될까요?

```python
# 4_slow.py
@app.get("/slow")
def slow():
    import time; time.sleep(5)
    return {"ok": True}
```

```python
requests.get("http://127.0.0.1:8001/slow", timeout=1)
# requests.exceptions.ReadTimeout
```

타임아웃이 없으면 호출은 5초 동안 그대로 붙잡힙니다. 분산 시스템에서 타임아웃은 옵션이 아니라 기본 장치입니다.

### 5단계 — 노드 간 시계 어긋남

```python
# 5_clock.py
import time
print("server time:", time.time())
# 같은 코드를 다른 머신에서 실행하면 두 값은
# NTP를 사용해도 정확히 일치하지 않음(ms 단위 drift)
```

실제 순서를 벽시계 시간으로 결정하면 안 됩니다. 6편의 합의와 8편의 메시지 순서 이야기가 이 문제를 직접 다룹니다.

## 이 코드에서 먼저 봐야 할 점

- 네트워크가 끼는 순간 같은 호출도 전혀 다른 종류의 오류를 갖게 됩니다.
- 타임아웃, 재시도, 멱등성은 단일 머신에서는 거의 의식하지 않던 개념입니다.
- 시계는 정확히 일치하지 않습니다.
- 성공과 실패 사이에 모름이라는 상태가 새로 생깁니다.

## 자주 하는 실수 5가지

1. **타임아웃 없이 호출합니다.** 응답이 영원히 오지 않을 수 있습니다.
2. **멱등성 없이 재시도합니다.** 대표적인 결과가 중복 결제입니다.
3. **벽시계 시간으로 순서를 정합니다.** 노드마다 보는 시간이 다릅니다.
4. **부분 장애를 무시합니다.** 실제 운영에서 가장 흔한 상태입니다.
5. **단일 머신 지연으로 용량을 계산합니다.** 네트워크 지연이 전체 예산을 지배합니다.

## 지연의 물리적 한계와 설계 예산

지연은 "빠르게 하면 해결된다"는 문제가 아닙니다. 물리적으로 제거할 수 없는 하한이 존재하기 때문입니다.

| 구간 | 하한 | 전형적 값 | 최악 (P99.9) |
|------|------|-----------|-------------|
| 같은 프로세스 함수 호출 | < 1ns | < 1us | < 10us |
| 같은 머신 IPC (Unix socket) | ~1us | 10~50us | 200us |
| 같은 AZ 내 TCP RPC | ~100us | 0.5~2ms | 10ms |
| 같은 리전 다른 AZ | ~0.5ms | 1~3ms | 20ms |
| 서울 → 도쿄 (크로스 리전) | ~10ms | 30~50ms | 200ms |
| 서울 → US-East | ~60ms | 80~120ms | 500ms |

빛의 속도는 서울에서 버지니아까지 편도 약 60ms를 강제합니다. 이것은 코드 최적화로 줄일 수 없는 물리 법칙입니다. 분산 시스템 설계에서 "지연 예산"이란 SLO(예: P99 응답 200ms)에서 이 물리적 하한을 빼고 남은 시간을 어떻게 쓸 것인가의 문제입니다.

### 지연 예산 계산 예시

```text
SLO: P99 응답 시간 200ms
─────────────────────────────────
네트워크 (클라이언트→LB→서버): 5ms
DB 쿼리 (같은 AZ, 인덱스 hit):  3ms
외부 API (같은 리전):            30ms
직렬화/역직렬화:                  2ms
비즈니스 로직:                    10ms
─────────────────────────────────
합계:                            50ms
여유:                            150ms  ← 재시도 1회 가능
```

여유가 충분하면 재시도 정책을 공격적으로 잡을 수 있습니다. 여유가 없으면 재시도 대신 빠른 실패(fail-fast)와 비동기 보정을 선택해야 합니다. 이 판단이 분산 시스템 설계의 일상입니다.

### 꼬리 지연(tail latency)이 위험한 이유

P50이 2ms인 서비스라도 P99.9가 500ms라면, fan-out 호출에서 문제가 됩니다. 예를 들어 하나의 요청이 10개의 백엔드를 병렬로 호출하면, 그중 하나라도 꼬리 지연에 걸릴 확률은 다음과 같습니다.

```text
P(하나라도 느림) = 1 - (1 - 0.001)^10 ≈ 1%
```

10개 중 하나가 느릴 확률 1%는 무시할 수 없습니다. 호출 대상이 100개로 늘어나면 확률은 약 9.5%가 됩니다. 이것이 fan-out 시스템에서 꼬리 지연 관리가 필수인 이유이고, hedged request나 backup request 같은 기법이 존재하는 이유입니다.

## 부분 장애의 분류

단일 머신에서 장애는 "전부 정상" 또는 "전부 정지" 둘 중 하나에 가깝습니다. 분산 환경에서는 상태 공간이 훨씬 넓어집니다.

| 장애 유형 | 설명 | 예시 | 호출자가 아는 것 |
|-----------|------|------|-----------------|
| 노드 크래시 | 프로세스가 종료됨 | OOM kill, segfault | connection refused |
| 네트워크 단절 | 노드는 살아 있으나 메시지 도달 불가 | 스위치 장애, 보안 그룹 오설정 | timeout |
| 느린 노드 | 응답은 하지만 매우 느림 | GC pause, 디스크 I/O 포화 | timeout (크래시와 구분 불가) |
| 비잔틴 | 잘못된 응답을 반환 | 메모리 비트 플립, 악의적 노드 | 잘못된 데이터를 정상으로 믿음 |
| 부분 성공 | 일부 복제본만 기록 성공 | 3노드 중 2노드만 write ack | 다수결 기준에 따라 다름 |

호출자 관점에서 가장 어려운 것은 "느린 노드"와 "네트워크 단절"을 구분할 수 없다는 점입니다. 둘 다 타임아웃으로 나타나기 때문입니다. 이 모호함을 다루는 방법이 2편(장애 모델)의 핵심 주제입니다.

### 장애가 전파되는 경로

```text
DB 슬로우 쿼리 (디스크 I/O 포화)
  → 커넥션 풀 고갈 (대기 큐 증가)
    → 서비스 A 응답 지연 (스레드 점유)
      → 서비스 B 타임아웃 증가
        → 로드밸런서 헬스체크 실패
          → 트래픽 재분배 → 남은 노드 과부하
            → 전체 장애 (cascading failure)
```

이 전파 경로가 보여주는 것은 부분 장애가 방치되면 전체 장애로 확대된다는 사실입니다. 10편에서 다루는 circuit breaker, bulkhead, load shedding은 이 전파를 차단하는 운영 패턴입니다.

## 분산 시스템의 시간 문제

단일 머신에서는 `time.time()`이 사건의 순서를 결정하는 데 충분합니다. 분산 환경에서는 그렇지 않습니다.

### 벽시계와 단조 시계의 차이

| 속성 | 벽시계 (wall clock) | 단조 시계 (monotonic clock) |
|------|--------------------|-----------------------------|
| 용도 | "지금 몇 시인가" | "얼마나 시간이 흘렀는가" |
| NTP 보정 | 받음 (점프 가능) | 받지 않음 |
| 뒤로 갈 수 있음 | 예 (NTP step) | 아니오 |
| 노드 간 비교 | 불가 (drift 존재) | 불가 (로컬 전용) |
| Python API | `time.time()` | `time.monotonic()` |

NTP는 보통 수 밀리초 이내로 시계를 맞추지만, 네트워크 장애나 VM 마이그레이션 후에는 수 초 이상 점프할 수 있습니다. 이벤트 순서를 벽시계로 결정하면, 시계가 뒤로 간 노드가 "과거에" 쓴 것처럼 보이는 쓰기를 만들 수 있습니다.

실제 사례로, AWS에서 VM을 라이브 마이그레이션하면 벽시계가 수백 밀리초 점프하는 경우가 보고됩니다. Google Spanner가 TrueTime API를 만든 이유도 이 문제를 하드웨어 수준에서 해결하기 위해서입니다.

### 논리 시계: 순서를 보장하는 대안

```python
# Lamport clock 최소 구현
class LamportClock:
    def __init__(self):
        self._counter = 0

    def tick(self) -> int:
        """로컬 이벤트 발생 시 호출"""
        self._counter += 1
        return self._counter

    def send(self) -> int:
        """메시지 전송 시 호출, 반환값을 메시지에 첨부"""
        self._counter += 1
        return self._counter

    def receive(self, remote_ts: int) -> int:
        """메시지 수신 시 호출"""
        self._counter = max(self._counter, remote_ts) + 1
        return self._counter
```

Lamport clock은 인과 관계(causality)를 보장합니다. a가 b보다 먼저 발생했다면 `clock(a) < clock(b)`입니다. 단, 역은 성립하지 않습니다. 이 한계를 해결한 것이 Vector Clock이고, 5편(복제)에서 상세히 다룹니다.

## 실무에서는 이렇게 드러납니다

현실의 거의 모든 웹 백엔드는 분산 시스템입니다. 복제와 장애 조치를 갖춘 관계형 데이터베이스도 그렇고, Redis Cluster, Kafka, Cassandra는 더 말할 것도 없습니다. 클라우드의 AZ 이중화, 멀티 리전 배치, CDN도 모두 분산 시스템 설계의 일부입니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 단일 머신의 직관을 의도적으로 의심합니다.
- 타임아웃, 재시도, 멱등성을 첫 줄부터 설계합니다.
- 시스템 모델 안에 모름 상태를 반드시 넣습니다.
- 벽시계는 표시용으로만 보고, 단조 증가 시계를 더 신뢰합니다.
- 굳이 분산이 필요 없는 상황에서는 단일 머신으로 끝내는 선택도 합니다.

## 분산 컴퓨팅의 8가지 허상

1991년에 정리된 이 목록은 30년이 지난 지금도 분산 시스템 입문자가 가장 자주 밟는 지뢰입니다.

| 허상 | 현실 |
|------|------|
| 네트워크는 신뢰할 수 있다 | 패킷 유실, 지연, 순서 역전이 일상입니다 |
| 지연은 0이다 | 같은 데이터센터 안에서도 수백 마이크로초가 걸립니다 |
| 대역폭은 무한하다 | 직렬화 크기가 커지면 병목이 됩니다 |
| 네트워크는 안전하다 | 중간자 공격, 인증 미비가 실제로 발생합니다 |
| 토폴로지는 변하지 않는다 | 클라우드에서 IP와 경로는 수시로 바뀝니다 |
| 관리자가 한 명이다 | 여러 팀, 여러 클라우드 계정이 얽힙니다 |
| 전송 비용은 0이다 | 직렬화, 역직렬화 자체가 CPU를 먹습니다 |
| 네트워크는 균일하다 | 리전 간 RTT는 AZ 내부의 수십 배입니다 |

이 허상을 코드 리뷰 체크리스트로 바꾸면 아래처럼 됩니다.

```text
- [ ] 모든 외부 호출에 타임아웃이 있는가?
- [ ] 네트워크 실패 시 재시도 정책이 멱등성과 함께 설계되었는가?
- [ ] 직렬화 크기가 측정되고 있는가?
- [ ] TLS/인증이 적용되었는가?
- [ ] 배포 스크립트가 IP 변경을 가정하고 있는가?
```

## 세 축을 하나의 설계 언어로 엮기

지연, 장애, 조정은 독립적으로 존재하지 않습니다. 지연이 길어지면 장애처럼 보이고, 장애를 감지하려면 조정이 필요하며, 조정에는 다시 지연이 붙습니다. 이 세 축이 순환적으로 작용한다는 사실을 이해하면, 분산 시스템의 거의 모든 설계 결정이 왜 트레이드오프인지 자연스럽게 보입니다.

### 지연이 장애로 보이는 상황

```python
# timeout_ambiguity.py
import requests

try:
    r = requests.get("http://payment-service/charge", timeout=2.0)
except requests.exceptions.ReadTimeout:
    # 서버가 죽은 건지, 느린 건지, 응답이 유실된 건지 알 수 없습니다.
    # 이 세 가지는 호출자 관점에서 구분 불가능합니다.
    pass
```

이 모호함은 분산 시스템의 가장 근본적인 어려움 중 하나입니다. 타임아웃이 발생하면 호출자는 세 가지 가능성을 모두 열어 두어야 합니다. 첫째, 서버가 요청을 받지 못했을 수 있습니다. 둘째, 서버가 요청을 처리했지만 응답이 유실되었을 수 있습니다. 셋째, 서버가 아직 처리 중일 수 있습니다.

이 세 가지를 구분하지 못하는 상태가 바로 **모름(unknown)**이고, 분산 시스템에서 모름은 성공이나 실패보다 더 다루기 어려운 상태입니다. 재시도가 안전하려면 서버 쪽이 멱등적이어야 하고, 멱등성 없이 재시도하면 중복 부작용이 생깁니다.

### 장애 감지에 조정이 필요한 이유

노드가 죽었는지 살았는지를 판단하려면 적어도 한 노드 이상이 합의해야 합니다. 혼자서 "상대가 죽었다"고 판단하면 split-brain이 생깁니다. 이것이 6편에서 다루는 합의가 필요한 근본 이유입니다.

```text
Node A: "B는 타임아웃이니 죽었다"
Node C: "B는 나한테 방금 응답했다"
→ 누구 말이 맞는가? → 합의 필요
```

### 조정에 지연이 붙는 현실

합의를 맺으려면 메시지를 주고받아야 하고, 메시지에는 지연이 붙습니다. 5노드 Raft 클러스터에서 한 번의 쓰기를 확정하려면 최소 1 RTT(리더→팔로워→리더)가 필요하고, 리더 선출에는 더 많은 라운드가 필요합니다.

| 동작 | 필요한 네트워크 라운드트립 | 예상 지연 (같은 리전) |
|------|--------------------------|---------------------|
| 로컬 함수 호출 | 0 | < 1us |
| 같은 AZ 내 RPC | 1 | 0.5~2ms |
| Raft 쓰기 확정 | 1~2 | 2~10ms |
| 리더 선출 | 2~4 | 150~600ms |
| 크로스 리전 합의 | 2~4 | 50~200ms |

이 표가 보여주는 것은 단순합니다. 조정은 비쌉니다. 조정이 필요 없는 설계가 가능하면 그것이 최선이고, 조정이 반드시 필요하면 그 비용을 SLO에 반영해야 합니다.

## 분산 시스템을 만들어야 할 때와 만들지 말아야 할 때

분산 시스템은 목적이 아니라 수단입니다. 다음 조건 중 하나라도 해당되지 않으면 단일 머신으로 충분한지 먼저 확인해야 합니다.

**분산이 필요한 신호:**

- 단일 머신의 CPU, 메모리, 디스크 I/O가 워크로드를 감당하지 못합니다.
- 한 머신이 죽어도 서비스가 계속 살아 있어야 합니다(가용성 요구).
- 데이터가 물리적으로 여러 위치에 있어야 법적 요건을 충족합니다.
- 여러 팀이 독립 배포해야 하는 조직 구조입니다.

**단일 머신으로 충분한 신호:**

- 트래픽이 단일 서버로 감당됩니다.
- 잠깐의 다운타임이 비즈니스에 치명적이지 않습니다.
- 데이터 크기가 한 디스크에 들어갑니다.
- 팀 규모가 작아 배포 병목이 없습니다.

SQLite로 충분한 워크로드에 Kafka와 Cassandra를 붙이면 복잡도만 올라갑니다. "왜 분산해야 하는가"를 먼저 대답하지 못하면, 시스템을 나누는 행위 자체가 비용입니다.

## 운영 시나리오: 첫 번째 분산 장애를 만났을 때

실제 운영에서 분산 시스템의 본질을 처음 체감하는 순간은 대개 이런 식입니다.

1. 주문 서비스가 결제 서비스를 호출합니다. 타임아웃 2초.
2. 결제 서비스가 외부 PG사 API를 호출합니다. PG사 응답이 3초로 늦어집니다.
3. 주문 서비스는 타임아웃을 받고 사용자에게 "실패"를 반환합니다.
4. 하지만 PG사는 실제로 결제를 승인했고, 결제 서비스도 곧 성공 응답을 받습니다.
5. 결과: 사용자는 "실패"를 봤지만 돈은 빠져나간 상태입니다.

이 시나리오가 보여주는 교훈은 명확합니다.

- 타임아웃은 "상대가 실패했다"는 의미가 아니라 "모른다"는 의미입니다.
- 모름 상태를 처리하려면 멱등성 키 기반 조회나 보상 로직이 필요합니다.
- 단일 머신에서는 함수 반환값이 곧 결과였지만, 분산 환경에서는 반환값과 실제 상태가 다를 수 있습니다.

```python
# 멱등성 키로 결과를 확인하는 패턴
def place_order(idempotency_key: str):
    # 1. 이미 처리된 키인지 확인
    existing = db.get_order_by_key(idempotency_key)
    if existing:
        return existing  # 중복 요청은 기존 결과를 반환

    # 2. 결제 호출
    try:
        payment = payment_service.charge(
            amount=10000,
            idempotency_key=idempotency_key,
            timeout=2.0,
        )
    except TimeoutError:
        # 모름 상태: 나중에 확인해야 합니다
        return {"status": "pending", "key": idempotency_key}

    # 3. 성공 시 주문 저장
    order = db.create_order(key=idempotency_key, payment_id=payment.id)
    return order
```

이 코드에서 핵심은 세 가지입니다. 첫째, 멱등성 키로 중복 호출을 안전하게 흡수합니다. 둘째, 타임아웃 시 "실패"가 아니라 "보류"로 응답합니다. 셋째, 별도 보정 워커가 보류 상태를 주기적으로 확인해 최종 상태로 수렴시킵니다.

## 정리와 다음 글

분산 시스템은 지연, 장애, 조정이라는 세 축에서 단일 머신 프로그램과 본질적으로 다릅니다. 다음 글에서는 그중 장애를 더 정밀하게 다루기 위해 crash, omission, Byzantine 같은 장애 모델을 살펴봅니다.

## 처음 질문으로 돌아가기

- **분산 시스템은 정확히 무엇이며 단일 머신 프로그램과 무엇이 다를까요?**
  - 독립 노드들이 메시지로 협력하는 시스템이며, 지연·부분 장애·시계 불일치라는 세 가지 조건이 단일 머신과 본질적으로 다릅니다. 함수 호출의 "즉시·항상·하나의 시계" 가정이 모두 깨집니다.
- **지연, 장애, 조정이라는 세 축은 왜 분산 시스템의 기본 언어일까요?**
  - 지연이 장애처럼 보이고, 장애 감지에 조정이 필요하며, 조정에 지연이 붙는 순환 구조이기 때문입니다. 이 세 축을 빼면 분산 시스템의 어떤 설계 결정도 설명할 수 없습니다.
- **분산 컴퓨팅의 8가지 허상은 왜 지금도 반복될까요?**
  - 단일 머신 경험이 직관을 만들고, 그 직관이 네트워크 너머에서도 유효하다고 무의식적으로 가정하기 때문입니다. 허상을 깨는 유일한 방법은 실제 장애를 경험하거나, 장애를 주입해 미리 경험하는 것입니다.
