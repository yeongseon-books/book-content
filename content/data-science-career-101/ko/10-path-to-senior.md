# Data Science Career 101 (10/10): 시니어 데이터 직무로 가는 길

이 글은 Data Science Career 101 시리즈의 마지막 글입니다.


시니어로 성장한다는 말을 들으면 많은 사람이 먼저 기술 난도를 떠올립니다. 더 어려운 모델을 다루고, 더 복잡한 파이프라인을 만들고, 더 빨리 문제를 해결하는 모습이 시니어의 전부처럼 보이기 쉽습니다. 물론 그런 역량도 중요하지만, 실제로는 거기서부터 차이가 시작되지는 않습니다.

더 큰 변화는 문제를 보는 관점에서 생깁니다. 무엇을 만들었는지보다 무엇을 바꿨는지, 내가 혼자 잘한 일보다 팀이 더 잘 움직이게 되었는지, 당장 급한 일보다 반복될 문제의 구조를 먼저 잡았는지가 시니어의 행동을 가릅니다.


## 먼저 던지는 질문

- 시니어 데이터 직무가 무엇이 다른지 설명합니다?
- 문제 정의 능력이 왜 시니어의 핵심 역량인지 정리합니다?
- 영향 측정을 어떻게 계획해야 설득력이 생기는지 짚습니다?

## 왜 중요한가

시니어는 직함보다 행동에 가깝습니다. 같은 기술 수준이어도 어떤 문제를 고르고, 어떻게 영향력을 넓히는지에 따라 역할의 무게가 달라집니다.

그래서 시니어 성장은 단순한 기술 숙련의 연장이 아닙니다. 문제 정의, 영향 측정, 문서화, 동료 증폭, 시스템 관점이 함께 붙으면서 같은 기술 역량도 훨씬 더 큰 결과로 이어집니다.

## 한눈에 보는 개념

이 흐름은 단순 승진 사다리가 아니라 관점의 확장을 뜻합니다. 주니어가 자신의 작업을 안정적으로 끝내는 데 집중한다면, 시니어는 문제 선정과 영향 측정, 동료 성장, 시스템 전체를 함께 봅니다.

## 핵심 용어

- **scope**: 책임이 미치는 범위입니다.
- **impact**: 결과가 제품과 조직에 미치는 효과입니다.
- **multiplier**: 동료의 성과까지 키우는 증폭 효과입니다.
- **systems thinking**: 끝에서 끝까지 연결해 보는 관점입니다.
- **mentorship**: 동료의 성장을 돕는 활동입니다.

## 전후 비교

**Before**: "주어진 티켓을 빠르게 끝내는 것이 가장 중요하다고 생각했다."

**After**: "문제를 정의하고, 측정 계획을 붙여 제안하고, 동료 성과까지 키우는 쪽으로 시야를 넓힐 수 있다."

## 실습: 다섯 가지 시니어 행동

### Step 1 — Define the Problem

```text
Write the "problem", not the "symptom".
```

시니어는 증상과 문제를 구분합니다. 증상을 바로 고치기보다 왜 그런 현상이 생겼는지를 문제 정의 단계에서 분리해 내야 합니다.

### Step 2 — Measure Impact

```markdown
- hypothesis
- measurement plan
- result
```

좋은 제안은 실행 전에 이미 측정 계획이 붙어 있습니다. 무엇이 좋아질 것이고, 어떻게 확인할지까지 말할 수 있어야 설득력이 생깁니다.

### Step 3 — Mentor

```text
- monthly 1:1
- mentee sets the agenda
```

시니어는 자기 성과만 내는 사람이 아닙니다. 동료의 성장을 돕는 행동이 팀 전체의 생산성을 올립니다.

### Step 4 — Decision Records

```text
- decision / alternatives / rationale / impact
```

의사결정 기록은 단순 메모가 아닙니다. 나중에 맥락을 복원하고 같은 논의를 반복하는 비용을 줄이는 장치입니다.

### Step 5 — Systems Thinking

```text
- data -> model -> decision -> operation -> retro
```

시니어는 한 부분만 최적화하지 않습니다. 데이터 수집부터 운영과 회고까지 전체 흐름을 보고 병목을 찾습니다.

## 이 예시에서 먼저 봐야 할 점

- 문제 정의가 곧 일의 품질입니다.
- 측정이 있어야 영향이 증명됩니다.
- 동료 성장이 곧 팀의 증폭 효과입니다.

주니어에서 시니어로 가는 가장 큰 변화는 “무엇을 만들었는가”에서 “무엇이 달라졌는가”로 관심이 이동하는 점입니다. 구현보다 영향, 재사용성, 팀 확장이 더 중요해집니다.

## 자주 하는 실수 5가지

1. **기술만 깊게 파는 실수**
2. **영향 측정을 빼먹는 실수**
3. **멘토링을 나중으로 미루는 실수**
4. **결정 기록을 남기지 않는 실수**
5. **시스템 전체를 보지 않는 실수**

## 실무에서는 이렇게 나타납니다

큰 회사의 시니어 트랙은 Staff, Principal 같은 단계로 이어지기도 합니다. 공통점은 직함보다 영향 범위가 커진다는 점입니다. 더 모호한 문제를 다루고, 더 많은 이해관계자를 조율하고, 더 긴 시간 축에서 결과를 측정합니다.

## 시니어는 이렇게 생각합니다

- 문제 정의를 먼저 합니다.
- 영향 측정을 계획 단계에서 붙입니다.
- 동료를 증폭시키는 행동을 의식합니다.
- 중요한 결정을 문서로 남깁니다.
- 시스템 전체 관점에서 병목과 리스크를 봅니다.

## 체크리스트

- [ ] 문제 정의 템플릿을 만들었다.
- [ ] 측정 계획이 포함된 제안을 한 번 써 봤다.
- [ ] 멘토링 대상 한 명을 정했다.
- [ ] 의사결정 기록 템플릿을 준비했다.

## 정리 및 다음 단계

시니어 데이터 직무로 성장한다는 것은 더 많은 일을 혼자 처리하는 사람이 되는 것이 아닙니다. 더 좋은 문제를 정의하고, 영향 측정을 붙여 제안하고, 동료와 시스템까지 함께 보는 사람으로 바뀌는 일에 가깝습니다.

이 시리즈는 여기서 마무리됩니다. 데이터 커리어는 기술, 도메인, 사람, 그리고 문제를 보는 관점이 함께 쌓이면서 길이 만들어집니다.


## 5년 커리어 로드맵 예시와 학습 계획 표

시니어로 가는 길은 한 번의 승진 이벤트가 아니라 5년 단위 누적 전략에 가깝습니다. 단기 성과만 쫓으면 경력은 바쁘지만 방향이 흐려질 수 있습니다. 그래서 아래 로드맵은 1년 단위로 역량 목표, 프로젝트 유형, 학습 계획을 연결해 장기 성장 축을 명확히 잡도록 구성했습니다.

| 연차 | 핵심 목표 | 권장 프로젝트 | 학습 초점 | 검증 방식 |
| --- | --- | --- | --- | --- |
| 1년차 | 실행 안정성 확보 | 분석 리포트, 기본 모델, 데이터 정리 | SQL/Python/통계 기초 | 재현 가능한 산출물 3개 |
| 2년차 | 독립 과제 리드 | 실험 분석, 모델 개선, 파이프라인 자동화 | 지표 설계, 실험 해석, 코드 품질 | 분기 단위 오너십 과제 |
| 3년차 | 영향 범위 확장 | 타팀 협업 프로젝트, 도메인 심화 | 커뮤니케이션, 문서화, 도메인 지식 | 의사결정 반영률 상승 |
| 4년차 | 시스템 관점 강화 | 엔드투엔드 개선 과제 | 아키텍처 사고, 운영 안정화 | 성능/비용/속도 동시 개선 |
| 5년차 | 시니어 역할 정착 | 조직 표준화, 멘토링, 전략 과제 | 리더십, 우선순위 설계 | 팀 생산성 증폭 증거 |

학습 계획은 "무엇을 공부할까"보다 "무엇을 만들고 검증할까"로 적어야 실행력이 올라갑니다. 예를 들어 2년차 목표가 독립 과제 리드라면, 학습 항목은 모델 이론 20시간보다 "실험 설계 문서 1건 + 분석 리뷰 2회"처럼 행동 단위로 쓰는 편이 효과적입니다.

아래는 연간 계획을 분기 단위로 쪼갠 예시입니다.

| 분기 | 계획 예시 |
| --- | --- |
| Q1 | 핵심 역량 진단, 학습 공백 정리, 멘토 피드백 수집 |
| Q2 | 중간 난도 프로젝트 1건 완료, 결과 발표 1회 |
| Q3 | 협업 범위 확장, 타팀 과제 1건 참여 |
| Q4 | 연간 회고, 다음 해 성장 목표 재설계 |

또한 5년 로드맵은 하나의 직무에 고정된 문서가 아니라 "역할 진화"를 반영해야 합니다. 예를 들어 Analyst에서 시작해 Analytics Engineer로 확장하거나, DS에서 MLE로 이동하는 경로도 자연스럽습니다. 중요한 것은 역할 변경 시에도 공통 기반(문제 정의, 지표 사고, 문서화, 협업)을 계속 누적하는 것입니다.

실전 운영 팁도 제안합니다. 첫째, 반기마다 "하지 않을 것" 목록을 작성해 우선순위를 정리하십시오. 둘째, 연 1회는 외부 벤치마크(시장 요구 역량, 도메인 변화)를 반영해 계획을 업데이트하십시오. 셋째, 성과를 숫자와 문장으로 함께 기록해 다음 기회(승진, 이직, 리더십 과제)에 바로 사용할 수 있게 하십시오.

결국 시니어 성장은 시간의 문제가 아니라 설계의 문제입니다. 5년 로드맵을 통해 매년 어떤 문제를 풀고 어떤 역량을 증명할지 명확히 하면, 경력은 우연이 아니라 의도된 누적으로 바뀝니다. 이 시리즈를 마무리하는 시점에서 가장 중요한 다음 행동은 거창한 목표가 아니라, 다음 90일 계획을 실제 일정으로 옮기는 일입니다.


## 실행용 점검 프레임: 10-path-to-senior.md 편

이 섹션은 앞에서 다룬 내용을 실제 행동으로 옮기기 위한 점검 프레임입니다. 많은 독자가 글을 읽을 때는 이해했다고 느끼지만, 일주일 뒤 실행 단계에서 다시 막히는 이유는 "다음 행동"이 문장으로 고정되어 있지 않기 때문입니다. 그래서 아래 프레임은 학습, 실습, 회고를 연결하는 최소 단위로 구성했습니다.

첫째, 이번 주 목표를 한 문장으로 적습니다. 목표는 도구가 아니라 문제 기준으로 써야 합니다. 예를 들어 "SQL 윈도우 함수 공부"보다 "사용자별 누적 결제를 계산해 이탈 신호를 찾는다"처럼 적는 편이 실행력이 높습니다. 같은 시간이라도 문제 중심 목표가 결과 해석까지 자연스럽게 이어집니다.

둘째, 입력과 출력을 명확히 구분합니다. 입력은 데이터, 로그, 문서, 인터뷰 메모처럼 내가 활용할 재료입니다. 출력은 쿼리, 분석 노트, 비교표, 의사결정 문서처럼 다른 사람이 검토 가능한 결과물입니다. 입력을 많이 모으는 것과 출력을 만드는 것은 다르며, 성장 속도는 출력 빈도에 더 강하게 좌우됩니다.

셋째, 판단 기준을 미리 적습니다. 많은 작업이 끝에서 흔들리는 이유는 성공 기준을 사전에 합의하지 않았기 때문입니다. "정확한가"라는 문장 대신 "분모 정의가 명확한가", "누락 세그먼트가 없는가", "후속 액션이 있는가"처럼 점검 문항으로 바꾸면 품질이 안정됩니다.

넷째, 협업 단위를 작게 쪼개 공유합니다. 완성본만 보여 주려는 습관은 피드백 타이밍을 늦춥니다. 중간 결과를 1페이지 요약으로 공유하고, 가정과 리스크를 먼저 드러내면 이후 수정 비용이 크게 줄어듭니다. 데이터 직무는 혼자 완벽하게 만든 뒤 공개하는 방식보다, 가정 기반으로 빠르게 정렬하는 방식이 장기적으로 더 강합니다.

다섯째, 회고는 감상이 아니라 의사결정 기록으로 남깁니다. "어려웠다"로 끝내지 말고 무엇이 병목이었는지, 다음에는 어떤 순서로 접근할지, 어떤 지표를 추가로 볼지를 적어야 합니다. 같은 유형의 문제를 다시 만났을 때 과거 기록이 바로 템플릿이 되면 성장 속도는 가속됩니다.

아래 표는 매주 반복 가능한 최소 운영 템플릿입니다.

| 항목 | 작성 질문 | 예시 출력 |
| --- | --- | --- |
| 목표 | 이번 주에 어떤 문제를 풀 것인가 | 문제 정의 3줄 |
| 입력 | 어떤 데이터/자료를 사용할 것인가 | 데이터 출처 목록 |
| 방법 | 어떤 접근 순서로 진행할 것인가 | 작업 단계 체크리스트 |
| 검증 | 무엇을 성공 기준으로 볼 것인가 | 품질 점검 표 |
| 결과 | 무엇이 바뀌었는가 | 한 문장 권고안 |
| 회고 | 다음 주에 무엇을 바꿀 것인가 | 개선 액션 3개 |

이 템플릿의 핵심은 완벽함이 아니라 반복 가능성입니다. 한 번에 거대한 문서를 만들 필요는 없습니다. 중요한 것은 매주 같은 구조로 남겨서 비교 가능성을 확보하는 일입니다. 비교가 가능해지면 성장도 측정 가능해지고, 측정 가능해지면 커리어 결정의 품질이 올라갑니다.

마지막으로, 실행 프레임은 개인 메모로 끝내지 말고 가능한 범위에서 팀 공유 자산으로 전환하는 것이 좋습니다. 같은 질문이 반복되는 조직일수록 표준 템플릿의 가치가 큽니다. 템플릿 하나를 잘 만드는 사람은 단순히 일을 수행하는 사람이 아니라, 팀이 더 빠르고 정확하게 일하도록 환경을 설계하는 사람으로 인식됩니다.


## 월간 점검 질문 세트

아래 질문은 월말 회고 때 그대로 사용할 수 있는 점검 세트입니다. 첫 번째 질문은 "이번 달에 내가 해결한 문제는 무엇이며, 그 문제는 누구의 의사결정에 영향을 주었는가"입니다. 두 번째 질문은 "내 결과물의 재현 가능성을 다른 동료가 확인할 수 있는가"입니다. 세 번째 질문은 "결과를 한 문장 권고안으로 요약했는가"입니다. 네 번째 질문은 "가정이 틀렸을 때 대안 경로를 제시했는가"입니다. 다섯 번째 질문은 "다음 달에는 무엇을 하지 않을 것인가"입니다.

이 다섯 질문을 매달 반복하면 학습과 실행이 분리되지 않습니다. 특히 마지막 질문은 과부하를 줄이는 데 효과적입니다. 커리어 성장은 더 많은 일을 하는 것만으로 만들어지지 않습니다. 우선순위를 분명히 하고, 덜 중요한 일을 의도적으로 줄일 때 더 빠르게 만들어집니다.

| 질문 | Pass 기준 |
| --- | --- |
| 문제 정의가 명확한가 | 이해관계자가 같은 문장으로 설명 가능 |
| 결과가 재현 가능한가 | 실행 명령과 데이터 기준이 문서화됨 |
| 해석이 행동으로 연결되는가 | 후속 액션이 1개 이상 명시됨 |
| 리스크를 선제적으로 다뤘는가 | 가정과 한계를 함께 기술함 |
| 다음 달 우선순위가 보이는가 | 할 일/하지 않을 일 목록이 분리됨 |

## 시니어 승격 준비표: 영향력 증거 정리

시니어 승격 준비에서 가장 자주 부족한 것은 역량이 아니라 "증거의 정리 방식"입니다. 일을 많이 했더라도 영향이 구조적으로 정리되어 있지 않으면 평가 시 설득력이 약해집니다. 아래 표는 승격 리뷰 전에 정리하면 좋은 증거 항목입니다.

| 증거 영역 | 정리 질문 | 예시 증거 |
| --- | --- | --- |
| 문제 정의 | 모호한 문제를 어떻게 구조화했는가 | 문제 프레이밍 문서, 대안 비교표 |
| 영향 측정 | 결과가 무엇을 얼마나 바꿨는가 | KPI 전후 비교, 비용/시간 절감 수치 |
| 조직 증폭 | 동료가 더 잘 일하게 만든 행동은 무엇인가 | 리뷰 가이드, 멘토링 기록, 표준 템플릿 |
| 시스템 개선 | 반복 문제를 어떻게 줄였는가 | 운영 자동화, 장애 재발 방지 문서 |

승격은 일회성 발표가 아니라 누적 기록의 결과입니다. 따라서 분기마다 위 표를 업데이트하면 연말에 급하게 성과를 모으는 부담이 줄어듭니다. 무엇보다 "내가 잘했다"가 아니라 "팀이 더 좋아졌다"는 증거를 남기는 습관이 시니어 역할 전환을 가장 빠르게 만듭니다.


## 시니어 커리어 래더: 개인 기여자에서 문제 오너로

시니어로 가는 길은 기술 난이도보다 책임 범위의 확장으로 설명하는 편이 정확합니다. 아래 래더는 데이터 직무에서 공통으로 관찰되는 전환 기준입니다.

| 레벨 | 핵심 책임 | 평가 기준 |
| --- | --- | --- |
| Mid | 맡은 문제를 안정적으로 해결 | 품질, 속도, 재현성 |
| Senior | 모호한 문제를 구조화하고 팀을 정렬 | 의사결정 영향력, 우선순위 설계 |
| Staff+ | 조직 단위 기준과 시스템을 설계 | 장기 성과, 팀 생산성 증폭 |

중요한 지점은 Senior부터 "내가 얼마나 많이 했는가"보다 "팀이 더 잘하게 만들었는가"가 평가 중심으로 이동한다는 사실입니다.

## 연봉 협상 고급 스크립트: 역할 범위 기반으로 제안하기

```text
1) 현재 기여: 지난 12개월 동안 만든 핵심 영향 3가지를 수치로 제시
2) 확장 책임: 다음 12개월에 맡을 문제 범위를 명확히 제시
3) 시장 정렬: 동등 역할의 보상 밴드와 차이를 설명
4) 제안안: 기본급/성과급/주식 중 우선 조정 항목을 제시
```

시니어 협상은 "더 달라"가 아니라 "확장된 책임과 보상의 일치"를 만드는 과정입니다. 역할 범위를 문서로 제시하면 협상 난도가 크게 낮아집니다.

## 최종 자기 점검 매트릭스

- 나는 문제를 푸는 사람인가, 문제를 정의하는 사람인가
- 나는 결과를 내는 사람인가, 팀의 기준을 만드는 사람인가
- 나는 프로젝트를 완료하는 사람인가, 시스템을 개선하는 사람인가

이 질문에 대한 답이 선명해질수록 다음 레벨로의 전환이 빨라집니다.


## 실전 부록: 지원 문서와 면접 답변을 한 세트로 준비하는 방법

글을 읽고 이해한 내용을 실제 커리어 행동으로 바꾸려면, 이력서 문장·포트폴리오 증거·면접 답변이 서로 같은 문제를 가리키도록 정렬해야 합니다. 아래 구성은 데이터 직무 입문자와 주니어가 가장 빠르게 효과를 보는 준비 방식입니다.

### 이력서 불릿 템플릿

```text
- [문제] 어떤 비즈니스 또는 운영 문제를 맡았는가
- [행동] 어떤 데이터 접근(지표/실험/파이프라인/모델)을 적용했는가
- [결과] 무엇이 얼마나 개선되었는가
- [의미] 그 결과가 팀 의사결정에 어떤 영향을 주었는가
```

예시 문장: "결제 퍼널 이탈 구간을 코호트 기준으로 분해해 주요 병목을 식별했고, 온보딩 실험 우선순위를 재정렬해 4주 내 전환율을 개선했습니다." 핵심은 기술 이름보다 책임과 결과를 문장으로 남기는 일입니다.

### 포트폴리오 프로젝트 선정 체크

| 점검 항목 | Pass 기준 |
| --- | --- |
| 문제 명확성 | 한 문장으로 문제와 영향 범위를 설명 가능 |
| 데이터 현실성 | 공개 데이터라도 결측/편향/한계를 문서화 |
| 재현 가능성 | 실행 절차와 검증 기준을 README에 명시 |
| 의사결정 연결 | 결과가 다음 액션으로 이어짐 |

이 표를 기준으로 프로젝트를 고르면 "보여주기용 분석"을 줄이고 "실무형 문제 해결"을 남길 수 있습니다.

### 인터뷰 Q&A 3문장 프레임

1. 문제를 어떻게 정의했는지 말합니다.
2. 어떤 접근을 선택했고 왜 그렇게 판단했는지 말합니다.
3. 결과와 한계를 함께 설명하고 다음 액션을 제시합니다.

이 세 문장으로 답변하면 면접관이 가장 궁금해하는 판단 과정이 빠르게 드러납니다. 특히 한계를 먼저 말하는 태도는 신뢰를 높입니다.

### 보상 협상 기본 스크립트

```text
- 제안 감사합니다. 기대 역할과 성공 기준을 먼저 정렬하고 싶습니다.
- 제가 제공할 수 있는 기여 범위와 최근 증거를 공유드리겠습니다.
- 시장 밴드와 총보상 구조를 기준으로 조정 가능한 항목을 논의하고 싶습니다.
```

협상은 대립이 아니라 역할-보상 정렬입니다. 역할 범위를 수치와 산출물로 제시하면 대화가 감정에서 사실로 이동합니다.

### 역량 점검 매트릭스

| 영역 | 현재 수준(1~5) | 목표 수준(1~5) | 다음 2주 액션 |
| --- | --- | --- | --- |
| 문제 정의 |  |  |  |
| 데이터 처리 |  |  |  |
| 검증 설계 |  |  |  |
| 전달력 |  |  |  |
| 협업 운영 |  |  |  |

월 1회 이 표를 채우면 성장 체감이 감정이 아니라 기록으로 남습니다. 커리어 의사결정의 품질은 결국 기록의 품질에서 올라갑니다.
