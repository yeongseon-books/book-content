# Data Science Career 101 (5/10): SQL과 분석 인터뷰

이 글은 Data Science Career 101 시리즈의 5번째 글입니다.


SQL 인터뷰를 준비할 때 많은 지원자가 처음에는 문법 문제집처럼 접근합니다. 어떤 JOIN이 있는지, 윈도우 함수 문법이 무엇인지 외우는 데 집중하지만, 실제 면접에서는 그보다 먼저 질문을 어떻게 분해하고 결과를 어떻게 해석하는지가 드러납니다.

분석 인터뷰도 마찬가지입니다. 쿼리를 쓸 줄 아는 것만으로는 부족하고, 지표 정의를 어떻게 세우는지, NULL과 시간대를 어떻게 다루는지, 숫자를 보고 다음 질문을 무엇으로 이어 갈지까지 보여 줘야 실무 감각이 읽힙니다.

## 왜 중요한가

SQL은 거의 모든 데이터 직무의 공용어입니다. 따라서 SQL 인터뷰는 단순한 쿼리 시험이 아니라, 질문을 숫자로 번역하는 사고력을 보는 자리이기도 합니다.

실제로 많은 면접은 SQL 문제와 제품 해석 질문을 붙여서 냅니다. 쿼리 자체보다도 어떤 가정을 두었는지, 계산 기준을 어떻게 명시하는지, 결과를 보고 어떤 후속 검증을 제안하는지가 함께 평가됩니다.

## 한눈에 보는 개념

좋은 답변은 결과를 뽑는 데서 끝나지 않습니다. 결과가 무엇을 뜻하는지, 다음으로 무엇을 확인해야 하는지까지 말할 수 있어야 완성됩니다.

## 핵심 용어

- **JOIN**: 여러 테이블을 결합하는 연산입니다.
- **GROUP BY**: 기준별로 데이터를 묶어 집계하는 구문입니다.
- **window function**: 행 맥락을 유지하면서 프레임 단위 집계를 수행하는 함수입니다.
- **CTE**: common table expression으로, 쿼리를 단계별로 읽기 쉽게 나누는 방법입니다.
- **funnel**: 단계별 전환율을 보는 분석 방식입니다.

## 전후 비교

**Before**: "문제가 나오면 일단 SELECT *부터 쓴다."

**After**: "질문을 분해하고 CTE로 구조를 나눈 뒤 결과를 해석할 수 있다."

## 실습: 다섯 가지 질문 패턴

### 단계 1 — 단일 테이블 집계
```sql
SELECT date, COUNT(*) AS dau
FROM events
WHERE event = 'login'
GROUP BY date
ORDER BY date;
```

가장 기본적인 패턴입니다. 무엇을 세는지, 어떤 기준으로 묶는지를 명확히 말할 수 있어야 합니다.

### 단계 2 — JOIN

```sql
SELECT u.country, COUNT(o.id) AS orders
FROM users u
LEFT JOIN orders o ON o.user_id = u.id
GROUP BY u.country;
```

JOIN 문제에서는 연결 기준과 NULL 처리까지 같이 봐야 합니다. 왜 `LEFT JOIN`인지 설명할 수 있어야 사고 과정이 드러납니다.

### 단계 3 — 윈도우 함수
```sql
SELECT user_id, amount,
       SUM(amount) OVER (PARTITION BY user_id ORDER BY ts) AS cum
FROM payments;
```

윈도우 함수는 누적 합, 순위, 이동 평균처럼 맥락이 있는 계산에 자주 쓰입니다. 인터뷰에서는 문법보다 `PARTITION BY`와 `ORDER BY`의 의미를 설명하는 힘이 중요합니다.

### 단계 4 — Funnel

```sql
WITH steps AS (
  SELECT user_id,
         MAX(CASE WHEN step='visit' THEN 1 ELSE 0 END) AS s1,
         MAX(CASE WHEN step='signup' THEN 1 ELSE 0 END) AS s2,
         MAX(CASE WHEN step='purchase' THEN 1 ELSE 0 END) AS s3
  FROM funnel GROUP BY user_id
)
SELECT SUM(s1), SUM(s2), SUM(s3) FROM steps;
```

퍼널 문제는 제품 분석 직군에서 특히 자주 나옵니다. 단계 정의와 사용자 중복 처리 방식에 따라 결론이 크게 달라집니다.

### 단계 5 — 한 문장 해석
```text
"Conversion fell from X to Y, hypothesis Z."
```

마지막 한 문장이 중요합니다. 숫자가 어떻게 달라졌고, 어떤 가설이 가능한지 말할 수 있어야 분석 답변으로 완성됩니다.

## 이 예시에서 먼저 봐야 할 점

- CTE는 가독성을 높입니다.
- 지표 정의가 결론을 바꿉니다.
- 해석까지 가야 답변이 끝납니다.

면접관은 SQL 자체보다 사고 과정을 함께 봅니다. 어떤 가정을 두는지, 시간대를 고려하는지, NULL을 어떻게 처리하는지가 모두 드러나기 때문입니다.

## 자주 하는 실수 5가지

1. **SELECT *를 남발하는 실수**
2. **NULL 처리를 무시하는 실수**
3. **시간대를 무시하는 실수**
4. **지표 정의를 모호하게 두는 실수**
5. **해석 없이 끝내는 실수**

## 실무에서는 이렇게 나타납니다

분석 인터뷰는 보통 SQL 문제 하나와 케이스 질문 하나를 함께 묶습니다. SQL로 기본기를 보고, 이어서 그 결과를 제품이나 비즈니스 판단으로 연결할 수 있는지 확인하는 방식입니다.

## 시니어는 이렇게 생각합니다

- 지표를 먼저 정의합니다.
- CTE로 동료가 읽기 쉬운 쿼리를 만듭니다.
- 해석이 실제 가치라고 봅니다.
- NULL을 먼저 의심합니다.
- 시간대를 명시합니다.

## 정리 및 다음 단계

SQL과 분석 인터뷰는 쿼리 시험이면서 동시에 사고력 시험입니다. 질문을 분해하고, 적절한 JOIN과 집계를 고르고, 필요하면 CTE로 구조를 정리한 뒤, 마지막에 결과를 해석하는 흐름이 핵심입니다.

다음 글에서는 머신러닝 인터뷰에서 어떤 질문이 나오고 어떻게 답해야 하는지 봅니다.


## 주니어에서 시니어로 가는 성장 마일스톤 표

SQL과 분석 인터뷰를 준비하는 시점은 동시에 장기 성장 축을 점검하기 좋은 시기입니다. 인터뷰 합격만 목표로 하면 단기 최적화에 머물기 쉽기 때문에, 주니어에서 시니어로 이어지는 마일스톤을 함께 설정해 두는 편이 좋습니다. 아래 표는 데이터 직무 공통으로 적용할 수 있는 성장 단계 기준입니다.

| 단계 | 핵심 역량 | 대표 행동 | 산출물 품질 기준 | 다음 단계로 가는 신호 |
| --- | --- | --- | --- | --- |
| 주니어 초기 | 정확한 실행 | 요구사항을 따라 분석/구현 | 재현 가능한 결과 제출 | 반복 과제에서 실수율 감소 |
| 주니어 중기 | 문제 분해 | 모호한 요청을 질문으로 정리 | 지표 정의와 가정 명시 | 독립적으로 작은 과제 리드 |
| 미드레벨 | 우선순위 판단 | 여러 요청 중 임팩트가 큰 일 선택 | 해석과 제안까지 포함 | 분기 단위 과제 오너십 확보 |
| 시니어 초기 | 시스템 관점 | 데이터-제품-운영 연결 | 측정 계획이 포함된 제안 | 타팀 협업 과제 주도 |
| 시니어 이상 | 조직 증폭 | 동료 성장/표준화 주도 | 재사용 가능한 프레임 구축 | 팀 전체 속도와 품질 향상 |

많은 사람이 시니어를 "어려운 기술을 잘하는 사람"으로만 이해하지만, 실제 평가는 책임 범위와 영향 방식에서 갈립니다. 예를 들어 주니어는 정확한 실행이 강점이라면, 시니어는 문제 선택과 조직 확장에 더 큰 가치를 둡니다. 그래서 인터뷰 준비 중에도 "나는 지금 어느 단계의 행동을 보여 주고 있는가"를 점검해야 합니다.

성장 마일스톤을 운영 가능한 습관으로 바꾸는 방법도 중요합니다. 아래 체크표를 월 단위로 반복하면 진척을 객관적으로 볼 수 있습니다.

| 월간 점검 항목 | 확인 질문 |
| --- | --- |
| 실행 품질 | 같은 실수를 줄이고 있는가 |
| 문제 정의 | 요청을 받은 뒤 가정을 명확히 했는가 |
| 영향 측정 | 결과를 수치로 설명할 수 있는가 |
| 협업 | 이해관계자 기대를 사전에 조율했는가 |
| 지식 공유 | 배운 내용을 문서나 세션으로 남겼는가 |

또한 단계별로 "버려야 할 습관"을 명확히 하는 것이 빠른 성장에 도움이 됩니다. 주니어 단계에서는 완벽주의로 시작을 늦추는 습관을 버려야 하고, 미드 단계에서는 요청을 그대로 처리하는 습관을 버려야 하며, 시니어 단계에서는 혼자 해결하려는 습관을 버려야 합니다. 성장의 본질은 새로운 기술 추가만이 아니라, 현재 단계에 맞지 않는 행동을 줄이는 데도 있습니다.

면접 답변에서도 마일스톤 언어를 활용할 수 있습니다. "저는 SQL을 할 수 있습니다" 대신 "요청을 받으면 먼저 지표 정의와 분모 기준을 맞추고, 결과를 한 문장 권고안으로 정리하는 습관을 갖고 있습니다"처럼 행동 기반으로 말하면 신뢰가 높아집니다. 특히 시니어 잠재력을 보려는 면접에서는 기술 나열보다 행동의 일관성이 더 큰 신호가 됩니다.

장기적으로는 6개월 단위 개인 성장 문서를 운영해 보십시오. 내가 맡은 문제 유형, 만든 산출물, 실패에서 배운 교훈, 다음 반기에 강화할 역량을 기록하면 본인의 성장 곡선이 선명해집니다. 인터뷰 준비와 커리어 성장은 분리된 일이 아니라 같은 루프입니다. 오늘의 분석 문제를 푸는 방식이 내일의 시니어 행동을 미리 만들어 줍니다.


## 실행용 점검 프레임: 05-sql-and-analytics-interview.md 편

이 섹션은 앞에서 다룬 내용을 실제 행동으로 옮기기 위한 점검 프레임입니다. 많은 독자가 글을 읽을 때는 이해했다고 느끼지만, 일주일 뒤 실행 단계에서 다시 막히는 이유는 "다음 행동"이 문장으로 고정되어 있지 않기 때문입니다. 그래서 아래 프레임은 학습, 실습, 회고를 연결하는 최소 단위로 구성했습니다.

첫째, 이번 주 목표를 한 문장으로 적습니다. 목표는 도구가 아니라 문제 기준으로 써야 합니다. 예를 들어 "SQL 윈도우 함수 공부"보다 "사용자별 누적 결제를 계산해 이탈 신호를 찾는다"처럼 적는 편이 실행력이 높습니다. 같은 시간이라도 문제 중심 목표가 결과 해석까지 자연스럽게 이어집니다.

둘째, 입력과 출력을 명확히 구분합니다. 입력은 데이터, 로그, 문서, 인터뷰 메모처럼 내가 활용할 재료입니다. 출력은 쿼리, 분석 노트, 비교표, 의사결정 문서처럼 다른 사람이 검토 가능한 결과물입니다. 입력을 많이 모으는 것과 출력을 만드는 것은 다르며, 성장 속도는 출력 빈도에 더 강하게 좌우됩니다.

셋째, 판단 기준을 미리 적습니다. 많은 작업이 끝에서 흔들리는 이유는 성공 기준을 사전에 합의하지 않았기 때문입니다. "정확한가"라는 문장 대신 "분모 정의가 명확한가", "누락 세그먼트가 없는가", "후속 액션이 있는가"처럼 점검 문항으로 바꾸면 품질이 안정됩니다.

넷째, 협업 단위를 작게 쪼개 공유합니다. 완성본만 보여 주려는 습관은 피드백 타이밍을 늦춥니다. 중간 결과를 1페이지 요약으로 공유하고, 가정과 리스크를 먼저 드러내면 이후 수정 비용이 크게 줄어듭니다. 데이터 직무는 혼자 완벽하게 만든 뒤 공개하는 방식보다, 가정 기반으로 빠르게 정렬하는 방식이 장기적으로 더 강합니다.

다섯째, 회고는 감상이 아니라 의사결정 기록으로 남깁니다. "어려웠다"로 끝내지 말고 무엇이 병목이었는지, 다음에는 어떤 순서로 접근할지, 어떤 지표를 추가로 볼지를 적어야 합니다. 같은 유형의 문제를 다시 만났을 때 과거 기록이 바로 템플릿이 되면 성장 속도는 가속됩니다.

| 입력 | 어떤 데이터/자료를 사용할 것인가 | 데이터 출처 목록 |
| 방법 | 어떤 접근 순서로 진행할 것인가 | 작업 단계 체크리스트 |
| 검증 | 무엇을 성공 기준으로 볼 것인가 | 품질 점검 표 |
| 결과 | 무엇이 바뀌었는가 | 한 문장 권고안 |
| 회고 | 다음 주에 무엇을 바꿀 것인가 | 개선 액션 3개 |

이 템플릿의 핵심은 완벽함이 아니라 반복 가능성입니다. 한 번에 거대한 문서를 만들 필요는 없습니다. 중요한 것은 매주 같은 구조로 남겨서 비교 가능성을 확보하는 일입니다. 비교가 가능해지면 성장도 측정 가능해지고, 측정 가능해지면 커리어 결정의 품질이 올라갑니다.

마지막으로, 실행 프레임은 개인 메모로 끝내지 말고 가능한 범위에서 팀 공유 자산으로 전환하는 것이 좋습니다. 같은 질문이 반복되는 조직일수록 표준 템플릿의 가치가 큽니다. 템플릿 하나를 잘 만드는 사람은 단순히 일을 수행하는 사람이 아니라, 팀이 더 빠르고 정확하게 일하도록 환경을 설계하는 사람으로 인식됩니다.


## 월간 점검 질문 세트

아래 질문은 월말 회고 때 그대로 사용할 수 있는 점검 세트입니다. 첫 번째 질문은 "이번 달에 내가 해결한 문제는 무엇이며, 그 문제는 누구의 의사결정에 영향을 주었는가"입니다. 두 번째 질문은 "내 결과물의 재현 가능성을 다른 동료가 확인할 수 있는가"입니다. 세 번째 질문은 "결과를 한 문장 권고안으로 요약했는가"입니다. 네 번째 질문은 "가정이 틀렸을 때 대안 경로를 제시했는가"입니다. 다섯 번째 질문은 "다음 달에는 무엇을 하지 않을 것인가"입니다.

이 다섯 질문을 매달 반복하면 학습과 실행이 분리되지 않습니다. 특히 마지막 질문은 과부하를 줄이는 데 효과적입니다. 커리어 성장은 더 많은 일을 하는 것만으로 만들어지지 않습니다. 우선순위를 분명히 하고, 덜 중요한 일을 의도적으로 줄일 때 더 빠르게 만들어집니다.

| 질문 | Pass 기준 |
| --- | --- |
| 문제 정의가 명확한가 | 이해관계자가 같은 문장으로 설명 가능 |
| 결과가 재현 가능한가 | 실행 명령과 데이터 기준이 문서화됨 |
| 해석이 행동으로 연결되는가 | 후속 액션이 1개 이상 명시됨 |
| 리스크를 선제적으로 다뤘는가 | 가정과 한계를 함께 기술함 |
| 다음 달 우선순위가 보이는가 | 할 일/하지 않을 일 목록이 분리됨 |

## 실전 SQL 면접 문제 3선과 해설 포인트

면접 준비 효율을 높이려면 문제 수를 늘리기 전에 "반복 출제되는 사고 패턴"을 먼저 익혀야 합니다. 아래 세 문제는 거의 모든 SQL 면접에서 변형되어 나오는 핵심 유형입니다.

| 문제 유형 | 면접 질문 예시 | 해설 핵심 |
| --- | --- | --- |
| 재구매율 계산 | "첫 구매 후 30일 내 재구매율을 구해 보세요" | 분모 정의, 기간 경계, 중복 사용자 처리 |
| 퍼널 전환 | "방문-가입-결제 전환율을 구해 보세요" | 단계 정의, 누락 단계 처리, 집계 단위 일관성 |
| 누적 매출 추세 | "일자별 누적 매출과 7일 이동평균을 구해 보세요" | 윈도우 프레임 설정, 결측 일자 보정 |

답변 구조는 항상 같습니다. 문제를 재정의하고, 집계 단위를 명시하고, 예외 케이스를 먼저 언급한 뒤, 마지막에 비즈니스 해석을 한 문장으로 닫습니다. 이 흐름을 반복하면 쿼리 문법 실수보다 더 치명적인 해석 실수를 줄일 수 있습니다.


## 에스큐엘 인터뷰 질의응답 샘플: 질문 의도를 먼저 해석하기

SQL 인터뷰는 정답 쿼리만 맞히는 시험이 아닙니다. 질문 의도와 데이터 함정을 먼저 설명할 수 있어야 실무형 답변으로 평가됩니다.

### 질문 1
"최근 30일 기준으로 재구매율을 계산해 주세요."

- 먼저 확인할 점: 재구매 정의, 분모 기간, 취소 주문 처리 기준
- 답변 구조: 정의 합의 -> CTE로 중간 집계 -> 최종 비율 계산 -> 예외 케이스 설명

### 질문 2
"국가별 매출 상위 3개 상품을 구해 주세요."

- 핵심 포인트: `ROW_NUMBER()` 또는 `DENSE_RANK()` 선택 이유 설명
- 실무 보강: 동률 처리 정책, 환불 반영 기준, 날짜 필터 범위

이런 방식으로 설명하면 쿼리 문법 실수 하나가 있더라도 문제 해결 역량을 충분히 보여줄 수 있습니다.

## 분석 인터뷰 답변 스크립트: 90초 구조

```text
1) 문제 재정의: 요청을 지표 언어로 바꾼다.
2) 접근 계획: 필요한 데이터와 검증 순서를 말한다.
3) 결과 해석: 숫자의 의미와 리스크를 분리해 설명한다.
4) 액션 제안: 다음 실험/개선안을 제시한다.
```

면접에서 가장 강한 답변은 빠른 계산보다 "해석과 행동 연결"이 분명한 답변입니다. 90초 구조를 반복 연습하면 전달력이 안정됩니다.


## 실전 부록: 지원 문서와 면접 답변을 한 세트로 준비하는 방법

글을 읽고 이해한 내용을 실제 커리어 행동으로 바꾸려면, 이력서 문장·포트폴리오 증거·면접 답변이 서로 같은 문제를 가리키도록 정렬해야 합니다. 아래 구성은 데이터 직무 입문자와 주니어가 가장 빠르게 효과를 보는 준비 방식입니다.

### 이력서 불릿 템플릿

```text
- [문제] 어떤 비즈니스 또는 운영 문제를 맡았는가
- [행동] 어떤 데이터 접근(지표/실험/파이프라인/모델)을 적용했는가
- [결과] 무엇이 얼마나 개선되었는가
- [의미] 그 결과가 팀 의사결정에 어떤 영향을 주었는가
```

예시 문장: "결제 퍼널 이탈 구간을 코호트 기준으로 분해해 주요 병목을 식별했고, 온보딩 실험 우선순위를 재정렬해 4주 내 전환율을 개선했습니다." 핵심은 기술 이름보다 책임과 결과를 문장으로 남기는 일입니다.

### 포트폴리오 프로젝트 선정 체크

| 점검 항목 | Pass 기준 |
| --- | --- |
| 문제 명확성 | 한 문장으로 문제와 영향 범위를 설명 가능 |
| 데이터 현실성 | 공개 데이터라도 결측/편향/한계를 문서화 |
| 재현 가능성 | 실행 절차와 검증 기준을 README에 명시 |
| 의사결정 연결 | 결과가 다음 액션으로 이어짐 |

이 표를 기준으로 프로젝트를 고르면 "보여주기용 분석"을 줄이고 "실무형 문제 해결"을 남길 수 있습니다.

### 인터뷰 Q&A 3문장 프레임

1. 문제를 어떻게 정의했는지 말합니다.
2. 어떤 접근을 선택했고 왜 그렇게 판단했는지 말합니다.
3. 결과와 한계를 함께 설명하고 다음 액션을 제시합니다.

이 세 문장으로 답변하면 면접관이 가장 궁금해하는 판단 과정이 빠르게 드러납니다. 특히 한계를 먼저 말하는 태도는 신뢰를 높입니다.

### 보상 협상 기본 스크립트

```text
- 제안 감사합니다. 기대 역할과 성공 기준을 먼저 정렬하고 싶습니다.
- 제가 제공할 수 있는 기여 범위와 최근 증거를 공유드리겠습니다.
- 시장 밴드와 총보상 구조를 기준으로 조정 가능한 항목을 논의하고 싶습니다.
```

협상은 대립이 아니라 역할-보상 정렬입니다. 역할 범위를 수치와 산출물로 제시하면 대화가 감정에서 사실로 이동합니다.

| 영역 | 현재 수준(1~5) | 목표 수준(1~5) | 다음 2주 액션 |
| --- | --- | --- | --- |

| 데이터 처리 |  |  |  |
| 검증 설계 |  |  |  |
| 전달력 |  |  |  |

월 1회 이 표를 채우면 성장 체감이 감정이 아니라 기록으로 남습니다. 커리어 의사결정의 품질은 결국 기록의 품질에서 올라갑니다.
