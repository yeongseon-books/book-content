# Data Science Career 101 (8/10): 첫 직장 적응

이 글은 Data Science Career 101 시리즈의 8번째 글입니다.


첫 데이터 직무에 합류하면 대부분은 기술보다 속도에서 압박을 느낍니다. 빨리 성과를 보여 줘야 할 것 같고, 아는 것이 부족하다는 사실이 더 크게 보이기 때문입니다. 그래서 데이터 지도도 없이 대시보드 수정이나 티켓 처리부터 반복하다가, 정작 중요한 맥락은 놓치는 경우가 많습니다.

하지만 첫 90일의 핵심은 존재감을 과시하는 데 있지 않습니다. 어떤 데이터가 어디서 오고, 누가 무엇을 중요하게 보고, 작은 신뢰를 어디서 만들 수 있는지 읽는 데 있습니다. 초반에 이 지도를 그려 두면 이후 제안의 방향도 훨씬 안정적이 됩니다.

이 글은 Data Science Career 101 시리즈의 여덟 번째 글입니다.

> 첫 직장 적응에서 가장 중요한 것은 무엇을 할 수 있는가가 아니라 무엇을 물어봐야 하고 어떻게 답해야 하는가입니다.

## 먼저 던지는 질문

- 첫 90일을 어떤 단계로 나눠 봐야 하는지 정리합니다?
- 초반에 무엇을 듣고 기록해야 하는지 설명합니다?
- 데이터 지도와 이해관계자를 왜 빨리 파악해야 하는지 짚습니다?

## 이 글에서 배우는 내용

- 30/60/90 온보딩 구간
- 데이터 지도 그리기
- 이해관계자 파악
- 빠른 성과 만들기
- 문서화 습관

## 왜 중요한가

첫 90일은 평판의 씨앗이 심어지는 시기입니다. 이 시기에 무엇을 듣고, 무엇을 남기고, 어디에서 작은 성과를 만들었는지가 이후의 속도를 크게 바꿉니다.

특히 데이터 직무는 문맥 의존성이 높아서, 도메인과 데이터 출처를 모른 채 기술만 앞세우면 방향이 쉽게 어긋납니다. 반대로 이해관계자와 데이터 지도를 먼저 정리하면 작은 개선도 훨씬 더 적절한 문제를 겨냥하게 됩니다.

## 한눈에 보는 개념

온보딩의 우선순위는 이 순서로 이해하면 좋습니다. 지도를 그리고, 사람을 파악하고, 작은 승리를 만들고, 그 과정을 문서화해야 자산이 남습니다.

## 핵심 용어

- **data map**: 데이터 소스와 쓰임새를 정리한 지도입니다.
- **stakeholder**: 내 작업의 영향을 받거나 방향을 정하는 사람입니다.
- **quick win**: 초기에 만들 수 있는 작지만 분명한 성과입니다.
- **postmortem**: 문제를 되짚어 배우는 회고 문서입니다.
- **decision log**: 결정과 근거를 남기는 기록입니다.

## 전후 비교

**Before**: "들어가자마자 티켓만 빨리 처리하면 된다고 생각했다."

**After**: "지도와 관계와 지표와 문서를 남기는 일이 적응의 핵심이라는 점을 이해했다."

## 실습: 90일 계획

### 단계 1 — 0-30일차: 경청하기
```text
- ten 1:1s
- catalog the data
- learn five key dashboards
```

처음 한 달은 가능한 한 많이 듣는 편이 좋습니다. 팀이 중요하게 보는 문제와 데이터 소스를 파악해야 이후 제안의 품질이 달라집니다.

### 단계 2 — 31-60일차: 빠른 성과
```text
- one analysis or model PoC
- one SQL efficiency win
```

작은 승리는 신뢰의 출발점입니다. 거창한 변화보다 팀이 바로 체감할 수 있는 개선이 훨씬 효과적입니다.

### 단계 3 — 61-90일차: 제안하기
```text
- one quarterly agenda item
- include measurement plan
```

이 단계부터는 제안이 필요합니다. 다만 아이디어만 말하지 말고 무엇을 어떻게 측정할지까지 붙여야 설득력이 생깁니다.

### 단계 4 — 데이터 맵
```markdown
| Source | Owner | Freshness | Use |
```

데이터 지도는 개인 메모가 아니라 팀 자산입니다. 누가 소유자인지, 얼마나 자주 갱신되는지, 어디에 쓰이는지 정리해 두면 반복 질문이 크게 줄어듭니다.

### 단계 5 — 결정 로그
```text
- decision / alternatives / rationale / impact
```

결정 기록은 초반일수록 중요합니다. 맥락을 복원하기 쉽고, 협업 상대와의 인식 차이도 줄여 줍니다.

## 이 예시에서 먼저 봐야 할 점

- 잘 듣는 일이 영향력의 출발점입니다.
- 작은 승리가 신뢰를 만듭니다.
- 문서화는 시간이 갈수록 더 큰 자산이 됩니다.

입사 초기에 실력을 과시하는 것보다 팀이 무엇을 중요하게 여기는지 파악하는 편이 더 낫습니다. 데이터 직무는 특히 문맥 의존성이 높아 방향이 틀리면 노력도 쉽게 헛수고가 됩니다.

## 자주 하는 실수 5가지

1. **첫날부터 큰 제안을 밀어붙이는 실수**
2. **이해관계자를 무시하는 실수**
3. **데이터 출처를 모르고 움직이는 실수**
4. **대시보드 수정만 반복하는 실수**
5. **문서를 남기지 않는 실수**

## 실무에서는 이렇게 나타납니다

많은 매니저는 30일, 60일, 90일 체크인으로 적응 속도를 봅니다. 이때 단순히 바빴는지가 아니라 누구를 만났고, 어떤 문서를 남겼고, 어떤 작은 성과를 냈는지가 더 중요하게 평가됩니다.

## 시니어는 이렇게 생각합니다

- 잘 듣는 일도 전략입니다.
- 지도는 개인 메모가 아니라 팀 자산입니다.
- 작은 승리가 이후 제안의 신뢰를 만듭니다.
- 결정 기록은 팀 문화를 만듭니다.
- 겸손이 성장 속도를 높입니다.

## 정리 및 다음 단계

첫 데이터 직무의 90일은 큰 프로젝트를 단번에 성공시키는 시기가 아니라, 팀과 데이터와 사람의 지형을 읽는 시기입니다. 1:1, 데이터 지도, 작은 승리, 문서화라는 네 축을 잘 잡아야 이후 제안과 실행도 안정적으로 붙습니다.

다음 글에서는 기술 역량과는 또 다른 축인 도메인 전문성을 어떻게 쌓을지 봅니다.


## 이직 타이밍 판단 기준과 연봉 밴드 해석

첫 직장 적응 이후 많은 데이터 실무자가 고민하는 주제는 이직 타이밍입니다. 단순히 연봉 숫자만 보고 움직이면 커리어 누적 방향이 흔들릴 수 있고, 반대로 너무 오래 머물면 학습 기회가 정체될 수 있습니다. 그래서 이직 판단은 감정이 아니라 기준표로 점검하는 편이 좋습니다.

| 판단 항목 | 점검 질문 | 이직 신호 | 잔류 신호 |
| --- | --- | --- | --- |
| 학습 곡선 | 지난 6개월간 새로운 문제를 배웠는가 | 학습 정체가 지속됨 | 난도 높은 과제가 꾸준히 제공됨 |
| 영향 범위 | 내 제안이 실제 의사결정으로 이어지는가 | 반복 실행만 하고 영향 확대 없음 | 영향 범위가 분기마다 확대됨 |
| 보상 구조 | 역할 대비 보상이 시장과 크게 차이 나는가 | 시장 대비 과소 보상 | 성장과 보상이 함께 개선됨 |
| 팀 건강도 | 리더십/협업이 지속 가능한가 | 반복적 소모와 불신 | 갈등은 있어도 해결 구조가 있음 |
| 역할 적합도 | 원하는 축(분석/모델/엔지니어링)과 맞는가 | 장기 방향과 업무가 불일치 | 장기 목표와 현재 역할이 정렬됨 |

연봉 밴드는 절대값보다 "역할과 기대치"를 함께 봐야 합니다. 높은 연봉이더라도 과도한 책임과 불명확한 성공 기준이 붙으면 성장 효율이 떨어질 수 있습니다. 반대로 현재 연봉이 다소 낮아도 학습 곡선과 영향 범위가 빠르게 커지는 환경이라면 단기 잔류가 더 유리할 수 있습니다.

실무적으로는 아래 3단계로 판단하면 안정적입니다. 첫째, 현재 역할에서 6개월 내 확장 가능한 책임이 있는지 확인합니다. 둘째, 외부 시장에서 같은 경력 대비 기대 역할을 조사합니다. 셋째, 금전 보상 외에 도메인/멘토링/프로세스 성숙도를 함께 비교합니다.

| 연차 구간 | 시장에서 자주 묻는 기준 | 협상 포인트 예시 |
| --- | --- | --- |
| 0-2년 | 기본기, 실행 안정성 | 온보딩 지원, 멘토링, 명확한 목표 |
| 2-4년 | 독립 과제 리드, 해석 역량 | 직무 범위, 승격 기준, 평가 지표 |
| 4-7년 | 조직 영향, 협업 리드 | 책임 범위, 팀 구성, 보상 체계 |

이직 타이밍을 잘 잡는 사람은 "불만"이 아니라 "증거"로 판단합니다. 예를 들어 최근 3개 프로젝트에서 제안이 실행되지 않은 이유, 학습 정체의 원인, 보상 격차를 문서로 정리하면 결정이 훨씬 명확해집니다. 감정 기반 결정보다 후회가 적습니다.

면접에서도 이 기준은 도움이 됩니다. 이직 사유를 물을 때 "연봉이 낮아서"만 말하면 약합니다. 대신 "분석 결과가 제품 의사결정까지 연결되는 환경에서 영향 범위를 확장하고 싶습니다"처럼 성장 기준으로 설명하면 훨씬 설득력이 높습니다. 결국 좋은 이직은 더 높은 숫자만이 아니라, 더 좋은 문제를 풀 수 있는 위치를 선택하는 일입니다.


## 실행용 점검 프레임: 08-first-job.md 편

이 섹션은 앞에서 다룬 내용을 실제 행동으로 옮기기 위한 점검 프레임입니다. 많은 독자가 글을 읽을 때는 이해했다고 느끼지만, 일주일 뒤 실행 단계에서 다시 막히는 이유는 "다음 행동"이 문장으로 고정되어 있지 않기 때문입니다. 그래서 아래 프레임은 학습, 실습, 회고를 연결하는 최소 단위로 구성했습니다.

첫째, 이번 주 목표를 한 문장으로 적습니다. 목표는 도구가 아니라 문제 기준으로 써야 합니다. 예를 들어 "SQL 윈도우 함수 공부"보다 "사용자별 누적 결제를 계산해 이탈 신호를 찾는다"처럼 적는 편이 실행력이 높습니다. 같은 시간이라도 문제 중심 목표가 결과 해석까지 자연스럽게 이어집니다.

둘째, 입력과 출력을 명확히 구분합니다. 입력은 데이터, 로그, 문서, 인터뷰 메모처럼 내가 활용할 재료입니다. 출력은 쿼리, 분석 노트, 비교표, 의사결정 문서처럼 다른 사람이 검토 가능한 결과물입니다. 입력을 많이 모으는 것과 출력을 만드는 것은 다르며, 성장 속도는 출력 빈도에 더 강하게 좌우됩니다.

셋째, 판단 기준을 미리 적습니다. 많은 작업이 끝에서 흔들리는 이유는 성공 기준을 사전에 합의하지 않았기 때문입니다. "정확한가"라는 문장 대신 "분모 정의가 명확한가", "누락 세그먼트가 없는가", "후속 액션이 있는가"처럼 점검 문항으로 바꾸면 품질이 안정됩니다.

넷째, 협업 단위를 작게 쪼개 공유합니다. 완성본만 보여 주려는 습관은 피드백 타이밍을 늦춥니다. 중간 결과를 1페이지 요약으로 공유하고, 가정과 리스크를 먼저 드러내면 이후 수정 비용이 크게 줄어듭니다. 데이터 직무는 혼자 완벽하게 만든 뒤 공개하는 방식보다, 가정 기반으로 빠르게 정렬하는 방식이 장기적으로 더 강합니다.

다섯째, 회고는 감상이 아니라 의사결정 기록으로 남깁니다. "어려웠다"로 끝내지 말고 무엇이 병목이었는지, 다음에는 어떤 순서로 접근할지, 어떤 지표를 추가로 볼지를 적어야 합니다. 같은 유형의 문제를 다시 만났을 때 과거 기록이 바로 템플릿이 되면 성장 속도는 가속됩니다.

아래 표는 매주 반복 가능한 최소 운영 템플릿입니다.

| 항목 | 작성 질문 | 예시 출력 |
| --- | --- | --- |
| 목표 | 이번 주에 어떤 문제를 풀 것인가 | 문제 정의 3줄 |
| 입력 | 어떤 데이터/자료를 사용할 것인가 | 데이터 출처 목록 |
| 방법 | 어떤 접근 순서로 진행할 것인가 | 작업 단계 체크리스트 |
| 검증 | 무엇을 성공 기준으로 볼 것인가 | 품질 점검 표 |
| 결과 | 무엇이 바뀌었는가 | 한 문장 권고안 |
| 회고 | 다음 주에 무엇을 바꿀 것인가 | 개선 액션 3개 |

이 템플릿의 핵심은 완벽함이 아니라 반복 가능성입니다. 한 번에 거대한 문서를 만들 필요는 없습니다. 중요한 것은 매주 같은 구조로 남겨서 비교 가능성을 확보하는 일입니다. 비교가 가능해지면 성장도 측정 가능해지고, 측정 가능해지면 커리어 결정의 품질이 올라갑니다.

마지막으로, 실행 프레임은 개인 메모로 끝내지 말고 가능한 범위에서 팀 공유 자산으로 전환하는 것이 좋습니다. 같은 질문이 반복되는 조직일수록 표준 템플릿의 가치가 큽니다. 템플릿 하나를 잘 만드는 사람은 단순히 일을 수행하는 사람이 아니라, 팀이 더 빠르고 정확하게 일하도록 환경을 설계하는 사람으로 인식됩니다.


## 월간 점검 질문 세트

아래 질문은 월말 회고 때 그대로 사용할 수 있는 점검 세트입니다. 첫 번째 질문은 "이번 달에 내가 해결한 문제는 무엇이며, 그 문제는 누구의 의사결정에 영향을 주었는가"입니다. 두 번째 질문은 "내 결과물의 재현 가능성을 다른 동료가 확인할 수 있는가"입니다. 세 번째 질문은 "결과를 한 문장 권고안으로 요약했는가"입니다. 네 번째 질문은 "가정이 틀렸을 때 대안 경로를 제시했는가"입니다. 다섯 번째 질문은 "다음 달에는 무엇을 하지 않을 것인가"입니다.

이 다섯 질문을 매달 반복하면 학습과 실행이 분리되지 않습니다. 특히 마지막 질문은 과부하를 줄이는 데 효과적입니다. 커리어 성장은 더 많은 일을 하는 것만으로 만들어지지 않습니다. 우선순위를 분명히 하고, 덜 중요한 일을 의도적으로 줄일 때 더 빠르게 만들어집니다.

| 질문 | Pass 기준 |
| --- | --- |
| 문제 정의가 명확한가 | 이해관계자가 같은 문장으로 설명 가능 |
| 결과가 재현 가능한가 | 실행 명령과 데이터 기준이 문서화됨 |
| 해석이 행동으로 연결되는가 | 후속 액션이 1개 이상 명시됨 |
| 리스크를 선제적으로 다뤘는가 | 가정과 한계를 함께 기술함 |
| 다음 달 우선순위가 보이는가 | 할 일/하지 않을 일 목록이 분리됨 |


## 보강 메모: 실전 적용 예시

실전에서는 한 가지 프레임만으로 모든 상황을 풀 수 없습니다. 그래서 동일한 문제를 최소 두 가지 관점으로 다시 써 보는 연습이 필요합니다. 예를 들어 케이스 문제라면 지표 관점과 사용자 여정 관점을 각각 분리해 적어 보고, 온보딩 문제라면 개인 생산성과 팀 신뢰 형성 관점을 분리해 점검합니다. 도메인 학습 문제라면 용어 정확성과 의사결정 연결성을 분리해 체크합니다.

이 보강 연습의 목적은 정답을 늘리는 일이 아니라 시야를 넓히는 일입니다. 같은 데이터라도 질문이 달라지면 결론이 달라질 수 있기 때문입니다. 따라서 최종 결론을 내리기 전에 "내가 놓친 관점이 있는가"를 한 번 더 점검하는 습관이 중요합니다.

## 첫 90일 성과표: 주차별 실행 체크

온보딩 단계에서 가장 중요한 것은 바쁨이 아니라 진척의 가시성입니다. 따라서 첫 90일은 주차별로 작은 결과를 남기고, 그 결과가 어떤 신뢰를 만들었는지 기록하는 방식이 효과적입니다.

| 기간 | 필수 결과물 | 신뢰 신호 |
| --- | --- | --- |
| 1-4주 | 이해관계자 맵, 데이터 소스 목록 | "상황을 빨리 이해한다"는 평가 |
| 5-8주 | quick win 1건, 분석 메모 1건 | "작은 문제를 닫을 수 있다"는 평가 |
| 9-12주 | 분기 제안서 1건, 측정 계획 포함 | "앞으로를 설계할 수 있다"는 평가 |

이 표를 주간 1:1에서 공유하면 기대치 정렬이 쉬워집니다. 특히 매니저와 "이번 주 결과가 다음 단계에 어떤 영향을 주는지"를 함께 점검하면, 초반 불안감이 줄고 실행 우선순위가 더 선명해집니다. 첫 직장 적응은 속도 경쟁이 아니라 신뢰 축적의 설계라는 점을 잊지 않는 것이 중요합니다.


## 첫 직장 90일 계획: 생존이 아니라 신뢰 구축입니다

첫 직장에서는 빠르게 많이 하는 사람보다, 작은 결과를 반복적으로 검증해 신뢰를 쌓는 사람이 오래갑니다. 아래 30-60-90일 계획은 주니어 데이터 직무의 초기 적응에 맞춘 실행 프레임입니다.

| 기간 | 핵심 목표 | 필수 산출물 |
| --- | --- | --- |
| 1-30일 | 도메인과 데이터 맥락 이해 | 용어집, 지표 맵, 질문 리스트 |
| 31-60일 | 독립 분석 1~2건 수행 | 분석 노트, 의사결정 요약문 |
| 61-90일 | 반복 업무 개선 제안 | 자동화 초안, 운영 체크리스트 |

이 프레임의 핵심은 "첫 90일에 대형 프로젝트를 완성한다"가 아닙니다. 팀이 당신에게 반복적으로 일을 맡길 수 있는 신뢰를 만드는 일입니다.

## 연봉 협상 스크립트: 첫 오퍼에서 기준 세우기

```text
1) 감사: 제안과 기대 역할을 명확히 확인했습니다.
2) 근거: 유사 직무 밴드, 지역/시장 데이터, 제 역량 증거를 공유합니다.
3) 요청: 기본급 또는 사인온 중 어떤 항목을 조정할지 구체적으로 제안합니다.
4) 유연성: 총보상 구조를 함께 보며 현실적인 대안을 논의합니다.
```

협상은 공격이 아니라 정렬입니다. 감정 표현보다 역할 책임, 기대 성과, 시장 근거를 중심으로 말하면 대화가 훨씬 생산적으로 진행됩니다.


## 실전 부록: 지원 문서와 면접 답변을 한 세트로 준비하는 방법

글을 읽고 이해한 내용을 실제 커리어 행동으로 바꾸려면, 이력서 문장·포트폴리오 증거·면접 답변이 서로 같은 문제를 가리키도록 정렬해야 합니다. 아래 구성은 데이터 직무 입문자와 주니어가 가장 빠르게 효과를 보는 준비 방식입니다.

### 이력서 불릿 템플릿

```text
- [문제] 어떤 비즈니스 또는 운영 문제를 맡았는가
- [행동] 어떤 데이터 접근(지표/실험/파이프라인/모델)을 적용했는가
- [결과] 무엇이 얼마나 개선되었는가
- [의미] 그 결과가 팀 의사결정에 어떤 영향을 주었는가
```

예시 문장: "결제 퍼널 이탈 구간을 코호트 기준으로 분해해 주요 병목을 식별했고, 온보딩 실험 우선순위를 재정렬해 4주 내 전환율을 개선했습니다." 핵심은 기술 이름보다 책임과 결과를 문장으로 남기는 일입니다.

### 포트폴리오 프로젝트 선정 체크

| 점검 항목 | Pass 기준 |
| --- | --- |
| 문제 명확성 | 한 문장으로 문제와 영향 범위를 설명 가능 |
| 데이터 현실성 | 공개 데이터라도 결측/편향/한계를 문서화 |
| 재현 가능성 | 실행 절차와 검증 기준을 README에 명시 |
| 의사결정 연결 | 결과가 다음 액션으로 이어짐 |

이 표를 기준으로 프로젝트를 고르면 "보여주기용 분석"을 줄이고 "실무형 문제 해결"을 남길 수 있습니다.

### 인터뷰 Q&A 3문장 프레임

1. 문제를 어떻게 정의했는지 말합니다.
2. 어떤 접근을 선택했고 왜 그렇게 판단했는지 말합니다.
3. 결과와 한계를 함께 설명하고 다음 액션을 제시합니다.

이 세 문장으로 답변하면 면접관이 가장 궁금해하는 판단 과정이 빠르게 드러납니다. 특히 한계를 먼저 말하는 태도는 신뢰를 높입니다.

### 보상 협상 기본 스크립트

```text
- 제안 감사합니다. 기대 역할과 성공 기준을 먼저 정렬하고 싶습니다.
- 제가 제공할 수 있는 기여 범위와 최근 증거를 공유드리겠습니다.
- 시장 밴드와 총보상 구조를 기준으로 조정 가능한 항목을 논의하고 싶습니다.
```

협상은 대립이 아니라 역할-보상 정렬입니다. 역할 범위를 수치와 산출물로 제시하면 대화가 감정에서 사실로 이동합니다.

### 역량 점검 매트릭스

| 영역 | 현재 수준(1~5) | 목표 수준(1~5) | 다음 2주 액션 |
| --- | --- | --- | --- |
| 문제 정의 |  |  |  |
| 데이터 처리 |  |  |  |
| 검증 설계 |  |  |  |
| 전달력 |  |  |  |
| 협업 운영 |  |  |  |

월 1회 이 표를 채우면 성장 체감이 감정이 아니라 기록으로 남습니다. 커리어 의사결정의 품질은 결국 기록의 품질에서 올라갑니다.
