# 데이터 직무란 무엇인가

처음 데이터 커리어를 알아볼 때 가장 많이 생기는 오해는 이것입니다. 데이터 직무는 통계를 잘하는 사람 한 종류만 있는 세계라고 생각하는 것입니다. 그런데 실제 채용 공고와 현업 팀을 들여다보면 분석가, 데이터 사이언티스트, 데이터 엔지니어, 머신러닝 엔지니어, 애널리틱스 엔지니어처럼 역할이 꽤 세분화되어 있습니다.

겉으로는 모두 데이터를 다루는 것처럼 보여도, 각 역할이 풀어야 하는 문제와 기대받는 산출물은 다릅니다. 어떤 역할은 의사결정을 돕는 데 집중하고, 어떤 역할은 가설 검증과 모델링에 강해야 하며, 또 어떤 역할은 데이터가 안정적으로 흐르도록 기반을 만듭니다. 이 차이를 먼저 잡아 두어야 학습 순서도 정리되고, 지원할 직무도 훨씬 선명해집니다.

## 이 글에서 다룰 문제

- 데이터 직무라고 부를 때 실제로 어떤 역할들이 묶여 있을까요?
- 직함보다 책임과 산출물을 봐야 하는 이유는 무엇일까요?
- 입문자 관점에서 먼저 구분해야 할 핵심 역할은 무엇일까요?
- 데이터 직무를 준비할 때 기술만 외우면 왜 방향을 잃기 쉬울까요?
- 내가 어느 역할에 더 끌리는지 판단하려면 무엇을 기준으로 봐야 할까요?

> 데이터 직무는 하나의 이름이 아니라, 질문을 데이터로 풀고 제품이나 의사결정에 연결하는 여러 역할의 묶음입니다.

## 한눈에 보는 전체 흐름

```mermaid
flowchart LR
    Q[질문] --> D[데이터]
    D --> A[분석]
    A --> P[제품]
```

이 그림은 데이터 직무 전반을 아주 단순하게 압축한 것입니다. 누군가는 질문을 정의하고, 누군가는 데이터를 준비하고, 누군가는 분석이나 모델링을 수행하며, 누군가는 그 결과를 제품과 운영으로 연결합니다. 결국 데이터 직무는 숫자 자체보다 **질문에서 결과까지 이어지는 흐름을 어떻게 맡느냐**로 이해하는 편이 더 정확합니다.

## 핵심 용어

- **data analyst**: 데이터를 해석해 의사결정을 돕는 역할입니다.
- **data scientist**: 가설을 세우고 실험이나 모델로 검증하는 역할입니다.
- **data engineer**: 데이터 파이프라인과 저장 구조를 만드는 역할입니다.
- **ML engineer**: 모델을 학습·배포·운영하는 역할입니다.
- **analytics engineer**: 분석용 데이터 모델을 안정적으로 만드는 역할입니다.

## Before/After

**Before**: "데이터 일은 통계만 잘하면 되는 줄 알았다."

**After**: "역할마다 목적과 산출물이 다르다는 점을 기준으로 다섯 가지 직무를 구분할 수 있다."

## 실습: 역할별 한 줄 정의를 먼저 익혀 보기

직무 구분은 긴 설명보다 짧은 문장으로 먼저 잡는 편이 좋습니다. 아래 한 줄 요약은 각 역할의 중심을 빠르게 기억하는 데 도움이 됩니다.

### Step 1 — Analyst

```text
SQL과 대시보드로 의사결정을 돕는다.
```

### Step 2 — Scientist

```text
실험과 모델로 가설을 검증한다.
```

### Step 3 — Engineer

```text
파이프라인과 저장소를 구축해 데이터 기반을 만든다.
```

### Step 4 — ML Engineer

```text
모델을 학습시키고 배포하며 모니터링한다.
```

### Step 5 — Analytics Engineer

```text
dbt 같은 도구로 신뢰할 수 있는 분석 기반을 만든다.
```

## 이 예시에서 봐야 할 점

- 직함보다 **팀이 기대하는 책임**을 먼저 봐야 합니다.
- 같은 직함이라도 회사 규모와 조직 구조에 따라 실제 업무는 크게 달라질 수 있습니다.
- 현실에서는 역할 경계가 완전히 분리되지 않고 겹치는 경우가 많습니다.

입문자가 흔히 놓치는 부분도 여기입니다. 직무명을 외우는 데 시간을 많이 쓰지만, 실제로는 어떤 질문을 풀고 어떤 결과물을 내야 하는지가 더 중요합니다. 작은 회사에서는 한 사람이 분석과 간단한 데이터 엔지니어링까지 같이 맡기도 하고, 큰 회사에서는 같은 Analyst라는 이름 아래에서도 제품 분석과 재무 분석의 결이 완전히 다를 수 있습니다.

## 자주 하는 실수 5가지

1. **직함만 보고 역할을 판단하는 실수**
2. **모든 도구를 한 번에 배우려는 실수**
3. **문맥 없이 도구 이름만 외우는 실수**
4. **비즈니스 도메인을 가볍게 보는 실수**
5. **측정과 검증을 빼먹는 실수**

## 실무에서는 이렇게 나타납니다

큰 조직일수록 역할이 더 분리됩니다. Analyst는 지표와 대시보드에 집중하고, Scientist는 실험과 모델링을, ML Engineer는 배포와 운영을, Analytics Engineer는 신뢰할 수 있는 분석용 데이터 모델을 맡는 식입니다. 반대로 작은 조직에서는 한 사람이 두세 역할을 함께 수행하기도 합니다. 그래서 입문 단계에서는 “이 직함이 정확히 무엇이냐”보다 “이 역할이 무엇을 책임지느냐”를 읽는 눈을 먼저 갖추는 편이 낫습니다.

## 시니어는 이렇게 생각합니다

- 질문이 먼저이고, 데이터는 그 질문에 답하기 위한 수단입니다.
- 측정 가능한 결과가 있어야 일이 끝난 것입니다.
- 도메인을 이해할수록 같은 데이터에서도 더 좋은 판단이 나옵니다.
- 역할을 연결해서 보는 사람이 팀에서 더 큰 영향력을 냅니다.
- 도구보다 문제 구조를 이해하는 사람이 오래 갑니다.

## 체크리스트

- [ ] 다섯 가지 주요 데이터 역할을 구분할 수 있다.
- [ ] 내가 가장 관심 있는 역할 한 가지를 골랐다.
- [ ] 각 역할의 기본 도구를 한 번씩 적어 봤다.
- [ ] 앞으로 1주일 동안 어떤 직무를 더 탐색할지 정했다.

## 정리 및 다음 단계

데이터 직무를 이해하는 첫걸음은 “모두 데이터만 본다”는 뭉뚱그린 시선을 버리는 일입니다. 역할마다 질문, 산출물, 평가 기준, 협업 상대가 다르기 때문입니다. 이 구분이 잡혀야 학습 계획도 현실적으로 세울 수 있고, 포트폴리오도 더 설득력 있게 구성할 수 있습니다.

다음 글에서는 분석가, 사이언티스트, 엔지니어를 조금 더 직접적으로 비교하면서, 각 역할이 무엇을 중심에 두고 일하는지 살펴보겠습니다.
