# Data Science Career 101 (1/10): 데이터 직무란 무엇인가

처음 데이터 커리어를 알아볼 때 가장 자주 생기는 혼란은 직함은 많은데, 실제로 누가 무슨 문제를 푸는지 잘 보이지 않는다는 사실입니다. 채용 공고에는 analyst, scientist, engineer가 모두 등장하지만, 막상 입문자 입장에서는 무엇부터 구분해야 하는지 막막해지기 쉽습니다.

이 글은 Data Science Career 101 시리즈의 첫 번째 글입니다.

특히 같은 회사 안에서도 데이터 직무 정의가 제각각이라 더 헷갈립니다. 어떤 팀은 분석가가 실험 설계까지 맡고, 어떤 팀은 엔지니어가 지표 테이블까지 책임집니다. 그래서 처음에는 직함보다 질문, 산출물, 협업 상대를 기준으로 지형을 읽는 편이 훨씬 안전합니다.

![Data Science Career 101 1장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/data-science-career-101/01/01-01-concept-at-a-glance.ko.png)
*Data Science Career 101 1장 흐름 개요*
> 데이터 직무의 핵심은 직함이 아니라 책임 경계입니다. 각 역할이 어떤 문제를 맡고 어떤 산출물을 만드는지가 진입 기준이 되어야 합니다.

## 먼저 던지는 질문

- 데이터 직무라고 할 때 실제로 어떤 역할들이 함께 묶이는지 정리합니다?
- 직함보다 책임과 산출물로 역할을 구분해야 하는 이유를 설명합니다?
- 입문자가 먼저 구분해야 할 다섯 가지 핵심 역할을 살펴봅니다?

## 이 글에서 배우는 내용

- 데이터 커리어의 전체 지형
- 다섯 가지 주요 역할
- 역할을 가르는 핵심 역량
- 대표적인 진입 경로
- 입문자가 자주 하는 오해

## 왜 중요한가

같은 데이터라도 역할이 달라지면 기대되는 질문, 산출물, 협업 방식이 모두 달라집니다. 이 차이를 초반에 구분하지 못하면 학습 계획도 흐려지고, 지원서와 포트폴리오도 애매해집니다.

예를 들어 SQL을 더 파야 하는지, 모델링을 먼저 연습해야 하는지, 대시보드와 지표 설계를 강조해야 하는지는 지원하려는 역할에 따라 달라집니다. 입문 단계에서 이 기준을 세워 두면 학습 시간도 아끼고, 포트폴리오 방향도 더 또렷하게 잡을 수 있습니다.

## 한눈에 보는 개념

데이터 직무를 가장 단순하게 보면 질문에서 출발해 데이터를 거쳐 분석과 제품 변화로 이어지는 흐름입니다. 차이는 이 흐름의 어느 구간을 책임지느냐에 있습니다.

## 핵심 용어

- **data analyst**: 데이터를 해석해 의사결정을 돕는 역할입니다.
- **data scientist**: 가설을 세우고 실험과 모델로 검증하는 역할입니다.
- **data engineer**: 데이터 파이프라인과 저장 구조를 만드는 역할입니다.
- **ML engineer**: 모델을 학습·배포·운영하는 역할입니다.
- **analytics engineer**: 신뢰할 수 있는 분석용 데이터 모델을 만드는 역할입니다.

## 전후 비교

**Before**: "데이터 일은 통계만 잘하면 되는 줄 알았다."

**After**: "다섯 가지 역할을 목적과 산출물 기준으로 구분할 수 있다."

## 실습: 역할별 한 줄 정의

### 단계 1 — Analyst

```text
Drives decisions with SQL and dashboards.
```

분석가는 데이터를 해석해 팀의 판단 속도를 높이는 역할입니다. SQL, 지표, 대시보드가 중심 도구가 되는 이유도 여기에 있습니다.

### 단계 2 — Scientist

```text
Tests hypotheses with experiments and models.
```

사이언티스트는 질문을 가설로 바꾸고, 실험과 모델로 그 가설을 검증합니다. 그래서 설명력과 검증력이 함께 요구됩니다.

### 단계 3 — Engineer

```text
Builds pipelines and storage as data infrastructure.
```

엔지니어는 데이터가 안정적으로 흐르도록 기반을 만듭니다. 파이프라인과 스키마 품질이 무너지면 나머지 역할도 함께 흔들립니다.

### 단계 4 — ML 엔지니어
```text
Trains, deploys, and monitors models.
```

ML 엔지니어는 모델을 연구 대상이 아니라 운영 대상로 다룹니다. 학습, 배포, 모니터링을 하나의 시스템으로 보는 감각이 중요합니다.

### 단계 5 — 분석 엔지니어
```text
Builds reliable analytics models with dbt and friends.
```

애널리틱스 엔지니어는 분석가와 엔지니어 사이에서 재현 가능하고 일관된 분석 기반을 만듭니다. 지표 신뢰도를 높이는 역할이라고 보면 이해가 빠릅니다.

## 이 예시에서 먼저 봐야 할 점

- 직함보다 책임을 봐야 합니다.
- 같은 이름의 직무라도 회사마다 정의가 다를 수 있습니다.
- 실무에서는 역할 경계가 생각보다 자주 겹칩니다.

입문자가 가장 많이 놓치는 부분은 도구가 아니라 목적입니다. SQL을 쓴다고 모두 같은 역할이 아니고, Python을 쓴다고 모두 모델링 중심 직무도 아닙니다. 어떤 질문을 맡고 어떤 결과를 남기는지가 훨씬 더 본질적입니다.

## 자주 하는 실수 5가지

1. **직함만 보고 역할을 판단하는 실수**
2. **모든 도구를 한 번에 배우려는 실수**
3. **문맥 없이 도구 이름만 외우는 실수**
4. **비즈니스 도메인을 가볍게 보는 실수**
5. **측정과 검증을 건너뛰는 실수**

## 실무에서는 이렇게 나타납니다

큰 조직일수록 Analyst, Scientist, ML Engineer, Analytics Engineer가 더 선명하게 분리됩니다. 반대로 작은 조직에서는 한 사람이 두세 역할을 겸하기도 합니다. 그래서 입문 단계에서는 직함보다 실제 책임을 읽는 눈을 먼저 갖추는 편이 안전합니다.

## 시니어는 이렇게 생각합니다

- 질문이 먼저이고 데이터는 수단입니다.
- 측정 가능한 결과가 있어야 일이 끝납니다.
- 도메인을 이해할수록 같은 데이터에서도 더 좋은 판단이 나옵니다.
- 역할을 연결해서 보는 사람이 더 큰 영향력을 냅니다.
- 문제 구조를 이해하는 사람이 오래 갑니다.

## 체크리스트

- [ ] 다섯 가지 주요 역할을 구분할 수 있다.
- [ ] 내가 더 끌리는 역할 한 가지를 골랐다.
- [ ] 각 역할의 기본 도구를 한 번씩 적어 봤다.
- [ ] 앞으로 1주일 동안 탐색할 방향을 정했다.

## 정리 및 다음 단계

데이터 직무를 이해하는 첫걸음은 “모두 데이터를 본다”는 뭉뚱그린 시선을 버리는 일입니다. 역할마다 풀어야 하는 문제, 만드는 산출물, 협업 상대, 평가 기준이 다르기 때문입니다.

다음 글에서는 분석가, 사이언티스트, 엔지니어를 더 직접적으로 비교하면서 세 역할의 차이를 구조적으로 정리하겠습니다.

## 데이터 직무 유형 비교표: DA/DE/DS/MLE를 한 번에 정리

입문 단계에서 가장 많이 도움이 되는 학습은 개별 도구를 더 외우는 일이 아니라, 역할 간 차이를 같은 축으로 비교해 보는 일입니다. 같은 SQL을 써도 질문이 다르고, 같은 Python을 써도 산출물과 검증 기준이 다르기 때문입니다. 아래 표는 데이터 분석가(DA), 데이터 엔지니어(DE), 데이터 사이언티스트(DS), 머신러닝 엔지니어(MLE)를 같은 프레임으로 비교해 직무 선택 기준을 명확히 잡도록 돕는 실무형 정리입니다.

| 구분 | DA (Data Analyst) | DE (Data Engineer) | DS (Data Scientist) | MLE (ML Engineer) |
| --- | --- | --- | --- | --- |
| 핵심 질문 | 무슨 일이 일어났는가, 왜 일어났는가 | 데이터가 안정적으로 흐르는가 | 어떤 가설이 맞는가, 무엇을 예측할 수 있는가 | 모델이 운영에서 안정적으로 동작하는가 |
| 대표 산출물 | 대시보드, 분석 리포트, KPI 정의서 | 데이터 파이프라인, 테이블 스키마, 품질 모니터링 | 실험 결과, 모델 프로토타입, 피처 분석 | 추론 API, 배포 파이프라인, 모니터링 대시보드 |
| 주 협업 상대 | PM, 마케팅, 사업팀 | 백엔드, 플랫폼, 분석팀 | PM, 리서치, 분석가 | 백엔드, 플랫폼, 데이터 사이언티스트 |
| 성공 지표 | 의사결정 반영률, 분석 재사용률 | 파이프라인 안정성, 지연 시간, 데이터 품질 | 실험 신뢰도, 모델 성능, 인사이트 채택률 | 서빙 안정성, 지연 시간, 성능 유지율 |
| 실패 패턴 | 지표 정의 모호, 해석 누락 | 스키마 관리 부실, 운영 자동화 부족 | 과적합, 누수, 비즈니스 맥락 부족 | 학습-서빙 불일치, 드리프트 대응 부재 |
| 초반 학습 우선순위 | SQL, 지표 설계, 스토리텔링 | SQL, 데이터 모델링, 오케스트레이션 | 통계, 실험 설계, 모델 평가 | 서빙 구조, 배포, 모니터링 |

이 비교표를 읽을 때 중요한 포인트는 "어느 직무가 더 상위인가"가 아니라 "내가 어떤 문제를 오래 풀고 싶은가"입니다. 예를 들어 숫자의 의미를 해석하고 의사결정 회의를 이끄는 데 흥미가 크다면 DA 축이 맞습니다. 반대로 데이터가 깨지지 않게 흐르게 만드는 구조 설계에 몰입이 잘 된다면 DE 축이 더 자연스럽습니다. DS는 가설과 검증을 중심으로 사고하고, MLE는 모델을 운영 시스템으로 관리한다는 점에서 관점이 분명히 다릅니다.

또 하나 자주 놓치는 지점은 경력 초반에는 역할이 섞여 나타난다는 사실입니다. 작은 조직에서는 DA가 실험 분석을 겸하거나, DS가 간단한 배포를 맡는 경우가 흔합니다. 이때 중요한 것은 "내가 지금 하고 있는 일이 어느 축의 역량을 누적하고 있는가"를 추적하는 것입니다. 직함은 하나여도 실제 업무 비중이 DA 60, DS 40처럼 혼합될 수 있습니다. 혼합 자체는 문제가 아니지만, 본인이 원하는 장기 방향과 누적 역량이 엇갈리면 성장 체감이 떨어집니다.

실행 가능한 방법도 정리해 두겠습니다. 첫째, 최근 4주 동안 한 작업을 모두 적고 표의 네 직무 축에 분류해 보십시오. 둘째, 시간이 가장 많이 들어간 축과 에너지가 가장 살아나는 축을 분리해서 보십시오. 셋째, 다음 8주 학습 계획은 "흥미가 큰 축"을 기준으로 재배열하십시오. 예를 들어 MLE 지향이라면 모델 실험보다 서빙 지연 시간 측정과 배포 자동화 연습 비중을 높이는 편이 효과적입니다.

마지막으로, 면접 준비 관점에서도 이 표는 강력한 기준이 됩니다. "제가 SQL을 쓸 수 있습니다"보다 "DA 관점에서는 지표 정의와 해석, DE 관점에서는 데이터 정합성까지 같이 봅니다"처럼 역할 언어로 설명할 때 훨씬 설득력이 높아집니다. 즉, 도구를 말하는 사람이 아니라 책임을 말하는 사람으로 보이게 됩니다. 데이터 커리어의 첫 단추는 결국 이 책임 언어를 빨리 익히는 일입니다.

## 실행용 점검 프레임: 01-what-is-data-career.md 편

이 섹션은 앞에서 다룬 내용을 실제 행동으로 옮기기 위한 점검 프레임입니다. 많은 독자가 글을 읽을 때는 이해했다고 느끼지만, 일주일 뒤 실행 단계에서 다시 막히는 이유는 "다음 행동"이 문장으로 고정되어 있지 않기 때문입니다. 그래서 아래 프레임은 학습, 실습, 회고를 연결하는 최소 단위로 구성했습니다.

첫째, 이번 주 목표를 한 문장으로 적습니다. 목표는 도구가 아니라 문제 기준으로 써야 합니다. 예를 들어 "SQL 윈도우 함수 공부"보다 "사용자별 누적 결제를 계산해 이탈 신호를 찾는다"처럼 적는 편이 실행력이 높습니다. 같은 시간이라도 문제 중심 목표가 결과 해석까지 자연스럽게 이어집니다.

둘째, 입력과 출력을 명확히 구분합니다. 입력은 데이터, 로그, 문서, 인터뷰 메모처럼 내가 활용할 재료입니다. 출력은 쿼리, 분석 노트, 비교표, 의사결정 문서처럼 다른 사람이 검토 가능한 결과물입니다. 입력을 많이 모으는 것과 출력을 만드는 것은 다르며, 성장 속도는 출력 빈도에 더 강하게 좌우됩니다.

셋째, 판단 기준을 미리 적습니다. 많은 작업이 끝에서 흔들리는 이유는 성공 기준을 사전에 합의하지 않았기 때문입니다. "정확한가"라는 문장 대신 "분모 정의가 명확한가", "누락 세그먼트가 없는가", "후속 액션이 있는가"처럼 점검 문항으로 바꾸면 품질이 안정됩니다.

넷째, 협업 단위를 작게 쪼개 공유합니다. 완성본만 보여 주려는 습관은 피드백 타이밍을 늦춥니다. 중간 결과를 1페이지 요약으로 공유하고, 가정과 리스크를 먼저 드러내면 이후 수정 비용이 크게 줄어듭니다. 데이터 직무는 혼자 완벽하게 만든 뒤 공개하는 방식보다, 가정 기반으로 빠르게 정렬하는 방식이 장기적으로 더 강합니다.

다섯째, 회고는 감상이 아니라 의사결정 기록으로 남깁니다. "어려웠다"로 끝내지 말고 무엇이 병목이었는지, 다음에는 어떤 순서로 접근할지, 어떤 지표를 추가로 볼지를 적어야 합니다. 같은 유형의 문제를 다시 만났을 때 과거 기록이 바로 템플릿이 되면 성장 속도는 가속됩니다.

아래 표는 매주 반복 가능한 최소 운영 템플릿입니다.

| 항목 | 작성 질문 | 예시 출력 |
| --- | --- | --- |
| 목표 | 이번 주에 어떤 문제를 풀 것인가 | 문제 정의 3줄 |
| 입력 | 어떤 데이터/자료를 사용할 것인가 | 데이터 출처 목록 |
| 방법 | 어떤 접근 순서로 진행할 것인가 | 작업 단계 체크리스트 |
| 검증 | 무엇을 성공 기준으로 볼 것인가 | 품질 점검 표 |
| 결과 | 무엇이 바뀌었는가 | 한 문장 권고안 |
| 회고 | 다음 주에 무엇을 바꿀 것인가 | 개선 액션 3개 |

이 템플릿의 핵심은 완벽함이 아니라 반복 가능성입니다. 한 번에 거대한 문서를 만들 필요는 없습니다. 중요한 것은 매주 같은 구조로 남겨서 비교 가능성을 확보하는 일입니다. 비교가 가능해지면 성장도 측정 가능해지고, 측정 가능해지면 커리어 결정의 품질이 올라갑니다.

마지막으로, 실행 프레임은 개인 메모로 끝내지 말고 가능한 범위에서 팀 공유 자산으로 전환하는 것이 좋습니다. 같은 질문이 반복되는 조직일수록 표준 템플릿의 가치가 큽니다. 템플릿 하나를 잘 만드는 사람은 단순히 일을 수행하는 사람이 아니라, 팀이 더 빠르고 정확하게 일하도록 환경을 설계하는 사람으로 인식됩니다.

## 월간 점검 질문 세트

아래 질문은 월말 회고 때 그대로 사용할 수 있는 점검 세트입니다. 첫 번째 질문은 "이번 달에 내가 해결한 문제는 무엇이며, 그 문제는 누구의 의사결정에 영향을 주었는가"입니다. 두 번째 질문은 "내 결과물의 재현 가능성을 다른 동료가 확인할 수 있는가"입니다. 세 번째 질문은 "결과를 한 문장 권고안으로 요약했는가"입니다. 네 번째 질문은 "가정이 틀렸을 때 대안 경로를 제시했는가"입니다. 다섯 번째 질문은 "다음 달에는 무엇을 하지 않을 것인가"입니다.

이 다섯 질문을 매달 반복하면 학습과 실행이 분리되지 않습니다. 특히 마지막 질문은 과부하를 줄이는 데 효과적입니다. 커리어 성장은 더 많은 일을 하는 것만으로 만들어지지 않습니다. 우선순위를 분명히 하고, 덜 중요한 일을 의도적으로 줄일 때 더 빠르게 만들어집니다.

| 질문 | Pass 기준 |
| --- | --- |
| 문제 정의가 명확한가 | 이해관계자가 같은 문장으로 설명 가능 |
| 결과가 재현 가능한가 | 실행 명령과 데이터 기준이 문서화됨 |
| 해석이 행동으로 연결되는가 | 후속 액션이 1개 이상 명시됨 |
| 리스크를 선제적으로 다뤘는가 | 가정과 한계를 함께 기술함 |
| 다음 달 우선순위가 보이는가 | 할 일/하지 않을 일 목록이 분리됨 |

## 채용 공고 읽기 연습: 역할 신호를 문장으로 분해하기

입문 단계에서는 채용 공고를 많이 읽어도 무엇을 핵심 신호로 봐야 하는지 모르면 학습 계획으로 연결되지 않습니다. 그래서 공고를 읽을 때는 기술 키워드 개수보다 문제 유형, 산출물, 협업 상대, 성공 지표를 먼저 추출하는 방식이 효과적입니다. 아래 표처럼 공고 문장을 역할 언어로 번역하면 직함이 달라도 실제 책임을 비교하기 쉬워집니다.

| 공고 문장 | 숨은 역할 신호 | 준비 액션 |
| --- | --- | --- |
| "비즈니스 지표를 설계하고 대시보드를 운영합니다" | Analyst 중심 | 지표 정의서 1개와 대시보드 해석 노트 작성 |
| "데이터 파이프라인 안정화와 품질 모니터링" | Engineer 중심 | 데이터 품질 체크 규칙 5개 정리 |
| "실험 설계와 모델 성능 개선" | Scientist 또는 MLE 중심 | 가설-지표-검증 템플릿으로 모의 답변 연습 |

실행 팁도 간단합니다. 이번 주에는 공고 5개를 골라 표를 채우고, 각 공고에 대해 "내가 바로 보여 줄 수 있는 증거"를 한 줄씩 적어 보십시오. 예를 들어 Analyst 공고라면 코호트 분석 링크, Engineer 공고라면 파이프라인 재실행 절차 문서처럼 구체적 증거를 연결해야 합니다. 이 연습을 반복하면 기술 목록 중심 자기소개에서 벗어나 책임 중심 자기소개로 빠르게 전환할 수 있습니다.

## 커리어 지도: 역할-역량 매트릭스로 시작점 고정하기

직무 이름만 보고 학습을 시작하면 한 달 뒤에 방향을 잃기 쉽습니다. 그래서 첫 글에서는 시작점을 문장으로 고정하는 작업이 중요합니다. 아래 매트릭스는 "내가 어떤 문제를 풀고 싶은가"를 기준으로 역할 적합도를 빠르게 점검하기 위한 도구입니다.

| 관심 문제 | 가장 맞는 시작 역할 | 초반 증거 산출물 | 8주 학습 우선순위 |
| --- | --- | --- | --- |
| 지표 해석과 의사결정 지원 | Data Analyst | KPI 정의서, 코호트 분석 노트 | SQL, 지표 설계, 스토리텔링 |
| 데이터 흐름 안정화 | Data Engineer | 파이프라인 재실행 절차서 | 데이터 모델링, 품질 규칙 |
| 가설 검증과 예측 | Data Scientist | 실험 리포트, 모델 비교표 | 통계, 실험 설계, 평가 |
| 모델 운영과 배포 | ML Engineer | 추론 API 데모, 모니터링 표 | 서빙, 배포, 관측 |

이 표를 채운 뒤에는 "내가 잘하는 것"과 "계속 하고 싶은 것"을 분리해서 적어야 합니다. 잘하는 일은 단기 성과를 만들고, 계속 하고 싶은 일은 장기 커리어를 결정합니다. 두 축이 다를 수 있다는 사실을 초반에 인정하면 방향 전환 비용을 크게 줄일 수 있습니다.

## 이력서 템플릿: 책임 언어로 한 페이지 만들기

데이터 직무 이력서는 기술 목록보다 책임 문장이 먼저 와야 합니다. 아래 템플릿은 신입 또는 주니어가 바로 사용할 수 있는 한 페이지 구조입니다.

```text
[요약]
- 비즈니스 질문을 데이터로 구조화하고, 재현 가능한 분석 결과로 연결합니다.

[핵심 역량]
- SQL: 집계, 윈도우 함수, 지표 검증
- Python: 데이터 정제, EDA, 시각화
- 커뮤니케이션: 분석 결과를 의사결정 문장으로 번역

[대표 프로젝트]
- 문제: 결제 전환율 하락 원인 파악
- 접근: 퍼널 분해 + 세그먼트 비교 + 이상 구간 검증
- 결과: 이탈 구간 특정, 실험안 2개 제시, 주간 지표 반영

[협업 방식]
- PM/마케팅/개발과 가설-지표-실험 기준 정렬
```

핵심은 "무엇을 배웠다"가 아니라 "무슨 문제를 어떻게 풀어 어떤 결정을 가능하게 했는가"를 남기는 일입니다. 이 템플릿으로 문장을 바꾸면 같은 경험도 훨씬 명확하게 전달됩니다.

## 실전 부록: 지원 문서와 면접 답변을 한 세트로 준비하는 방법

글을 읽고 이해한 내용을 실제 커리어 행동으로 바꾸려면, 이력서 문장·포트폴리오 증거·면접 답변이 서로 같은 문제를 가리키도록 정렬해야 합니다. 아래 구성은 데이터 직무 입문자와 주니어가 가장 빠르게 효과를 보는 준비 방식입니다.

### 이력서 불릿 템플릿

```text
- [문제] 어떤 비즈니스 또는 운영 문제를 맡았는가
- [행동] 어떤 데이터 접근(지표/실험/파이프라인/모델)을 적용했는가
- [결과] 무엇이 얼마나 개선되었는가
- [의미] 그 결과가 팀 의사결정에 어떤 영향을 주었는가
```

예시 문장: "결제 퍼널 이탈 구간을 코호트 기준으로 분해해 주요 병목을 식별했고, 온보딩 실험 우선순위를 재정렬해 4주 내 전환율을 개선했습니다." 핵심은 기술 이름보다 책임과 결과를 문장으로 남기는 일입니다.

### 포트폴리오 프로젝트 선정 체크

| 점검 항목 | Pass 기준 |
| --- | --- |
| 문제 명확성 | 한 문장으로 문제와 영향 범위를 설명 가능 |
| 데이터 현실성 | 공개 데이터라도 결측/편향/한계를 문서화 |
| 재현 가능성 | 실행 절차와 검증 기준을 README에 명시 |
| 의사결정 연결 | 결과가 다음 액션으로 이어짐 |

이 표를 기준으로 프로젝트를 고르면 "보여주기용 분석"을 줄이고 "실무형 문제 해결"을 남길 수 있습니다.

### 인터뷰 Q&A 3문장 프레임

1. 문제를 어떻게 정의했는지 말합니다.
2. 어떤 접근을 선택했고 왜 그렇게 판단했는지 말합니다.
3. 결과와 한계를 함께 설명하고 다음 액션을 제시합니다.

이 세 문장으로 답변하면 면접관이 가장 궁금해하는 판단 과정이 빠르게 드러납니다. 특히 한계를 먼저 말하는 태도는 신뢰를 높입니다.

### 보상 협상 기본 스크립트

```text
- 제안 감사합니다. 기대 역할과 성공 기준을 먼저 정렬하고 싶습니다.
- 제가 제공할 수 있는 기여 범위와 최근 증거를 공유드리겠습니다.
- 시장 밴드와 총보상 구조를 기준으로 조정 가능한 항목을 논의하고 싶습니다.
```

협상은 대립이 아니라 역할-보상 정렬입니다. 역할 범위를 수치와 산출물로 제시하면 대화가 감정에서 사실로 이동합니다.

### 역량 점검 매트릭스

| 영역 | 현재 수준(1~5) | 목표 수준(1~5) | 다음 2주 액션 |
| --- | --- | --- | --- |
| 문제 정의 |  |  |  |
| 데이터 처리 |  |  |  |
| 검증 설계 |  |  |  |
| 전달력 |  |  |  |
| 협업 운영 |  |  |  |

월 1회 이 표를 채우면 성장 체감이 감정이 아니라 기록으로 남습니다. 커리어 의사결정의 품질은 결국 기록의 품질에서 올라갑니다.

## 처음 질문으로 돌아가기

- **데이터 직무라고 할 때 실제로 어떤 역할들이 함께 묶이는지 정리합니다?**
  - 본문에서는 analyst, scientist, engineer, ML engineer, analytics engineer 다섯 가지 역할을 구분하고, 각각이 어떤 질문에 답하며 어떤 산출물을 만드는지 정리했습니다.
