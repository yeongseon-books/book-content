# Data Science Career 101 (7/10): 케이스 인터뷰

케이스 인터뷰는 많은 지원자에게 가장 추상적으로 느껴지는 단계입니다. SQL처럼 정답 형태가 명확하지도 않고, 머신러닝처럼 알고리즘 이름을 기준으로 준비할 수도 없기 때문입니다. 그래서 질문을 들으면 곧바로 아이디어를 던지다가 구조 없이 끝나는 경우가 많습니다.

하지만 면접관이 보고 싶은 것은 아이디어의 양이 아니라 사고의 질입니다. 문제를 어떻게 명확히 하고, 어떤 지표를 중심에 둘지 정하고, 어떤 가설을 세운 뒤, 어떤 데이터로 검증하고, 마지막에 어떤 결정을 내릴지까지 보여 줘야 답변이 단단해집니다.

이 글은 Data Science Career 101 시리즈의 일곱 번째 글입니다.

## 먼저 던지는 질문

- 케이스 인터뷰가 실제로 무엇을 평가하는지 설명합니다?
- 문제를 들었을 때 왜 먼저 명확화 질문을 해야 하는지 짚습니다?
- 북극성 지표와 보조 지표를 어떻게 고를지 정리합니다?

## 이 글에서 배우는 내용

- 케이스 유형
- 답변 프레임
- 지표 설계
- 가설 검증
- 의사결정 제안

## 왜 중요한가

기술 역량만으로는 제품 문제를 풀 수 없습니다. 데이터 직무가 커질수록 질문을 구조화하고, 모호한 상황에서 판단 기준을 세우는 힘이 더 중요해집니다.

실무에서는 문제 정의가 늘 불완전한 상태에서 시작합니다. 숫자 하나가 떨어졌을 때도 세그먼트, 기간, 최근 변경 사항, 지표의 의미를 먼저 정리해야 합니다. 케이스 인터뷰는 바로 그 초기 구조화 능력을 압축해서 보는 방식이라고 생각하면 이해가 쉽습니다.

## 한눈에 보는 개념

이 다섯 단계는 답변 템플릿으로 그대로 써도 좋습니다. 문제를 명확히 하지 않으면 뒤의 지표와 데이터 계획이 모두 흔들립니다.

## 핵심 용어

- **product sense**: 기능과 지표가 사용자 경험에 미치는 영향을 읽는 감각입니다.
- **north star metric**: 제품의 핵심 가치를 대표하는 최상위 지표입니다.
- **hypothesis**: 데이터로 검증 가능한 설명입니다.
- **A/B test**: 두 조건을 비교하는 통제 실험입니다.
- **trade-off**: 하나를 얻으면 다른 하나를 잃는 관계입니다.

## 전후 비교

**Before**: "문제를 들으면 숫자부터 아무거나 던진다."

**After**: "Problem to Metric to Hypothesis to Data to Decision 순서로 답을 구성할 수 있다."

## 실습: 다섯 단계 프레임

### 단계 1 — Clarify

```text
"DAU dropped" = which segment, what period?
```

명확화 질문은 시간을 버는 장치가 아니라 잘못된 문제를 푸는 일을 막는 장치입니다. 세그먼트, 기간, 최근 변경 사항을 먼저 확인해야 합니다.

### 단계 2 — Metric

```text
- north star metric
- two supporting metrics
```

핵심 지표 하나와 보조 지표 둘 정도를 잡으면 답변의 중심이 생깁니다. 지표가 없으면 대화는 금방 추상적으로 흩어집니다.

### 단계 3 — Hypothesis

```text
- product change
- external event
- data pipeline
```

가설은 하나만 두면 위험합니다. 제품 변경, 외부 이벤트, 데이터 파이프라인처럼 성격이 다른 가설을 함께 세워야 탐색이 균형을 잡습니다.

### 단계 4 — Data

```text
- which query, which comparison
- A/B feasibility
```

“데이터를 보겠습니다”로는 약합니다. 어떤 비교를 할지, 어떤 로그를 볼지, 실험이 가능한지까지 말해야 실행 계획처럼 들립니다.

### 단계 5 — Decision

```text
- recommended action
- risks
- follow-up measurement
```

케이스 답변은 결론이 있어야 끝납니다. 추천 액션과 리스크, 후속 측정이 함께 나와야 실제 의사결정 제안이 됩니다.

## 이 예시에서 먼저 봐야 할 점

- 명확화가 답변 품질을 끌어올립니다.
- 지표가 대화의 중심축이 됩니다.
- 마지막에는 반드시 결정을 내려야 합니다.

많은 지원자가 케이스 인터뷰를 브레인스토밍처럼 답하다가 끝냅니다. 하지만 면접관이 보고 싶은 것은 아이디어 양보다 구조화된 사고 방식입니다.

## 자주 하는 실수 5가지

1. **질문을 듣자마자 바로 답하는 실수**
2. **지표를 정의하지 않는 실수**
3. **가설을 하나만 두는 실수**
4. **데이터 계획을 추상적으로 말하는 실수**
5. **결정 없이 끝내는 실수**

## 실무에서는 이렇게 나타납니다

빅테크의 PM, 분석가, 사이언티스트 면접에서 케이스 스타일 질문 비중은 높습니다. 실제 업무가 늘 애매한 문제를 구조화하는 과정이기 때문입니다.

## 시니어는 이렇게 생각합니다

- 명확화부터 합니다.
- 북극성 지표를 먼저 고릅니다.
- 세 가지 가설을 세웁니다.
- 데이터 계획을 구체적으로 말합니다.
- 마지막에 결정을 닫습니다.

## 체크리스트

- [ ] 명확화 질문 세 가지를 준비했다.
- [ ] 핵심 지표와 보조 지표를 구분할 수 있다.
- [ ] 서로 다른 성격의 가설 세 가지를 세울 수 있다.
- [ ] 추천 액션과 후속 측정 계획까지 말할 수 있다.

## 정리 및 다음 단계

케이스 인터뷰는 숫자를 많이 아는 사람보다, 애매한 상황을 구조화할 줄 아는 사람에게 유리합니다. 문제를 명확히 하고, 지표를 정하고, 가설과 데이터 계획을 세운 뒤, 마지막에 결정을 내려야 답변이 완성됩니다.

다음 글에서는 첫 데이터 직무에 들어간 뒤 90일 동안 무엇을 먼저 해야 하는지 봅니다.

## 데이터 팀 구조 유형과 협업 패턴

케이스 인터뷰에서는 개인 역량만 보는 것이 아니라 팀 안에서 문제를 풀 수 있는지 함께 평가합니다. 따라서 데이터 팀 구조가 어떻게 생겨 있고, 구조별로 협업 패턴이 어떻게 달라지는지 이해하면 답변 품질이 높아집니다. 특히 "누가 의사결정을 내리고, 누가 실행 책임을 가지는가"를 명확히 말할 수 있어야 합니다.

| 팀 구조 | 특징 | 장점 | 리스크 | 적합한 상황 |
| --- | --- | --- | --- | --- |
| 중앙집중형 | 데이터 조직이 공통 서비스 제공 | 표준화와 품질 관리 용이 | 현업 반응 속도 저하 | 거버넌스가 중요한 대기업 |
| 임베디드형 | 데이터 인력이 제품팀에 직접 배치 | 문제 맥락 이해와 속도 우수 | 팀 간 기준 불일치 가능 | 제품 실험이 빠른 조직 |
| 하이브리드형 | 중앙 플랫폼 + 임베디드 분석 결합 | 표준과 민첩성 균형 | 역할 경계 모호화 | 중대형 성장 조직 |

협업 패턴도 구조별로 달라집니다. 중앙집중형에서는 요청 티켓과 우선순위 정렬이 중요하고, 임베디드형에서는 제품 회의 동시 참여와 빠른 반복이 중요합니다. 하이브리드형에서는 공통 지표 정의와 팀별 실험 속도를 동시에 맞춰야 하므로 문서화와 조율 역량이 핵심이 됩니다.

케이스 답변에서 자주 쓰는 협업 프레임도 정리합니다.

| 단계 | 질문 | 산출물 |
| --- | --- | --- |
| 문제 정렬 | 이 문제의 최종 의사결정자는 누구인가 | 책임자/의사결정 경로 표 |
| 지표 합의 | 성공/실패를 무엇으로 볼 것인가 | KPI 정의 문서 |
| 실행 분담 | 분석, 구현, 검증을 누가 맡는가 | 역할 분담 매트릭스 |
| 결과 공유 | 결과를 어떤 리듬으로 공유할 것인가 | 주간 리포트/리뷰 노트 |

면접에서 "협업"을 말할 때 추상적으로 "소통을 잘합니다"라고 답하면 약합니다. 대신 "중앙팀과 제품팀의 지표 정의가 어긋날 수 있어 먼저 metric spec을 합의하겠습니다"처럼 구조 기반으로 답해야 신뢰가 생깁니다. 케이스 인터뷰는 결국 기술 + 조직 이해를 함께 요구합니다.

실무에서는 구조가 고정되지 않고 성장 단계에 따라 변합니다. 초기에는 임베디드형으로 빠르게 가다가 규모가 커지면 중앙 플랫폼 기능이 필요해집니다. 이 변화 구간에서 중요한 사람은 특정 구조만 고집하는 사람이 아니라, 구조 전환 시 생기는 마찰을 줄이는 사람입니다.

정리하면, 팀 구조를 이해한 답변은 현실성이 높습니다. 같은 분석 아이디어라도 "어떤 구조에서 어떻게 실행 가능한가"를 말할 수 있으면 케이스 면접에서 차별화가 가능합니다.

## 실행용 점검 프레임: 07-case-interview.md 편

이 섹션은 앞에서 다룬 내용을 실제 행동으로 옮기기 위한 점검 프레임입니다. 많은 독자가 글을 읽을 때는 이해했다고 느끼지만, 일주일 뒤 실행 단계에서 다시 막히는 이유는 "다음 행동"이 문장으로 고정되어 있지 않기 때문입니다. 그래서 아래 프레임은 학습, 실습, 회고를 연결하는 최소 단위로 구성했습니다.

첫째, 이번 주 목표를 한 문장으로 적습니다. 목표는 도구가 아니라 문제 기준으로 써야 합니다. 예를 들어 "SQL 윈도우 함수 공부"보다 "사용자별 누적 결제를 계산해 이탈 신호를 찾는다"처럼 적는 편이 실행력이 높습니다. 같은 시간이라도 문제 중심 목표가 결과 해석까지 자연스럽게 이어집니다.

둘째, 입력과 출력을 명확히 구분합니다. 입력은 데이터, 로그, 문서, 인터뷰 메모처럼 내가 활용할 재료입니다. 출력은 쿼리, 분석 노트, 비교표, 의사결정 문서처럼 다른 사람이 검토 가능한 결과물입니다. 입력을 많이 모으는 것과 출력을 만드는 것은 다르며, 성장 속도는 출력 빈도에 더 강하게 좌우됩니다.

셋째, 판단 기준을 미리 적습니다. 많은 작업이 끝에서 흔들리는 이유는 성공 기준을 사전에 합의하지 않았기 때문입니다. "정확한가"라는 문장 대신 "분모 정의가 명확한가", "누락 세그먼트가 없는가", "후속 액션이 있는가"처럼 점검 문항으로 바꾸면 품질이 안정됩니다.

넷째, 협업 단위를 작게 쪼개 공유합니다. 완성본만 보여 주려는 습관은 피드백 타이밍을 늦춥니다. 중간 결과를 1페이지 요약으로 공유하고, 가정과 리스크를 먼저 드러내면 이후 수정 비용이 크게 줄어듭니다. 데이터 직무는 혼자 완벽하게 만든 뒤 공개하는 방식보다, 가정 기반으로 빠르게 정렬하는 방식이 장기적으로 더 강합니다.

다섯째, 회고는 감상이 아니라 의사결정 기록으로 남깁니다. "어려웠다"로 끝내지 말고 무엇이 병목이었는지, 다음에는 어떤 순서로 접근할지, 어떤 지표를 추가로 볼지를 적어야 합니다. 같은 유형의 문제를 다시 만났을 때 과거 기록이 바로 템플릿이 되면 성장 속도는 가속됩니다.

아래 표는 매주 반복 가능한 최소 운영 템플릿입니다.

| 항목 | 작성 질문 | 예시 출력 |
| --- | --- | --- |
| 목표 | 이번 주에 어떤 문제를 풀 것인가 | 문제 정의 3줄 |
| 입력 | 어떤 데이터/자료를 사용할 것인가 | 데이터 출처 목록 |
| 방법 | 어떤 접근 순서로 진행할 것인가 | 작업 단계 체크리스트 |
| 검증 | 무엇을 성공 기준으로 볼 것인가 | 품질 점검 표 |
| 결과 | 무엇이 바뀌었는가 | 한 문장 권고안 |
| 회고 | 다음 주에 무엇을 바꿀 것인가 | 개선 액션 3개 |

이 템플릿의 핵심은 완벽함이 아니라 반복 가능성입니다. 한 번에 거대한 문서를 만들 필요는 없습니다. 중요한 것은 매주 같은 구조로 남겨서 비교 가능성을 확보하는 일입니다. 비교가 가능해지면 성장도 측정 가능해지고, 측정 가능해지면 커리어 결정의 품질이 올라갑니다.

마지막으로, 실행 프레임은 개인 메모로 끝내지 말고 가능한 범위에서 팀 공유 자산으로 전환하는 것이 좋습니다. 같은 질문이 반복되는 조직일수록 표준 템플릿의 가치가 큽니다. 템플릿 하나를 잘 만드는 사람은 단순히 일을 수행하는 사람이 아니라, 팀이 더 빠르고 정확하게 일하도록 환경을 설계하는 사람으로 인식됩니다.

## 월간 점검 질문 세트

아래 질문은 월말 회고 때 그대로 사용할 수 있는 점검 세트입니다. 첫 번째 질문은 "이번 달에 내가 해결한 문제는 무엇이며, 그 문제는 누구의 의사결정에 영향을 주었는가"입니다. 두 번째 질문은 "내 결과물의 재현 가능성을 다른 동료가 확인할 수 있는가"입니다. 세 번째 질문은 "결과를 한 문장 권고안으로 요약했는가"입니다. 네 번째 질문은 "가정이 틀렸을 때 대안 경로를 제시했는가"입니다. 다섯 번째 질문은 "다음 달에는 무엇을 하지 않을 것인가"입니다.

이 다섯 질문을 매달 반복하면 학습과 실행이 분리되지 않습니다. 특히 마지막 질문은 과부하를 줄이는 데 효과적입니다. 커리어 성장은 더 많은 일을 하는 것만으로 만들어지지 않습니다. 우선순위를 분명히 하고, 덜 중요한 일을 의도적으로 줄일 때 더 빠르게 만들어집니다.

| 질문 | Pass 기준 |
| --- | --- |
| 문제 정의가 명확한가 | 이해관계자가 같은 문장으로 설명 가능 |
| 결과가 재현 가능한가 | 실행 명령과 데이터 기준이 문서화됨 |
| 해석이 행동으로 연결되는가 | 후속 액션이 1개 이상 명시됨 |
| 리스크를 선제적으로 다뤘는가 | 가정과 한계를 함께 기술함 |
| 다음 달 우선순위가 보이는가 | 할 일/하지 않을 일 목록이 분리됨 |

## 보강 메모: 실전 적용 예시

실전에서는 한 가지 프레임만으로 모든 상황을 풀 수 없습니다. 그래서 동일한 문제를 최소 두 가지 관점으로 다시 써 보는 연습이 필요합니다. 예를 들어 케이스 문제라면 지표 관점과 사용자 여정 관점을 각각 분리해 적어 보고, 온보딩 문제라면 개인 생산성과 팀 신뢰 형성 관점을 분리해 점검합니다. 도메인 학습 문제라면 용어 정확성과 의사결정 연결성을 분리해 체크합니다.

이 보강 연습의 목적은 정답을 늘리는 일이 아니라 시야를 넓히는 일입니다. 같은 데이터라도 질문이 달라지면 결론이 달라질 수 있기 때문입니다. 따라서 최종 결론을 내리기 전에 "내가 놓친 관점이 있는가"를 한 번 더 점검하는 습관이 중요합니다.

## 실전 케이스 분해: 실제 공고 기반 모의 질문

케이스 인터뷰는 추상적인 연습보다 실제 채용 공고의 문제 문장을 가져와 분해할 때 학습 효과가 큽니다. 공고에 자주 등장하는 문제를 케이스 질문으로 바꾸면 준비 방향이 분명해집니다.

| 공고 문장 | 모의 케이스 질문 | 답변 시 필수 요소 |
| --- | --- | --- |
| "리텐션 개선을 위한 데이터 분석" | "월간 리텐션이 하락했을 때 무엇부터 확인하겠습니까?" | 세그먼트 분해, 코호트 기준, 이벤트 정의 |
| "신규 기능 성과 측정" | "출시 2주 후 성과를 어떻게 판단하겠습니까?" | 북극성 지표 + 가드레일 지표 + 기간 설정 |
| "마케팅 효율 최적화" | "CAC가 상승했을 때 어떤 가설을 세우겠습니까?" | 채널별 가설, 실험 가능성, 의사결정 조건 |

면접 연습은 질문 1개당 3분 답변, 2분 반박 대응, 1분 결론 요약으로 구성하면 좋습니다. 이 구조를 반복하면 즉흥적으로 아이디어를 던지는 습관을 줄이고, 구조화된 문제 해결력을 안정적으로 보여 줄 수 있습니다.

## 케이스 인터뷰 실전 스크립트: 질문-구조-계산-의사결정

케이스 인터뷰는 빠른 암산보다 문제 구조화 능력을 확인하는 과정입니다. 아래 스크립트를 몸에 익히면 낯선 도메인에서도 안정적으로 답변할 수 있습니다.

```text
1) 질문 재진술: 성공 지표와 시간 범위를 합의한다.
2) 분해: 사용자, 퍼널, 가격, 채널 등 3~4축으로 쪼갠다.
3) 가정: 필요한 수치를 명시적으로 선언한다.
4) 계산: 중간 결과를 확인하며 단계적으로 계산한다.
5) 결론: 실행 가능한 액션 2개와 리스크 1개를 제시한다.
```

면접관은 계산 정확도뿐 아니라 "가정 관리"를 봅니다. 가정을 말로 공개하면 논리적 신뢰도가 크게 올라갑니다.

## 케이스 연습 문제 예시

- 구독 서비스의 2개월 차 이탈이 증가한 원인을 진단하고 우선 대응안을 제시하십시오.
- 광고 예산이 20% 줄어든 상황에서 신규 가입 목표를 유지하는 방법을 제안하십시오.
- 가격 인상 후 매출은 증가했지만 재구매율이 하락한 이유를 설명하십시오.

각 문제를 풀 때 정답 하나를 찾으려 하지 말고, 가정과 선택 근거를 문서로 남기십시오. 케이스 인터뷰 점수는 "생각 과정의 가시성"에서 갈립니다.

## 실전 부록: 지원 문서와 면접 답변을 한 세트로 준비하는 방법

글을 읽고 이해한 내용을 실제 커리어 행동으로 바꾸려면, 이력서 문장·포트폴리오 증거·면접 답변이 서로 같은 문제를 가리키도록 정렬해야 합니다. 아래 구성은 데이터 직무 입문자와 주니어가 가장 빠르게 효과를 보는 준비 방식입니다.

### 이력서 불릿 템플릿

```text
- [문제] 어떤 비즈니스 또는 운영 문제를 맡았는가
- [행동] 어떤 데이터 접근(지표/실험/파이프라인/모델)을 적용했는가
- [결과] 무엇이 얼마나 개선되었는가
- [의미] 그 결과가 팀 의사결정에 어떤 영향을 주었는가
```

예시 문장: "결제 퍼널 이탈 구간을 코호트 기준으로 분해해 주요 병목을 식별했고, 온보딩 실험 우선순위를 재정렬해 4주 내 전환율을 개선했습니다." 핵심은 기술 이름보다 책임과 결과를 문장으로 남기는 일입니다.

### 포트폴리오 프로젝트 선정 체크

| 점검 항목 | Pass 기준 |
| --- | --- |
| 문제 명확성 | 한 문장으로 문제와 영향 범위를 설명 가능 |
| 데이터 현실성 | 공개 데이터라도 결측/편향/한계를 문서화 |
| 재현 가능성 | 실행 절차와 검증 기준을 README에 명시 |
| 의사결정 연결 | 결과가 다음 액션으로 이어짐 |

이 표를 기준으로 프로젝트를 고르면 "보여주기용 분석"을 줄이고 "실무형 문제 해결"을 남길 수 있습니다.

### 인터뷰 Q&A 3문장 프레임

1. 문제를 어떻게 정의했는지 말합니다.
2. 어떤 접근을 선택했고 왜 그렇게 판단했는지 말합니다.
3. 결과와 한계를 함께 설명하고 다음 액션을 제시합니다.

이 세 문장으로 답변하면 면접관이 가장 궁금해하는 판단 과정이 빠르게 드러납니다. 특히 한계를 먼저 말하는 태도는 신뢰를 높입니다.

### 보상 협상 기본 스크립트

```text
- 제안 감사합니다. 기대 역할과 성공 기준을 먼저 정렬하고 싶습니다.
- 제가 제공할 수 있는 기여 범위와 최근 증거를 공유드리겠습니다.
- 시장 밴드와 총보상 구조를 기준으로 조정 가능한 항목을 논의하고 싶습니다.
```

협상은 대립이 아니라 역할-보상 정렬입니다. 역할 범위를 수치와 산출물로 제시하면 대화가 감정에서 사실로 이동합니다.

### 역량 점검 매트릭스

| 영역 | 현재 수준(1~5) | 목표 수준(1~5) | 다음 2주 액션 |
| --- | --- | --- | --- |
| 문제 정의 |  |  |  |
| 데이터 처리 |  |  |  |
| 검증 설계 |  |  |  |
| 전달력 |  |  |  |
| 협업 운영 |  |  |  |

월 1회 이 표를 채우면 성장 체감이 감정이 아니라 기록으로 남습니다. 커리어 의사결정의 품질은 결국 기록의 품질에서 올라갑니다.

## 추가 점검 메모

케이스 인터뷰에서는 완벽한 정답보다 구조화된 접근이 더 중요합니다. 질문 재정의, 가정 공개, 단계적 계산, 실행 제안의 네 단계를 꾸준히 반복하면 낯선 문제에서도 안정적으로 답변할 수 있습니다.
