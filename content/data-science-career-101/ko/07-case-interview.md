# 케이스 인터뷰

데이터 직무 면접에서 케이스 인터뷰는 많은 지원자를 당황하게 만듭니다. SQL처럼 정답이 보이는 문제도 아니고, 머신러닝처럼 공식 용어를 설명하는 문제도 아닙니다. 대신 제품 지표가 떨어졌을 때 어떻게 접근할지, 어떤 가설을 세울지, 어떤 데이터를 먼저 볼지, 마지막에 어떤 결정을 내릴지 묻습니다.

그래서 케이스 인터뷰는 기술만으로 풀리지 않습니다. 제품 감각과 사고 구조가 함께 필요합니다. 당장 답을 내놓기보다 문제를 먼저 명확히 하고, 어떤 지표를 기준으로 볼지 정하고, 여러 가설을 세운 뒤, 데이터를 통해 우선순위를 좁혀 나가는 방식이 중요합니다.

## 이 글에서 다룰 문제

- 케이스 인터뷰는 무엇을 평가하려고 할까요?
- 질문을 받으면 왜 바로 답하지 말아야 할까요?
- 북극성 지표와 보조 지표는 어떻게 고를 수 있을까요?
- 가설과 데이터 계획은 어떤 수준으로 구체적이어야 할까요?
- 답변을 결정으로 끝내야 하는 이유는 무엇일까요?

> 케이스 인터뷰의 핵심은 똑똑한 추측이 아니라, 문제에서 결정까지 가는 사고 구조를 보여 주는 일입니다.

## 한눈에 보는 전체 흐름

```mermaid
flowchart LR
    Q[문제 명확화] --> M[지표]
    M --> H[가설]
    H --> D[데이터]
    D --> Dec[결정]
```

이 다섯 단계는 사실상 답변 템플릿입니다. 특히 데이터 직무 면접에서는 숫자를 잘 아는 사람보다, 애매한 문제를 구조화할 줄 아는 사람이 더 높은 평가를 받는 경우가 많습니다.

## 핵심 용어

- **product sense**: 기능과 지표가 사용자 경험에 미치는 영향을 읽는 감각입니다.
- **north star metric**: 제품의 핵심 가치를 대표하는 최상위 지표입니다.
- **hypothesis**: 데이터로 검증할 수 있는 설명 가설입니다.
- **A/B test**: 두 조건을 비교하는 통제 실험입니다.
- **trade-off**: 하나를 얻으면 다른 하나가 줄어드는 관계입니다.

## Before/After

**Before**: "문제를 들으면 숫자부터 아무거나 던지는 편이다."

**After**: "문제 명확화 → 지표 → 가설 → 데이터 → 결정 순서로 답을 구성할 수 있다."

## 실습: 다섯 단계 답변 프레임

### Step 1 — Clarify

```text
"DAU dropped" = which segment, what period?
```

문제 명확화는 시간을 버는 단계가 아니라, 잘못된 문제를 푸는 일을 막는 단계입니다. 어느 지역인지, 어느 사용자군인지, 어느 기간인지, 제품 변경이 있었는지부터 물어야 같은 단어를 놓고도 서로 다른 상황을 상상하는 문제를 줄일 수 있습니다.

### Step 2 — Metric

```text
- north star metric
- two supporting metrics
```

핵심 지표 하나와 보조 지표 둘 정도를 잡아 두면 답변의 중심이 생깁니다. 북극성 지표만 말하고 끝내면 원인 분해가 어렵고, 반대로 보조 지표만 나열하면 방향이 흐려집니다.

### Step 3 — Hypothesis

```text
- product change
- external event
- data pipeline
```

가설은 하나만 두면 위험합니다. 제품 변경, 외부 이벤트, 데이터 수집 이슈처럼 성격이 다른 가설을 함께 세워야 탐색 범위가 균형을 잡습니다. 특히 데이터 직군 면접에서는 파이프라인 문제를 가설에 넣는 습관이 좋은 신호가 됩니다.

### Step 4 — Data

```text
- which query, which comparison
- A/B feasibility
```

이 단계에서는 “데이터를 보겠습니다”라고만 말하면 약합니다. 어떤 세그먼트 비교를 할지, 어떤 기간 비교를 할지, 어떤 쿼리나 로그를 볼지, 추가 실험이 가능한지까지 이야기해야 실제 실행 계획처럼 들립니다.

### Step 5 — Decision

```text
- recommended action
- risks
- follow-up measurement
```

케이스 답변은 결론이 있어야 끝납니다. 추천 액션과 리스크, 그리고 후속 측정 계획까지 함께 말하면 단순 아이디어가 아니라 의사결정 제안으로 바뀝니다.

## 이 예시에서 봐야 할 점

- 문제 명확화는 답변 품질을 끌어올리는 첫 단계입니다.
- 지표가 대화의 중심축이 됩니다.
- 마지막에 결정을 내려야 비로소 답변이 닫힙니다.

많은 지원자가 케이스 인터뷰를 브레인스토밍처럼 답하다가 끝냅니다. 하지만 면접관이 보고 싶은 것은 아이디어 양이 아니라, 생각을 구조화하는 방식입니다. 질문을 정리하고, 지표를 정하고, 세 가지 정도의 가설을 세우고, 데이터를 통해 좁힌 뒤, 무엇을 할지 말할 수 있어야 합니다.

## 자주 하는 실수 5가지

1. **문제를 듣자마자 바로 답하는 실수**
2. **지표를 정의하지 않는 실수**
3. **가설을 하나만 세우는 실수**
4. **데이터 계획을 추상적으로 말하는 실수**
5. **결정 없이 아이디어만 말하고 끝내는 실수**

## 실무에서는 이렇게 나타납니다

빅테크의 PM·분석가·사이언티스트 면접에서 케이스 스타일 질문 비중은 꽤 높습니다. 실제 업무가 늘 애매한 문제를 구조화하는 과정이기 때문입니다. 지표 하락, 기능 론치, 실험 결과 해석, 세그먼트별 차이 분석 같은 주제가 면접 문제로 자주 바뀝니다.

## 시니어는 이렇게 생각합니다

- 먼저 문제를 명확히 합니다.
- 북극성 지표와 보조 지표를 같이 봅니다.
- 가설을 세 가지 이상 둡니다.
- 데이터 계획은 구체적으로 말합니다.
- 마지막에는 행동 가능한 결정을 제안합니다.

## 체크리스트

- [ ] 명확화 질문 세 가지를 준비했다.
- [ ] 핵심 지표와 보조 지표를 구분할 수 있다.
- [ ] 서로 다른 성격의 가설 세 가지를 세울 수 있다.
- [ ] 추천 액션과 후속 측정 계획까지 말할 수 있다.

## 연습 문제

1. north star metric을 한 줄로 설명해 보세요.
2. trade-off의 예를 한 줄로 적어 보세요.
3. 좋은 명확화 질문의 기준을 한 줄로 정리해 보세요.

## 정리 및 다음 단계

케이스 인터뷰는 숫자를 많이 아는 사람보다, 애매한 상황을 구조화할 줄 아는 사람에게 유리합니다. 문제를 먼저 명확히 하고, 지표를 정하고, 여러 가설을 세우고, 데이터를 통해 우선순위를 좁힌 뒤, 마지막에 결정을 내려야 답변이 완성됩니다. 이 구조를 몸에 익혀 두면 면접뿐 아니라 실제 분석 업무에서도 큰 도움이 됩니다.
