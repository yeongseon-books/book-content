# Data Science Career 101 (6/10): ML 인터뷰

이 글은 Data Science Career 101 시리즈의 6번째 글입니다.


머신러닝 인터뷰를 처음 준비할 때는 보통 모델 이름과 알고리즘 개념부터 외우기 시작합니다. Random Forest, XGBoost, Logistic Regression, Precision, Recall 같은 키워드는 익숙해지지만, 막상 면접에서 “이 문제에서는 왜 이 지표를 보겠는가?”라는 질문이 나오면 답변이 급격히 얕아지기 쉽습니다.

실무 인터뷰는 모델 암기보다 판단 구조를 더 많이 봅니다. 어떤 문제를 풀고 있는지, 데이터 누수나 드리프트 같은 운영 함정을 알고 있는지, 그리고 모델을 서비스 안에서 어떻게 모니터링할지를 연결해 말할 수 있어야 깊이가 드러납니다.

이 글은 Data Science Career 101 시리즈의 여섯 번째 글입니다.

## 먼저 던지는 질문

- ML 인터뷰가 실제로 어떤 영역을 묻는지 정리합니다?
- 모델 선택을 설명할 때 무엇을 먼저 말해야 하는지 설명합니다?
- 평가 지표를 문제 정의와 함께 봐야 하는 이유를 짚습니다?

## 이 글에서 배우는 내용

- 기초 개념 질문
- 모델 선택 논리
- 평가 지표
- 운영 함정
- ML 시스템 설계 관점

## 왜 중요한가

머신러닝 면접은 모델 암기 시험이 아닙니다. 어떤 문제를 풀고 있는지, 어떤 지표를 우선해야 하는지, 운영에서 어떤 리스크가 있는지를 함께 생각할 수 있는지 보는 자리입니다.

특히 최근 면접은 모델 선택보다 배포 이후를 더 자주 묻습니다. 데이터가 언제 들어오는지, 재학습 주기는 어떻게 잡는지, 성능 저하를 어떻게 감지하는지 설명해야 실제 서비스 감각이 있는 지원자로 보입니다.

## 한눈에 보는 개념

이 흐름을 답변 구조로 잡으면 훨씬 단단해집니다. 문제를 정의하고, 모델을 선택하고, 평가 기준을 설명한 뒤, 배포와 모니터링까지 연결해야 합니다.

## 핵심 용어

- **bias-variance**: 과소적합과 과적합 사이의 균형입니다.
- **overfitting**: 학습 데이터를 지나치게 외워 일반화가 약해진 상태입니다.
- **AUC**: ROC 곡선 아래 면적입니다.
- **precision/recall**: 오탐과 미탐 사이의 균형을 보여 주는 지표입니다.
- **drift**: 시간이 지나며 데이터 분포가 변하는 현상입니다.

## 전후 비교

**Before**: "Random Forest가 늘 무난한 정답이라고 생각했다."

**After**: "문제 정의와 지표를 기준으로 모델을 고를 수 있다."

## 실습: 다섯 가지 답변 패턴

### 단계 1 — Fundamentals

```text
Explain bias-variance in one line.
```

기초 개념 질문은 짧지만 강합니다. 한 줄로 설명할 수 있을 정도로 개념을 압축해 두어야 응용 질문도 흔들리지 않습니다.

### 단계 2 — 모델 선택
```text
- assumptions: linear vs tree vs neural
- data size, interpretability
```

모델 선택 문제에서는 이름보다 기준이 중요합니다. 데이터 크기, 가정, 해석 가능성, 지연 요구사항 같은 판단 기준을 먼저 말해야 합니다.

### 단계 3 — Evaluation

```python
from sklearn.metrics import precision_score, recall_score, roc_auc_score
```

평가 지표는 문제 맥락과 분리해서 말할 수 없습니다. 비용 구조가 다른 문제에서는 precision과 recall의 우선순위도 달라집니다.

### 단계 4 — 운영 환경 함정
```text
- data leakage
- class imbalance
- time leakage
```

이 함정들을 먼저 언급하면 답변의 깊이가 달라집니다. 특히 leakage를 알고 있다는 사실 자체가 실무 감각 신호가 됩니다.

### 단계 5 — 시스템 설계
```text
- data -> train -> serve -> monitor
- retraining cadence
- drift detection
```

좋은 답변은 모델 하나를 잘 고르는 데서 멈추지 않습니다. 데이터 수집부터 서빙과 모니터링까지 하나의 시스템으로 설명할 수 있어야 합니다.

## 이 예시에서 먼저 봐야 할 점

- 지표가 답변의 방향을 결정합니다.
- 함정을 먼저 언급하면 시니어리티가 드러납니다.
- 모델을 시스템 일부로 보는 관점이 중요합니다.

많은 지원자가 정확도나 AUC만 말하고 끝내지만, 서비스 환경에서는 그것만으로 충분하지 않습니다. 데이터 분포 변화와 운영 비용까지 생각해야 실제 답이 됩니다.

## 자주 하는 실수 5가지

1. **모든 문제에 같은 모델을 답으로 내는 실수**
2. **AUC만 보고 끝내는 실수**
3. **leakage를 모르는 실수**
4. **재학습 계획이 없는 실수**
5. **해석 가능성을 무시하는 실수**

## 실무에서는 이렇게 나타납니다

실제 인터뷰에서는 정확도보다 운영 질문이 더 길어지는 경우가 많습니다. 데이터가 언제 들어오고, 예측이 어디서 쓰이고, 분포가 바뀌면 어떻게 대응할지를 설명할 수 있어야 하기 때문입니다.

## 시니어는 이렇게 생각합니다

- 문제 정의에서 시작합니다.
- 지표가 모델 선택을 이끕니다.
- 함정을 먼저 떠올립니다.
- 개별 모델보다 시스템 수준에서 봅니다.
- 드리프트와 재학습 계획까지 포함합니다.

## 정리 및 다음 단계

ML 인터뷰에서 중요한 것은 모델 이름을 많이 아는 일이 아니라, 문제와 지표와 운영 리스크를 함께 묶어 설명하는 일입니다. 문제 정의, 모델 선택, 평가, 함정, 시스템까지 하나의 흐름으로 답할 수 있어야 합니다.

다음 글에서는 제품 감각과 사고 구조를 많이 보는 케이스 인터뷰를 다루겠습니다.


## 도메인별 데이터 직무 비교: 금융, 헬스, 커머스

ML 인터뷰를 준비할 때 모델 지식만큼 중요한 축이 도메인 맥락입니다. 같은 모델이라도 금융, 헬스, 커머스에서는 실패 비용과 규제, 지표 해석 방식이 다르기 때문에 답변 구조가 달라져야 합니다. 면접관도 보통 "어떤 도메인에서 어떻게 판단을 바꿀 것인가"를 통해 실무 적합성을 확인합니다.

| 도메인 | 대표 문제 | 우선 지표 | 리스크 | 모델링 관점 |
| --- | --- | --- | --- | --- |
| 금융 | 부정 거래 탐지, 신용 리스크 | 정밀도/재현율 균형, 손실 비용 | 오탐 비용, 규제 준수 | 설명 가능성, 감사 가능성 강조 |
| 헬스케어 | 재입원 예측, 진단 보조 | 재현율, 민감도, 안전성 | 미탐 위험, 윤리/법적 책임 | 보수적 임계값, 임상 맥락 반영 |
| 커머스 | 추천, 이탈 예측, 수요 예측 | 전환율, 매출, 리텐션 | 과최적화, 시즌성 편향 | 실험 반복, 빠른 배포-검증 |

금융 도메인에서는 모델 성능 수치만으로는 부족합니다. "왜 이 예측이 나왔는지"를 설명할 수 있어야 하며, 규제 감사나 내부 통제 문서화가 중요합니다. 따라서 인터뷰 답변에서도 feature importance, 임계값 근거, 오탐/미탐 비용을 반드시 함께 말하는 편이 좋습니다.

헬스케어에서는 정확도보다 안전성 우선 원칙이 자주 등장합니다. 예를 들어 미탐의 비용이 매우 큰 문제에서는 재현율을 더 높게 가져가고, 후속 확인 절차를 포함하는 운영 설계를 제안해야 합니다. 또한 데이터 편향과 윤리 이슈를 알고 있다는 점을 명확히 드러내야 실무 감각이 높게 평가됩니다.

커머스는 실험 속도와 반복 학습 루프가 강점인 도메인입니다. 모델을 완벽하게 만들기보다 빠르게 배포하고 지표로 검증하면서 개선하는 방식이 일반적입니다. 다만 시즌성, 프로모션, 재고 변화 같은 외부 요인을 지표 해석에 반영하지 않으면 잘못된 결론으로 이어질 수 있습니다.

아래는 도메인별 면접 답변 템플릿입니다.

| 도메인 | 답변 시작 문장 예시 |
| --- | --- |
| 금융 | "이 문제는 모델 성능뿐 아니라 규제 감사 가능성과 비용 구조를 함께 봐야 합니다." |
| 헬스케어 | "이 문제에서는 미탐 비용이 커서 재현율과 안전 절차를 우선하겠습니다." |
| 커머스 | "빠른 실험과 롤백 가능한 배포 전략으로 학습 루프를 짧게 가져가겠습니다." |

도메인 관점은 모델 선택에도 영향을 줍니다. 금융에서는 고성능 블랙박스 모델이 항상 정답이 아닐 수 있으며, 헬스케어에서는 데이터 품질과 라벨 신뢰도가 모델 복잡도보다 중요할 때가 많습니다. 커머스에서는 온라인 실험 인프라가 모델 자체만큼 큰 경쟁력이 됩니다.

정리하면, ML 인터뷰에서 도메인 맥락을 말할 수 있다는 것은 "모델을 아는 사람"을 넘어 "운영 현실을 이해하는 사람"이라는 신호입니다. 같은 알고리즘 지식이라도 도메인별 위험과 지표를 연결해 설명하면 답변 깊이가 분명히 달라집니다.


## 실행용 점검 프레임: 06-ml-interview.md 편

이 섹션은 앞에서 다룬 내용을 실제 행동으로 옮기기 위한 점검 프레임입니다. 많은 독자가 글을 읽을 때는 이해했다고 느끼지만, 일주일 뒤 실행 단계에서 다시 막히는 이유는 "다음 행동"이 문장으로 고정되어 있지 않기 때문입니다. 그래서 아래 프레임은 학습, 실습, 회고를 연결하는 최소 단위로 구성했습니다.

첫째, 이번 주 목표를 한 문장으로 적습니다. 목표는 도구가 아니라 문제 기준으로 써야 합니다. 예를 들어 "SQL 윈도우 함수 공부"보다 "사용자별 누적 결제를 계산해 이탈 신호를 찾는다"처럼 적는 편이 실행력이 높습니다. 같은 시간이라도 문제 중심 목표가 결과 해석까지 자연스럽게 이어집니다.

둘째, 입력과 출력을 명확히 구분합니다. 입력은 데이터, 로그, 문서, 인터뷰 메모처럼 내가 활용할 재료입니다. 출력은 쿼리, 분석 노트, 비교표, 의사결정 문서처럼 다른 사람이 검토 가능한 결과물입니다. 입력을 많이 모으는 것과 출력을 만드는 것은 다르며, 성장 속도는 출력 빈도에 더 강하게 좌우됩니다.

셋째, 판단 기준을 미리 적습니다. 많은 작업이 끝에서 흔들리는 이유는 성공 기준을 사전에 합의하지 않았기 때문입니다. "정확한가"라는 문장 대신 "분모 정의가 명확한가", "누락 세그먼트가 없는가", "후속 액션이 있는가"처럼 점검 문항으로 바꾸면 품질이 안정됩니다.

넷째, 협업 단위를 작게 쪼개 공유합니다. 완성본만 보여 주려는 습관은 피드백 타이밍을 늦춥니다. 중간 결과를 1페이지 요약으로 공유하고, 가정과 리스크를 먼저 드러내면 이후 수정 비용이 크게 줄어듭니다. 데이터 직무는 혼자 완벽하게 만든 뒤 공개하는 방식보다, 가정 기반으로 빠르게 정렬하는 방식이 장기적으로 더 강합니다.

다섯째, 회고는 감상이 아니라 의사결정 기록으로 남깁니다. "어려웠다"로 끝내지 말고 무엇이 병목이었는지, 다음에는 어떤 순서로 접근할지, 어떤 지표를 추가로 볼지를 적어야 합니다. 같은 유형의 문제를 다시 만났을 때 과거 기록이 바로 템플릿이 되면 성장 속도는 가속됩니다.

아래 표는 매주 반복 가능한 최소 운영 템플릿입니다.

| 항목 | 작성 질문 | 예시 출력 |
| --- | --- | --- |
| 목표 | 이번 주에 어떤 문제를 풀 것인가 | 문제 정의 3줄 |
| 입력 | 어떤 데이터/자료를 사용할 것인가 | 데이터 출처 목록 |
| 방법 | 어떤 접근 순서로 진행할 것인가 | 작업 단계 체크리스트 |
| 검증 | 무엇을 성공 기준으로 볼 것인가 | 품질 점검 표 |
| 결과 | 무엇이 바뀌었는가 | 한 문장 권고안 |
| 회고 | 다음 주에 무엇을 바꿀 것인가 | 개선 액션 3개 |

이 템플릿의 핵심은 완벽함이 아니라 반복 가능성입니다. 한 번에 거대한 문서를 만들 필요는 없습니다. 중요한 것은 매주 같은 구조로 남겨서 비교 가능성을 확보하는 일입니다. 비교가 가능해지면 성장도 측정 가능해지고, 측정 가능해지면 커리어 결정의 품질이 올라갑니다.

마지막으로, 실행 프레임은 개인 메모로 끝내지 말고 가능한 범위에서 팀 공유 자산으로 전환하는 것이 좋습니다. 같은 질문이 반복되는 조직일수록 표준 템플릿의 가치가 큽니다. 템플릿 하나를 잘 만드는 사람은 단순히 일을 수행하는 사람이 아니라, 팀이 더 빠르고 정확하게 일하도록 환경을 설계하는 사람으로 인식됩니다.


## 월간 점검 질문 세트

아래 질문은 월말 회고 때 그대로 사용할 수 있는 점검 세트입니다. 첫 번째 질문은 "이번 달에 내가 해결한 문제는 무엇이며, 그 문제는 누구의 의사결정에 영향을 주었는가"입니다. 두 번째 질문은 "내 결과물의 재현 가능성을 다른 동료가 확인할 수 있는가"입니다. 세 번째 질문은 "결과를 한 문장 권고안으로 요약했는가"입니다. 네 번째 질문은 "가정이 틀렸을 때 대안 경로를 제시했는가"입니다. 다섯 번째 질문은 "다음 달에는 무엇을 하지 않을 것인가"입니다.

이 다섯 질문을 매달 반복하면 학습과 실행이 분리되지 않습니다. 특히 마지막 질문은 과부하를 줄이는 데 효과적입니다. 커리어 성장은 더 많은 일을 하는 것만으로 만들어지지 않습니다. 우선순위를 분명히 하고, 덜 중요한 일을 의도적으로 줄일 때 더 빠르게 만들어집니다.

| 질문 | Pass 기준 |
| --- | --- |
| 문제 정의가 명확한가 | 이해관계자가 같은 문장으로 설명 가능 |
| 결과가 재현 가능한가 | 실행 명령과 데이터 기준이 문서화됨 |
| 해석이 행동으로 연결되는가 | 후속 액션이 1개 이상 명시됨 |
| 리스크를 선제적으로 다뤘는가 | 가정과 한계를 함께 기술함 |
| 다음 달 우선순위가 보이는가 | 할 일/하지 않을 일 목록이 분리됨 |

## 연봉 밴드와 기대 역할: ML 포지션 해석 표

ML 포지션 지원 시 연봉 숫자만 비교하면 의사결정이 왜곡될 수 있습니다. 연봉은 역할 기대치, 온콜 책임, 배포 범위와 함께 읽어야 실제 성장성과 부담을 정확히 판단할 수 있습니다. 아래 표는 국내외 공고에서 자주 보이는 패턴을 단순화한 참고 프레임입니다.

| 포지션 레벨 | 자주 요구되는 책임 | 보상 해석 포인트 |
| --- | --- | --- |
| 주니어 ML 엔지니어 | 모델 개선, 오프라인 평가, 기본 배포 지원 | 멘토링 체계와 학습 기회가 보상만큼 중요 |
| 미드레벨 ML 엔지니어 | 서빙 안정화, 모니터링, 재학습 자동화 | 온콜 범위와 운영 책임 대비 보상 확인 |
| 시니어 ML 엔지니어 | 아키텍처 결정, 팀 표준화, 타팀 협업 리드 | 영향 범위와 의사결정 권한이 핵심 |

면접에서 연봉 질문이 나오면 금액 중심 답변보다 책임-성과 중심으로 대화하는 편이 좋습니다. 예를 들어 "배포 이후 모니터링 책임과 재학습 주기를 제가 오너십 있게 운영할 수 있는 환경인지"를 먼저 확인하면, 단기 보상과 장기 성장의 균형을 더 정확히 맞출 수 있습니다.


## 머신러닝 인터뷰 대비: 모델링보다 검증 설계가 먼저입니다

ML 인터뷰에서 자주 나오는 실수는 모델 이름을 길게 나열하는 일입니다. 실무 면접관은 "어떤 상황에서 어떤 검증 전략을 택했는가"를 더 중요하게 봅니다.

| 질문 유형 | 좋은 답변의 핵심 | 피해야 할 답변 |
| --- | --- | --- |
| 데이터 누수 탐지 | 누수 경로 가설과 차단 방법 제시 | "성능이 높으면 괜찮다" |
| 불균형 데이터 | 지표 선택 이유(PR-AUC, Recall 등) 설명 | Accuracy만 제시 |
| 배포 후 성능 저하 | 드리프트 지표와 대응 절차 제시 | 재학습만 반복 언급 |

특히 "왜 그 지표를 선택했는가"를 말하지 못하면 모델링 경험이 있어도 실무 적합도가 낮게 평가됩니다.

## 모델 운영 체크리스트: 면접 답변용

- 입력 데이터 스키마 검증 규칙이 있는가
- 학습/서빙 피처 변환이 동일한가
- 배포 버전 롤백 절차가 있는가
- 성능 저하 알림 임계값이 정의되어 있는가
- 실패 시 책임자와 대응 시간 기준이 있는가

이 체크리스트를 실제 프로젝트에 적용한 경험이 있으면, "모델을 만든 사람"이 아니라 "모델을 운영한 사람"으로 인식됩니다.


## 실전 부록: 지원 문서와 면접 답변을 한 세트로 준비하는 방법

글을 읽고 이해한 내용을 실제 커리어 행동으로 바꾸려면, 이력서 문장·포트폴리오 증거·면접 답변이 서로 같은 문제를 가리키도록 정렬해야 합니다. 아래 구성은 데이터 직무 입문자와 주니어가 가장 빠르게 효과를 보는 준비 방식입니다.

### 이력서 불릿 템플릿

```text
- [문제] 어떤 비즈니스 또는 운영 문제를 맡았는가
- [행동] 어떤 데이터 접근(지표/실험/파이프라인/모델)을 적용했는가
- [결과] 무엇이 얼마나 개선되었는가
- [의미] 그 결과가 팀 의사결정에 어떤 영향을 주었는가
```

예시 문장: "결제 퍼널 이탈 구간을 코호트 기준으로 분해해 주요 병목을 식별했고, 온보딩 실험 우선순위를 재정렬해 4주 내 전환율을 개선했습니다." 핵심은 기술 이름보다 책임과 결과를 문장으로 남기는 일입니다.

### 포트폴리오 프로젝트 선정 체크

| 점검 항목 | Pass 기준 |
| --- | --- |
| 문제 명확성 | 한 문장으로 문제와 영향 범위를 설명 가능 |
| 데이터 현실성 | 공개 데이터라도 결측/편향/한계를 문서화 |
| 재현 가능성 | 실행 절차와 검증 기준을 README에 명시 |
| 의사결정 연결 | 결과가 다음 액션으로 이어짐 |

이 표를 기준으로 프로젝트를 고르면 "보여주기용 분석"을 줄이고 "실무형 문제 해결"을 남길 수 있습니다.

### 인터뷰 Q&A 3문장 프레임

1. 문제를 어떻게 정의했는지 말합니다.
2. 어떤 접근을 선택했고 왜 그렇게 판단했는지 말합니다.
3. 결과와 한계를 함께 설명하고 다음 액션을 제시합니다.

이 세 문장으로 답변하면 면접관이 가장 궁금해하는 판단 과정이 빠르게 드러납니다. 특히 한계를 먼저 말하는 태도는 신뢰를 높입니다.

### 보상 협상 기본 스크립트

```text
- 제안 감사합니다. 기대 역할과 성공 기준을 먼저 정렬하고 싶습니다.
- 제가 제공할 수 있는 기여 범위와 최근 증거를 공유드리겠습니다.
- 시장 밴드와 총보상 구조를 기준으로 조정 가능한 항목을 논의하고 싶습니다.
```

협상은 대립이 아니라 역할-보상 정렬입니다. 역할 범위를 수치와 산출물로 제시하면 대화가 감정에서 사실로 이동합니다.

### 역량 점검 매트릭스

| 영역 | 현재 수준(1~5) | 목표 수준(1~5) | 다음 2주 액션 |
| --- | --- | --- | --- |
| 문제 정의 |  |  |  |
| 데이터 처리 |  |  |  |
| 검증 설계 |  |  |  |
| 전달력 |  |  |  |
| 협업 운영 |  |  |  |

월 1회 이 표를 채우면 성장 체감이 감정이 아니라 기록으로 남습니다. 커리어 의사결정의 품질은 결국 기록의 품질에서 올라갑니다.


## 추가 점검 메모

ML 인터뷰는 모델 성능 숫자보다 검증 설계의 일관성을 봅니다. 데이터 분할 기준, 지표 선택 이유, 배포 후 모니터링 계획까지 한 세트로 말할 수 있어야 운영 역량이 드러납니다.


모델 선택 근거를 설명할 때는 성능 숫자 하나보다 비용, 지연 시간, 유지보수 난도를 함께 제시해야 합니다. 같은 정확도라면 운영 비용이 낮고 장애 대응이 쉬운 선택이 장기적으로 더 높은 평가를 받습니다. 이 관점을 답변에 포함하면 연구 경험과 운영 감각을 동시에 보여줄 수 있습니다.

## 처음 질문으로 돌아가기

- **ML 인터뷰가 실제로 어떤 영역을 묻는지 정리합니다?**
  - 본문에서는 ML 인터뷰 준비을 위해 알아야 할 핵심 개념과 전략을 정리했습니다.
- **모델 선택을 설명할 때 무엇을 먼저 말해야 하는지 설명합니다?**
  - 예제를 통해 실제 상황에서 어떻게 접근하는지, 무엇을 놓치기 쉬운지 보았습니다.
- **평가 지표를 문제 정의와 함께 봐야 하는 이유를 짚습니다?**
  - 실제 준비할 때는 단기(2주)와 장기(3개월) 전략을 함께 세워 꾸준히 개선해야 합니다.
