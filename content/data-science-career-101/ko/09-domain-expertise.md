# Data Science Career 101 (9/10): 도메인 전문성 쌓기

이 글은 Data Science Career 101 시리즈의 9번째 글입니다.


데이터 직무에서 어느 순간부터는 기술만으로 설명되지 않는 차이가 벌어집니다. 같은 대시보드를 보고도 어떤 사람은 “숫자가 조금 흔들렸네”에서 멈추고, 어떤 사람은 “이건 프로모션 구조와 환불 정책 변화까지 같이 봐야 한다”는 식으로 더 좋은 질문을 던집니다. 그 차이를 만드는 축이 바로 도메인 전문성입니다.

입문자일수록 기술 학습만으로도 바쁘기 때문에 도메인 공부를 나중으로 미루기 쉽습니다. 하지만 실제로는 용어, KPI, 현장 흐름을 조금만 이해해도 질문 수준과 해석 품질이 빠르게 달라집니다. 도메인은 나중에 붙는 장식이 아니라, 기술을 실제 가치로 연결하는 번역기입니다.

이 글은 Data Science Career 101 시리즈의 아홉 번째 글입니다.

> 도메인 전문성에서 가장 중요한 것은 무엇을 할 수 있는가가 아니라 무엇을 물어봐야 하고 어떻게 답해야 하는가입니다.

## 먼저 던지는 질문

- 데이터 직무에서 도메인 전문성이 왜 중요한지 설명합니다?
- 용어집과 KPI를 어떻게 익히면 좋은지 정리합니다?
- 현장 관찰이 숫자 해석 능력을 왜 높이는지 짚습니다?

## 이 글에서 배우는 내용

- 도메인의 정의
- 용어집 학습
- 핵심 지표 이해
- 현장 관찰
- 지속적 학습 루프

## 왜 중요한가

기술만으로는 비즈니스 의미를 해석할 수 없습니다. 업종과 제품 맥락을 이해해야 같은 그래프를 봐도 중요한 변화인지 아닌지 구분할 수 있습니다.

특히 KPI는 업종에 따라 정상 범위와 의미가 다릅니다. 게임의 리텐션과 핀테크의 리텐션은 해석 기준이 다르고, 헬스케어의 전환율과 커머스의 전환율도 위험과 비용 구조가 전혀 다릅니다. 도메인을 알아야 숫자를 읽는 해상도가 올라갑니다.

## 한눈에 보는 개념

도메인 학습은 보통 이 순서로 쌓입니다. 용어를 알아야 회의가 들리고, 지표를 알아야 숫자가 읽히고, 현장을 알아야 왜 그런 숫자가 나오는지 감이 생깁니다.

## 핵심 용어

- **domain**: 특정 산업이나 비즈니스 영역입니다.
- **glossary**: 용어와 정의를 정리한 목록입니다.
- **KPI**: 핵심 성과 지표입니다.
- **playbook**: 반복 상황에서 따르는 운영 대응 가이드입니다.
- **shadowing**: 현장 업무를 옆에서 관찰하며 배우는 방식입니다.

## 전후 비교

**Before**: "업계를 잘 모르는 상태에서 대시보드 숫자만 바꿔 왔다."

**After**: "KPI와 업계 용어를 기준으로 비즈니스 대화를 이어 갈 수 있다."

## 실습: 다섯 단계 도메인 학습

### 단계 1 — 용어집 만들기

```text
- 30 words
- definition + example
```

도메인 학습의 시작은 어휘입니다. 회의에서 자주 나오는 단어 30개만 정리해도 이해도와 질문 수준이 눈에 띄게 달라집니다.

### Step 2 — Five Key Metrics

```text
- definition
- formula
- consuming team
```

지표는 이름만 알면 부족합니다. 계산식, 사용하는 팀, 어떤 의사결정에 연결되는지까지 알아야 해석이 깊어집니다.

### Step 3 — Field Shadowing

```text
- spend a day with sales or operations
- notes and questions
```

현장을 보면 숫자 뒤의 맥락이 빨리 보입니다. 운영팀이나 영업팀의 실제 흐름을 한 번만 봐도 왜 특정 KPI가 중요한지 훨씬 잘 이해됩니다.

### Step 4 — External Study

```text
- one industry conference per quarter
- industry news RSS
```

내부 데이터만 보면 조직 내부 시야에 갇히기 쉽습니다. 외부 학습은 업계 변화가 내부 지표에 어떻게 반영되는지 읽게 해 줍니다.

### Step 5 — Quarterly Retro

```text
- ten new words
- three new metrics
```

도메인 전문성도 회고가 필요합니다. 분기마다 새로 익힌 용어와 지표를 점검해야 지식이 흩어지지 않고 쌓입니다.

## 이 예시에서 먼저 봐야 할 점

- 어휘가 입구입니다.
- 지표가 방향을 잡아 줍니다.
- 현장이 진실을 보여 줍니다.

기술 공부만으로도 바쁜 입문자일수록 도메인 학습을 뒤로 미루기 쉽습니다. 하지만 도메인을 모르면 좋은 질문을 만들기 어렵고, 숫자의 중요도를 판단하기도 어렵습니다.

## 자주 하는 실수 5가지

1. **기술만 깊게 파는 실수**
2. **용어집 없이 회의를 버티는 실수**
3. **지표 정의를 모호하게 아는 실수**
4. **현장을 전혀 보지 않는 실수**
5. **외부 학습을 건너뛰는 실수**

## 실무에서는 이렇게 나타납니다

핀테크, 헬스케어, 게임처럼 규제와 사용자 행동 패턴이 강한 산업일수록 도메인이 답을 바꿉니다. 같은 리텐션 지표라도 업종마다 정상 범위와 해석이 달라질 수 있습니다.

## 시니어는 이렇게 생각합니다

- 용어부터 정확히 익힙니다.
- 지표 정의를 흐리지 않습니다.
- 현장을 직접 관찰합니다.
- 외부 변화도 같이 공부합니다.
- 도메인 학습을 반복 루프로 만듭니다.

## 정리 및 다음 단계

데이터 직무에서 기술은 출발점이지만, 도메인 전문성은 해석의 질을 바꾸는 힘입니다. 용어집을 만들고, 핵심 KPI를 정리하고, 현장을 보고, 외부 학습을 병행하면 숫자 뒤의 맥락이 훨씬 선명해집니다.

다음 글에서는 이런 경험을 바탕으로 시니어 데이터 직무로 성장할 때 무엇이 달라지는지 살펴보겠습니다.


## 리더십 역할 비교: IC Staff와 Manager

도메인 전문성을 쌓다 보면 다음 단계에서 자주 마주치는 선택이 있습니다. 개인 기여자(IC)로 깊이를 확장할지, 매니저로 팀 운영 책임을 넓힐지 결정해야 하는 시점입니다. 이 선택은 우열의 문제가 아니라 영향 방식의 차이입니다. 본인의 강점과 에너지 패턴에 맞춰 선택해야 장기적으로 지속 가능합니다.

| 구분 | IC Staff 트랙 | Manager 트랙 |
| --- | --- | --- |
| 핵심 미션 | 어려운 문제 해결과 기술 방향 제시 | 팀 성과와 사람 성장 책임 |
| 영향 방식 | 설계/표준/기술적 의사결정 | 우선순위 조율/코칭/채용 |
| 주요 산출물 | 아키텍처 제안, 기술 표준, 고난도 프로젝트 | 목표 설정, 인력 운영, 프로세스 개선 |
| 성공 지표 | 재사용성, 기술 리스크 감소, 성능 개선 | 팀 생산성, 이직률/성장률, 목표 달성 |
| 자주 하는 일 | 리뷰, 설계 문서, 난제 디버깅 | 1:1, 피드백, 갈등 조정, 리소스 배분 |
| 위험 요소 | 고립, 우선순위 영향력 부족 | 기술 감각 약화, 과도한 회의 |

IC Staff는 "내가 얼마나 많이 코딩했는가"보다 "조직 전체가 얼마나 더 쉽게 일하게 되었는가"로 평가받습니다. 즉, 복잡한 문제를 해결하고 그 방식을 표준화해 팀 전체 속도를 높여야 합니다. Manager는 "내가 무엇을 직접 만들었는가"보다 "팀이 어떤 결과를 지속적으로 만들고 있는가"가 중심입니다.

아래는 두 트랙 선택 전에 점검할 질문입니다.

| 점검 질문 | IC 적합 신호 | Manager 적합 신호 |
| --- | --- | --- |
| 에너지가 오르는 순간은 언제인가 | 기술 난제 해결 시 | 동료 성장과 팀 정렬 시 |
| 반복 업무를 견디는 방식은 | 깊이 파고 자동화하는 편 | 사람/프로세스 조정으로 푸는 편 |
| 갈등 상황에서 선호하는 행동은 | 사실/구조 중심 조정 | 관계/맥락 포함 조정 |

실무에서는 두 역할이 완전히 분리되지 않습니다. Staff도 멘토링과 조율을 하고, Manager도 기술 판단을 이해해야 합니다. 다만 "최종 책임"이 다릅니다. IC는 기술적 올바름과 확장성을, Manager는 팀의 지속 가능한 실행력을 최종 책임집니다.

커리어 전략 측면에서는 한 번 선택했다고 영구 고정할 필요가 없습니다. 일정 기간 Manager 역할을 하다가 IC로 복귀하거나, 반대로 IC에서 Manager로 이동하는 사례도 많습니다. 중요한 것은 현재 조직에서 어떤 문제를 풀 때 본인의 강점이 가장 크게 증폭되는지 파악하는 일입니다.

정리하면, IC Staff와 Manager는 서로 다른 리더십 언어를 가집니다. 하나는 기술 시스템을 통해 영향력을 만들고, 다른 하나는 사람과 운영 시스템을 통해 영향력을 만듭니다. 도메인 전문성이 쌓일수록 이 선택은 더 전략적으로 해야 하며, 자신의 강점이 어디서 가장 크게 확장되는지 기준을 분명히 두어야 합니다.


## 실행용 점검 프레임: 09-domain-expertise.md 편

이 섹션은 앞에서 다룬 내용을 실제 행동으로 옮기기 위한 점검 프레임입니다. 많은 독자가 글을 읽을 때는 이해했다고 느끼지만, 일주일 뒤 실행 단계에서 다시 막히는 이유는 "다음 행동"이 문장으로 고정되어 있지 않기 때문입니다. 그래서 아래 프레임은 학습, 실습, 회고를 연결하는 최소 단위로 구성했습니다.

첫째, 이번 주 목표를 한 문장으로 적습니다. 목표는 도구가 아니라 문제 기준으로 써야 합니다. 예를 들어 "SQL 윈도우 함수 공부"보다 "사용자별 누적 결제를 계산해 이탈 신호를 찾는다"처럼 적는 편이 실행력이 높습니다. 같은 시간이라도 문제 중심 목표가 결과 해석까지 자연스럽게 이어집니다.

둘째, 입력과 출력을 명확히 구분합니다. 입력은 데이터, 로그, 문서, 인터뷰 메모처럼 내가 활용할 재료입니다. 출력은 쿼리, 분석 노트, 비교표, 의사결정 문서처럼 다른 사람이 검토 가능한 결과물입니다. 입력을 많이 모으는 것과 출력을 만드는 것은 다르며, 성장 속도는 출력 빈도에 더 강하게 좌우됩니다.

셋째, 판단 기준을 미리 적습니다. 많은 작업이 끝에서 흔들리는 이유는 성공 기준을 사전에 합의하지 않았기 때문입니다. "정확한가"라는 문장 대신 "분모 정의가 명확한가", "누락 세그먼트가 없는가", "후속 액션이 있는가"처럼 점검 문항으로 바꾸면 품질이 안정됩니다.

넷째, 협업 단위를 작게 쪼개 공유합니다. 완성본만 보여 주려는 습관은 피드백 타이밍을 늦춥니다. 중간 결과를 1페이지 요약으로 공유하고, 가정과 리스크를 먼저 드러내면 이후 수정 비용이 크게 줄어듭니다. 데이터 직무는 혼자 완벽하게 만든 뒤 공개하는 방식보다, 가정 기반으로 빠르게 정렬하는 방식이 장기적으로 더 강합니다.

다섯째, 회고는 감상이 아니라 의사결정 기록으로 남깁니다. "어려웠다"로 끝내지 말고 무엇이 병목이었는지, 다음에는 어떤 순서로 접근할지, 어떤 지표를 추가로 볼지를 적어야 합니다. 같은 유형의 문제를 다시 만났을 때 과거 기록이 바로 템플릿이 되면 성장 속도는 가속됩니다.

아래 표는 매주 반복 가능한 최소 운영 템플릿입니다.

| 항목 | 작성 질문 | 예시 출력 |
| --- | --- | --- |
| 목표 | 이번 주에 어떤 문제를 풀 것인가 | 문제 정의 3줄 |
| 입력 | 어떤 데이터/자료를 사용할 것인가 | 데이터 출처 목록 |
| 방법 | 어떤 접근 순서로 진행할 것인가 | 작업 단계 체크리스트 |
| 검증 | 무엇을 성공 기준으로 볼 것인가 | 품질 점검 표 |
| 결과 | 무엇이 바뀌었는가 | 한 문장 권고안 |
| 회고 | 다음 주에 무엇을 바꿀 것인가 | 개선 액션 3개 |

이 템플릿의 핵심은 완벽함이 아니라 반복 가능성입니다. 한 번에 거대한 문서를 만들 필요는 없습니다. 중요한 것은 매주 같은 구조로 남겨서 비교 가능성을 확보하는 일입니다. 비교가 가능해지면 성장도 측정 가능해지고, 측정 가능해지면 커리어 결정의 품질이 올라갑니다.

마지막으로, 실행 프레임은 개인 메모로 끝내지 말고 가능한 범위에서 팀 공유 자산으로 전환하는 것이 좋습니다. 같은 질문이 반복되는 조직일수록 표준 템플릿의 가치가 큽니다. 템플릿 하나를 잘 만드는 사람은 단순히 일을 수행하는 사람이 아니라, 팀이 더 빠르고 정확하게 일하도록 환경을 설계하는 사람으로 인식됩니다.


## 월간 점검 질문 세트

아래 질문은 월말 회고 때 그대로 사용할 수 있는 점검 세트입니다. 첫 번째 질문은 "이번 달에 내가 해결한 문제는 무엇이며, 그 문제는 누구의 의사결정에 영향을 주었는가"입니다. 두 번째 질문은 "내 결과물의 재현 가능성을 다른 동료가 확인할 수 있는가"입니다. 세 번째 질문은 "결과를 한 문장 권고안으로 요약했는가"입니다. 네 번째 질문은 "가정이 틀렸을 때 대안 경로를 제시했는가"입니다. 다섯 번째 질문은 "다음 달에는 무엇을 하지 않을 것인가"입니다.

이 다섯 질문을 매달 반복하면 학습과 실행이 분리되지 않습니다. 특히 마지막 질문은 과부하를 줄이는 데 효과적입니다. 커리어 성장은 더 많은 일을 하는 것만으로 만들어지지 않습니다. 우선순위를 분명히 하고, 덜 중요한 일을 의도적으로 줄일 때 더 빠르게 만들어집니다.

| 질문 | Pass 기준 |
| --- | --- |
| 문제 정의가 명확한가 | 이해관계자가 같은 문장으로 설명 가능 |
| 결과가 재현 가능한가 | 실행 명령과 데이터 기준이 문서화됨 |
| 해석이 행동으로 연결되는가 | 후속 액션이 1개 이상 명시됨 |
| 리스크를 선제적으로 다뤘는가 | 가정과 한계를 함께 기술함 |
| 다음 달 우선순위가 보이는가 | 할 일/하지 않을 일 목록이 분리됨 |


## 보강 메모: 실전 적용 예시

실전에서는 한 가지 프레임만으로 모든 상황을 풀 수 없습니다. 그래서 동일한 문제를 최소 두 가지 관점으로 다시 써 보는 연습이 필요합니다. 예를 들어 케이스 문제라면 지표 관점과 사용자 여정 관점을 각각 분리해 적어 보고, 온보딩 문제라면 개인 생산성과 팀 신뢰 형성 관점을 분리해 점검합니다. 도메인 학습 문제라면 용어 정확성과 의사결정 연결성을 분리해 체크합니다.

이 보강 연습의 목적은 정답을 늘리는 일이 아니라 시야를 넓히는 일입니다. 같은 데이터라도 질문이 달라지면 결론이 달라질 수 있기 때문입니다. 따라서 최종 결론을 내리기 전에 "내가 놓친 관점이 있는가"를 한 번 더 점검하는 습관이 중요합니다.

## 도메인 학습 프로젝트 예시: 4주 집중 플랜

도메인 전문성은 추상적으로 "공부해야지"라고 생각하면 잘 쌓이지 않습니다. 짧은 기간에 결과물을 남기는 프로젝트형 학습이 훨씬 효과적입니다. 아래는 4주 집중 플랜 예시입니다.

| 주차 | 목표 | 산출물 |
| --- | --- | --- |
| 1주차 | 핵심 용어 30개 정리 | 도메인 용어집 v1 |
| 2주차 | KPI 5개 정의와 계산식 파악 | KPI 정의 문서 |
| 3주차 | 현업 인터뷰 또는 shadowing 수행 | 관찰 노트와 질문 리스트 |
| 4주차 | 숫자 해석 리포트 작성 | "지표 변화와 비즈니스 의미" 1페이지 |

이 플랜의 핵심은 도메인 지식을 즉시 해석 능력으로 연결하는 것입니다. 마지막 주 리포트에서 반드시 "무엇이 바뀌었고, 어떤 의사결정을 제안하는가"를 포함하면 기술 설명에서 비즈니스 언어로 전환하는 연습이 됩니다. 이 전환 능력이 커리어 중반 이후 차이를 크게 만듭니다.


## 도메인 전문성 설계: 산업 지식을 기술 역량과 연결하기

데이터 직무의 중반 이후 성장은 도구 숙련도만으로 설명되지 않습니다. 같은 SQL 실력이라도 도메인 질문을 더 잘 정의하는 사람이 더 큰 영향력을 냅니다.

| 도메인 | 반드시 이해할 지표 | 자주 나오는 오해 | 학습 방법 |
| --- | --- | --- | --- |
| 커머스 | 전환율, 객단가, 재구매율 | 매출 증가=건강한 성장 | 퍼널과 코호트 동시 분석 |
| 핀테크 | 승인율, 연체율, LTV | 단기 지표만 최적화 | 리스크 지표와 수익성 동시 추적 |
| SaaS | 활성 사용자, 잔존율, NRR | 가입자 수만 강조 | 사용량 기반 세그먼트 분석 |

도메인 전문성은 자격증으로 완성되지 않습니다. 실제 의사결정 문서를 읽고, 지표가 어떻게 행동으로 바뀌는지 추적할 때 빠르게 쌓입니다.

## 전문가형 커뮤니케이션 템플릿

```text
- 관찰: 어떤 변화가 발생했는가
- 해석: 왜 그런 변화가 생겼다고 보는가
- 한계: 어떤 가정이 들어갔는가
- 제안: 지금 당장 할 액션 2개는 무엇인가
```

이 템플릿을 회의 메모와 리포트에 반복 적용하면, 도메인 지식이 "아는 수준"에서 "영향을 주는 수준"으로 전환됩니다.


## 실전 부록: 지원 문서와 면접 답변을 한 세트로 준비하는 방법

글을 읽고 이해한 내용을 실제 커리어 행동으로 바꾸려면, 이력서 문장·포트폴리오 증거·면접 답변이 서로 같은 문제를 가리키도록 정렬해야 합니다. 아래 구성은 데이터 직무 입문자와 주니어가 가장 빠르게 효과를 보는 준비 방식입니다.

### 이력서 불릿 템플릿

```text
- [문제] 어떤 비즈니스 또는 운영 문제를 맡았는가
- [행동] 어떤 데이터 접근(지표/실험/파이프라인/모델)을 적용했는가
- [결과] 무엇이 얼마나 개선되었는가
- [의미] 그 결과가 팀 의사결정에 어떤 영향을 주었는가
```

예시 문장: "결제 퍼널 이탈 구간을 코호트 기준으로 분해해 주요 병목을 식별했고, 온보딩 실험 우선순위를 재정렬해 4주 내 전환율을 개선했습니다." 핵심은 기술 이름보다 책임과 결과를 문장으로 남기는 일입니다.

### 포트폴리오 프로젝트 선정 체크

| 점검 항목 | Pass 기준 |
| --- | --- |
| 문제 명확성 | 한 문장으로 문제와 영향 범위를 설명 가능 |
| 데이터 현실성 | 공개 데이터라도 결측/편향/한계를 문서화 |
| 재현 가능성 | 실행 절차와 검증 기준을 README에 명시 |
| 의사결정 연결 | 결과가 다음 액션으로 이어짐 |

이 표를 기준으로 프로젝트를 고르면 "보여주기용 분석"을 줄이고 "실무형 문제 해결"을 남길 수 있습니다.

### 인터뷰 Q&A 3문장 프레임

1. 문제를 어떻게 정의했는지 말합니다.
2. 어떤 접근을 선택했고 왜 그렇게 판단했는지 말합니다.
3. 결과와 한계를 함께 설명하고 다음 액션을 제시합니다.

이 세 문장으로 답변하면 면접관이 가장 궁금해하는 판단 과정이 빠르게 드러납니다. 특히 한계를 먼저 말하는 태도는 신뢰를 높입니다.

### 보상 협상 기본 스크립트

```text
- 제안 감사합니다. 기대 역할과 성공 기준을 먼저 정렬하고 싶습니다.
- 제가 제공할 수 있는 기여 범위와 최근 증거를 공유드리겠습니다.
- 시장 밴드와 총보상 구조를 기준으로 조정 가능한 항목을 논의하고 싶습니다.
```

협상은 대립이 아니라 역할-보상 정렬입니다. 역할 범위를 수치와 산출물로 제시하면 대화가 감정에서 사실로 이동합니다.

### 역량 점검 매트릭스

| 영역 | 현재 수준(1~5) | 목표 수준(1~5) | 다음 2주 액션 |
| --- | --- | --- | --- |
| 문제 정의 |  |  |  |
| 데이터 처리 |  |  |  |
| 검증 설계 |  |  |  |
| 전달력 |  |  |  |
| 협업 운영 |  |  |  |

월 1회 이 표를 채우면 성장 체감이 감정이 아니라 기록으로 남습니다. 커리어 의사결정의 품질은 결국 기록의 품질에서 올라갑니다.


## 추가 점검 메모

도메인 전문성은 기술 외우기로 생기지 않습니다. 지표가 실제 의사결정과 연결되는 순간을 반복적으로 추적할 때 비로소 같은 데이터에서도 더 좋은 질문을 만들 수 있습니다.
