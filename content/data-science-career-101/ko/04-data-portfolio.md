# Data Science Career 101 (4/10): 데이터 포트폴리오

데이터 포트폴리오를 준비할 때 가장 흔한 오해는 “코드를 올리면 곧 실력 증명이 된다”는 생각입니다. 하지만 채용 관점에서 더 중요한 것은 코드량보다 문제를 어떻게 정의했고, 어떤 판단을 했고, 결과를 어떻게 설명했는지입니다.

특히 데이터 프로젝트는 노트북 몇 개만 남겨 두면 맥락이 금방 사라집니다. 데이터 출처는 무엇인지, 비교 기준은 무엇인지, 재현은 가능한지, 결과가 왜 의미 있는지까지 같이 보여 줘야 포트폴리오가 단순 저장소가 아니라 실무 증거로 읽힙니다.

이 글은 Data Science Career 101 시리즈의 네 번째 글입니다.

![Data Science Career 101 4장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/data-science-career-101/04/04-01-concept-at-a-glance.ko.png)
*Data Science Career 101 4장 흐름 개요*
> 포트폴리오에서 가장 중요한 것은 무엇을 할 수 있는가가 아니라 무엇을 물어봐야 하고 어떻게 답해야 하는가입니다.

## 먼저 던지는 질문

- 데이터 포트폴리오를 어떤 프로젝트 조합으로 구성하면 좋은지 설명합니다?
- 코드만 있는 저장소가 왜 약하게 보이는지 짚습니다?
- README가 어떤 구조를 가져야 하는지 정리합니다?

## 이 글에서 배우는 내용

- 세 가지 프로젝트 구성
- README 템플릿
- 재현 가능성
- 시각화 원칙
- 문서화 방식

## 왜 중요한가

채용에서 기억에 남는 것은 숫자 자체보다 서사입니다. 무슨 문제를 풀었고, 어떤 판단을 했고, 어떤 결론에 도달했는지를 빠르게 전달할 수 있어야 인터뷰로 이어집니다.

작은 프로젝트라도 질문, 데이터, 접근 방식, 결론, 재현 경로가 분명하면 훨씬 신뢰도가 높아집니다. 반대로 모델 성능 수치만 강조하면 인터뷰어는 “이 사람이 실제로 문제를 이해했는가?”를 판단하기 어려워집니다.

## 한눈에 보는 개념

좋은 포트폴리오는 이 흐름을 빠짐없이 보여 줍니다. 질문이 없으면 맥락이 없고, 결과만 있으면 신뢰가 약하며, 재현 방법이 없으면 협업 가능성이 잘 드러나지 않습니다.

## 핵심 용어

- **portfolio**: 가장 좋은 작업을 선별해 보여 주는 묶음입니다.
- **reproducible**: 다른 사람도 같은 결과를 다시 실행할 수 있는 상태입니다.
- **storytelling**: 문제에서 결론까지의 경로를 설득력 있게 전달하는 방식입니다.
- **README**: 저장소를 처음 열었을 때 가장 먼저 읽는 문서입니다.
- **notebook**: 분석 과정을 순서대로 기록한 작업 노트입니다.

## 전후 비교

**Before**: "모델 코드만 GitHub에 올리면 포트폴리오가 되는 줄 알았다."

**After**: "문제와 결론, 재현 방법까지 함께 정리해야 한다는 점을 이해했다."

## 실습: 포트폴리오 구성

### 단계 1 — 세 가지 프로젝트
```text
- one analytics (dashboard)
- one model (classification or regression)
- one data engineering (pipeline)
```

세 프로젝트는 서로 다른 역량을 보여 줍니다. 분석 프로젝트는 질문과 해석, 모델 프로젝트는 평가와 검증, 데이터 엔지니어링 프로젝트는 흐름과 재현성을 보여 주기에 좋습니다.

### 단계 2 — README 템플릿
```markdown
# Title
## 문제
## 데이터
## 접근
## 결과
## 재현 방법
```

README는 저장소의 첫인상입니다. 무엇을 만들었는지보다 왜 만들었는지, 무엇을 배웠는지를 먼저 보여 주는 편이 더 강합니다.

### 단계 3 — 재현 가능한 환경
```bash
uv pip install -r requirements.txt
make data
make run
```

실행 방법이 명확하면 프로젝트의 신뢰도가 크게 올라갑니다. 재현 가능성은 실무 감각을 드러내는 가장 값싼 신호 중 하나입니다.

### 단계 4 — Visualization

```text
- one key chart
- one comparison table
- one one-line conclusion
```

시각화는 많다고 좋은 것이 아닙니다. 핵심 차트 하나, 비교 표 하나, 한 줄 결론 하나가 오히려 더 오래 남습니다.

### 단계 5 — Documentation

```text
- enough markdown cells in the notebook
- decision notes
```

문서화는 단순 장식이 아닙니다. 어떤 선택을 왜 했는지 남겨 두면 협업 가능성과 사고 과정이 함께 드러납니다.

## 이 예시에서 먼저 봐야 할 점

- 재현 가능성은 신뢰를 만듭니다.
- 서사는 기억을 만듭니다.
- 결론 한 줄이 인상을 남깁니다.

입문자 포트폴리오일수록 화려함보다 완결성이 중요합니다. 작은 프로젝트라도 문제 정의, 접근 방식, 결과, 재현 방법이 모두 정리되어 있으면 훨씬 강한 인상을 줍니다.

## 자주 하는 실수 5가지

1. **문제 설명 없이 모델만 올리는 실수**
2. **데이터 출처를 불분명하게 두는 실수**
3. **재현할 수 없게 만드는 실수**
4. **README를 비워 두는 실수**
5. **시각화를 과하게 넣는 실수**

## 실무에서는 이렇게 나타납니다

면접관은 보통 몇 분 안에 프로젝트의 문제와 결론을 훑습니다. 이때 README 첫 화면, 핵심 시각화 한두 장, 결론 한 줄이 가장 큰 역할을 합니다. 복잡한 프로젝트보다 빠르게 이해되는 프로젝트가 더 강할 때가 많습니다.

## 시니어는 이렇게 생각합니다

- 재현 가능성이 먼저입니다.
- 문제를 먼저 정의합니다.
- 결론을 한 줄로 남깁니다.
- 서사가 곧 증거입니다.
- 링크 하나하나가 내 명함입니다.

## 체크리스트

- [ ] 다섯 섹션 README 구조를 만들었다.
- [ ] 재현 명령을 적어 두었다.
- [ ] 한 문장 결론을 남겼다.

## 정리 및 다음 단계

좋은 데이터 포트폴리오는 코드보다 문제 정의와 결론 전달이 더 또렷합니다. 분석, 모델, 데이터 엔지니어링 프로젝트를 균형 있게 고르고 README와 재현 방법까지 갖추면 입문자 포트폴리오의 설득력이 크게 올라갑니다.

다음 글에서는 데이터 직무 면접에서 빠지지 않는 SQL과 분석 인터뷰를 다루겠습니다.

## 데이터 직무 면접 유형별 준비 전략

포트폴리오를 잘 준비해도 면접 유형별로 답변 방식이 달라지지 않으면 성과가 제한됩니다. 데이터 직무 면접은 보통 SQL/분석, ML 이론, 케이스 인터뷰, 협업/행동 면접으로 구성되며, 각 면접은 서로 다른 신호를 확인합니다. 아래 표는 유형별 평가 포인트와 준비 전략을 정리한 실무용 체크리스트입니다.

| 면접 유형 | 면접관이 보는 핵심 | 준비 전략 | 답변 구조 |
| --- | --- | --- | --- |
| SQL/분석 | 질문 분해력, 정확한 집계, 해석력 | CTE 기반 풀이 연습, NULL/시간대 체크 | 문제 정의 -> 쿼리 -> 검증 -> 해석 |
| ML 이론/적용 | 모델 선택 근거, 지표 이해, 리스크 인식 | 알고리즘 암기보다 비교표 작성 | 문제 유형 -> 후보 모델 -> 지표 -> 한계 |
| 케이스 | 구조화 사고, 지표 설계, 의사결정 | 명확화 질문 템플릿 준비 | 맥락 확인 -> 지표 -> 가설 -> 데이터 -> 액션 |
| 행동/협업 | 커뮤니케이션, 갈등 해결, 책임감 | STAR 사례 6~8개 준비 | Situation -> Task -> Action -> Result |

면접 준비 시간 배분도 중요합니다. 입문자의 경우 SQL/분석 40%, 케이스 25%, 행동 20%, ML 15%처럼 기본기를 우선하는 배분이 안정적입니다. 지원 직무가 MLE라면 ML과 시스템 질문 비중을 늘리되, SQL과 행동 면접을 줄이지 않는 것이 좋습니다. 실제 불합격 사유는 고급 기술 부족보다 기본 커뮤니케이션과 문제 정의 부족에서 더 자주 발생합니다.

면접 질문에 대한 답변 길이도 조절해야 합니다. SQL 문제는 논리적이고 짧게, 케이스는 구조적으로, 행동 질문은 구체적 사건 중심으로 답하는 편이 효과적입니다. 아래는 60초/120초 답변 템플릿 예시입니다.

| 답변 길이 | 권장 사용 상황 | 구성 |
| --- | --- | --- |
| 60초 | 사실 확인 질문, 개념 질문 | 정의 1문장 + 예시 1개 + 주의점 1개 |
| 120초 | 케이스/경험 질문 | 맥락 + 행동 + 결과 + 회고 |

또한 면접 전날 점검은 "지식"보다 "실수 방지" 관점으로 하는 편이 좋습니다. 예를 들어 SQL에서는 join key 확인, 집계 단위 확인, 분모 정의 확인을 체크하십시오. 케이스에서는 명확화 질문을 최소 2개 이상 먼저 던지도록 루틴화하십시오. 행동 면접에서는 결과 수치를 준비하되 과장하지 말아야 신뢰가 유지됩니다.

아래는 실전에서 자주 쓰는 면접 준비표입니다.

| 준비 항목 | 최소 목표 | 권장 목표 |
| --- | --- | --- |
| SQL 문제 | 20문제 풀이 | 40문제 + 오답 노트 |
| 케이스 프레임 | 5개 암기 | 10개 상황별 변형 |
| STAR 사례 | 4개 준비 | 8개 준비(실패 사례 포함) |
| 포트폴리오 발표 | 1개 5분 발표 | 2개 5분/10분 버전 |

면접은 지식을 보여 주는 시험이면서 동시에 함께 일할 사람인지 판단하는 과정입니다. 따라서 정답만 맞추는 태도보다 사고 과정을 투명하게 보여 주는 태도가 중요합니다. "제가 바로 정답은 모르지만, 이런 가정으로 접근하겠습니다"라는 문장은 약점 고백이 아니라 문제 해결 능력의 신호입니다. 면접 유형별 전략을 분리해서 준비하면 포트폴리오의 강점도 훨씬 또렷하게 전달됩니다.

## 실행용 점검 프레임: 04-data-portfolio.md 편

이 섹션은 앞에서 다룬 내용을 실제 행동으로 옮기기 위한 점검 프레임입니다. 많은 독자가 글을 읽을 때는 이해했다고 느끼지만, 일주일 뒤 실행 단계에서 다시 막히는 이유는 "다음 행동"이 문장으로 고정되어 있지 않기 때문입니다. 그래서 아래 프레임은 학습, 실습, 회고를 연결하는 최소 단위로 구성했습니다.

첫째, 이번 주 목표를 한 문장으로 적습니다. 목표는 도구가 아니라 문제 기준으로 써야 합니다. 예를 들어 "SQL 윈도우 함수 공부"보다 "사용자별 누적 결제를 계산해 이탈 신호를 찾는다"처럼 적는 편이 실행력이 높습니다. 같은 시간이라도 문제 중심 목표가 결과 해석까지 자연스럽게 이어집니다.

둘째, 입력과 출력을 명확히 구분합니다. 입력은 데이터, 로그, 문서, 인터뷰 메모처럼 내가 활용할 재료입니다. 출력은 쿼리, 분석 노트, 비교표, 의사결정 문서처럼 다른 사람이 검토 가능한 결과물입니다. 입력을 많이 모으는 것과 출력을 만드는 것은 다르며, 성장 속도는 출력 빈도에 더 강하게 좌우됩니다.

셋째, 판단 기준을 미리 적습니다. 많은 작업이 끝에서 흔들리는 이유는 성공 기준을 사전에 합의하지 않았기 때문입니다. "정확한가"라는 문장 대신 "분모 정의가 명확한가", "누락 세그먼트가 없는가", "후속 액션이 있는가"처럼 점검 문항으로 바꾸면 품질이 안정됩니다.

넷째, 협업 단위를 작게 쪼개 공유합니다. 완성본만 보여 주려는 습관은 피드백 타이밍을 늦춥니다. 중간 결과를 1페이지 요약으로 공유하고, 가정과 리스크를 먼저 드러내면 이후 수정 비용이 크게 줄어듭니다. 데이터 직무는 혼자 완벽하게 만든 뒤 공개하는 방식보다, 가정 기반으로 빠르게 정렬하는 방식이 장기적으로 더 강합니다.

다섯째, 회고는 감상이 아니라 의사결정 기록으로 남깁니다. "어려웠다"로 끝내지 말고 무엇이 병목이었는지, 다음에는 어떤 순서로 접근할지, 어떤 지표를 추가로 볼지를 적어야 합니다. 같은 유형의 문제를 다시 만났을 때 과거 기록이 바로 템플릿이 되면 성장 속도는 가속됩니다.

아래 표는 매주 반복 가능한 최소 운영 템플릿입니다.

| 항목 | 작성 질문 | 예시 출력 |
| --- | --- | --- |
| 목표 | 이번 주에 어떤 문제를 풀 것인가 | 문제 정의 3줄 |
| 입력 | 어떤 데이터/자료를 사용할 것인가 | 데이터 출처 목록 |
| 방법 | 어떤 접근 순서로 진행할 것인가 | 작업 단계 체크리스트 |
| 검증 | 무엇을 성공 기준으로 볼 것인가 | 품질 점검 표 |
| 결과 | 무엇이 바뀌었는가 | 한 문장 권고안 |
| 회고 | 다음 주에 무엇을 바꿀 것인가 | 개선 액션 3개 |

이 템플릿의 핵심은 완벽함이 아니라 반복 가능성입니다. 한 번에 거대한 문서를 만들 필요는 없습니다. 중요한 것은 매주 같은 구조로 남겨서 비교 가능성을 확보하는 일입니다. 비교가 가능해지면 성장도 측정 가능해지고, 측정 가능해지면 커리어 결정의 품질이 올라갑니다.

마지막으로, 실행 프레임은 개인 메모로 끝내지 말고 가능한 범위에서 팀 공유 자산으로 전환하는 것이 좋습니다. 같은 질문이 반복되는 조직일수록 표준 템플릿의 가치가 큽니다. 템플릿 하나를 잘 만드는 사람은 단순히 일을 수행하는 사람이 아니라, 팀이 더 빠르고 정확하게 일하도록 환경을 설계하는 사람으로 인식됩니다.

## 월간 점검 질문 세트

아래 질문은 월말 회고 때 그대로 사용할 수 있는 점검 세트입니다. 첫 번째 질문은 "이번 달에 내가 해결한 문제는 무엇이며, 그 문제는 누구의 의사결정에 영향을 주었는가"입니다. 두 번째 질문은 "내 결과물의 재현 가능성을 다른 동료가 확인할 수 있는가"입니다. 세 번째 질문은 "결과를 한 문장 권고안으로 요약했는가"입니다. 네 번째 질문은 "가정이 틀렸을 때 대안 경로를 제시했는가"입니다. 다섯 번째 질문은 "다음 달에는 무엇을 하지 않을 것인가"입니다.

이 다섯 질문을 매달 반복하면 학습과 실행이 분리되지 않습니다. 특히 마지막 질문은 과부하를 줄이는 데 효과적입니다. 커리어 성장은 더 많은 일을 하는 것만으로 만들어지지 않습니다. 우선순위를 분명히 하고, 덜 중요한 일을 의도적으로 줄일 때 더 빠르게 만들어집니다.

| 질문 | Pass 기준 |
| --- | --- |
| 문제 정의가 명확한가 | 이해관계자가 같은 문장으로 설명 가능 |
| 결과가 재현 가능한가 | 실행 명령과 데이터 기준이 문서화됨 |
| 해석이 행동으로 연결되는가 | 후속 액션이 1개 이상 명시됨 |
| 리스크를 선제적으로 다뤘는가 | 가정과 한계를 함께 기술함 |
| 다음 달 우선순위가 보이는가 | 할 일/하지 않을 일 목록이 분리됨 |

## 포트폴리오 프로젝트 예시: 채용 공고와 1:1 매핑

좋은 포트폴리오는 "멋진 프로젝트"가 아니라 "지원 직무와 직접 연결되는 프로젝트"입니다. 아래는 채용 공고 요구사항을 포트폴리오 구성으로 매핑한 예시입니다.

| 공고 요구사항 | 포트폴리오 증거 | 제출 형태 |
| --- | --- | --- |
| 지표 분석 및 대시보드 운영 | 퍼널/코호트 분석 프로젝트 | README + 대시보드 스크린샷 + 해석 문서 |
| 실험 설계와 성과 검증 | A/B 테스트 분석 프로젝트 | 실험 가정/결과/한계 문서 |
| 데이터 파이프라인 개선 | ETL 자동화 프로젝트 | DAG 설명 + 품질 체크 규칙 + 장애 대응 노트 |

면접에서는 "무엇을 만들었다"보다 "왜 이 프로젝트를 선택했고 채용 요구와 어떻게 맞췄는가"를 설명하면 설득력이 크게 올라갑니다. 따라서 저장소 첫 화면에 대상 직무, 핵심 역량, 확인 가능한 링크를 명시해 면접관이 3분 안에 강점을 파악할 수 있게 구성하는 것이 좋습니다.

## 포트폴리오 프로젝트 예시 3종: 채용 공고와 직접 연결하기

포트폴리오는 "잘 만든 분석"보다 "지원 직무와 맞는 문제 해결"을 보여줘야 합니다. 아래 세 프로젝트는 데이터 직무 입문자가 실제로 제출 가능한 구조로 정리한 예시입니다.

### 프로젝트 A: 이탈 분석 대시보드
- 문제: 신규 사용자 2주차 이탈 급증 원인 파악
- 데이터: 이벤트 로그, 결제 테이블, 유입 채널
- 접근: 퍼널 분해, 세그먼트 비교, 코호트 분석
- 결과: 온보딩 단계 병목 확인, 개선 실험안 제안

### 프로젝트 B: 수요 예측 미니 모델
- 문제: 주간 재고 소진율 예측
- 데이터: 판매 이력, 프로모션, 시즌 변수
- 접근: 베이스라인과 트리 모델 비교, 오차 분석
- 결과: 예측 오차 구간 명시, 운영 적용 조건 제시

### 프로젝트 C: 데이터 품질 모니터링
- 문제: 대시보드 수치 불일치 반복
- 데이터: 원천-집계 테이블 매핑 정보
- 접근: 품질 규칙 정의, 누락/중복 탐지 자동화
- 결과: 오류 탐지 시간 단축, 재실행 절차 표준화

이 세 가지를 모두 넣을 필요는 없습니다. 목표 직무에 맞춰 2개만 깊게 만드는 편이 효과적입니다.

## 포트폴리오 README 템플릿

```text
1) Problem: 어떤 의사결정을 돕기 위한 문제인가
2) Data: 출처, 기간, 주요 컬럼, 한계
3) Method: 접근 순서, 가정, 대안 비교
4) Result: 수치 결과와 행동 제안
5) Reproduce: 실행 명령, 환경, 검증 절차
```

면접관은 코드 양보다 사고 과정을 봅니다. 그래서 README는 결과 요약문이 아니라 "판단 기록"이어야 합니다.

## 실전 부록: 지원 문서와 면접 답변을 한 세트로 준비하는 방법

글을 읽고 이해한 내용을 실제 커리어 행동으로 바꾸려면, 이력서 문장·포트폴리오 증거·면접 답변이 서로 같은 문제를 가리키도록 정렬해야 합니다. 아래 구성은 데이터 직무 입문자와 주니어가 가장 빠르게 효과를 보는 준비 방식입니다.

### 이력서 불릿 템플릿

```text
- [문제] 어떤 비즈니스 또는 운영 문제를 맡았는가
- [행동] 어떤 데이터 접근(지표/실험/파이프라인/모델)을 적용했는가
- [결과] 무엇이 얼마나 개선되었는가
- [의미] 그 결과가 팀 의사결정에 어떤 영향을 주었는가
```

예시 문장: "결제 퍼널 이탈 구간을 코호트 기준으로 분해해 주요 병목을 식별했고, 온보딩 실험 우선순위를 재정렬해 4주 내 전환율을 개선했습니다." 핵심은 기술 이름보다 책임과 결과를 문장으로 남기는 일입니다.

### 포트폴리오 프로젝트 선정 체크

| 점검 항목 | Pass 기준 |
| --- | --- |
| 문제 명확성 | 한 문장으로 문제와 영향 범위를 설명 가능 |
| 데이터 현실성 | 공개 데이터라도 결측/편향/한계를 문서화 |
| 재현 가능성 | 실행 절차와 검증 기준을 README에 명시 |
| 의사결정 연결 | 결과가 다음 액션으로 이어짐 |

이 표를 기준으로 프로젝트를 고르면 "보여주기용 분석"을 줄이고 "실무형 문제 해결"을 남길 수 있습니다.

### 인터뷰 Q&A 3문장 프레임

1. 문제를 어떻게 정의했는지 말합니다.
2. 어떤 접근을 선택했고 왜 그렇게 판단했는지 말합니다.
3. 결과와 한계를 함께 설명하고 다음 액션을 제시합니다.

이 세 문장으로 답변하면 면접관이 가장 궁금해하는 판단 과정이 빠르게 드러납니다. 특히 한계를 먼저 말하는 태도는 신뢰를 높입니다.

### 보상 협상 기본 스크립트

```text
- 제안 감사합니다. 기대 역할과 성공 기준을 먼저 정렬하고 싶습니다.
- 제가 제공할 수 있는 기여 범위와 최근 증거를 공유드리겠습니다.
- 시장 밴드와 총보상 구조를 기준으로 조정 가능한 항목을 논의하고 싶습니다.
```

협상은 대립이 아니라 역할-보상 정렬입니다. 역할 범위를 수치와 산출물로 제시하면 대화가 감정에서 사실로 이동합니다.

### 역량 점검 매트릭스

| 영역 | 현재 수준(1~5) | 목표 수준(1~5) | 다음 2주 액션 |
| --- | --- | --- | --- |
| 문제 정의 |  |  |  |
| 데이터 처리 |  |  |  |
| 검증 설계 |  |  |  |
| 전달력 |  |  |  |
| 협업 운영 |  |  |  |

월 1회 이 표를 채우면 성장 체감이 감정이 아니라 기록으로 남습니다. 커리어 의사결정의 품질은 결국 기록의 품질에서 올라갑니다.

## 추가 점검 메모

포트폴리오 품질 기준은 산출물 개수가 아니라 문제 정의의 선명도와 재현성입니다. 프로젝트를 하나 더 추가하기보다 기존 프로젝트의 가정, 실패 원인, 개선 반복을 정리하면 면접 신뢰도가 크게 높아집니다.
