# Data Science Career 101 (3/10): 학습 경로 설계

데이터 직무를 준비하는 입문자가 가장 자주 빠지는 함정은 열심히 공부하는데도 순서가 없다는 점입니다. 강의를 몇 개 듣고, 책을 몇 권 사고, Kaggle 노트북을 따라 해 봐도 어느 순간 “그래서 지금 나는 무엇을 쌓고 있는가?”라는 질문 앞에서 멈추게 됩니다.

좋은 학습 경로는 더 많은 자료를 소비하게 만드는 계획이 아니라, 다음 단계로 넘어가기 위한 기준을 만들어 주는 계획입니다. 무엇을 먼저 익히고, 어떤 결과물을 남기고, 어느 시점에 회고할지 정해 두어야 공부가 취미가 아니라 자산으로 남습니다.

이 글은 Data Science Career 101 시리즈의 세 번째 글입니다.

## 먼저 던지는 질문

- 데이터 직무 입문자는 무엇을 어떤 순서로 배워야 하는지 정리합니다?
- 12주 로드맵을 기초, 분석, 모델링 단계로 나누는 이유를 설명합니다?
- 매주 남기는 산출물이 왜 중요한지 짚습니다?

## 왜 중요한가

무작정 공부하면 의욕보다 먼저 방향을 잃습니다. 반대로 경로가 있으면 배운 내용이 다음 단계의 재료가 되고, 중간에 지치더라도 다시 돌아올 기준점이 남습니다.

특히 데이터 직무는 SQL, Python, 통계, 시각화, 실험, 모델링처럼 배워야 할 항목이 많아서 순서를 잘못 잡으면 체감 난도가 급격히 올라갑니다. 그래서 “무엇을 더 많이 하느냐”보다 “무엇을 어떤 순서로 쌓느냐”가 더 중요합니다.

## 한눈에 보는 개념

입문 단계에서는 이 순서가 중요합니다. 기초가 받쳐 줘야 분석이 흔들리지 않고, 분석 감각이 있어야 모델링 결과를 해석할 수 있으며, 마지막에는 제품과 사용자 맥락으로 연결해야 합니다.

## 핵심 용어

- **fundamentals**: Python, SQL, 통계 같은 핵심 기초입니다.
- **analytics**: 질문을 중심으로 데이터를 읽고 해석하는 단계입니다.
- **modeling**: 예측과 분류 같은 모델링 단계입니다.
- **product sense**: 결과가 사용자와 제품에 미치는 영향을 읽는 감각입니다.
- **retro**: 일정 주기로 학습을 돌아보는 회고입니다.

## 실습: 12주 학습 경로

### Step 1 — Fundamentals (Weeks 1-4)

```text
- Python syntax, pandas
- SQL JOIN, GROUP BY
- Statistics basics (mean, variance, p-value)
- Visualization (matplotlib)
```

처음 4주는 화려하지 않아도 괜찮습니다. 이 구간을 건너뛰면 뒤에서 더 큰 비용을 치르게 됩니다.

### Step 2 — Analytics (Weeks 5-8)

```text
- Data cleaning
- A/B test design
- One dashboard
- One case study
```

분석 단계에서는 질문을 구조화하고 결과를 설명하는 연습이 필요합니다. 지표와 해석을 같이 다뤄야 실전 감각이 붙습니다.

### Step 3 — Modeling (Weeks 9-12)

```text
- One regression, one classification
- scikit-learn pipeline
- Model evaluation metrics
- One mini project
```

모델링은 기초와 분석 위에 올라갑니다. 그래서 이 단계에서는 모델 자체보다 문제 정의와 평가 지표를 함께 보는 습관이 중요합니다.

### Step 4 — Weekly Artifact

```markdown
- README
- Data source
- Code
- Result
- Retro
```

주간 산출물은 “공부했다”가 아니라 “남겼다”는 증거입니다. 나중에 포트폴리오로 연결할 수 있다는 점에서도 중요합니다.

### Step 5 — Retro Template

```text
What went well / Improve / Next
```

회고는 의욕을 다지는 의식이 아니라 방향을 조정하는 장치입니다. 무엇이 잘됐고 무엇을 줄여야 하는지 짧게라도 남겨야 같은 실수를 반복하지 않습니다.

## 이 예시에서 먼저 봐야 할 점

- 주간 산출물은 성장의 증거입니다.
- 기초가 있어야 모델링도 흔들리지 않습니다.
- 회고가 학습 루프를 닫아 줍니다.

입문자가 자주 하는 실수는 모델부터 시작하는 일입니다. 하지만 실제로 더 자주 막히는 곳은 SQL, 데이터 정리, 기본 통계입니다. 기초를 건너뛰지 않는 편이 결국 더 빠릅니다.

## 자주 하는 실수 5가지

1. **책을 처음부터 끝까지 읽은 뒤 시작하려는 실수**
2. **모델링부터 먼저 시작하는 실수**
3. **산출물을 남기지 않는 실수**
4. **회고를 건너뛰는 실수**
5. **도구를 자주 바꾸는 실수**

## 실무에서는 이렇게 나타납니다

많은 부트캠프와 사내 교육도 대체로 이 12주 구조를 닮아 있습니다. 기초를 다지고, 분석 문제를 풀고, 마지막에 작은 모델링 프로젝트를 붙이는 흐름이 가장 재현 가능하기 때문입니다.

## 시니어는 이렇게 생각합니다

- 기초는 시간이 갈수록 복리처럼 쌓입니다.
- 매주 남는 증거가 있어야 성장도 설명할 수 있습니다.
- 회고가 다음 주의 방향을 정합니다.
- 프로젝트가 개별 기술을 하나로 묶어 줍니다.
- 지속 가능성이 곧 전략입니다.

## 정리 및 다음 단계

좋은 학습 계획은 더 많은 자료를 소비하는 계획이 아니라, 매주 작은 결과물을 만들며 다음 단계로 넘어가는 계획입니다. 기초, 분석, 모델링을 순서대로 밟고 매주 흔적을 남기면 공부가 자산으로 바뀝니다.

다음 글에서는 이렇게 쌓은 결과물을 어떻게 포트폴리오로 구성해야 하는지 살펴보겠습니다.


## 포트폴리오 프로젝트 유형별 예시 표와 선택 기준

학습 경로를 만들 때 가장 큰 난관은 "무엇을 만들면 좋은 포트폴리오가 되는가"입니다. 많은 입문자가 난이도 높은 프로젝트를 고르지만, 평가자는 화려함보다 완결성과 문제 정의의 명확성을 먼저 봅니다. 그래서 아래 표는 직무별로 바로 연결되는 프로젝트 유형을 정리한 실전 목록입니다.

| 프로젝트 유형 | 목표 직무 | 핵심 질문 | 권장 산출물 | 최소 검증 기준 |
| --- | --- | --- | --- | --- |
| 퍼널/코호트 분석 | Analyst | 전환이 어디서 떨어지는가 | SQL 쿼리, 대시보드, 해석 메모 | 지표 정의와 세그먼트 비교 포함 |
| A/B 테스트 분석 | Analyst/Scientist | 변화가 유의미한가 | 실험 설계 문서, 결과 해석 | 가정/한계/후속 액션 명시 |
| 수요 예측 모델 | Scientist | 다음 기간 값을 얼마나 잘 예측하는가 | 모델 코드, 피처 설명, 오류 분석 | 기준선 대비 개선 수치 제시 |
| 이탈 예측 모델 | Scientist/MLE | 어떤 사용자가 이탈 위험이 높은가 | 분류 모델, 임계값 전략 | precision-recall 트레이드오프 설명 |
| ELT 파이프라인 구축 | Engineer | 데이터 적재와 변환이 안정적인가 | DAG, 모델링 문서, 품질 체크 | 재실행 가능성과 장애 복구 절차 |
| 추천 모델 서빙 PoC | MLE | 예측을 서비스로 제공할 수 있는가 | 추론 API, 모니터링 지표 | 응답 시간, 오류율, 배포 절차 |

프로젝트 선택에서 자주 발생하는 오류는 "기술 트렌드"를 기준으로 주제를 고르는 일입니다. 트렌드를 완전히 무시할 필요는 없지만, 입문자 단계에서는 데이터 출처의 신뢰성, 재현 가능성, 질문의 구체성이 훨씬 중요합니다. 예를 들어 공개 데이터셋으로 만든 코호트 분석 프로젝트는 복잡한 딥러닝 프로젝트보다도 채용 관점에서 더 명확한 신호를 줄 수 있습니다.

다음은 한 프로젝트를 실무형 포트폴리오로 바꾸기 위한 구성 체크입니다.

| 항목 | 반드시 포함할 내용 | 흔한 누락 |
| --- | --- | --- |
| 문제 정의 | 왜 이 문제를 푸는지, 누구의 의사결정에 쓰이는지 | 문제 대신 기술 소개만 작성 |
| 데이터 설명 | 출처, 기간, 전처리 기준 | 결측/이상치 처리 기준 미기록 |
| 접근 방식 | 왜 이 방법을 선택했는지 | 대안 비교 없이 단일 방법만 기술 |
| 결과 해석 | 수치의 의미, 비즈니스 함의 | 성능 수치만 제시하고 해석 없음 |
| 재현 방법 | 실행 명령, 의존성, 폴더 구조 | 실행 불가능한 저장소 상태 |

프로젝트 난이도는 "새로운 알고리즘 수"가 아니라 "의사결정 연결도"로 판단하는 편이 좋습니다. 즉, 결과를 읽고 다음 행동을 제안할 수 있으면 좋은 프로젝트입니다. 반대로 모델 점수는 높아도 무엇을 바꿔야 하는지 말할 수 없으면 포트폴리오 힘이 약합니다.

12주 학습 경로와 연결하는 방법도 간단합니다. 1-4주는 데이터 정리와 SQL 분석형 프로젝트, 5-8주는 실험 또는 기본 모델 프로젝트, 9-12주는 재현성과 문서화 개선에 집중하십시오. 한 번에 세 개를 병렬로 하지 말고, 하나를 끝까지 닫는 경험을 먼저 만드는 것이 중요합니다. 완료의 경험이 쌓여야 다음 프로젝트 품질도 올라갑니다.

마지막으로 면접 대비 관점에서 프로젝트를 정리할 때는 "내가 한 일"과 "그 일이 만든 변화"를 분리해 말해야 합니다. "모델을 만들었습니다"가 아니라 "기준선 대비 오류를 12% 낮췄고, 어떤 세그먼트에서 성능이 약한지 확인해 후속 실험 계획을 제안했습니다"처럼 말해야 합니다. 프로젝트 유형을 잘 고르는 일은 결국 본인의 문제 해결 스타일을 명확히 보여 주는 작업입니다.


## 실행용 점검 프레임: 03-learning-path.md 편

이 섹션은 앞에서 다룬 내용을 실제 행동으로 옮기기 위한 점검 프레임입니다. 많은 독자가 글을 읽을 때는 이해했다고 느끼지만, 일주일 뒤 실행 단계에서 다시 막히는 이유는 "다음 행동"이 문장으로 고정되어 있지 않기 때문입니다. 그래서 아래 프레임은 학습, 실습, 회고를 연결하는 최소 단위로 구성했습니다.

첫째, 이번 주 목표를 한 문장으로 적습니다. 목표는 도구가 아니라 문제 기준으로 써야 합니다. 예를 들어 "SQL 윈도우 함수 공부"보다 "사용자별 누적 결제를 계산해 이탈 신호를 찾는다"처럼 적는 편이 실행력이 높습니다. 같은 시간이라도 문제 중심 목표가 결과 해석까지 자연스럽게 이어집니다.

둘째, 입력과 출력을 명확히 구분합니다. 입력은 데이터, 로그, 문서, 인터뷰 메모처럼 내가 활용할 재료입니다. 출력은 쿼리, 분석 노트, 비교표, 의사결정 문서처럼 다른 사람이 검토 가능한 결과물입니다. 입력을 많이 모으는 것과 출력을 만드는 것은 다르며, 성장 속도는 출력 빈도에 더 강하게 좌우됩니다.

셋째, 판단 기준을 미리 적습니다. 많은 작업이 끝에서 흔들리는 이유는 성공 기준을 사전에 합의하지 않았기 때문입니다. "정확한가"라는 문장 대신 "분모 정의가 명확한가", "누락 세그먼트가 없는가", "후속 액션이 있는가"처럼 점검 문항으로 바꾸면 품질이 안정됩니다.

넷째, 협업 단위를 작게 쪼개 공유합니다. 완성본만 보여 주려는 습관은 피드백 타이밍을 늦춥니다. 중간 결과를 1페이지 요약으로 공유하고, 가정과 리스크를 먼저 드러내면 이후 수정 비용이 크게 줄어듭니다. 데이터 직무는 혼자 완벽하게 만든 뒤 공개하는 방식보다, 가정 기반으로 빠르게 정렬하는 방식이 장기적으로 더 강합니다.

다섯째, 회고는 감상이 아니라 의사결정 기록으로 남깁니다. "어려웠다"로 끝내지 말고 무엇이 병목이었는지, 다음에는 어떤 순서로 접근할지, 어떤 지표를 추가로 볼지를 적어야 합니다. 같은 유형의 문제를 다시 만났을 때 과거 기록이 바로 템플릿이 되면 성장 속도는 가속됩니다.

아래 표는 매주 반복 가능한 최소 운영 템플릿입니다.

| 항목 | 작성 질문 | 예시 출력 |
| --- | --- | --- |
| 목표 | 이번 주에 어떤 문제를 풀 것인가 | 문제 정의 3줄 |
| 입력 | 어떤 데이터/자료를 사용할 것인가 | 데이터 출처 목록 |
| 방법 | 어떤 접근 순서로 진행할 것인가 | 작업 단계 체크리스트 |
| 검증 | 무엇을 성공 기준으로 볼 것인가 | 품질 점검 표 |
| 결과 | 무엇이 바뀌었는가 | 한 문장 권고안 |
| 회고 | 다음 주에 무엇을 바꿀 것인가 | 개선 액션 3개 |

이 템플릿의 핵심은 완벽함이 아니라 반복 가능성입니다. 한 번에 거대한 문서를 만들 필요는 없습니다. 중요한 것은 매주 같은 구조로 남겨서 비교 가능성을 확보하는 일입니다. 비교가 가능해지면 성장도 측정 가능해지고, 측정 가능해지면 커리어 결정의 품질이 올라갑니다.

마지막으로, 실행 프레임은 개인 메모로 끝내지 말고 가능한 범위에서 팀 공유 자산으로 전환하는 것이 좋습니다. 같은 질문이 반복되는 조직일수록 표준 템플릿의 가치가 큽니다. 템플릿 하나를 잘 만드는 사람은 단순히 일을 수행하는 사람이 아니라, 팀이 더 빠르고 정확하게 일하도록 환경을 설계하는 사람으로 인식됩니다.


## 월간 점검 질문 세트

아래 질문은 월말 회고 때 그대로 사용할 수 있는 점검 세트입니다. 첫 번째 질문은 "이번 달에 내가 해결한 문제는 무엇이며, 그 문제는 누구의 의사결정에 영향을 주었는가"입니다. 두 번째 질문은 "내 결과물의 재현 가능성을 다른 동료가 확인할 수 있는가"입니다. 세 번째 질문은 "결과를 한 문장 권고안으로 요약했는가"입니다. 네 번째 질문은 "가정이 틀렸을 때 대안 경로를 제시했는가"입니다. 다섯 번째 질문은 "다음 달에는 무엇을 하지 않을 것인가"입니다.

이 다섯 질문을 매달 반복하면 학습과 실행이 분리되지 않습니다. 특히 마지막 질문은 과부하를 줄이는 데 효과적입니다. 커리어 성장은 더 많은 일을 하는 것만으로 만들어지지 않습니다. 우선순위를 분명히 하고, 덜 중요한 일을 의도적으로 줄일 때 더 빠르게 만들어집니다.

| 질문 | Pass 기준 |
| --- | --- |
| 문제 정의가 명확한가 | 이해관계자가 같은 문장으로 설명 가능 |
| 결과가 재현 가능한가 | 실행 명령과 데이터 기준이 문서화됨 |
| 해석이 행동으로 연결되는가 | 후속 액션이 1개 이상 명시됨 |
| 리스크를 선제적으로 다뤘는가 | 가정과 한계를 함께 기술함 |
| 다음 달 우선순위가 보이는가 | 할 일/하지 않을 일 목록이 분리됨 |

## 스킬 매트릭스: 12주 이후 다음 분기 계획

12주를 마친 뒤에는 "무엇을 더 공부할까"보다 "어떤 스킬이 아직 약한가"를 먼저 진단해야 다음 분기가 효율적입니다. 아래 매트릭스는 입문자가 자주 사용하는 자기 진단 표입니다.

| 역량 영역 | 현재 수준(1-5) | 증거 산출물 | 다음 4주 액션 |
| --- | --- | --- | --- |
| SQL 문제 분해 |  | 쿼리 리뷰 노트 | 주 2문제 풀이 + 오답 패턴 기록 |
| 해석/스토리텔링 |  | 한 페이지 분석 메모 | 결과를 권고안 문장으로 요약하는 연습 |
| 모델 평가 |  | 모델 비교표 | 기준선 대비 개선 근거 작성 |
| 재현성/문서화 |  | README와 실행 스크립트 | 실행 실패 케이스까지 문서화 |

핵심은 점수 자체가 아니라 증거 기반 평가입니다. "잘하는 것 같다" 대신 실제 산출물로 수준을 판정해야 계획이 현실적으로 바뀝니다. 분기마다 같은 표를 업데이트하면 본인의 성장 궤적을 명확히 설명할 수 있어 면접과 성과 리뷰에서도 강한 자료가 됩니다.

## 처음 질문으로 돌아가기

- **데이터 직무 입문자는 무엇을 어떤 순서로 배워야 하는지 정리합니다?**
  - 매기 4주 구간(Python, SQL, 통계), 4주 단순(EDA, 대시보드, A/B 테스트), 4주 모델링(간단한 모델, 평가), 그리고 루프를 닫는 주간 회고가 핵심 약속되는 것입니다.
- **12주 로드맵을 기초, 분석, 모델링 단계로 나누는 이유를 설명합니다?**
  - 기초 역할을 떼줄 좋단 단계를 앞냥기면 동이, 단계를 덥나나달 때 단계는 문제를 분석시 만드기 받고 결과 파악도 달래봅니다.
- **매주 남기는 산출물이 왜 중요한지 짚습니다?**
  - 매단라 가는 비등자를 브랑하고 습격을 난단는 쨈, 저제 노출답단 매 단계에서 달래 단계로 돈아가기 날 아싶니다.
