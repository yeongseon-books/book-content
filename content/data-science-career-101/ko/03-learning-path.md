# 학습 경로 설계

데이터 커리어 입문자에게 가장 흔한 실패 패턴은 열심히 하지 않아서가 아니라, 순서 없이 공부해서 지치는 것입니다. 책은 몇 권 사 두고, 강의도 여기저기 담아 두고, SQL과 Python과 머신러닝과 통계를 동시에 건드리다 보면 어느 순간 “뭘 하고 있는지 모르겠다”는 느낌이 옵니다.

그래서 입문 단계에서는 속도보다 경로가 더 중요합니다. 기초를 먼저 잡고, 분석 감각을 붙이고, 그다음 모델링과 제품 감각으로 넘어가는 식의 순서가 있어야 학습이 쌓입니다. 특히 매주 손에 남는 산출물을 남기면, 막연한 공부가 아니라 앞으로 나아가고 있다는 증거를 확보할 수 있습니다.

## 이 글에서 다룰 문제

- 데이터 직무 입문자는 무엇을 어떤 순서로 배우는 편이 좋을까요?
- 무작정 공부하면 왜 쉽게 번아웃이 올까요?
- 기초, 분석, 모델링을 12주 안에 어떻게 나눌 수 있을까요?
- 매주 남기는 산출물은 왜 중요할까요?
- 회고를 학습 계획에 넣어야 하는 이유는 무엇일까요?

> 좋은 학습 경로는 더 많은 자료를 보는 계획이 아니라, 매주 작게 완성하면서 다음 단계로 넘어가는 계획입니다.

## 한눈에 보는 전체 흐름

```mermaid
flowchart LR
    F[기초] --> A[분석]
    A --> M[모델링]
    M --> P[제품 감각]
```

이 흐름은 단순하지만 꽤 실용적입니다. 기초가 없으면 분석이 흔들리고, 분석 감각이 없으면 모델링 결과를 제대로 해석하기 어렵고, 제품 감각이 없으면 만든 결과를 어디에 써야 할지 감이 오지 않습니다. 결국 입문자에게 필요한 것은 모든 것을 한 번에 배우는 일이 아니라, 다음 단계를 떠받칠 바닥을 순서대로 만드는 일입니다.

## 핵심 용어

- **fundamentals**: Python, SQL, 통계처럼 모든 단계의 바탕이 되는 기초입니다.
- **analytics**: 질문을 기준으로 데이터를 읽고 해석하는 단계입니다.
- **modeling**: 예측과 분류 같은 모델링을 다루는 단계입니다.
- **product sense**: 결과가 사용자와 제품에 어떤 영향을 주는지 읽는 감각입니다.
- **retro**: 일정 주기로 학습을 돌아보는 회고입니다.

## Before/After

**Before**: "책과 강의만 계속 늘어나고 실제로 끝나는 것은 없다."

**After**: "12주짜리 로드맵과 주간 산출물을 기준으로 공부를 이어 갈 수 있다."

## 실습: 12주 학습 경로 짜기

아래 계획은 완벽한 커리큘럼이라기보다, 초보자가 흔들리지 않고 앞으로 나아가기 위한 최소 구조입니다. 중요한 것은 매주 하나씩 남기고, 다음 주로 넘어갈 때 손에 쥔 결과물이 있어야 한다는 점입니다.

### Step 1 — Fundamentals (Weeks 1-4)

```text
- Python 문법, pandas
- SQL의 JOIN, GROUP BY
- 통계 기초 (평균, 분산, p-value)
- 시각화 (matplotlib)
```

### Step 2 — Analytics (Weeks 5-8)

```text
- 데이터 정제
- A/B 테스트 설계
- 대시보드 하나
- 케이스 스터디 하나
```

### Step 3 — Modeling (Weeks 9-12)

```text
- 회귀 하나, 분류 하나
- scikit-learn 파이프라인
- 모델 평가 지표
- 미니 프로젝트 하나
```

### Step 4 — Weekly Artifact

```markdown
- README
- 데이터 소스
- 코드
- 결과
- 회고
```

### Step 5 — Retro Template

```text
잘된 점 / 아쉬운 점 / 다음 단계
```

## 이 예시에서 봐야 할 점

- 주간 산출물은 “공부했다”가 아니라 “남겼다”는 증거가 됩니다.
- 기초를 튼튼히 해야 뒤 단계의 모델링도 덜 흔들립니다.
- 회고를 넣어야 학습 루프가 닫히고, 다음 주 계획이 현실적으로 바뀝니다.

많은 입문자가 머신러닝 모델부터 시작하고 싶어 하지만, 실제로는 데이터 정리와 SQL, 기본 통계에서 더 자주 막힙니다. 그래서 처음 4주는 화려하지 않아도 괜찮습니다. 오히려 이 구간을 건너뛰지 않아야 나중에 포트폴리오를 만들 때도 설명이 훨씬 자연스러워집니다.

## 자주 하는 실수 5가지

1. **책을 처음부터 끝까지 읽은 뒤 시작하려는 실수**
2. **기초 없이 모델링부터 들어가는 실수**
3. **주간 산출물을 남기지 않는 실수**
4. **회고 없이 같은 방식만 반복하는 실수**
5. **도구를 자주 바꾸며 흐름을 잃는 실수**

## 실무에서는 이렇게 나타납니다

많은 부트캠프와 사내 교육 프로그램도 비슷한 흐름을 탑니다. 처음에는 Python·SQL·통계 같은 기초를 다지고, 그다음 분석 문제를 풀고, 마지막에 작은 모델링 프로젝트를 붙이는 식입니다. 이유는 간단합니다. 이런 순서가 가장 재현 가능하고, 입문자가 중간에 무너지지 않기 때문입니다.

## 시니어는 이렇게 생각합니다

- 기초는 쌓일수록 복리처럼 돌아옵니다.
- 매주 남는 증거가 있어야 성장도 설명할 수 있습니다.
- 회고는 의욕이 아니라 방향을 조정하는 장치입니다.
- 프로젝트는 개별 기술을 하나로 묶어 주는 단계입니다.
- 오래 가는 학습은 무리한 속도보다 지속 가능한 리듬에서 나옵니다.

## 정리 및 다음 단계

좋은 학습 계획은 더 많은 자료를 소비하는 계획이 아니라, 매주 작은 결과물을 만들며 다음 단계로 넘어가는 계획입니다. 기초, 분석, 모델링을 순서대로 밟고, 매주 README와 코드와 결과를 남기면 공부가 눈에 보이는 자산으로 바뀝니다. 이 자산은 나중에 포트폴리오와 면접 답변의 재료가 됩니다.

다음 글에서는 이렇게 쌓은 결과물을 어떻게 포트폴리오로 구성해야 하는지 살펴보겠습니다.
