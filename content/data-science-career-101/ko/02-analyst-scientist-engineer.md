# Data Science Career 101 (2/10): 분석가 vs 사이언티스트 vs 엔지니어

이 글은 Data Science Career 101 시리즈의 2번째 글입니다.


입문자가 데이터 직무를 알아볼 때 가장 자주 묻는 질문은 결국 이것입니다. “분석가와 사이언티스트와 엔지니어는 정확히 뭐가 다른가요?” 겉으로 보면 모두 SQL과 Python을 쓰고, 모두 데이터를 다루기 때문에 차이가 흐릿하게 느껴질 수 있습니다.

하지만 실무에서는 누가 어떤 질문을 맡고, 어떤 결과물을 남기고, 무엇으로 평가받는지가 분명히 다릅니다. 이 차이를 모르고 준비하면 학습 계획도 넓게만 퍼지고, 지원서도 애매해지기 쉽습니다.

이 글은 Data Science Career 101 시리즈의 두 번째 글입니다.

## 먼저 던지는 질문

- 분석가, 사이언티스트, 엔지니어가 각각 어떤 목적을 중심에 두는지 비교합니다?
- 세 역할이 만드는 대표 산출물을 정리합니다?
- 주요 도구와 성과 지표가 왜 달라지는지 설명합니다?

## 이 글에서 배우는 내용

- 각 역할의 목적
- 대표 산출물
- 주요 도구
- 성과 지표
- 협업 방식

## 왜 중요한가

역할 차이를 잘못 이해하면 학습 방향도 엇나가고 지원 전략도 모호해집니다. 특히 입문자일수록 “무엇을 잘해야 하는 직무인가”를 먼저 잡아야 불필요한 우회를 줄일 수 있습니다.

예를 들어 실험 설계와 해석이 강점인 사람에게는 사이언티스트 트랙이 맞을 수 있지만, 데이터 모델과 파이프라인 안정화에 더 끌리는 사람에게는 엔지니어 트랙이 더 자연스러울 수 있습니다. 같은 노력도 어디에 쓰느냐에 따라 체감 성장 속도가 크게 달라집니다.

## 한눈에 보는 개념

이 그림은 세 역할의 중심축을 압축합니다. 분석가는 의사결정 지원, 사이언티스트는 가설 검증, 엔지니어는 데이터 흐름 보장에 무게를 둡니다.

## 핵심 용어

- **decision support**: 데이터를 바탕으로 의사결정을 돕는 일입니다.
- **A/B test**: 통제된 실험을 통해 효과를 비교하는 방식입니다.
- **ETL**: 데이터를 추출하고 변환하고 적재하는 흐름입니다.
- **feature store**: 모델 특징값을 재사용하기 위한 공유 저장소입니다.
- **SLA**: 서비스가 지켜야 하는 수준에 대한 약속입니다.

## 전후 비교

**Before**: "셋 다 데이터를 보는 사람이니 큰 차이가 없는 줄 알았다."

**After**: "목적과 산출물 기준으로 세 역할을 구분할 수 있다."

## 실습: 비교표 만들기

### 단계 1 — Purpose

```text
Analyst: answer questions
Scientist: validate hypotheses
Engineer: guarantee data flow
```

목적은 역할의 나침반입니다. 질문에 답하는 역할인지, 가설을 검증하는 역할인지, 데이터 흐름을 보장하는 역할인지에 따라 필요한 역량이 달라집니다.

### 단계 2 — Deliverables

```text
Analyst: dashboards, reports
Scientist: experiments, models
Engineer: pipelines, schemas
```

직무는 산출물로 가장 선명하게 드러납니다. 보고서와 대시보드가 중심인지, 실험 설계와 모델이 중심인지, 파이프라인과 스키마가 중심인지 보면 역할 구분이 쉬워집니다.

### 단계 3 — 주요 도구
```text
Analyst: SQL, BI tool
Scientist: Python, notebook, Spark
Engineer: Airflow, dbt, Kafka
```

도구는 목적을 따라옵니다. 같은 Python을 써도 분석가는 가공과 해석에, 사이언티스트는 모델링에, 엔지니어는 파이프라인 자동화에 더 자주 씁니다.

### 단계 4 — Metrics

```text
Analyst: decision adoption rate
Scientist: experimental significance
Engineer: SLA, data quality
```

어떤 지표로 평가받는지 알면 역할의 본질이 보입니다. 분석가는 의사결정 반영률, 사이언티스트는 검증의 신뢰도, 엔지니어는 안정성과 품질이 더 중요합니다.

### 단계 5 — Collaboration

```text
Analyst <-> PM/marketing
Scientist <-> PM/research
Engineer <-> backend/platform
```

협업 상대도 다릅니다. 누구와 자주 대화하는지 보면 그 역할이 팀에서 어떤 위치를 차지하는지 이해하기 쉽습니다.

## 이 예시에서 먼저 봐야 할 점

- 목적이 도구를 결정합니다.
- 성과 지표가 행동을 바꿉니다.
- 회사마다 경계는 달라져도 기본 축은 유지됩니다.

입문자가 흔히 빠지는 함정은 도구 목록만 보고 역할을 판단하는 일입니다. 하지만 같은 SQL을 써도 질문을 푸는 사람과 데이터 흐름을 설계하는 사람의 일은 완전히 다릅니다.

## 자주 하는 실수 5가지

1. **도구만 보고 직무를 분류하는 실수**
2. **산출물을 무시하는 실수**
3. **성과 지표를 정의하지 않는 실수**
4. **세 역할을 동시에 완벽하게 준비하려는 실수**
5. **도메인을 빼고 기술만 보는 실수**

## 실무에서는 이렇게 나타납니다

큰 조직일수록 역할이 더 세밀하게 분리되고, 작은 조직일수록 한 사람이 둘 이상의 역할을 겸합니다. 그렇더라도 실제 업무를 보면 차이는 남습니다. 무엇을 책임지고 무엇으로 평가받는지 보면 역할은 결국 드러납니다.

## 시니어는 이렇게 생각합니다

- 역할의 목적을 먼저 명확히 합니다.
- 산출물 합의를 먼저 맞춥니다.
- 성과 지표를 팀과 공유합니다.
- 역할 경계는 유연하게 보되 책임은 분명히 둡니다.
- 한 축을 깊게 파면서도 옆 역할과 대화하는 T자형 역량을 키웁니다.

## 체크리스트

- [ ] 세 역할의 목적을 각각 설명할 수 있다.
- [ ] 각 역할의 대표 산출물을 하나씩 안다.
- [ ] 역할별 주요 도구를 하나씩 적어 봤다.
- [ ] 역할별 핵심 지표를 하나씩 정리했다.

## 정리 및 다음 단계

세 역할은 경쟁 관계라기보다 분업 구조에 가깝습니다. 분석가는 질문을 구조화하고, 사이언티스트는 실험과 모델로 검증하고, 엔지니어는 그 과정이 안정적으로 돌아가게 받쳐 줍니다.

다음 글에서는 입문자 관점에서 실제 학습 순서를 어떻게 짜야 하는지 12주 로드맵으로 정리하겠습니다.


## 직무별 필수 기술 스택 로드맵과 학습 우선순위

분석가, 사이언티스트, 엔지니어를 비교할 때 실제 준비에 가장 도움이 되는 질문은 "무엇을 먼저 익혀야 실무에 빨리 연결되는가"입니다. 도구 목록을 길게 나열하면 오히려 실행이 멈춥니다. 그래서 아래 로드맵은 각 역할별로 0-3개월, 3-6개월, 6-12개월 단계에서 우선순위를 나눈 실전 기준입니다.

| 기간 | Analyst 우선순위 | Scientist 우선순위 | Engineer 우선순위 |
| --- | --- | --- | --- |
| 0-3개월 | SQL 기본/집계, 지표 정의, 대시보드 읽기 | Python 기초, 통계 기초, 실험 해석 | SQL + 데이터 모델링, 배치 파이프라인 개념 |
| 3-6개월 | 코호트/퍼널 분석, 리포트 구조화, 스토리텔링 | 회귀/분류 기본, 피처 엔지니어링, 평가 지표 | Airflow/dbt 기본, 스키마 관리, 품질 체크 |
| 6-12개월 | 실험 분석, 의사결정 문서화, 이해관계자 커뮤니케이션 | 실험 설계, 모델 비교, 오류 분석 루프 | 스트리밍 기초, 장애 대응, 모니터링 자동화 |

학습 우선순위를 정할 때는 "난이도"보다 "재사용성"을 기준으로 보는 편이 안정적입니다. 예를 들어 Analyst 트랙에서 SQL 집계와 지표 정의는 거의 모든 과제에서 재사용됩니다. Scientist 트랙에서는 통계적 사고와 평가 지표 해석이 모델 종류보다 먼저 재사용됩니다. Engineer 트랙에서는 스키마 설계와 품질 관리가 특정 도구보다 오래 가는 기반입니다.

다음 표는 역할별 필수 스택을 도구명 중심이 아니라 업무 행위 중심으로 재정리한 것입니다.

| 역할 | 반드시 익혀야 할 행위 | 연결 도구 예시 |
| --- | --- | --- |
| Analyst | 질문을 쿼리로 분해하기 | SQL, BI Tool |
| Analyst | 지표를 문장으로 정의하기 | Metric Spec, Dashboard |
| Scientist | 가설을 실험으로 바꾸기 | Python, Notebook, Stats |
| Scientist | 모델 결과를 비즈니스 언어로 해석하기 | sklearn, 시각화 도구 |
| Engineer | 데이터 경로를 안정화하기 | Airflow, dbt, Spark |
| Engineer | 오류 탐지와 복구 경로 설계하기 | Monitoring, Alerting |

실제로는 세 트랙이 일부 겹치므로 "공통 기반"을 먼저 확보하면 이동 비용이 줄어듭니다. 공통 기반은 SQL, 데이터 모델의 기본 이해, 통계적 사고, 문서화 습관입니다. 이 네 가지는 어떤 역할을 선택해도 손해가 없습니다. 반대로 너무 이른 시점에 고급 모델링이나 복잡한 분산 처리에만 매달리면, 실무에서 바로 쓰이는 연결고리가 약해질 수 있습니다.

학습 계획을 주 단위로 바꿔 쓰는 방법도 제안합니다. 월요일에는 개념 정리, 수요일에는 실습, 금요일에는 결과 정리와 회고를 고정하십시오. 주간 산출물은 "쿼리 1개 + 해석 1문단 + 다음 질문 1개"처럼 작게 설정하는 편이 유지율이 높습니다. 중요한 것은 매주 끝에서 "무엇을 할 수 있게 되었는가"를 문장으로 남기는 일입니다.

면접 준비 단계에서는 역할별 우선순위를 답변 구조에 그대로 반영할 수 있습니다. 예를 들어 Scientist 지원이라면 "문제 정의 -> 가설 -> 지표 -> 검증" 순서로 말하고, Engineer 지원이라면 "요구사항 -> 데이터 경로 -> 품질/운영 -> 장애 대응" 순서로 말하면 일관성이 생깁니다. 결국 학습 우선순위와 면접 답변 우선순위가 맞아야 준비가 실제 성과로 연결됩니다.


## 실행용 점검 프레임: 02-analyst-scientist-engineer.md 편

이 섹션은 앞에서 다룬 내용을 실제 행동으로 옮기기 위한 점검 프레임입니다. 많은 독자가 글을 읽을 때는 이해했다고 느끼지만, 일주일 뒤 실행 단계에서 다시 막히는 이유는 "다음 행동"이 문장으로 고정되어 있지 않기 때문입니다. 그래서 아래 프레임은 학습, 실습, 회고를 연결하는 최소 단위로 구성했습니다.

첫째, 이번 주 목표를 한 문장으로 적습니다. 목표는 도구가 아니라 문제 기준으로 써야 합니다. 예를 들어 "SQL 윈도우 함수 공부"보다 "사용자별 누적 결제를 계산해 이탈 신호를 찾는다"처럼 적는 편이 실행력이 높습니다. 같은 시간이라도 문제 중심 목표가 결과 해석까지 자연스럽게 이어집니다.

둘째, 입력과 출력을 명확히 구분합니다. 입력은 데이터, 로그, 문서, 인터뷰 메모처럼 내가 활용할 재료입니다. 출력은 쿼리, 분석 노트, 비교표, 의사결정 문서처럼 다른 사람이 검토 가능한 결과물입니다. 입력을 많이 모으는 것과 출력을 만드는 것은 다르며, 성장 속도는 출력 빈도에 더 강하게 좌우됩니다.

셋째, 판단 기준을 미리 적습니다. 많은 작업이 끝에서 흔들리는 이유는 성공 기준을 사전에 합의하지 않았기 때문입니다. "정확한가"라는 문장 대신 "분모 정의가 명확한가", "누락 세그먼트가 없는가", "후속 액션이 있는가"처럼 점검 문항으로 바꾸면 품질이 안정됩니다.

넷째, 협업 단위를 작게 쪼개 공유합니다. 완성본만 보여 주려는 습관은 피드백 타이밍을 늦춥니다. 중간 결과를 1페이지 요약으로 공유하고, 가정과 리스크를 먼저 드러내면 이후 수정 비용이 크게 줄어듭니다. 데이터 직무는 혼자 완벽하게 만든 뒤 공개하는 방식보다, 가정 기반으로 빠르게 정렬하는 방식이 장기적으로 더 강합니다.

다섯째, 회고는 감상이 아니라 의사결정 기록으로 남깁니다. "어려웠다"로 끝내지 말고 무엇이 병목이었는지, 다음에는 어떤 순서로 접근할지, 어떤 지표를 추가로 볼지를 적어야 합니다. 같은 유형의 문제를 다시 만났을 때 과거 기록이 바로 템플릿이 되면 성장 속도는 가속됩니다.

아래 표는 매주 반복 가능한 최소 운영 템플릿입니다.

| 항목 | 작성 질문 | 예시 출력 |
| --- | --- | --- |
| 목표 | 이번 주에 어떤 문제를 풀 것인가 | 문제 정의 3줄 |
| 입력 | 어떤 데이터/자료를 사용할 것인가 | 데이터 출처 목록 |
| 방법 | 어떤 접근 순서로 진행할 것인가 | 작업 단계 체크리스트 |
| 검증 | 무엇을 성공 기준으로 볼 것인가 | 품질 점검 표 |
| 결과 | 무엇이 바뀌었는가 | 한 문장 권고안 |
| 회고 | 다음 주에 무엇을 바꿀 것인가 | 개선 액션 3개 |

이 템플릿의 핵심은 완벽함이 아니라 반복 가능성입니다. 한 번에 거대한 문서를 만들 필요는 없습니다. 중요한 것은 매주 같은 구조로 남겨서 비교 가능성을 확보하는 일입니다. 비교가 가능해지면 성장도 측정 가능해지고, 측정 가능해지면 커리어 결정의 품질이 올라갑니다.

마지막으로, 실행 프레임은 개인 메모로 끝내지 말고 가능한 범위에서 팀 공유 자산으로 전환하는 것이 좋습니다. 같은 질문이 반복되는 조직일수록 표준 템플릿의 가치가 큽니다. 템플릿 하나를 잘 만드는 사람은 단순히 일을 수행하는 사람이 아니라, 팀이 더 빠르고 정확하게 일하도록 환경을 설계하는 사람으로 인식됩니다.


## 월간 점검 질문 세트

아래 질문은 월말 회고 때 그대로 사용할 수 있는 점검 세트입니다. 첫 번째 질문은 "이번 달에 내가 해결한 문제는 무엇이며, 그 문제는 누구의 의사결정에 영향을 주었는가"입니다. 두 번째 질문은 "내 결과물의 재현 가능성을 다른 동료가 확인할 수 있는가"입니다. 세 번째 질문은 "결과를 한 문장 권고안으로 요약했는가"입니다. 네 번째 질문은 "가정이 틀렸을 때 대안 경로를 제시했는가"입니다. 다섯 번째 질문은 "다음 달에는 무엇을 하지 않을 것인가"입니다.

이 다섯 질문을 매달 반복하면 학습과 실행이 분리되지 않습니다. 특히 마지막 질문은 과부하를 줄이는 데 효과적입니다. 커리어 성장은 더 많은 일을 하는 것만으로 만들어지지 않습니다. 우선순위를 분명히 하고, 덜 중요한 일을 의도적으로 줄일 때 더 빠르게 만들어집니다.

| 질문 | Pass 기준 |
| --- | --- |
| 문제 정의가 명확한가 | 이해관계자가 같은 문장으로 설명 가능 |
| 결과가 재현 가능한가 | 실행 명령과 데이터 기준이 문서화됨 |
| 해석이 행동으로 연결되는가 | 후속 액션이 1개 이상 명시됨 |
| 리스크를 선제적으로 다뤘는가 | 가정과 한계를 함께 기술함 |
| 다음 달 우선순위가 보이는가 | 할 일/하지 않을 일 목록이 분리됨 |

## 면접 질문 샘플: 역할별 답변 기준

같은 질문이라도 역할에 따라 좋은 답변의 기준이 달라집니다. 그래서 면접 준비에서는 정답을 외우기보다 역할별 관점으로 답변 구조를 바꾸는 연습이 중요합니다. 아래 예시는 자주 나오는 질문을 세 역할 관점으로 분해한 표입니다.

| 공통 질문 | Analyst 관점 | Scientist 관점 | Engineer 관점 |
| --- | --- | --- | --- |
| "전환율이 떨어졌습니다. 무엇부터 보시겠습니까?" | 세그먼트/기간 분해와 지표 정의 확인 | 가설 설정과 검증 계획 제시 | 데이터 파이프라인/로그 품질 이상 여부 확인 |
| "좋은 결과란 무엇인가요?" | 의사결정 반영률과 해석 가능성 | 통계적 신뢰도와 일반화 성능 | SLA, 지연 시간, 품질 안정성 |

실전에서는 위 표를 자기 경험과 연결해 말해야 합니다. 예를 들어 "저는 Analyst 역할에서 먼저 분모 정의를 맞추고, Scientist 협업 시 검증 지표를 명시하고, Engineer와는 데이터 신선도 기준을 합의합니다"처럼 역할 간 연결 능력을 보여 주면 답변의 밀도가 올라갑니다.


## 역할 선택 인터뷰: 세 직무를 같은 질문으로 비교하기

분석가, 사이언티스트, 엔지니어를 비교할 때는 같은 질문을 던져야 차이가 보입니다. 아래 Q&A 샘플은 면접과 자기소개서에서 바로 쓸 수 있도록 구성한 비교 프롬프트입니다.

### 샘플 질문 1
"최근에 해결한 문제를 설명해 주세요."

- Analyst 답변 축: 문제를 KPI 변화로 정의하고, 원인을 세그먼트와 퍼널로 좁힌 과정
- Scientist 답변 축: 가설 정의, 검증 설계, 대안 모델 비교 근거
- Engineer 답변 축: 데이터 누락/지연을 줄이기 위해 파이프라인 구조를 바꾼 과정

### 샘플 질문 2
"성과를 어떻게 측정했나요?"

- Analyst: 의사결정 반영률, 리포트 재사용률
- Scientist: 실험 신뢰도, 모델 개선 폭
- Engineer: 실패율, 복구 시간, 지연 시간

이 프레임을 쓰면 "세 직무가 비슷해 보인다"는 감각이 줄어듭니다. 같은 문제라도 측정 지표와 성공 기준이 다르기 때문입니다.

## 커리어 래더 다이어그램: 주니어에서 미드레벨로 넘어가는 기준

```text
Junior -> Mid-level 전환 기준
1) 지시된 작업 수행
2) 문제를 재정의하고 우선순위를 제안
3) 결과를 문서화하고 재사용 가능하게 정리
4) 팀 공통 기준(지표/모델/파이프라인)에 기여
```

전환의 핵심은 난이도 높은 기술 하나가 아니라, "문제 정의와 협업 구조화" 역량입니다. 역할별 기술 스택은 달라도, 이 전환 기준은 공통으로 적용됩니다.


## 실전 부록: 지원 문서와 면접 답변을 한 세트로 준비하는 방법

글을 읽고 이해한 내용을 실제 커리어 행동으로 바꾸려면, 이력서 문장·포트폴리오 증거·면접 답변이 서로 같은 문제를 가리키도록 정렬해야 합니다. 아래 구성은 데이터 직무 입문자와 주니어가 가장 빠르게 효과를 보는 준비 방식입니다.

### 이력서 불릿 템플릿

```text
- [문제] 어떤 비즈니스 또는 운영 문제를 맡았는가
- [행동] 어떤 데이터 접근(지표/실험/파이프라인/모델)을 적용했는가
- [결과] 무엇이 얼마나 개선되었는가
- [의미] 그 결과가 팀 의사결정에 어떤 영향을 주었는가
```

예시 문장: "결제 퍼널 이탈 구간을 코호트 기준으로 분해해 주요 병목을 식별했고, 온보딩 실험 우선순위를 재정렬해 4주 내 전환율을 개선했습니다." 핵심은 기술 이름보다 책임과 결과를 문장으로 남기는 일입니다.

### 포트폴리오 프로젝트 선정 체크

| 점검 항목 | Pass 기준 |
| --- | --- |
| 문제 명확성 | 한 문장으로 문제와 영향 범위를 설명 가능 |
| 데이터 현실성 | 공개 데이터라도 결측/편향/한계를 문서화 |
| 재현 가능성 | 실행 절차와 검증 기준을 README에 명시 |
| 의사결정 연결 | 결과가 다음 액션으로 이어짐 |

이 표를 기준으로 프로젝트를 고르면 "보여주기용 분석"을 줄이고 "실무형 문제 해결"을 남길 수 있습니다.

### 인터뷰 Q&A 3문장 프레임

1. 문제를 어떻게 정의했는지 말합니다.
2. 어떤 접근을 선택했고 왜 그렇게 판단했는지 말합니다.
3. 결과와 한계를 함께 설명하고 다음 액션을 제시합니다.

이 세 문장으로 답변하면 면접관이 가장 궁금해하는 판단 과정이 빠르게 드러납니다. 특히 한계를 먼저 말하는 태도는 신뢰를 높입니다.

### 보상 협상 기본 스크립트

```text
- 제안 감사합니다. 기대 역할과 성공 기준을 먼저 정렬하고 싶습니다.
- 제가 제공할 수 있는 기여 범위와 최근 증거를 공유드리겠습니다.
- 시장 밴드와 총보상 구조를 기준으로 조정 가능한 항목을 논의하고 싶습니다.
```

협상은 대립이 아니라 역할-보상 정렬입니다. 역할 범위를 수치와 산출물로 제시하면 대화가 감정에서 사실로 이동합니다.

### 역량 점검 매트릭스

| 영역 | 현재 수준(1~5) | 목표 수준(1~5) | 다음 2주 액션 |
| --- | --- | --- | --- |
| 문제 정의 |  |  |  |
| 데이터 처리 |  |  |  |
| 검증 설계 |  |  |  |
| 전달력 |  |  |  |
| 협업 운영 |  |  |  |

월 1회 이 표를 채우면 성장 체감이 감정이 아니라 기록으로 남습니다. 커리어 의사결정의 품질은 결국 기록의 품질에서 올라갑니다.

## 처음 질문으로 돌아가기

- **분석가, 사이언티스트, 엔지니어가 각각 어떤 목적을 중심에 두는지 비교합니다?**
  - 데이터 직무링크로 보면 세 역할이 다른 맨답과 동맥을 링크로 닻다는 랜 나안니다. 즉, 분석가는 질문에 답하고, 사이언티스트는 가설을 검증하고, 엔지니어는 데이터 흐름을 보장하는 부역 역할로 날라 서로 다를 도구를 잊니다.
- **세 역할이 만드는 대표 산출물을 정리합니다?**
  - 각 역할의 대표 산출물이 다르다는 것은 그들이 기대하는 답과 결과도 다르다는 저제 노출답단, 인서서도 다른 답을 기대한다는 것임니다.
- **주요 도구와 성과 지표가 왜 달라지는지 설명합니다?**
  - 도구를 맞이 보는 대신 '각이 글을 빼나고 뭘나 산출물을 만드는가'를 보면 실라 달른 역할을 구분할 수 있습니다.
