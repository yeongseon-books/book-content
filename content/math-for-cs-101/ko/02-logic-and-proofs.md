# 논리와 증명

개발자는 매일 판단을 합니다. 이 조건이 충분한가, 이 가정이 안전한가, 이 구현이 모든 입력에서 맞는가 같은 질문입니다. 처음에는 예제 몇 개를 돌려 보고 넘어갈 수 있지만, 중요한 로직일수록 언젠가는 왜 맞는지 설명해야 합니다.

특히 조건문이 많아지고 상태 전이가 복잡해질수록 테스트를 더 많이 쓰는 것만으로는 부족해집니다. 어떤 주장을 하고 있는지, 그 주장에 필요한 가정은 무엇인지, 반례가 있는지 분리해서 봐야 하기 때문입니다.

이 글은 Math for CS 101 시리즈의 2번째 글입니다.

여기서는 논리와 증명을 참과 거짓을 다루는 추상 수학이 아니라, 프로그램과 알고리즘의 올바름을 설명하는 실전 문법으로 다뤄 보겠습니다.

---

## 이 글에서 다룰 문제

- 테스트와 증명은 무엇이 다를까요?
- 명제, 함의, 동치는 코드와 어떤 식으로 연결될까요?
- 직접 증명과 귀류법은 언제 쓰면 좋을까요?
- 수학적 귀납법은 반복문 검증과 어떻게 닮아 있을까요?
- 반례 하나가 왜 전체 주장을 무너뜨릴 수 있을까요?

> 논리는 참과 거짓을 다루는 문법이고, 증명은 그 문법으로 프로그램과 알고리즘의 올바름을 설명하는 절차입니다. 테스트가 일부 사례를 비추는 손전등이라면, 증명은 주장 전체의 경계를 그리는 지도에 가깝습니다.

---

## 왜 중요한가

테스트는 중요한 도구이지만 본질적으로 일부 입력만 확인합니다. 반면 증명은 가능한 모든 경우를 겨냥합니다. 예제가 세 개 맞았다고 해서 모든 n에 대해 맞는 것은 아닙니다. 그래서 논리와 증명은 코드를 더 많이 짜기 전에, 내가 정확히 어떤 명제를 주장하고 있는지부터 분명하게 만듭니다.

개발에서도 이 감각은 자주 필요합니다. 타입 시스템은 어떤 값이 들어올 수 있는지 논리적으로 제한하고, 분산 합의 알고리즘은 특정 장애 조건에서도 안전성이 유지되는지 증명하려고 합니다. 인증과 권한 처리처럼 조건 조합이 많은 영역일수록 명제와 함의를 분명히 다루는 능력이 바로 설계 품질로 이어집니다.

---

## 머릿속에 먼저 둘 관점

논리와 증명을 처음 배울 때 가장 도움이 되는 문장은 이것입니다. **증명은 예시를 많이 모으는 작업이 아니라, 가정에서 결론까지의 연결을 빠짐없이 적는 작업**이라는 점입니다.

명제는 참과 거짓을 가질 수 있는 문장이고, 함의는 어떤 조건이 참일 때 다른 결론이 따라오는 관계입니다. 직접 증명은 가정에서 결론으로 곧장 가는 방식이고, 귀류법은 결론의 반대를 가정해 모순을 이끌어 냅니다. 수학적 귀납법은 한 단계가 다음 단계로 이어진다는 사실을 통해 전체 자연수 구간을 다루는 방법입니다.

이 도구들을 외워야 해서 배우는 것이 아닙니다. 같은 주장을 여러 방식으로 검토할 수 있게 만들기 때문에 배웁니다. 어떤 주장은 직접 밀고 가는 편이 가장 짧고, 어떤 주장은 반대를 가정하는 편이 더 잘 보입니다. 반복 구조가 들어가면 귀납법이 사실상 기본 문법이 됩니다.

---

## 작은 증명 도구 상자

### 첫 번째 단계 — 진리표로 함의를 펼칩니다

```python
def truth_imply():
    return [(p, q, (not p) or q) for p in (False, True) for q in (False, True)]
```

함의 `p → q`는 코드로 옮기면 `not p or q`가 됩니다. 낯선 기호처럼 보여도 결국 불리언 표현입니다. 이 감각이 있으면 조건문이 언제 참이 되는지 훨씬 또렷하게 읽을 수 있습니다.

### 두 번째 단계 — 동치를 확인합니다

```python
def equiv(p, q):
    return p == q
```

동치는 두 표현이 항상 같은 진리값을 갖는다는 뜻입니다. 리팩터링을 할 때 두 조건식이 보기만 다른지, 실제 의미도 같은지 확인하는 작업과 아주 가깝습니다.

### 세 번째 단계 — 직접 증명으로 밀고 갑니다

```python
def even_sum(a, b):
    assert a % 2 == 0 and b % 2 == 0
    return (a + b) % 2 == 0
```

짝수 두 개의 합이 짝수라는 사실은 직접 증명의 전형적인 예입니다. 가정이 분명할수록 결론까지의 경로도 짧아집니다. 코드에서는 선행 조건을 명확히 적는 습관과 맞닿아 있습니다.

### 네 번째 단계 — 귀류법으로 반대를 흔듭니다

```python
def assume_not(claim):
    return f"suppose not {claim}, derive contradiction"
```

직접 밀고 가기 어려운 주장은 반대를 가정하는 편이 더 잘 보일 때가 많습니다. 동시성 제약이나 상태 불변식을 검토할 때도 이런 방식이 자주 쓰입니다. 불가능하다고 느끼는 이유를 감으로 두지 않고, 모순으로 드러내는 것입니다.

### 다섯 번째 단계 — 귀납법으로 전체를 엮습니다

```python
def sum_to(n):
    return n * (n + 1) // 2
```

귀납법은 반복 구조와 특히 잘 맞습니다. 기저 단계가 맞고, n에서 n+1로 넘어갈 수 있으면 전체 자연수 구간으로 확장됩니다. 그래서 저는 귀납법을 수학의 별도 기술보다, 반복문 검증의 가까운 친척으로 보는 편이 실용적이라고 생각합니다.

---

## 이 코드에서 먼저 볼 점

- 함의는 낯선 기호가 아니라 불리언 표현으로 옮길 수 있습니다.
- 직접 증명은 가정이 분명할수록 읽기 쉬워집니다.
- 귀류법은 반대를 가정했을 때 무엇이 깨지는지 보는 방식입니다.
- 귀납법은 반복문이 단계적으로 이어지는 구조와 닮아 있습니다.
- 반례 하나는 보편 명제를 무너뜨리기에 충분합니다.

---

## 어디서 자주 헷갈릴까요?

가장 흔한 실수는 예시 몇 개를 증명으로 착각하는 일입니다. 예시는 이해를 돕지만, 보편 명제를 보장하지는 않습니다. 테스트 세트가 통과했다는 말과 모든 입력에서 맞다는 말은 다릅니다.

함의와 역을 같은 것으로 보는 실수도 많습니다. `p → q`가 참이라고 해서 `q → p`가 자동으로 참이 되지는 않습니다. 권한 검증, 캐시 조건, 에러 핸들링처럼 가드가 많은 코드에서는 이 혼동이 버그로 이어지기 쉽습니다.

귀납법에서 기저 단계를 빼먹는 것도 전형적인 함정입니다. 출발점이 비어 있으면 아무리 귀납 단계가 매끈해도 전체 증명이 서지 않습니다. 반복문 초기화가 틀리면 이후 단계가 아무리 좋아도 소용없는 것과 같습니다.

---

## 실무에서는 이렇게 생각한다

컴파일러의 타입 검사기는 허용 가능한 상태를 논리적으로 제한합니다. 분산 시스템의 합의 알고리즘은 특정 장애 조건에서도 안전성이 유지되는지 증명합니다. 인증과 권한 처리에서도 조건 조합이 복잡해질수록 논리적 사고가 직접 도움이 됩니다.

좋은 엔지니어는 반례를 공격으로 받아들이지 않습니다. 오히려 설계의 경계를 명확하게 해 주는 빠른 피드백으로 봅니다. 또 증명을 수학 숙제처럼 다루기보다, 가정을 드러내고 결론을 제한하는 문서화 습관으로 가져갑니다.
