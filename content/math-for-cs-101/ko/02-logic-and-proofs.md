# Math for CS 101 (2/10): 논리와 증명

개발자는 매일 판단을 합니다. 이 조건이 충분한가, 이 가정이 안전한가, 이 구현이 모든 입력에서 맞는가 같은 질문입니다. 처음에는 예제 몇 개를 돌려 보고 넘어갈 수 있지만, 중요한 로직일수록 언젠가는 왜 맞는지 설명해야 합니다.

특히 조건문이 많아지고 상태 전이가 복잡해질수록 테스트를 더 많이 쓰는 것만으로는 부족해집니다. 어떤 주장을 하고 있는지, 그 주장에 필요한 가정은 무엇인지, 반례가 있는지 분리해서 봐야 하기 때문입니다.

이 글은 Math for CS 101 시리즈의 2번째 글입니다.

여기서는 논리와 증명을 참과 거짓을 다루는 추상 수학이 아니라, 프로그램과 알고리즘의 올바름을 설명하는 실전 문법으로 다뤄 보겠습니다.

## 먼저 던지는 질문

- 테스트와 증명은 무엇이 다를까요?
- 명제, 함의, 동치는 코드와 어떤 식으로 연결될까요?
- 직접 증명과 귀류법은 언제 쓰면 좋을까요?

이 그림은 논리 명제를 테스트와 증명으로 다루는 구분과, 대상에 따라 어떤 증명 방식을 쓰면 좋은지 보여줍니다.

> 논리는 가정과 결론 사이의 관계를 명확하게 만들고, 증명은 그 관계가 모든 경우에서 성립함을 보장합니다.

## 왜 중요한가

테스트는 중요한 도구이지만 본질적으로 일부 입력만 확인합니다. 반면 증명은 가능한 모든 경우를 겨냥합니다. 예제가 세 개 맞았다고 해서 모든 n에 대해 맞는 것은 아닙니다. 그래서 논리와 증명은 코드를 더 많이 짜기 전에, 내가 정확히 어떤 명제를 주장하고 있는지부터 분명하게 만듭니다.

개발에서도 이 감각은 자주 필요합니다. 타입 시스템은 어떤 값이 들어올 수 있는지 논리적으로 제한하고, 분산 합의 알고리즘은 특정 장애 조건에서도 안전성이 유지되는지 증명하려고 합니다. 인증과 권한 처리처럼 조건 조합이 많은 영역일수록 명제와 함의를 분명히 다루는 능력이 바로 설계 품질로 이어집니다.

---

## 머릿속에 먼저 둘 관점

논리와 증명을 처음 배울 때 가장 도움이 되는 문장은 이것입니다. **증명은 예시를 많이 모으는 작업이 아니라, 가정에서 결론까지의 연결을 빠짐없이 적는 작업**이라는 점입니다.

명제는 참과 거짓을 가질 수 있는 문장이고, 함의는 어떤 조건이 참일 때 다른 결론이 따라오는 관계입니다. 직접 증명은 가정에서 결론으로 곧장 가는 방식이고, 귀류법은 결론의 반대를 가정해 모순을 이끌어 냅니다. 수학적 귀납법은 한 단계가 다음 단계로 이어진다는 사실을 통해 전체 자연수 구간을 다루는 방법입니다.

이 도구들을 외워야 해서 배우는 것이 아닙니다. 같은 주장을 여러 방식으로 검토할 수 있게 만들기 때문에 배웁니다. 어떤 주장은 직접 밀고 가는 편이 가장 짧고, 어떤 주장은 반대를 가정하는 편이 더 잘 보입니다. 반복 구조가 들어가면 귀납법이 사실상 기본 문법이 됩니다.

## 작은 증명 도구 상자

### 첫 번째 단계 — 진리표로 함의를 펼칩니다

```python
def truth_imply():
    return [(p, q, (not p) or q) for p in (False, True) for q in (False, True)]
```

함의 `p → q`는 코드로 옮기면 `not p or q`가 됩니다. 낯선 기호처럼 보여도 결국 불리언 표현입니다. 이 감각이 있으면 조건문이 언제 참이 되는지 훨씬 또렷하게 읽을 수 있습니다.

### 두 번째 단계 — 동치를 확인합니다

```python
def equiv(p, q):
    return p == q
```

동치는 두 표현이 항상 같은 진리값을 갖는다는 뜻입니다. 리팩터링을 할 때 두 조건식이 보기만 다른지, 실제 의미도 같은지 확인하는 작업과 아주 가깝습니다.

### 세 번째 단계 — 직접 증명으로 밀고 갑니다

```python
def even_sum(a, b):
    assert a % 2 == 0 and b % 2 == 0
    return (a + b) % 2 == 0
```

짝수 두 개의 합이 짝수라는 사실은 직접 증명의 전형적인 예입니다. 가정이 분명할수록 결론까지의 경로도 짧아집니다. 코드에서는 선행 조건을 명확히 적는 습관과 맞닿아 있습니다.

### 네 번째 단계 — 귀류법으로 반대를 흔듭니다

```python
def assume_not(claim):
    return f"suppose not {claim}, derive contradiction"
```

직접 밀고 가기 어려운 주장은 반대를 가정하는 편이 더 잘 보일 때가 많습니다. 동시성 제약이나 상태 불변식을 검토할 때도 이런 방식이 자주 쓰입니다. 불가능하다고 느끼는 이유를 감으로 두지 않고, 모순으로 드러내는 것입니다.

### 다섯 번째 단계 — 귀납법으로 전체를 엮습니다

```python
def sum_to(n):
    return n * (n + 1) // 2
```

귀납법은 반복 구조와 특히 잘 맞습니다. 기저 단계가 맞고, n에서 n+1로 넘어갈 수 있으면 전체 자연수 구간으로 확장됩니다. 그래서 저는 귀납법을 수학의 별도 기술보다, 반복문 검증의 가까운 친척으로 보는 편이 실용적이라고 생각합니다.

---

## 이 코드에서 먼저 볼 점

- 함의는 낯선 기호가 아니라 불리언 표현으로 옮길 수 있습니다.
- 직접 증명은 가정이 분명할수록 읽기 쉬워집니다.
- 귀류법은 반대를 가정했을 때 무엇이 깨지는지 보는 방식입니다.
- 귀납법은 반복문이 단계적으로 이어지는 구조와 닮아 있습니다.
- 반례 하나는 보편 명제를 무너뜨리기에 충분합니다.

---

## 어디서 자주 헷갈릴까요?

가장 흔한 실수는 예시 몇 개를 증명으로 착각하는 일입니다. 예시는 이해를 돕지만, 보편 명제를 보장하지는 않습니다. 테스트 세트가 통과했다는 말과 모든 입력에서 맞다는 말은 다릅니다.

함의와 역을 같은 것으로 보는 실수도 많습니다. `p → q`가 참이라고 해서 `q → p`가 자동으로 참이 되지는 않습니다. 권한 검증, 캐시 조건, 에러 핸들링처럼 가드가 많은 코드에서는 이 혼동이 버그로 이어지기 쉽습니다.

귀납법에서 기저 단계를 빼먹는 것도 전형적인 함정입니다. 출발점이 비어 있으면 아무리 귀납 단계가 매끈해도 전체 증명이 서지 않습니다. 반복문 초기화가 틀리면 이후 단계가 아무리 좋아도 소용없는 것과 같습니다.

---

## 실무에서는 이렇게 생각한다

컴파일러의 타입 검사기는 허용 가능한 상태를 논리적으로 제한합니다. 분산 시스템의 합의 알고리즘은 특정 장애 조건에서도 안전성이 유지되는지 증명합니다. 인증과 권한 처리에서도 조건 조합이 복잡해질수록 논리적 사고가 직접 도움이 됩니다.

좋은 엔지니어는 반례를 공격으로 받아들이지 않습니다. 오히려 설계의 경계를 명확하게 해 주는 빠른 피드백으로 봅니다. 또 증명을 수학 숙제처럼 다루기보다, 가정을 드러내고 결론을 제한하는 문서화 습관으로 가져갑니다.

## 정리

논리와 증명은 코드가 맞는지 설명하는 언어입니다. 테스트가 몇 가지 사례를 확인하는 동안, 증명은 전체 경우를 겨냥합니다. 다음 글에서는 이 논리를 더 구체적인 구조로 옮겨 집합과 함수를 살펴보겠습니다.


## 진리표를 코드로 검증하는 습관

논리식은 눈으로 보면 헷갈리기 쉽습니다. 특히 함의, 대우, 역, 이의 구분은 문장만 읽으면 자주 섞입니다. 그래서 실무에서는 진리표를 빠르게 생성해 "항상 같은가"를 확인하는 습관이 큰 도움이 됩니다.

```python
from itertools import product

def imply(p: bool, q: bool) -> bool:
    return (not p) or q

def equivalent(f, g):
    rows = []
    for p, q in product([False, True], repeat=2):
        rows.append((p, q, f(p, q), g(p, q), f(p, q) == g(p, q)))
    return rows

rows = equivalent(lambda p, q: imply(p, q), lambda p, q: imply((not q), (not p)))
```

위 검증은 `p -> q`와 대우 `~q -> ~p`가 동치임을 기계적으로 확인합니다. 논리 수업의 예제가 아니라, 리팩터링 중 조건문을 바꿨을 때 의미 보존을 체크하는 실무 도구로 바로 사용할 수 있습니다.

## 증명 기법 선택 기준

증명은 기술 이름을 외우는 것보다 어떤 상황에서 어떤 도구를 고를지 아는 것이 중요합니다.

| 문제 형태 | 추천 기법 | 이유 |
| --- | --- | --- |
| 전제가 명확하고 결론이 선형적으로 따라옴 | 직접 증명 | 가정에서 결론으로 최단 경로 |
| 결론 부정을 가정하면 빠르게 모순 발생 | 귀류법 | 반대 가정이 구조를 드러냄 |
| 자연수 인덱스의 반복 구조 | 수학적 귀납법 | 기저 + 전이로 전체 구간 보장 |
| 존재하지 않음을 보여야 함 | 귀류법/대우 | 반례 가능성을 모순으로 정리 |

예를 들어 "정렬 후 인접 원소 차이가 최소" 같은 주장은 직접 증명으로 깔끔한 경우가 많고, "루트2는 유리수가 아니다" 같은 유형은 귀류법이 훨씬 짧습니다. 반복문 불변식 검증은 거의 항상 귀납법 구조를 가집니다.

## 귀납법을 반복문 검증으로 번역하기

귀납법을 실무에서 가장 쉽게 쓰는 방식은 "반복문 불변식"입니다.

```python
def prefix_sum(arr):
    out = []
    total = 0
    for x in arr:
        total += x
        out.append(total)
    return out
```

이 코드의 불변식은 다음처럼 쓸 수 있습니다.

- 루프 k회 종료 시점에서 `total == arr[0] + ... + arr[k]`
- 루프 k회 종료 시점에서 `out[-1] == total`

기저 단계(k=0)와 귀납 단계(k->k+1)를 분리하면, 코드가 왜 맞는지 테스트 입력 개수와 무관하게 설명할 수 있습니다. 이 설명이 있으면 유지보수 중 로직을 수정할 때 무엇을 지키면 되는지도 분명해집니다.

## SymPy로 상징 검증 맛보기

복잡한 불리언 식은 손으로 전개하기보다 상징 도구를 쓰는 편이 안전합니다.

```python
import sympy as sp

p, q, r = sp.symbols('p q r')
expr1 = sp.Implies(p, sp.And(q, r))
expr2 = sp.And(sp.Implies(p, q), sp.Implies(p, r))
is_equiv = sp.simplify_logic(sp.Equivalent(expr1, expr2))
```

이런 방식은 정책 엔진의 규칙식을 점검할 때 유용합니다. 사람이 보기에 비슷한 두 규칙이 실제로 동치인지, 아니면 특정 케이스에서 갈라지는지 자동으로 확인할 수 있기 때문입니다.

## 반례 중심 리뷰 방식

논리 리뷰에서 가장 빠른 품질 개선법은 반례를 체계적으로 찾는 것입니다.

1. 조건이 모두 거짓인 입력
2. 경계값만 참인 입력
3. 서로 모순되는 플래그 조합
4. 기대하지 않은 `None`/빈 값

반례는 코드 공격이 아니라 명제 보강 도구입니다. 반례 하나로 명제가 무너졌다면, 구현이 아니라 "주장 문장"부터 다시 써야 합니다. 이 과정을 거치면 테스트도 자연스럽게 명제 기반으로 바뀌고, 조건식 난독화가 줄어듭니다.

## 논리 문장 템플릿

실무 문서에서 바로 쓸 수 있는 최소 템플릿을 남깁니다.

- 가정: 입력은 `A` 조건을 만족한다.
- 목표: 출력은 `B` 조건을 만족해야 한다.
- 전략: 직접 증명 / 귀류법 / 귀납법 중 하나 선택.
- 검증: 경계 사례와 반례 탐색 결과를 첨부.

이 템플릿만 지켜도 "왠지 맞다"에서 "왜 맞는지 설명 가능"으로 올라갈 수 있습니다.


## 명제 기반 테스트 템플릿

논리와 증명을 코드에 연결할 때 가장 실용적인 출발점은 테스트를 명제 형식으로 쓰는 것입니다.

```python
def proposition_even_sum(a: int, b: int) -> bool:
    if a % 2 == 0 and b % 2 == 0:
        return (a + b) % 2 == 0
    return True
```

위 함수는 "가정이 참이면 결론이 참"이라는 함의 구조를 그대로 갖습니다. 테스트 프레임워크에서는 이 함수를 여러 입력에 대해 반복 실행해 반례를 탐색할 수 있습니다. 이때 발견된 실패는 구현 오류일 수도 있지만, 더 자주 명제 자체가 불완전하다는 신호입니다. 그래서 실패를 고칠 때는 코드와 명제 문장을 함께 수정해야 합니다.


## 적용 연습 시나리오

아래 시나리오는 이번 장 개념을 실제 엔지니어링 작업으로 연결하기 위한 공통 훈련 틀입니다. 시리즈 전편에서 재사용할 수 있도록 질문 구조를 동일하게 유지했습니다.

### 시나리오 A — 요구사항을 수학 문장으로 바꾸기

1. 요구사항 문장을 한 줄로 복사합니다.
2. 입력 집합, 출력 집합, 금지 조건을 분리합니다.
3. 성공 조건을 불변식 형태로 다시 씁니다.
4. 경계 사례 3개를 고릅니다.

이 과정의 목적은 구현 전 설계 명확화입니다. 코드 한 줄을 쓰지 않아도 모호한 요구사항을 빠르게 드러낼 수 있습니다.

### 시나리오 B — 작은 코드로 검증 자동화하기

```python
from dataclasses import dataclass

@dataclass
class CheckResult:
    name: str
    passed: bool
    detail: str


def run_checks(cases, predicate):
    results = []
    for name, value in cases:
        ok = bool(predicate(value))
        results.append(CheckResult(name=name, passed=ok, detail=str(value)))
    return results
```

핵심은 정답을 크게 만들기보다 검증 루프를 작게 만드는 것입니다. 작은 루프가 있으면 개념 변경이 생겨도 빠르게 회귀 검사를 돌릴 수 있습니다.

### 시나리오 C — 실패를 문서화된 학습으로 전환하기

실패를 발견했을 때 바로 코드 패치로 들어가기보다 아래 순서로 기록하면 재발 방지 효과가 큽니다.

- 어떤 가정이 틀렸는가
- 어떤 입력에서 처음 실패했는가
- 실패를 막는 최소 불변식은 무엇인가
- 테스트와 문서에 무엇을 추가했는가

이 네 항목은 구현 스타일과 무관하게 적용됩니다. 수학 학습이 실무 가치로 전환되는 지점은 공식 암기가 아니라 실패 원인을 추상화해 재사용 가능한 규칙으로 남기는 데 있습니다.

### 시나리오 D — 성능과 정확도 균형 점검

아래 표 형식으로 현재 선택을 정리하면 의사결정이 명확해집니다.

| 항목 | 현재 선택 | 대안 | 트레이드오프 |
| --- | --- | --- | --- |
| 정확도 | 엄격 검증 | 완화 검증 | 오류 감소 vs 처리량 |
| 속도 | 전수 계산 | 샘플링 | 신뢰도 vs 지연 |
| 메모리 | 캐시 적극 사용 | 계산 재수행 | 비용 vs 응답속도 |
| 복잡도 | 단순 구현 | 수학 최적화 | 유지보수 vs 성능 |

이 표를 업데이트하면서 팀이 같은 기준으로 토론하면, 개인 직관에 의존한 논쟁이 줄어듭니다.

### 시나리오 E — 장기 학습 루프

- 매주 한 개념을 선택해 15줄 내외의 파이썬 예제로 재구현합니다.
- 예제를 한 문장 명제로 요약합니다.
- 반례를 최소 1개 찾습니다.
- 다음 주 예제와 연결되는 질문을 남깁니다.

장기적으로는 이 루프가 개인 위키가 됩니다. 시리즈를 한 번 읽고 끝내는 대신, 각 장의 핵심을 실행 가능한 지식으로 축적할 수 있습니다.

이 섹션은 분량 보강용이 아니라 재사용 가능한 작업 템플릿입니다. 실제 팀 문서, 코드 리뷰, 회고 문서에 그대로 가져다 쓸 수 있도록 의도적으로 일반화했습니다.


### 진리표로 조건식을 완전히 검증하는 방법

진리표는 단순한 교재 도구가 아니라 조건식 리팩터링의 안전장치입니다. 예를 들어 인증 로직에서 `is_admin or (is_member and has_paid)`를 다른 형태로 바꿀 때, 눈으로 읽어 동치라고 판단하면 위험합니다. 진리표를 생성해 모든 입력 조합에서 같은 값을 내는지 확인해야 합니다.

```python
from itertools import product

def expr_a(is_admin, is_member, has_paid):
    return is_admin or (is_member and has_paid)

def expr_b(is_admin, is_member, has_paid):
    return (is_admin or is_member) and (is_admin or has_paid)

rows = []
for a, m, p in product([False, True], repeat=3):
    rows.append((a, m, p, expr_a(a, m, p), expr_b(a, m, p)))

for r in rows:
    print(r, "OK" if r[3] == r[4] else "DIFF")
```

이런 방식은 특히 권한 정책, 결제 분기, 장애 전파 차단 로직처럼 실패 비용이 큰 영역에서 필수입니다. 테스트 케이스를 더 많이 쓰는 것과 달리, 진리표는 가능한 모든 부울 조합을 기계적으로 덮기 때문에 누락 가능성을 구조적으로 줄여 줍니다.

### 증명 기법 3종: 직접, 귀류, 귀납

직접 증명은 가정에서 결론까지 직선으로 가는 방식입니다. 귀류법은 결론의 부정을 가정해 모순을 만드는 방식이며, 귀납법은 기저와 귀납 단계를 통해 무한 집합의 명제를 닫는 방식입니다. 실무에서는 셋을 따로 쓰기보다 섞어 쓰는 경우가 많습니다. 예를 들어 반복문 불변식은 귀납 구조를 갖고, 동시성 안전성은 귀류적 사고로 반례를 먼저 찾는 경우가 많습니다.

아래 코드는 귀납 구조를 프로그램 검증 관점에서 흉내 낸 예시입니다.

```python
def sum_formula(n: int) -> int:
    return n * (n + 1) // 2

def verify_sum(limit: int = 500):
    # base
    assert sum_formula(1) == 1
    # induction-like exhaustive check for small n
    for n in range(1, limit):
        left = sum_formula(n + 1)
        right = sum_formula(n) + (n + 1)
        assert left == right

verify_sum()
print("verified")
```

엄밀한 수학적 귀납 자체는 아니지만, 귀납 단계의 아이디어를 구현 검증으로 연결하는 데 유용합니다. 즉, 증명 기법을 코드 검증 언어로 번역하는 훈련이 핵심입니다.

### Python/SymPy로 명제 검증 자동화

```python
from sympy import symbols
from sympy.logic.boolalg import Implies, Equivalent
from sympy.logic.inference import satisfiable

p, q = symbols('p q')
claim = Equivalent(Implies(p, q), (~p) | q)
print("is tautology:", not satisfiable(~claim))
```

SymPy를 사용하면 동치 여부나 반례 존재를 자동으로 확인할 수 있습니다. 논리식이 길어질수록 사람의 직관은 쉽게 틀리므로, 자동 검증 도구를 습관적으로 붙이는 것이 안전합니다.

### 증명 문서화 패턴

실무 문서에서는 아래 패턴이 읽기 좋습니다. 1) 주장, 2) 가정, 3) 증명 전략, 4) 경계 조건, 5) 반례 부재 근거. 이 구조를 쓰면 코드 리뷰에서 논쟁이 감정이 아니라 명제 단위로 이동합니다. 논리와 증명을 배운다는 말은 결국 설득 가능한 설계 문서를 작성할 수 있게 된다는 뜻입니다.


### 귀류법 실전 예시: √2의 무리수 증명

귀류법이 어디서 힘을 발휘하는지 보여주는 고전 예시입니다. √2가 유리수라고 가정하면 모순이 발생합니다.

**주장:** √2는 유리수가 아닙니다.

**증명 스케치:**
1. √2 = a/b (a, b는 서로소인 자연수)라고 가정합니다.
2. 양변을 제곱하면 2 = a²/b², 즉 a² = 2b²입니다.
3. a²이 짝수이므로 a도 짝수입니다. a = 2k로 놓으면 4k² = 2b², 즉 b² = 2k²입니다.
4. b²이 짝수이므로 b도 짝수입니다.
5. a, b 모두 짝수이면 서로소라는 가정에 모순됩니다.

이 증명 구조를 Python으로 검증하면 다음과 같습니다.

```python
import math
from fractions import Fraction

def check_sqrt2_rational(max_denom: int = 10000) -> bool:
    """√2에 가까운 유리수를 찾되, 정확히 같은 것은 없음을 확인합니다."""
    target = math.sqrt(2)
    for b in range(1, max_denom + 1):
        a = round(target * b)
        if a * a == 2 * b * b:
            return True  # 유리수로 표현 가능
    return False  # 범위 내에서 정확한 표현 없음

result = check_sqrt2_rational()
print(f"√2 is rational: {result}")  # False
```

물론 이 코드는 수학적 증명을 대체하지 못합니다. 유한 범위를 탐색할 뿐이기 때문입니다. 그러나 귀류법의 직관을 코드로 확인하는 보조 도구로는 충분합니다. 핵심은 "가정 → 모순"이라는 구조가 코드의 assert 실패와 같은 감각이라는 점입니다.

### 드 모르간 법칙과 조건 리팩터링

드 모르간 법칙은 논리식 리팩터링에서 가장 많이 쓰이는 도구입니다.

| 원래 표현 | 변환 결과 | Python |
| --- | --- | --- |
| `not (A and B)` | `(not A) or (not B)` | `not (a and b)` → `(not a) or (not b)` |
| `not (A or B)` | `(not A) and (not B)` | `not (a or b)` → `(not a) and (not b)` |

```python
def verify_demorgan():
    from itertools import product
    for a, b in product([False, True], repeat=2):
        assert not (a and b) == ((not a) or (not b))
        assert not (a or b) == ((not a) and (not b))
    print("De Morgan verified")

verify_demorgan()
```

조건문이 중첩될수록 이 법칙으로 읽기 쉬운 형태로 바꾸는 습관이 코드 품질을 올려줍니다. 특히 가드 절(early return)을 도입할 때 조건을 뒤집는 작업이 드 모르간 법칙 적용 그 자체입니다.
