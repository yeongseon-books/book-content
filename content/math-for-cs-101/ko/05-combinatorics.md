# Math for CS 101 (5/10): 조합

알고리즘이 왜 갑자기 느려지는지, 해시 충돌이 왜 피할 수 없는지, 테스트 케이스 수가 왜 금방 폭발하는지 설명하려면 결국 셈이 필요합니다. 하지만 실무 문제에서 모든 경우를 손으로 나열하는 일은 거의 불가능합니다.

조합론은 바로 이 지점에서 힘을 발휘합니다. 순서가 중요한지, 중복이 허용되는지, 선택이 서로 겹치는지 같은 구조만 정확히 잡으면, 경우의 수를 직접 늘어놓지 않고도 계산할 수 있습니다.

이 글은 Math for CS 101 시리즈의 5번째 글입니다.

여기서는 조합론을 공식 암기 과목으로 보지 않고, 복잡도 분석과 확률 계산의 바닥에서 쓰는 셈의 언어로 정리해 보겠습니다.

## 왜 중요한가

알고리즘이 왜 느려지는지 설명하려면 가능한 경우의 수가 얼마나 빨리 늘어나는지 알아야 합니다. 해시 충돌 가능성을 가늠할 때도, 테스트 케이스 수를 정할 때도, 샘플링 전략을 세울 때도 결국 셈이 필요합니다.

조합론의 장점은 모든 경우를 손으로 나열하지 않아도 된다는 점입니다. 순서가 중요한지, 중복을 허용하는지, 선택이 서로 배타적인지 같은 구조만 파악하면 공식과 원리로 훨씬 빠르게 판단할 수 있습니다. 그래서 조합론은 계산 기술이면서 동시에 모델링 기술이기도 합니다.

---

## 머릿속에 먼저 둘 관점

조합론을 처음 배울 때 가장 도움이 되는 관점은 **무엇을 세는지보다 어떤 구조로 세는지 먼저 묻는 것**입니다. 같은 문제도 순서가 중요하면 순열이 되고, 순서가 중요하지 않으면 조합이 됩니다. 서로 독립적인 연속 선택이면 곱의 법칙이 들어오고, 서로 배타적인 선택이면 합의 법칙이 들어옵니다.

또 하나 중요한 도구가 비둘기집 원리입니다. 직접 경우를 세지 않아도 충돌이나 중복이 반드시 생긴다는 사실을 보여 줄 수 있습니다. 해시 버킷보다 키가 많으면 충돌이 생긴다는 직관이 여기서 바로 나옵니다.

이항계수는 조합론이 확률로 넘어가는 다리 역할을 합니다. 파스칼 삼각형, 이항 전개, 이산 확률 분포가 모두 이 한 구조 위에서 연결됩니다.

## 다섯 단계로 보는 셈의 기초

### 첫 번째 단계 — 팩토리얼을 익힙니다

```python
def fact(n):
    r = 1
    for i in range(2, n + 1):
        r *= i
    return r
```

팩토리얼은 조합론에서 가장 자주 재사용되는 부품입니다. 순열과 조합 공식을 이해하려면 먼저 익숙해져야 합니다. 단순 반복처럼 보여도, 감소하는 선택 수를 한 줄로 압축한 표현입니다.

### 두 번째 단계 — 순서를 세어 봅니다

```python
def nPr(n, r):
    return fact(n) // fact(n - r)
```

순열은 순서가 바뀌면 다른 경우로 셉니다. 비밀번호 자리 배치, 작업 실행 순서, 발표 순서처럼 배열 자체가 의미를 갖는 문제에서 이 감각이 중요합니다.

### 세 번째 단계 — 선택만 남깁니다

```python
def nCr(n, r):
    return fact(n) // (fact(r) * fact(n - r))
```

조합은 고르는 것 자체만 중요합니다. 팀 구성, 샘플 선택, 피처 집합 선택처럼 순서가 필요 없는 문제에서 자주 나옵니다. 순열과 조합을 구분하는 가장 쉬운 질문은 이것입니다. 순서가 바뀌면 다른 경우인가요?

### 네 번째 단계 — 충돌의 필연을 봅니다

```python
def pigeon(items, holes):
    return items > holes
```

놀랄 만큼 단순한 부등식이지만, 충돌이 반드시 생긴다는 사실을 증명할 수 있습니다. 긴 계산보다 구조 파악이 더 중요하다는 점에서 조합론다운 도구입니다.

### 다섯 번째 단계 — 이항계수를 펼칩니다

```python
def row(n):
    return [nCr(n, k) for k in range(n + 1)]
```

이항계수는 파스칼 삼각형, 확률 분포, 다항식 전개와 모두 연결됩니다. 조합론이 더 이상 셈만의 과목이 아니라는 사실을 여기서 체감하게 됩니다.

---

## 이 코드에서 먼저 볼 점

- 팩토리얼은 여러 공식의 공통 재료입니다.
- 순열과 조합의 차이는 결국 순서의 의미입니다.
- `nCr`은 `nCr(n, r) = nCr(n, n-r)` 대칭성을 가집니다.
- 비둘기집 원리는 긴 계산보다 구조 파악이 더 중요합니다.
- 큰 n에서는 직접 계산 방식의 비용과 정수 크기도 함께 봐야 합니다.

---

## 어디서 자주 헷갈릴까요?

순열과 조합을 바꿔 쓰는 실수가 가장 흔합니다. 계산 실수 이전에 모델링 실수입니다. 문제를 읽고도 무엇을 세는지 구조를 구분하지 못하면 공식은 맞아도 답이 틀립니다.

중복 허용 여부를 확인하지 않는 것도 자주 나옵니다. 같은 선택을 여러 번 할 수 있느냐에 따라 경우의 수가 완전히 달라집니다. 그래서 공식을 외우기 전에 먼저 규칙을 적는 편이 안전합니다.

`0! = 1`을 놓치거나, 아주 큰 n에 팩토리얼을 그대로 적용하는 것도 전형적인 함정입니다. 수학적 정의와 계산 비용을 함께 봐야 한다는 말입니다.

## 실무에서는 이렇게 생각한다

A/B 테스트 버킷 수를 설계할 때, 해시 충돌 가능성을 대략 볼 때, 샘플 수를 정할 때, 경우의 수 폭발이 예상되는 탐색 문제를 점검할 때 조합론은 실용적인 판단 기준을 줍니다. 저는 조합론이 복잡도 분석의 서문이라고 생각합니다. 얼마나 많은 길이 열려 있는지 모르면, 어떤 알고리즘이 현실적으로 가능한지도 판단하기 어렵기 때문입니다.

시니어 엔지니어는 공식을 바로 꺼내기보다 먼저 구조를 묻습니다. 순서가 중요한가, 독립적인 선택인가, 충돌이 피할 수 없는가, 경우의 수가 이미 감당 불가능한가를 먼저 봅니다. 그다음에야 공식이 들어옵니다.

## 곱의 법칙과 합의 법칙을 코드로 분리하기

조합 문제를 풀 때 가장 먼저 해야 할 일은 선택 구조를 곱인지 합인지로 분해하는 것입니다. 서로 독립적으로 연속 선택하면 곱의 법칙, 서로 배타적인 선택지 중 하나를 고르면 합의 법칙입니다.

```python
def multiplication_rule(*counts: int) -> int:
    out = 1
    for c in counts:
        out *= c
    return out

def addition_rule(*counts: int) -> int:
    return sum(counts)
```

예를 들어 비밀번호가 영문 소문자 26개, 숫자 10개에서 길이 4로 구성되고 각 자리가 독립이라면 `26 * 10 * 26 * 10`처럼 곱의 법칙을 씁니다. 반면 로그인 실패 원인이 "비밀번호 오류" 또는 "계정 잠금"처럼 동시에 성립하지 않는 범주라면 합의 법칙으로 합칩니다.

## 순열/조합/중복 조합 비교

| 유형 | 질문 | 공식 |
| --- | --- | --- |
| 순열 `nPr` | 순서까지 구분하는가 | `n! / (n-r)!` |
| 조합 `nCr` | 순서 무시하고 고르기만 하는가 | `n! / (r!(n-r)!)` |
| 중복 조합 | 같은 원소를 여러 번 고를 수 있는가 | `C(n+r-1, r)` |

이 표를 빠르게 떠올릴 수 있으면 모델링 실수를 크게 줄일 수 있습니다.

## itertools로 결과를 검증하는 방법

공식 계산만 믿기보다 작은 입력에서 열거와 대조하면 학습과 검증이 동시에 됩니다.

```python
import itertools
import math

items = ['A', 'B', 'C', 'D']
perms = list(itertools.permutations(items, 2))
combs = list(itertools.combinations(items, 2))

assert len(perms) == math.perm(4, 2)
assert len(combs) == math.comb(4, 2)
```

이 방식은 테스트 코드로도 쓸 수 있습니다. 복잡한 경우의 수 계산 함수를 작성했다면 작은 n에서는 열거 결과와 일치하는지 자동 검증하면 신뢰도가 올라갑니다.

## 비둘기집 원리를 충돌 분석에 적용하기

해시 버킷이 100개인데 키가 101개면 충돌은 반드시 발생합니다. 이 명제는 확률 추정이 아니라 결정적 사실입니다. 설계 리뷰에서 "충돌 가능성"을 말하는 단계에서 멈추지 말고 "충돌 불가 여부"를 먼저 판정하는 데 비둘기집 원리를 사용하면 좋습니다.

```python
def must_collide(num_keys: int, num_buckets: int) -> bool:
    return num_keys > num_buckets
```

충돌이 필연이라면 관심사는 충돌 회피가 아니라 충돌 처리 전략으로 이동해야 합니다. 예를 들어 재해시 정책, 체이닝 길이 상한, 리사이징 임계값 같은 운영 규칙이 본질이 됩니다.

## 이항계수와 확률의 연결

동전 n번 던져 앞면이 k번 나올 경우의 수는 `C(n, k)`입니다. 확률 챕터에서 등장하는 이항분포의 핵심 항이 바로 이 계수입니다. 즉 조합론을 건너뛰면 확률 분포를 공식으로만 외우게 되고, 의미를 놓치기 쉽습니다.

## 실무 적용 체크포인트

1. 순서 중요 여부를 문서에 명시했는가
2. 중복 허용 여부를 명시했는가
3. 작은 입력에서 열거 검증을 붙였는가
4. 경우의 수 폭발 구간에서 근사/샘플링 전략이 있는가

조합론의 목적은 정답 숫자 하나를 얻는 것이 아니라, 계산 가능성과 불가능성의 경계를 조기에 드러내는 것입니다.

## 조합 폭발을 조기에 감지하는 휴리스틱

완전 탐색을 시작하기 전에 탐색 공간을 거칠게 추정하면 시간 낭비를 줄일 수 있습니다.

```python
import math

def search_space_bits(count: int) -> float:
    return math.log2(count)
```

경우의 수가 `2^40` 수준만 넘어도 단일 머신에서 전수 탐색은 사실상 불가능합니다. 이때는 분기 한정, 동적 계획법, 근사, 랜덤 샘플링 같은 전략으로 바로 전환해야 합니다. 조합론은 정답 계산 도구이면서 동시에 "지금 방식이 안 된다"를 빠르게 선언하는 도구입니다.

### 시나리오 B — 작은 코드로 검증 자동화하기

```python
from dataclasses import dataclass

@dataclass
class CheckResult:
    name: str
    passed: bool
    detail: str

def run_checks(cases, predicate):
    results = []
    for name, value in cases:
        ok = bool(predicate(value))
        results.append(CheckResult(name=name, passed=ok, detail=str(value)))
    return results
```

핵심은 정답을 크게 만들기보다 검증 루프를 작게 만드는 것입니다. 작은 루프가 있으면 개념 변경이 생겨도 빠르게 회귀 검사를 돌릴 수 있습니다.

### 시나리오 C — 실패를 문서화된 학습으로 전환하기

실패를 발견했을 때 바로 코드 패치로 들어가기보다 아래 순서로 기록하면 재발 방지 효과가 큽니다.

- 어떤 가정이 틀렸는가

- 테스트와 문서에 무엇을 추가했는가

이 네 항목은 구현 스타일과 무관하게 적용됩니다. 수학 학습이 실무 가치로 전환되는 지점은 공식 암기가 아니라 실패 원인을 추상화해 재사용 가능한 규칙으로 남기는 데 있습니다.

| 정확도 | 엄격 검증 | 완화 검증 | 오류 감소 vs 처리량 |
| 속도 | 전수 계산 | 샘플링 | 신뢰도 vs 지연 |
| 메모리 | 캐시 적극 사용 | 계산 재수행 | 비용 vs 응답속도 |
| 복잡도 | 단순 구현 | 수학 최적화 | 유지보수 vs 성능 |

이 표를 업데이트하면서 팀이 같은 기준으로 토론하면, 개인 직관에 의존한 논쟁이 줄어듭니다.

### 순열과 조합 Python 코드 확장

```python
import math

def npr(n, r):
    return math.factorial(n) // math.factorial(n-r)

def ncr(n, r):
    return math.factorial(n) // (math.factorial(r) * math.factorial(n-r))

print(npr(10, 3), ncr(10, 3))
```

순열은 순서가 중요할 때, 조합은 선택만 중요할 때 사용합니다. 문제 문장을 읽고 "배열"인지 "선택"인지 먼저 구분해야 합니다.

### 포함-배제 원리

겹치는 사건을 단순 합으로 세면 중복 계산이 생깁니다. 포함-배제 원리는 이 중복을 체계적으로 제거합니다.

```python
def inclusion_exclusion(a, b, inter):
    return a + b - inter

print(inclusion_exclusion(120, 90, 30))
```

세 집합 이상에서는 교집합 차수를 번갈아 더하고 빼야 합니다. 접근 제어, 규칙 엔진, 모니터링 알람 집계에서 매우 자주 등장합니다.

### 카탈란 수와 응용

카탈란 수는 "유효한 괄호열", "BST 구조 수", "교차 없는 연결" 같은 구조를 셀 때 등장합니다.

```python
def catalan(n):
    dp = [0] * (n + 1)
    dp[0] = 1
    for i in range(1, n + 1):
        dp[i] = sum(dp[j] * dp[i-1-j] for j in range(i))
    return dp[n]

for i in range(6):
    print(i, catalan(i))
```

문제를 카탈란 구조로 매핑할 수 있으면 지수적 완전탐색 대신 닫힌 형태 계산이나 DP로 비용을 크게 줄일 수 있습니다.

### 경우의 수 폭발 진단 체크

탐색 알고리즘 설계 전에는 반드시 분기 계수와 깊이를 곱셈 관점으로 추정해야 합니다. 이 추정이 없으면 성능 문제를 구현 이후에야 발견하게 됩니다. 조합론은 사후 최적화가 아니라 사전 리스크 관리 도구입니다.

### 생일 문제(Birthday Problem)

조합론의 직관적 예시로 가장 유명한 문제입니다. n명 중 생일이 같은 쌍이 있을 확률은 생각보다 빨리 커집니다.

```python
def birthday_collision_prob(n: int, days: int = 365) -> float:
    """
    n명 중 생일이 같은 쌍이 있을 확률을 계산합니다.
    P(충돌) = 1 - P(모두 다름)
    """
    if n > days:
        return 1.0
    no_collision = 1.0
    for i in range(n):
        no_collision *= (days - i) / days
    return 1 - no_collision

# 23명이면 생일 충돌 확률이 50%를 넘습니다
for n in [10, 23, 50, 70]:
    print(f"n={n:3d}: {birthday_collision_prob(n):.4f}")
# n= 10: 0.1169
# n= 23: 0.5073
# n= 50: 0.9704
# n= 70: 0.9992
```

이 문제는 해시 충돌, UUID 중복, 랜덤 ID 부여 등에서 직접적으로 등장합니다. 128비트 UUID의 충돌 확률을 예측할 때도 같은 논리를 적용합니다. 도메인이 365일이 아니라 2¹²⁸이 될 뿐입니다.

### 별과 막대(Stars and Bars) 정리

n개의 동일한 물건을 k개의 구분된 상자에 나누는 방법의 수는 C(n+k-1, k-1)입니다. 이 공식은 리소스 분배, 로드 밸런싱, 파티션 문제에서 자주 쓰입니다.

```python
import math

def distribute(items: int, bins: int) -> int:
    """items개를 bins개 상자에 나누는 방법의 수 (빈 상자 허용)"""
    return math.comb(items + bins - 1, bins - 1)

# 10개의 요청을 3개의 서버에 분배하는 방법
print(distribute(10, 3))  # 66

# 5개의 크레딧을 4개의 프로젝트에 할당하는 방법
print(distribute(5, 4))   # 56
```

실무에서 로드 밸런서가 요청을 백엔드 서버에 분배하는 경우의 수를 이해하면, 특정 분배 전략이 왜 편향되는지 정량적으로 설명할 수 있습니다.

### 조합론 연습 문제 세트

아래 문제들은 직접 풀어 보면 개념 정착에 도움이 됩니다.

| 문제 | 힌트 | 답 |
| --- | --- | --- |
| 8명에서 3명 선발(순서 무관) | 조합 C(8,3) | 56 |
| 4자리 비밀번호(0-9, 중복 허용) | 곱의 법칙 10⁴ | 10,000 |
| 5개 동전 던지기에서 앞면 3개 | C(5,3) × (1/2)⁵ | 5/16 |
| 10층 건물에서 3층 선택(순서 있음) | 순열 P(10,3) | 720 |
| MISSISSIPPI 문자 배열 | 다항 순열 11!/(4!4!2!) | 34,650 |

```python
# 다항 순열 계산
import math

def multinomial_permutation(total: int, *groups: int) -> int:
    denom = 1
    for g in groups:
        denom *= math.factorial(g)
    return math.factorial(total) // denom

# MISSISSIPPI: 11글자, M=1, I=4, S=4, P=2
print(multinomial_permutation(11, 1, 4, 4, 2))  # 34650
```

조합론 문제를 풀 때 가장 중요한 첫 단계는 "순서가 중요한가?"입니다. 중요하면 순열, 아니면 조합입니다. 다음으로 "중복을 허용하는가?"를 묻습니다. 이 두 질문만으로 대부분의 문제를 올바른 공식으로 라우팅할 수 있습니다.

### 포함-배제 원리 확장: 세 집합 이상

두 집합일 때는 |A ∪ B| = |A| + |B| - |A ∩ B|로 간단하지만, 세 집합 이상이면 교차 항을 번갈아 더하고 빼야 합니다.

```python
def inclusion_exclusion_3(a: int, b: int, c: int,
                          ab: int, ac: int, bc: int,
                          abc: int) -> int:
    """세 집합의 합집합 크기를 계산합니다."""
    return a + b + c - ab - ac - bc + abc

# 예시: 1000명 중 Python 사용자 400, Java 300, Go 200
# Python&Java 100, Python&Go 50, Java&Go 30, 세 모두 언어 10
print(inclusion_exclusion_3(400, 300, 200, 100, 50, 30, 10))  # 730
```

모니터링 시스템에서 "어떤 알람 규칙에도 해당하지 않는 이벤트 수"를 구할 때도 같은 원리를 적용합니다. 전체에서 각 규칙에 걸리는 수를 빼되, 중복 제거를 포함-배제 구조로 처리합니다.

### 조합론으로 보는 암호 강도

비밀번호나 토큰의 강도는 결국 경우의 수입니다. 공격자가 무차별 대입으로 맞출 확률을 계산할 때 조합론을 직접 사용합니다.

| 토큰 구성 | 경우의 수 | 무차별 탐색 시간(1M/초) |
| --- | --- | --- |
| 숫자 4자리 | 10⁴ = 10,000 | 0.01초 |
| 영소문자 8자리 | 26⁸ ≈ 2×10¹¹ | ~58시간 |
| 영소+대+숫자+특수 12자리 | 94¹² ≈ 5×10²³ | 우주 나이 초과 |

이 표를 보면 문자 종류를 늘리는 것보다 길이를 늘리는 편이 경우의 수를 더 빠르게 폭발시킵니다. 이것이 바로 곱의 법칙의 실무 적용입니다. 보안 감사에서 "토큰 엔트로피가 충분한가?"를 평가할 때 이 계산이 근거가 됩니다. 조합론을 모르면 "길면 안전하다"라는 막연한 직관에 의존하게 되고, 정량적 논거를 대지 못합니다.
