# Developer Career 101 (3/10): 학습 계획 세우기

배워야 할 기술은 계속 늘어나는데 일상은 늘 바쁩니다. 그래서 많은 사람이 좋은 책과 강의를 잔뜩 모아 두고도 실제로는 끝까지 가져가지 못합니다.

이 글은 Developer Career 101 시리즈의 3번째 글입니다.


![Developer Career 101 3장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/developer-career-101/03/03-01-concept-at-a-glance.ko.png)
*Developer Career 101 3장 흐름 개요*
> 학습 계획을 세울 때는 기술 깊이, 새로운 도메인, 그리고 외부에 드러나는 결과물이 함께 균형을 이루도록 해야 합니다.

## 먼저 던지는 질문

- 바쁜 일상 속에서도 학습을 오래 지속하려면 어떤 구조가 필요할까요?
- 분기 목표와 주간 루틴은 어떻게 연결되어야 할까요?
- 책, 강의, 코드베이스를 고를 때 어떤 기준이 있어야 입력만 쌓이지 않을까요?

## 이 글에서 배울 것

- 분기 목표를 세우는 법
- 주간 루틴을 고정하는 법
- 학습 자료를 고르는 기준
- 결과물을 만드는 방식
- 회고를 운영하는 법

## 왜 중요한가

계획 없는 학습은 쉽게 흩어집니다. 반대로 분기 목표와 주간 시간 블록, 결과물, 회고가 연결되어 있으면 바쁜 시기에도 학습이 끊기지 않습니다.

> 학습은 의욕이 생길 때만 하는 이벤트가 아니라, 미리 예약된 시간과 결과물로 굴리는 시스템입니다.

## 핵심 개념 한눈에 보기

학습은 입력을 많이 넣는다고 자동으로 쌓이지 않습니다. 목표가 주간 루틴으로 내려오고, 그 루틴이 결과물로 이어지며, 결과물이 다시 회고로 연결될 때 비로소 다음 분기의 학습이 더 정교해집니다.

## 핵심 용어

- 목표: 도달하려는 지점입니다.
- 루틴: 주 단위로 반복되는 리듬입니다.
- 결과물: 바깥에 남는 산출물입니다.
- 회고: 과정을 돌아보는 점검입니다.
- **의도적 연습**: 구조화된 집중 훈련입니다.

## 개선 전/후

**Before**: “책과 강의를 사 두기만 하고 끝냅니다.”

**After**: “분기마다 하나의 결과물을 실제로 내보냅니다.”

## 직접 해보기: 학습 루틴 만들기

### 1단계 — 분기 목표

```markdown
2026 Q2: Build a CLI tool in Rust
```

목표는 막연한 공부가 아니라 끝이 있는 문장이어야 합니다. “Rust 공부하기”보다 “Rust로 CLI 도구 하나 만들기”가 훨씬 운영하기 쉽습니다.

### 2단계 — 주간 루틴

```text
Tue/Thu 21:00-22:00 (60 min)
Sat 09:00-11:00 (120 min)
```

시간 블록은 기분이 아니라 일정에 붙여야 합니다. 평일 짧은 집중과 주말 긴 몰입을 조합하면 지속성이 훨씬 높아집니다.

### 3단계 — 학습 자료 고르기

```text
- 1 book
- 1 course
- 1 codebase
```

자료를 많이 고르는 것이 능사는 아닙니다. 책 하나, 강의 하나, 실제 코드베이스 하나처럼 성격이 다른 자료를 조합하면 입력이 서로 보완됩니다.

### 4단계 — 결과물

```text
- repo URL
- blog post
- talk slides
```

결과물은 학습을 증거로 바꿉니다. 저장소, 글, 발표 자료처럼 바깥에 남는 형태가 있어야 학습이 커리어 자산으로 연결됩니다.

### 5단계 — 분기 회고

```markdown
- achieved: 90%
- blocker: ownership
- next quarter: async/await depth
```

회고는 잘했는지 못했는지만 적는 문서가 아닙니다. 달성률, 방해 요인, 다음 분기 보정 방향을 함께 적어야 다음 사이클이 더 좋아집니다.

| --- | --- | --- |
| 분기 목표 | 결과물과 마감이 한 문장에 같이 있다 | "공부하기"처럼 끝이 없다 |
| 주간 루틴 | 요일, 시간, 길이가 캘린더에 박혀 있다 | 남는 시간에 하겠다고 남겨 둔다 |

## 학습 로드맵 표

학습 분야를 CS 기초, 언어, 프레임워크, 시스템 설계, 소프트스킬로 나누면 각 분야마다 기간, 자원, 검증 방법을 구체적으로 세울 수 있습니다.

| 분야 | 기간 | 학습 자원 | 검증 방법 |
| --- | --- | --- | --- |
| CS 기초 | 3개월 | MIT OCW, 교재, 강의 | 문제 풀이 저장소, 정리 글 |
| 언어 (Python) | 2개월 | 공식 문서, Fluent Python | CLI 도구 하나, pytest |
| 프레임워크 (FastAPI) | 2개월 | 공식 문서, FastAPI 예제 | API 3개, 배포 링크 |
| 시스템 설계 | 3개월 | DDIA, 시스템 디자인 인터뷰 | 모의 설계 문서 3개 |
| 소프트스킬 | 지속 | 팀 협업, 코드 리뷰, 글 | 발표 1회, 블로그 글 3개 |

CS 기초는 진입 장벽이 높지만 한 번 쌓으면 평생 갑니다. 언어와 프레임워크는 2~3개월 집중으로 기본을 잡을 수 있습니다. 시스템 설계는 실무 경험이 쌓여야 하는 영역이고, 소프트스킬은 평생 다듬어야 할 부분입니다.

## 학습 함정 피하기

학습할 때 가장 흔한 실수는 튀토리얼 지옥과 완벽주의입니다. 이 두 가지는 실력 향상을 막는 가장 큰 장벽물입니다.

### 튀토리얼 지옥

튀토리얼만 계속 따라 하면 응용 능력이 생기지 않습니다. 튀토리얼은 시작점이지 끝이 아닙니다.

**빠지는 방법:**

1. 튀토리얼을 따라 한 번 만든다
2. 코드를 보지 않고 처음부터 다시 만든다
3. 기능을 하나 추가하거나 바꾹다

```python
# 튀토리얼: FastAPI TODO API
# 1단계: 튀토리얼을 그대로 따라 함
from fastapi import FastAPI

app = FastAPI()
todos = []

@app.post("/todos")
def create_todo(title: str):
    todos.append({"title": title})
    return {"ok": True}

# 2단계: 코드 없이 처음부터 다시 만듦
# 3단계: 완료 상태를 추가하거나 DB 연결로 바꾸기
```

### 완벽주의

모든 것을 완벽하게 이해하려 하면 아무것도 시작하지 못합니다. 70% 이해로 먼저 만들고, 막히면 그때 찾아보세요.

**빠지는 방법:**

1. 전체 개념을 70% 이해한다
2. 작은 프로젝트를 시작한다
3. 막히면 공식 문서로 돌아간다

```python
# 70% 이해로 시작한 프로젝트
# "SQLAlchemy를 완벽히 이해하고 시작하기" → 실패
# "70% 이해하고 먼저 API 하나 만들기" → 성공

# 기본 CRUD를 먼저 만든다
@app.post("/users")
def create_user(name: str):
    # 나중에 relationship, join, transaction을 추가한다
    user = User(name=name)
    session.add(user)
    session.commit()
    return user
```

## 파이썬 학습 트래커 예제

학습 트래커는 분기 목표를 주간 단위로 쪼개어 진행상황을 기록하는 도구입니다. Markdown 파일 하나로 충분합니다.

```python
# 학습_추적기.py
from pathlib import Path
from datetime import date

def update_tracker(week: int, topic: str, status: str):
    """
    학습 트래커 업데이트

    Args:
        week: 주차 (1-12)
        topic: 학습 주제
        status: done / in-progress / planned
    """
    tracker_file = Path("learning_tracker.md")

    # 파일이 없으면 생성
    if not tracker_file.exists():
        tracker_file.write_text(
            "# Python 학습 트래커 2026 Q2\n\n"
            "| 주차 | 주제 | 상태 | 날짜 |\n"
            "| --- | --- | --- | --- |\n"
        )

    # 레코드 추가
    with tracker_file.open("a") as f:
        f.write(f"| {week} | {topic} | {status} | {date.today()} |\n")

    print(f"Added: Week {week}, {topic}, {status}")


# 사용 예시
update_tracker(1, "Python 기초 문법", "done")
update_tracker(2, "List comprehension", "in-progress")
update_tracker(3, "Generator", "planned")

# 출력 파일: learning_tracker.md
# | 주차 | 주제 | 상태 | 날짜 |
# | --- | --- | --- | --- |
# | 1 | Python 기초 문법 | done | 2026-05-21 |
# | 2 | List comprehension | in-progress | 2026-05-21 |
# | 3 | Generator | planned | 2026-05-21 |
```

매주 금요일에 트래커를 업데이트하면 학습 진행상황이 한눈에 보입니다. 달성률이 70% 이하로 떨어지면 다음 주 계획을 조정하세요.

## 이 예시에서 먼저 볼 점

- 목표는 결과물로 드러나야 합니다.
- 루틴은 추상적 의지가 아니라 시간 블록입니다.
- 회고는 계획을 고치는 장치입니다.

## 자주 하는 실수 5가지

1. **목표를 모호하게 잡는 일입니다.**
2. **기분에 따라 움직이는 루틴을 만드는 일입니다.**
3. **입력만 쌓는 일입니다.**
4. **결과물을 정하지 않는 일입니다.**
5. **회고를 건너뛰는 일입니다.**

## 실무에서는 이렇게 드러납니다

승진 준비나 성과 개선 계획도 결국 분기 목표와 결과물로 평가되는 경우가 많습니다. 학습 계획을 운영하는 습관은 일상 업무 평가 구조와도 자연스럽게 이어집니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 학습은 계획입니다.
- 결과물은 증거입니다.
- 시간 블록은 습관이 됩니다.
- 회고는 편집 도구입니다.
- 깊이는 누적됩니다.

## 정리

지속 가능한 학습 계획은 의욕보다 구조에 더 많이 의존합니다. 분기 목표를 세우고, 주간 시간을 예약하고, 결과물을 만들고, 회고로 조정하는 흐름이 생기면 바쁜 시기에도 학습이 사라지지 않습니다. 다음 글에서는 이 학습과 경험을 채용 담당자가 빠르게 이해할 수 있는 이력서와 포트폴리오로 어떻게 바꾸는지 정리하겠습니다.

## 이력서로 연결되는 학습 계획 설계

학습 계획은 지식을 쌓는 일정표가 아니라, 채용 문서로 연결되는 증거 생성 시스템이어야 합니다. 즉 분기 목표를 세울 때부터 "이 결과가 이력서의 어떤 줄이 되는가"를 함께 설계해야 합니다. 아래 템플릿은 학습-실행-증거를 한 흐름으로 묶기 위한 구조입니다.

### 학습 계획 템플릿

| 항목 | 작성 질문 | 예시 |
| --- | --- | --- |
| 목표 역량 | 이번 분기에 강화할 한 가지는 무엇인가 | API 성능 최적화 |
| 학습 입력 | 어떤 자료 3개로 시작할 것인가 | 책 1, 강의 1, 실무 코드 1 |
| 실험 과제 | 무엇을 직접 만들어 볼 것인가 | 캐시 적용 전후 벤치마크 |
| 산출물 | 외부/내부에 무엇을 남길 것인가 | 기술 노트, PR, 발표 슬라이드 |
| 검증 지표 | 진짜 향상을 어떻게 확인할 것인가 | p95 220ms→140ms |

이 템플릿의 강점은 학습의 시작점부터 검증과 기록을 강제한다는 점입니다. 덕분에 분기 말에 "열심히 했다"가 아니라 "무엇이 얼마나 개선됐다"를 말할 수 있습니다.

## 상황-과제-행동-결과 기법을 학습 회고에 쓰는 방법

STAR는 면접 답변에만 쓰는 프레임이 아닙니다. 주간 회고를 STAR로 적으면 학습이 경험으로 변환됩니다.

- Situation: 어떤 배경과 제약에서 시작했는가
- Task: 이번 주 해결하려던 핵심 과제는 무엇인가
- Action: 실제로 시도한 방법은 무엇인가
- Result: 수치 또는 행동 변화로 무엇이 달라졌는가

예시:

- Situation: 배치 작업 지연으로 대시보드 갱신이 늦어졌습니다.
- Task: 파이프라인 시간을 30% 줄이는 것이 목표였습니다.
- Action: 병목 쿼리 인덱스 추가와 배치 분할을 적용했습니다.
- Result: 실행 시간이 42분에서 24분으로 감소했습니다.

이렇게 기록된 STAR는 나중에 이력서 불릿, 포트폴리오 사례, 면접 답변의 재료로 바로 재활용됩니다.

## 주간 운영 보드 예시

| 요일 | 학습 블록 | 실행 블록 | 기록 블록 |
| --- | --- | --- | --- |
| 화요일 | 개념 학습 60분 | 실습 30분 | 메모 15분 |
| 목요일 | 코드 리딩 60분 | 리팩터링 30분 | 회고 15분 |
| 토요일 | 집중 실험 120분 | 결과 정리 60분 | 블로그 초안 30분 |

학습이 오래 가려면 "언제 할지"보다 "언제 기록할지"를 먼저 고정해야 합니다. 기록 없는 학습은 누적되지 않습니다.

## 학습 로드맵 타임라인: 16주 운영안

학습 계획은 주제를 나열하는 문서가 아니라 시간과 에너지를 배분하는 운영 문서입니다. 아래 표는 16주 기준으로 백엔드 트랙을 운영하는 예시입니다.

| 주차 | 초점 | 실습 산출물 | 검증 질문 |
| --- | --- | --- | --- |
| 1-4주 | Python/HTTP 기초 강화 | FastAPI API 2개 | 요청-응답 흐름을 설명할 수 있는가 |
| 5-8주 | 데이터 저장/트랜잭션 | PostgreSQL 연동 프로젝트 | 장애 시 롤백 전략을 말할 수 있는가 |
| 9-12주 | 테스트/관측성 | pytest 30개, 로그 대시보드 | 실패 원인을 30분 내 좁힐 수 있는가 |
| 13-16주 | 배포/운영 루틴 | CI 파이프라인 + 문서화 | 새 환경에서 재배포가 가능한가 |

각 주차는 학습 60%, 구현 30%, 회고 10%로 고정합니다. 이 비율을 지키면 “공부는 많이 했는데 남는 결과물이 없다”는 상태를 방지할 수 있습니다. 또한 매주 금요일에는 반드시 다음 세 문장을 기록합니다. “이번 주에 만든 것”, “막힌 이유”, “다음 주에 제거할 리스크”. 이 세 줄이 누적되면 면접 답변과 포트폴리오 설명 자료로 바로 전환됩니다.

## 매니저 1:1 학습 점검 템플릿

혼자 학습하는 시기에도 분기별로 멘토 또는 매니저와 1:1 점검을 하면 방향 이탈을 크게 줄일 수 있습니다. 아래 형식은 30분 미팅 기준 템플릿입니다.

| 구간 | 시간 | 질문 | 준비 자료 |
| --- | --- | --- | --- |
| 진행 공유 | 10분 | 지난 4주 동안 무엇을 완주했는가 | 주간 로그 4개 |
| 병목 분석 | 10분 | 반복 실패 패턴은 무엇인가 | 실패한 PR/문제 목록 |
| 다음 실험 | 10분 | 다음 4주에 무엇을 줄이고, 무엇을 늘릴까 | 다음 스프린트 계획표 |

1:1이 끝나면 액션 아이템을 세 개 이하로 제한합니다. “테스트 코드 추가”, “DB 인덱스 학습”, “모의 면접 1회”처럼 구체 항목을 정하면 실행률이 올라갑니다. 너무 많은 과제를 잡으면 결국 하나도 끝내지 못하고 자신감만 낮아집니다.

## 처음 질문으로 돌아가기

- **바쁜 일상 속에서도 학습을 오래 지속하려면 어떤 구조가 필요할까요?**
  - 학습 계획은 현재 약한 역량 하나, 새로운 도메인 하나, 그리고 외부에 드러나는 결과물로 구성되어야 합니다.
- **분기 목표와 주간 루틴은 어떻게 연결되어야 할까요?**
  - 책이나 강의를 보는 것만으로는 부족하고, 그것을 직접 코드나 프로젝트로 만들어 봐야 배움이 실제 역량이 됩니다.
- **책, 강의, 코드베이스를 고를 때 어떤 기준이 있어야 입력만 쌓이지 않을까요?**
  - 학습 목표를 정하고, 분기별로 달성 여부를 확인하며, 예상 외의 시간이 걸린 부분을 기록하면 다음 분기 계획이 훨씬 현실적이 됩니다.
