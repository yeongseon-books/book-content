# Developer Career 101 (5/10): 코딩 인터뷰 준비

문제를 많이 풀었다고 해서 인터뷰에서 바로 좋은 평가를 받는 것은 아닙니다. 실제 면접에서는 풀이 아이디어를 말로 설명해야 하고, 제한 시간 안에 판단과 복잡도를 함께 보여 줘야 합니다.

- 문제를 많이 푸는 것만으로 코딩 인터뷰 준비가 충분하지 않은 이유는 무엇일까요?
- 자주 나오는 문제 패턴을 어떻게 묶어 공부해야 시간 낭비를 줄일 수 있을까요?
- 풀이 과정에서 말하기, 복잡도 설명, 시간 관리가 왜 점수에 큰 영향을 줄까요?

- UMPIRE 절차

## 왜 중요한가

패턴이 없으면 면접 시간 대부분을 문제 해석에 써 버립니다. 반대로 패턴과 절차가 몸에 익으면 풀이 품질뿐 아니라 설명의 안정감도 함께 올라갑니다.

> 코딩 인터뷰 실력은 문제 수집량보다 패턴 인식, 설명 능력, 시간 감각이 함께 움직일 때 올라갑니다.

## 핵심 개념 한눈에 보기

코딩 인터뷰는 문제 풀이 시험이면서 동시에 커뮤니케이션 시험입니다. 패턴으로 접근하고, 절차에 따라 풀고, 말로 설명하고, 모의 면접과 피드백으로 다듬는 흐름이 함께 돌아야 합니다.

## 핵심 용어

- 패턴: 반복해서 등장하는 풀이 형태입니다.
- **UMPIRE**: Understand, Match, Plan, Implement, Review, Evaluate 절차입니다.
- **모의 면접**: 실제 인터뷰처럼 연습하는 방식입니다.
- **엣지 케이스**: 경계 조건입니다.
- 복잡도: 시간과 공간 비용입니다.

## 개선 전/후

**Before**: “문제를 랜덤으로 풉니다.”

**After**: “패턴별로 세 문제씩 깊게 연습합니다.”

## 직접 해보기: 인터뷰 루틴 만들기

### 1단계 — 여덟 가지 패턴

```text
two pointers, sliding window,
binary search, BFS/DFS,
heap, dp, greedy, backtracking
```

패턴 단위로 묶어 보면 새로운 문제도 낯설지 않게 들어옵니다. 무작위 풀이보다 패턴별 반복이 훨씬 빠르게 근육을 만듭니다.

### 2단계 — UMPIRE 절차

```text
Understand → Match → Plan
Implement → Review → Evaluate
```

좋은 면접 답변은 코드부터 치지 않습니다. 먼저 문제를 이해하고, 어떤 패턴인지 맞춰 보고, 계획을 말한 뒤 구현과 검토, 복잡도 평가로 이어집니다.

### 3단계 — 예시 풀이

```python
def two_sum(nums, target):
    seen = {}
    for i, n in enumerate(nums):
        if target - n in seen:
            return [seen[target - n], i]
        seen[n] = i
```

코드 자체보다 중요한 것은 이 풀이를 어떻게 설명하는가입니다. 해시맵을 써서 한 번 순회로 줄였고, 시간 복잡도는 O(n)이라는 점을 자연스럽게 말할 수 있어야 합니다.

### 4단계 — 모의 면접

```text
- twice a week, 45 minutes
- a friend or pramp.com
```

모의 면접은 실제 긴장감을 미리 연습하는 장치입니다. 혼자 푸는 연습과 달리 말하기, 시간 압박, 질문 대응까지 함께 점검할 수 있습니다.

### 5단계 — 회고

```markdown
- stuck pattern: dp
- next week: 5 dp problems + voice recording
```

회고는 약한 패턴을 드러내고 다음 주 계획을 구체화합니다. 특히 말하기가 약했다면 음성 녹음까지 붙여 보는 편이 좋습니다.

## 코딩 인터뷰 유형 비교

코딩 인터뷰는 알고리즘, 시스템 디자인, 라이브 코딩, 과제로 나눥니다. 각 유형마다 준비 전략과 평가 기준이 다릅니다.

| 유형 | 준비 전략 | 평가 기준 |
| --- | --- | --- |
| 알고리즘 | 패턴별 3문제씩, 복잡도 설명 | 정확성, 효율성, 설명 능력 |
| 시스템 디자인 | 대표 사례 5개, 트레이드오프 기록 | 제약 조건 파악, 확장 가능성, 모니터링 |
| 라이브 코딩 | 화면 공유 연습, 말하며 치기 | 커뮤니케이션, 실수 복구, 협업 자세 |
| 과제 | 요구사항 정확히 따르기, README | 코드 품질, 테스트, 문서화, 커밋 히스토리 |

알고리즘은 정해진 시간 안에 정확한 해법을 찾는 능력을 평가합니다. 시스템 디자인은 큰 그림을 그리는 능력을, 라이브 코딩은 협업 자세를, 과제는 실무 코드 품질을 봅니다.

## 45분 인터뷰 시간 배분

코딩 인터뷰는 보통 45분에서 1시간이 주어집니다. 시간을 미리 분배해 두면 막힐 때 당황하지 않습니다.

### 표준 45분 분배

| 단계 | 시간 | 할 일 |
| --- | --- | --- |
| 문제 이해 | 5분 | 제약, 예외, 예시 확인 |
| 패턴 매칭 | 3분 | 어떤 패턴인지 말하기 |
| 접근법 설명 | 5분 | 브루트포스 → 최적화 경로 설명 |
| 구현 | 25분 | 코드 작성, 말하면서 치기 |
| 검토 | 5분 | 엣지 케이스 확인, 복잡도 설명 |
| 여유 | 2분 | 질문 대비 |

```python
# 45분 인터뷰 타이머 예시
import time

def interview_timer():
    """45분 인터뷰 단계별 타이머"""
    stages = [
        ("1. 문제 이해", 5),
        ("2. 패턴 매칭", 3),
        ("3. 접근법 설명", 5),
        ("4. 구현", 25),
        ("5. 검토", 5),
        ("6. 여유", 2),
    ]

    total = 0
    for stage, minutes in stages:
        total += minutes
        print(f"{stage}: {minutes}분 (~ {total}분까지)")


interview_timer()
# 출력:
# 1. 문제 이해: 5분 (~5분까지)
# 2. 패턴 매칭: 3분 (~8분까지)
# 3. 접근법 설명: 5분 (~13분까지)
# 4. 구현: 25분 (~38분까지)
# 5. 검토: 5분 (~43분까지)
# 6. 여유: 2분 (~45분까지)
```

### 시간 부족 시 우선순위

시간이 부족하면 다음 순서로 포기하세요.

1. **여유 시간** — 제일 먼저 포기
2. **최적화** — brute force를 먼저 만들고 설명
3. **엣지 케이스** — 면접관과 함께 논의

완벽한 풀이보다 말하는 풀이가 더 좋은 평가를 받습니다.

## 파이썬 인터뷰 문제 풀이 예제 (투포인터)

투포인터는 코딩 인터뷰에서 가장 자주 나오는 패턴 중 하나입니다. 문제를 한 번 훑어보면서 해결하는 기법입니다.

### 문제: 정렬된 배열에서 타겟 합 찾기

```python
# 문제: 정렬된 배열에서 두 수의 합이 target이 되는 인덱스를 반환하세요.
# 예시: nums = [2, 7, 11, 15], target = 9
# 결과: [0, 1] (nums[0] + nums[1] = 2 + 7 = 9)

def two_sum_sorted(nums: list[int], target: int) -> list[int]:
    """
    정렬된 배열에서 투포인터로 타겟 합 찾기

    접근법:
    1. 왼쪽과 오른쪽에 포인터를 하나씩 둔다
    2. 합이 target보다 작으면 왼쪽을 오른쪽으로
    3. 합이 target보다 크면 오른쪽을 왼쪽으로

    시간 복잡도: O(n)
    공간 복잡도: O(1)
    """
    left = 0
    right = len(nums) - 1

    while left < right:
        current_sum = nums[left] + nums[right]

        if current_sum == target:
            return [left, right]
        elif current_sum < target:
            left += 1  # 합이 작으면 왼쪽을 키운다
        else:
            right -= 1  # 합이 크면 오른쪽을 줘인다

    return []  # 해가 없는 경우


# 테스트
assert two_sum_sorted([2, 7, 11, 15], 9) == [0, 1]
assert two_sum_sorted([2, 3, 4], 6) == [0, 2]
assert two_sum_sorted([1, 2, 3], 10) == []
print("Pass")
```

### 면접에서 말하는 방법

1. **문제 확인**: "배열이 정렬되어 있다고 가정해도 될까요?"
2. **패턴 설명**: "정렬된 배열이므로 투포인터로 풀겠습니다."
3. **접근법**: "왼쪽과 오른쪽에서 시작해서 합을 비교하며 좌우로 이동합니다."
4. **복잡도**: "시간 O(n), 공간 O(1)입니다."
5. **엣지 케이스**: "해가 없는 경우 빈 배열을 반환합니다."

말하면서 풀면 면접관이 사고 과정을 이해하기 쉽습니다.

## 이 예시에서 먼저 볼 점

- 패턴은 지름길입니다.
- 말하기는 평가의 일부입니다.
- 모의 면접이 실전 연습입니다.

1. **조용히 코딩만 하는 일입니다.**
2. **엣지 케이스를 놓치는 일입니다.**
3. **복잡도를 말하지 않는 일입니다.**
4. **모의 면접을 건너뛰는 일입니다.**

| 디버깅 인터뷰 | 문제 분석, 가설 검증 | 버그 재현-격리 연습 | 추측으로 고치지 않기 |

면접 준비는 문제 풀이 수집이 아니라 "평가 맥락별 루틴"을 만드는 작업입니다.

## 4주 준비 스케줄 예시

| 주차 | 핵심 목표 | 세부 실행 |
| --- | --- | --- |
| 1주차 | 패턴 지도 만들기 | 투포인터, 슬라이딩윈도우, BFS/DFS 복습 |
| 2주차 | 구현 안정성 | 제한시간 35분 실전 풀이 6회 |
| 3주차 | 말하기 강화 | 풀이 음성 녹음과 복기 |
| 4주차 | 모의 면접 | 주 2회 실전 시뮬레이션 |

준비가 잘된 후보는 정답률보다 설명 품질에서 차이가 납니다. 막히는 구간을 숨기지 않고 가설과 대안을 말할 수 있어야 면접관이 협업 가능성을 높게 평가합니다.

## 복잡도 설명 템플릿

- 접근 이유: 왜 이 패턴을 골랐는지
- 시간 복잡도: 입력 크기 n 기준 계산
- 공간 복잡도: 추가 메모리 사용량
- 대안 비교: 더 단순한 방법 대비 장단점

이 템플릿을 습관화하면 "코드를 작성하는 사람"에서 "판단을 설명하는 엔지니어"로 보입니다.

## 면접 준비 주간 운영표

코딩 인터뷰 준비는 문제 수집보다 운영 리듬이 중요합니다. 아래 표는 직장 병행 기준 4주 운영 예시입니다.

| 요일 | 60분 블록 1 | 60분 블록 2 | 기록 항목 |
| --- | --- | --- | --- |
| 월 | 배열/문자열 2문제 | 해설 복기 | 막힌 지점 3줄 |
| 화 | 해시/맵 2문제 | 시간 복잡도 설명 연습 | Big-O 음성 녹음 |
| 수 | 모의 인터뷰 1회 | 코드 리팩터링 | 개선 전/후 비교 |
| 목 | 트리/그래프 2문제 | 오답 노트 정리 | 재시도 날짜 지정 |
| 금 | 주간 종합 3문제 | 회고 | 약점 카테고리 점수 |
| 토 | 45분 실전 세트 | 리뷰 | 인터뷰 스크립트 개선 |
| 일 | 휴식/가벼운 복습 | 다음 주 계획 | 우선순위 3개 |

핵심은 매일 문제를 많이 푸는 것이 아니라, 약점 카테고리를 줄이는 것입니다. 예를 들어 "그래프에서 방문 처리 누락"이 반복된다면 문제를 늘리지 말고 동일 패턴을 3회 연속으로 재연습해야 합니다. 면접에서 불합격을 만드는 것은 지식 부족보다 반복 실수인 경우가 많습니다.

## 면접관 질문 대응 템플릿

코드 구현 후 설명 단계에서 점수가 갈립니다. 아래 템플릿으로 답변을 구조화하면 전달력이 안정됩니다.

1. 문제 재정의: 입력 조건과 예외를 한 문장으로 정리합니다.
2. 접근법 근거: 왜 이 자료구조를 택했는지 말합니다.
3. 복잡도: 시간/공간 복잡도를 수치로 제시합니다.
4. 트레이드오프: 대안 접근법과 포기한 이유를 설명합니다.
5. 검증: 경계값 테스트 2개를 직접 실행합니다.

이 구조를 고정하면 긴장 상황에서도 말이 흔들리지 않습니다. 특히 4번 트레이드오프를 말할 수 있으면 면접관은 단순 암기형 지원자가 아니라 판단 가능한 엔지니어로 인식합니다.

## 처음 질문으로 돌아가기

- **문제를 많이 푸는 것만으로 코딩 인터뷰 준비가 충분하지 않은 이유는 무엇일까요?**
  - 코딩 인터뷰의 절반은 답을 맞히는 것이고, 절반은 문제를 이해하고 경계를 명시하는 과정입니다.
- **자주 나오는 문제 패턴을 어떻게 묶어 공부해야 시간 낭비를 줄일 수 있을까요?**
  - 시간 복잡도와 공간 복잡도 중 무엇을 우선할 것인지, 어느 부분부터 구현할 것인지 명시해야 합니다.
- **풀이 과정에서 말하기, 복잡도 설명, 시간 관리가 왜 점수에 큰 영향을 줄까요?**
  - 인터뷰에서는 조용히 풀기보다 사고 과정을 말로 설명하고, 막히면 면접관과 함께 풀어가는 태도가 중요합니다.
