# Developer Career 101 (5/10): 코딩 인터뷰 준비

문제를 많이 풀었다고 해서 인터뷰에서 바로 좋은 평가를 받는 것은 아닙니다. 실제 면접에서는 풀이 아이디어를 말로 설명해야 하고, 제한 시간 안에 판단과 복잡도를 함께 보여 줘야 합니다.

이 글은 Developer Career 101 시리즈의 5번째 글입니다.

## 먼저 던지는 질문

- 문제를 많이 푸는 것만으로 코딩 인터뷰 준비가 충분하지 않은 이유는 무엇일까요?
- 자주 나오는 문제 패턴을 어떻게 묶어 공부해야 시간 낭비를 줄일 수 있을까요?
- 풀이 과정에서 말하기, 복잡도 설명, 시간 관리가 왜 점수에 큰 영향을 줄까요?

## 큰 그림

![Developer Career 101 5장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/developer-career-101/05/05-01-concept-at-a-glance.ko.png)

*Developer Career 101 5장 흐름 개요*

이 그림에서는 코딩 인터뷰 준비를 운영 흐름 안에서 어디에 배치해야 하는지 봅니다. 핵심은 개념을 따로 외우는 것이 아니라 입력, 처리, 검증, 운영 신호가 어떤 경계로 이어지는지 확인하는 데 있습니다.

> 코딩 인터뷰 준비의 핵심은 기능 이름이 아니라, 어떤 경계에서 무엇을 검증하고 어떤 신호를 남길지 정하는 데 있습니다.

## 이 글에서 배울 것

- 여덟 가지 문제 패턴
- UMPIRE 절차
- 모의 면접 루틴
- 시간 관리 방법
- 피드백을 흡수하는 방식

## 왜 중요한가

패턴이 없으면 면접 시간 대부분을 문제 해석에 써 버립니다. 반대로 패턴과 절차가 몸에 익으면 풀이 품질뿐 아니라 설명의 안정감도 함께 올라갑니다.

> 코딩 인터뷰 실력은 문제 수집량보다 패턴 인식, 설명 능력, 시간 감각이 함께 움직일 때 올라갑니다.

## 핵심 개념 한눈에 보기

코딩 인터뷰는 문제 풀이 시험이면서 동시에 커뮤니케이션 시험입니다. 패턴으로 접근하고, 절차에 따라 풀고, 말로 설명하고, 모의 면접과 피드백으로 다듬는 흐름이 함께 돌아야 합니다.

## 핵심 용어

- 패턴: 반복해서 등장하는 풀이 형태입니다.
- **UMPIRE**: Understand, Match, Plan, Implement, Review, Evaluate 절차입니다.
- **모의 면접**: 실제 인터뷰처럼 연습하는 방식입니다.
- **엣지 케이스**: 경계 조건입니다.
- 복잡도: 시간과 공간 비용입니다.

## Before/After

**Before**: “문제를 랜덤으로 풉니다.”

**After**: “패턴별로 세 문제씩 깊게 연습합니다.”

## 직접 해보기: 인터뷰 루틴 만들기

### 1단계 — 여덟 가지 패턴

```text
two pointers, sliding window,
binary search, BFS/DFS,
heap, dp, greedy, backtracking
```

패턴 단위로 묶어 보면 새로운 문제도 낯설지 않게 들어옵니다. 무작위 풀이보다 패턴별 반복이 훨씬 빠르게 근육을 만듭니다.

### 2단계 — UMPIRE 절차

```text
Understand → Match → Plan
Implement → Review → Evaluate
```

좋은 면접 답변은 코드부터 치지 않습니다. 먼저 문제를 이해하고, 어떤 패턴인지 맞춰 보고, 계획을 말한 뒤 구현과 검토, 복잡도 평가로 이어집니다.

### 3단계 — 예시 풀이

```python
def two_sum(nums, target):
    seen = {}
    for i, n in enumerate(nums):
        if target - n in seen:
            return [seen[target - n], i]
        seen[n] = i
```

코드 자체보다 중요한 것은 이 풀이를 어떻게 설명하는가입니다. 해시맵을 써서 한 번 순회로 줄였고, 시간 복잡도는 O(n)이라는 점을 자연스럽게 말할 수 있어야 합니다.

### 4단계 — 모의 면접

```text
- twice a week, 45 minutes
- a friend or pramp.com
```

모의 면접은 실제 긴장감을 미리 연습하는 장치입니다. 혼자 푸는 연습과 달리 말하기, 시간 압박, 질문 대응까지 함께 점검할 수 있습니다.

### 5단계 — 회고

```markdown
- stuck pattern: dp
- next week: 5 dp problems + voice recording
```

회고는 약한 패턴을 드러내고 다음 주 계획을 구체화합니다. 특히 말하기가 약했다면 음성 녹음까지 붙여 보는 편이 좋습니다.

## 이 예시에서 먼저 볼 점

- 패턴은 지름길입니다.
- 말하기는 평가의 일부입니다.
- 모의 면접이 실전 연습입니다.

## 자주 하는 실수 5가지

1. **조용히 코딩만 하는 일입니다.**
2. **엣지 케이스를 놓치는 일입니다.**
3. **복잡도를 말하지 않는 일입니다.**
4. **모의 면접을 건너뛰는 일입니다.**
5. **피드백을 거부하는 일입니다.**

## 실무에서는 이렇게 드러납니다

많은 회사가 내부 레벨링이나 승진 평가에서도 주기적으로 코딩 평가를 운영합니다. 인터뷰 준비 방식은 입사 이후에도 계속 쓰이는 훈련입니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 패턴은 전략입니다.
- 설명은 증거입니다.
- 시간은 제약 조건입니다.
- 모의 면접이 근육을 만듭니다.
- 피드백은 편집 도구입니다.

## 연습 문제

1. 투 포인터를 한 줄로 설명해 보세요.
2. O(n log n) 예시를 한 줄로 적어 보세요.
3. 엣지 케이스 예시를 한 줄로 적어 보세요.

## 정리

코딩 인터뷰 준비는 문제를 많이 푸는 일보다 패턴을 묶어 이해하고, 절차대로 설명하고, 피드백으로 다듬는 훈련에 더 가깝습니다. 랜덤 풀이를 줄이고 패턴·말하기·모의 면접 루틴을 함께 돌리면 실전 안정감이 빠르게 올라갑니다. 다음 글에서는 코딩 문제보다 더 큰 그림을 평가하는 시스템 디자인 인터뷰를 정리하겠습니다.

## 처음 질문으로 돌아가기

  - 본문의 기준은 코딩 인터뷰 준비를 한 덩어리 개념으로 보지 않고 입력, 처리, 검증, 운영 신호가 만나는 경계로 나누어 확인하는 것입니다.
- **자주 나오는 문제 패턴을 어떻게 묶어 공부해야 시간 낭비를 줄일 수 있을까요?**
  - 예제와 그림에서는 어떤 값이 들어오고, 어느 단계에서 바뀌며, 어떤 기준으로 통과 또는 실패하는지를 먼저 확인해야 합니다.
- **풀이 과정에서 말하기, 복잡도 설명, 시간 관리가 왜 점수에 큰 영향을 줄까요?**
