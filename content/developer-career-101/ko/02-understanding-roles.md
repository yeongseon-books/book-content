# 직무 이해하기

개발자를 하고 싶다고 말해도 실제로 마주하는 일은 모두 같지 않습니다. 사용자 경험을 다루는 사람도 있고, 데이터 흐름을 설계하는 사람도 있으며, 운영 안정성과 모델 품질을 책임지는 사람도 있습니다.

이 글은 Developer Career 101 시리즈의 2번째 글입니다.

## 이 글에서 다룰 문제

- 프론트엔드, 백엔드, 데이터, SRE, ML, 모바일 직무는 실제로 무엇이 다를까요?
- 같은 개발자라는 이름 아래에서도 책임, 도구, 지표가 왜 완전히 달라질까요?
- 지금 역할이 나와 맞는지, 아니면 전환을 고민해야 하는지 어떻게 판단할 수 있을까요?
- 새로운 직무로 옮기려면 무엇부터 파악해야 할까요?

## 이 글에서 배울 것

- 자주 만나는 여섯 가지 직무
- 직무별 핵심 책임
- 직무별 대표 도구
- 직무별 평가 지표
- 직무 전환 때 따져 볼 기준

## 왜 중요한가

맞지 않는 직무를 오래 붙잡으면 성장보다 소진이 먼저 옵니다. 반대로 책임과 도구, 지표의 차이를 정확히 이해하면 지금의 역할을 더 잘 해낼 수도 있고, 전환이 필요할 때도 훨씬 전략적으로 움직일 수 있습니다.

> 좋은 직무 선택은 어떤 도구를 쓰는가보다 어떤 책임을 기꺼이 맡을 수 있는가에서 시작합니다.

## 핵심 개념 한눈에 보기

![주요 개발 직무 비교 흐름](https://yeongseon-books.github.io/book-public-assets/assets/developer-career-101/02/02-01-concept-at-a-glance.ko.png)

*주요 개발 직무 비교 흐름*

직무를 가르는 첫 기준은 도구가 아니라 책임입니다. 도구는 바뀔 수 있지만, 무엇을 최적화하고 어떤 실패를 가장 먼저 책임지는지는 직무의 성격을 오래 결정합니다.

## 핵심 용어

- 프론트엔드: 사용자 경험을 책임지는 영역입니다.
- 백엔드: 시스템 로직과 데이터 흐름을 책임지는 영역입니다.
- **SRE**: 운영 안정성과 신뢰성을 책임지는 역할입니다.
- 데이터: 분석과 파이프라인을 다루는 역할입니다.
- **ML**: 모델 기반 예측 품질을 다루는 역할입니다.

## Before/After

**Before**: “직무는 대체로 비슷해 보입니다.”

**After**: “책임과 도구, 지표가 완전히 다르다는 걸 압니다.”

## 직접 해보기: 직무 비교표 만들기

### 1단계 — Frontend

```text
responsibility: UX
tools: React, CSS
metrics: LCP, INP
```

프론트엔드는 사용자가 바로 체감하는 속도와 흐름을 다룹니다. 같은 기능이라도 체감 성능과 인터랙션 품질이 좋지 않으면 제품 전체 인상이 무너질 수 있습니다.

### 2단계 — Backend

```text
responsibility: data flow
tools: Python, SQL
metrics: p95, error rate
```

백엔드는 기능이 실제로 돌아가게 만드는 기반입니다. 응답 시간, 오류율, 데이터 정합성 같은 운영 지표를 함께 봐야 역할이 선명해집니다.

### 3단계 — Data

```text
responsibility: pipelines
tools: Airflow, dbt
metrics: freshness, accuracy
```

데이터 직무는 데이터를 쌓는 일보다 신뢰할 수 있게 흐르게 만드는 일이 중요합니다. 분석이든 모델 학습이든 입력 품질이 흔들리면 이후 단계가 전부 흔들립니다.

### 4단계 — SRE

```text
responsibility: operations
tools: Prometheus, K8s
metrics: SLO, MTTR
```

SRE는 장애를 덜 나게 만드는 역할이 아니라, 장애를 다루는 기준을 팀 차원에서 정하는 역할에 가깝습니다. 신뢰성과 운영 자동화 감각이 핵심입니다.

### 5단계 — ML

```text
responsibility: model quality
tools: PyTorch, MLflow
metrics: AUC, latency
```

ML은 모델 정확도만 보는 역할이 아닙니다. 예측 품질과 지연 시간, 데이터 품질, 재현성과 배포 운영까지 함께 다뤄야 실제 서비스에서 의미가 생깁니다.

## 직무를 고를 때 보는 판단 기준

| 기준 | 프론트엔드 | 백엔드 | 데이터/SRE/ML |
| --- | --- | --- | --- |
| 가장 자주 만나는 실패 | 느린 화면, 깨진 상호작용 | 오류율 상승, 데이터 정합성 문제 | 신선도 저하, 장애 대응, 모델 품질 저하 |
| 잘했을 때 드러나는 신호 | 사용자가 흐름을 덜 헤맨다 | 기능이 안정적으로 오래 간다 | 운영 비용과 리스크가 함께 줄어든다 |
| 초반에 확인할 증거 | 성능 지표, 디자인 협업 방식 | API 계약, 저장소 구조, 운영 지표 | 배치/실시간 흐름, 온콜 구조, 재현성 도구 |
| 나와 맞는지 보는 질문 | UI 디테일을 오래 다듬는 일이 즐거운가 | 데이터 흐름과 경계 조건을 좋아하는가 | 품질 기준을 수치로 다루는 일이 편한가 |

## 이 예시에서 먼저 볼 점

- 책임이 직무를 정의합니다.
- 지표가 그 직무의 문화와 우선순위를 보여 줍니다.
- 도구는 수단일 뿐입니다.

## 자주 하는 실수 5가지

1. **도구만 보고 직무를 고르는 일입니다.**
2. **핵심 지표를 모르는 일입니다.**
3. **직무 경계를 충동적으로 넘나드는 일입니다.**
4. **전환을 가볍게 결정하는 일입니다.**
5. **도메인 차이를 무시하는 일입니다.**

## 실무에서는 이렇게 드러납니다

많은 회사가 직무 전환에 약 6개월 정도의 적응 기간을 잡습니다. 사용하는 기술보다 책임과 판단 기준이 달라지기 때문입니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 직무는 도구가 아니라 책임입니다.
- 지표는 그 역할의 언어입니다.
- 전환은 전략적으로 해야 합니다.
- 도메인은 깊이를 만듭니다.
- 경계가 있어야 협업도 선명해집니다.

## 체크리스트

- [ ] 현재 직무의 핵심 지표 세 가지를 적었습니다.
- [ ] 관심 있는 직무의 책임을 적었습니다.
- [ ] 직무 전환 학습 계획을 잡았습니다.

## 정리

직무를 이해한다는 말은 이름을 외우는 일이 아니라 각 역할이 책임지는 문제와 성공 기준을 구분하는 일입니다. 무엇을 만들고 싶은지보다 어떤 책임을 오래 맡고 싶은지를 먼저 보면 선택이 훨씬 선명해집니다. 다음 글에서는 바쁜 일상 안에서도 지속 가능한 학습 계획을 어떻게 세우는지 정리하겠습니다.
