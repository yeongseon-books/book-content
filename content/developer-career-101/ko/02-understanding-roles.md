# 직무 이해하기

## 이 글에서 다룰 문제

- 프론트엔드, 백엔드, 데이터, SRE, ML, 모바일 직무는 실제로 무엇이 다를까요?
- 같은 개발자라는 이름 아래에서도 책임, 도구, 평가 지표가 왜 완전히 달라질까요?
- 지금 역할이 나와 맞는지, 아니면 전환을 고민해야 하는지 어떻게 판단할 수 있을까요?
- 관심 있는 직무로 옮기려면 무엇부터 배워야 할까요?

> Developer Career 101 시리즈 (2/10)

직무를 고를 때 기술 스택 이름만 보고 결정하면 꽤 자주 후회합니다. React가 재미있어서 프론트엔드를 택했는데 사용자 경험과 브라우저 디버깅이 맞지 않을 수도 있고, Python을 좋아해서 데이터 직무를 생각했는데 실제로는 파이프라인 운영과 품질 검증이 더 큰 비중일 수도 있습니다.

직무는 도구보다 책임으로 구분하는 편이 정확합니다. 어떤 문제를 해결하고, 어떤 지표를 지키며, 어떤 실패를 줄이는지가 직무의 본체입니다. 이 글에서는 그 기준으로 주요 역할을 비교해 보겠습니다.

## 왜 이 주제가 중요한가

직무 적합도가 맞지 않으면 성장도 느리고 소진도 빨라집니다. 반대로 내 성향과 직무의 책임 구조가 잘 맞으면 학습 속도와 성취감이 함께 올라갑니다. 이직이나 커리어 전환을 고민할 때도 먼저 알아야 하는 것은 기술 목록이 아니라 역할의 성격입니다.

> 좋은 직무 선택은 어떤 도구를 쓸지보다 어떤 책임을 기꺼이 떠안을 수 있는지에서 시작합니다.

## 핵심 개념 한눈에 보기

```mermaid
flowchart LR
    F[프론트엔드] --> B[백엔드]
    B --> D[데이터]
    D --> S[SRE]
    S --> M[ML]
    M --> Mo[모바일]
```

화살표는 우열이 아니라 관점의 이동을 뜻합니다. 사용자 화면에 가까운 역할이 있고, 시스템 내부 흐름을 책임지는 역할이 있으며, 운영 안정성이나 모델 품질처럼 다른 축을 보는 역할도 있습니다. 어느 직무가 더 낫다기보다, 문제를 바라보는 초점이 다르다고 이해하면 됩니다.

## 핵심 용어

- **프론트엔드**: 사용자 경험과 인터랙션 품질을 책임지는 역할입니다.
- **백엔드**: 데이터 흐름, 비즈니스 로직, API 안정성을 책임지는 역할입니다.
- **데이터 엔지니어링**: 데이터 파이프라인과 분석 신뢰성을 만드는 역할입니다.
- **SRE**: 시스템 운영 안정성과 복구 가능성을 높이는 역할입니다.
- **ML 엔지니어링**: 모델 학습과 서빙 품질을 제품 수준으로 끌어올리는 역할입니다.

## Before / After

**Before**: 개발자 직무는 다 비슷하고, 쓰는 도구만 조금 다르다고 생각합니다.

**After**: 직무마다 책임, 주요 지표, 협업 방식이 뚜렷하게 다르다는 사실을 이해합니다.

## 직접 해보기: 직무 비교표 만들기

### 1단계 — 프론트엔드 보기

```text
responsibility: UX
tools: React, CSS
metrics: LCP, INP
```

프론트엔드는 눈에 보이는 화면만 다루는 직무가 아닙니다. 로딩 속도, 상호작용 품질, 접근성처럼 사용자가 직접 체감하는 문제를 다룹니다.

### 2단계 — 백엔드 보기

```text
responsibility: data flow
tools: Python, SQL
metrics: p95, error rate
```

백엔드는 요청을 정확하고 빠르게 처리하는 구조를 만듭니다. 응답 지연, 오류율, 데이터 일관성처럼 겉으로 보이지 않는 품질이 핵심입니다.

### 3단계 — 데이터 직무 보기

```text
responsibility: pipelines
tools: Airflow, dbt
metrics: freshness, accuracy
```

데이터 직무는 대시보드 한 장을 만드는 일을 넘어서, 데이터가 제때 들어오고 믿을 수 있게 유지되도록 전체 흐름을 설계합니다.

### 4단계 — SRE 보기

```text
responsibility: operations
tools: Prometheus, K8s
metrics: SLO, MTTR
```

SRE는 장애가 나지 않게 하는 일만 하지 않습니다. 장애가 나더라도 빨리 탐지하고 복구할 수 있는 구조를 만드는 역할입니다.

### 5단계 — ML 직무 보기

```text
responsibility: model quality
tools: PyTorch, MLflow
metrics: AUC, latency
```

ML 직무에서는 모델 정확도만으로 충분하지 않습니다. 서빙 지연 시간, 실험 재현성, 데이터 드리프트 대응까지 함께 고려해야 합니다.

## 이 예시에서 읽어야 할 포인트

- 직무의 본질은 책임이며, 도구는 그 책임을 수행하기 위한 수단입니다.
- 각 직무가 중요하게 보는 지표를 알면 그 조직 문화도 함께 읽을 수 있습니다.
- 직무 전환은 기술 스택 변경이 아니라 문제 공간 변경에 가깝습니다.

## 자주 하는 실수 5가지

1. **도구 이름만 보고 직무를 고르는 실수**: 좋아하는 프레임워크와 맞는 직무는 다를 수 있습니다.
2. **핵심 지표를 모르는 실수**: 어떤 숫자를 지켜야 하는지 모르면 역할 이해가 반쪽짜리입니다.
3. **직무 경계를 무시하는 실수**: 협업이 필요한 지점과 책임 경계를 구분하지 못하게 됩니다.
4. **충동적으로 전환하는 실수**: 흥미만으로 옮기면 적응 비용을 과소평가하기 쉽습니다.
5. **도메인을 가볍게 보는 실수**: 기술이 같아도 도메인이 달라지면 요구되는 판단이 크게 달라집니다.

## 실무에서는 이렇게 이어집니다

많은 회사가 직무 전환에 최소 몇 개월의 온보딩을 잡는 이유가 여기에 있습니다. 역할이 바뀌면 기술만 새로 배우는 것이 아니라, 성공 기준과 실패 비용을 다시 익혀야 하기 때문입니다.

## 시니어는 이렇게 생각합니다

- 직무는 도구가 아니라 책임의 묶음입니다.
- 지표는 그 역할이 세상을 읽는 언어입니다.
- 전환은 전략적으로 해야 합니다.
- 도메인 이해가 쌓여야 깊이가 생깁니다.
- 경계를 이해해야 협업도 더 잘됩니다.

## 체크리스트

- [ ] 현재 직무의 핵심 지표 세 가지를 적었다.
- [ ] 관심 있는 직무의 책임과 산출물을 정리했다.
- [ ] 직무 전환 시 필요한 학습 항목을 목록으로 만들었다.

## 정리 및 다음 글

직무를 이해한다는 것은 프레임워크 목록을 외우는 일이 아니라, 어떤 책임을 맡고 어떤 실패를 줄여야 하는지 파악하는 일입니다. 내 성향과 잘 맞는 책임 구조를 찾으면 커리어 선택이 훨씬 선명해집니다.

다음 글에서는 이런 직무 이해를 바탕으로, 실제로 지속 가능한 학습 계획을 어떻게 세우는지 살펴보겠습니다.
