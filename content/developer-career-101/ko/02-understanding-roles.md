# Developer Career 101 (2/10): 직무 이해하기

개발자를 하고 싶다고 말해도 실제로 마주하는 일은 모두 같지 않습니다. 사용자 경험을 다루는 사람도 있고, 데이터 흐름을 설계하는 사람도 있으며, 운영 안정성과 모델 품질을 책임지는 사람도 있습니다.

이 글은 Developer Career 101 시리즈의 2번째 글입니다.

![Developer Career 101 2장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/developer-career-101/02/02-01-concept-at-a-glance.ko.png)
*Developer Career 101 2장 흐름 개요*
> 직무를 선택할 때 중요한 것은 회사 규모가 아니라, 주어지는 문제의 크기와 그 문제를 혼자 풀 수 있는지, 팀과 함께 풀어야 하는지입니다.

## 먼저 던지는 질문

- 프론트엔드, 백엔드, 데이터, SRE, ML, 모바일 직무는 실제로 무엇이 다를까요?
- 같은 개발자라는 이름 아래에서도 책임, 도구, 지표가 왜 완전히 달라질까요?
- 지금 역할이 나와 맞는지, 아니면 전환을 고민해야 하는지 어떻게 판단할 수 있을까요?

## 이 글에서 배울 것

- 자주 만나는 여섯 가지 직무
- 직무별 핵심 책임
- 직무별 대표 도구
- 직무별 평가 지표
- 직무 전환 때 따져 볼 기준

## 왜 중요한가

맞지 않는 직무를 오래 붙잡으면 성장보다 소진이 먼저 옵니다. 반대로 책임과 도구, 지표의 차이를 정확히 이해하면 지금의 역할을 더 잘 해낼 수도 있고, 전환이 필요할 때도 훨씬 전략적으로 움직일 수 있습니다.

> 좋은 직무 선택은 어떤 도구를 쓰는가보다 어떤 책임을 기꺼이 맡을 수 있는가에서 시작합니다.

## 핵심 개념 한눈에 보기

직무를 가르는 첫 기준은 도구가 아니라 책임입니다. 도구는 바뀔 수 있지만, 무엇을 최적화하고 어떤 실패를 가장 먼저 책임지는지는 직무의 성격을 오래 결정합니다.

## 핵심 용어

- 프론트엔드: 사용자 경험을 책임지는 영역입니다.
- 백엔드: 시스템 로직과 데이터 흐름을 책임지는 영역입니다.
- **SRE**: 운영 안정성과 신뢰성을 책임지는 역할입니다.
- 데이터: 분석과 파이프라인을 다루는 역할입니다.
- **ML**: 모델 기반 예측 품질을 다루는 역할입니다.

## 개선 전/후

**Before**: “직무는 대체로 비슷해 보입니다.”

**After**: “책임과 도구, 지표가 완전히 다르다는 걸 압니다.”

## 직접 해보기: 직무 비교표 만들기

### 1단계 — Frontend

```text
responsibility: UX
tools: React, CSS
metrics: LCP, INP
```

프론트엔드는 사용자가 바로 체감하는 속도와 흐름을 다룹니다. 같은 기능이라도 체감 성능과 인터랙션 품질이 좋지 않으면 제품 전체 인상이 무너질 수 있습니다.

### 2단계 — Backend

```text
responsibility: data flow
tools: Python, SQL
metrics: p95, error rate
```

백엔드는 기능이 실제로 돌아가게 만드는 기반입니다. 응답 시간, 오류율, 데이터 정합성 같은 운영 지표를 함께 봐야 역할이 선명해집니다.

### 3단계 — Data

```text
responsibility: pipelines
tools: Airflow, dbt
metrics: freshness, accuracy
```

데이터 직무는 데이터를 쌓는 일보다 신뢰할 수 있게 흐르게 만드는 일이 중요합니다. 분석이든 모델 학습이든 입력 품질이 흔들리면 이후 단계가 전부 흔들립니다.

### 4단계 — SRE

```text
responsibility: operations
tools: Prometheus, K8s
metrics: SLO, MTTR
```

SRE는 장애를 덜 나게 만드는 역할이 아니라, 장애를 다루는 기준을 팀 차원에서 정하는 역할에 가깝습니다. 신뢰성과 운영 자동화 감각이 핵심입니다.

### 5단계 — ML

```text
responsibility: model quality
tools: PyTorch, MLflow
metrics: AUC, latency
```

ML은 모델 정확도만 보는 역할이 아닙니다. 예측 품질과 지연 시간, 데이터 품질, 재현성과 배포 운영까지 함께 다뤄야 실제 서비스에서 의미가 생깁니다.

## 직무를 고를 때 보는 판단 프레임워크

| 기준 | 프론트엔드 | 백엔드 | 데이터/SRE/ML |
| --- | --- | --- | --- |
| 가장 자주 만나는 실패 | 느린 화면, 깨진 상호작용 | 오류율 상승, 데이터 정합성 문제 | 신선도 저하, 장애 대응, 모델 품질 저하 |
| 잘했을 때 드러나는 신호 | 사용자가 흐름을 덜 헤맨다 | 기능이 안정적으로 오래 간다 | 운영 비용과 리스크가 함께 줄어든다 |
| 초반에 확인할 증거 | 성능 지표, 디자인 협업 방식 | API 계약, 저장소 구조, 운영 지표 | 배치/실시간 흐름, 온콜 구조, 재현성 도구 |
| 나와 맞는지 보는 질문 | UI 디테일을 오래 다듬는 일이 즐거운가 | 데이터 흐름과 경계 조건을 좋아하는가 | 품질 기준을 수치로 다루는 일이 편한가 |

## 직무 비교 표

주요 개발 직무를 핵심 기술, 시장 수요, 진입 난이도를 기준으로 비교하면 본인의 적성과 시장 현실을 함께 볼 수 있습니다.

| 직무 | 핵심 기술 | 시장 수요 | 진입 난이도 |
| --- | --- | --- | --- |
| 프론트엔드 | React, TypeScript, CSS, 빌드 도구 | 높음 | 중간 (학습 진입장벽 낮음) |
| 백엔드 | Python/Java, SQL, REST API, 테스트 | 매우 높음 | 중간 (언어 선택지 다양) |
| 풀스택 | 프론트+백엔드 모두 | 중간 (스타트업 선호) | 높음 (넓은 범위) |
| DevOps/SRE | Kubernetes, Terraform, 모니터링 | 높음 | 높음 (운영 경험 필요) |
| 데이터 엔지니어 | Airflow, dbt, SQL, Spark | 중간 | 중간 (통계 기초 필요) |
| ML 엔지니어 | PyTorch, MLflow, 모델 배포 | 중간 (성장 중) | 높음 (수학+코딩) |

프론트엔드와 백엔드는 시장 수요가 가장 높고, 학습 자료도 풍부합니다. 풀스택은 넓은 범위를 경험할 수 있지만, 깊이가 없다는 평가를 받기 쉽습니다. DevOps와 ML은 진입 난이도가 높지만, 단단한 기초를 쌓으면 경쟁력이 오래 유지됩니다.

## T자형 역량

T자형 인재는 한 분야는 깊게(세로), 나머지는 넓게(가로) 아는 구조입니다. 예를 들어, 백엔드를 깊게 알면서 프론트엔드와 데이터 파이프라인 용어를 함께 이해하는 사람을 말합니다.

### 세로: 깊이

한 분야에서 경계 조건, 성능 병목, 운영 이슈를 직접 해결할 수 있는 수준입니다.

```python
# 백엔드 깊이 예시: 커넥션 풀 튜닝
from sqlalchemy.pool import QueuePool
from sqlalchemy import create_engine

engine = create_engine(
    DATABASE_URL,
    poolclass=QueuePool,
    pool_size=10,
    max_overflow=20,
    pool_pre_ping=True
)

# p95 응답 시간이 느릴 때 커넥션 풀 크기와 초과 한도를 조절합니다.
```

### 가로: 넓이

다른 직무의 용어와 우선순위를 이해하고, 협업할 때 빨리 조율하는 능력입니다.

- 프론트엔드가 말하는 LCP, INP를 이해하고, API 응답 크기를 줄여 협업합니다.
- 데이터 팀이 요청하는 비정규화 로그 형식을 제공하며 협업합니다.

깊이가 없으면 문제를 끝까지 풀 수 없고, 넓이가 없으면 협업 속도가 느려집니다. T자형 구조는 변화하는 기술 스택에도 버틸 수 있게 합니다.

## 직무 전환 전략

직무를 바꾸는 것은 새 언어를 배우는 일보다 더 많은 시간이 걸립니다. 책임과 지표가 달라지기 때문입니다.

### 1단계 — 현재 직무에서 검증

백엔드에서 데이터 엔지니어로 전환하고 싶다면, 지금 회사에서 Airflow 파이프라인을 먼저 만들어 보세요.

```python
# 백엔드에서 데이터 엔지니어로 전환 예시
# 1단계: 현재 회사에서 작은 데이터 파이프라인 구축
from airflow import DAG
from airflow.operators.python import PythonOperator

def extract_orders():
    # 주문 데이터 추출
    pass

dag = DAG("daily_orders", schedule="@daily")
task = PythonOperator(task_id="extract", python_callable=extract_orders, dag=dag)
```

### 2단계 — 3개월 프로젝트

이력서에 넣을 수 있는 프로젝트를 하나 완성하세요. 사이드 프로젝트나 회사 내부 작업이면 충분합니다.

```markdown
## 프로젝트: 일리 주문 데이터 파이프라인

- 목표: 주문 데이터를 매일 추출하여 DW에 적재
- 기간: 3개월
- 결과: 매일 오전 9시에 신선한 데이터 제공
```

### 3단계 — 내부 전환 또는 이직

회사가 내부 이동을 지원하면 먼저 시도하고, 그렇지 않다면 포트폴리오를 갖춘 상태에서 외부 이직을 진행하세요.

전환은 6개월에서 1년이 걸릴 수 있습니다. 서두르지 말고, 현재 직무에서 경험을 먼저 만드세요.

## 이 예시에서 먼저 볼 점

- 책임이 직무를 정의합니다.
- 지표가 그 직무의 문화와 우선순위를 보여 줍니다.
- 도구는 수단일 뿐입니다.

## 자주 하는 실수 5가지

1. **도구만 보고 직무를 고르는 일입니다.**
2. **핵심 지표를 모르는 일입니다.**
3. **직무 경계를 충동적으로 넘나드는 일입니다.**
4. **전환을 가볍게 결정하는 일입니다.**
5. **도메인 차이를 무시하는 일입니다.**

## 실무에서는 이렇게 드러납니다

많은 회사가 직무 전환에 약 6개월 정도의 적응 기간을 잡습니다. 사용하는 기술보다 책임과 판단 기준이 달라지기 때문입니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 직무는 도구가 아니라 책임입니다.
- 지표는 그 역할의 언어입니다.
- 전환은 전략적으로 해야 합니다.
- 도메인은 깊이를 만듭니다.
- 경계가 있어야 협업도 선명해집니다.

## 체크리스트

- [ ] 관심 있는 직무의 책임을 적었습니다.
- [ ] 직무 전환 학습 계획을 잡았습니다.

## 기술 스택 선택 의사결정 표

직무를 선택할 때 특정 프레임워크의 인기만 보고 결정하면 1~2년 뒤 방향을 다시 틀어야 하는 경우가 많습니다. 기술 스택은 유행보다 문제 적합성과 팀 운영성, 학습 곡선까지 함께 비교해야 합니다.

| 판단 항목 | 질문 | 좋은 신호 | 경고 신호 |
| --- | --- | --- | --- |
| 문제 적합성 | 지금 풀 문제에 이 스택이 직접적인가 | 요구사항과 기능이 명확히 맞음 | 단지 "요즘 많이 씀" |
| 팀 운영성 | 팀 내 리뷰·운영 경험이 있는가 | 사내 레퍼런스와 온콜 경험 존재 | 혼자만 이해하는 구조 |
| 학습 곡선 | 3개월 내 실무 투입 가능한가 | 문서/예제/멘토가 충분 | 입문 자료가 부족 |
| 채용 시장성 | 채용 공고에서 지속 수요가 보이는가 | 여러 산업에서 반복 등장 | 특정 회사에서만 사용 |
| 장기 유지보수 | 2년 뒤 교체 비용이 감당 가능한가 | 표준 생태계, 버전 정책 안정 | 업데이트 중단 위험 |

이 표의 목적은 완벽한 답을 찾는 것이 아니라, 의사결정의 근거를 명확히 남기는 데 있습니다. 스택 선택이 어려울수록 "왜 이 선택을 했는지"를 문장으로 기록해야 미래의 자신이 판단을 수정하기 쉽습니다.

## 시장 수요를 읽는 기본 프레임

시장 수요 분석은 단순히 공고 건수만 세는 작업이 아닙니다. 다음 세 가지를 함께 봐야 실제 수요를 이해할 수 있습니다.

- 공고 밀도: 기술 키워드가 얼마나 자주 반복되는가
- 역할 깊이: 단순 사용 경험인지 운영·아키텍처 수준까지 요구하는가
- 도메인 확산: 특정 산업에 갇혀 있는지 여러 산업으로 확산되는가

예를 들어 "Python"은 공고 수 자체가 많고, 백엔드·데이터·자동화 등 도메인 확산도 넓습니다. 반면 특정 도구는 공고 수는 많아도 유지보수 중심 수요만 강할 수 있습니다. 따라서 단일 지표로 판단하지 말고, 공고 문장 안의 책임 범위를 함께 읽어야 합니다.

## 직무 전환 시뮬레이션 예시

| 현재 역할 | 전환 목표 | 90일 준비 항목 | 리스크 완화 |
| --- | --- | --- | --- |
| 프론트엔드 | 백엔드 | API 설계, SQL, 인증 기본 | 작은 내부 API 기여 |
| 백엔드 | 데이터 | ETL, 모델링, 품질 검증 | 주간 리포트 자동화 |
| 데이터 | ML 엔지니어 | 실험 추적, 배포 파이프라인 | 미니 모델 서비스화 |

전환의 핵심은 "완벽히 준비된 뒤 이동"이 아니라, 현재 역할 안에서 목표 역할의 작은 책임을 먼저 가져오는 것입니다. 이 과정을 통해 실제 적합성도 확인하고, 이력서에 남길 증거도 동시에 확보할 수 있습니다.

## 실천 메모

이 섹션은 본문의 핵심을 실무 행동으로 전환하기 위한 메모입니다. 이번 주에 바로 실행할 한 가지를 정하고, 실행 결과를 수치 또는 관찰 가능한 문장으로 남기면 다음 회고에서 방향을 빠르게 수정할 수 있습니다. 계획은 길수록 좋은 것이 아니라, 반복 가능한 행동으로 내려올수록 강해집니다.

## 채용 공고 기반 직무 적합도 매트릭스

직무를 선택할 때는 느낌보다 데이터가 필요합니다. 같은 백엔드 포지션이라도 요구하는 시스템 복잡도, 협업 방식, 장애 대응 강도가 크게 다르기 때문입니다. 아래 매트릭스는 공고 세 개를 비교할 때 쓰는 최소 기준입니다.

| 평가 축 | 질문 | 1점 기준 | 3점 기준 | 5점 기준 |
| --- | --- | --- | --- | --- |
| 문제 난이도 | 어떤 종류의 문제를 푸는가 | CRUD 중심 | API + 배치 혼합 | 분산 시스템/성능 최적화 |
| 운영 책임 | 온콜/장애 대응 범위는 | 없음 | 월 1회 로테이션 | 주 단위 로테이션 + RCA 필수 |
| 협업 밀도 | 타 직군 협업 빈도는 | 요청 기반 | 주간 정렬 미팅 | 일일 동기화 + 의사결정 문서화 |
| 성장 경로 | 1년 뒤 역할 확장성은 | 동일 업무 반복 | 부분 리드 경험 | 서비스 오너십 부여 |

이 매트릭스로 점수를 만든 뒤에는 직무별로 가중치를 다르게 둡니다. 예를 들어 데이터 엔지니어 전환을 원하면 운영 책임보다 데이터 모델링 비중을 높입니다. 프론트엔드 전환을 원하면 협업 밀도와 제품 실험 속도 비중을 높입니다. 같은 표를 쓰더라도 가중치 설정이 목표를 반영하도록 설계해야 합니다.

## 직무 전환 로드맵 타임라인

전환은 한 번의 결심으로 끝나지 않습니다. 현재 역할의 신뢰를 유지하면서 점진적으로 새 역할 증거를 쌓아야 합니다.

| 기간 | 실행 항목 | 결과물 | 관리자와 합의할 내용 |
| --- | --- | --- | --- |
| 1-2개월 | 현재 팀에서 인접 과제 참여 | 데이터 파이프라인 보조 태스크 1개 | 업무 시간 20% 전환 학습 허용 |
| 3-4개월 | 작은 주도 과제 수행 | 운영 문서 + 대시보드 구축 | 평가 지표를 기존 직무와 분리 |
| 5-6개월 | 공식 역할 조정 제안 | 직무 설명서 초안, 기대 역량 표 | 다음 반기 목표에 전환 항목 반영 |

이 과정에서 중요한 것은 “새로운 일을 했다”가 아니라 “새로운 역할에서 재현 가능한 성과를 냈다”는 증거입니다. 따라서 프로젝트 결과를 수치로 남겨야 합니다. 예를 들어 “배치 실패율 12%→3%” 같은 수치가 있으면 전환 협의가 훨씬 현실적으로 진행됩니다.

## 정리

직무를 이해한다는 말은 이름을 외우는 일이 아니라 각 역할이 책임지는 문제와 성공 기준을 구분하는 일입니다. 무엇을 만들고 싶은지보다 어떤 책임을 오래 맡고 싶은지를 먼저 보면 선택이 훨씬 선명해집니다. 다음 글에서는 바쁜 일상 안에서도 지속 가능한 학습 계획을 어떻게 세우는지 정리하겠습니다.

## 처음 질문으로 돌아가기

- **프론트엔드, 백엔드, 데이터, SRE, ML, 모바일 직무는 실제로 무엇이 다를까요?**
  - 각 직무는 맡는 문제의 종류, 사용하는 도구, 성공을 측정하는 지표가 다릅니다. 프론트엔드는 UI·성능, 백엔드는 기능·안정성, 데이터는 정확성·활용성을 우선합니다.
- **같은 개발자라는 이름 아래에서도 책임, 도구, 지표가 왜 완전히 달라질까요?**
  - 직급은 같지만 문제의 크기, 책임 범위, 요구되는 커뮤니케이션 방식이 다릅니다. 이를 명시적으로 이해하면 자신의 강점과 약점이 명확해집니다.
- **지금 역할이 나와 맞는지, 아니면 전환을 고민해야 하는지 어떻게 판단할 수 있을까요?**
  - 현재 직무의 성공 지표를 정의하고, 잘하는 일과 어려운 일을 구분한 후, 그 기준으로 전환 시기와 학습 계획을 결정하면 됩니다.
