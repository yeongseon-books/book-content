# 첫 직장 적응

## 이 글에서 다룰 문제

- 첫 개발자 직장에서 처음 90일은 무엇에 집중해야 할까요?
- 많이 아는 척하기보다 질문하고 기록하는 태도가 왜 더 중요할까요?
- 첫 PR과 1:1은 어떻게 신뢰를 만드는 장치가 될까요?
- 온보딩 초기에 자주 하는 실수는 무엇이고, 어떻게 줄일 수 있을까요?

> Developer Career 101 시리즈 (7/10)

첫 직장에 들어가면 누구나 긴장합니다. 빨리 기여하고 싶고, 모르는 것을 드러내고 싶지 않기도 합니다. 하지만 온보딩 초반에는 똑똑해 보이는 것보다 학습 속도를 높이는 편이 훨씬 중요합니다. 현업에서는 모르는 것을 숨기는 사람보다, 질문을 정리하고 작은 기여를 쌓는 사람이 더 빨리 신뢰를 얻습니다.

이 글에서는 첫 90일을 학습, 질문, 작은 기여, 피드백의 흐름으로 나누어 보겠습니다. 처음부터 큰 성과를 내려 하기보다, 오래 갈 수 있는 습관을 만드는 것이 핵심입니다.

## 왜 이 주제가 중요한가

첫인상은 생각보다 오래 갑니다. 초기에 어떤 태도로 일했는지, 질문을 어떻게 했는지, 협업을 얼마나 성실하게 했는지가 이후 1년 이상 평가에 영향을 주는 경우가 많습니다. 그래서 첫 90일은 성과 경쟁보다 적응 전략이 중요합니다.

> 첫 직장의 초반 90일은 실력을 증명하는 시기라기보다, 배우는 방식과 협업 태도를 보여 주는 시기입니다.

## 핵심 개념 한눈에 보기

```mermaid
flowchart LR
    L[학습] --> A[질문]
    A --> C[기여]
    C --> F[피드백]
```

이 흐름은 단순하지만 실전적입니다. 먼저 맥락을 배우고, 질문을 모으고, 작은 PR로 기여하고, 피드백을 다시 학습으로 연결해야 적응이 빨라집니다. 이 순서를 무시하고 바로 큰 기능부터 맡으려 하면 오히려 불안만 커질 수 있습니다.

## 핵심 용어

- **온보딩**: 새로운 조직과 코드베이스에 적응하는 기간입니다.
- **버디**: 가까운 거리에서 질문을 받아 주는 동료형 안내자입니다.
- **1:1**: 매니저나 멘토와 정기적으로 대화하는 시간입니다.
- **섀도잉**: 다른 사람의 작업이나 운영 흐름을 옆에서 관찰하며 배우는 방식입니다.
- **심리적 안전감**: 모르는 것을 말해도 괜찮다고 느끼는 팀 분위기입니다.

## Before / After

**Before**: 혼자 공부만 하면서 실수하지 않기 위해 조용히 버팁니다.

**After**: 질문을 정리해 묻고, 작은 PR을 보내며, 1:1에서 피드백을 주고받습니다.

## 직접 해보기: 첫 90일 계획 세우기

### 1단계 — 30일 차: 배우기

```text
- explore the codebase
- shadow an oncall
- write a glossary
```

초기 한 달은 빨리 이해하려는 욕심보다 꾸준히 구조를 익히는 태도가 중요합니다. 용어집을 만들면 회의에서 나오는 약어와 배경 맥락이 훨씬 빨리 잡힙니다.

### 2단계 — 60일 차: 질문 정리하기

```markdown
## question notebook
- why is X faster than Y?
- what is Z's retention policy?
```

질문을 바로바로 던지는 것도 좋지만, 한 번 정리해서 묻는 습관이 더 좋습니다. 맥락이 붙고, 상대 시간도 아낄 수 있으며, 나중에 같은 질문을 반복하지 않게 됩니다.

### 3단계 — 90일 차: 첫 PR 보내기

```bash
git checkout -b first-fix
# 범위를 작게 잡은 변경
```

첫 PR은 작을수록 좋습니다. 오탈자 수정, 로그 개선, 작은 버그 수정처럼 범위가 명확한 작업이 코드 리뷰 흐름과 배포 문화를 익히는 데 더 도움이 됩니다.

### 4단계 — 1:1 활용하기

```text
weekly with manager, biweekly with mentor
```

1:1은 단순 보고 시간이 아닙니다. 막히는 지점, 기대치, 우선순위를 조정하는 중요한 채널입니다. 초반일수록 더 자주 활용하는 편이 좋습니다.

### 5단계 — 회고 남기기

```markdown
- went well: code reading
- gap: question frequency
- next: one question per day
```

처음 90일의 회고는 앞으로의 적응 속도를 바꿉니다. 무엇이 잘 먹혔고, 어떤 방식이 잘 안 맞았는지 기록해 두면 다음 조직에서도 큰 자산이 됩니다.

## 이 예시에서 읽어야 할 포인트

- 질문은 무능의 표시가 아니라 학습 속도를 높이는 기술입니다.
- 작은 PR은 신뢰와 협업 감각을 함께 쌓아 줍니다.
- 1:1은 혼자 끙끙대지 않게 해 주는 공식 채널입니다.

## 자주 하는 실수 5가지

1. **모르는 것을 숨기는 실수**: 초반에는 이해하지 못한 부분이 더 빨리 커집니다.
2. **질문을 너무 크게 묻는 실수**: 범위를 좁히지 않으면 답변도 추상적이 됩니다.
3. **PR을 미루는 실수**: 코드베이스와 리뷰 문화를 익힐 기회를 놓칩니다.
4. **1:1을 수동적으로 보내는 실수**: 고민을 정리하지 않으면 좋은 피드백도 얻기 어렵습니다.
5. **기록을 남기지 않는 실수**: 같은 혼란을 반복하고 학습 속도도 떨어집니다.

## 실무에서는 이렇게 이어집니다

많은 조직이 첫 90일 목표, 첫 PR, 코드 리뷰 참여 같은 항목을 온보딩 지표로 둡니다. 초반 적응을 운에 맡기지 않고 구조적으로 관리하기 때문입니다. 개인도 같은 방식으로 접근하면 훨씬 안정적입니다.

## 시니어는 이렇게 생각합니다

- 질문하는 태도는 예의이자 효율입니다.
- 작은 PR이 진짜 진전입니다.
- 1:1은 신뢰를 만드는 시간입니다.
- 기록은 커리어 자산입니다.
- 관찰하는 힘도 실력의 일부입니다.

## 체크리스트

- [ ] 팀 용어집이나 메모 문서를 만들었다.
- [ ] 질문 노트를 따로 운영하고 있다.
- [ ] 범위가 작은 첫 PR을 보냈다.
- [ ] 매니저 또는 멘토와 1:1 리듬을 잡았다.

## 연습 문제

1. 심리적 안전감을 한 줄로 설명해 보세요.
2. 버디와 멘토의 차이를 한 줄로 적어 보세요.
3. 좋은 첫 PR의 예시를 하나 떠올려 보세요.

## 정리 및 다음 글

첫 직장 적응의 핵심은 크게 기여하는 척하는 데 있지 않습니다. 배우고, 묻고, 기록하고, 작은 기여를 반복하는 데 있습니다. 이 루틴이 자리를 잡으면 초반 불안이 빠르게 자신감으로 바뀝니다.
