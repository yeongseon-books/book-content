# Developer Career 101 (8/10): 사이드 프로젝트와 학습

사이드 프로젝트를 시작할 때는 대개 아이디어가 너무 큽니다. 하지만 본업과 함께 가져가야 하는 프로젝트는 재미보다 지속 가능성이 더 중요하고, 멋진 비전보다 작게 끝내는 구조가 더 중요합니다.

이 글은 Developer Career 101 시리즈의 8번째 글입니다.

## 먼저 던지는 질문

- 본업과 충돌하지 않는 좋은 사이드 프로젝트는 어떤 조건을 가져야 할까요?
- 너무 큰 아이디어 대신 작은 MVP로 시작해야 하는 이유는 무엇일까요?
- 시간 박스, 공개, 피드백은 사이드 프로젝트를 어떻게 지속 가능하게 만들까요?

## 큰 그림

![Developer Career 101 8장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/developer-career-101/08/08-01-concept-at-a-glance.ko.png)

*Developer Career 101 8장 흐름 개요*

사이드 프로젝트는 새로운 기술을 배우거나 포트폴리오를 채우는 수단이 아니라, 본업에서 경험하지 못한 전체 사이클을 직접 주도하는 경험입니다.

> 사이드 프로젝트의 가치는 얼마나 정교한 기술을 썼느냐가 아니라, 요구사항 정의부터 배포와 유지보수까지 전체 책임을 져본 경험입니다.

## 이 글에서 배울 것

- 프로젝트를 고르는 기준
- 시간 박스 운영법
- 공개와 피드백 수집법
- 회사 업무와 분리하는 법
- 지속 가능성 전략

## 왜 중요한가

좋은 사이드 프로젝트는 학습과 증거를 동시에 남깁니다. 반대로 범위가 크고 경계가 모호하면 피로만 쌓이고 결과물이 남지 않기 쉽습니다.

> 사이드 프로젝트의 핵심은 대단한 아이디어가 아니라, 작게 끝내고 배움을 남기는 구조입니다.

## 핵심 개념 한눈에 보기

사이드 프로젝트는 길게 버티는 마라톤이 아닙니다. 아이디어를 고르고, 시간을 제한하고, 작게 내보내고, 피드백을 받아 다음 사이클로 넘어가는 구조가 있어야 지치지 않습니다.

## 핵심 용어

- **사이드 프로젝트**: 취미 또는 보조 성격의 개인 프로젝트입니다.
- **시간 박스**: 경계를 둔 시간 슬롯입니다.
- **MVP**: 최소 기능 제품입니다.
- 문라이팅: 공개하지 않은 부업입니다.
- **이해 충돌**: 의무가 충돌하는 상태입니다.

## Before/After

**Before**: “아이디어는 많지만 끝낸 것이 없습니다.”

**After**: “분기마다 작은 MVP 하나를 실제로 배포합니다.”

## 직접 해보기: 사이드 프로젝트 운영하기

### 1단계 — Pick the Idea

```text
criteria:
- interest
- learning value
- no conflict with day job
```

좋은 아이디어는 화려한 아이디어가 아닙니다. 흥미가 있고, 배울 것이 분명하며, 본업과 충돌하지 않아야 오래 가져갈 수 있습니다.

### 2단계 — Time Box

```text
4 hours/week (Sat 09-13)
```

사이드 프로젝트는 남는 시간에 하는 일이 아니라 미리 예약된 시간에 하는 일이어야 합니다. 시간을 무한정 쓰기 시작하면 건강과 본업 둘 다 흔들리기 쉽습니다.

### 3단계 — Define MVP

```markdown
- 1 core feature
- 1 command
- 1 README
```

MVP는 끝선을 만드는 장치입니다. 핵심 기능 하나와 최소한의 문서만으로도 첫 공개는 충분히 가능합니다.

### 4단계 — Publish

```bash
gh repo create --public
# README + LICENSE + first release
```

공개하지 않으면 피드백도 늦고 동기부여도 약해집니다. 작은 저장소라도 README와 LICENSE, 첫 릴리스를 남기면 프로젝트가 실제 자산으로 바뀝니다.

### 5단계 — Separation Policy

```text
- no company assets
- IP review with employer
```

본업과의 분리는 안전장치입니다. 회사 자산을 섞지 말고, 지식재산권 규정을 확인해 두어야 나중에 불필요한 문제를 피할 수 있습니다.

## 이 예시에서 먼저 볼 점

- 시간 박스가 지속 가능성을 만듭니다.
- MVP가 끝선을 만듭니다.
- 분리가 안전입니다.

## 자주 하는 실수 5가지

1. **회사 코드를 섞는 일입니다.**
2. **시간을 무한정 쓰는 일입니다.**
3. **MVP를 너무 크게 잡는 일입니다.**
4. **라이선스를 빼먹는 일입니다.**
5. **끝내 배포하지 않는 일입니다.**

## 실무에서는 이렇게 드러납니다

많은 회사가 고용 계약서에 오픈소스 기여나 개인 프로젝트 관련 규정을 명시합니다. 사이드 프로젝트는 학습 도구이면서 동시에 직업적 경계 감각을 시험하는 활동이기도 합니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 작게 시작합니다.
- 배포가 동기부여를 만듭니다.
- 시간 박스가 건강을 지킵니다.
- 분리는 직업 안전입니다.
- 지속성이 결국 차이를 만듭니다.

## 정리

좋은 사이드 프로젝트는 큰 꿈을 오래 품는 프로젝트가 아니라, 본업과 충돌하지 않는 범위 안에서 작게 만들고 실제로 끝내는 프로젝트입니다. 아이디어, 시간, 공개, 경계가 정리되면 사이드 프로젝트는 학습과 포트폴리오를 동시에 남겨 줍니다. 다음 글에서는 혼자 공부하는 한계를 넘기 위해 멘토링과 네트워킹을 어떻게 운영할지 정리하겠습니다.

## 사이드 프로젝트 계획 템플릿

사이드 프로젝트가 중도 포기로 끝나는 가장 큰 이유는 범위와 시간 경계가 없기 때문입니다. 계획 단계에서 최소 템플릿을 고정하면 완주 확률이 크게 올라갑니다.

| 항목 | 작성 기준 | 예시 |
| --- | --- | --- |
| 문제 정의 | 한 문장으로 적기 | "개인 지출 기록 자동 분류" |
| 사용자 | 딱 한 부류만 지정 | 본인 + 1명의 테스트 사용자 |
| MVP 범위 | 핵심 기능 1개 | 영수증 입력/분류만 제공 |
| 일정 | 4~6주 제한 | 토요일 4시간 x 5주 |
| 공개 방식 | 검증 가능한 형태 | GitHub README + 데모 영상 |

## 시간 관리 표

본업과 병행할 때는 의욕이 아니라 시간 예산이 기준입니다.

| 주차 | 개발 시간 | 문서 시간 | 회고 시간 | 목표 |
| --- | --- | --- | --- | --- |
| 1주차 | 3시간 | 1시간 | 30분 | 요구사항/MVP 고정 |
| 2주차 | 4시간 | 1시간 | 30분 | 핵심 기능 구현 |
| 3주차 | 4시간 | 1시간 | 30분 | 테스트/버그 수정 |
| 4주차 | 3시간 | 2시간 | 30분 | 배포/README 완성 |
| 5주차 | 2시간 | 2시간 | 1시간 | 회고/다음 계획 |

시간표를 보면 코드 작성 시간만큼 문서와 회고 시간이 중요합니다. 실제로 포트폴리오 가치를 높이는 부분은 구현 그 자체보다 의사결정 기록과 공개 품질인 경우가 많습니다.

## 작은 완주를 반복하는 전략

- "한 번에 크게" 대신 "4~6주 단위 완주"를 반복합니다.
- 배포 가능한 최소 버전을 먼저 내고, 기능은 릴리스 단위로 추가합니다.
- 공개 후 피드백 3개를 받아 다음 사이클의 요구사항으로 연결합니다.
- 본업 피로도가 높은 주는 유지보수·문서 주간으로 전환합니다.

지속 가능한 사이드 프로젝트는 고강도 스퍼트가 아니라 리듬 관리에서 나옵니다.

- **본업과 충돌하지 않는 좋은 사이드 프로젝트는 어떤 조건을 가져야 할까요?**
  - 사이드 프로젝트의 가치는 기술이 아니라 요구사항 정의부터 배포, 유지보수까지 전체 사이클을 직접 주도한 경험입니다.
- **너무 큰 아이디어 대신 작은 MVP로 시작해야 하는 이유는 무엇일까요?**
  - 작은 프로젝트라도 사용자 피드백을 받고, 버그를 수정하고, 새로운 기능을 추가하는 경험이 실무적 성장을 만듭니다.
- **시간 박스, 공개, 피드백은 사이드 프로젝트를 어떻게 지속 가능하게 만들까요?**
  - 사이드 프로젝트는 GitHub에 공개하고 README를 충실히 작성하면, 이직이나 평가에서 강력한 증거가 됩니다.
