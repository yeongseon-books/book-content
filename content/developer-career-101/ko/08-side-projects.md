# Developer Career 101 (8/10): 사이드 프로젝트와 학습

사이드 프로젝트를 시작할 때는 대개 아이디어가 너무 큽니다. 하지만 본업과 함께 가져가야 하는 프로젝트는 재미보다 지속 가능성이 더 중요하고, 멋진 비전보다 작게 끝내는 구조가 더 중요합니다.

이 글은 Developer Career 101 시리즈의 8번째 글입니다.

![Developer Career 101 8장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/developer-career-101/08/08-01-concept-at-a-glance.ko.png)
*Developer Career 101 8장 흐름 개요*
> 사이드 프로젝트의 가치는 얼마나 정교한 기술을 썼느냐가 아니라, 요구사항 정의부터 배포와 유지보수까지 전체 책임을 져본 경험입니다.

## 먼저 던지는 질문

- 본업과 충돌하지 않는 좋은 사이드 프로젝트는 어떤 조건을 가져야 할까요?
- 너무 큰 아이디어 대신 작은 MVP로 시작해야 하는 이유는 무엇일까요?
- 시간 박스, 공개, 피드백은 사이드 프로젝트를 어떻게 지속 가능하게 만들까요?

## 이 글에서 배울 것

- 프로젝트를 고르는 기준
- 시간 박스 운영법
- 공개와 피드백 수집법
- 회사 업무와 분리하는 법
- 지속 가능성 전략

## 왜 중요한가

좋은 사이드 프로젝트는 학습과 증거를 동시에 남깁니다. 반대로 범위가 크고 경계가 모호하면 피로만 쌓이고 결과물이 남지 않기 쉽습니다.

> 사이드 프로젝트의 핵심은 대단한 아이디어가 아니라, 작게 끝내고 배움을 남기는 구조입니다.

## 핵심 개념 한눈에 보기

사이드 프로젝트는 길게 버티는 마라톤이 아닙니다. 아이디어를 고르고, 시간을 제한하고, 작게 내보내고, 피드백을 받아 다음 사이클로 넘어가는 구조가 있어야 지치지 않습니다.

## 핵심 용어

- **사이드 프로젝트**: 취미 또는 보조 성격의 개인 프로젝트입니다.
- **시간 박스**: 경계를 둔 시간 슬롯입니다.
- **MVP**: 최소 기능 제품입니다.
- 문라이팅: 공개하지 않은 부업입니다.
- **이해 충돌**: 의무가 충돌하는 상태입니다.

## 개선 전/후

**Before**: “아이디어는 많지만 끝낸 것이 없습니다.”

**After**: “분기마다 작은 MVP 하나를 실제로 배포합니다.”

## 직접 해보기: 사이드 프로젝트 운영하기

### 1단계 — Pick the Idea

```text
criteria:
- interest
- learning value
- no conflict with day job
```

좋은 아이디어는 화려한 아이디어가 아닙니다. 흥미가 있고, 배울 것이 분명하며, 본업과 충돌하지 않아야 오래 가져갈 수 있습니다.

### 2단계 — Time Box

```text
4 hours/week (Sat 09-13)
```

사이드 프로젝트는 남는 시간에 하는 일이 아니라 미리 예약된 시간에 하는 일이어야 합니다. 시간을 무한정 쓰기 시작하면 건강과 본업 둘 다 흔들리기 쉽습니다.

### 3단계 — Define MVP

```markdown
- 1 core feature
- 1 command
- 1 README
```

MVP는 끝선을 만드는 장치입니다. 핵심 기능 하나와 최소한의 문서만으로도 첫 공개는 충분히 가능합니다.

### 4단계 — Publish

```bash
gh repo create --public
# 프로젝트 소개문 + 라이선스 + 첫 릴리스
```

공개하지 않으면 피드백도 늦고 동기부여도 약해집니다. 작은 저장소라도 README와 LICENSE, 첫 릴리스를 남기면 프로젝트가 실제 자산으로 바뀝니다.

### 5단계 — Separation Policy

```text
- no company assets
- IP review with employer
```

본업과의 분리는 안전장치입니다. 회사 자산을 섞지 말고, 지식재산권 규정을 확인해 두어야 나중에 불필요한 문제를 피할 수 있습니다.

## 사이드 프로젝트를 고를 때 쓰는 점수표

| 기준 | 높은 점수를 주는 경우 | 경고 신호 |
| --- | --- | --- |
| 학습 가치 | 지금 커리어 공백을 직접 메운다 | 이미 익숙한 것만 반복한다 |
| 완주 가능성 | 4~6주 안에 MVP를 낼 수 있다 | 핵심 기능이 세 개 이상이다 |
| 공개 가능성 | README와 데모로 설명할 수 있다 | 결과가 눈에 보이지 않는다 |
| 경계 관리 | 회사 자산, 고객 데이터와 분리된다 | 계약 조항이 모호하다 |

## 이 예시에서 먼저 볼 점

- 시간 박스가 지속 가능성을 만듭니다.
- MVP가 끝선을 만듭니다.
- 분리가 안전입니다.

## 자주 하는 실수 5가지

1. **회사 코드를 섞는 일입니다.**
2. **시간을 무한정 쓰는 일입니다.**
3. **MVP를 너무 크게 잡는 일입니다.**
4. **라이선스를 빼먹는 일입니다.**
5. **끝내 배포하지 않는 일입니다.**

## 실무에서는 이렇게 드러납니다

많은 회사가 고용 계약서에 오픈소스 기여나 개인 프로젝트 관련 규정을 명시합니다. 사이드 프로젝트는 학습 도구이면서 동시에 직업적 경계 감각을 시험하는 활동이기도 합니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 작게 시작합니다.
- 배포가 동기부여를 만듭니다.
- 시간 박스가 건강을 지킵니다.
- 분리는 직업 안전입니다.
- 지속성이 결국 차이를 만듭니다.

## 체크리스트

- [ ] 주 4시간 시간 박스를 잡았습니다.
- [ ] MVP를 정의했습니다.
- [ ] 릴리스 절차를 정했습니다.
- [ ] 지식재산권 검토 기준을 확인했습니다.

## 연습 문제

1. 문라이팅을 한 줄로 설명해 보세요.
2. 이해 충돌 예시를 한 줄로 적어 보세요.
3. MVP 기준을 한 줄로 적어 보세요.

## 사이드 프로젝트 계획 템플릿

사이드 프로젝트가 중도 포기로 끝나는 가장 큰 이유는 범위와 시간 경계가 없기 때문입니다. 계획 단계에서 최소 템플릿을 고정하면 완주 확률이 크게 올라갑니다.

| 항목 | 작성 기준 | 예시 |
| --- | --- | --- |
| 문제 정의 | 한 문장으로 적기 | "개인 지출 기록 자동 분류" |
| 사용자 | 딱 한 부류만 지정 | 본인 + 1명의 테스트 사용자 |
| MVP 범위 | 핵심 기능 1개 | 영수증 입력/분류만 제공 |
| 일정 | 4~6주 제한 | 토요일 4시간 x 5주 |
| 공개 방식 | 검증 가능한 형태 | GitHub README + 데모 영상 |

## 시간 관리 표

본업과 병행할 때는 의욕이 아니라 시간 예산이 기준입니다.

| 주차 | 개발 시간 | 문서 시간 | 회고 시간 | 목표 |
| --- | --- | --- | --- | --- |
| 1주차 | 3시간 | 1시간 | 30분 | 요구사항/MVP 고정 |
| 2주차 | 4시간 | 1시간 | 30분 | 핵심 기능 구현 |
| 3주차 | 4시간 | 1시간 | 30분 | 테스트/버그 수정 |
| 4주차 | 3시간 | 2시간 | 30분 | 배포/README 완성 |
| 5주차 | 2시간 | 2시간 | 1시간 | 회고/다음 계획 |

시간표를 보면 코드 작성 시간만큼 문서와 회고 시간이 중요합니다. 실제로 포트폴리오 가치를 높이는 부분은 구현 그 자체보다 의사결정 기록과 공개 품질인 경우가 많습니다.

## 작은 완주를 반복하는 전략

- "한 번에 크게" 대신 "4~6주 단위 완주"를 반복합니다.
- 배포 가능한 최소 버전을 먼저 내고, 기능은 릴리스 단위로 추가합니다.
- 공개 후 피드백 3개를 받아 다음 사이클의 요구사항으로 연결합니다.
- 본업 피로도가 높은 주는 유지보수·문서 주간으로 전환합니다.

지속 가능한 사이드 프로젝트는 고강도 스퍼트가 아니라 리듬 관리에서 나옵니다.

## 사이드 프로젝트 10주 로드맵

사이드 프로젝트가 중간에 멈추는 가장 큰 이유는 목표가 너무 크기 때문입니다. 아래 로드맵은 10주 안에 배포 가능한 결과물을 만드는 데 초점을 둡니다.

| 주차 | 목표 | 산출물 | 완료 기준 |
| --- | --- | --- | --- |
| 1-2주 | 문제 정의/범위 고정 | PRD 1페이지, 핵심 사용자 시나리오 | 제외 기능 목록이 명확함 |
| 3-4주 | MVP 구현 | 핵심 기능 1개 동작 | 데모 영상 3분 촬영 가능 |
| 5-6주 | 데이터/로그 추가 | DB 스키마, 구조화 로그 | 오류 재현 경로 추적 가능 |
| 7-8주 | 테스트/배포 | 테스트 20개, CI 파이프라인 | 재배포 30분 내 가능 |
| 9-10주 | 문서화/공유 | README, 회고, 발표 자료 | 타인이 1시간 내 실행 가능 |

완주 확률을 높이려면 매주 "삭제할 기능"을 하나 정해야 합니다. 추가보다 삭제를 잘해야 프로젝트가 살아남습니다. 특히 주 7-8시간 이하로 투자하는 경우에는 기능 수보다 운영 품질이 이력서 신뢰도에 더 큰 영향을 줍니다.

## 시간 예산표: 직장 병행형

현실적인 시간 배분이 없으면 사이드 프로젝트는 죄책감 루틴이 됩니다. 아래 예시는 주 8시간 예산 모델입니다.

| 활동 | 시간 | 메모 |
| --- | ---: | --- |
| 구현 | 4시간 | 가장 에너지 높은 시간대 배치 |
| 테스트/디버깅 | 2시간 | 토요일 오전 고정 |
| 문서/회고 | 1시간 | 일요일 저녁 30분 x 2 |
| 학습/조사 | 1시간 | 구현에 직접 필요한 주제만 |

이 배분의 핵심은 학습 시간을 무한대로 늘리지 않는 것입니다. 프로젝트를 위한 학습인지, 학습을 위한 학습인지 구분해야 합니다. 결과물이 목표라면 조사 시간은 전체의 20%를 넘기지 않는 편이 좋습니다.

## 포트폴리오 연결용 결과 정리 템플릿

프로젝트를 끝낸 뒤에는 다음 형식으로 정리합니다.

| 질문 | 답변 예시 |
| --- | --- |
| 어떤 문제를 풀었는가 | "소규모 팀의 일정 공유 지연 문제" |
| 왜 이 아키텍처를 골랐는가 | "초기에는 단일 서비스로 운영 복잡도 최소화" |
| 무엇을 개선했는가 | "알림 실패율 9%→2%" |
| 다음 단계는 무엇인가 | "비동기 큐 도입, 알림 재시도 정책 추가" |

이 템플릿으로 정리된 문장은 이력서 불릿, 포트폴리오 본문, 면접 답변으로 그대로 재사용할 수 있습니다.

## 학습 로드맵과 사이드 프로젝트 연결표

학습과 프로젝트가 분리되면 둘 다 중단되기 쉽습니다. 아래 연결표처럼 각 학습 주제를 즉시 프로젝트 과제로 붙여야 유지됩니다.

| 학습 주제 | 프로젝트 적용 과제 | 완료 신호 |
| --- | --- | --- |
| 인증/인가 | 로그인/권한 분리 구현 | 권한 테스트 통과 |
| 데이터 모델링 | 핵심 엔티티 스키마 재설계 | 쿼리 응답 속도 개선 |
| 관측성 | 에러 로깅/알람 추가 | 장애 원인 10분 내 파악 |
| 배포 자동화 | CI 파이프라인 구축 | 태그 기반 배포 성공 |

이 방식은 "공부했지만 구현하지 못했다"는 단절을 줄여 줍니다.

## 공개 배포 전 점검표

프로젝트를 공개하기 전에는 기능보다 신뢰 요소를 점검해야 합니다.

| 항목 | 확인 질문 |
| --- | --- |
| 실행 재현성 | 다른 사람이 10분 안에 실행 가능한가 |
| 오류 처리 | 실패 시 사용자에게 행동 지침이 보이는가 |
| 관측성 | 최소한의 로그와 상태 페이지가 있는가 |
| 문서 품질 | 문제 정의와 아키텍처 선택 이유가 명확한가 |

이 점검표를 통과한 프로젝트는 면접에서 설명할 때도 구조가 흔들리지 않습니다.

## 학습 과부하를 막는 삭제 규칙

사이드 프로젝트를 오래 유지하려면 추가 규칙보다 삭제 규칙이 필요합니다. 주간 회고 때 다음 세 가지를 확인합니다.

1. 이번 주에 추가한 기능이 핵심 사용자 시나리오를 직접 개선했는가
2. 유지 비용이 높은 기능을 과감히 제거했는가
3. 다음 주 우선순위를 세 개 이하로 제한했는가

이 원칙을 지키면 프로젝트가 길어져도 방향이 흐려지지 않습니다.

## 릴리스 노트 템플릿

작은 프로젝트도 릴리스 노트를 남기면 성장 기록의 품질이 올라갑니다. 이번 버전에서 무엇이 바뀌었는지, 왜 바꿨는지, 사용자 영향이 무엇인지 세 줄로 정리하면 이후 면접에서 프로젝트 진행력을 설명하기가 쉬워집니다.

## 데모 준비 체크포인트

데모 직전에는 기능 추가를 멈추고 안정화에 집중합니다. 시작 시나리오, 실패 시나리오, 복구 시나리오를 각각 한 번씩 실행해 보면 발표 중 돌발 오류를 크게 줄일 수 있습니다.

### 실행 메모

이 섹션은 읽고 끝내는 지식이 아니라 실제 행동으로 옮기기 위한 메모입니다. 오늘 바로 적용할 한 가지 행동을 정하고, 적용 전후의 차이를 짧게 기록해 두면 다음 의사결정의 품질이 올라갑니다. 커리어 글의 가치는 정보량보다 실행 전환율에서 결정되므로, 작은 행동을 반복해 결과를 누적하는 방식으로 운영하는 것이 가장 효과적입니다.

추가로, 다음 주 시작 전에 지난 주 액션 아이템의 완료 여부를 체크하면 계획이 선언으로 끝나지 않습니다. 완료율을 숫자로 기록하면 스스로의 실행 패턴을 더 정확히 보정할 수 있습니다.

## 정리

좋은 사이드 프로젝트는 큰 꿈을 오래 품는 프로젝트가 아니라, 본업과 충돌하지 않는 범위 안에서 작게 만들고 실제로 끝내는 프로젝트입니다. 아이디어, 시간, 공개, 경계가 정리되면 사이드 프로젝트는 학습과 포트폴리오를 동시에 남겨 줍니다. 다음 글에서는 혼자 공부하는 한계를 넘기 위해 멘토링과 네트워킹을 어떻게 운영할지 정리하겠습니다.

## 처음 질문으로 돌아가기

- **본업과 충돌하지 않는 좋은 사이드 프로젝트는 어떤 조건을 가져야 할까요?**
  - 사이드 프로젝트의 가치는 기술이 아니라 요구사항 정의부터 배포, 유지보수까지 전체 사이클을 직접 주도한 경험입니다.
- **너무 큰 아이디어 대신 작은 MVP로 시작해야 하는 이유는 무엇일까요?**
  - 작은 프로젝트라도 사용자 피드백을 받고, 버그를 수정하고, 새로운 기능을 추가하는 경험이 실무적 성장을 만듭니다.
- **시간 박스, 공개, 피드백은 사이드 프로젝트를 어떻게 지속 가능하게 만들까요?**
  - 사이드 프로젝트는 GitHub에 공개하고 README를 충실히 작성하면, 이직이나 평가에서 강력한 증거가 됩니다.
