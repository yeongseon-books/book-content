# 사이드 프로젝트와 학습

## 이 글에서 다룰 문제

- 본업과 충돌하지 않는 좋은 사이드 프로젝트는 어떤 조건을 가져야 할까요?
- 너무 큰 아이디어 대신 작은 MVP로 시작해야 하는 이유는 무엇일까요?
- 시간 박스, 공개, 피드백은 사이드 프로젝트를 어떻게 지속 가능하게 만들까요?
- 회사 자산, 지식재산권, 이해 충돌은 왜 초반부터 분리해서 생각해야 할까요?

> Developer Career 101 시리즈 (8/10)

사이드 프로젝트는 많은 개발자에게 매력적입니다. 직접 배우고 싶은 기술을 써 볼 수 있고, 작은 제품이나 도구를 끝까지 만들어 볼 수도 있기 때문입니다. 하지만 의욕만으로 시작하면 생각보다 쉽게 지칩니다. 범위가 너무 크거나, 본업과 섞이거나, 공개 시점을 미루다 보면 몇 달 뒤에는 저장소만 남기 쉽습니다.

좋은 사이드 프로젝트는 화려한 아이디어보다 운영 방식이 중요합니다. 작은 범위, 분명한 목적, 지속 가능한 시간 사용, 본업과의 명확한 분리가 있어야 오래 갑니다. 이 글에서는 그 기준을 실전적으로 정리해 보겠습니다.

## 왜 이 주제가 중요한가

사이드 프로젝트는 학습과 증거를 동시에 남길 수 있는 드문 장치입니다. 새 기술을 익힐 수 있을 뿐 아니라, 이력서와 포트폴리오에서 설명할 실제 결과물이 생깁니다. 다만 건강과 본업을 해치지 않는 방식으로 운영해야 장점이 살아납니다.

> 사이드 프로젝트의 핵심은 대단한 아이디어가 아니라, 작게 끝내고 배움을 남기는 구조입니다.

## 핵심 개념 한눈에 보기

```mermaid
flowchart LR
    P[선정] --> T[시간 박스]
    T --> S[출시]
    S --> F[피드백]
```

프로젝트를 고르고, 시간을 제한하고, 작게라도 공개하고, 반응을 받아 다음 판단에 쓰는 흐름이 있어야 합니다. 이 네 단계 중 하나라도 빠지면 프로젝트는 공부 메모나 미완성 아이디어로 끝나기 쉽습니다.

## 핵심 용어

- **사이드 프로젝트**: 본업 외 시간에 진행하는 학습형 또는 실험형 프로젝트입니다.
- **시간 박스**: 미리 정해 둔 제한된 작업 시간입니다.
- **MVP**: 꼭 필요한 핵심 기능만 담은 최소 실행 제품입니다.
- **문라이트닝**: 회사에 알리지 않은 채 사실상 부업 형태로 일하는 상황을 뜻합니다.
- **이해 충돌**: 본업의 책임과 개인 프로젝트가 부딪히는 상태입니다.

## Before / After

**Before**: 아이디어만 많이 떠올리고, 완성 없이 새 저장소만 계속 만듭니다.

**After**: 분기마다 하나의 작은 MVP를 정해 끝까지 만들고 공개합니다.

## 직접 해보기: 사이드 프로젝트 운영하기

### 1단계 — 아이디어 고르기

```text
criteria:
- interest
- learning value
- no conflict with day job
```

좋은 아이디어는 멋진 시장 규모보다 지금의 나에게 맞는지부터 봐야 합니다. 흥미가 있고, 학습 가치가 있으며, 본업과 충돌하지 않는지가 최소 기준입니다.

### 2단계 — 시간 박스 잡기

```text
4 hours/week (Sat 09-13)
```

프로젝트가 건강을 해치기 시작하면 오래 가지 못합니다. 주 4시간처럼 명확한 제한을 두면 오히려 범위 조절이 쉬워지고, 본업과 생활도 지킬 수 있습니다.

### 3단계 — MVP 정의하기

```markdown
- 1 core feature
- 1 command
- 1 README
```

MVP는 기능을 많이 넣는 계획이 아닙니다. 사용자나 미래의 내가 실제로 써 볼 수 있을 만큼만 만드는 일입니다. 핵심 기능 하나와 README 하나만 있어도 충분히 배울 수 있습니다.

### 4단계 — 공개하기

```bash
gh repo create --public
# README + LICENSE + first release
```

공개가 중요한 이유는 완성 기준을 만들어 주기 때문입니다. 저장소를 열고 README와 라이선스를 붙이고 첫 릴리스를 내면 프로젝트가 공부 메모에서 실제 산출물로 바뀝니다.

### 5단계 — 분리 정책 세우기

```text
- no company assets
- IP review with employer
```

회사 노트북, 내부 코드, 회사 시간, 회사 도메인 지식이 프로젝트와 섞이지 않도록 초반부터 선을 그어야 합니다. 고용 계약의 오픈소스나 지식재산권 조항도 꼭 확인하는 편이 안전합니다.

## 이 예시에서 읽어야 할 포인트

- 시간 박스가 있어야 프로젝트가 생활을 해치지 않습니다.
- MVP가 작을수록 완주 확률이 올라갑니다.
- 본업과의 분리는 학습 문제가 아니라 직업 안전 문제이기도 합니다.

## 자주 하는 실수 5가지

1. **회사 코드나 자산을 섞는 실수**: 가장 피해야 할 위험입니다.
2. **시간 제한 없이 몰입하는 실수**: 초반 속도는 빨라도 금방 지치기 쉽습니다.
3. **MVP를 너무 크게 잡는 실수**: 시작은 하지만 끝이 보이지 않게 됩니다.
4. **라이선스를 빼먹는 실수**: 공개 프로젝트의 기본 신뢰 장치가 빠집니다.
5. **끝까지 공개하지 않는 실수**: 결과물이 남지 않아 학습과 증거가 약해집니다.

## 실무에서는 이렇게 이어집니다

많은 회사가 고용 계약서에 오픈소스 기여와 개인 프로젝트 원칙을 적어 둡니다. 그만큼 개인 프로젝트와 회사 업무의 경계를 명확히 보는 문화가 일반적입니다. 사이드 프로젝트를 오래 하고 싶다면 이 현실을 무시하면 안 됩니다.

## 시니어는 이렇게 생각합니다

- 작게 시작해야 오래 갑니다.
- 출시가 동기 부여를 만들어 줍니다.
- 시간 박스는 건강을 지켜 줍니다.
- 분리는 커리어 안전장치입니다.
- 지속 가능한 반복이 실력을 키웁니다.

## 체크리스트

- [ ] 주간 시간 박스를 정했다.
- [ ] MVP 범위를 한 문장으로 적었다.
- [ ] 공개 절차와 첫 릴리스 계획을 세웠다.
- [ ] 회사 정책과 지식재산권 조항을 확인했다.

## 연습 문제

1. 문라이트닝을 한 줄로 설명해 보세요.
2. 이해 충돌의 예시를 하나 적어 보세요.
3. 좋은 MVP 기준을 한 줄로 정리해 보세요.

## 정리 및 다음 글

사이드 프로젝트는 커리어를 빠르게 바꾸는 마법이 아니라, 작게 만들고 끝까지 공개하면서 학습과 증거를 함께 남기는 습관입니다. 범위를 줄이고, 시간을 제한하고, 본업과 분리하면 직장 생활과도 충분히 병행할 수 있습니다.

다음 글에서는 이런 성장 흐름을 혼자만의 것으로 두지 않고, 멘토링과 네트워킹으로 확장하는 방법을 다루겠습니다.
