# Developer Career 101 (4/10): 이력서와 포트폴리오

좋은 경험이 있어도 짧은 시간 안에 전달하지 못하면 기회로 이어지지 않는 경우가 많습니다. 채용 담당자는 수많은 지원서를 빠르게 훑기 때문에, 내가 무엇을 했는지보다 어떤 결과를 냈는지가 먼저 보여야 합니다.

이 글은 Developer Career 101 시리즈의 4번째 글입니다.

## 먼저 던지는 질문

- 채용 담당자가 짧은 시간 안에 나를 이해하려면 이력서는 어떤 구조여야 할까요?
- 담당 업무 나열이 아니라 결과와 영향 중심으로 쓰려면 무엇이 달라져야 할까요?
- 숫자와 STAR 구조는 왜 강한 설득 도구가 될까요?

## 큰 그림

![Developer Career 101 4장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/developer-career-101/04/04-01-concept-at-a-glance.ko.png)

*Developer Career 101 4장 흐름 개요*

이력서와 포트폴리오는 회사 평가자에게 '당신이 어떤 문제를 풀었고, 그 결과가 무엇인지'를 보여주는 증거입니다.

> 이력서와 포트폴리오의 핵심은 기술 목록이 아니라, 맡은 문제의 크기와 해결 과정에서 새로 배운 것을 명확히 기록하는 데 있습니다.

## 이 글에서 배울 것

- 이력서의 기본 구조
- 불릿을 쓰는 방식
- 포트폴리오를 연결하는 법
- 키워드와 ATS를 고려하는 법
- 지원 흐름 전체에서 이력서가 맡는 역할

## 왜 중요한가

이력서는 면접으로 들어가는 입장권입니다. 같은 경험이라도 결과와 영향이 잘 드러나면 읽는 사람이 이해하는 속도와 신뢰도가 크게 달라집니다.

> 이력서는 내가 어떤 사람인지 길게 설명하는 문서가 아니라, 면접을 열어 주는 짧은 광고입니다.

## 핵심 개념 한눈에 보기

좋은 이력서는 위에서 아래로 읽히는 흐름이 자연스럽습니다. 누구인지, 무엇을 잘하는지, 어떤 결과를 냈는지, 그 결과를 어디서 검증할 수 있는지가 짧은 시간 안에 이어져야 합니다.

## 핵심 용어

- 요약문: 세 줄 안팎의 짧은 소개입니다.
- 불릿: 결과를 담는 한 문장입니다.
- **STAR**: Situation, Task, Action, Result 구조입니다.
- **ATS**: 지원서를 자동 분류하는 시스템입니다.
- 포트폴리오: 경험을 증명하는 자료 묶음입니다.

## 개선 전/후

**Before**: “무슨 일을 했는지 나열합니다.”

**After**: “숫자와 함께 결과와 영향을 씁니다.”

## 직접 해보기: 이력서 만들기

### 1단계 — Header

```text
Name | Email | GitHub | LinkedIn
```

헤더는 단순해야 합니다. 연락 가능 정보와 검증 가능한 링크만 빠르게 보이게 두는 편이 좋습니다.

### 2단계 — Summary

```text
Backend, 3 yrs. Cut payment p95 from 200ms to 80ms.
```

요약문은 경력 소개가 아니라 핵심 성과의 압축본입니다. 역할과 연차, 가장 강한 숫자 하나만 있어도 인상이 달라집니다.

### 3단계 — Experience Bullets (STAR)

```markdown
- Cut p95 latency from 200ms to 80ms by introducing read replicas, serving 5M req/day.
```

불릿은 업무 설명보다 결과 문장이어야 합니다. 어떤 상황에서 무엇을 했고, 결과가 어떻게 달라졌는지 숫자로 보여 주면 읽는 사람이 바로 이해할 수 있습니다.

### 4단계 — Projects

```markdown
- tinytool: 1.2k stars, used by 30 orgs
```

프로젝트는 취미 목록이 아니라 증거입니다. 사용 조직 수, 스타 수, 배포 링크처럼 검증 가능한 단서가 들어가면 포트폴리오의 신뢰도가 올라갑니다.

### 5단계 — Skills

```text
Python, PostgreSQL, AWS, Kubernetes
```

기술 스택은 경험과 분리되어 떠 있으면 약합니다. 앞선 경력과 프로젝트 섹션에서 실제로 등장한 기술만 묶어 적는 편이 훨씬 강합니다.

## 이력서 섹션별 작성법

이력서는 요약, 경력, 프로젝트, 기술 섹션으로 구성됩니다. 각 섹션마다 DO와 DON'T를 명확히 구분하면 이력서 품질이 크게 달라집니다.

| 섹션 | DO | DON'T |
| --- | --- | --- |
| 요약 | 역할, 연차, 가장 강한 숫자 하나 | 추상적 형용사 (열정적인, 창의적인) |
| 경력 | STAR 구조, 숫자, 결과 | 담당 업무 나열 (참여, 협업, 지원) |
| 프로젝트 | 검증 가능한 링크, 사용 조직 수, star | 기술 나열만 |
| 기술 | 경력에서 실제 등장한 기술만 | 경험 없는 기술 나열 |

요약은 세 줄로 제한하고, 경력은 불릿당 하나의 결과를 담습니다. 프로젝트는 배포된 링크나 저장소 URL을 넣고, 기술은 앞에서 등장한 기술만 모아 적습니다.

## 경력 기술 Before/After 예시

경력 기술은 담당 업무 나열이 아니라 결과와 영향을 담는 문장입니다. 나쁨 예시와 좋은 예시를 비교하면 차이가 명확합니다.

### 예시 1: API 성능 개선

**Bad:**

```markdown
- 백엔드 API 개발에 참여했습니다.
- DB 쿼리 최적화를 진행했습니다.
```

**Good:**

```markdown
- API p95 응답 시간을 200ms에서 80ms로 개선하여 일 500만 건 요청을 안정적으로 처리했습니다. (read replica + 쿼리 최적화)
```

### 예시 2: 배포 자동화

**Bad:**

```markdown
- CI/CD 파이프라인을 구축했습니다.
- GitHub Actions를 사용했습니다.
```

**Good:**

```markdown
- GitHub Actions로 CI/CD 파이프라인을 구축하여 배포 주기를 2주에서 3일로 단축하고, 수동 배포 실수를 90% 감소시켰습니다.
```

### 예시 3: 코드 품질 개선

**Bad:**

```markdown
- 코드 리뷰를 진행했습니다.
- 테스트 커버리지를 올렸습니다.
```

**Good:**

```markdown
- 코드 리뷰 문화를 도입하고 pytest 커버리지를 40%에서 75%로 높여, 프로덕션 버그를 60% 감소시켰습니다.
```

## 자동 서류 심사 통과 전략

ATS(Applicant Tracking System)는 이력서를 자동 분류하는 시스템입니다. 큰 회사일수록 ATS를 사용할 확률이 높습니다.

### ATS 친화적 형식

1. **텍스트 기반 PDF** — 이미지 PDF는 ATS가 읽지 못합니다.
2. **표준 섹션 명** — Experience, Education, Skills 같은 일반적 이름을 사용합니다.
3. **키워드 포함** — JD(Job Description)에서 언급된 기술을 그대로 사용합니다.

```python
# 자동 심사 키워드 매칭 예시
job_description = ["Python", "FastAPI", "PostgreSQL", "Docker", "AWS"]

# 이력서에도 동일한 키워드를 포함
resume_skills = ["Python", "FastAPI", "PostgreSQL", "Docker", "AWS"]

# 매칭률 계산
matched = len(set(job_description) & set(resume_skills))
total = len(job_description)
match_rate = (matched / total) * 100

print(f"키워드 매칭률: {match_rate}%")
# 출력: 키워드 매칭률: 100%
```

### ATS를 피해야 할 형식

- 2단 레이아웃 (ATS가 순서를 헷갈림)
- 헤더/푸터에 중요 정보 배치 (ATS가 건너뜀)
- 이미지 형태의 텍스트 (OCR 실패 가능성)

이력서는 ATS를 통과하고, 인간 평가자가 읽어야 비로소 면접 기회를 얻습니다. 두 단계를 모두 고려하세요.

## 이 예시에서 먼저 볼 점

- 불릿은 결과 문장입니다.
- 숫자는 증거입니다.
- STAR 구조가 짧은 서사를 만들어 줍니다.

## 자주 하는 실수 5가지

1. **담당 업무만 나열하는 일입니다.**
2. **숫자를 전혀 쓰지 않는 일입니다.**
3. **포트폴리오와 이력서가 어긋나는 일입니다.**
4. **ATS가 읽기 어려운 형식을 쓰는 일입니다.**
5. **이력서가 세 페이지를 넘는 일입니다.**

## 실무에서는 이렇게 드러납니다

큰 회사는 내부 이동 때도 짧은 내부용 이력서를 요구하는 경우가 많습니다. 이력서를 잘 쓰는 능력은 외부 지원에만 필요한 기술이 아닙니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 이력서는 광고입니다.
- 숫자가 가장 강합니다.
- STAR는 결과를 정리하는 틀입니다.
- ATS는 현실입니다.
- 포트폴리오는 증거입니다.

## 체크리스트

- [ ] PDF와 텍스트 친화 형식을 준비했습니다.

## 정리

좋은 이력서와 포트폴리오는 경험을 예쁘게 포장하는 문서가 아니라, 결과와 증거를 빠르게 전달하는 문서입니다. 역할, 숫자, 결과, 링크가 선명하게 보이면 읽는 사람이 다음 질문을 떠올리기 쉬워집니다. 다음 글에서는 이렇게 얻은 기회를 실제 면접으로 연결하는 코딩 인터뷰 준비 방법을 정리하겠습니다.

## 이력서 구조 템플릿: 30초 스캔 통과형

이력서는 읽는 사람이 천천히 감상하는 문서가 아닙니다. 보통 30초 안에 "면접으로 넘길지"를 판단합니다. 따라서 정보의 순서가 설득력의 절반입니다.

| 섹션 | 권장 길이 | 포함 요소 | 피해야 할 요소 |
| --- | --- | --- | --- |
| 헤더 | 1줄 | 이름, 연락처, GitHub/LinkedIn | 주소, 불필요 개인정보 |
| 요약 | 2~3줄 | 역할, 연차, 대표 성과 숫자 | 추상적 성격 묘사 |
| 경력 | 회사별 3~5불릿 | 문제, 행동, 결과, 수치 | 담당업무 나열 |
| 프로젝트 | 2~4개 | 링크, 기여 범위, 검증 지표 | 실행 불가한 데모 |
| 기술 | 1블록 | 실제 사용한 기술만 | 키워드 과다 나열 |

요약문은 특히 중요합니다. "백엔드 3년, 결제 API p95 210ms에서 95ms로 개선"처럼 역할과 숫자를 같이 배치하면 읽는 사람이 즉시 맥락을 잡을 수 있습니다.

## 상황-과제-행동-결과 불릿 예시 모음

아래는 같은 경험을 약한 문장과 강한 문장으로 비교한 예시입니다.

| 약한 문장 | 강한 문장 |
| --- | --- |
| 결제 API 개발을 담당했습니다. | 결제 API의 병목 쿼리를 재설계해 p95 응답 시간을 210ms에서 95ms로 단축했습니다. |
| 장애 대응을 했습니다. | 주 2회 반복되던 배치 실패를 재시도 정책과 알람 튜닝으로 월 1회 이하로 줄였습니다. |
| 코드 리뷰를 진행했습니다. | 리뷰 템플릿을 도입해 PR 재수정 횟수를 평균 3.1회에서 1.7회로 낮췄습니다. |

핵심은 "무엇을 했다"가 아니라 "무엇이 달라졌다"를 쓰는 것입니다.

## 포트폴리오 연결 구조

포트폴리오는 프로젝트 목록이 아니라 "증거 링크 모음"입니다. 아래 순서로 구성하면 이력서와 정합성이 높아집니다.

1. 문제 정의: 어떤 문제를 해결했는가
2. 설계 선택: 왜 그 구조를 선택했는가
3. 결과 수치: 성능·안정성·사용량 중 무엇이 개선됐는가
4. 재현 링크: 저장소, 데모, 문서, 발표 자료

이 구조를 따르면 채용 담당자가 "주장-근거"를 빠르게 대조할 수 있습니다.

## 채용 공고-이력서 매핑 시트

서류 합격률을 올리려면 이력서를 먼저 쓰는 대신 채용 공고를 구조화해야 합니다. 같은 경험도 어떤 문장으로 연결하느냐에 따라 해석이 달라집니다.

| 공고 요구사항 | 내 경험 근거 | 이력서 문장 형태 | 증거 링크 |
| --- | --- | --- | --- |
| 대용량 트래픽 API 운영 | 주문 API p95 420ms→260ms 개선 | "캐시 정책과 인덱스 최적화로 p95 38% 개선" | 프로젝트 README, 대시보드 캡처 |
| 테스트 자동화 | pytest 도입으로 회귀 버그 감소 | "핵심 모듈 테스트 0→65개로 확대" | 테스트 리포트 |
| 협업/커뮤니케이션 | PM과 릴리즈 기준 합의 | "배포 체크리스트 표준화" | ADR 문서 |

이 표를 작성하면 공고별 맞춤 이력서 작성 시간이 줄어듭니다. 중요한 것은 한 번 만든 문장을 복붙하는 것이 아니라, 요구사항 우선순위에 따라 불릿 순서를 바꾸는 것입니다. 서류 스크리닝은 상단 5개 불릿에서 결정되는 경우가 많기 때문입니다.

## 포트폴리오 품질 점검표

포트폴리오는 "무엇을 만들었는가"보다 "어떤 문제를 어떻게 검증했는가"를 보여줘야 합니다. 아래 점검표는 제출 전 최종 검토용입니다.

| 항목 | Pass 기준 | Fail 신호 |
| --- | --- | --- |
| 문제 정의 | 사용자/업무 문제 한 문장 명시 | 기능 소개만 있음 |
| 기술 선택 이유 | 대안 비교 1개 이상 제시 | "익숙해서 사용"만 기재 |
| 운영 관점 | 로그/에러 처리/배포 전략 포함 | 로컬 실행 설명만 있음 |
| 결과 지표 | 속도, 오류율, 완료율 등 수치 제시 | "개선됨" 같은 추상 표현 |
| 재현 가능성 | README만으로 실행 가능 | 환경 설정 누락 |

이 점검표를 기준으로 최소 2회 리라이트를 하면 문서의 밀도가 크게 올라갑니다. 특히 결과 지표 항목은 준비가 늦어지기 쉬우므로, 프로젝트를 진행할 때부터 측정 지표를 같이 설계해야 합니다.



## 면접관 관점에서 포트폴리오를 읽는 순서

면접관은 보통 포트폴리오를 5분 내로 스캔합니다. 이때 보는 순서를 고려해 정보 배치를 설계해야 합니다.

1. 문제 정의가 명확한가
2. 선택한 기술의 근거가 있는가
3. 결과를 수치로 증명했는가
4. 실패와 수정 과정이 기록되어 있는가

이 순서에 맞춰 문서를 배치하면 포트폴리오의 신뢰도가 크게 올라갑니다.

## 처음 질문으로 돌아가기

- **채용 담당자가 짧은 시간 안에 나를 이해하려면 이력서는 어떤 구조여야 할까요?**
  - 이력서는 과거 회사와 기술 목록이 아니라, 각 회사에서 맡은 문제의 크기와 해결 과정에서 배운 것을 기록해야 합니다.
- **담당 업무 나열이 아니라 결과와 영향 중심으로 쓰려면 무엇이 달라져야 할까요?**
  - 포트폴리오는 기술이 얼마나 정교한지보다, 당신이 어떤 문제를 어떻게 풀었는지를 명확히 보여주는 것이 중요합니다.
- **숫자와 STAR 구조는 왜 강한 설득 도구가 될까요?**
  - 회사 평가자는 당신의 이력서를 3~5분 안에 스캔하고, 포트폴리오에서는 README와 배포된 링크를 먼저 봅니다.
