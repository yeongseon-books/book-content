# Developer Career 101 (6/10): 시스템 디자인 인터뷰

시스템 디자인 인터뷰를 처음 보면 커다란 박스를 몇 개 그리고 끝내는 시험처럼 느껴지기 쉽습니다. 하지만 실제로는 요구사항을 정리하고, 대략 계산하고, 구성 요소를 세우고, 트레이드오프를 설명하는 순서를 얼마나 안정적으로 가져가는지가 더 중요합니다.

이 글은 Developer Career 101 시리즈의 6번째 글입니다.

![Developer Career 101 6장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/developer-career-101/06/06-01-concept-at-a-glance.ko.png)
*Developer Career 101 6장 흐름 개요*
> 시스템 디자인에서 가장 중요한 것은 정답보다는, 요구사항을 확인하고 제약 조건을 명시한 후 트레이드오프를 설명하는 과정입니다.

## 먼저 던지는 질문

- 시스템 디자인 인터뷰에서 면접관은 무엇을 실제로 평가할까요?
- 요구사항 확인, 추정, 상위 설계, 심화 설명은 어떤 순서로 진행해야 할까요?
- 트레이드오프와 병목을 말하지 못하면 왜 답변이 얕아 보일까요?

## 이 글에서 배울 것

- 네 단계 절차
- 요구사항을 끌어내는 법
- 러프한 추정 방법
- 컴포넌트 설계 방법
- 심화 설명과 트레이드오프 정리 방식

## 왜 중요한가

시스템 디자인은 시니어 레벨을 가르는 대표 필터입니다. 면접관은 정답 그림보다, 제약 속에서 우선순위를 세우고 판단을 설명하는 방식을 더 유심히 봅니다.

> 시스템 디자인 인터뷰는 큰 그림을 예쁘게 그리는 시험이 아니라, 제약 속에서 판단을 설명하는 시험입니다.

## 핵심 개념 한눈에 보기

디자인 답변은 순서가 중요합니다. 요구사항을 정리하지 않고 박스부터 그리면 과하게 설계하거나 엉뚱한 방향으로 갈 수 있습니다. 반대로 요구사항, 추정, 상위 구조, 심화 설명의 순서를 지키면 답변이 훨씬 안정적으로 들립니다.

## 핵심 용어

- **기능 요구사항**: 시스템이 해야 하는 기능입니다.
- **비기능 요구사항**: 성능과 가용성 같은 제약입니다.
- 추정: 거칠지만 방향을 잡는 계산입니다.
- 트레이드오프: 선택의 비용입니다.
- 병목: 전체를 제한하는 지점입니다.

## 개선 전/후

**Before**: “박스를 몇 개 그리면 된다고 생각합니다.”

**After**: “요구사항, 추정, 설계, 심화 설명 순서로 답합니다.”

## 직접 해보기: URL 단축기 설계하기

### 1단계 — Requirements

```text
functional: shorten, redirect, analytics
non-functional: 100M URLs, 99.99% availability
```

기능 요구사항과 비기능 요구사항을 먼저 나눠야 합니다. 단축과 리다이렉트, 통계가 필요한지부터 확인하고, 규모와 가용성 기준을 붙여야 설계 범위가 정해집니다.

### 2단계 — Estimate

```text
QPS: 1000 reads/s, 10 writes/s
storage: 100M * 500 bytes = 50 GB
```

추정은 정답을 맞히는 문제가 아니라 설계의 바닥을 만드는 단계입니다. 읽기와 쓰기 비중, 저장 용량 정도만 잡아도 캐시와 저장소 선택의 감이 생깁니다.

### 3단계 — Components

```text
LB → API → KV store
analytics → Kafka → DW
```

상위 구조에서는 주요 흐름이 보이면 충분합니다. 요청이 어디로 들어와 저장되고, 분석 데이터가 어디로 빠지는지 정도를 선명하게 설명하는 편이 좋습니다.

### 4단계 — Deep Dive

```text
- collision: base62 + counter
- cache: redis (LRU)
- DR: multi-AZ
```

심화 설명은 하나의 병목이나 핵심 설계 결정을 깊게 파는 단계입니다. 충돌 처리, 캐시 전략, 장애 복구 같은 지점에서 시니어다운 깊이가 드러납니다.

### 5단계 — Trade-offs

```text
SQL vs KV: consistency vs performance
```

트레이드오프를 말하지 않으면 설계가 평면적으로 보입니다. 어떤 선택이 어떤 비용을 가져오는지 설명해야 판단의 질이 드러납니다.

## 시스템 디자인 답변에서 깊이를 만드는 질문

| 단계 | 스스로 던질 질문 | 답변에 꼭 남길 것 |
| --- | --- | --- |
| 요구사항 | 지금 풀어야 할 핵심 시나리오는 무엇인가 | 기능/비기능 분리 |
| 추정 | 읽기와 쓰기 비율이 저장소 선택을 어떻게 바꾸는가 | QPS, 용량, 성장 가정 |
| 상위 설계 | 병목이 생기면 어디부터 흔들리는가 | 주요 흐름과 저장 경계 |
| 심화 설명 | 어떤 선택이 가장 큰 비용을 만드는가 | 캐시, 일관성, 장애 복구 트레이드오프 |

## 이 예시에서 먼저 볼 점

- 요구사항이 항상 먼저입니다.
- 추정이 설계를 땅에 붙입니다.
- 심화 설명이 깊이를 보여 줍니다.

## 자주 하는 실수 5가지

1. **요구사항을 건너뛰는 일입니다.**
2. **추정을 전혀 하지 않는 일입니다.**
3. **트레이드오프를 말하지 않는 일입니다.**
4. **병목을 짚지 못하는 일입니다.**
5. **시간 관리를 못하는 일입니다.**

## 실무에서는 이렇게 드러납니다

회사 안에서 RFC를 쓰거나 큰 기능을 설계할 때도 같은 틀이 반복됩니다. 요구사항, 추정, 상위 구조, 심화 논의, 트레이드오프는 면접 전용 스킬이 아닙니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 디자인은 대화입니다.
- 추정은 근육입니다.
- 트레이드오프는 설계의 언어입니다.
- 심화 설명이 깊이를 보여 줍니다.
- 시간 관리도 훈련입니다.

## 체크리스트

- [ ] 기능과 비기능 요구사항을 나눴습니다.
- [ ] 추정을 명시했습니다.
- [ ] 트레이드오프를 말했습니다.
- [ ] 병목 하나를 깊게 설명했습니다.

## 연습 문제

2. 비기능 요구사항 예시를 한 줄로 적어 보세요.
3. 병목 예시를 한 줄로 적어 보세요.

## 시니어 역량 체크리스트

시스템 디자인 인터뷰는 아키텍처 지식 암기 시험이 아닙니다. 시니어 역량의 핵심인 문제 구조화, 우선순위 판단, 트레이드오프 설명을 검증하는 자리입니다. 아래 체크리스트는 면접 전에 스스로 점검할 최소 기준입니다.

| 역량 축 | 스스로 점검할 질문 | 준비 산출물 |
| --- | --- | --- |
| 요구사항 정제 | 기능/비기능 요구를 명확히 분리하는가 | 요구사항 템플릿 |
| 추정 감각 | QPS, 저장량, 성장률을 빠르게 계산하는가 | 러프 계산 노트 |
| 구조화 능력 | 핵심 컴포넌트를 단계적으로 설명하는가 | 블록 다이어그램 3개 |
| 트레이드오프 | 선택의 비용을 말할 수 있는가 | 대안 비교 표 |
| 운영 관점 | 장애, 모니터링, 복구를 포함하는가 | SLO/런북 초안 |

## 개인 기여자와 매니저 관점 비교

시니어 구간에서는 개인 기여자(IC)와 매니저 관점이 함께 요구될 때가 많습니다. 시스템 디자인 답변에서도 두 관점을 동시에 드러내면 평가가 좋아집니다.

| 관점 | 답변에서 보여야 할 포인트 |
| --- | --- |
| IC 관점 | 자료구조·저장소·캐시·일관성 선택 근거 |
| 매니저 관점 | 위험 우선순위, 일정 현실성, 운영 책임 분담 |

예를 들어 캐시 도입을 설명할 때도 "응답 시간을 줄인다"에서 끝내지 말고 "캐시 무효화 정책 실패 시 어떤 장애가 생기고 누가 대응하는지"까지 말하면 깊이가 달라집니다.

## 답변 품질을 높이는 미니 프레임

1. 지금 풀 문제를 한 문장으로 재정의합니다.
2. 가장 중요한 비기능 요구사항 2개를 고릅니다.
3. 단순한 기본안 하나를 먼저 제시합니다.
4. 병목 1개를 깊게 파고 대안을 비교합니다.
5. 운영 리스크와 모니터링 지표를 붙입니다.

이 다섯 단계만 지켜도 산만한 답변을 구조화된 답변으로 바꿀 수 있습니다.

## 실천 메모

이 섹션은 본문의 핵심을 실무 행동으로 전환하기 위한 메모입니다. 이번 주에 바로 실행할 한 가지를 정하고, 실행 결과를 수치 또는 관찰 가능한 문장으로 남기면 다음 회고에서 방향을 빠르게 수정할 수 있습니다. 계획은 길수록 좋은 것이 아니라, 반복 가능한 행동으로 내려올수록 강해집니다.

## 시스템 디자인 면접 일정표(6주)

시스템 디자인 준비는 개념 암기보다 답변 구조를 반복 훈련하는 과정입니다. 아래 일정표는 6주 기준 운영안입니다.

| 주차 | 핵심 주제 | 실전 연습 | 회고 포인트 |
| --- | --- | --- | --- |
| 1주 | 요구사항 정리 프레임 | 채팅 시스템 요구사항 정리 2회 | 질문 누락 개수 |
| 2주 | 용량 추정 | URL 단축기 QPS/스토리지 계산 3회 | 단위 변환 실수 |
| 3주 | 데이터 모델링 | 피드 서비스 스키마 설계 2회 | 정규화/비정규화 근거 |
| 4주 | 병목/확장 전략 | 캐시 계층 설계 2회 | 장애 지점 식별 |
| 5주 | 일관성/가용성 | 주문 시스템 트랜잭션 설계 | 보상 트랜잭션 설명력 |
| 6주 | 모의 면접 | 60분 실전 2회 | 시간 배분 안정성 |

이 일정표에서 가장 중요한 것은 매주 "실패 로그"를 남기는 것입니다. 예를 들어 요구사항 질문을 8개 중 3개 누락했다면 다음 주 시작 전에 동일 질문 세트를 다시 진행해야 합니다. 새 주제로 넘어가는 기준은 이해감이 아니라 재현 가능성입니다.

## 승진 기준과 연결되는 답변 매트릭스

시스템 디자인 답변은 채용 면접뿐 아니라 레벨 평가 기준과도 연결됩니다. 아래 매트릭스를 보면 어떤 깊이가 어느 레벨에서 기대되는지 명확히 파악할 수 있습니다.

| 평가 항목 | 주니어 기준 | 미들 기준 | 시니어 기준 |
| --- | --- | --- | --- |
| 문제 분해 | 기능 단위로 설명 | 트래픽/장애 시나리오 포함 | 비즈니스 리스크까지 구조화 |
| 기술 선택 | 단일 솔루션 제시 | 대안 비교 제시 | 비용/운영/조직 영향까지 비교 |
| 확장 전략 | Scale Out 언급 | 병목 포인트 기반 확장 | 단계별 마이그레이션 계획 제시 |
| 운영 관점 | 로그/모니터링 언급 | SLI/SLO 기준 제안 | 온콜/복구 절차까지 설계 |

이 표를 가지고 모의 면접 녹화를 보면 본인의 답변 수준을 훨씬 객관적으로 평가할 수 있습니다. "알고 있다"와 "설명할 수 있다" 사이의 간극을 메우는 데 효과적입니다.

## 설계 답변 기본 템플릿

아래 템플릿은 45~60분 설계 면접에서 자주 쓰이는 순서입니다.

1. 문제 경계 확정: 사용자, 기능, 비기능 요구사항 확인
2. 대략적 용량 산정: QPS, 저장 용량, 읽기/쓰기 비율
3. API와 데이터 모델: 핵심 엔티티와 계약 정의
4. 고수준 아키텍처: 요청 경로, 캐시, 비동기 큐 배치
5. 병목/장애 대응: 단일 장애점, 백프레셔, 재시도 정책
6. 마무리 트레이드오프: CAP/비용/운영 복잡도 정리

면접관은 완벽한 정답보다 문제를 다루는 방식의 일관성을 봅니다. 따라서 템플릿을 입으로 말해보는 훈련을 반복하면 답변의 안정성이 크게 올라갑니다.

## 아키텍처 트레이드오프 설명 훈련

시스템 디자인 면접에서 점수를 크게 올리는 구간은 정답 제시가 아니라 트레이드오프 설명입니다. 예를 들어 캐시를 도입한다면 읽기 성능 향상뿐 아니라 무효화 비용, 일관성 저하 가능성까지 함께 말해야 합니다.

| 선택지 | 장점 | 리스크 | 언제 적합한가 |
| --- | --- | --- | --- |
| 단일 DB 확장 | 단순한 운영 | 수직 확장 한계 | 초기 단계, 낮은 QPS |
| 읽기 복제본 | 조회 확장 용이 | 복제 지연 | 읽기 비중이 높은 서비스 |
| 샤딩 | 큰 확장성 | 라우팅 복잡도 증가 | 데이터 폭증 구간 |

면접 준비에서는 이 표를 서비스 유형별로 3세트 이상 만들어 보관하면 설명 속도와 정확도가 올라갑니다.

## 면접 후 복기 시트

모의 면접과 실전 면접 후에는 같은 형식으로 복기해야 개선 속도가 빨라집니다.

| 항목 | 기록 예시 |
| --- | --- |
| 잘한 점 | 요구사항 확인 질문 6개 수행 |
| 부족한 점 | 용량 추정에서 단위 변환 실수 2회 |
| 즉시 개선 | 추정 템플릿을 카드로 만들어 연습 |
| 다음 연습 일정 | 48시간 내 동일 문제 재시도 |

## 화이트보드 설명 훈련 규칙

설계 면접에서 판서 품질은 사고 품질을 드러냅니다. 구성 요소 이름을 먼저 고정하고, 화살표마다 요청/응답 방향을 명시하며, 병목 지점에 숫자 근거를 적는 습관을 들이면 설명력이 크게 올라갑니다.

## 실전 답변 속도 개선 팁

답변 속도는 암기량보다 순서 고정에서 나옵니다. 요구사항 확인, 용량 추정, 데이터 모델, 확장 전략 순서를 항상 같은 패턴으로 말하면 긴장 상황에서도 흐름이 무너지지 않습니다.

### 실행 메모

이 섹션은 읽고 끝내는 지식이 아니라 실제 행동으로 옮기기 위한 메모입니다. 오늘 바로 적용할 한 가지 행동을 정하고, 적용 전후의 차이를 짧게 기록해 두면 다음 의사결정의 품질이 올라갑니다. 커리어 글의 가치는 정보량보다 실행 전환율에서 결정되므로, 작은 행동을 반복해 결과를 누적하는 방식으로 운영하는 것이 가장 효과적입니다.

## 정리

시스템 디자인 인터뷰는 정답 도안을 맞히는 시험이 아니라, 제한된 시간 안에 요구사항을 구조화하고 판단을 설명하는 시험입니다. 요구사항, 추정, 상위 설계, 심화 설명의 순서를 몸에 익히면 답변이 훨씬 안정적으로 정리됩니다. 다음 글에서는 면접을 넘어 실제 첫 직장에서 처음 90일을 어떻게 보내야 하는지 정리하겠습니다.

## 처음 질문으로 돌아가기

- **시스템 디자인 인터뷰에서 면접관은 무엇을 실제로 평가할까요?**
  - 시스템 디자인은 한 가지 정답이 없고, 요구사항을 확인하고 제약을 명시한 후 트레이드오프를 설명하는 과정이 중요합니다.
- **요구사항 확인, 추정, 상위 설계, 심화 설명은 어떤 순서로 진행해야 할까요?**
  - 확장성, 안정성, 비용 중 무엇을 우선할 것인지, 캐시와 데이터베이스 사이 일관성을 어떻게 유지할 것인지를 결정해야 합니다.
- **트레이드오프와 병목을 말하지 못하면 왜 답변이 얕아 보일까요?**
  - 면접관의 질문을 기다리기보다 먼저 병목 지점을 발견하고, 그 이유를 설명하고, 개선책을 제시하는 주도성이 평가됩니다.
