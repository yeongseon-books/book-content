# 튜토리얼 작성하기

## 이 글에서 다룰 문제

- 튜토리얼은 설명 글이나 레퍼런스와 무엇이 다를까요?
- 따라 하기 글에서 왜 모든 단계가 실제로 검증되어야 할까요?
- 전제 조건, 작은 성공, 오류 복구, 다음 단계는 어떤 순서로 배치해야 할까요?
- 짧은 튜토리얼이 오히려 더 좋은 경우는 언제일까요?

> 튜토리얼은 정보를 주는 글이 아니라, 독자가 처음 성공을 경험하게 만드는 글입니다.

> 기술 글쓰기 101 시리즈 (8/10)

튜토리얼은 많이 설명하는 글이 아닙니다. 독자가 실제로 따라 하면서 결과를 얻는 글입니다. 그래서 설명이 좋아도 단계가 빠지거나 순서가 틀리면 좋은 튜토리얼이 될 수 없습니다. 반대로 문장은 조금 거칠어도, 독자가 정확히 따라 해 성공할 수 있다면 훨씬 쓸모 있습니다.

튜토리얼의 핵심은 첫 성공입니다. 처음 보는 기술을 배울 때 독자가 가장 필요한 것은 완전한 이해가 아니라, 작더라도 손에 잡히는 성공 경험입니다. 그 경험이 생겨야 다음 문서와 더 깊은 학습으로 넘어갑니다.

## 왜 중요한가

첫 성공은 학습 의지를 만듭니다. 설치가 되고, Hello World가 뜨고, 응답이 돌아오고, 눈앞에서 작은 결과가 확인되면 독자는 더 배울 힘을 얻습니다. 반대로 첫 10분 안에 막히면, 기술 자체보다 문서에 대한 신뢰를 먼저 잃습니다.

그래서 튜토리얼은 친절함보다 검증 가능성이 더 중요합니다. 각 단계가 실제로 동작해야 하고, 오류가 나면 어디서 확인할지 안내해야 합니다. 독자는 강의를 듣는 것이 아니라 직접 손을 움직이고 있기 때문입니다.

## 한눈에 보는 흐름

```mermaid
flowchart LR
    P[전제 조건] --> S[단계]
    S --> W[작은 성공]
    W --> N[다음 단계]
```

전제 조건을 먼저 밝히고, 단계별로 따라 하게 하고, 중간에 작은 성공을 보여 주고, 마지막에 다음 단계로 연결합니다. 이 구조가 튜토리얼의 기본 뼈대입니다.

## 핵심 용어

- **튜토리얼**: 학습을 목적으로 한 따라 하기 글입니다.
- **전제 조건**: 시작 전에 이미 갖춰져 있어야 하는 환경이나 지식입니다.
- **작은 성공**: 짧은 시간 안에 눈으로 확인할 수 있는 첫 결과입니다.
- **복구 경로**: 오류가 났을 때 다시 정상 흐름으로 돌아오는 방법입니다.
- **다음 단계**: 첫 성공 뒤에 이어서 볼 학습 주제입니다.

특히 작은 성공은 튜토리얼의 품질을 가르는 핵심입니다. 끝까지 가야만 결과가 나오는 구조보다, 중간에 한 번이라도 확인 가능한 지점을 만드는 편이 훨씬 좋습니다.

## Before / After

**Before**: "FastAPI를 배워 봅시다." (강의형)

**After**: "5분 안에 Hello World를 실행해 봅시다." (튜토리얼형)

앞 문장은 주제 소개에 가깝고, 뒤 문장은 행동과 결과가 분명합니다. 튜토리얼은 후자여야 합니다.

## 실습: 5분 튜토리얼 만들기

### 1단계 — 전제 조건 적기

```bash
python3 --version  # 3.11 or newer
```

전제 조건은 반드시 맨 앞에 둡니다. 독자가 지금 자기 환경으로 시작할 수 있는지 먼저 판단해야 하기 때문입니다. 버전 정보가 빠지면 동일한 튜토리얼도 사람마다 다른 결과를 낼 수 있습니다.

### 2단계 — 설치 단계 적기

```bash
pip install "fastapi[standard]"
```

설치 단계는 짧고 분명해야 합니다. 여러 선택지를 한 번에 제시하기보다, 지금 이 튜토리얼에 필요한 경로 하나만 안내하는 편이 낫습니다.

### 3단계 — 코드 보여 주기

```python
from fastapi import FastAPI
app = FastAPI()

@app.get("/")
def root():
    return {"hello": "world"}
```

코드는 튜토리얼의 목표를 드러내는 최소 수준이면 충분합니다. 여기서는 첫 응답을 확인하는 것이 목적이므로, 부가 설정을 늘어놓지 않습니다.

### 4단계 — 실행 단계 적기

```bash
fastapi dev main.py
```

실행 단계에서는 파일명, 명령 순서, 현재 위치가 흐리지 않게 써야 합니다. 튜토리얼이 막히는 지점은 대개 여기입니다.

### 5단계 — 결과 확인하기

```text
{"hello":"world"}
```

작은 성공은 눈으로 보이는 결과여야 합니다. 브라우저 응답, 콘솔 출력, 생성된 파일처럼 즉시 확인 가능한 기준이 있으면 독자는 스스로 맞게 따라왔는지 판단할 수 있습니다.

## 이 예시에서 봐야 할 점

- 전제 조건이 가장 먼저 나옵니다.
- 명령은 순서대로 나옵니다.
- 결과가 분명하게 보입니다.
- 작은 성공이 빠르게 도착합니다.

이 구조가 튜토리얼에서 중요한 이유는, 독자가 중간에 멈춰도 어디까지 왔는지 확인할 수 있기 때문입니다.

## 자주 하는 실수 다섯 가지

1. 전제 조건이 없어 시작부터 환경 차이로 막힙니다.
2. 명령 순서가 뒤섞여 독자가 그대로 따라 할 수 없습니다.
3. 작은 성공이 없어 끝까지 갔는지 판단하기 어렵습니다.
4. 오류가 났을 때 돌아갈 복구 안내가 없습니다.
5. 다음 단계가 없어 학습 흐름이 끊깁니다.

튜토리얼은 정보량이 많다고 좋은 글이 되지 않습니다. 잘게 쪼개고, 짧게 성공하게 하고, 다음으로 연결할수록 더 좋아집니다.

## 실무에서는 이렇게 드러납니다

좋은 라이브러리와 프레임워크의 공식 튜토리얼은 대체로 5분 안에 첫 성공을 보여 줍니다. 그 뒤에 개념 문서와 레퍼런스 문서가 이어집니다. 튜토리얼이 모든 것을 설명하려 들지 않는 이유가 여기 있습니다.

사내 온보딩 문서도 마찬가지입니다. 새 팀원이 30분 안에 로컬 환경을 띄우고 첫 요청을 보내 볼 수 있어야 이후 문서가 살아납니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 전제 조건을 숨기지 않습니다.
- 작은 성공은 3분 안팎에 도착하게 설계합니다.
- 오류마다 복구 한 줄을 붙입니다.
- 다음 단계는 너무 큰 점프가 아니어야 한다고 봅니다.
- 튜토리얼은 학습용이지 레퍼런스가 아니라고 구분합니다.

시니어가 튜토리얼을 짧게 끊는 이유는, 독자를 과소평가해서가 아니라 첫 성공의 속도를 지키기 위해서입니다.

## 체크리스트

- [ ] 전제 조건이 처음에 적혀 있는가
- [ ] 단계 수가 다섯 개 안팎인가
- [ ] 중간에 작은 성공이 보이는가
- [ ] 오류 복구 한 줄이 필요한 곳에 있는가
- [ ] 마지막에 다음 단계가 이어지는가

## 연습 문제

1. 튜토리얼과 설명 글의 차이를 한 줄로 적어 보세요.
2. 작은 성공이 왜 중요한지 한 줄로 적어 보세요.
3. 자신이 아는 도구 하나를 골라 3단계 튜토리얼 개요를 써 보세요.

## 정리 및 다음 단계

튜토리얼은 독자가 읽는 글이 아니라 따라 하는 글입니다. 그래서 전제 조건, 단계, 작은 성공, 복구 안내, 다음 단계가 모두 필요합니다. 특히 첫 성공을 빨리 보여 줄수록 더 좋은 튜토리얼이 됩니다.

다음 글에서는 개인 경험을 담는 블로그 글과 팀의 공식 진실을 담는 문서를 어떻게 구분해야 하는지 살펴보겠습니다. 다음 글은 **블로그와 문서 차이**입니다.
