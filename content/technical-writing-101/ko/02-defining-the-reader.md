# Technical Writing 101 (2/10): 독자 정의하기

같은 FastAPI 예제라도 처음 API를 만드는 주니어에게는 친절한 입문이 될 수 있고, 운영 장애를 잡는 온콜 엔지니어에게는 쓸모없는 장황한 설명이 될 수 있습니다. 글이 틀린 게 아니라 독자가 빗나간 것입니다.

문서를 읽는 사람을 한 덩어리로 놓고 쓰기 시작하면 예제 길이, 용어 수준, 생략 가능한 배경지식이 모두 흔들립니다. 반대로 독자를 한 사람처럼 구체화하면 무엇을 설명하고 무엇을 과감히 버려야 할지가 빠르게 정리됩니다.

이 글은 기술 글쓰기 101 시리즈의 2번째 글입니다.

여기서는 독자 모델을 페르소나, 전제 지식, 목표, 비목표로 나누어 글의 범위를 좁히는 방법을 다룹니다.

![Technical Writing 101 2장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/technical-writing-101/02/02-01-concept-at-a-glance.ko.png)
*Technical Writing 101 2장 흐름 개요*
> 선명한 독자 한 명을 먼저 세우면, 글의 지식 수준과 목표와 범위가 그 독자 주변으로 자동 정렬됩니다.

## 먼저 던지는 질문

- 왜 모두를 위한 글은 결국 아무도 제대로 돕지 못할까요?
- 페르소나를 만들면 문장이 왜 더 선명해질까요?
- 전제 지식과 목표를 먼저 적어 두면 범위가 어떻게 줄어들까요?

## 이 글에서 배울 것

- 페르소나 만들기
- 전제 지식 정리하기
- 목표 맞추기
- 범위 좁히기
- 예제 난이도 맞추기

## 왜 중요한가

독자가 흐리면 문장도 흐려집니다. 누구를 돕는 글인지 정하지 않으면 설명 깊이, 예제 길이, 용어 선택, 비목표가 모두 흔들립니다.

## 한눈에 보는 멘탈 모델

> 멘탈 모델: 선명한 독자 한 명을 먼저 세우면, 글의 지식 수준과 목표와 범위가 그 독자 주변으로 자동 정렬됩니다.

## 핵심 용어

- **persona**: 독자의 모델입니다.
- **prerequisite**: 전제 지식입니다.
- **goal**: 독자가 다음에 하게 될 일입니다.
- **scope**: 글이 다루는 범위입니다.
- **non-goal**: 글이 다루지 않는 범위입니다.

## 전후 비교

**Before**: "A post for developers."

**After**: "A post for a first-year Python engineer learning FastAPI."

## 같은 기능도 독자에 따라 완전히 달라집니다

| 독자 | 알고 있는 것 | 지금 필요한 것 | 이 글에서 빼야 할 것 |
| --- | --- | --- | --- |
| 입문자 | Python 문법, `pip` | 첫 FastAPI 엔드포인트 | 배포 전략, 성능 튜닝 |
| 리뷰어 | API 기본기 | 문서의 빠진 전제 지식 | 설치 세부 과정 전체 |
| 온콜 엔지니어 | 운영 환경 | 장애 대응 절차와 점검 순서 | 기초 개념 재설명 |

같은 `/health` 엔드포인트 예제라도 입문자에게는 라우트 선언과 실행 명령이 중요하고, 온콜 엔지니어에게는 로그 위치와 재현 경로가 더 중요합니다. 그래서 페르소나는 문체 취향을 고르는 장식이 아니라, 무엇을 남기고 무엇을 잘라낼지 결정하는 편집 기준입니다.

## 현실적인 페르소나 사례 3가지

실제 프로젝트에서 사용할 수 있는 구체적인 페르소나 예시를 봅니다.

### 페르소나 A: 신입 주니어 개발자

```python
persona_junior = {
    "name": "민지",
    "role": "주니어 백엔드 개발자 (경력 6개월)",
    "knows": ["Python 기본 문법", "pip 설치", "git add/commit/push"],
    "unknown": ["async/await", "ORM", "테스트 작성", "Docker"],
    "goal": "회사에서 배정받은 사용자 CRUD API를 3일 안에 만들기",
    "non_goal": ["프로덕션 배포", "성능 튜닝", "CI/CD 설정"],
    "pain_point": "FastAPI 공식 문서가 너무 방대해 어디서부터 시작할지 모르겠음"
}
```

**이 페르소나를 위한 글의 특징:**

- 모든 단계를 상세히 설명
- 코드 예제는 10줄 이하로 유지
- 예상 출력을 반드시 표시
- "왜 이렇게 해야 하나요?" 섹션 필수

### 페르소나 B: 중견 개발자

```python
persona_mid = {
    "name": "지훈",
    "role": "백엔드 개발자 (경력 3년)",
    "knows": ["Flask 경험", "REST API 설계", "PostgreSQL", "Docker 기초"],
    "unknown": ["FastAPI 의존성 주입", "Pydantic 고급 기능", "WebSocket"],
    "goal": "기존 Flask 프로젝트를 FastAPI로 마이그레이션하기",
    "non_goal": ["기초 문법 학습", "Python 설치"],
    "pain_point": "Flask와 FastAPI의 차이점을 모르겠고, 특히 의존성 주입 개념이 낯설음"
}
```

**이 페르소나를 위한 글의 특징:**

- Flask와의 비교 위주로 설명
- 기초 개념 설명은 생략 가능
- 트레이드오프 중심으로 작성
- 마이그레이션 스크립트 예시 포함

### 페르소나 C: 온콜 엔지니어

```python
persona_oncall = {
    "name": "예진",
    "role": "SRE 엔지니어 (경력 5년)",
    "knows": ["다양한 프레임워크 경험", "K8s 운영", "모니터링"],
    "unknown": ["FastAPI 내부 동작"],
    "goal": "새벽 3시 프로덕션 장애를 5분 안에 해결하기",
    "non_goal": ["튜토리얼 따라하기", "기초 개념 학습"],
    "pain_point": "빠른 시간 안에 로그에서 문제를 찾고 복구하는 법을 알아야 함"
}
```

**이 페르소나를 위한 글의 특징:**

- 체크리스트 형식
- 기초 설명 완전 생략
- 로그 위치, 메트릭 확인법 중심
- "이 증상이면 이 해결책" 패턴
## 독자 매트릭스: 경험과 공감 수준으로 독자 나누기

독자를 입문/중급/고급으로만 나누면 중요한 측면을 놀칩니다. 경험과 공감 수준 두 축으로 나누면 더 정밀한 독자 모델을 만들 수 있습니다.

| 경험 \ 공감 | 문제를 모릅니다 | 문제를 알고 있습니다 | 문제를 겪고 있습니다 |
| --- | --- | --- | --- |
| **입문** | 입문 튜토리얼 | 문제 중심 입문 가이드 | 트러블슈팅 가이드 |
| **중급** | 심화 가이드 | 비교 가이드 | 경험 공유 블로그 |
| **고급** | 레퍼런스 | 설계 논의 | 시스템 설계 블로그 |

**예시:**

- **입문 + 문제 모름**: "FastAPI로 첫 API 만들기" (기초 튜토리얼)
- **입문 + 문제 인지**: "왜 FastAPI가 Flask보다 빠를까" (비교)
- **입문 + 문제 겪는 중**: "FastAPI 422 오류 해결하기" (트러블슈팅)

같은 입문자라도 현재 겪고 있는 문제에 따라 필요한 글의 유형이 달라집니다. 공감 수준을 함께 고려하면 독자의 현재 상황에 맞는 글을 쓸 수 있습니다.

## 페르소나 질문 테스트

페르소나를 만든 후 아래 질문에 답할 수 있는지 확인하면 페르소나의 품질을 검증할 수 있습니다.

1. **이 독자는 지금 무엇을 할 수 있나요?**
   → 전제 지식을 구체적으로 적었는지 확인

2. **이 독자는 무엇을 할 수 없나요?**
   → 빈칸을 구체적으로 적었는지 확인

3. **이 독자는 이 글을 읽은 후 무엇을 할 거예요?**
   → 목표가 명확한지 확인

4. **이 독자는 이 글을 읽은 후 무엇을 하지 않아도 되나요?**
   → 비목표가 명확한지 확인

5. **이 독자가 막힐 곳은 어디인가요?**
   → 트러블슈팅 섹션에 필요한 내용 확인

이 다섯 가지 질문에 구체적으로 답할 수 없다면 페르소나를 더 구체화해야 합니다.

## 비목표 작성 패턴 3가지

비목표는 범위를 좋히는 가장 강력한 도구입니다. 아래 세 가지 패턴으로 비목표를 작성할 수 있습니다.

### 패턴 1: 단계 범위 제한

**예시**: "이 글은 로컬 개발 환경까지만 다루고, 프로덕션 배포는 다루지 않습니다."

**효과**: 독자가 어디까지 진행할지 명확히 알고, 그 다음은 다른 글에서 다룰 것으로 예측합니다.

### 패턴 2: 기술 스택 제한

**예시**: "이 글은 FastAPI만 다루고, Django와의 비교는 포함하지 않습니다."

**효과**: 독자가 글의 기술 범위를 명확히 이해하고, 비교 내용을 기대하지 않습니다.

### 패턴 3: 수준 제한

**예시**: "이 글은 성능 최적화, 보안 설정, 커스텀 미들웨어 제작은 다루지 않습니다."

**효과**: 고급 독자가 이 글이 입문용임을 빠르게 판단하고 넘어갑니다.
## 독자 프로필 템플릿

페르소나를 만들 때 매번 빈 종이에서 시작하면 시간이 오래 걸립니다. 아래 템플릿을 채우면 독자 정의가 훨씬 빨라집니다.

```python
persona = {
    "name": "지민",
    "role": "주니어 백엔드 개발자 (경력 1년)",
    "knows": [
        "Python 기본 문법",
        "함수와 클래스",
        "pip로 패키지 설치하기",
        "git add, commit, push"
    ],
    "unknown": [
        "async/await",
        "type hints",
        "FastAPI 라우터 구조",
        "의존성 주입"
    ],
    "goal": "FastAPI로 첫 번째 REST API 엔드포인트를 만들고 로컬에서 실행하기",
    "non_goal": [
        "프로덕션 배포",
        "데이터베이스 마이그레이션",
        "인증 로직",
        "성능 튜닝"
    ],
    "context": "회사에서 기존 Flask 프로젝트를 FastAPI로 전환하라는 과제를 받았고, 첫 프로토타입을 이번 주 안에 만들어야 합니다."
}
```

이 템플릿에서 가장 중요한 필드는 `non_goal`입니다. 비목표를 명시하면 글이 훨씬 짧아지고, 독자는 이 글이 자기 질문에 답하는지 빠르게 판단할 수 있습니다.

## 독자 수준별 문서 전략

같은 주제라도 독자의 지식 수준에 따라 용어, 설명 깊이, 예제 유형이 완전히 달라집니다.

| 독자 수준 | 용어 수준 | 설명 깊이 | 예제 유형 | 생략 가능 요소 |
| --- | --- | --- | --- | --- |
| 입문 (0-6개월) | 기술 용어마다 한 줄 설명 필요 | 모든 단계 상세히 설명 | Hello World, 최소 실행 예제 | 없음. 모든 전제를 명시 |
| 중급 (6개월-2년) | 표준 용어는 설명 생략 가능 | 핵심 단계만 설명 | 실무 시나리오, 확장 예제 | 설치, 기본 문법 |
| 고급 (2년+) | 도메인 전문 용어 사용 가능 | 개념보다 트레이드오프 중심 | 최적화, 에지 케이스, 아키텍처 | 기초 개념, 일반적 사용법 |

입문자를 위한 글에서 중급 독자의 용어 수준을 쓰면 독자는 첫 문단에서 이탈합니다. 반대로 고급 독자를 위한 글에서 기초 개념을 반복하면 독자는 인내심을 잃습니다. 글을 쓰기 전에 독자 수준을 먼저 고정해야 합니다.

## 흔한 실수: 모든 독자를 만족시키려는 글

"이 글은 입문자부터 고급 사용자까지 모두를 위한 것입니다"는 가장 위험한 문장입니다. 모두를 대상으로 잡으면 글은 다음과 같은 문제를 겪습니다.

### 문제 1: 설명 깊이를 결정할 수 없습니다

입문자를 위해 기초 개념을 설명하면 고급 독자는 지루해하고, 고급 독자를 위해 개념을 생략하면 입문자는 길을 잃습니다. 결국 둘 다 만족시키지 못합니다.

### 문제 2: 예제 난이도가 흔들립니다

입문자에게는 Hello World가 필요하고, 고급 독자에게는 복잡한 실무 시나리오가 필요합니다. 하나의 글에 두 종류 예제를 모두 넣으면 구조가 무너집니다.

### 문제 3: 비목표를 정할 수 없습니다

독자 범위가 넓으면 "이 글에서 다루지 않는 것"을 정할 수 없습니다. 결국 글은 계속 길어지고, 범위는 끝없이 확장됩니다.

### 해결책: 독자를 쪼개고 글을 나눕니다

모든 독자를 한 글에 담으려 하지 말고, 수준별로 글을 나누는 편이 훨씬 효과적입니다.

- **입문편**: "FastAPI 시작하기: 첫 번째 엔드포인트 만들기"
- **중급편**: "FastAPI 의존성 주입으로 깔끔한 라우터 구조 만들기"
- **고급편**: "FastAPI 비동기 처리 최적화: 병목 지점 찾기"

이렇게 나누면 각 글의 독자, 전제 지식, 목표, 비목표가 명확해지고, 독자는 자기 수준에 맞는 글을 빠르게 찾을 수 있습니다.
## 실습: 페르소나 카드 하나 만들기

### 1단계 — 이름과 역할

```python
persona = {"name": "Jimin", "role": "First-year Python backend"}
```

### 2단계 — 전제 지식

```python
knows = ["variables", "functions", "git basics"]
```

### 3단계 — 빈칸

```python
unknown = ["async", "type hints"]
```

### 4단계 — 목표

```python
goal = "Ship the first FastAPI endpoint"
```

### 5단계 — 비목표

```python
non_goal = ["deployment", "DB migrations"]
```

## 이 코드에서 먼저 볼 점

- 페르소나에 이름이 있습니다.
- 페르소나에 빈칸이 있습니다.
- 페르소나에 비목표가 있습니다.

## 자주 하는 실수 5가지

1. **모두를 대상으로 잡습니다.**
2. **전제 지식을 건너뜁니다.**
3. **목표가 모호합니다.**
4. **비목표가 없습니다.**
5. **예제가 너무 어렵습니다.**

## 실무에서는 이렇게 드러납니다

API 레퍼런스, 사용자 가이드, 튜토리얼은 모두 페르소나에 따라 갈라집니다. 독자가 다르면 필요한 배경과 예제가 달라지기 때문입니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 독자는 한 사람처럼 느껴져야 합니다.
- 비목표가 글을 줄여 줍니다.
- 예제는 전제 지식 범위에서 돌아야 합니다.
- 목표는 동사로 적습니다.
- 2주 뒤의 나도 독자입니다.

## 체크리스트

- [ ] 페르소나가 한 명인가
- [ ] 전제 지식이 세 가지 있는가
- [ ] 목표가 한 줄로 적혀 있는가
- [ ] 비목표가 하나 이상 있는가

## 연습 문제

1. persona의 뜻을 한 줄로 적어 보세요.
2. non-goal의 뜻을 한 줄로 적어 보세요.
3. prerequisite의 예시를 한 줄로 적어 보세요.

## 실전 앵커: 독자 정의 문서 한 장으로 범위 고정하기

팀 문서에서 독자 정의를 말로만 합의하면 곧바로 범위가 흔들립니다. 아래처럼 문서 한 장으로 고정하면 작성자와 리뷰어가 같은 기준을 공유할 수 있습니다.

### 독자 정의 카드 예시

| 항목 | 내용 |
| --- | --- |
| 독자 | FastAPI로 첫 API를 만드는 주니어 백엔드 개발자 |
| 전제 지식 | Python 기본 문법, `pip`, Git 기본 명령 |
| 목표 | 로컬에서 `GET /health` 엔드포인트를 실행하고 응답 확인 |
| 비목표 | 배포, 인증, 성능 최적화 |
| 실패 지점 | 가상환경 누락, 포트 충돌, 패키지 버전 불일치 |

### 같은 주제, 다른 독자 비교

| 주제: 인증 추가 | 입문자 글 | 온콜 엔지니어 글 |
| --- | --- | --- |
| 시작점 | 인증 개념 설명 | 장애 증상 분류 |
| 코드 길이 | 10-20줄 MWE | 진단 명령 + 로그 위치 |
| 성공 기준 | 로그인 성공 화면 | 401/403 원인 분리 완료 |
| 톤 | 단계별 안내 | 체크리스트 중심 |

### 리뷰어 질문 세트

- 이 독자가 모르는 용어를 본문이 설명했는가?
- 이 독자가 이미 아는 내용으로 지면을 낭비하지 않았는가?
- 목표 달성에 필요 없는 섹션이 들어갔는가?

이 세 질문만 적용해도 문서 길이는 줄고 완주율은 올라갑니다.

## 페르소나 기반 문장 변환 예시

- 모호한 문장: 이 문서는 API 개발자를 위한 문서입니다.
- 교정 문장: 이 문서는 Flask 경험은 있지만 FastAPI 의존성 주입이 낯선 1-3년차 백엔드 개발자를 위한 문서입니다.

- 모호한 문장: 배포를 진행합니다.
- 교정 문장: 이 글에서는 배포를 다루지 않으며, 로컬 실행 확인까지만 진행합니다.

## 정리

독자를 분명히 적으면 문장도, 예제도, 범위도 함께 선명해집니다. 특히 전제 지식과 목표와 비목표를 같이 적어 두면 글이 훨씬 짧고 유용해집니다. 다음 글에서는 이렇게 정한 독자를 기준으로 제목과 구조를 어떻게 설계해야 하는지 다룹니다.

## 처음 질문으로 돌아가기

- **왜 모두를 위한 글은 결국 아무도 제대로 돕지 못할까요?**
  독자를 넓게 잡으면 설명 깊이, 용어 수준, 예제 난이도, 비목표를 정할 수 없고, 결국 글은 친절해 보이지만 아무도 제대로 돕지 못하는 상태가 됩니다.
- **페르소나를 만들면 문장이 왜 더 선명해질까요?**
  페르소나를 세우면 글을 쓰는 동안 향하는 독자가 명확한 기준으로 고정되고, 문장도 그 독자를 중심으로 정렬됩니다.
- **전제 지식과 목표를 먼저 적어 두면 범위가 어떻게 줄어들까요?**
  비목표를 함께 적어 두면 글에서 다루지 않을 영역이 명확해지고, 불필요한 설명을 과감히 잘라낼 수 있습니다.
