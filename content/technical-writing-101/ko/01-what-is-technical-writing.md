# Technical Writing 101 (1/10): 기술 글쓰기란 무엇인가

기술 글쓰기를 처음 배울 때는 문장을 매끈하게 다듬는 일부터 떠올리기 쉽습니다. 하지만 현업에서 더 자주 문제를 만드는 것은 문체보다 방향입니다. 독자가 누구인지, 무엇을 하려는지, 어디서 막힐지를 잡지 못하면 문장이 아무리 자연스러워도 실행 가능한 문서가 되지 않습니다.

팀 문서, README, 튜토리얼, 장애 대응 메모가 모두 같은 이유로 중요합니다. 읽는 사람이 그 글을 바탕으로 설치하고, 실행하고, 비교하고, 복구해야 하기 때문입니다. 기술 글은 정보를 전달하는 글이면서 동시에 행동을 설계하는 글입니다.

여기서는 기술 글을 독자, 작업, 결과, 범위로 나누어 보는 기본 틀을 먼저 세웁니다.

*Technical Writing 101 1장 흐름 개요*
> 기술 글은 정보 전달을 넘어 독자의 행동을 설계하는 글입니다.

## 먼저 던지는 질문

- 기술 글쓰기는 일반 글쓰기와 무엇이 다를까요?
- 기술 글은 왜 설명에서 끝나지 않고 독자의 행동까지 이어져야 할까요?
- 기술 글의 목적을 독자, 작업, 결과, 범위로 나누면 무엇이 또렷해질까요?

## 이 글에서 배울 것

- 기술 글쓰기의 정의
- 일반 글과의 차이
- 세 가지 목적
- 독자의 행동
- 시리즈의 형태

## 왜 중요한가

기술 글은 설명하는 코드를 따라 오래 남는 경우가 많습니다. 그래서 오늘 적은 문장이 몇 주 뒤, 몇 달 뒤에도 누군가의 설치, 실행, 판단, 복구 작업에 직접 영향을 줍니다.

## 한눈에 보는 멘탈 모델

> 멘탈 모델: 기술 글은 정보를 나열하는 글이 아니라, 독자의 질문을 답으로 연결하고 그 답을 실제 행동으로 넘기는 글입니다.

## 핵심 용어

- **technical writing**: 기술 정보를 전달하는 산문입니다.
- **audience**: 독자입니다.
- **task**: 해야 할 일입니다.
- **outcome**: 결과입니다.
- **scope**: 경계입니다.

## 전후 비교

**Before**: "Python is a great language."

**After**: "A beginner can run Hello World in five minutes."

## 독자, 작업, 결과, 범위를 한 번에 맞추는 표

| 항목 | 약한 문장 | 더 나은 문장 |
| --- | --- | --- |
| 독자 | 개발자 | FastAPI로 첫 API를 만드는 주니어 백엔드 엔지니어 |
| 작업 | 환경 설정 | 가상환경을 만들고 의존성을 설치하기 |
| 결과 | 성공 | `(.venv)` 프롬프트가 보이고 `pip list`에 패키지가 나타나기 |
| 범위 | 파이썬 시작 | 로컬 개발 환경 준비까지만 다루기 |

이 표가 중요한 이유는 기술 글의 모호함이 대개 첫 문장에서 시작되기 때문입니다. 독자를 넓게 잡으면 작업도 넓어지고, 결과는 눈에 보이지 않게 되며, 결국 문서는 친절해 보이지만 실행은 어려운 상태로 남습니다. 한 문단을 쓰기 전에 이 네 칸을 먼저 채워 두면 글의 방향이 훨씬 또렷해집니다.

## 기술 글의 수명 주기와 유지보수 전략

기술 글은 발행 후에도 계속 관리해야 합니다. 특히 코드 예제를 포함한 글은 라이브러리 버전이 올라갈 때마다 검증해야 합니다.

### 단계 1: 발행 직후 (0-1주)

- 독자 피드백 수집
- 오타 수정
- 누락된 예제 보강

### 단계 2: 안정화 (1주-3개월)

- 라이브러리 메이저 버전 업데이트 시 코드 검증
- 새로 발견된 트러블슈팅 사례 추가
- 독자 질문 기반 FAQ 섹션 추가

### 단계 3: 유지보수 (3-12개월)

- 분기별 링크 체크
- 분기별 코드 예제 실행 검증
- 독자 피드백 수집 및 반영

### 단계 4: 아카이브 (12개월+)

- 더 이상 유지보수하지 않으면 글 상단에 "아카이브" 경고 표시
- "이 글은 [version] 기준으로 작성되었습니다" 명시

## 협업 문서를 위한 리뷰 체크리스트

팀에서 글을 작성할 때는 동료에게 리뷰를 요청하기 전에 아래 항목을 점검합니다.

- [ ] 모든 명령을 처음부터 따라 해 보았는가
- [ ] 링크가 모두 살아 있는가
- [ ] 코드 예제가 최신 버전에서 돌아가는가
- [ ] 오타가 없는가
- [ ] 독자, 전제 지식, 목표, 비목표가 명시되어 있는가
- [ ] 제목이 글의 내용과 일치하는가
- [ ] 각 섹션이 계층 구조를 따르는가 (What, How, Why)

이 체크리스트를 통과한 글은 리뷰어의 시간을 아껐 주고, 피드백 품질도 향상됩니다.

## 기술 글쓰기의 계층 모델

기술 글은 단일 계층이 아니라 세 계층으로 나눈어 생각하면 훨씬 구조화하기 쉽습니다.

### 계층 1: 무엇(What)을 하는가

독자가 달성할 최종 결과를 명시합니다. "가상환경을 만들고 활성화하기"처럼 동사형으로 적습니다.

### 계층 2: 어떻게(How) 하는가

단계별 명령과 예상 결과를 제시합니다. 복사해 붙여 넣을 수 있는 코드와 확인 방법을 함께 둡니다.

### 계층 3: 왜(Why) 하는가

이 작업이 필요한 이유, 다른 방법과의 차이, 하지 않았을 때 발생할 문제를 설명합니다. 이 계층이 없으면 독자는 명령은 따라 하지만 응용하지 못합니다.

**예시: 가상환경 설명**

**What**: 가상환경은 프로젝트별로 분리된 Python 환경입니다.

**How**: `python3 -m venv .venv && source .venv/bin/activate`

**Why**: 전역 패키지 충돌을 피하고, 프로젝트별 의존성을 분리하기 위해서입니다.

## 기술 글 작성 순서

많은 사람이 글을 위에서부터 아래로 순서대로 쓰기 시작합니다. 하지만 기술 글은 다음 순서로 작성하는 편이 훨씬 효율적입니다.

### 1단계: 독자와 목표 정하기

글을 쓰기 전에 독자, 전제 지식, 목표, 비목표를 먼저 적습니다. 이 네 가지를 적어 두면 본문을 쓰는 동안 방향이 흔들리지 않습니다.

### 2단계: 개요부터 만들기

제목과 H2 헤딩만 먼저 나열합니다. 그 헤딩만 읽어도 독자가 흐름을 이해할 수 있는지 확인합니다.

### 3단계: 코드 예제 확보하기

본문을 쓰기 전에 실행 가능한 코드 예제를 먼저 만듭니다. 예제가 실제로 돌아가는지 확인하고, 출력을 확보합니다.

### 4단계: 본문 쓰기

개요를 재우고 코드를 배치하며 설명을 채웁니다. 각 섹션마다 What, How, Why를 모두 다룹니다.

### 5단계: 검증하기

동료에게 리뷰를 요청하거나, 글의 독자와 비슷한 사람에게 테스트를 부탁합니다. 모든 명령을 처음부터 따라 해 보고, 막히는 지점을 기록합니다.

## 기술 글에서 피해야 할 AI 표현 10가지

기술 글은 구체적이고 실행 가능해야 합니다. 아래는 삭제해야 할 모호한 표현들입니다.

1. **"robust"** → 대신 어떤 오류를 처리하는지 명시하기
2. **"seamlessly"** → 대신 통합 단계 명시하기
3. **"powerful"** → 대신 구체적 기능 나열하기
4. **"easily"** → 대신 몇 줄의 코드로 가능하다고 쓰기
5. **"intuitive"** → 대신 설명 없이 사용 가능한 부분 짚기
6. **"simply"** → 대신 실제 단계 수 명시하기
7. **"leverage"** → 대신 "사용하기" 또는 "활용하기"
8. **"cutting-edge"** → 대신 버전 번호와 출시일 명시하기
9. **"혁신적"** → 대신 기존 방법과의 구체적 차이 설명하기
10. **"효율적"** → 대신 실행 시간, 메모리 사용량 등 지표 제시하기

## 테크니컬 라이팅의 유형

기술 글쓰기는 목적과 독자와 형식에 따라 여러 유형으로 나뉩니다. 각 유형은 작성 난이도, 유지보수 빈도, 요구되는 문체가 다릅니다.

| 유형 | 목적 | 독자 | 분량 | 특징 |
| --- | --- | --- | --- | --- |
| API 문서 | 함수, 클래스 레퍼런스 | 통합 개발자 | 짧고 반복적 | 정확성, 완전성, 검색 가능성 |
| 튜토리얼 | 학습 경로 제공 | 입문자 | 중간 길이 | 단계별 구조, 실행 예제, 검증 지점 |
| README | 프로젝트 소개 | 기여자, 사용자 | 짧음 | Quick Start, 설치, 예제, 라이선스 |
| 블로그 | 경험 공유 | 동료 엔지니어 | 중간~긴 길이 | 스토리, 트러블슈팅, 의사결정 배경 |

각 유형이 요구하는 문체와 구조는 다르지만, 공통점은 분명합니다. 모두 독자가 다음 행동을 할 수 있게 만들어야 합니다. API 문서는 함수 호출을, 튜토리얼은 프로젝트 완성을, README는 설치를, 블로그는 문제 해결을 돕습니다.

## 좋은 기술 문서의 공통 특징

형식은 달라도 좋은 기술 글은 다섯 가지 공통 특징을 보입니다.

### 1. 독자가 분명합니다

"개발자"가 아니라 "FastAPI로 첫 API를 만드는 주니어"처럼 구체적입니다. 독자를 좁게 잡을수록 설명 깊이, 용어 선택, 전제 지식이 또렷해집니다.

### 2. 명령이 실행됩니다

복사해 붙여 넣으면 바로 돌아가야 합니다. 환경 변수, 버전, 경로가 모두 갖춰져 있어야 합니다. 실행 불가능한 예제는 독자를 가르치지 못하고 좌절시킵니다.

### 3. 결과가 눈에 보입니다

"성공했습니다"가 아니라 "터미널에 `(.venv)` 프롬프트가 나타납니다"처럼 검증 가능해야 합니다. 결과를 눈으로 확인할 수 없으면 독자는 다음 단계로 넘어갈 확신을 갖지 못합니다.

### 4. 범위가 명확합니다

무엇을 다루고, 무엇을 다루지 않는지 미리 밝힙니다. 비목표를 적어 두면 독자는 이 글이 자기 질문에 답하는지 빠르게 판단할 수 있습니다.

### 5. 낡은 정보를 지웁니다

오래된 버전, 비추천 방법, 깨진 링크는 신뢰를 깎습니다. 최신 상태를 유지하거나, 유지할 수 없다면 아카이브 경고를 붙여야 합니다.

## 나쁜 문장과 좋은 문장 비교

같은 내용이라도 문장 구조에 따라 명확성이 크게 달라집니다. 아래는 흔히 보이는 나쁜 문장을 좋은 문장으로 고친 예시입니다.

### 예시 1: 독자가 모호한 문장

**나쁜 문장**: "이 가이드는 개발자를 위한 것입니다."

**좋은 문장**: "이 가이드는 Python을 처음 배우는 주니어 백엔드 개발자를 위한 것입니다."

**이유**: 독자가 구체적일수록 전제 지식, 예제 난이도, 생략 가능한 배경이 명확해집니다.

### 예시 2: 결과가 보이지 않는 문장

**나쁜 문장**: "설치가 성공하면 준비가 끝납니다."

**좋은 문장**: "설치가 성공하면 터미널에 `(.venv)` 프롬프트가 나타나고, `pip list`를 실행했을 때 패키지 목록이 출력됩니다."

**이유**: 독자가 눈으로 확인할 수 있는 결과를 제시해야 다음 단계로 안전하게 넘어갈 수 있습니다.

### 예시 3: 명령이 실행되지 않는 문장

**나쁜 문장**: "가상환경을 만드세요."

**좋은 문장**: "터미널에서 `python3 -m venv .venv`를 실행하세요."

**이유**: 독자가 복사해 붙여 넣을 수 있는 명령을 제시해야 실제 행동으로 이어집니다.

## 기술 글쓰기의 평가 기준

기술 글은 다음 네 가지 기준으로 평가할 수 있습니다.

| 기준 | 설명 | 검증 방법 |
| --- | --- | --- |
| 실행 가능성 | 독자가 명령을 그대로 따라 할 수 있는가 | 명령을 복사해 붙여 넣고 실행해 보기 |
| 결과 검증 가능성 | 독자가 성공 여부를 눈으로 확인할 수 있는가 | 결과 문장이 구체적인지 확인하기 |
| 범위 명확성 | 글이 다루는 것과 다루지 않는 것이 분명한가 | 비목표 섹션이 있는지 확인하기 |
| 유지 가능성 | 글을 오래 유지보수할 수 있는 구조인가 | 링크와 코드를 분기별로 점검할 수 있는지 확인하기 |

이 기준은 글을 발행하기 전뿐 아니라, 동료에게 리뷰를 요청하거나, 기존 문서를 개선할 때도 사용할 수 있습니다.
## 실습: 짧은 기술 단락 하나 만들기

### 1단계 — 독자 고르기

```python
audience = "Python beginners"
```

### 2단계 — 작업 고르기

```python
task = "Create and activate a virtual environment"
```

### 3단계 — 명령 제시하기

```bash
python3 -m venv .venv
source .venv/bin/activate
```

### 4단계 — 결과 적기

```python
result = "(.venv) shows up in the prompt"
```

### 5단계 — 다음 행동 적기

```python
next_step = "pip install requests"
```

## 이 코드에서 먼저 볼 점

- 독자가 가장 먼저 나옵니다.
- 명령은 짧습니다.
- 결과는 눈으로 확인할 수 있습니다.

## 기술 글의 실패 사례 3가지

### 사례 1: 독자를 너무 넓게 잡은 글

"이 튜토리얼은 개발자를 위한 것입니다"라고 시작한 글은 초보자에게는 어렵고, 숙련자에게는 불필요합니다. 독자를 좁게 잡으면 설명 깊이가 명확해집니다.

### 사례 2: 결과를 명시하지 않은 글

"설치가 완료되었습니다"로 끝나는 단계는 독자에게 확신을 주지 못합니다. 터미널 프롬프트, 로그 출력, 파일 존재, HTTP 응답처럼 눈으로 확인할 수 있는 결과를 제시해야 합니다.

### 사례 3: 오래된 정보를 방치한 글

라이브러리가 메이저 버전을 올리면 API가 바뀌고, 설치 명령이 바뀌며, 설정 구조가 달라집니다. 이때 글을 업데이트하지 않으면 독자는 글을 따라 했지만 실패하고, 글의 신뢰를 잃게 됩니다.
## 자주 하는 실수 5가지

1. **독자가 모호합니다.**
2. **이론이 너무 많습니다.**
3. **복사해 붙여 넣을 수 없는 명령을 올립니다.**
4. **눈에 보이는 결과가 없습니다.**
5. **다음 단계가 없습니다.**

## 실무에서는 이렇게 드러납니다

회사 내부 문서, 오픈소스 README, 발표 슬라이드도 모두 기술 글쓰기입니다. 형식은 달라도 공통점은 하나입니다. 읽는 사람이 그 글을 바탕으로 무언가를 해야 한다는 점입니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 독자의 시간을 아낍니다.
- 명령은 적힌 그대로 돌아가야 합니다.
- 결과는 눈으로 보여야 합니다.
- 오래된 정보는 지웁니다.
- 링크는 살아 있어야 합니다.

## 실전 앵커: 모호한 글을 실행 문서로 바꾸는 전환 예시

현업 문서에서 가장 자주 보이는 문제는 정의는 있는데 실행이 없는 상태입니다. 아래 예시는 같은 내용을 어떻게 실행 문서로 바꾸는지 보여 줍니다.

| 항목 | 바꾸기 전 | 바꾼 뒤 |
| --- | --- | --- |
| 목표 문장 | Python 환경을 준비합니다. | Python 3.11 가상환경을 만들고 `requests` 설치까지 확인합니다. |
| 성공 기준 | 설치가 끝납니다. | 프롬프트에 `(.venv)`가 보이고 `pip show requests`가 출력됩니다. |
| 범위 | 입문 가이드 | 로컬 개발 환경 준비까지만 다룹니다. |
| 비목표 | 없음 | 배포, CI, Docker는 다루지 않습니다. |

### 문장 교정 패턴

- 약한 문장: 개발에 필요한 환경을 설정합니다.
- 교정 문장: `python3 -m venv .venv`를 실행해 가상환경을 만들고 활성화합니다.
- 확인 문장: `which python` 결과가 `.venv` 경로를 가리키면 성공입니다.

### 적용 템플릿

```text
독자:

```

이 여섯 줄을 먼저 채우고 본문을 작성하면, 글이 설명 중심에서 행동 중심으로 자연스럽게 이동합니다.

## 공통 실수 교정표

| 실수 | 왜 문제인가 | 바로 고치는 방법 |
| --- | --- | --- |
| 독자 누락 | 난이도와 용어가 흔들립니다. | 도입부에 독자와 전제 지식을 1문장으로 고정합니다. |

## 정리

기술 글은 정보를 설명하는 데서 멈추지 않고, 독자가 바로 행동할 수 있게 만들어야 합니다. 그래서 독자, 작업, 명령, 결과, 다음 단계가 모두 분명해야 합니다. 다음 글에서는 같은 내용을 써도 누구를 위해 쓰느냐에 따라 문장이 어떻게 달라지는지 봅니다.
