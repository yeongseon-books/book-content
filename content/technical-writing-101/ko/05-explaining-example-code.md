# 예제 코드 설명하기

## 이 글에서 다룰 문제

- 코드 블록을 그대로 붙였는데도 독자가 왜 핵심을 놓칠까요?
- 최소 예제, 설명 줄, 실행 명령, 출력 결과는 어떤 순서로 보여 줘야 할까요?
- 코드 안 주석과 코드 밖 설명은 언제 나눠 쓰는 편이 좋을까요?
- 전체 코드 링크를 따로 주는 방식이 왜 본문 가독성을 높일까요?

> 예제 코드의 길이보다 중요한 것은, 독자가 무엇을 봐야 하는지 한 줄로 짚어 주는 일입니다.

> 기술 글쓰기 101 시리즈 (5/10)

기술 글에서 코드가 나온다고 해서 설명이 저절로 된 것은 아닙니다. 오히려 코드가 길수록 독자는 어디를 봐야 하는지 놓치기 쉽습니다. 글쓴이는 친절하게 많이 보여 줬다고 생각하지만, 독자는 중요한 두세 줄을 찾다가 지쳐 버립니다.

좋은 예제 코드는 전체 구현을 다 보여 주는 코드가 아니라, 지금 설명하려는 개념이나 동작을 가장 짧게 드러내는 코드입니다. 그리고 그 짧은 코드 옆에는 “이 줄이 왜 중요한가”를 짚는 설명이 있어야 합니다.

## 왜 중요한가

실행 가능한 예제는 독자의 손으로 도달해야 비로소 학습이 됩니다. 코드를 읽는 것과 코드를 직접 돌려 보는 것은 완전히 다른 경험입니다. 기술 글에서 예제 코드는 독자의 이해를 확인하는 가장 빠른 수단이기도 합니다.

그래서 예제 코드는 완성도보다 전달력이 먼저입니다. 적은 줄 수, 선명한 입력과 출력, 복사해 붙여 넣을 수 있는 실행 형태가 중요합니다. 이 세 가지가 갖춰지면 독자는 본문을 읽으면서 동시에 검증할 수 있습니다.

## 한눈에 보는 흐름

예제 코드를 설명할 때는 다음 흐름이 안정적입니다.

```mermaid
flowchart LR
    M[최소 예제] --> H[핵심 설명]
    H --> R[실행]
    R --> O[출력]
```

최소 예제를 먼저 보여 주고, 코드 밖에서 핵심을 짚고, 직접 실행하게 하고, 마지막에 출력으로 성공을 확인합니다. 이 순서가 있어야 독자가 예제를 읽기만 하지 않고 실제로 써 보게 됩니다.

## 핵심 용어

- **MWE**: 동작을 재현하는 최소 실행 예제입니다.
- **설명 줄(callout)**: 코드 밖에서 핵심을 짚는 짧은 문장입니다.
- **인라인 주석**: 코드 안에 붙는 주석입니다.
- **fixture**: 예제에서 사용하는 샘플 데이터입니다.
- **snippet**: 전체 코드 중 필요한 일부만 떼어 낸 짧은 조각입니다.

이 용어를 구분해 두면 글을 쓸 때 “무엇을 코드로 보여 줄지”와 “무엇을 문장으로 설명할지”를 더 쉽게 나눌 수 있습니다.

## Before / After

**Before**: 200줄짜리 코드 덤프

**After**: 8줄짜리 MWE와 2줄짜리 설명

앞 방식은 전체 맥락은 많지만 초점이 없습니다. 뒤 방식은 적은 코드로 핵심을 뚜렷하게 보여 줍니다. 기술 글은 대개 후자를 더 필요로 합니다.

## 실습: 예제 하나 설명해 보기

### 1단계 — 최소 코드 보여 주기

```python
def add(a, b):
    return a + b
```

이 예제는 짧고 바로 이해할 수 있습니다. 설명하려는 것이 “두 값을 받아 결과를 돌려주는 함수”라면 이 정도면 충분합니다. 처음부터 입력 검증, 타입 힌트, 예외 처리까지 모두 넣을 필요는 없습니다.

### 2단계 — 코드 밖에서 핵심 짚기

```python
# Highlight: take two numbers and return their sum
```

핵심 설명을 코드 밖에 두면 독자가 코드와 설명을 분리해 읽을 수 있습니다. 모든 줄에 주석을 달기 시작하면 오히려 시선이 분산됩니다. 코드 안 주석은 정말 필요한 경우에만 쓰고, 기본 설명은 바깥 문장으로 처리하는 편이 낫습니다.

### 3단계 — 실행 방법 보여 주기

```bash
python3 -c "from m import add; print(add(2, 3))"
```

예제를 읽은 뒤 곧바로 실행할 수 있어야 합니다. “이 코드는 적당히 파일에 넣고 실행해 보세요”보다, 바로 붙여 넣을 수 있는 한 줄 명령이 훨씬 낫습니다.

### 4단계 — 출력 보여 주기

```text
5
```

출력은 예제의 종착점입니다. 출력이 없으면 독자는 성공 기준을 모릅니다. 특히 입문 글에서는 예상 출력이 있어야 자신이 맞게 따라왔는지 바로 확인할 수 있습니다.

### 5단계 — 전체 코드 링크 연결하기

```python
full_code_url = "https://github.com/example/repo/blob/main/m.py"
```

본문은 핵심만 보여 주고, 전체 구현은 링크로 분리하는 편이 좋습니다. 그래야 본문 가독성을 지키면서도 궁금한 독자에게 충분한 맥락을 줄 수 있습니다.

## 이 예시에서 봐야 할 점

- 코드는 최소한으로 유지했습니다.
- 핵심 설명은 코드 밖에 두었습니다.
- 실행 명령과 출력이 모두 있습니다.
- 전체 구현은 링크로 분리했습니다.

이렇게 하면 본문은 가볍게 읽히고, 예제는 실제로 따라 할 수 있습니다. 기술 글에서 코드의 양보다 더 중요한 것은 독자가 헤매지 않는 흐름입니다.

## 자주 하는 실수 다섯 가지

1. 예제가 너무 길어 핵심이 묻힙니다.
2. 주석이 많아 코드 자체를 읽기 어려워집니다.
3. 출력이 없어 성공 여부를 확인할 수 없습니다.
4. 버전 정보가 없어 같은 명령이 환경마다 다르게 동작합니다.
5. 복사해 붙여 넣으면 깨지는 조각 코드를 그대로 올립니다.

예제 코드는 보여 주는 자료이면서 동시에 검증 도구입니다. 그래서 읽기 쉬워야 하고, 실제로도 돌아가야 합니다.

## 실무에서는 이렇게 드러납니다

오픈소스 README의 Quick Start 절을 보면 거의 항상 비슷한 패턴이 나옵니다. 짧은 설치 명령, 최소 코드, 실행 방법, 실제 출력, 전체 문서 링크입니다. 이 구조가 널리 쓰이는 이유는 독자의 진입 비용을 가장 낮추기 때문입니다.

사내 문서나 블로그에서도 마찬가지입니다. 예제 코드를 친절하게 많이 붙이는 것보다, 지금 문단이 무엇을 보여 주는 예제인지 정확히 설명하는 편이 더 효과적입니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 예제는 최소 단위로 잘라 보여 줍니다.
- 설명은 코드 밖에서 짧게 짚습니다.
- 출력은 실제 값이어야 한다고 봅니다.
- 버전은 가능한 한 고정합니다.
- 전체 코드는 링크로 분리해 본문 집중도를 지킵니다.

시니어가 예제를 짧게 유지하는 이유는 게으르기 때문이 아니라, 독자가 무엇을 배워야 하는지 더 분명히 만들기 위해서입니다.

## 체크리스트

- [ ] 예제가 열 줄 안팎으로 짧은가
- [ ] 코드 밖 설명이 한두 줄로 핵심을 짚는가
- [ ] 실행 명령을 그대로 복사할 수 있는가
- [ ] 출력 결과가 눈에 보이는가
- [ ] 전체 구현은 별도 링크로 제공되는가

## 연습 문제

1. MWE가 무엇인지 한 줄로 적어 보세요.
2. 설명 줄을 코드 밖에 두는 이유를 한 줄로 적어 보세요.
3. fixture 예시를 하나 만들어 한 줄로 적어 보세요.

## 정리 및 다음 단계

좋은 예제 코드는 많은 코드를 보여 주는 예제가 아니라, 핵심을 가장 짧게 드러내는 예제입니다. 최소 코드, 핵심 설명, 실행 명령, 출력, 전체 코드 링크가 갖춰지면 독자는 읽기와 실행을 함께 할 수 있습니다.
