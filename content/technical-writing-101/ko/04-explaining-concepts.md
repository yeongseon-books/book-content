# 개념 설명하기

## 이 글에서 다룰 문제

- 처음 보는 독자가 추상적인 기술 개념을 바로 이해하게 하려면 무엇부터 설명해야 할까요?
- 정의, 비유, 반례, 예시는 왜 함께 움직여야 할까요?
- 비유는 도움이 되지만 왜 경계를 잘못 그리면 오히려 오해를 키울까요?
- 짧은 코드와 흔한 오해를 함께 보여 주면 이해가 어떻게 빨라질까요?

> 좋은 개념 설명은 비유로 문을 열고, 반례로 경계를 세웁니다.

> 기술 글쓰기 101 시리즈 (4/10)

기술 문서에서 가장 자주 보는 문제 중 하나는, 용어 정의는 있는데 이해는 남지 않는다는 점입니다. 글쓴이는 설명했다고 느끼지만 독자는 여전히 흐릿합니다. 대개는 정의만 있고, 그 개념이 어디까지 맞고 어디서부터 틀리는지 보여 주지 않았기 때문입니다.

개념 설명은 단순한 사전 풀이가 아닙니다. 독자의 머릿속에 사용할 수 있는 모델을 심어 주는 일입니다. 그래서 한 줄 정의만으로는 부족하고, 익숙한 비유와 반례, 그리고 실제 예시가 함께 있어야 합니다.

## 왜 중요한가

개념이 흐리면 그 뒤 설명은 모두 모래성처럼 무너집니다. 캐시, 비동기, 메시지 큐, 인증, 인덱스처럼 기본 개념을 잘못 이해한 채 읽으면 이후의 코드와 아키텍처 설명도 계속 비틀립니다.

좋은 개념 설명은 이후 문단의 비용을 줄입니다. 한 번 분명하게 이해시켜 두면 반복 설명이 줄고, 독자도 스스로 새 예시를 해석할 수 있습니다. 그래서 입문 글일수록 개념 설명을 서두르지 말고 더 정교하게 다뤄야 합니다.

## 한눈에 보는 흐름

개념 설명은 보통 다음 순서가 가장 안정적입니다.

```mermaid
flowchart LR
    D[정의] --> A[비유]
    A --> C[반례]
    C --> E[예시]
```

정의는 핵심을 짧게 잡아 주고, 비유는 익숙한 감각으로 끌어오고, 반례는 오해를 막고, 예시는 실제 동작을 보여 줍니다. 이 네 가지가 모이면 독자는 개념을 기억하기 쉬워집니다.

## 핵심 용어

- **정의**: 개념을 한 줄로 묶어 주는 문장입니다.
- **비유**: 낯선 개념을 익숙한 사물이나 상황에 빗대는 설명입니다.
- **반례**: 이 개념이 적용되지 않는 경우를 보여 주는 예입니다.
- **풀이 예시**: 단계가 드러나는 실제 예시입니다.
- **오해**: 독자가 가장 쉽게 잘못 이해하는 지점입니다.

기술 글에서 특히 중요한 것은 반례와 오해입니다. “무엇인가”만 알려 주면 독자는 대충 비슷한 것까지 모두 같은 범주로 묶어 버리기 쉽습니다.

## Before / After

**Before**: "비동기는 동시에 실행하는 것이다." (오해)

**After**: "비동기는 기다리는 동안 다른 일을 할 수 있게 만드는 방식이다."

앞 문장은 동시성, 병렬성, 비동기를 한데 섞어 버릴 위험이 큽니다. 뒤 문장은 독자가 체감할 수 있는 행동으로 개념을 옮겨 줍니다.

## 실습: 개념 하나 설명해 보기

### 1단계 — 정의 쓰기

```python
definition = "A cache stores frequent answers ahead of time"
```

정의는 한 줄이면 충분합니다. 길어진 정의는 대개 핵심이 흐려졌다는 뜻입니다. 처음 읽는 독자에게는 짧고 단단한 문장이 더 잘 남습니다.

### 2단계 — 비유 붙이기

```python
analogy = "Side dishes you keep at the front of the fridge"
```

비유는 익숙해야 합니다. 냉장고 앞칸처럼 손이 자주 가는 위치를 떠올리면 캐시의 목적이 빠르게 전달됩니다. 다만 비유는 개념 전체와 일치하지 않으므로, 지나치게 밀어붙이면 오히려 오해를 부릅니다.

### 3단계 — 반례 제시하기

```python
counterexample = "Data you read only once does not belong in a cache"
```

반례는 경계를 그립니다. 모든 데이터를 캐시에 넣는 것이 아니라는 사실을 바로 짚어 주면, 독자는 캐시를 “빠른 저장소” 정도로 뭉뚱그리지 않게 됩니다.

### 4단계 — 코드 예시 보여 주기

```python
cache = {}
cache["user:1"] = {"name": "Jimin"}
```

짧은 예시는 개념을 손에 잡히게 만듭니다. 여기서는 딕셔너리 두 줄만으로도 캐시가 “자주 찾는 값을 미리 들고 있는 곳”이라는 느낌이 전달됩니다.

### 5단계 — 흔한 오해 적기

```python
misconception = "A cache can grow forever"
```

오해를 먼저 깨 주면 설명이 오래 갑니다. 캐시는 무한히 쌓아 두는 창고가 아니라, 크기와 만료 정책을 함께 고려해야 하는 장치라는 점을 이어서 설명할 수 있습니다.

## 이 예시에서 봐야 할 점

- 정의는 한 줄입니다.
- 비유는 일상 경험에 닿아 있습니다.
- 반례가 개념의 경계를 분명히 합니다.
- 코드는 짧아도 의미가 전달됩니다.
- 오해를 먼저 다뤄 뒤 문단의 혼란을 줄입니다.

개념 설명은 많이 적는다고 좋아지지 않습니다. 독자가 어디에서 막힐지를 먼저 예상하고, 그 지점을 해소하는 순서로 짜야 합니다.

## 자주 하는 실수 다섯 가지

1. 정의가 길어져 핵심이 첫 문장에서 사라집니다.
2. 비유를 지나치게 끌고 가다 실제 개념과 어긋납니다.
3. 반례가 없어 독자가 경계를 잘못 잡습니다.
4. 예시 코드가 너무 커져 개념보다 구현 디테일이 먼저 보입니다.
5. 흔한 오해를 건너뛰어 같은 질문이 반복됩니다.

개념 설명은 친절한 번역이 아니라, 오해를 줄이는 설계에 가깝습니다.

## 실무에서는 이렇게 드러납니다

좋은 사내 위키나 설계 문서는 대개 정의, 비유, 반례, 예시의 순서를 자연스럽게 갖고 있습니다. 읽는 사람도 그 구조를 따라가며 자기 머릿속 모델을 만듭니다.

반대로 “정의만 있고 예시는 없다”거나 “예시는 많은데 경계가 없다”면 팀 안에서도 같은 용어를 서로 다르게 쓰기 쉽습니다. 기술 개념은 공유 언어이기 때문에, 설명이 흐리면 협업 비용도 같이 올라갑니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 정의는 짧게 잡습니다.
- 비유는 익숙한 범위 안에서만 씁니다.
- 반례로 개념의 경계를 세웁니다.
- 예시는 바로 실행하거나 상상할 수 있어야 한다고 봅니다.
- 흔한 오해를 먼저 깨는 편이 전체 설명을 더 단단하게 만든다고 봅니다.

시니어가 개념 설명에 공을 들이는 이유는, 그 한 번의 설명이 이후 수많은 대화와 문서를 바꾸기 때문입니다.

## 체크리스트

- [ ] 한 줄 정의가 있는가
- [ ] 익숙한 비유가 하나 있는가
- [ ] 반례로 경계를 보여 주는가
- [ ] 예시 코드가 짧고 자급적인가
- [ ] 흔한 오해를 먼저 짚는가

## 연습 문제

1. 반례가 왜 필요한지 한 줄로 적어 보세요.
2. 풀이 예시가 단순 예시와 다른 점을 한 줄로 적어 보세요.
3. 자신이 설명하고 싶은 개념 하나를 골라 정의와 비유를 한 줄씩 써 보세요.

## 정리 및 다음 단계

개념 설명은 한 줄 정의로 시작하지만 거기서 끝나지 않습니다. 비유로 감각을 만들고, 반례로 경계를 세우고, 예시로 손에 잡히게 해야 독자가 실제로 이해합니다. 오해를 먼저 예상해 부수는 글일수록 오래 남습니다.

다음 글에서는 개념 설명에서 한 걸음 더 나아가, 짧은 코드 예제를 어떻게 보여 주고 어떻게 설명해야 독자가 따라오기 쉬운지 살펴보겠습니다. 다음 글은 **예제 코드 설명하기**입니다.
