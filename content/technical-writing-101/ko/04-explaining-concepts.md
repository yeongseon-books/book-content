# Technical Writing 101 (4/10): 개념 설명하기

새 개념을 설명할 때 가장 흔한 실수는 정의를 길게 늘어놓는 일입니다. 독자는 단어장을 원하지 않는데, 작성자는 정확성을 지키겠다는 마음으로 추상적인 문장을 계속 보태곤 합니다. 그 결과 개념은 더 선명해지지 않고 오히려 더 멀어집니다.

이 글은 기술 글쓰기 101 시리즈의 4번째 글입니다.

현업에서 통하는 개념 설명은 정의 하나로 끝나지 않습니다. 어디까지가 맞는지, 어디서부터 틀리는지, 실제 코드나 운영 판단에서는 어떤 모양으로 드러나는지를 함께 보여 줘야 개념이 손에 잡힙니다.

여기서는 정의, 비유, 반례, 예시를 묶어 처음 보는 독자에게 개념을 설명하는 방법을 다룹니다.

![Technical Writing 101 4장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/technical-writing-101/04/04-01-concept-at-a-glance.ko.png)
*Technical Writing 101 4장 흐름 개요*
> 좋은 개녕 설명은 정의로 중심을 잡고, 비유로 문을 열고, 반례로 경계를 그리고, 예시로 손에 잡히게 만듭니다.

## 먼저 던지는 질문

- 처음 보는 독자가 개념을 바로 이해하게 하려면 무엇부터 보여 줘야 할까요?
- 비유와 반례는 왜 같이 움직여야 할까요?
- 한 줄 정의만으로는 왜 설명이 오래 남지 않을까요?

## 이 글에서 배울 것

- 한 줄 정의
- 비유 쓰기
- 반례 쓰기
- 다이어그램 쓰기
- 풀이 예시 만들기

## 왜 중요한가

개념이 흐리면 그 뒤에 붙는 모든 설명은 모래성처럼 무너집니다. 한 번 잘못 잡힌 개념은 코드와 설계와 운영 판단까지 계속 비틀어 놓습니다.

## 한눈에 보는 멘탈 모델

> 멘탈 모델: 좋은 개념 설명은 정의로 중심을 잡고, 비유로 문을 열고, 반례로 경계를 긋고, 예시로 손에 잡히게 만듭니다.

## 핵심 용어

- **definition**: 한 줄 정의입니다.
- **analogy**: 비유입니다.
- **counterexample**: 반례입니다.
- **worked example**: 풀이 과정을 보여 주는 예시입니다.
- **misconception**: 흔한 오해입니다.

## 전후 비교

**Before**: "Async means running at the same time." (Wrong.)

**After**: "Async means doing other work while you wait."

## 정의가 통했는지 확인하는 가장 빠른 방법

캐시를 이렇게만 설명하면 부족합니다.

```text
캐시는 자주 쓰는 데이터를 저장하는 기술입니다.
```

이 문장만으로는 독자가 무엇을 넣고, 무엇을 빼야 하는지 판단하기 어렵습니다. 아래처럼 경계와 반례를 함께 두면 설명이 훨씬 실용적으로 바뀝니다.

```text
캐시는 자주 다시 계산하거나 다시 읽는 값을 잠시 앞당겨 저장하는 계층입니다.
한 번만 읽고 끝나는 대용량 파일은 캐시에 두지 않는 편이 낫습니다.
```

좋은 개념 설명은 독자가 다음 사례를 봤을 때 스스로 분류할 수 있게 만들어야 합니다. 즉, 정의를 암기하게 만드는 것이 아니라 예외를 판단하게 만드는 쪽이 더 중요합니다.

## 개념 설명의 깊이 조절 기법

같은 개념이라도 독자 수준에 따라 설명 깊이를 조절할 수 있습니다.

### 입문 독자를 위한 설명

**개념**: 캐시

**접근:**

1. **비유로 시작**: "냉장고 앞에 자주 먹는 반찬을 두는 것처럼..."
2. **간단한 정의**: "캐시는 자주 쓰는 값을 빠른 저장소에 둡는 기술입니다."
3. **간단한 코드 예시**:

```python
cache = {}
cache["user:1"] = {"name": "Jimin"}
result = cache.get("user:1")
```

4. **반례**: "한 번만 쓰는 데이터는 캐시하지 않는 편이 낫습니다."

### 중급 독자를 위한 설명

**개념**: 캐시

**접근:**

1. **정의 바로 제시**: "캐시는 원본보다 빠른 저장소에 복사본을 두는 계층입니다."
2. **Redis 예시로 바로 이동**:

```python
import redis

r = redis.Redis()
r.set("user:1", '{"name": "Jimin"}')
r.get("user:1")
```

3. **트레이드오프 제시**: "메모리를 더 쓰지만 시간을 절약합니다."

### 고급 독자를 위한 설명

**개념**: 캐시

**접근:**

1. **비교 테이블**:

| 전략 | 비용 | 복잡도 | 적합한 상황 |
| --- | --- | --- | --- |
| In-memory (dict) | 낮음 | 낮음 | 단일 프로세스 |
| Redis | 중간 | 중간 | 분산 시스템 |
| CDN | 높음 | 높음 | 정적 자산 |

2. **캐시 무효화 전략 비교**: LRU vs TTL vs 수동 삭제

## 전문 용어 도입 패턴 3가지

전문 용어를 처음 소개할 때는 다음 세 가지 패턴 중 하나를 선택합니다.

### 패턴 1: 즉시 정의형

**예시**: "FastAPI는 ASGI(비동기 서버 게이트웨이 인터페이스) 프레임워크입니다."

**사용 시점**: 용어가 글의 핵심이고 바로 이해해야 할 때

### 패턴 2: 미루기형

**예시**: "이 글에서 ASGI의 세부 동작은 다루지 않습니다. ASGI에 대해 더 알고 싶다는 [공식 문서](https://asgi.readthedocs.io/)를 참고하세요."

**사용 시점**: 용어가 비목표 영역에 있지만 언급은 해야 할 때

### 패턴 3: 점진적 노출형

**예시**: "첫 번째 글에서는 ASGI를 인급하지 않고, 두 번째 글에서 간단히 소개하고, 세 번째 글에서 상세히 다룹니다."

**사용 시점**: 시리즈 글에서 복잡한 개념을 여러 편에 걸쳐 설명할 때
## 추상 개념 설명 패턴 4가지

처음 보는 개념을 설명할 때는 정의만으로 끝내지 말고, 아래 네 가지 패턴을 조합해 독자가 여러 각도로 이해할 수 있게 만듭니다.

### 패턴 1: 비유로 낯선 것을 익숙한 것에 연결하기

**개념**: 캐시

**비유**: "냉장고 앞쪽에 자주 먹는 반찬을 두는 것처럼, 자주 계산하는 값을 빠른 저장소에 미리 둡니다."

**효과**: 독자가 일상 경험을 통해 개념의 목적을 빠르게 이해할 수 있습니다. 하지만 비유를 너무 멀리 끌고 가면 오히려 혼란을 줄 수 있으므로, 비유는 도입부에만 쓰고 이후 정확한 정의로 넘어가야 합니다.

### 패턴 2: 대조로 경계 그리기

**개념**: 동기 vs 비동기

**대조**:

- **동기**: 한 작업이 끝날 때까지 기다린 뒤 다음 작업을 시작합니다.
- **비동기**: 한 작업이 끝나길 기다리는 동안 다른 작업을 먼저 처리할 수 있습니다.

**효과**: 두 개념을 나란히 놓으면 차이점이 선명해지고, 독자는 어느 상황에서 어느 개념을 써야 할지 판단할 수 있습니다.

### 패턴 3: 시각화로 구조 보여 주기

**개념**: FastAPI 요청 흐름

**시각화**: 다이어그램으로 "클라이언트 → 라우터 → 의존성 → 비즈니스 로직 → 응답" 흐름을 그립니다.

**효과**: 텍스트로 설명하면 여러 문단이 필요한 구조를 한 장의 그림으로 압축할 수 있습니다. 특히 계층, 흐름, 경계가 중요한 개념은 다이어그램이 필수입니다.

### 패턴 4: 단계 분해로 과정 보여 주기

**개념**: HTTP 요청-응답 사이클

**단계 분해**:

1. 클라이언트가 HTTP 요청을 보냅니다.
2. 서버가 요청을 받아 라우터로 전달합니다.
3. 라우터가 적절한 핸들러 함수를 호출합니다.
4. 핸들러 함수가 응답 데이터를 만듭니다.
5. 서버가 HTTP 응답으로 클라이언트에 돌려줍니다.

**효과**: 복잡한 과정을 번호 붙인 단계로 나누면 독자가 전체 흐름을 순서대로 이해할 수 있습니다.

## 설명 깊이 판단 기준

같은 개념이라도 독자의 배경 지식에 따라 설명 깊이를 조절해야 합니다. 너무 자세히 설명하면 고급 독자는 지루해하고, 너무 생략하면 입문 독자는 길을 잃습니다.

| 독자 수준 | 설명 깊이 | 예시 분량 | 비유 사용 | 전제 지식 |
| --- | --- | --- | --- | --- |
| 입문 | 모든 단계 상세히 설명 | Hello World 수준 | 적극 사용 | Python 기본 문법만 가정 |
| 중급 | 핵심 개념만 설명 | 실무 시나리오 | 선택적 사용 | 웹 프레임워크 경험 가정 |
| 고급 | 개념보다 트레이드오프 중심 | 복잡한 최적화 사례 | 거의 쓰지 않음 | 아키텍처 설계 경험 가정 |

입문 독자에게는 "라우터는 URL을 함수에 연결하는 계층입니다"처럼 한 줄 정의부터 시작하고, 고급 독자에게는 "라우터 계층을 분리하면 의존성 주입 범위를 모듈별로 제한할 수 있지만, 초기 설정 복잡도가 올라갑니다"처럼 트레이드오프를 중심으로 설명합니다.

## 개념 설명의 세 가지 계층

개념을 설명할 때는 표면(표현), 중간(맥락), 깊은 층(함의)으로 나눈어 설명하면 독자가 점진적으로 이해할 수 있습니다.

### 예시: "async" 개념

**계층 1: 표면(표현)**

"async는 비동기를 의미합니다."

**계층 2: 중간(맥락)**

"async는 한 작업이 끄나길 기다리는 동안 다른 작업을 먼저 처리할 수 있는 프로그래밍 방식입니다. 네트워크 요청이나 파일 읽기처럼 대기 시간이 긴 작업에서 효과적입니다."

**계층 3: 깊은 층(함의)**

"async는 CPU 경합 작업이 아니라 I/O 경합 작업에서만 성능을 향상시킵니다. 만약 모든 작업이 CPU 만을 사용한다면 async를 써도 병렬 성능이 나아지지 않습니다."

### 효과

입문 독자는 계층 2까지만 읽어도 충분하고, 고급 독자는 계층 3에서 함의를 얻습니다.
## 전문 용어 처리 전략

기술 글에서 전문 용어를 완전히 피할 수는 없지만, 독자가 처음 보는 용어를 무방비로 만나게 해서는 안 됩니다.

### 전략 1: 처음 등장할 때 한 줄 정의 붙이기

**나쁜 예**: "FastAPI는 ASGI를 지원합니다."

**좋은 예**: "FastAPI는 ASGI(비동기 웹 서버 인터페이스)를 지원합니다."

### 전략 2: 약어보다 전체 이름 우선

첫 등장 시에는 전체 이름을 쓰고, 이후 반복 시 약어를 씁니다.

**예시**: "FastAPI는 Pydantic 모델로 요청 본문을 검증합니다. 이 Pydantic 모델은 타입 힌트를 바탕으로 자동 검증을 수행합니다."

### 전략 3: 독자가 알아야 하는 용어와 몰라도 되는 용어 구분

모든 용어를 설명하려 하지 말고, 독자의 목표에 필수적인 용어만 설명합니다.

- **알아야 하는 용어**: 라우터, 의존성, 엔드포인트
- **몰라도 되는 용어**: ASGI 내부 구현, Starlette 세부 동작

비목표 영역의 용어는 과감히 생략하거나, "이 글에서는 다루지 않지만, 궁금하면 [링크] 참고"로 처리합니다.

## 개념 설명에서 피해야 할 패턴 5가지

### 1. 정의를 계속 반복하기

나쁨 예: "캐시는 캐시이기 때문에..."

좋은 예: "캐시는 자주 쓰는 값을 빠른 저장소에 둘는 기술입니다."

### 2. 비유를 너무 멀리 끌고 가기

나쁨 예: "캐시는 냉장고 앞에 두는 반찬과 같습니다. 그리고 냉장고 문을 여는 것은..."

좋은 예: "캐시는 냉장고 앞에 두는 반찬과 같습니다. (이제 정확한 정의로 넣어갑니다)"

### 3. 반례 없이 정의만 나열하기

나셨 예: "캐시는 빠른 저장소에 값을 둔는 기술입니다."

좋은 예: "캐시는 빠른 저장소에 값을 둔는 기술입니다. 하지만 한 번만 쓰는 데이터는 캐시하지 않는 편이 낫습니다."

### 4. 코드 예시 없이 추상적 설명만 하기

나셨 예: "캐시는 메모리에 값을 저장합니다."

좋은 예: "캐시는 메모리에 값을 저장합니다. `cache = {}; cache['user:1'] = {'name': 'Jimin'}`"

### 5. 흔한 오해를 건너뛰기

나셨 예: "캐시는 빠릅니다. (오해를 다루지 않음)"

좋은 예: "캐시는 빠릅니다. 하지만 캐시가 무한정 커지는 않습니다. 메모리가 부족해지면 오래된 값을 지워야 합니다."
## 실습: 개념 하나 설명해 보기

### 1단계 — 정의

- 정의: `A cache stores frequent answers ahead of time`

### 2단계 — 비유

- 비유: `Side dishes you keep at the front of the fridge`

### 3단계 — 반례

- 반례: `Data you read only once does not belong in a cache`

### 4단계 — 코드 예시

```python
cache = {}
cache["user:1"] = {"name": "Jimin"}
```

### 5단계 — 흔한 오해

- 흔한 오해: `A cache can grow forever`

## 이 코드에서 먼저 볼 점

- 정의는 한 줄입니다.
- 비유는 일상에 닿아 있습니다.
- 반례가 경계를 그립니다.

## 자주 하는 실수 5가지

1. **정의가 너무 깁니다.**
2. **비유를 너무 멀리 끌고 갑니다.**
3. **반례가 없습니다.**
4. **코드 예시가 너무 큽니다.**
5. **흔한 오해를 건너뜁니다.**

## 실무에서는 이렇게 드러납니다

좋은 사내 위키 문서는 대개 정의, 비유, 반례, 예시 순서로 열립니다. 그 구조가 있어야 팀 안에서 같은 단어를 비슷한 뜻으로 쓰기 쉬워집니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 정의는 한 줄입니다.
- 비유는 익숙한 영역에 둡니다.
- 반례가 경계입니다.
- 예시는 실행할 수 있어야 합니다.
- 오해를 먼저 깨뜨립니다.

### 예시: 비동기 처리

- 한 줄 정의: 비동기는 대기 시간 동안 다른 작업을 처리하는 방식입니다.
- 비유: 빨래가 돌아가는 동안 설거지를 하는 것과 같습니다.
- 반례: CPU 계산만 많은 작업에는 성능 이득이 제한적입니다.

```python
import asyncio

async def fetch_user():
    await asyncio.sleep(1)
    return {"id": 1}
```

- 흔한 오해: 비동기를 쓰면 모든 코드가 자동으로 빨라진다.

| 입문 | 필수 | 필수 | 짧게 | 선택 |
| 중급 | 필수 | 필수 | 중간 | 필수 |
| 고급 | 요약 | 필수 | 핵심만 | 필수 |

이 표를 기준으로 쓰면 과설명과 과생략을 동시에 줄일 수 있습니다.

## 오해 제거 문장 패턴

- 오해 유발: 캐시는 항상 성능을 높입니다.
- 오해 제거: 캐시는 반복 조회가 많은 데이터에서만 효과적이며, 무효화 전략이 없으면 오히려 오류를 키울 수 있습니다.

## 정리

개념 설명은 정의 하나로 끝나지 않습니다. 비유로 감각을 만들고, 반례로 오해를 막고, 예시로 손에 잡히게 해야 독자가 실제로 이해합니다. 다음 글에서는 이렇게 잡은 개념을 코드 예제로 어떻게 설명해야 하는지 이어서 봅니다.

## 처음 질문으로 돌아가기

- **처음 보는 독자가 개념을 바로 이해하게 하려면 무엇부터 보여 줘야 할까요?**
  정의로 중심을 잡고, 비유로 문을 열고, 반례로 경계를 긋고, 예시로 손에 잡히게 만들어야 합니다.
- **비유와 반례는 왜 같이 움직여야 할까요?**
  비유가 감각을 만들면 반례가 오해를 막아 주기 때문입니다. 비유만 있으면 독자는 경계를 넘어서 잘못 이해할 수 있습니다.
- **한 줄 정의만으로는 왜 설명이 오래 남지 않을까요?**
  정의는 추상적이기 때문에 독자가 실제 상황에 적용하기 어렵습니다. 비유, 반례, 예시가 함께 있어야 개념이 오래 남습니다.
