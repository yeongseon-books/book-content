# Data Science란 무엇인가?

## 이 글에서 다룰 문제

- 데이터 사이언스를 한 문장으로 어떻게 이해하면 좋을까요?
- 데이터 분석가, 데이터 사이언티스트, ML 엔지니어, 데이터 엔지니어는 무엇이 다를까요?
- 데이터 작업은 왜 늘 문제 정의에서 시작해야 할까요?
- 모델이나 대시보드는 왜 의사결정으로 이어지지 않으면 가치가 약해질까요?
- 입문자가 가장 먼저 가져가야 할 전체 흐름은 무엇일까요?

> Data Science 101 시리즈 (1/10)

처음 데이터 분야를 접하면 용어부터 헷갈립니다. 데이터 분석도 있고, 데이터 사이언스도 있고, 머신러닝도 있고, 직무 이름도 제각각입니다. 이때 가장 먼저 필요한 것은 세부 도구가 아니라 큰 그림입니다. 어떤 문제가 데이터 문제로 바뀌고, 그 데이터를 어떻게 읽고, 마지막에 어떤 결정으로 닫히는지 이해하면 이후에 배우는 Pandas, SQL, 시각화, 모델링이 각각 어디에 놓이는지 보이기 시작합니다.

이 글에서는 데이터 사이언스를 “문제와 데이터를 연결하는 일”이라는 관점에서 설명하겠습니다. 이 관점을 잡아 두면 직무 경계도 훨씬 또렷해지고, 왜 EDA나 지표 설계가 모델링만큼 중요하다고 말하는지도 자연스럽게 이해할 수 있습니다.

## 왜 중요한가

직무 이름이 겹쳐 보이는 분야일수록 초반 학습 경로가 쉽게 흔들립니다. 큰 그림 없이 도구부터 배우면 조각은 쌓이는데 연결이 되지 않습니다. 반대로 데이터 작업이 결국 *문제 → 데이터 → 모델 또는 인사이트 → 결정*으로 이어진다는 흐름을 먼저 이해하면, 어떤 도구를 왜 배우는지 기준이 생깁니다.

저는 입문자에게 이 지점을 가장 먼저 강조합니다. 데이터 작업은 “데이터를 많이 다루는 일”이 아니라, “문제를 데이터로 답할 수 있는 형태로 바꾸는 일”에 더 가깝기 때문입니다.

> 큰 그림 없이 배우는 공부는 조각을 모으는 일에 머무르기 쉽습니다.

## 데이터 사이언스를 이해하는 가장 짧은 정의

데이터 사이언스는 **문제와 그 뒤에 있는 데이터를 연결해, 실제 의사결정에 쓸 수 있는 결과를 만드는 일**입니다. 여기서 중요한 것은 “데이터를 본다”가 아니라 “문제를 푼다”입니다. 데이터는 재료이고, 모델이나 대시보드는 도구이며, 최종 산출물은 대개 어떤 행동을 바꿀 수 있는 판단입니다.

예를 들어 “이탈 사용자를 줄이고 싶다”는 문제를 보겠습니다. 이 문장을 그대로 두면 너무 넓습니다. 하지만 “다음 30일 안에 이탈할 가능성이 높은 사용자를 매주 100명 추려서 캠페인 대상으로 삼는다”로 바꾸면 그때부터는 필요한 데이터, 확인할 지표, 만들 수 있는 모델, 검증할 실험이 한 줄로 이어집니다. 데이터 사이언스는 바로 이 연결 작업입니다.

## 전체 흐름 한눈에 보기

```mermaid
flowchart LR
    Problem["비즈니스 문제"] --> Data["데이터"]
    Data --> Model["모델 / 인사이트"]
    Model --> Decision["의사결정"]
```

이 그림이 단순해 보여도 중요한 이유가 있습니다. 많은 입문자가 데이터 수집이나 모델링을 독립된 기술처럼 배우기 때문입니다. 하지만 실제 업무에서는 늘 이 흐름의 일부로 움직입니다. 문제 없는 데이터 분석은 방향이 없고, 결정으로 이어지지 않는 모델은 오래 살아남기 어렵습니다.

## 핵심 용어

- **Data Analyst**: 지표, 리포트, 대시보드를 통해 비즈니스 질문에 답하는 역할입니다.
- **Data Scientist**: 모델을 만들고 실험을 설계해 예측이나 최적화를 시도하는 역할입니다.
- **ML Engineer**: 모델을 서비스 환경에 올리고 운영 가능하게 만드는 역할입니다.
- **Data Engineer**: 데이터 파이프라인과 웨어하우스를 설계하고 유지하는 역할입니다.
- **EDA**: Exploratory Data Analysis, 모델링 전에 데이터를 먼저 읽어 보는 탐색 단계입니다.

이 직무들은 서로 완전히 분리되지 않습니다. 작은 조직에서는 한 사람이 두세 역할을 겸하기도 합니다. 다만 역할의 중심축은 다릅니다. 분석가는 질문과 지표 쪽에, 데이터 사이언티스트는 가설과 모델 쪽에, 엔지니어는 안정적인 운영과 재현성 쪽에 더 무게가 실립니다.

## Before / After

**Before**: 데이터가 있으니 일단 뭔가 분석해 봅니다. 질문이 넓고 결과도 흐릿해서, 보고서를 만들고 나서도 “그래서 무엇을 해야 하죠?”라는 말이 남습니다.

**After**: 문제를 한 문장으로 좁히고, 그 문제에 필요한 데이터만 봅니다. 그다음 결과를 의사결정 문장으로 닫습니다. 같은 데이터라도 해석과 실행의 선명도가 완전히 달라집니다.

## 5단계 워크플로

### 1단계 — 문제 정의

```text
"Each week, pick 100 users most likely to churn"
```

좋은 문제 정의는 구체적입니다. 주기, 대상, 결과물이 들어가야 합니다. “이탈을 줄이자”보다 “매주 이탈 가능성이 높은 사용자 100명을 고른다”가 훨씬 낫습니다. 이렇게 써야 이후 단계가 흔들리지 않습니다.

### 2단계 — 데이터 수집

```python
import pandas as pd
df = pd.read_csv("users.csv")
print(df.shape, df.columns.tolist())
```

여기서 중요한 것은 단순히 파일을 읽는 기술이 아닙니다. 지금 보고 있는 데이터가 어느 시점의 어떤 원본에서 왔는지 추적 가능해야 합니다. 같은 분석을 두 달 뒤 다시 할 수 없다면 그 결과는 신뢰를 얻기 어렵습니다.

### 3단계 — 정제와 EDA

```python
df = df.dropna(subset=["last_login"])
df["days_since_login"] = (
    pd.Timestamp.today() - pd.to_datetime(df["last_login"])
).dt.days
print(df["days_since_login"].describe())
```

수집한 데이터는 거의 항상 바로 쓰기 어렵습니다. 결측치가 있고, 타입이 틀리고, 분포를 먼저 읽어 봐야 합니다. EDA는 모델링 전의 준비 운동이 아니라, 데이터가 스스로 자신을 소개하는 시간에 가깝습니다.

### 4단계 — 모델 또는 규칙 만들기

```python
candidates = (
    df[df["days_since_login"] > 30]
      .sort_values("amount_total", ascending=False)
      .head(100)
)
```

모든 데이터 문제에 복잡한 모델이 필요한 것은 아닙니다. 때로는 간단한 규칙 기반 우선순위만으로도 충분한 가치를 만들 수 있습니다. 핵심은 정교함보다 문제 적합성입니다.

### 5단계 — 의사결정으로 연결

```text
Email campaign for 100 users → measure conversion → adjust next week
```

마지막 단계가 빠지면 데이터 작업은 자주 보고서에서 멈춥니다. 결과는 반드시 “무엇을 할 것인가”로 닫혀야 합니다. 그래야 다음 주에 다시 측정하고, 개선하고, 같은 루프를 돌릴 수 있습니다.

## 이 코드에서 주목할 점

- 문제를 한 문장으로 쓰는 순간 이후의 모든 코드가 방향을 갖습니다.
- 데이터 흐름은 대개 읽기 → 정제 → 탐색 → 결정으로 이어집니다.
- 모델이나 대시보드는 결정으로 연결될 때 비로소 실무 가치가 생깁니다.

## 자주 하는 실수 5가지

1. **도구부터 배웁니다.** 문제 없이 도구만 익히면 언제 무엇을 써야 할지 기준이 생기지 않습니다.
2. **모델 정확도만 봅니다.** 비즈니스 지표가 함께 좋아졌는지 보지 않으면 엉뚱한 최적화가 됩니다.
3. **EDA를 건너뜁니다.** 데이터가 실제로 어떤 모습인지 확인하지 않으면 가설을 너무 빨리 굳히게 됩니다.
4. **역할 경계를 무시합니다.** 분석가, 사이언티스트, 엔지니어의 중심 역할이 다르다는 점을 놓치면 협업과 채용이 꼬이기 쉽습니다.
5. **결과를 결정과 연결하지 않습니다.** 보고서는 남는데 행동은 바뀌지 않는 전형적인 상황이 됩니다.

## 실무에서는 이렇게 드러납니다

초기 스타트업 데이터 팀은 종종 작은 조직 하나처럼 움직입니다. 한쪽에서는 대시보드와 핵심 지표를 보고, 다른 쪽에서는 모델을 만들고, 또 다른 쪽에서는 데이터 파이프라인과 운영을 맡습니다. 역할은 겹칠 수 있지만, 모든 작업은 결국 한 가지 질문으로 모입니다. “이 결과가 어떤 결정을 바꾸는가?” 이 질문이 없는 데이터 작업은 오래 유지되기 어렵습니다.

## 시니어는 이렇게 생각합니다

- 문제를 잘 정의하는 일이 모델링보다 더 비쌀 때가 많습니다.
- EDA를 대충 하면 뒤의 모든 단계가 흔들립니다.
- 결과는 반드시 결정 문장으로 닫혀야 합니다.
- 역할 경계는 문서로 남겨야 협업이 단순해집니다.
- 데이터 품질은 특정 직군만의 일이 아니라 팀 전체의 책임입니다.

## 정리 및 다음 글

데이터 사이언스는 데이터 자체를 다루는 기술이 아니라, 문제와 데이터를 연결해 의사결정으로 닫는 일입니다. 이 관점을 잡고 나면 수집, 정제, EDA, 시각화, 모델링이 각각 어디에 놓이는지 훨씬 또렷해집니다. 다음 글에서는 막연한 문제를 실제로 데이터가 답할 수 있는 질문으로 어떻게 바꾸는지 살펴보겠습니다.
