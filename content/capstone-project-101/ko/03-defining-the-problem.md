# Capstone Project 101 (3/10): 문제 정의

이 글은 캡스톤 프로젝트 101 시리즈의 3번째 글입니다.

문제 정의가 흐리면 팀은 계속 해결책만 바꿉니다. 기능은 늘어나는데 왜 만드는지 설명은 점점 약해지는 경우가 많습니다.

문제 문장은 프로젝트의 기준점입니다. 요구사항을 넣을지 뺄지, MVP를 어디까지 자를지, 발표에서 무엇을 강조할지 모두 이 문장으로 다시 돌아와야 합니다.

이 글은 Capstone Project 101 시리즈의 3번째 글입니다. 여기서는 기능 설명과 문제 정의를 구분하고, 사용자·가정·지표가 함께 들어 있는 문제 카드를 만드는 방법을 다룹니다.

> 멘탈 모델: 문제 문장은 기능 목록의 앞줄이 아니라, 이후 모든 요구사항과 우선순위를 묶어 주는 기준 문장입니다.

> 문제 정의는 팀이 같은 상황을 같은 말로 설명할 수 있도록 하는 기준입니다. 그 기준이 안정적이면 요구사항도, MVP도, 발표도 흔들리지 않습니다.

## 먼저 던지는 질문

- 왜 문제 문장이 흐리면 해결책도 같이 흔들릴까요?
- 기능 설명과 문제 정의는 어떻게 다를까요?
- 사용자, 가치, 가정, 측정 기준을 한 장에 담으려면 무엇을 적어야 할까요?

## 이 글에서 배우는 내용

- 문제 진술 작성법
- 사용자 가설 정리
- 가치 명세
- 기준선 지표 설정
- 문제 재정의 감각

## 왜 중요한가

문제 정의는 프로젝트 품질의 절반을 좌우합니다. 문제가 선명하면 어떤 요구사항을 넣어야 하는지, 어떤 기능을 빼야 하는지, 데모에서 무엇을 보여 줘야 하는지가 자연스럽게 정리됩니다. 반대로 문제 문장이 흐리면 새 기능 제안이 나올 때마다 다 중요해 보이고, 진행 상황도 계속 다시 정의됩니다.

좋은 팀은 문제 정의를 한 번에 완성하려 하지 않습니다. 다만 관찰, 사용자, 가치, 가정, 지표를 분리해서 적어 두기 때문에 중간 조정이 와도 기준이 무너지지 않습니다.

## 한눈에 보는 흐름

## 실전 문서 예시: 문제 카드

짧은 문제 카드 하나만 잘 써도 해결책 논의의 질이 크게 달라집니다. 아래 항목은 최소 구성입니다.

```text
관찰: 수강 신청 직전에 시간표 충돌을 수동으로 계산하느라 시간이 오래 걸린다
핵심 사용자: 복수전공 학생, 신입생
문제 문장: 수강 신청 전 30초 안에 충돌 여부를 확인하기 어렵다
핵심 가정: 시간표 데이터를 텍스트나 CSV로 쉽게 넣을 수 있다
성공 지표: 첫 사용자가 30초 안에 충돌 여부를 확인한다
```

## 이 문서로 먼저 확인할 것

- 문제 문장이 해결책 이름으로 시작하지 않는지 확인합니다.
- 핵심 사용자를 좁혀 쓰고 있는지 점검합니다.
- 숨겨진 가정을 별도 항목으로 끌어올립니다.
- 지표가 시간, 정확도, 완료율처럼 측정 가능한 값인지 봅니다.

## 핵심 용어

- **statement**: 문제를 설명하는 문장입니다.
- **persona**: 핵심 사용자 집단입니다.
- **value**: 문제를 풀었을 때 생기는 변화입니다.
- **assumption**: 아직 검증되지 않은 전제입니다.
- **metric**: 성공 여부를 판단하는 측정 기준입니다.

## 이전과 이후

**Before**: 기능이 곧 문제라고 생각합니다.

**After**: 문제가 기능의 근거라고 생각합니다.

## 실습: 문제 카드

### 1단계 — 관찰

```python
obs = "schedule conflicts during course registration"
```

관찰은 추상적 주장보다 실제 상황에 가까워야 합니다. 출발점이 구체적일수록 뒤 문장도 구체적으로 이어집니다.

### 2단계 — 사용자

```python
user = "freshmen plus double-major students"
```

사용자를 좁히는 이유는 배제가 아니라 집중입니다. 모두를 사용자로 잡는 순간 맥락이 사라집니다.

### 3단계 — 가치

```python
value = "spot conflicts fast"
```

가치는 기능 설명과 다릅니다. 달력 화면이 아니라 충돌을 빨리 찾게 해 주는 변화가 가치입니다.

### 4단계 — 가정

```python
assume = "users can paste timetables as text"
```

가정을 숨기면 구현 단계에서 갑자기 복잡도가 커집니다. 입력 방식, 데이터 출처, 사용자 행동처럼 숨어 있는 전제를 문서에 올려야 합니다.

### 5단계 — 지표

```python
metric = "conflict found within 30s"
```

지표는 잘 만들었는지 판단하는 최소 기준입니다. 숫자가 들어가야 데모와 테스트가 쉬워집니다.

## 이 코드에서 먼저 볼 점

- 관찰이 문제 진술보다 먼저 나옵니다.
- 사용자 범위는 구체적일수록 좋습니다.
- 가정은 숨기지 않고 드러내야 합니다.
- 지표가 들어가야 해결 여부를 설명할 수 있습니다.

## 자주 하는 실수 5가지

1. 해결책을 문제처럼 적습니다.
2. 사용자를 모두라고 적습니다.
3. 중요한 가정을 문서 밖에 둡니다.
4. 지표를 모호한 표현으로 남깁니다.
5. 문제 재정의를 실패라고 생각합니다.

## 실무에서는 이렇게 이어집니다

PRD의 첫머리나 프로젝트 제안서의 초반에는 거의 항상 문제 진술이 들어갑니다. 구현 전에 이 문장을 맞춰 두어야 뒤에서 요구사항과 설계가 흔들리지 않습니다. 캡스톤에서 문제 정의를 제대로 해 본 경험은 제품 문서를 읽고 쓰는 기초 체력으로 이어집니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 문제 문장은 짧게 씁니다.
- 가정을 숨기지 않습니다.
- 지표는 숫자로 적습니다.
- 문제 재정의는 건강한 조정이라고 봅니다.
- 사용자는 가능한 한 구체적으로 잡습니다.

## 정리와 다음 글

문제 정의는 기능 목록의 서문이 아니라 프로젝트의 기준점입니다. 사용자, 가정, 지표를 함께 적어 두면 요구사항과 MVP를 자를 때도 훨씬 단단한 판단이 가능합니다. 다음 글에서는 이 기준을 실제 요구사항 문서로 바꾸겠습니다.

## 실전 확장: MVP 범위 설정 체크리스트와 기능 우선순위 매트릭스

문제 정의가 끝났다고 해서 범위가 자동으로 정리되지는 않습니다. 오히려 이 시점부터 팀은 "이것도 필요하다"는 제안을 연속으로 받습니다. 따라서 문제 정의 문장을 기준으로 MVP 범위를 자르는 운영 도구가 필요합니다. 아래 체크리스트와 매트릭스는 구현 시작 전에 반드시 한 번 통과시키는 장치입니다.

### MVP 범위 설정 체크리스트

| 점검 항목 | 예/아니오 | 근거 문장 |
| --- | --- | --- |
| 핵심 사용자가 한 문장으로 정의되어 있는가 |  |  |
| 핵심 흐름이 3~5단계 이내로 설명되는가 |  |  |
| 데모에서 재현 가능한 입력 데이터가 준비되는가 |  |  |
| 성공 지표가 시간/완료율 등 수치로 표현되는가 |  |  |
| 외부 의존성 실패 시 대체 경로가 있는가 |  |  |
| 이번 학기 범위 밖 항목이 문서화되어 있는가 |  |  |
| Must 기능이 전체 후보의 절반 이하인가 |  |  |
| 발표 리허설 기준으로 5분 내 시연 가능한가 |  |  |

체크리스트는 점수표가 아니라 경보 장치입니다. "아니오"가 3개 이상이면 구현을 멈추고 범위를 다시 잘라야 합니다. 캡스톤에서는 빠른 시작보다 빠른 범위 정리가 훨씬 높은 성공 확률을 만듭니다.

### 기능 우선순위 매트릭스

| 기능 | 사용자 가치 | 구현 난도 | 실패 리스크 | 데모 기여도 | 분류 |
| --- | --- | --- | --- | --- | --- |
| 시간표 입력 | 높음 | 중간 | 중간 | 높음 | Must |
| 충돌 계산 | 매우 높음 | 중간 | 높음 | 매우 높음 | Must |
| 결과 시각화 | 높음 | 중간 | 낮음 | 높음 | Must |
| 소셜 공유 | 중간 | 낮음 | 낮음 | 중간 | Should |
| 알림 기능 | 중간 | 중간 | 중간 | 낮음 | Could |
| 추천 알고리즘 고도화 | 낮음 | 높음 | 높음 | 낮음 | Won't |

분류 규칙은 간단해야 합니다. `Must`는 핵심 흐름을 성립시키는 기능, `Should`는 품질을 높이는 기능, `Could`는 여유가 있을 때 추가 가능한 기능, `Won't`는 이번 학기 제외 기능입니다. 이 규칙을 문서 상단에 고정해 두면 팀 내 해석 차이를 줄일 수 있습니다.

### 범위 컷팅 결정 로그 예시

```text
2026-05-20
- 결정: 추천 알고리즘 고도화는 Won't로 이동
- 이유: 학습 비용 대비 데모 기여도가 낮음
- 영향: 데이터 파이프라인 단순화, 테스트 범위 축소
- 대안: 발표 Q&A에서 확장 계획으로만 설명
```

결정 로그를 남기면 나중에 "왜 이 기능을 뺐는가"라는 질문에 즉시 답할 수 있습니다. 특히 발표 자료와 회고 문서에서 매우 강한 근거로 작동합니다.

### 적용 순서 제안

1. 문제 정의 문장을 문서 상단에 고정합니다.
2. 후보 기능을 전부 나열합니다.
3. 매트릭스로 1차 분류를 수행합니다.
4. 체크리스트를 통과하지 못한 항목을 제거합니다.
5. 결정 로그를 남기고 구현 이슈로 전환합니다.

핵심은 "기능을 많이 담는 것"이 아니라 "핵심 흐름을 완결하는 것"입니다. 문제 정의 단계에서 이 원칙이 지켜지면 이후 일정 관리와 발표 준비가 크게 안정됩니다.

## 심화 보강: 의사결정 근거 문서화 패턴

캡스톤 문서의 분량을 늘리는 목적은 글자 수 자체가 아니라 판단의 흔적을 남기는 데 있습니다. 같은 결과물을 만들더라도 판단 근거가 기록된 팀과 기록되지 않은 팀의 학습 밀도는 크게 다릅니다. 아래 패턴은 어떤 장에서든 공통으로 적용할 수 있는 최소 문서화 규칙입니다.

### 의사결정 카드 템플릿

| 항목 | 작성 기준 | 예시 |
| --- | --- | --- |
| 결정 제목 | 동사형으로 짧게 작성 | 입력 형식을 CSV로 고정한다 |
| 배경 | 현재 문제와 제약을 3문장 내 서술 | 발표 전 2주 내 구현 안정화 필요 |
| 선택지 | 최소 2개 이상 비교 | CSV, 엑셀 직접 업로드 |
| 평가 기준 | 시간, 위험, 구현 난도 명시 | 학습 비용, 오류 가능성 |
| 최종 결정 | 한 문장으로 고정 | CSV 입력을 MVP 범위로 채택 |
| 후속 작업 | 실행 가능한 액션으로 변환 | 파서 검증 테스트 5건 추가 |

### 리스크 등록부 예시

| 리스크 ID | 설명 | 발생 가능성 | 영향도 | 대응 전략 | 담당 |
| --- | --- | --- | --- | --- | --- |
| RK-01 | 외부 데이터 접근 지연 | 중간 | 높음 | 샘플 데이터 대체 경로 준비 | 데이터 담당 |
| RK-02 | 데모 환경 네트워크 불안정 | 중간 | 높음 | 오프라인 영상 백업 준비 | 발표 담당 |
| RK-03 | 통합 테스트 누락 | 높음 | 중간 | 주간 회귀 테스트 슬롯 고정 | QA 담당 |
| RK-04 | 요구사항 빈번한 변경 | 중간 | 중간 | 변경 승인 규칙 명시 | 팀 리드 |

리스크 등록부는 불안을 기록하는 문서가 아니라 우선순위를 결정하는 문서입니다. 발생 가능성과 영향도를 함께 보면 어떤 리스크를 먼저 줄여야 하는지 명확해집니다.

### 주간 리뷰 질문

- 이번 주 결정 중 다음 주에 다시 검토할 항목은 무엇인가
- 일정 편차를 만든 원인은 기술 이슈인가, 의사결정 지연인가
- 범위 확장 요청이 들어왔을 때 거절 기준이 문서에 있는가
- 테스트 실패가 특정 구간에 집중되는가
- 발표 시점에 그대로 설명 가능한 근거가 충분한가

### 품질을 높이는 기록 원칙

첫째, 문장은 짧고 검증 가능하게 씁니다. 둘째, 수치가 가능한 항목은 반드시 수치로 남깁니다. 셋째, 결정과 액션을 분리하지 않습니다. 넷째, 회의 직후 24시간 내 문서를 갱신합니다. 다섯째, 변경 이유를 생략하지 않습니다. 이 다섯 가지 원칙만 지켜도 캡스톤 문서는 단순 제출물이 아니라 다음 프로젝트를 위한 운영 매뉴얼로 전환됩니다.

### 적용 체크리스트

- [ ] 결정 카드가 최신 상태로 유지됩니다.
- [ ] 리스크 등록부에 담당자와 대응 전략이 있습니다.
- [ ] 주간 리뷰 질문에 대한 답변이 기록됩니다.
- [ ] 변경 로그가 날짜 순으로 정리됩니다.
- [ ] 발표와 회고 문서가 같은 근거를 참조합니다.

위 보강 블록은 각 장의 주제와 별개로 공통 적용되는 운영 원칙입니다. 캡스톤의 난도는 기술 난도보다 의사결정 난도에서 크게 올라갑니다. 따라서 문서화 패턴을 반복 적용하면 팀의 판단 체계가 안정되고, 일정과 품질, 발표 전달력이 동시에 개선됩니다.

### 추가 보강: 발표 직전 최종 점검 루틴

발표 직전에는 새 기능 추가를 중단하고 안정화 루틴을 고정합니다. 최종 점검 루틴은 기술 완성도를 높이는 절차가 아니라 실패 확률을 낮추는 절차입니다. 핵심 흐름 시연, 오류 메시지 점검, 백업 자료 확인, 예상 질문 재점검, 역할별 멘트 정렬을 반복하면 전달 품질이 크게 향상됩니다.

| 점검 항목 | 확인 내용 | 통과 기준 |
| --- | --- | --- |
| 핵심 시연 | 시작부터 결과 화면까지 흐름 유지 | 3회 연속 성공 |
| 오류 처리 | 비정상 입력 시 안내 문구 노출 | 사용자 행동 유도 가능 |
| 백업 자료 | 네트워크 실패 대비 영상/스크린샷 | 즉시 전환 가능 |
| Q&A 근거 | 기술/범위/테스트 답변 문서화 | 질문 10개 대응 가능 |
| 역할 분담 | 발표자, 데모 조작자, 시간 관리자 확정 | 리허설 동일 운영 |

이 루틴을 문서로 남기면 회고 단계에서 무엇이 실제로 효과가 있었는지 비교할 수 있습니다.

### 보강 메모: 다음 프로젝트로 이관할 핵심 문장

이번 장의 핵심 판단 기준을 3문장으로 요약해 저장소 루트의 프로젝트 로그에 남깁니다. 다음 프로젝트 킥오프에서 이 문장을 먼저 읽으면 재시작 비용을 크게 줄일 수 있습니다. 문장 요약은 길이보다 재사용 가능성이 중요합니다.

## 실전 앵커: 코드 리뷰 체크리스트와 CI 파이프라인 최소 구성

문제 정의가 문서로 끝나면 실행력이 떨어집니다. 문제 문장을 개발 흐름으로 연결하려면 코드 리뷰 기준과 자동 검증 파이프라인을 함께 설계해야 합니다. 이 단계에서 품질 규칙을 고정해 두면 이후 범위 조정에도 일관된 판단을 유지할 수 있습니다.

### 코드 리뷰 체크리스트(문제 정의 연계형)

- 변경 내용이 문제 문장과 직접 연결되는지 확인합니다.
- 새 기능이 `MVP 범위` 안에 있는지 확인합니다.
- 입력/출력 예시가 데모 시나리오와 일치하는지 점검합니다.
- 실패 케이스 테스트가 최소 1건 이상 포함되어 있는지 확인합니다.
- 문서(README/결정 로그)가 함께 갱신되었는지 확인합니다.

### 깃허브 액션 CI 예시

```yaml
name: capstone-ci
on:
  pull_request:
    branches: [main]
jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.11'
      - run: pip install -r requirements.txt
      - run: pytest -q
      - run: python scripts/check_problem_scope.py
```

이 파이프라인의 핵심은 복잡한 배포 자동화가 아닙니다. 문제 정의와 범위 검증을 PR 단계에서 자동으로 통과시키는 데 있습니다. 팀원이 많아질수록 사람의 기억보다 자동화된 규칙이 프로젝트 품질을 안정적으로 유지합니다.

## 처음 질문으로 돌아가기

- **왜 문제 문장이 흐리면 해결책도 같이 흔들릴까요?**
  - 이 글은 문제 정의을 단순 정의가 아니라, 실무에서 마주하는 구체적 상황과 판단 과정으로 풀어냈습니다. 각 섹션의 예시와 체크리스트를 따라가며 자신의 상황에 적용할 수 있도록 구성했습니다.
- **기능 설명과 문제 정의는 어떻게 다를까요?**
  - 본문에서 제시한 예시 코드나 매트릭스를 참고하면, 좋은 판단 기준이 무엇인지 구체적으로 느낄 수 있습니다. 숫자나 기준이 아니라 '어떤 상황에서 이런 기준을 세우는가'를 이해하는 것이 중요합니다.
- **사용자, 가치, 가정, 측정 기준을 한 장에 담으려면 무엇을 적어야 할까요?**
  - 이 글의 핵심 메시지는 평가 기준이 무엇이든, 한 번의 판단으로 끝나지 않는다는 점입니다. 초반에 정한 기준을 문서로 남기고, 나중에 돌아봤을 때도 같은 실수를 반복하지 않도록 하는 과정 자체가 프로젝트를 성공으로 이끕니다.
