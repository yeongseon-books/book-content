# Capstone Project 101 (6/10): MVP 설계

많은 팀이 MVP를 '기능을 조금만 만드는 것'으로 이해합니다. 그래서 모든 기능을 반쯤 만들고도 핵심 흐름은 끝내지 못하는 일이 자주 생깁니다.

MVP는 작은 완성품이 아니라 가장 중요한 가설을 가장 빨리 검증하는 설계입니다. 그래서 무엇을 넣을지보다 무엇을 의도적으로 빼는지가 더 중요할 때가 많습니다.

이 글은 Capstone Project 101 시리즈의 6번째 글입니다. 여기서는 핵심 사용자 흐름, 범위 제외 목록, 데모 기준, 피드백 수집을 한 세트로 묶는 MVP 설계 방식을 다룹니다.

> 멘탈 모델: MVP의 목적은 기능 수를 줄이는 것이 아니라, 핵심 사용자 흐름 하나를 끝까지 연결해 팀이 실제로 배울 수 있는 상태를 만드는 데 있습니다.

## 먼저 던지는 질문

- MVP는 왜 작은 완성품보다 학습 도구에 가깝다고 할까요?
- 핵심 흐름 하나를 고른다는 말은 무엇을 남기고 무엇을 버린다는 뜻일까요?
- 범위 밖 항목을 문서로 적는 일은 왜 중요할까요?

## 이 글에서 배우는 내용

- MVP의 기본 정의
- 핵심 흐름 선정법
- 범위 밖 항목 정리법
- 데모 시나리오 작성
- 피드백 수집 방식

## 왜 중요한가

많은 학생 프로젝트가 기능 개수로 진도를 계산합니다. 하지만 캡스톤에서는 기능이 많아도 핵심 흐름이 살아 있지 않으면 설득력이 약합니다. 사용자가 처음 들어와 가장 중요한 행동 하나를 끝낼 수 있어야 하고, 그 과정에서 무엇을 배웠는지 설명할 수 있어야 합니다.

현업에서도 새 제품이나 새 기능은 처음부터 모든 예외를 처리하지 않습니다. 가장 중요한 흐름을 먼저 붙이고, 그 흐름이 작동하는지 확인한 뒤 확장합니다. MVP는 대충 만든 시제품이 아니라, 우선순위를 극단적으로 정리한 결과물입니다.

## 실전 문서 예시: MVP 계약서

MVP를 지킬 때 가장 도움이 되는 문서는 짧은 계약서입니다. 팀이 이번 주에 지킬 경계를 아래처럼 문장으로 적습니다.

```text
핵심 흐름: 로그인 → 시간표 입력 → 충돌 계산 → 결과 화면
범위 밖: 결제, 관리자 대시보드, 다국어, 추천 알고리즘 고도화
데모 기준: 샘플 데이터로 60초 안에 흐름 전체를 보여 준다
성공 기준: 첫 사용자 3명 중 2명 이상이 설명 없이 결과 화면에 도달한다
피드백 질문: 이해하기 쉬웠는가, 속도가 충분했는가, 실제로 쓸 의향이 있는가
```

## 이 문서로 먼저 확인할 것

- 핵심 흐름이 한 문장으로 읽히는지 확인합니다.
- 범위 밖 항목이 충분히 공격적으로 잘려 있는지 봅니다.
- 데모 기준이 실제 발표 환경에서 재현 가능한지 점검합니다.
- 피드백 질문이 다음 결정으로 이어질 수 있는 수준인지 살핍니다.

## 핵심 용어

- **MVP**: 가장 작은 검증 가능한 제품입니다.
- **happy path**: 사용자가 가장 정상적으로 밟는 핵심 흐름입니다.
- **out of scope**: 지금은 의도적으로 제외한 범위입니다.
- **demo**: 핵심 흐름을 따라가는 시연입니다.
- **feedback**: 다음 판단을 위한 구조화된 반응입니다.

## 이전과 이후

**Before**: 모든 기능을 조금씩 만듭니다.

**After**: 핵심 흐름 하나를 끝까지 완성합니다.

## 실습: MVP 표

### 1단계 — 핵심 흐름 선정

```python
flow = "register -> upload -> share"
```

핵심 흐름은 한 문장으로 적을 수 있어야 합니다. 한 줄로 안 적히면 아직 너무 큽니다.

### 2단계 — 제외 목록

```python
out = ["payment", "i18n", "admin"]
```

범위 밖 항목은 소극적인 문서가 아니라, MVP를 지키는 가장 강한 장치입니다. 빼 두어야 핵심이 살아납니다.

### 3단계 — 데모 시나리오

```python
demo = ["login_demo_user", "upload_sample", "show_share_link"]
```

데모 시나리오는 구현 우선순위를 정해 줍니다. 발표 당일 어떤 순서로 보여 줄지를 적어 보면 핵심과 부가 기능이 자연스럽게 갈립니다.

### 4단계 — 성공 기준

```python
success = {"happy_path": "<= 60s", "errors": 0}
```

성공 기준은 감상을 숫자로 바꾸는 작업입니다. 무엇을 성공으로 볼지 정해 두어야 데모 이후 평가도 가능합니다.

### 5단계 — 피드백 폼

```python
form = ["clarity", "speed", "value"]
```

피드백 항목이 없으면 발표는 박수로 끝나고 학습은 남지 않습니다. MVP는 결과물인 동시에 실험이기도 합니다.

## 이 코드에서 먼저 볼 점

- 핵심 흐름은 한 문장입니다.
- 범위 밖 항목이 명시적입니다.
- 성공 기준에는 숫자가 들어갑니다.
- 데모와 피드백이 함께 있어야 학습이 남습니다.

## 자주 하는 실수 5가지

1. 진행률을 기능 개수로 계산합니다.
2. 처음부터 모든 예외를 처리하려고 합니다.
3. 데모 시나리오 없이 구현만 먼저 합니다.
4. 피드백 항목을 만들지 않습니다.
5. 외부 의존성을 너무 많이 붙여 위험을 키웁니다.

## 실무에서는 이렇게 이어집니다

스타트업도 처음에는 한 줄짜리 happy path를 먼저 만듭니다. 회원가입 후 업로드가 되는지, 신청 후 승인 메일이 가는지처럼 가장 중요한 흐름부터 확인합니다. 캡스톤에서 MVP를 잘 설계해 본 경험은 이후 우선순위 조정과 스코프 컷팅 감각으로 이어집니다.

## 시니어 엔지니어는 이렇게 생각합니다

- MVP는 학습 도구입니다.
- 흐름은 하나로 유지합니다.
- 범위를 자르는 결정을 두려워하지 않습니다.
- 데모는 미리 각본을 씁니다.
- 피드백은 구조화해 받습니다.

## 정리와 다음 글

MVP는 작은 제품이 아니라 작은 실험입니다. 핵심 흐름, 범위 제외 목록, 데모 기준, 피드백 질문을 함께 적어 두면 학기 중간의 흔들림 속에서도 무엇을 지켜야 하는지 선명해집니다. 다음 글에서는 이 MVP를 어떤 기술 스택으로 구현할지 다룹니다.

## 실전 확장: 기술 설계 문서 구조와 ADR 작성 예시

MVP 범위를 정한 뒤에는 구현 전에 기술 설계 문서를 최소 형태로 남겨야 합니다. 설계 문서가 없는 팀은 같은 논쟁을 반복하고, 구현 중에 기준이 바뀌면서 일정이 크게 흔들립니다. 반대로 짧은 설계 문서와 ADR을 남긴 팀은 결정의 맥락을 보존하기 때문에 변경이 와도 조정 속도가 빠릅니다.

### 기술 설계 문서 기본 구조

| 섹션 | 포함해야 할 내용 | 길이 가이드 |
| --- | --- | --- |
| 배경과 목표 | 어떤 문제를 해결하려는지, MVP 범위는 무엇인지 | 5~8문장 |
| 핵심 시나리오 | 사용자가 밟는 핵심 흐름과 실패 흐름 | 다이어그램 1개 + 설명 |
| 시스템 구성 | 프런트/백엔드/데이터/배포 구성 | 컴포넌트 표 1개 |
| 데이터 모델 | 입력/출력 스키마, 필수 필드 | 표 1~2개 |
| API 계약 | 엔드포인트, 요청/응답, 오류 코드 | 표 또는 OpenAPI 초안 |
| 품질 기준 | 성능, 오류 처리, 테스트 기준 | 체크리스트 |
| 위험과 대안 | 예상 리스크, 대응 전략, 대안 비교 | 5~10문장 |

### 컴포넌트 책임 경계 예시

| 컴포넌트 | 책임 | 비책임 |
| --- | --- | --- |
| Web UI | 입력 수집, 결과 표시 | 데이터 영속화 로직 |
| API 서버 | 검증, 충돌 계산, 응답 생성 | 화면 상태 관리 |
| 데이터 레이어 | 시간표 데이터 저장/조회 | 비즈니스 규칙 결정 |
| 테스트 모듈 | 핵심 시나리오 자동 검증 | 운영 모니터링 |

경계가 분명하면 팀원 간 "이건 누구 일인가"라는 질문이 줄어듭니다. MVP 단계일수록 책임 경계가 흐려지기 쉽기 때문에 문서화가 더 중요합니다.

### ADR 예시

```text
ADR-003: 충돌 계산 엔진을 서버 측에서 수행한다

상태: Accepted
날짜: 2026-05-21

문맥:
- 프런트엔드와 백엔드 모두 계산 로직을 넣으면 불일치 위험이 커진다.
- 발표 데모에서 계산 결과의 일관성이 핵심이다.

결정:
- 충돌 계산은 서버 측 단일 모듈에서 수행한다.
- 프런트엔드는 입력 검증과 결과 표시만 담당한다.

대안:
1) 클라이언트 계산: 구현은 빠르나 일관성 관리 부담 증가
2) 서버 계산: 네트워크 왕복이 필요하나 규칙 일원화 가능

결과:
- 테스트 기준을 서버 테스트로 집중할 수 있다.
- API 지연에 대한 모니터링 기준이 추가로 필요하다.
```

### 설계 리뷰 질문

- 이 설계가 MVP 핵심 흐름을 직접 지원하는가
- 가장 복잡한 부분을 단순화할 대안이 있는가
- 테스트 가능한 단위로 경계가 분리되어 있는가
- 실패 시 사용자에게 보이는 증상이 정의되어 있는가
- 발표 데모에서 증명할 수 있는 설계 결정인가

### 문서 운영 원칙

설계 문서는 완벽해야 하는 문서가 아니라 살아 있는 기준 문서입니다. 변경이 생길 때마다 전체를 다시 쓰기보다, ADR로 결정을 누적하는 편이 효율적입니다. MVP 단계에서는 특히 "왜 이 결정을 했는가"를 남기는 것이 중요합니다. 그 기록이 일정 관리, 발표 Q&A, 최종 회고의 공통 근거가 되기 때문입니다.

## 심화 보강: 의사결정 근거 문서화 패턴

캡스톤 문서의 분량을 늘리는 목적은 글자 수 자체가 아니라 판단의 흔적을 남기는 데 있습니다. 같은 결과물을 만들더라도 판단 근거가 기록된 팀과 기록되지 않은 팀의 학습 밀도는 크게 다릅니다. 아래 패턴은 어떤 장에서든 공통으로 적용할 수 있는 최소 문서화 규칙입니다.

### 의사결정 카드 템플릿

| 항목 | 작성 기준 | 예시 |
| --- | --- | --- |
| 결정 제목 | 동사형으로 짧게 작성 | 입력 형식을 CSV로 고정한다 |
| 배경 | 현재 문제와 제약을 3문장 내 서술 | 발표 전 2주 내 구현 안정화 필요 |
| 선택지 | 최소 2개 이상 비교 | CSV, 엑셀 직접 업로드 |
| 평가 기준 | 시간, 위험, 구현 난도 명시 | 학습 비용, 오류 가능성 |
| 최종 결정 | 한 문장으로 고정 | CSV 입력을 MVP 범위로 채택 |
| 후속 작업 | 실행 가능한 액션으로 변환 | 파서 검증 테스트 5건 추가 |

### 리스크 등록부 예시

| 리스크 ID | 설명 | 발생 가능성 | 영향도 | 대응 전략 | 담당 |
| --- | --- | --- | --- | --- | --- |
| RK-01 | 외부 데이터 접근 지연 | 중간 | 높음 | 샘플 데이터 대체 경로 준비 | 데이터 담당 |
| RK-02 | 데모 환경 네트워크 불안정 | 중간 | 높음 | 오프라인 영상 백업 준비 | 발표 담당 |
| RK-03 | 통합 테스트 누락 | 높음 | 중간 | 주간 회귀 테스트 슬롯 고정 | QA 담당 |
| RK-04 | 요구사항 빈번한 변경 | 중간 | 중간 | 변경 승인 규칙 명시 | 팀 리드 |

리스크 등록부는 불안을 기록하는 문서가 아니라 우선순위를 결정하는 문서입니다. 발생 가능성과 영향도를 함께 보면 어떤 리스크를 먼저 줄여야 하는지 명확해집니다.

### 주간 리뷰 질문

- 이번 주 결정 중 다음 주에 다시 검토할 항목은 무엇인가
- 일정 편차를 만든 원인은 기술 이슈인가, 의사결정 지연인가
- 범위 확장 요청이 들어왔을 때 거절 기준이 문서에 있는가
- 테스트 실패가 특정 구간에 집중되는가
- 발표 시점에 그대로 설명 가능한 근거가 충분한가

### 품질을 높이는 기록 원칙

첫째, 문장은 짧고 검증 가능하게 씁니다. 둘째, 수치가 가능한 항목은 반드시 수치로 남깁니다. 셋째, 결정과 액션을 분리하지 않습니다. 넷째, 회의 직후 24시간 내 문서를 갱신합니다. 다섯째, 변경 이유를 생략하지 않습니다. 이 다섯 가지 원칙만 지켜도 캡스톤 문서는 단순 제출물이 아니라 다음 프로젝트를 위한 운영 매뉴얼로 전환됩니다.

### 적용 체크리스트

- [ ] 결정 카드가 최신 상태로 유지됩니다.
- [ ] 리스크 등록부에 담당자와 대응 전략이 있습니다.
- [ ] 주간 리뷰 질문에 대한 답변이 기록됩니다.
- [ ] 변경 로그가 날짜 순으로 정리됩니다.
- [ ] 발표와 회고 문서가 같은 근거를 참조합니다.

위 보강 블록은 각 장의 주제와 별개로 공통 적용되는 운영 원칙입니다. 캡스톤의 난도는 기술 난도보다 의사결정 난도에서 크게 올라갑니다. 따라서 문서화 패턴을 반복 적용하면 팀의 판단 체계가 안정되고, 일정과 품질, 발표 전달력이 동시에 개선됩니다.

### 추가 보강: 발표 직전 최종 점검 루틴

발표 직전에는 새 기능 추가를 중단하고 안정화 루틴을 고정합니다. 최종 점검 루틴은 기술 완성도를 높이는 절차가 아니라 실패 확률을 낮추는 절차입니다. 핵심 흐름 시연, 오류 메시지 점검, 백업 자료 확인, 예상 질문 재점검, 역할별 멘트 정렬을 반복하면 전달 품질이 크게 향상됩니다.

| 점검 항목 | 확인 내용 | 통과 기준 |
| --- | --- | --- |
| 핵심 시연 | 시작부터 결과 화면까지 흐름 유지 | 3회 연속 성공 |
| 오류 처리 | 비정상 입력 시 안내 문구 노출 | 사용자 행동 유도 가능 |
| 백업 자료 | 네트워크 실패 대비 영상/스크린샷 | 즉시 전환 가능 |
| Q&A 근거 | 기술/범위/테스트 답변 문서화 | 질문 10개 대응 가능 |
| 역할 분담 | 발표자, 데모 조작자, 시간 관리자 확정 | 리허설 동일 운영 |

이 루틴을 문서로 남기면 회고 단계에서 무엇이 실제로 효과가 있었는지 비교할 수 있습니다.

## 실전 앵커: MVP 수용 기준표와 CI/CD 파이프라인 구성

MVP 설계의 핵심은 기능 개수가 아니라 "완료 판정 기준"입니다. 완료 기준이 없으면 구현이 끝났는지 판단할 수 없고, 발표 리허설에서 계속 수정만 반복하게 됩니다.

### MVP 수용 기준표

| 기능 | 완료 조건 | 실패 조건 | 검증 방법 |
| --- | --- | --- | --- |
| 입력 처리 | 샘플 5건 정상 처리 | 필수 필드 누락 시 오작동 | 단위 테스트 + 수동 시연 |
| 핵심 계산 | 충돌 탐지 정확도 100% | 충돌 누락 1건 이상 | 테스트 데이터셋 검증 |
| 결과 화면 | 핵심 정보 1스크롤 내 표시 | 핵심 정보 누락 | 체크리스트 검토 |

### 캡스톤용 CI/CD 최소 파이프라인

```yaml
stages:
  - test
  - build
  - deploy

test:
  script:
    - pytest -q
build:
  script:
    - docker build -t capstone-app .
deploy:
  script:
    - ./scripts/deploy_staging.sh
```

이 구성은 복잡하지 않지만 효과가 큽니다. 테스트 통과 없이 빌드가 진행되지 않으므로 품질 하락을 초기에 차단할 수 있고, 스테이징 배포를 고정하면 데모 리허설 환경도 안정적으로 유지됩니다.

## 처음 질문으로 돌아가기

- **MVP는 왜 작은 완성품보다 학습 도구에 가깝다고 할까요?**
  - 이 글은 MVP 설계을 단순 정의가 아니라, 실무에서 마주하는 구체적 상황과 판단 과정으로 풀어냈습니다. 각 섹션의 예시와 체크리스트를 따라가며 자신의 상황에 적용할 수 있도록 구성했습니다.
- **핵심 흐름 하나를 고른다는 말은 무엇을 남기고 무엇을 버린다는 뜻일까요?**
  - 본문에서 제시한 예시 코드나 매트릭스를 참고하면, 좋은 판단 기준이 무엇인지 구체적으로 느낄 수 있습니다. 숫자나 기준이 아니라 '어떤 상황에서 이런 기준을 세우는가'를 이해하는 것이 중요합니다.
- **범위 밖 항목을 문서로 적는 일은 왜 중요할까요?**
  - 이 글의 핵심 메시지는 평가 기준이 무엇이든, 한 번의 판단으로 끝나지 않는다는 점입니다. 초반에 정한 기준을 문서로 남기고, 나중에 돌아봤을 때도 같은 실수를 반복하지 않도록 하는 과정 자체가 프로젝트를 성공으로 이끕니다.
