# Capstone Project 101 (8/10): 일정 관리

이 글은 캡스톤 프로젝트 101 시리즈의 8번째 글입니다.

대부분의 캡스톤 일정은 마지막 주에 갑자기 망가지지 않습니다. 실제로는 중간부터 조금씩 밀리는데, 그 차이를 일찍 드러내는 장치가 없어서 마지막에 한꺼번에 터집니다.

좋은 일정은 보기 좋은 계획표가 아니라 계획과 실제의 차이를 계속 드러내는 시스템입니다. 마일스톤, 주간 계획, 버퍼, 블로커 기록이 모두 필요한 이유도 여기에 있습니다.

이 글은 Capstone Project 101 시리즈의 8번째 글입니다. 여기서는 마일스톤, 주간 계획, 스탠드업, 위험 버퍼를 묶어 현실적인 일정 운영 리듬을 만드는 방법을 설명합니다.

> 멘탈 모델: 좋은 일정은 완벽한 예측이 아니라, 계획과 실제의 차이를 빨리 드러내고 조정할 수 있게 만드는 운영 리듬입니다.

## 먼저 던지는 질문

- 왜 완벽해 보이는 계획이 실제로는 자주 무너질까요?
- 마일스톤과 주간 계획은 어떻게 구분해서 써야 할까요?
- 데일리 스탠드업은 왜 짧고 반복 가능해야 할까요?

## 이 글에서 배우는 내용

- 마일스톤 정의법
- 주간 계획 작성법
- 데일리 스탠드업 형식
- 위험 버퍼 계산 감각
- 진행률 측정 기준

## 왜 중요한가

캡스톤 일정은 생각보다 쉽게 무너집니다. 구현보다 디버깅이 오래 걸리고, 팀원 일정이 엇갈리고, 발표 준비가 예상보다 늦게 시작되기 때문입니다. 그래서 마지막 마감일만 적어 두는 방식으로는 현실을 따라가기 어렵습니다.

좋은 일정은 세밀해서 좋은 것이 아니라, 조정 가능해서 좋습니다. 무엇이 밀렸는지 보이게 만들고, 블로커가 생겼을 때 바로 드러나게 해야 합니다.

## 실전 문서 예시: 주간 실행 보드

아래 같은 보드는 예쁜 간트 차트보다 훨씬 빨리 현실을 보여 줍니다.

```text
주차 | 목표 | 완료 조건 | 블로커 | 버퍼 사용
1주차 | 요구사항 확정 | Must 스토리 승인 | 없음 | 0일
2주차 | 핵심 흐름 구현 | 입력과 결과 화면 연결 | CSV 데이터 정리 지연 | 1일
3주차 | 데모 리허설 | 60초 시연 성공 | 로그인 오류 재현 | 2일
4주차 | 발표 자료 확정 | Q&A 준비 완료 | 없음 | 0.5일
```

## 이 문서로 먼저 확인할 것

- 주간 목표가 한 줄로 읽히는지 확인합니다.
- 완료 조건이 결과 중심인지 봅니다.
- 블로커를 숨기지 않고 기록합니다.
- 버퍼 사용량을 따로 적어 일정 체력을 추적합니다.

## 핵심 용어

- **milestone**: 큰 결과 단위입니다.
- **weekly plan**: 한 주 동안 끝내야 할 계획입니다.
- **standup**: 짧게 맞추는 일일 공유입니다.
- **buffer**: 예상 밖 문제를 위한 여유 시간입니다.
- **progress**: 측정 가능한 진척 상태입니다.

## 이전과 이후

**Before**: 마지막 마감일만 적어 둡니다.

**After**: 마일스톤, 주간 계획, 버퍼를 함께 둡니다.

## 실습: 일정 표

### 1단계 — 마일스톤

```python
milestones = ["MVP", "Demo", "Final"]
```

마일스톤은 너무 잘게 쪼개기보다, 결과가 보이는 단위로 세 개에서 다섯 개 정도 두는 편이 좋습니다.

### 2단계 — 주간 계획

```python
weeks = {1: "setup", 2: "core", 3: "polish"}
```

주간 계획은 길게 쓰기보다, 한 주에 무엇을 끝낼지 한 줄로 말할 수 있어야 합니다.

### 3단계 — 스탠드업 형식

```python
standup = ["yesterday", "today", "blockers"]
```

스탠드업은 짧고 반복 가능한 형식이어야 합니다. 길어지는 순간 공유보다 회의가 됩니다.

### 4단계 — 위험 버퍼

```python
buffer_days = 0.2 * 21
```

버퍼는 낭비가 아니라 안전장치입니다. 예상보다 느린 현실을 계획에 반영하는 최소 장치라고 보는 편이 좋습니다.

### 5단계 — 진척도 스냅샷

```python
progress = {"done": 12, "todo": 8, "blocked": 2}
```

진척도는 체감보다 숫자로 보는 편이 낫습니다. 막힌 항목이 얼마나 있는지 드러나야 조정이 빨라집니다.

## 이 코드에서 먼저 볼 점

- 마일스톤은 세 개에서 다섯 개 정도가 적당합니다.
- 주간 계획은 한 줄로 읽혀야 합니다.
- 버퍼는 전체 일정의 약 20% 수준입니다.
- 블로커는 숨기지 않고 드러내야 합니다.

## 자주 하는 실수 5가지

2. 스탠드업이 길어집니다.
3. 진행률을 감으로 판단합니다.
4. 주간 계획을 고정된 문서로 생각합니다.
5. 블로커를 늦게 공유합니다.

## 실전 확장: 테스트 전략 표와 통합 테스트 시나리오 예시

일정 관리에서 가장 자주 과소평가되는 작업은 테스트입니다. 구현 일정만 계획하고 테스트 일정을 별도로 확보하지 않으면, 마지막 주에 결함이 몰려 전체 계획이 흔들립니다. 따라서 일정 문서 안에 테스트 전략을 명시적으로 포함해야 합니다.

### 테스트 전략 표

| 테스트 수준 | 목적 | 담당 | 실행 시점 | 합격 기준 |
| --- | --- | --- | --- | --- |
| 단위 테스트 | 핵심 함수 로직 검증 | 백엔드/프런트 개발자 | 기능 개발 직후 | 핵심 함수 통과율 90% 이상 |
| 통합 테스트 | 컴포넌트 간 연결 검증 | QA + 개발자 | 주 1회 | 핵심 흐름 100% 통과 |
| 회귀 테스트 | 기존 기능 영향 확인 | QA | 배포 전 | 주요 시나리오 실패 0건 |
| 데모 리허설 테스트 | 발표 환경 재현 검증 | 팀 전체 | 발표 전 주 2회 | 60초 데모 3회 연속 성공 |

### 통합 테스트 시나리오 예시

| 시나리오 ID | 입력 | 기대 결과 | 실패 시 대응 |
| --- | --- | --- | --- |
| IT-01 | 정상 시간표 입력 | 충돌 없음 메시지 표시 | 입력 파서 로그 확인 |
| IT-02 | 충돌 시간표 입력 | 충돌 과목과 시간대 표시 | 계산 모듈 디버깅 |
| IT-03 | 빈 입력 제출 | 사용자 친화적 오류 메시지 | 프런트 검증 추가 |
| IT-04 | 잘못된 형식 CSV | 형식 오류 안내와 재입력 유도 | 예외 처리 강화 |
| IT-05 | 반복 요청 20회 | 평균 응답 시간 기준 충족 | 캐시/연산 최적화 |

### 일정과 테스트 연동 규칙

- 각 마일스톤마다 테스트 통과 기준을 별도로 둡니다.
- 신규 기능이 추가되면 관련 회귀 테스트를 즉시 갱신합니다.
- 블로커 이슈는 기능 이슈보다 우선 처리합니다.
- 발표 주차에는 테스트 실패 건수 추세를 매일 공유합니다.
- 테스트 통과율이 목표 이하이면 기능 추가를 중단합니다.

### 주간 테스트 리포트 포맷

```text
주차: 4주차
실행 테스트 수: 28
통과: 24
실패: 4
핵심 실패 원인: 입력 형식 예외 처리 누락
다음 액션: 파서 검증 로직 보강, 회귀 케이스 3건 추가
```

이 리포트는 복잡한 품질 도구 없이도 일정 리스크를 조기 감지하게 합니다. 캡스톤에서는 품질 지표가 단순할수록 팀이 빠르게 공유하고 행동으로 옮기기 쉽습니다.

### 발표 직전 안정화 체크

- 핵심 시나리오 5개를 고정합니다.
- 데모와 동일한 환경에서 테스트합니다.
- 네트워크 불안정 상황을 가정한 대체 시연 자료를 준비합니다.
- 결함 우선순위를 P1/P2로 분리해 대응합니다.
- 최종 리허설 로그를 회고 문서에 연결합니다.

일정 관리의 목적은 날짜를 채우는 것이 아니라 실패를 앞당겨 발견하는 것입니다. 테스트 전략을 일정 문서에 통합하면 계획과 실제 차이를 더 빠르게 파악할 수 있고, 후반부 급격한 일정 붕괴를 예방할 수 있습니다.

## 심화 보강: 의사결정 근거 문서화 패턴

캡스톤 문서의 분량을 늘리는 목적은 글자 수 자체가 아니라 판단의 흔적을 남기는 데 있습니다. 같은 결과물을 만들더라도 판단 근거가 기록된 팀과 기록되지 않은 팀의 학습 밀도는 크게 다릅니다. 아래 패턴은 어떤 장에서든 공통으로 적용할 수 있는 최소 문서화 규칙입니다.

### 의사결정 카드 템플릿

| 항목 | 작성 기준 | 예시 |
| --- | --- | --- |
| 결정 제목 | 동사형으로 짧게 작성 | 입력 형식을 CSV로 고정한다 |
| 배경 | 현재 문제와 제약을 3문장 내 서술 | 발표 전 2주 내 구현 안정화 필요 |
| 선택지 | 최소 2개 이상 비교 | CSV, 엑셀 직접 업로드 |
| 평가 기준 | 시간, 위험, 구현 난도 명시 | 학습 비용, 오류 가능성 |
| 최종 결정 | 한 문장으로 고정 | CSV 입력을 MVP 범위로 채택 |
| 후속 작업 | 실행 가능한 액션으로 변환 | 파서 검증 테스트 5건 추가 |

### 리스크 등록부 예시

| 리스크 ID | 설명 | 발생 가능성 | 영향도 | 대응 전략 | 담당 |
| --- | --- | --- | --- | --- | --- |
| RK-01 | 외부 데이터 접근 지연 | 중간 | 높음 | 샘플 데이터 대체 경로 준비 | 데이터 담당 |
| RK-02 | 데모 환경 네트워크 불안정 | 중간 | 높음 | 오프라인 영상 백업 준비 | 발표 담당 |
| RK-03 | 통합 테스트 누락 | 높음 | 중간 | 주간 회귀 테스트 슬롯 고정 | QA 담당 |
| RK-04 | 요구사항 빈번한 변경 | 중간 | 중간 | 변경 승인 규칙 명시 | 팀 리드 |

리스크 등록부는 불안을 기록하는 문서가 아니라 우선순위를 결정하는 문서입니다. 발생 가능성과 영향도를 함께 보면 어떤 리스크를 먼저 줄여야 하는지 명확해집니다.

### 주간 리뷰 질문

- 이번 주 결정 중 다음 주에 다시 검토할 항목은 무엇인가
- 일정 편차를 만든 원인은 기술 이슈인가, 의사결정 지연인가
- 범위 확장 요청이 들어왔을 때 거절 기준이 문서에 있는가
- 테스트 실패가 특정 구간에 집중되는가
- 발표 시점에 그대로 설명 가능한 근거가 충분한가

### 품질을 높이는 기록 원칙

첫째, 문장은 짧고 검증 가능하게 씁니다. 둘째, 수치가 가능한 항목은 반드시 수치로 남깁니다. 셋째, 결정과 액션을 분리하지 않습니다. 넷째, 회의 직후 24시간 내 문서를 갱신합니다. 다섯째, 변경 이유를 생략하지 않습니다. 이 다섯 가지 원칙만 지켜도 캡스톤 문서는 단순 제출물이 아니라 다음 프로젝트를 위한 운영 매뉴얼로 전환됩니다.

### 적용 체크리스트

- [ ] 결정 카드가 최신 상태로 유지됩니다.
- [ ] 리스크 등록부에 담당자와 대응 전략이 있습니다.
- [ ] 주간 리뷰 질문에 대한 답변이 기록됩니다.
- [ ] 변경 로그가 날짜 순으로 정리됩니다.
- [ ] 발표와 회고 문서가 같은 근거를 참조합니다.

위 보강 블록은 각 장의 주제와 별개로 공통 적용되는 운영 원칙입니다. 캡스톤의 난도는 기술 난도보다 의사결정 난도에서 크게 올라갑니다. 따라서 문서화 패턴을 반복 적용하면 팀의 판단 체계가 안정되고, 일정과 품질, 발표 전달력이 동시에 개선됩니다.

### 추가 보강: 발표 직전 최종 점검 루틴

발표 직전에는 새 기능 추가를 중단하고 안정화 루틴을 고정합니다. 최종 점검 루틴은 기술 완성도를 높이는 절차가 아니라 실패 확률을 낮추는 절차입니다. 핵심 흐름 시연, 오류 메시지 점검, 백업 자료 확인, 예상 질문 재점검, 역할별 멘트 정렬을 반복하면 전달 품질이 크게 향상됩니다.

| 점검 항목 | 확인 내용 | 통과 기준 |
| --- | --- | --- |
| 핵심 시연 | 시작부터 결과 화면까지 흐름 유지 | 3회 연속 성공 |
| 오류 처리 | 비정상 입력 시 안내 문구 노출 | 사용자 행동 유도 가능 |
| 백업 자료 | 네트워크 실패 대비 영상/스크린샷 | 즉시 전환 가능 |
| Q&A 근거 | 기술/범위/테스트 답변 문서화 | 질문 10개 대응 가능 |
| 역할 분담 | 발표자, 데모 조작자, 시간 관리자 확정 | 리허설 동일 운영 |

이 루틴을 문서로 남기면 회고 단계에서 무엇이 실제로 효과가 있었는지 비교할 수 있습니다.

### 보강 메모: 다음 프로젝트로 이관할 핵심 문장

이번 장의 핵심 판단 기준을 3문장으로 요약해 저장소 루트의 프로젝트 로그에 남깁니다. 다음 프로젝트 킥오프에서 이 문장을 먼저 읽으면 재시작 비용을 크게 줄일 수 있습니다. 문장 요약은 길이보다 재사용 가능성이 중요합니다.

## 실전 앵커: 간트 차트 운영 예시와 주간 점검 규칙

일정 관리는 일정표를 예쁘게 만드는 일이 아니라 편차를 조기에 발견하는 일입니다. 캡스톤에서는 1주 지연이 바로 발표 품질 저하로 이어지기 때문에, 매주 같은 형식으로 점검해야 합니다.

### 8주 간트 차트 예시

```text
주차        1 2 3 4 5 6 7 8
문제/요구사항 █ █
MVP 구현         █ █ █
통합 테스트             █ █
발표 자료 준비               █ █
리허설 및 수정                  █ █
```

### 주간 점검 보드 템플릿

| 항목 | 계획 | 실제 | 편차 | 조치 |
| --- | --- | --- | --- | --- |
| 구현 완료율 | 60% | 45% | -15% | 범위 축소 |
| 테스트 통과율 | 90% | 82% | -8% | 회귀 테스트 추가 |
| 리허설 횟수 | 2회 | 1회 | -1회 | 일정 재배치 |

편차를 기록할 때는 변명보다 조치가 중요합니다. "왜 늦었는가"를 1문장으로 적고, "무엇을 줄일 것인가"를 바로 결정해야 다음 주가 안정됩니다. 일정표는 예측 도구가 아니라 우선순위 조정 도구라는 점을 계속 유지해야 합니다.

## 처음 질문으로 돌아가기

- **왜 완벽해 보이는 계획이 실제로는 자주 무너질까요?**
  - 이 글은 일정 관리을 단순 정의가 아니라, 실무에서 마주하는 구체적 상황과 판단 과정으로 풀어냈습니다. 각 섹션의 예시와 체크리스트를 따라가며 자신의 상황에 적용할 수 있도록 구성했습니다.
- **마일스톤과 주간 계획은 어떻게 구분해서 써야 할까요?**
  - 본문에서 제시한 예시 코드나 매트릭스를 참고하면, 좋은 판단 기준이 무엇인지 구체적으로 느낄 수 있습니다. 숫자나 기준이 아니라 '어떤 상황에서 이런 기준을 세우는가'를 이해하는 것이 중요합니다.
- **데일리 스탠드업은 왜 짧고 반복 가능해야 할까요?**
  - 이 글의 핵심 메시지는 평가 기준이 무엇이든, 한 번의 판단으로 끝나지 않는다는 점입니다. 초반에 정한 기준을 문서로 남기고, 나중에 돌아봤을 때도 같은 실수를 반복하지 않도록 하는 과정 자체가 프로젝트를 성공으로 이끕니다.
