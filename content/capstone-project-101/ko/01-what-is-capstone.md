# 캡스톤 프로젝트란 무엇인가

> 캡스톤 프로젝트 101 시리즈 (1/10)

<!-- a-grade-intro:begin -->

**핵심 질문**: *캡스톤* 은 *학과 과제* 와 *무엇이* 다를까요?

> *문제 정의* 부터 *발표* 까지 *전 과정* 을 학생이 *주도* 한다는 점이 다릅니다.

<!-- a-grade-intro:end -->

## 이 글에서 배울 것

- 캡스톤의 *정의*
- *목표* 와 *평가* 기준
- 일반 *과제* 와의 차이
- *팀* 의 역할
- 시리즈의 *흐름*

## 왜 중요한가

캡스톤은 *학교* 와 *현장* 을 잇는 *마지막* 다리입니다.

## 개념 한눈에 보기

```mermaid
flowchart LR
    P[Problem] --> R[Requirement]
    R --> M[MVP]
    M --> D[Demo]
    D --> X[Retro]
```

## 핵심 용어 정리

- **capstone**: *졸업* 작품.
- **stakeholder**: *이해관계자*.
- **MVP**: *최소* 제품.
- **demo**: *시연*.
- **retro**: *회고*.

## Before/After

**Before**: *큰 과제* 로 본다.

**After**: *작은 제품* 으로 본다.

## 실습: 캡스톤 정의 카드

### 1단계 — 한 줄 정의

```python
title = "강의 시간표 충돌 검사기"
```

### 2단계 — 사용자

```python
users = ["student", "advisor"]
```

### 3단계 — 가치

```python
value = "수강 신청 시간을 줄인다"
```

### 4단계 — 측정

```python
metric = "사용자가 충돌을 30초 안에 확인"
```

### 5단계 — 데모

```python
demo = "demo.mp4 + readme.md"
```

## 이 코드에서 주목할 점

- *제목* 은 한 줄.
- *사용자* 와 *가치* 는 짝.
- *측정 가능* 한 목표.

## 자주 하는 실수 5가지

1. ***주제* 를 *너무 크게* 잡는다.**
2. ***사용자* 가 *모호* 하다.**
3. ***측정* 기준이 없다.**
4. ***데모* 를 *마지막* 에 만든다.**
5. ***회고* 를 생략한다.**

## 실무에서는 이렇게 쓰입니다

신입 *온보딩 과제* 가 캡스톤과 *거의 같은 모양* 으로 주어집니다.

## 시니어 엔지니어는 이렇게 생각합니다

- *문제* 가 우선.
- *작게* 시작.
- *측정* 가능.
- *공개* 한다.
- *회고* 한다.

## 체크리스트

- [ ] 한 줄 *정의*.
- [ ] *사용자* 명시.
- [ ] *측정* 기준.
- [ ] *데모* 형태.
