# Capstone Project 101 (1/10): 캡스톤 프로젝트란 무엇인가

캡스톤 프로젝트를 처음 설명할 때 많은 팀이 '졸업 전 큰 과제'라는 말로 끝냅니다. 틀린 말은 아니지만, 그 설명만으로는 왜 어떤 팀은 끝까지 완주하고 어떤 팀은 중간에 흔들리는지 설명하기 어렵습니다.

캡스톤은 기능 몇 개를 구현하는 수업이 아니라 문제를 정하고, 제한된 시간 안에 검증 가능한 결과를 만들고, 그 과정을 다시 설명하는 연습입니다. 이 관점이 없으면 초반의 주제 선정부터 마지막 발표와 회고까지 기준이 계속 바뀝니다.

이 글은 Capstone Project 101 시리즈의 첫 번째 글입니다. 여기서는 캡스톤을 일반 과제와 구분하는 기준과, 시리즈 전체를 읽을 때 붙잡아야 할 기본 흐름을 정리합니다.

> 멘탈 모델: 캡스톤은 큰 기능을 많이 만드는 수업이 아니라, 문제 정의부터 데모와 회고까지 하나의 실무 흐름으로 연결하는 작은 제품 개발 연습입니다.

## 먼저 던지는 질문

- 캡스톤 프로젝트는 일반 과제와 무엇이 다를까요?
- 왜 많은 학교가 졸업 직전 과목으로 캡스톤을 운영할까요?
- 좋은 캡스톤은 기능 수보다 무엇으로 평가해야 할까요?

## 이 글에서 배우는 내용

- 캡스톤의 기본 정의
- 목표와 평가 관점
- 일반 과제와의 차이
- 팀 기반 프로젝트 감각
- 시리즈 전체 흐름

## 왜 중요한가

캡스톤은 학교와 현장 사이를 잇는 마지막 다리에 가깝습니다. 일반 과제는 보통 정답과 채점 기준이 비교적 분명하지만, 캡스톤은 문제를 어떻게 정의할지, 누구를 사용자로 볼지, 어디까지 만들지, 무엇을 데모로 보여 줄지까지 스스로 정해야 합니다. 그래서 결과물뿐 아니라 판단 과정 자체가 훈련 대상이 됩니다.

현업에서도 첫 프로젝트의 구조는 비슷합니다. 문제를 설명하고, 사용자 가치를 정리하고, 제한된 시간 안에 설득력 있는 결과를 만들어야 합니다. 캡스톤을 잘 이해한 팀은 초반부터 기능 목록보다 문제와 데모를 먼저 붙잡습니다.

## 실전 문서 예시: 한 장짜리 프로젝트 브리프

캡스톤을 일반 과제와 구분하는 가장 쉬운 방법은, 프로젝트 시작 시점에 아래처럼 한 장짜리 브리프를 만들 수 있는지 보는 것입니다.

```text
프로젝트 제목: 수강 신청 시간표 충돌 검사기
대상 사용자: 복수전공 학생과 신입생
문제 문장: 수강 신청 직전에 시간표 충돌을 직접 계산하느라 시간이 오래 걸린다
핵심 가치: 30초 안에 충돌 여부를 확인한다
데모 기준: 샘플 시간표를 넣고 충돌 여부와 추천 대안을 보여 준다
성공 기준: 첫 사용자가 설명 없이 핵심 흐름을 끝낸다
```

## 이 문서로 먼저 확인할 것

- 문제 문장이 기능 설명이 아니라 사용자 상황으로 적혀 있는지 확인합니다.
- 핵심 가치가 시간, 정확도, 편의 같은 변화로 표현되어 있는지 봅니다.
- 데모 기준이 발표 당일 실제로 재현 가능한 수준인지 점검합니다.
- 성공 기준이 팀 내부 감상이 아니라 관찰 가능한 행동인지 살핍니다.

## 핵심 용어

- **capstone**: 졸업 직전 수행하는 종합 프로젝트입니다.
- **stakeholder**: 결과물에 관심을 갖거나 영향을 받는 이해관계자입니다.
- **MVP**: 가장 작은 검증 가능한 제품입니다.
- **demo**: 핵심 흐름을 보여 주는 시연입니다.
- **retro**: 프로젝트를 돌아보며 학습을 정리하는 회고입니다.

**Before**: 캡스톤을 큰 과제라고만 봅니다.

## 실습: 캡스톤 정의 카드

### 1단계 — 한 줄 제목

제목 초안은 `course schedule conflict checker`처럼 한 줄로 적습니다.

프로젝트 제목은 화려할수록 좋은 것이 아니라, 무엇을 하는지 바로 이해될수록 좋습니다. 한 줄 제목이 선명해야 팀의 대화도 선명해집니다.

### 2단계 — 사용자

첫 사용자 후보는 `student`, `advisor`처럼 짧게 적어 범위를 눈에 보이게 만듭니다.

사용자를 좁히면 요구사항도 함께 좁혀집니다. 반대로 모두를 사용자로 잡으면 프로젝트는 빠르게 커집니다.

### 3단계 — 가치

핵심 가치는 `cuts time spent on registration`처럼 변화 중심 문장으로 정리합니다.

가치는 기능 설명이 아니라 변화 설명이어야 합니다. 무엇을 만들었는지가 아니라, 무엇이 더 쉬워지는지를 적는 편이 좋습니다.

### 4단계 — 지표

성공 지표는 `users confirm conflicts in 30 seconds`처럼 시간과 행동이 함께 드러나게 적습니다.

지표가 들어가면 프로젝트의 성공 기준이 명확해집니다. 막연히 편하다는 표현보다, 얼마 안에 무엇을 할 수 있는지가 더 강한 기준입니다.

### 5단계 — 데모

데모 산출물은 `demo.mp4 + readme.md`처럼 미리 적어 두면 발표 준비 기준이 분명해집니다.

데모 형식을 미리 적어 두면 어떤 흐름을 반드시 완성해야 하는지가 자연스럽게 드러납니다. 발표 직전 급히 정리하는 팀일수록 핵심 흐름이 흔들리기 쉽습니다.

## 이 코드에서 먼저 볼 점

- 제목은 한 줄로 끝납니다.
- 사용자와 가치는 짝으로 움직입니다.
- 지표는 측정 가능해야 합니다.
- 데모 형식이 정해져야 우선순위가 분명해집니다.

## 자주 하는 실수 5가지

1. 주제를 너무 크게 잡습니다.
2. 사용자를 모호하게 적습니다.
3. 측정 기준 없이 감각적인 표현만 남깁니다.
4. 데모를 마지막 작업으로 미룹니다.
5. 회고를 형식적인 문서라고만 생각합니다.

## 실무에서는 이렇게 이어집니다

신입 온보딩 프로젝트나 사내 파일럿도 구조는 거의 같습니다. 문제를 한 줄로 설명하고, 누가 쓸지 정하고, 작은 데모를 만들고, 피드백을 통해 다음 단계를 정합니다. 캡스톤을 제대로 경험한 사람은 주어진 기능만 구현하는 사람보다, 무엇을 왜 만드는지 설명할 수 있는 사람에 더 가깝습니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 문제를 먼저 봅니다.
- 처음부터 작게 시작합니다.
- 성공 기준을 숫자로 둡니다.
- 결과를 보여 줄 데모를 초반부터 상상합니다.
- 회고를 다음 프로젝트의 자산으로 남깁니다.

## 정리와 다음 글

캡스톤의 출발점은 기능 목록이 아니라 프로젝트 전체 흐름을 보는 관점입니다. 문제 정의, 요구사항, MVP, 데모, 회고가 하나의 체인으로 이어져야 이후 판단도 흔들리지 않습니다. 다음 글에서는 이 흐름을 감당할 수 있는 주제를 어떻게 고를지 다룹니다.

## 실전 확장: 주제 선정 평가 매트릭스와 좋은/나쁜 주제 비교

캡스톤의 출발점은 결국 주제 선택입니다. 이 글이 1장인 이유도, 캡스톤의 본질을 이해한 팀만이 주제를 현실적으로 고를 수 있기 때문입니다. 주제는 한 번 고르고 끝나는 명사가 아니라, 학기 내내 의사결정을 통과시키는 기준 문장입니다. 따라서 초기에는 반드시 비교 가능한 틀을 먼저 만들고, 그 틀로 후보를 검증해야 합니다. 아래 매트릭스는 초보 팀이 가장 자주 놓치는 항목을 중심으로 구성했습니다.

### 주제 선정 평가 매트릭스

| 평가 항목 | 질문 | 1점 기준 | 3점 기준 | 5점 기준 | 가중치 |
| --- | --- | --- | --- | --- | --- |
| 사용자 문제 명확성 | 누가 어떤 불편을 겪는가가 선명한가 | 사용자 불명확 | 사용자 집단은 있으나 상황 모호 | 사용자와 상황이 한 문장으로 명확 | 0.25 |
| 검증 가능성 | 한 학기 안에 데모와 피드백이 가능한가 | 데모 기준 없음 | 데모 가능하지만 측정 약함 | 데모와 성공 지표가 모두 명확 | 0.20 |
| 구현 실현성 | 팀 기술 역량으로 구현 가능한가 | 핵심 기술 전원 미경험 | 일부 경험 보유 | 핵심 기술 다수 경험 보유 | 0.20 |
| 데이터/리소스 접근성 | 필요한 데이터와 환경을 확보 가능한가 | 접근 경로 없음 | 부분 접근 가능 | 시작 시점에 확보 가능 | 0.15 |
| 발표 전달력 | 문제-해결-결과를 짧게 설명 가능한가 | 설명이 추상적 | 설명 가능하나 핵심 약함 | 3분 내 설득 가능한 구조 | 0.10 |
| 확장 여지 | MVP 이후 확장 방향이 존재하는가 | 확장 경로 불명확 | 제한적 확장 가능 | 확장 가설 2개 이상 명확 | 0.10 |

합계 점수는 `각 항목 점수 x 가중치`의 합으로 계산합니다. 그러나 합계 숫자보다 더 중요한 것은 항목별 편차입니다. 예를 들어 총점이 높아도 `검증 가능성`이 낮다면 학기 후반 발표 준비 단계에서 위험이 크게 드러납니다. 반대로 총점이 약간 낮아도 사용자 문제 명확성과 데모 가능성이 높다면 캡스톤에서는 더 안전한 선택이 됩니다.

### 좋은 주제와 나쁜 주제 비교표

| 구분 | 좋은 주제 예시 | 나쁜 주제 예시 | 차이의 핵심 |
| --- | --- | --- | --- |
| 문제 정의 | 수강 신청 직전 시간표 충돌 확인이 느리다 | AI로 대학생활을 혁신한다 | 좋은 주제는 문제가 관찰 가능함 |
| 사용자 범위 | 신입생과 복수전공생 | 대학생 전체, 교직원, 외부 사용자 | 좋은 주제는 첫 사용자가 좁음 |
| 데모 가능성 | 샘플 데이터로 60초 시연 가능 | 대규모 실시간 데이터 연동 필수 | 좋은 주제는 발표 환경에서 재현 가능 |
| 기술 복잡도 | 기존 프레임워크로 구현 가능 | 다중 모델, 실시간 스트리밍, 추천 엔진 동시 구축 | 좋은 주제는 핵심 리스크가 제한됨 |
| 성공 기준 | 첫 사용자가 30초 내 충돌 여부 확인 | 사용자 만족도 향상 | 좋은 주제는 측정 기준이 명확함 |

### 평가 운영 절차

1. 후보를 최소 3개 이상 준비합니다.
2. 팀원 각자가 독립적으로 점수를 먼저 매깁니다.
3. 점수 차이가 큰 항목만 토론합니다.
4. 토론 결과를 반영해 2차 점수를 확정합니다.
5. 최종 선택과 탈락 이유를 3문장으로 기록합니다.

이 절차를 쓰는 이유는 의견 충돌을 없애기 위해서가 아니라, 충돌을 문서화 가능한 형태로 바꾸기 위해서입니다. 팀원이 서로 다른 점수를 주는 것은 자연스럽습니다. 중요한 것은 왜 다르게 봤는지 근거를 남기고, 다음 의사결정에서 같은 충돌을 반복하지 않게 만드는 일입니다.

### 실패를 줄이는 검증 질문

- 이 주제를 한 문장으로 설명할 때 기능명이 아니라 사용자 문제를 먼저 말할 수 있는가
- 첫 데모를 2주 안에 만들어볼 수 있는가
- 데이터가 막혔을 때 대체 입력 경로가 있는가
- 팀원이 빠져도 핵심 흐름을 유지할 수 있는가
- 발표 당일 인터넷 불안정 상황에서도 시연 가능한가

위 질문에서 두 개 이상이 "아니오"라면 주제를 축소하는 편이 좋습니다. 캡스톤의 목표는 가장 거대한 제품이 아니라, 가장 설득력 있게 완주한 제품입니다. 따라서 초반에는 야심보다 검증 구조를 먼저 선택해야 합니다. 이 기준이 잡히면 이후 요구사항, MVP, 일정, 발표 자료가 서로 충돌하지 않고 같은 방향으로 정렬됩니다.

## 심화 보강: 의사결정 근거 문서화 패턴

캡스톤 문서의 분량을 늘리는 목적은 글자 수 자체가 아니라 판단의 흔적을 남기는 데 있습니다. 같은 결과물을 만들더라도 판단 근거가 기록된 팀과 기록되지 않은 팀의 학습 밀도는 크게 다릅니다. 아래 패턴은 어떤 장에서든 공통으로 적용할 수 있는 최소 문서화 규칙입니다.

### 의사결정 카드 템플릿

| 항목 | 작성 기준 | 예시 |
| --- | --- | --- |
| 결정 제목 | 동사형으로 짧게 작성 | 입력 형식을 CSV로 고정한다 |
| 배경 | 현재 문제와 제약을 3문장 내 서술 | 발표 전 2주 내 구현 안정화 필요 |
| 선택지 | 최소 2개 이상 비교 | CSV, 엑셀 직접 업로드 |
| 평가 기준 | 시간, 위험, 구현 난도 명시 | 학습 비용, 오류 가능성 |
| 최종 결정 | 한 문장으로 고정 | CSV 입력을 MVP 범위로 채택 |
| 후속 작업 | 실행 가능한 액션으로 변환 | 파서 검증 테스트 5건 추가 |

### 리스크 등록부 예시

| 리스크 ID | 설명 | 발생 가능성 | 영향도 | 대응 전략 | 담당 |
| --- | --- | --- | --- | --- | --- |
| RK-01 | 외부 데이터 접근 지연 | 중간 | 높음 | 샘플 데이터 대체 경로 준비 | 데이터 담당 |
| RK-02 | 데모 환경 네트워크 불안정 | 중간 | 높음 | 오프라인 영상 백업 준비 | 발표 담당 |
| RK-03 | 통합 테스트 누락 | 높음 | 중간 | 주간 회귀 테스트 슬롯 고정 | QA 담당 |
| RK-04 | 요구사항 빈번한 변경 | 중간 | 중간 | 변경 승인 규칙 명시 | 팀 리드 |

리스크 등록부는 불안을 기록하는 문서가 아니라 우선순위를 결정하는 문서입니다. 발생 가능성과 영향도를 함께 보면 어떤 리스크를 먼저 줄여야 하는지 명확해집니다.

### 주간 리뷰 질문

- 이번 주 결정 중 다음 주에 다시 검토할 항목은 무엇인가
- 일정 편차를 만든 원인은 기술 이슈인가, 의사결정 지연인가
- 범위 확장 요청이 들어왔을 때 거절 기준이 문서에 있는가
- 테스트 실패가 특정 구간에 집중되는가
- 발표 시점에 그대로 설명 가능한 근거가 충분한가

### 품질을 높이는 기록 원칙

첫째, 문장은 짧고 검증 가능하게 씁니다. 둘째, 수치가 가능한 항목은 반드시 수치로 남깁니다. 셋째, 결정과 액션을 분리하지 않습니다. 넷째, 회의 직후 24시간 내 문서를 갱신합니다. 다섯째, 변경 이유를 생략하지 않습니다. 이 다섯 가지 원칙만 지켜도 캡스톤 문서는 단순 제출물이 아니라 다음 프로젝트를 위한 운영 매뉴얼로 전환됩니다.

### 적용 체크리스트

- [ ] 결정 카드가 최신 상태로 유지됩니다.
- [ ] 리스크 등록부에 담당자와 대응 전략이 있습니다.
- [ ] 주간 리뷰 질문에 대한 답변이 기록됩니다.
- [ ] 변경 로그가 날짜 순으로 정리됩니다.
- [ ] 발표와 회고 문서가 같은 근거를 참조합니다.

위 보강 블록은 각 장의 주제와 별개로 공통 적용되는 운영 원칙입니다. 캡스톤의 난도는 기술 난도보다 의사결정 난도에서 크게 올라갑니다. 따라서 문서화 패턴을 반복 적용하면 팀의 판단 체계가 안정되고, 일정과 품질, 발표 전달력이 동시에 개선됩니다.

## 실전 앵커: 프로젝트 제안서 초안과 브랜치 운영 규칙

캡스톤을 정의한 직후에 가장 먼저 만들어야 할 문서는 제안서 초안입니다. 제안서는 길게 쓰는 문서가 아니라 팀이 같은 문제를 같은 단어로 합의했는지 확인하는 문서입니다. 아래 템플릿은 1페이지 안에서 끝내는 형태로 구성했습니다.

### 프로젝트 제안서 1페이지 템플릿

```text
프로젝트명:
문제 요약(3문장):
핵심 사용자:
핵심 시나리오(5단계 이내):
MVP 범위(포함):
MVP 비범위(제외):
성공 지표(수치 2개):
데모 방식(라이브/녹화/혼합):
주요 리스크와 대응:
```

이 템플릿의 장점은 빠른 합의입니다. 팀원마다 문서 스타일이 달라도 항목이 고정되어 있으면 논점이 흩어지지 않습니다. 특히 `MVP 비범위`를 명시하면 학기 중반에 발생하는 범위 확장을 미리 제어할 수 있습니다.

### Git 브랜치 전략 최소 규칙

문서 합의가 끝났으면 바로 저장소 운영 규칙을 정해야 합니다. 초반에는 단순한 Git 브랜치 전략이 가장 안정적입니다.

| 항목 | 규칙 | 이유 |
| --- | --- | --- |
| 기본 브랜치 | `main` 보호 | 직접 푸시를 막아 품질 유지 |
| 작업 브랜치 | `feat/*`, `fix/*`, `docs/*` | 변경 목적을 이름에서 즉시 식별 |
| 병합 방식 | PR + 1인 승인 | 단독 판단으로 인한 누락 방지 |
| 머지 조건 | 테스트 통과, 충돌 없음 | 데모 직전 회귀 실패 예방 |

```bash
git checkout -b feat/problem-card
git add .
git commit -m "add initial problem card and scope"
git push -u origin feat/problem-card
```

브랜치 전략은 고급 기능보다 일관성이 중요합니다. 캡스톤 팀에서 가장 큰 손실은 잘못된 기술 선택보다 변경 이력이 남지 않는 상태입니다. 따라서 작은 변경이라도 PR을 거쳐 기록을 남기는 습관이 필요합니다.

## 처음 질문으로 돌아가기

- **캡스톤 프로젝트는 일반 과제와 무엇이 다를까요?**
  - 일반 과제는 정답과 채점 기준이 정해져 있지만, 캡스톤은 문제 정의부터 데모까지 스스로 기준을 만들어야 합니다.
  - 이 글은 캡스톤 프로젝트란 무엇인가을 단순 정의가 아니라, 실무에서 마주하는 구체적 상황과 판단 과정으로 풀어냈습니다. 각 섹션의 예시와 체크리스트를 따라가며 자신의 상황에 적용할 수 있도록 구성했습니다.
- **왜 많은 학교가 졸업 직전 과목으로 캡스톤을 운영할까요?**
  - 현업의 초기 프로젝트 경험을 학교에서 체험해야 하기 때문입니다. 문제를 정의하고, 시간 안에 결과를 만들고, 의사결정을 설명하는 경험은 취업 직후 그대로 반복됩니다.
  - 본문에서 제시한 예시 코드나 매트릭스를 참고하면, 좋은 판단 기준이 무엇인지 구체적으로 느낄 수 있습니다. 숫자나 기준이 아니라 '어떤 상황에서 이런 기준을 세우는가'를 이해하는 것이 중요합니다.
- **좋은 캡스톤은 기능 수보다 무엇으로 평가해야 할까요?**
  - 기능 수가 아니라 문제를 얼마나 명확히 정의했는지, 데모를 얼마나 설듍력 있게 만들었는지, 팀의 판단 과정을 얼마나 잘 기록했는지를 봐야 합니다.
  - 이 글의 핵심 메시지는 평가 기준이 무엇이든, 한 번의 판단으로 끝나지 않는다는 점입니다. 초반에 정한 기준을 문서로 남기고, 나중에 돌아봤을 때도 같은 실수를 반복하지 않도록 하는 과정 자체가 프로젝트를 성공으로 이끕니다.
