# Capstone Project 101 (2/10): 주제 선정

이 글은 캡스톤 프로젝트 101 시리즈의 2번째 글입니다.

좋아 보이는 아이디어는 많지만, 한 학기 안에 끝까지 밀 수 있는 아이디어는 훨씬 적습니다. 캡스톤 주제 선정이 어려운 이유도 여기에 있습니다.

좋은 주제는 멋있어 보이는 주제가 아니라 팀이 끝까지 책임질 수 있는 주제입니다. 그래서 주제를 고를 때는 흥미, 사용자 가치, 구현 가능성, 발표 난도를 함께 봐야 합니다.

이 글은 Capstone Project 101 시리즈의 2번째 글입니다. 여기서는 후보를 비교하는 기준을 만들고, 범위가 큰 아이디어를 학기 안에 맞는 주제로 줄이는 방법을 설명합니다.

> 멘탈 모델: 좋은 주제는 새롭기만 한 주제가 아니라, 팀이 시간 안에 설명하고 구현하고 발표할 수 있는 범위로 정리된 주제입니다.

## 왜 중요한가

주제가 흔들리면 남은 학기도 같이 흔들립니다. 초반에 잘 고른 주제는 이후 요구사항 정리, MVP 설계, 발표 준비까지 계속 팀을 도와줍니다. 반대로 멋있어 보인다는 이유만으로 주제를 정하면 구현을 시작하기도 전에 범위와 난도가 팀을 압도하기 쉽습니다.

좋은 팀은 유행보다 완주 가능성을 먼저 봅니다. 캡스톤은 아이디어 경연이 아니라, 제한된 시간 안에 작지만 설득력 있는 결과물을 만드는 과정이기 때문입니다.

## 실전 문서 예시: 주제 비교 매트릭스

후보를 세 개 이상 두고 아래처럼 같은 축으로 점수를 매기면, 감으로 정한 결정인지 근거가 있는 결정인지 바로 드러납니다.

```text
후보 | 사용자 문제 | 영향 | 구현 가능성 | 데이터 접근성 | 데모 선명도
수강 충돌 검사기 | 높음 | 5 | 5 | 4 | 5
감정 일기 추천기 | 중간 | 3 | 4 | 3 | 3
캠퍼스 길찾기 | 중간 | 4 | 2 | 2 | 4

결론: 수강 충돌 검사기가 가장 작고, 가장 빨리 데모 가치가 드러난다.
```

## 이 문서로 먼저 확인할 것

- 점수축이 서로 겹치지 않는지 확인합니다.
- 데이터 접근성처럼 뒤늦게 터지는 리스크를 별도 축으로 둡니다.
- 최종 결론 옆에 한두 문장 이유를 남깁니다.
- 탈락한 후보도 보존해 두면 범위 조정 때 대안으로 되살릴 수 있습니다.

## 핵심 용어

- **idea**: 검토할 수 있는 주제 후보입니다.
- **filter**: 먼저 탈락시킬 기준입니다.
- **matrix**: 후보를 나란히 비교하는 표입니다.
- **score**: 각 기준에 대한 점수입니다.
- **pick**: 최종 선택입니다.

## 이전과 이후

**Before**: 멋진 주제만 찾으면 된다고 생각합니다.

**After**: 우리 팀에 맞는 주제를 고르는 일이 더 중요하다고 봅니다.

## 실습: 주제 비교 매트릭스

### 1단계 — 후보 정리

```python
ideas = ["schedule_checker", "mood_diary", "campus_map"]
```

후보는 적어도 세 개는 두는 편이 좋습니다. 둘만 놓으면 찬반 구도가 되기 쉽고, 세 개 이상이면 비교가 시작됩니다.

### 2단계 — 점수 축

```python
axes = ["impact", "feasibility", "interest"]
```

축은 팀의 기준을 드러냅니다. 사용자 가치, 실현 가능성, 팀의 흥미처럼 성격이 다른 항목을 두면 균형을 보기 좋습니다.

### 3단계 — 점수표

```python
score = {"schedule_checker": [4, 5, 4], "mood_diary": [3, 4, 5], "campus_map": [4, 3, 3]}
```

숫자는 완벽해서 쓰는 것이 아니라, 판단 이유를 드러내기 위해 씁니다. 누군가 실현 가능성을 5점으로 준 이유를 설명하는 순간 팀의 가정이 보입니다.

### 4단계 — 합계

```python
total = {k: sum(v) for k, v in score.items()}
```

합계는 빠른 비교에는 유용하지만 절대 기준은 아닙니다. 점수 분포의 균형도 함께 봐야 합니다.

### 5단계 — 선택

```python
pick = max(total, key=total.get)
```

최종 선택은 숫자가 대신해 주는 일이 아니라, 숫자를 근거로 팀 대화를 마무리하는 단계입니다. 왜 이 후보를 골랐는지 한두 문장으로 남겨 두면 이후 범위 조정이 쉬워집니다.

## 이 코드에서 먼저 볼 점

- 비교 구조가 있어야 감정 대신 기준으로 말할 수 있습니다.
- 축은 곧 팀의 판단 기준입니다.
- 합계뿐 아니라 균형도 함께 봐야 합니다.
- 선택 이유를 문장으로 남겨야 나중에 되돌아볼 수 있습니다.

## 자주 하는 실수 5가지

1. 트렌드만 따라갑니다.
2. 팀 역량을 실제보다 높게 잡습니다.
3. 비교표 없이 감으로 결정합니다.
4. 평가 축이 모호하거나 서로 겹칩니다.
5. 대안을 남기지 않고 첫 아이디어에 바로 묶입니다.

## 실무에서는 이렇게 이어집니다

제품 우선순위 회의도 본질은 비슷합니다. 후보를 모으고, 사용자 가치와 구현 비용을 비교하고, 지금 밀어야 할 하나를 고릅니다. 캡스톤에서 주제 비교 매트릭스를 써 보는 경험은 작은 제품 판단 연습으로 그대로 이어집니다.

## 정리와 다음 글

주제 선정은 흥미 경쟁이 아니라 완주 가능성을 고르는 일입니다. 비교 기준을 먼저 세우고, 작은 데모 가치가 가장 빨리 드러나는 후보를 고르면 이후 요구사항과 MVP도 훨씬 안정적으로 정리됩니다. 다음 글에서는 선택한 주제를 문제 문장으로 바꾸는 방법을 다룹니다.

## 실전 확장: 문제 정의 템플릿과 요구사항 명세 예시

주제를 골랐다면 바로 구현으로 넘어가기보다 문제 정의 문서를 먼저 고정해야 합니다. 주제 선정 직후 가장 많이 발생하는 오류는 "이미 모두가 같은 문제를 이해하고 있다"는 착각입니다. 실제로는 팀원마다 다른 사용자를 상정하고, 다른 성공 기준을 떠올립니다. 아래 템플릿은 그 차이를 빠르게 드러내는 최소 문서입니다.

### 문제 정의 템플릿

```text
[문제 이름]
한 줄 요약:

[사용자]
핵심 사용자 1:
핵심 사용자 2:
제외 사용자:

[관찰 사실]
현재 사용자가 겪는 불편:
불편이 발생하는 시점:
불편을 확인한 근거:

[문제 문장]
누가, 언제, 무엇을, 왜 어려워하는가:

[성공 기준]
정량 지표 1:
정량 지표 2:
정성 지표 1:

[핵심 가정]
가정 A:
가정 B:

[비범위]
이번 학기에서 하지 않을 것:
```

이 템플릿의 핵심은 작성 난이도가 아니라 검토 난이도를 낮춘다는 점입니다. 항목이 정해져 있으면 팀원이 서로 다른 전제를 가지고 있어도 어디에서 충돌하는지 바로 보입니다.

### 요구사항 명세 예시

| ID | 유형 | 요구사항 | 수용 기준 | 우선순위 |
| --- | --- | --- | --- | --- |
| RQ-01 | 기능 | 사용자는 시간표를 텍스트로 입력할 수 있어야 함 | 샘플 3개 입력 시 오류 없이 저장 | Must |
| RQ-02 | 기능 | 시스템은 충돌 과목을 식별해야 함 | 충돌 케이스 10개 중 10개 검출 | Must |
| RQ-03 | 기능 | 결과 화면은 충돌 구간을 시각적으로 보여야 함 | 핵심 정보 1스크롤 내 표시 | Should |
| RQ-04 | 비기능 | 결과 계산은 1초 이내 반환 | 평균 응답 시간 1초 이하 | Must |
| RQ-05 | 비기능 | 모바일 화면에서 사용 가능해야 함 | 390px 폭에서 레이아웃 깨짐 없음 | Should |

### 템플릿 적용 체크 포인트

- 문제 문장과 요구사항이 1:1로 연결되는지 확인합니다.
- 각 요구사항에 수용 기준이 빠지지 않았는지 점검합니다.
- Must가 전체의 60%를 넘으면 범위 과대 가능성을 의심합니다.
- 비기능 요구사항을 기능 요구사항과 분리합니다.
- 발표 데모에서 보여줄 항목을 별도로 표시합니다.

### 팀 리뷰 진행 방식

문서 리뷰는 20분 내로 끝내는 것이 좋습니다. 처음 5분은 작성자가 설명하고, 다음 10분은 질문만 받습니다. 마지막 5분은 수정 항목을 확정합니다. 이때 "좋다/나쁘다" 같은 감상 대신 "어떤 항목이 비어 있는가"로 말해야 문서 품질이 올라갑니다. 주제 선정 단계에서 이 습관을 만들면 이후 모든 장에서 의사결정 속도가 빨라집니다.



## 심화 보강: 의사결정 근거 문서화 패턴

캡스톤 문서의 분량을 늘리는 목적은 글자 수 자체가 아니라 판단의 흔적을 남기는 데 있습니다. 같은 결과물을 만들더라도 판단 근거가 기록된 팀과 기록되지 않은 팀의 학습 밀도는 크게 다릅니다. 아래 패턴은 어떤 장에서든 공통으로 적용할 수 있는 최소 문서화 규칙입니다.

### 의사결정 카드 템플릿

| 항목 | 작성 기준 | 예시 |
| --- | --- | --- |
| 결정 제목 | 동사형으로 짧게 작성 | 입력 형식을 CSV로 고정한다 |
| 배경 | 현재 문제와 제약을 3문장 내 서술 | 발표 전 2주 내 구현 안정화 필요 |
| 선택지 | 최소 2개 이상 비교 | CSV, 엑셀 직접 업로드 |
| 평가 기준 | 시간, 위험, 구현 난도 명시 | 학습 비용, 오류 가능성 |
| 최종 결정 | 한 문장으로 고정 | CSV 입력을 MVP 범위로 채택 |
| 후속 작업 | 실행 가능한 액션으로 변환 | 파서 검증 테스트 5건 추가 |

### 리스크 등록부 예시

| 리스크 ID | 설명 | 발생 가능성 | 영향도 | 대응 전략 | 담당 |
| --- | --- | --- | --- | --- | --- |
| RK-01 | 외부 데이터 접근 지연 | 중간 | 높음 | 샘플 데이터 대체 경로 준비 | 데이터 담당 |
| RK-02 | 데모 환경 네트워크 불안정 | 중간 | 높음 | 오프라인 영상 백업 준비 | 발표 담당 |
| RK-03 | 통합 테스트 누락 | 높음 | 중간 | 주간 회귀 테스트 슬롯 고정 | QA 담당 |
| RK-04 | 요구사항 빈번한 변경 | 중간 | 중간 | 변경 승인 규칙 명시 | 팀 리드 |

리스크 등록부는 불안을 기록하는 문서가 아니라 우선순위를 결정하는 문서입니다. 발생 가능성과 영향도를 함께 보면 어떤 리스크를 먼저 줄여야 하는지 명확해집니다.

### 주간 리뷰 질문

- 이번 주 결정 중 다음 주에 다시 검토할 항목은 무엇인가
- 일정 편차를 만든 원인은 기술 이슈인가, 의사결정 지연인가
- 범위 확장 요청이 들어왔을 때 거절 기준이 문서에 있는가
- 테스트 실패가 특정 구간에 집중되는가
- 발표 시점에 그대로 설명 가능한 근거가 충분한가

### 품질을 높이는 기록 원칙

첫째, 문장은 짧고 검증 가능하게 씁니다. 둘째, 수치가 가능한 항목은 반드시 수치로 남깁니다. 셋째, 결정과 액션을 분리하지 않습니다. 넷째, 회의 직후 24시간 내 문서를 갱신합니다. 다섯째, 변경 이유를 생략하지 않습니다. 이 다섯 가지 원칙만 지켜도 캡스톤 문서는 단순 제출물이 아니라 다음 프로젝트를 위한 운영 매뉴얼로 전환됩니다.

### 적용 체크리스트

- [ ] 결정 카드가 최신 상태로 유지됩니다.
- [ ] 리스크 등록부에 담당자와 대응 전략이 있습니다.
- [ ] 주간 리뷰 질문에 대한 답변이 기록됩니다.
- [ ] 변경 로그가 날짜 순으로 정리됩니다.
- [ ] 발표와 회고 문서가 같은 근거를 참조합니다.

위 보강 블록은 각 장의 주제와 별개로 공통 적용되는 운영 원칙입니다. 캡스톤의 난도는 기술 난도보다 의사결정 난도에서 크게 올라갑니다. 따라서 문서화 패턴을 반복 적용하면 팀의 판단 체계가 안정되고, 일정과 품질, 발표 전달력이 동시에 개선됩니다.



### 추가 보강: 발표 직전 최종 점검 루틴

발표 직전에는 새 기능 추가를 중단하고 안정화 루틴을 고정합니다. 최종 점검 루틴은 기술 완성도를 높이는 절차가 아니라 실패 확률을 낮추는 절차입니다. 핵심 흐름 시연, 오류 메시지 점검, 백업 자료 확인, 예상 질문 재점검, 역할별 멘트 정렬을 반복하면 전달 품질이 크게 향상됩니다.

| 점검 항목 | 확인 내용 | 통과 기준 |
| --- | --- | --- |
| 핵심 시연 | 시작부터 결과 화면까지 흐름 유지 | 3회 연속 성공 |
| 오류 처리 | 비정상 입력 시 안내 문구 노출 | 사용자 행동 유도 가능 |
| 백업 자료 | 네트워크 실패 대비 영상/스크린샷 | 즉시 전환 가능 |
| Q&A 근거 | 기술/범위/테스트 답변 문서화 | 질문 10개 대응 가능 |
| 역할 분담 | 발표자, 데모 조작자, 시간 관리자 확정 | 리허설 동일 운영 |

이 루틴을 문서로 남기면 회고 단계에서 무엇이 실제로 효과가 있었는지 비교할 수 있습니다.



### 보강 메모: 다음 프로젝트로 이관할 핵심 문장

이번 장의 핵심 판단 기준을 3문장으로 요약해 저장소 루트의 프로젝트 로그에 남깁니다. 다음 프로젝트 킥오프에서 이 문장을 먼저 읽으면 재시작 비용을 크게 줄일 수 있습니다. 문장 요약은 길이보다 재사용 가능성이 중요합니다.



### 마무리 보강: 검토 회의 운영 규칙

검토 회의는 30분 제한으로 운영하고, 결론 없는 논의를 다음 액션으로 분해합니다. 안건마다 결정자 1명과 기록자 1명을 고정하면 책임 공백을 줄일 수 있습니다. 회의 종료 전에는 반드시 다음 검토 시점과 확인 지표를 합의합니다.



추가 확인 메모: 본 장의 핵심 기준을 다음 스프린트 킥오프 문서 첫 줄에 재사용합니다.



보강 문장: 주제 선정의 품질은 아이디어의 화려함보다 검증 가능한 범위 설정에서 결정됩니다.

## 실전 앵커: 주제 제안서 비교표와 간트 차트 예시

주제 선정 단계에서는 아이디어 토론보다 제안서 비교가 더 중요합니다. 같은 길이의 제안서를 3개 이상 만들고 동일한 기준으로 평가하면 팀의 선호와 프로젝트 리스크를 분리해서 볼 수 있습니다.

### 주제 제안서 비교 템플릿

| 항목 | 후보 A | 후보 B | 후보 C |
| --- | --- | --- | --- |
| 사용자 문제 명확성 |  |  |  |
| 2주 내 데모 가능성 |  |  |  |
| 데이터 접근성 |  |  |  |
| 팀 기술 적합성 |  |  |  |
| 발표 전달력 |  |  |  |
| 총평(2문장) |  |  |  |

평가표를 채울 때는 "잘 될 것 같다" 대신 근거를 씁니다. 예를 들어 데이터 접근성 점수를 낮게 준 이유가 API 권한 이슈인지, 개인정보 이슈인지 구분하면 다음 단계에서 대응 전략이 달라집니다.

### 4주 사전 검증 간트 차트 예시

```text
주차      1주  2주  3주  4주
문제 정의 ████
데이터 확인   ████
핵심 기능 스파이크      ████
데모 초안 제작            ████
피드백 반영                 ████
```

간트 차트는 정교한 도구보다 의존 관계를 보여 주는 데 목적이 있습니다. 주제 선정 직후 이 정도 수준의 차트를 만들면 "언제까지 무엇이 준비되어야 하는가"가 명확해집니다. 일정이 밀릴 때도 어떤 항목을 먼저 줄여야 하는지 판단이 빨라집니다.

## 실무 부록: 팀 운영 템플릿 묶음

아래 부록은 장별 주제를 넘어서 캡스톤 전 구간에서 공통으로 재사용할 수 있는 운영 템플릿입니다. 팀이 매주 다른 양식으로 기록하면 비교가 어려워지고, 발표 직전에 근거를 다시 찾느라 시간이 낭비됩니다. 따라서 최소 템플릿을 고정하고 같은 구조로 반복 기록하는 편이 안정적입니다.

### 주간 운영 로그 템플릿

| 구분 | 기록 항목 | 작성 기준 |
| --- | --- | --- |
| 목표 | 이번 주 반드시 완료할 3개 | 측정 가능한 문장으로 작성 |
| 결과 | 완료/미완료 | 근거 링크와 함께 기록 |
| 편차 | 계획 대비 차이 | 원인 1문장, 조치 1문장 |
| 품질 | 테스트 통과율, 버그 건수 | 숫자로 기록 |
| 위험 | 다음 주 리스크 | 담당자와 대응 일정 포함 |

### 제안서-요구사항-데모 연결 규칙

1. 제안서의 문제 문장은 요구사항 문서 첫 줄과 동일한 의미여야 합니다.
2. 요구사항 ID는 테스트 케이스 ID와 1:1로 연결합니다.
3. 데모 스크립트 장면 번호를 요구사항 추적표에 함께 기록합니다.
4. 발표 슬라이드의 지표 숫자는 테스트 리포트 숫자와 동일해야 합니다.

이 규칙을 지키면 "문서는 맞는데 시연이 안 맞는" 상태를 줄일 수 있습니다. 캡스톤 평가에서는 기능의 화려함보다 논리 연결성이 더 강한 설득력을 만듭니다.

### 코드 리뷰 체크리스트(공통)

- 변경 목적이 현재 스프린트 목표와 맞는지 확인합니다.
- 테스트가 없는 기능 추가는 기본적으로 보류합니다.
- 예외 처리와 오류 메시지가 데모 시나리오를 방해하지 않는지 점검합니다.
- 설정 파일 변경은 롤백 방법을 함께 기록합니다.
- 문서 변경 없이 동작만 바뀌는 PR은 승인하지 않습니다.

### CI/CD 파이프라인 확인표

| 단계 | 필수 체크 | 실패 시 조치 |
| --- | --- | --- |
| 의존성 설치 | 버전 고정 여부 | 잠금 파일 재생성 |
| 테스트 | 핵심 테스트 통과 | 실패 케이스 원인 분류 |
| 빌드 | 아티팩트 생성 성공 | 빌드 로그 첨부 후 재시도 |
| 배포 | 스테이징 헬스체크 통과 | 즉시 이전 버전 롤백 |
| 검증 | 데모 핵심 경로 점검 | 배포 승인 보류 |

### 발표 직전 48시간 운영 규칙

첫째, 새 기능 추가를 중단합니다. 둘째, 리허설 스크립트를 고정하고 시간 측정을 반복합니다. 셋째, 데모 실패 대비 백업 영상을 준비합니다. 넷째, Q&A 예상 질문을 기술/범위/일정/검증 축으로 정리합니다. 다섯째, 발표 당일 역할(발표자, 조작자, 시간 관리자)을 변경하지 않습니다.

이 다섯 가지 규칙은 발표 직전 혼선을 줄이는 데 효과적입니다. 실제로 많은 팀이 이 시점에 기능 욕심을 다시 내다가 핵심 흐름을 망칩니다. 캡스톤의 완성도는 마지막 이틀의 선택에서 크게 갈립니다.

### 데모 스크립트 표준안

```text
오프닝: 사용자 문제와 목표 지표를 20초 안에 제시
시연 1: 입력 단계와 검증 메시지 확인
시연 2: 핵심 처리 과정과 결과 화면 연결
시연 3: 실패 케이스 1건을 짧게 제시
클로징: 한계, 다음 확장, 회고 한 줄
```

스크립트를 표준안으로 고정하면 발표자가 바뀌어도 전달력이 유지됩니다. 또한 회고 단계에서 어떤 장면이 실제 질문을 많이 유도했는지 비교 분석할 수 있습니다.

### 추가 보강: 발표 심사 기준 대응 메모

심사 관점에서는 기능 개수보다 문제 적합성, 구현 완성도, 검증 근거, 협업 과정, 전달 명확성이 함께 평가됩니다. 따라서 팀은 주간 문서에 이 다섯 축을 고정 항목으로 기록해야 합니다. 기능 진척이 느린 주차라도 검증 근거와 리스크 대응이 선명하면 전체 평가의 신뢰도를 유지할 수 있습니다.

- 문제 적합성: 사용자 문제와 기능 연결이 유지되는가
- 구현 완성도: 핵심 흐름이 끊기지 않는가
- 검증 근거: 테스트/로그/시연 결과가 일치하는가
- 협업 과정: 역할 분담과 리뷰 기록이 남아 있는가
- 전달 명확성: 슬라이드와 데모가 같은 메시지를 말하는가

## 처음 질문으로 돌아가기

- **좋은 캡스톤 주제는 어떤 조건을 갖춰야 할까요?**
  - 이 글은 주제 선정을 단순 정의가 아니라, 실무에서 마주하는 구체적 상황과 판단 과정으로 풀어냈습니다. 각 섹션의 예시와 체크리스트를 따라가며 자신의 상황에 적용할 수 있도록 구성했습니다.
- **흥미로운 아이디어와 실제로 해낼 수 있는 아이디어는 어떻게 구분할까요?**
  - 본문에서 제시한 예시 코드나 매트릭스를 참고하면, 좋은 판단 기준이 무엇인지 구체적으로 느낄 수 있습니다. 숫자나 기준이 아니라 '어떤 상황에서 이런 기준을 세우는가'를 이해하는 것이 중요합니다.
- **후보를 비교할 때 팀이 함께 볼 기준은 무엇일까요?**
  - 이 글의 핵심 메시지는 평가 기준이 무엇이든, 한 번의 판단으로 끝나지 않는다는 점입니다. 초반에 정한 기준을 문서로 남기고, 나중에 돌아봤을 때도 같은 실수를 반복하지 않도록 하는 과정 자체가 프로젝트를 성공으로 이끕니다.
