# Capstone Project 101 (10/10): 프로젝트 회고

이 글은 캡스톤 프로젝트 101 시리즈의 10번째 글입니다.

프로젝트가 끝나면 팀은 보통 안도감부터 느낍니다. 그 순간 바로 흩어지면 이번 학기의 시행착오는 다시 개인 기억으로만 남습니다.

회고의 목적은 감정을 털어놓는 데서 멈추지 않고, 사실과 원인과 다음 행동을 문서로 남기는 데 있습니다. 그래야 이번 프로젝트의 실패와 성공이 다음 프로젝트의 입력이 됩니다.

이 글은 Capstone Project 101 시리즈의 마지막 글입니다. 여기서는 KPT, 데이터, 원인 분석, 다음 행동을 묶어 회고를 학습 문서로 남기는 방법을 정리합니다.

> 멘탈 모델: 좋은 회고는 잘못한 사람을 찾는 문서가 아니라, 어떤 사실이 있었고 무엇을 바꿀지 정리하는 다음 프로젝트의 출발 문서입니다.

## 왜 중요한가

프로젝트는 끝났는데 학습이 남지 않으면 다음 프로젝트도 같은 패턴을 반복하기 쉽습니다. 반대로 잘한 점, 문제, 시도할 점을 구조적으로 남기면 이번 경험이 다음 팀 작업의 출발점이 됩니다.

실무 팀도 스프린트 회고와 포스트모템을 꾸준히 운영합니다. 중요한 것은 감정을 털어놓는 것보다, 어떤 사실이 있었고 다음에 무엇을 바꿀지를 작게라도 정하는 일입니다.

## 실전 문서 예시: 회고 액션 로그

회고가 실제 변화로 이어지려면 다음 행동을 아래처럼 작고 선명하게 적는 편이 좋습니다.

```text
구분 | 내용 | 담당자 | 기한
Keep | 요구사항 변경 기록을 남긴다 | 팀 리드 | 다음 프로젝트 시작 전
Problem | 데모 리허설이 부족해 발표 직전 오류를 발견했다 | 팀 전체 | 회고 시점 기록
Try | 매주 금요일 20분 리허설 슬롯을 고정한다 | QA 담당 | 다음 프로젝트 1주차
Action | 배포 체크리스트 템플릿을 저장소에 추가한다 | 백엔드 담당 | 다음 프로젝트 1주차
```

## 이 문서로 먼저 확인할 것

- 잘한 점, 문제, 시도를 구분해서 적는지 확인합니다.
- 데이터가 없는 주장에는 근거를 더 모읍니다.
- 다음 행동에 담당자와 기한이 있는지 봅니다.
- 다음 프로젝트에서 다시 열어볼 수 있도록 문서를 남깁니다.

## 핵심 용어

- **KPT**: Keep, Problem, Try 세 칸으로 정리하는 회고 방식입니다.
- **5 Whys**: 원인을 다섯 번 정도 연쇄적으로 묻는 방식입니다.
- **action**: 다음에 실제로 실행할 한 단계입니다.
- **data**: 수치로 남긴 근거입니다.
- **learning**: 문서로 남긴 교훈입니다.

## 이전과 이후

**Before**: 감정만 쏟아 놓고 끝냅니다.

**After**: 사실과 다음 행동을 함께 기록합니다.

## 실습: 회고 표

### 1단계 — KPT

```python
kpt = {"keep": [], "problem": [], "try": []}
```

KPT는 구조를 단순하게 유지해 줍니다. 초보 팀일수록 복잡한 형식보다 이 정도가 실용적입니다.

### 2단계 — 데이터

```python
metrics = {"velocity": 12, "bugs": 5, "review_time": 1.5}
```

데이터는 느낌보다 수치로 회고를 붙잡아 줍니다. 속도, 버그 수, 리뷰 시간 같은 지표는 논의를 훨씬 차분하게 만듭니다.

### 3단계 — 다섯 번 왜 묻기

```python
whys = ["bug_at_demo", "missed_test", "no_ci", "no_template", "first_time"]
```

Five Whys는 누가 잘못했는지보다 왜 이런 조건이 반복됐는지를 보는 데 도움이 됩니다. 원인을 시스템 관점으로 옮겨 주는 도구입니다.

### 4단계 — 다음 행동

```python
actions = [{"who": "A", "what": "add_ci", "by": "next_sprint"}]
```

다음 행동은 담당자와 마감이 함께 있어야 실행으로 이어집니다. 의지가 아니라 구조가 필요합니다.

### 5단계 — 학습 요약

```python
lessons = ["scope_first", "ci_early", "demo_dryrun"]
```

학습 요약은 다음 프로젝트에 그대로 들고 갈 수 있는 문장이어야 합니다. 짧더라도 재사용 가능해야 합니다.

## 이 코드에서 먼저 볼 점

- KPT는 세 칸 구조입니다.
- 데이터는 숫자로 남깁니다.
- 다음 행동에는 담당자와 마감이 있습니다.
- 학습은 재사용 가능한 문장으로 남깁니다.

## 자주 하는 실수 5가지

1. 누가 잘못했는지부터 묻습니다.
2. 감정만 기록합니다.
3. 다음 행동이 없습니다.
4. 데이터가 없습니다.
5. 다음 프로젝트와 연결하지 않습니다.

## 실무에서는 이렇게 이어집니다

실무 팀도 스프린트 회고와 포스트모템을 통해 다음 행동을 정합니다. 좋은 회고는 칭찬이나 비판에서 끝나지 않고, 작은 프로세스 개선으로 이어집니다. 캡스톤에서 이 감각을 익히면 프로젝트를 닫는 방식 자체가 달라집니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 사실에서 시작합니다.
- 원인은 개인보다 시스템에서 찾습니다.
- 다음 행동은 작게 쪼갭니다.
- 책임은 팀이 함께 집니다.
- 학습은 반드시 문서화합니다.

## 정리와 다음 글

회고는 프로젝트의 마지막 문서이면서 다음 프로젝트의 첫 문서입니다. KPT, 데이터, 원인 분석, 다음 행동을 함께 남기면 이번 캡스톤의 시행착오가 다음 작업의 자산으로 바뀝니다. 이것으로 Capstone Project 101 시리즈를 마칩니다.

## 실전 확장: KPT 회고 프레임워크와 개인 성장 기록 양식

회고 문서가 팀의 학습 자산이 되려면 팀 회고와 개인 회고를 분리해 기록하는 것이 효과적입니다. 팀 회고는 시스템과 프로세스 개선에 집중하고, 개인 회고는 역량 성장과 다음 학습 계획에 집중해야 합니다. 둘을 섞으면 책임 공방이 생기거나 실행 항목이 흐려지기 쉽습니다.

### KPT 회고 프레임워크 확장형

| 구분 | 질문 | 기록 예시 | 다음 액션 연결 |
| --- | --- | --- | --- |
| Keep | 무엇이 잘 작동했는가 | 요구사항 변경 로그 운영이 일정 충격을 줄임 | 동일 템플릿 재사용 |
| Problem | 반복된 문제는 무엇인가 | 데모 직전 통합 테스트 실패가 다수 발생 | 테스트 주기 앞당김 |
| Try | 다음에 시도할 실험은 무엇인가 | 매주 리허설 슬롯 고정 운영 | 캘린더 선점 |
| Learn | 이번에 새롭게 배운 원칙은 무엇인가 | 범위 제외 문서화가 품질을 높임 | MVP 문서 표준 반영 |
| Stop | 중단할 습관은 무엇인가 | 회의 중 결정을 구두로만 합의 | 결정 로그 의무화 |

KPT를 확장할 때는 항목을 늘리는 것보다 실행 항목과의 연결을 강화하는 것이 중요합니다. 각 항목 끝에 "다음 행동"이 없으면 회고는 기록으로만 끝납니다.

### 개인 성장 기록 양식

```text
[기본 정보]
이름:
역할:
기간:

[성과]
이번 프로젝트에서 가장 기여한 결과 2개:

[기술 성장]
새롭게 익힌 기술/도구:
익힌 방식(문서, 구현, 리뷰 등):
다음 프로젝트에서 재사용할 항목:

[협업 성장]
의사소통에서 잘한 점:
갈등 상황에서 배운 점:
다음에 개선할 협업 습관:

[실패와 교훈]
가장 큰 실수 1개:
원인:
재발 방지 행동:

[다음 4주 계획]
학습 목표 1:
실행 계획:
측정 기준:
```

### 팀 회고와 개인 회고 연결 규칙

- 팀 Problem 항목과 개인 실패 항목을 교차 검토합니다.
- 팀 Try 항목 중 개인 담당 액션을 명확히 지정합니다.
- 개인 학습 계획이 팀의 다음 프로젝트 준비와 연결되는지 확인합니다.
- 회고 문서는 저장소에 버전으로 남겨 재참조 가능하게 합니다.
- 다음 프로젝트 킥오프에서 지난 회고 핵심 5문장을 먼저 읽습니다.

### 액션 아이템 품질 기준

| 기준 | 나쁜 예시 | 좋은 예시 |
| --- | --- | --- |
| 구체성 | 테스트를 열심히 한다 | 매주 화요일 통합 테스트 10케이스 실행 |
| 책임성 | 팀이 확인한다 | QA 담당이 결과를 이슈로 보고 |
| 기한성 | 다음에 반영한다 | 다음 프로젝트 2주차까지 반영 |
| 측정성 | 품질을 높인다 | 결함 재오픈율 20% 이하 유지 |

### 회고 후 48시간 규칙

회고가 끝난 뒤 48시간 안에 액션 아이템을 실제 작업 도구로 옮기지 않으면 실행 확률이 급격히 낮아집니다. 따라서 회고 회의 종료 직후 다음 세 가지를 수행하는 규칙을 권장합니다.

1. 액션 아이템을 이슈로 등록합니다.
2. 담당자와 기한을 확정합니다.
3. 다음 점검 일정을 캘린더에 선점합니다.

프로젝트 회고의 품질은 문장의 아름다움이 아니라 다음 행동의 실행률로 판단됩니다. KPT와 개인 성장 기록을 함께 운용하면 캡스톤이 단발성 과제가 아니라 장기적인 성장 자산으로 전환됩니다.

## 심화 보강: 의사결정 근거 문서화 패턴

캡스톤 문서의 분량을 늘리는 목적은 글자 수 자체가 아니라 판단의 흔적을 남기는 데 있습니다. 같은 결과물을 만들더라도 판단 근거가 기록된 팀과 기록되지 않은 팀의 학습 밀도는 크게 다릅니다. 아래 패턴은 어떤 장에서든 공통으로 적용할 수 있는 최소 문서화 규칙입니다.

### 의사결정 카드 템플릿

| 항목 | 작성 기준 | 예시 |
| --- | --- | --- |
| 결정 제목 | 동사형으로 짧게 작성 | 입력 형식을 CSV로 고정한다 |
| 배경 | 현재 문제와 제약을 3문장 내 서술 | 발표 전 2주 내 구현 안정화 필요 |
| 선택지 | 최소 2개 이상 비교 | CSV, 엑셀 직접 업로드 |
| 평가 기준 | 시간, 위험, 구현 난도 명시 | 학습 비용, 오류 가능성 |
| 최종 결정 | 한 문장으로 고정 | CSV 입력을 MVP 범위로 채택 |
| 후속 작업 | 실행 가능한 액션으로 변환 | 파서 검증 테스트 5건 추가 |

### 리스크 등록부 예시

| 리스크 ID | 설명 | 발생 가능성 | 영향도 | 대응 전략 | 담당 |
| --- | --- | --- | --- | --- | --- |
| RK-01 | 외부 데이터 접근 지연 | 중간 | 높음 | 샘플 데이터 대체 경로 준비 | 데이터 담당 |
| RK-02 | 데모 환경 네트워크 불안정 | 중간 | 높음 | 오프라인 영상 백업 준비 | 발표 담당 |
| RK-03 | 통합 테스트 누락 | 높음 | 중간 | 주간 회귀 테스트 슬롯 고정 | QA 담당 |
| RK-04 | 요구사항 빈번한 변경 | 중간 | 중간 | 변경 승인 규칙 명시 | 팀 리드 |

리스크 등록부는 불안을 기록하는 문서가 아니라 우선순위를 결정하는 문서입니다. 발생 가능성과 영향도를 함께 보면 어떤 리스크를 먼저 줄여야 하는지 명확해집니다.

### 주간 리뷰 질문

- 이번 주 결정 중 다음 주에 다시 검토할 항목은 무엇인가
- 일정 편차를 만든 원인은 기술 이슈인가, 의사결정 지연인가
- 범위 확장 요청이 들어왔을 때 거절 기준이 문서에 있는가
- 테스트 실패가 특정 구간에 집중되는가
- 발표 시점에 그대로 설명 가능한 근거가 충분한가

### 품질을 높이는 기록 원칙

첫째, 문장은 짧고 검증 가능하게 씁니다. 둘째, 수치가 가능한 항목은 반드시 수치로 남깁니다. 셋째, 결정과 액션을 분리하지 않습니다. 넷째, 회의 직후 24시간 내 문서를 갱신합니다. 다섯째, 변경 이유를 생략하지 않습니다. 이 다섯 가지 원칙만 지켜도 캡스톤 문서는 단순 제출물이 아니라 다음 프로젝트를 위한 운영 매뉴얼로 전환됩니다.

### 적용 체크리스트

- [ ] 결정 카드가 최신 상태로 유지됩니다.
- [ ] 리스크 등록부에 담당자와 대응 전략이 있습니다.
- [ ] 주간 리뷰 질문에 대한 답변이 기록됩니다.
- [ ] 변경 로그가 날짜 순으로 정리됩니다.
- [ ] 발표와 회고 문서가 같은 근거를 참조합니다.

위 보강 블록은 각 장의 주제와 별개로 공통 적용되는 운영 원칙입니다. 캡스톤의 난도는 기술 난도보다 의사결정 난도에서 크게 올라갑니다. 따라서 문서화 패턴을 반복 적용하면 팀의 판단 체계가 안정되고, 일정과 품질, 발표 전달력이 동시에 개선됩니다.

### 추가 보강: 발표 직전 최종 점검 루틴

발표 직전에는 새 기능 추가를 중단하고 안정화 루틴을 고정합니다. 최종 점검 루틴은 기술 완성도를 높이는 절차가 아니라 실패 확률을 낮추는 절차입니다. 핵심 흐름 시연, 오류 메시지 점검, 백업 자료 확인, 예상 질문 재점검, 역할별 멘트 정렬을 반복하면 전달 품질이 크게 향상됩니다.

| 점검 항목 | 확인 내용 | 통과 기준 |
| --- | --- | --- |
| 핵심 시연 | 시작부터 결과 화면까지 흐름 유지 | 3회 연속 성공 |
| 오류 처리 | 비정상 입력 시 안내 문구 노출 | 사용자 행동 유도 가능 |
| 백업 자료 | 네트워크 실패 대비 영상/스크린샷 | 즉시 전환 가능 |
| Q&A 근거 | 기술/범위/테스트 답변 문서화 | 질문 10개 대응 가능 |
| 역할 분담 | 발표자, 데모 조작자, 시간 관리자 확정 | 리허설 동일 운영 |

이 루틴을 문서로 남기면 회고 단계에서 무엇이 실제로 효과가 있었는지 비교할 수 있습니다.

## 실전 앵커: 회고 체크리스트와 다음 프로젝트 이관 문서

회고의 품질은 감상문 길이가 아니라 다음 프로젝트에 재사용 가능한 규칙을 얼마나 남겼는지로 결정됩니다. 따라서 회고 문서는 "좋았다/아쉬웠다"에서 끝내지 말고 실행 항목으로 변환해야 합니다.

### 회고 체크리스트

- 제안서에서 세운 가정 중 검증된 항목과 실패한 항목을 분리합니다.
- 일정 지연의 원인을 기술 이슈와 의사결정 지연으로 구분합니다.
- Git 브랜치 전략이 실제로 지켜졌는지 로그로 확인합니다.
- CI/CD 파이프라인에서 반복 실패한 단계의 원인을 기록합니다.
- 발표 슬라이드와 데모 대본의 개선 포인트를 다음 템플릿에 반영합니다.

### 다음 프로젝트 이관 문서 템플릿

| 항목 | 이번 프로젝트 교훈 | 다음 프로젝트 적용 규칙 |
| --- | --- | --- |
| 문제 정의 | 사용자 범위를 넓게 잡아 초반 흔들림 발생 | 첫 주에 제외 사용자까지 문서화 |
| 일정 관리 | 테스트 버퍼 부족 | 매주 0.5일 회귀 테스트 고정 |
| 협업 방식 | PR 리뷰 기준 불일치 | 코드 리뷰 체크리스트 고정 |
| 발표 준비 | 리허설 횟수 부족 | 발표 2주 전부터 주 2회 리허설 |

회고 문서가 이 수준으로 남으면 캡스톤이 단발성 과제가 아니라 팀의 운영 자산으로 전환됩니다. 이것이 마지막 장에서 반드시 확보해야 할 결과입니다.

## 처음 질문으로 돌아가기

- **회고가 책임 공방으로 흐르지 않게 하려면 무엇이 필요할까요?**
  - 이 글은 프로젝트 회고을 단순 정의가 아니라, 실무에서 마주하는 구체적 상황과 판단 과정으로 풀어냈습니다. 각 섹션의 예시와 체크리스트를 따라가며 자신의 상황에 적용할 수 있도록 구성했습니다.
- **KPT 형식은 왜 초보 팀에게 특히 유용할까요?**
  - 본문에서 제시한 예시 코드나 매트릭스를 참고하면, 좋은 판단 기준이 무엇인지 구체적으로 느낄 수 있습니다. 숫자나 기준이 아니라 '어떤 상황에서 이런 기준을 세우는가'를 이해하는 것이 중요합니다.
- **데이터는 회고의 어떤 부분을 단단하게 만들어 줄까요?**
  - 이 글의 핵심 메시지는 평가 기준이 무엇이든, 한 번의 판단으로 끝나지 않는다는 점입니다. 초반에 정한 기준을 문서로 남기고, 나중에 돌아봤을 때도 같은 실수를 반복하지 않도록 하는 과정 자체가 프로젝트를 성공으로 이끕니다.
