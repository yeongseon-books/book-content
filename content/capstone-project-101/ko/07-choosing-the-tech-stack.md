# Capstone Project 101 (7/10): 기술 스택 선택

이 글은 캡스톤 프로젝트 101 시리즈의 7번째 글입니다.

기술 스택 선택이 어렵다는 말에는 보통 두 가지가 섞여 있습니다. 새로운 기술을 써 보고 싶은 마음과, 학기 안에 반드시 끝내야 한다는 현실입니다.

캡스톤에서는 이 둘이 자주 충돌합니다. 그래서 좋은 기술보다 지금 우리 팀이 무리 없이 다룰 수 있는 기술을 먼저 정의해야 합니다.

이 글은 Capstone Project 101 시리즈의 7번째 글입니다. 여기서는 친숙도, 학습 비용, 운영 부담, 문서 품질을 함께 보며 현실적인 기술 선택 기준을 정리합니다.

> 멘탈 모델: 좋은 기술 스택은 가장 화려한 조합이 아니라, 지금 팀이 일정 안에 배우고 구현하고 운영할 수 있는 조합입니다.

> 기술 스택 선택에서 중요한 것은 기능 이름이 아니라, 각 단계에서 '누가, 언제, 무엇으로' 의사결정을 내리는가입니다.

## 먼저 던지는 질문

- 왜 가장 새로운 스택이 항상 정답은 아닐까요?
- 팀의 친숙도는 왜 일정과 직접 연결될까요?
- 학습 비용과 운영 부담은 어떻게 따져야 할까요?

## 이 글에서 배우는 내용

- 친숙도 평가법
- 학습 곡선 비교
- 생태계 확인 포인트
- 운영 비용 감각
- 대안 비교 방식

## 왜 중요한가

캡스톤의 일정은 짧습니다. 그래서 새 기술을 익히는 시간, 디버깅 시간, 배포와 운영 시간을 모두 일정 안에 포함해서 봐야 합니다. 구현보다 도구 학습에 시간이 더 많이 들어가면 프로젝트는 빠르게 흔들립니다.

좋은 선택은 집중을 만듭니다. 반대로 과하게 욕심낸 선택은 학습 비용과 운영 부담이 일정 전체를 잠식하게 만듭니다.

## 실전 문서 예시: 간단한 ADR

기술 선택은 말로만 합의하면 나중에 다시 흔들리기 쉽습니다. 아래처럼 한 장짜리 ADR을 남기면 변경 판단이 빨라집니다.

```text
제목: 백엔드 프레임워크로 Flask를 선택한다
문맥: 팀원 4명 중 3명이 Flask 경험이 있고, 6주 안에 데모를 만들어야 한다
대안: FastAPI, Django
선택 이유: 학습 비용이 가장 낮고, 배포 절차가 단순하며, 문서가 충분하다
감수할 단점: 자동 문서화와 구조화 기능은 FastAPI보다 약하다
재검토 조건: 인증, 비동기 API, 관리자 기능 범위가 크게 늘어날 때
```

## 이 문서로 먼저 확인할 것

- 친숙도와 학습 비용을 분리해서 적었는지 확인합니다.
- 운영 부담에 배포와 장애 대응이 포함되어 있는지 봅니다.
- 대안이 두세 개 수준으로 정리되어 있는지 점검합니다.
- 재검토 조건이 있으면 나중에 감정적 논쟁을 줄일 수 있습니다.

## 핵심 용어

- **familiarity**: 팀이 이미 갖고 있는 경험치입니다.
- **learning curve**: 새 기술을 익히는 데 드는 비용입니다.
- **ecosystem**: 라이브러리, 문서, 커뮤니티를 포함한 주변 환경입니다.
- **ops**: 배포하고 운영하는 데 드는 부담입니다.
- **alternative**: 비교를 위해 남겨 둔 대안입니다.

## 이전과 이후

**Before**: 최신 스택이면 무조건 좋다고 생각합니다.

**After**: 친숙도와 비용을 함께 보고 선택합니다.

## 실습: 결정 표

### 1단계 — 후보 정리

```python
candidates = ["FastAPI", "Flask", "Django"]
```

후보는 너무 많이 늘리지 말고 비교 가능한 범위로 줄이는 편이 좋습니다.

### 2단계 — 친숙도 평가

```python
familiar = {"FastAPI": 4, "Flask": 5, "Django": 2}
```

친숙도는 단순 취향이 아니라 속도와 리스크에 직접 연결됩니다. 익숙한 도구는 구현뿐 아니라 디버깅도 빠릅니다.

### 3단계 — 학습 비용

```python
learning_cost = {"FastAPI": 2, "Flask": 1, "Django": 4}
```

새 기술은 배우는 재미가 있지만, 일정 안에서는 비용이기도 합니다. 캡스톤에서는 그 비용을 솔직하게 적는 편이 좋습니다.

### 4단계 — 운영 부담

```python
ops = {"FastAPI": 2, "Flask": 1, "Django": 3}
```

운영 부담은 배포, 설정, 모니터링, 오류 대응까지 포함합니다. 구현이 쉬워도 운영이 무거우면 전체 부담은 커질 수 있습니다.

### 5단계 — 점수 계산

```python
score = {k: familiar[k] - learning_cost[k] - ops[k] for k in candidates}
```

복잡한 모델보다 이런 단순한 계산이 오히려 팀 합의에 더 도움이 되는 경우가 많습니다. 중요한 것은 점수 자체보다 판단 기준을 공유하는 일입니다.

## 이 코드에서 먼저 볼 점

- 점수는 친숙도에서 비용을 빼는 구조입니다.
- 대안은 셋 이하로 유지하는 편이 좋습니다.
- 결정은 반드시 문서로 남겨야 합니다.
- 친숙도와 운영 부담은 동시에 봐야 합니다.

## 자주 하는 실수 5가지

1. 인기만 보고 선택합니다.
2. 팀의 친숙도를 무시합니다.
3. 운영 비용을 뒤늦게 발견합니다.
4. 결정 기록을 남기지 않습니다.
5. 대안 비교 없이 첫 아이디어를 확정합니다.

## 실무에서는 이렇게 이어집니다

실무 팀도 ADR 같은 문서에 결정 이유를 남깁니다. 특히 일정이 짧은 프로젝트에서는 왜 이 기술을 골랐는지를 짧게라도 기록해 두어야 범위 조정이나 기술 변경이 필요할 때 빠르게 판단할 수 있습니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 친숙도는 곧 속도입니다.
- 학습은 비용입니다.
- 운영 부담은 계속 이어집니다.
- 대안은 문서로 비교합니다.
- 결정은 나중에 되돌릴 수 있게 남깁니다.

## 체크리스트

- [ ] 학습 비용이 적혀 있습니다.
- [ ] 결정 기록이 있습니다.

## 정리와 다음 글

기술 스택 선택은 취향 경쟁이 아니라 리스크 조정입니다. 친숙도, 학습 비용, 운영 부담을 함께 보고 기록까지 남기면 기술 결정이 프로젝트를 돕는 방향으로 움직입니다. 다음 글에서는 이 선택을 일정으로 어떻게 관리할지 다룹니다.

## 실전 확장: 구현 체크리스트와 코드 품질 기준 표

기술 스택을 선택한 뒤에는 구현 단계의 품질 기준을 즉시 정의해야 합니다. 선택만 하고 기준이 없으면 팀원마다 완성의 정의가 달라지고, 리뷰 단계에서 충돌이 반복됩니다. 따라서 "어떤 코드를 합격으로 볼 것인가"를 간단한 체크리스트와 표로 고정해 두는 것이 좋습니다.

### 구현 체크리스트

- [ ] 핵심 기능마다 입력 검증이 존재합니다.
- [ ] 예외 상황에서 사용자 메시지가 명확합니다.
- [ ] 핵심 로직은 함수 단위로 분리되어 있습니다.
- [ ] 외부 의존성 호출에 타임아웃이 설정되어 있습니다.
- [ ] 로그에 요청 ID 또는 추적 키가 포함됩니다.
- [ ] 설정 값이 코드 하드코딩이 아니라 환경 변수로 분리됩니다.
- [ ] README에 실행 방법과 테스트 방법이 최신 상태로 반영됩니다.

체크리스트는 강한 기술 규칙이 아니라 팀의 최소 합의입니다. 모든 항목을 높은 수준으로 구현하기 어려운 경우에도, 최소선이 있으면 코드 품질이 급격히 무너지는 것을 막을 수 있습니다.

### 코드 품질 기준 표

| 항목 | 기준 | 측정 방법 | 실패 시 조치 |
| --- | --- | --- | --- |
| 가독성 | 함수가 한 가지 책임만 수행 | 코드 리뷰 | 함수 분리 |
| 오류 처리 | 예외가 사용자 메시지로 연결 | 시나리오 테스트 | 오류 핸들러 추가 |
| 테스트 | Must 기능 단위 테스트 존재 | 테스트 리포트 | 테스트 케이스 작성 |
| 성능 | 핵심 API 평균 응답 1초 이내 | 간단 부하 측정 | 쿼리/로직 최적화 |
| 일관성 | 포맷터/린터 규칙 준수 | 자동 검사 | 포맷/수정 |
| 변경 안정성 | 수정 시 회귀 테스트 통과 | CI 또는 수동 체크리스트 | 원인 분석 후 재수정 |

### 리뷰 프로세스 예시

```text
1) 작성자: 변경 목적과 영향 범위를 PR 설명에 기록
2) 리뷰어: 체크리스트 항목 기준으로 리뷰
3) 팀 리드: 우선순위와 일정 영향 확인
4) QA: 핵심 시나리오 회귀 여부 확인
5) 머지: 테스트 통과와 문서 업데이트 확인 후 진행
```

### 품질 기준을 지키기 위한 운영 팁

- 신규 기능보다 결함 수정 시간을 일정에 기본 포함합니다.
- 코드 리뷰는 스타일보다 위험도 높은 항목부터 확인합니다.
- 테스트가 없는 코드 경로를 목록으로 관리합니다.
- 데모 직전 주에는 구조 변경보다 안정화에 집중합니다.
- 품질 기준을 지키지 못한 경우 이유와 보완 계획을 기록합니다.

### 경량 품질 대시보드 예시

| 주차 | 신규 기능 수 | 결함 수 | 테스트 통과율 | 주요 리스크 |
| --- | --- | --- | --- | --- |
| 2주차 | 4 | 6 | 70% | 입력 검증 누락 |
| 3주차 | 3 | 5 | 78% | API 응답 지연 |
| 4주차 | 2 | 3 | 88% | 데모 시나리오 불안정 |
| 5주차 | 1 | 1 | 95% | 낮음 |

이 표는 완벽한 관측 체계가 없어도 팀 상태를 빠르게 공유하게 도와줍니다. 기술 스택 선택의 품질은 결국 구현 안정성으로 증명됩니다. 따라서 도구 선택 직후 품질 기준을 명확히 두는 것이 캡스톤 완주 확률을 크게 높입니다.

## 심화 보강: 의사결정 근거 문서화 패턴

캡스톤 문서의 분량을 늘리는 목적은 글자 수 자체가 아니라 판단의 흔적을 남기는 데 있습니다. 같은 결과물을 만들더라도 판단 근거가 기록된 팀과 기록되지 않은 팀의 학습 밀도는 크게 다릅니다. 아래 패턴은 어떤 장에서든 공통으로 적용할 수 있는 최소 문서화 규칙입니다.

### 의사결정 카드 템플릿

| 항목 | 작성 기준 | 예시 |
| --- | --- | --- |
| 결정 제목 | 동사형으로 짧게 작성 | 입력 형식을 CSV로 고정한다 |
| 배경 | 현재 문제와 제약을 3문장 내 서술 | 발표 전 2주 내 구현 안정화 필요 |
| 선택지 | 최소 2개 이상 비교 | CSV, 엑셀 직접 업로드 |
| 평가 기준 | 시간, 위험, 구현 난도 명시 | 학습 비용, 오류 가능성 |
| 최종 결정 | 한 문장으로 고정 | CSV 입력을 MVP 범위로 채택 |
| 후속 작업 | 실행 가능한 액션으로 변환 | 파서 검증 테스트 5건 추가 |

### 리스크 등록부 예시

| 리스크 ID | 설명 | 발생 가능성 | 영향도 | 대응 전략 | 담당 |
| --- | --- | --- | --- | --- | --- |
| RK-01 | 외부 데이터 접근 지연 | 중간 | 높음 | 샘플 데이터 대체 경로 준비 | 데이터 담당 |
| RK-02 | 데모 환경 네트워크 불안정 | 중간 | 높음 | 오프라인 영상 백업 준비 | 발표 담당 |
| RK-03 | 통합 테스트 누락 | 높음 | 중간 | 주간 회귀 테스트 슬롯 고정 | QA 담당 |
| RK-04 | 요구사항 빈번한 변경 | 중간 | 중간 | 변경 승인 규칙 명시 | 팀 리드 |

리스크 등록부는 불안을 기록하는 문서가 아니라 우선순위를 결정하는 문서입니다. 발생 가능성과 영향도를 함께 보면 어떤 리스크를 먼저 줄여야 하는지 명확해집니다.

### 주간 리뷰 질문

- 이번 주 결정 중 다음 주에 다시 검토할 항목은 무엇인가
- 일정 편차를 만든 원인은 기술 이슈인가, 의사결정 지연인가
- 범위 확장 요청이 들어왔을 때 거절 기준이 문서에 있는가
- 테스트 실패가 특정 구간에 집중되는가
- 발표 시점에 그대로 설명 가능한 근거가 충분한가

### 품질을 높이는 기록 원칙

첫째, 문장은 짧고 검증 가능하게 씁니다. 둘째, 수치가 가능한 항목은 반드시 수치로 남깁니다. 셋째, 결정과 액션을 분리하지 않습니다. 넷째, 회의 직후 24시간 내 문서를 갱신합니다. 다섯째, 변경 이유를 생략하지 않습니다. 이 다섯 가지 원칙만 지켜도 캡스톤 문서는 단순 제출물이 아니라 다음 프로젝트를 위한 운영 매뉴얼로 전환됩니다.

### 적용 체크리스트

- [ ] 결정 카드가 최신 상태로 유지됩니다.
- [ ] 리스크 등록부에 담당자와 대응 전략이 있습니다.
- [ ] 주간 리뷰 질문에 대한 답변이 기록됩니다.
- [ ] 변경 로그가 날짜 순으로 정리됩니다.
- [ ] 발표와 회고 문서가 같은 근거를 참조합니다.

위 보강 블록은 각 장의 주제와 별개로 공통 적용되는 운영 원칙입니다. 캡스톤의 난도는 기술 난도보다 의사결정 난도에서 크게 올라갑니다. 따라서 문서화 패턴을 반복 적용하면 팀의 판단 체계가 안정되고, 일정과 품질, 발표 전달력이 동시에 개선됩니다.

### 추가 보강: 발표 직전 최종 점검 루틴

발표 직전에는 새 기능 추가를 중단하고 안정화 루틴을 고정합니다. 최종 점검 루틴은 기술 완성도를 높이는 절차가 아니라 실패 확률을 낮추는 절차입니다. 핵심 흐름 시연, 오류 메시지 점검, 백업 자료 확인, 예상 질문 재점검, 역할별 멘트 정렬을 반복하면 전달 품질이 크게 향상됩니다.

| 점검 항목 | 확인 내용 | 통과 기준 |
| --- | --- | --- |
| 핵심 시연 | 시작부터 결과 화면까지 흐름 유지 | 3회 연속 성공 |
| 오류 처리 | 비정상 입력 시 안내 문구 노출 | 사용자 행동 유도 가능 |
| 백업 자료 | 네트워크 실패 대비 영상/스크린샷 | 즉시 전환 가능 |
| Q&A 근거 | 기술/범위/테스트 답변 문서화 | 질문 10개 대응 가능 |
| 역할 분담 | 발표자, 데모 조작자, 시간 관리자 확정 | 리허설 동일 운영 |

이 루틴을 문서로 남기면 회고 단계에서 무엇이 실제로 효과가 있었는지 비교할 수 있습니다.

### 보강 메모: 다음 프로젝트로 이관할 핵심 문장

이번 장의 핵심 판단 기준을 3문장으로 요약해 저장소 루트의 프로젝트 로그에 남깁니다. 다음 프로젝트 킥오프에서 이 문장을 먼저 읽으면 재시작 비용을 크게 줄일 수 있습니다. 문장 요약은 길이보다 재사용 가능성이 중요합니다.

## 실전 앵커: 기술 스택 결정표와 파이프라인 호환성 점검

기술 스택 선택은 선호도 투표가 아니라 운영 비용 계산입니다. 특히 캡스톤에서는 "배우고 싶은 기술"과 "학기 안에 안정적으로 시연 가능한 기술"을 분리해야 합니다.

### 기술 스택 결정표

| 영역 | 후보 | 장점 | 리스크 | 최종 선택 |
| --- | --- | --- | --- | --- |
| 백엔드 | FastAPI / Flask | 생산성 높음 | 팀 경험 편차 | FastAPI |
| 프론트엔드 | React / HTMX | 표현력 높음 | 초기 러닝커브 | React |
| 데이터 | SQLite / PostgreSQL | 설정 단순 | 확장 제약 | PostgreSQL |
| 배포 | Render / Railway | 설정 간단 | 무료 티어 제한 | Render |

### CI/CD 호환성 점검 항목

- 선택한 런타임 버전이 CI 환경과 동일한지 확인합니다.
- 빌드 아티팩트 생성 시간이 10분을 넘지 않는지 점검합니다.
- 시크릿 주입 방식이 로컬과 배포 환경에서 일관적인지 확인합니다.
- 롤백 절차를 문서로 기록하고 한 번 이상 실행해 봅니다.

스택 결정표와 호환성 점검표를 함께 운영하면 "기술은 좋아 보이는데 데모가 불안정한" 상황을 크게 줄일 수 있습니다.

## 처음 질문으로 돌아가기

- **왜 가장 새로운 스택이 항상 정답은 아닐까요?**
  - 이 글은 기술 스택 선택을 단순 정의가 아니라, 실무에서 마주하는 구체적 상황과 판단 과정으로 풀어냈습니다. 각 섹션의 예시와 체크리스트를 따라가며 자신의 상황에 적용할 수 있도록 구성했습니다.
- **팀의 친숙도는 왜 일정과 직접 연결될까요?**
  - 본문에서 제시한 예시 코드나 매트릭스를 참고하면, 좋은 판단 기준이 무엇인지 구체적으로 느낄 수 있습니다. 숫자나 기준이 아니라 '어떤 상황에서 이런 기준을 세우는가'를 이해하는 것이 중요합니다.
- **학습 비용과 운영 부담은 어떻게 따져야 할까요?**
  - 이 글의 핵심 메시지는 평가 기준이 무엇이든, 한 번의 판단으로 끝나지 않는다는 점입니다. 초반에 정한 기준을 문서로 남기고, 나중에 돌아봤을 때도 같은 실수를 반복하지 않도록 하는 과정 자체가 프로젝트를 성공으로 이끕니다.
