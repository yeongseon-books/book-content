# Capstone Project 101 (9/10): 발표 자료 만들기

발표 자료가 길어지는 가장 흔한 이유는, 팀이 만든 모든 기능을 다 보여 주려 하기 때문입니다. 하지만 청중은 보통 기능 개수보다 왜 이 프로젝트가 의미 있었는지부터 알고 싶어 합니다.

좋은 발표 자료는 기능 카탈로그가 아니라 선택과 결과를 설명하는 서사입니다. 무엇을 왜 만들었는지, 어떤 결과가 나왔는지, 다음에 무엇을 바꿀지까지 짧게 이어져야 설득력이 생깁니다.

이 글은 Capstone Project 101 시리즈의 9번째 글입니다. 여기서는 문제·해결·결과 구조로 슬라이드를 설계하고, 데모 실패 대비안과 Q&A까지 함께 준비하는 방법을 다룹니다.

> 멘탈 모델: 좋은 발표는 만든 기능을 다 보여 주는 자리가 아니라, 문제와 선택과 결과를 짧은 서사로 묶어 청중이 바로 이해하게 만드는 자리입니다.

*Capstone Project 101 9장 흐름 개요*
> 발표 자료 만들기에서 중요한 것은 기능 이름이 아니라, 각 단계에서 '누가, 언제, 무엇으로' 의사결정을 내리는가입니다.

## 먼저 던지는 질문

- 왜 기능 목록 중심 발표는 지루해질까요?
- 문제·해결·결과 구조는 왜 전달력이 높을까요?
- 슬라이드는 어떻게 구성해야 한 장당 메시지가 선명할까요?

## 이 글에서 배우는 내용

- 발표 서사 구조
- 슬라이드 구성 원칙
- 데모 각본 작성법
- Q&A 준비법
- 시간 배분 기준

## 왜 중요한가

발표는 결과물을 보여 주는 마지막 순간이자, 팀이 무엇을 배웠는지 설명하는 순간입니다. 같은 결과물이라도 서사가 없으면 단순 기능 나열로 보이고, 서사가 있으면 문제 해결 과정으로 보입니다.

실무 발표나 투자자 발표도 문제, 해결, 결과 구조를 자주 사용합니다. 듣는 사람은 기능 개수보다 어떤 문제를 풀었고 어떤 변화가 있었는지에 더 빠르게 반응합니다.

## 실전 문서 예시: 발표 운영 시트

슬라이드 파일보다 먼저 아래 같은 운영 시트를 만들면 발표 흐름이 훨씬 안정됩니다.

```text
구간 | 시간 | 핵심 메시지 | 백업 자료
문제 설명 | 2분 | 왜 이 문제가 불편한가 | 사용자 인터뷰 한 장
해결 방식 | 3분 | 왜 이 흐름을 선택했는가 | 요구사항 표 한 장
데모 | 4분 | 핵심 흐름 60초 시연 | 스크린샷 3장, 녹화 영상
결과와 학습 | 2분 | 무엇이 확인되었고 무엇이 남았는가 | 피드백 요약 표
Q&A | 4분 | 스택, 테스트, 범위 컷 설명 | ADR, 회고 메모
```

## 이 문서로 먼저 확인할 것

- 한 슬라이드에 한 메시지만 남겼는지 확인합니다.
- 데모 실패 시 바로 전환할 백업 자료가 있는지 점검합니다.
- 시간 분배가 실제 발표 시간과 맞는지 리허설합니다.
- 예상 질문에 대한 근거 문서를 따로 준비합니다.

## 핵심 용어

- **narrative**: 발표 전체를 묶는 이야기 흐름입니다.
- **slide**: 한 장에 하나의 메시지를 담는 화면입니다.
- **demo**: 미리 준비한 순서대로 보여 주는 시연입니다.
- **QnA**: 예상 질문과 답변 준비입니다.
- **timing**: 발표 시간 배분입니다.

## 이전과 이후

**Before**: 기능 목록 슬라이드를 만듭니다.

**After**: 문제, 해결, 결과 흐름으로 슬라이드를 구성합니다.

## 실습: 슬라이드 표

### 1단계 — 서사 만들기

발표 서사는 `problem`, `solution`, `demo`, `result`, `next`처럼 다섯 단계로 먼저 고정합니다.

이 다섯 단계만 정리해도 발표의 뼈대가 잡힙니다. 기능은 이 흐름 안에서 필요한 만큼만 보여 주면 됩니다.

### 2단계 — 슬라이드 수 배분

슬라이드 수는 `problem=2`, `solution=3`, `demo=1`, `result=2`, `next=1`처럼 구간별로 미리 배분합니다.

슬라이드 수를 미리 정해 두면 특정 구간이 과하게 길어지는 것을 막을 수 있습니다.

### 3단계 — 데모 각본

데모 장면은 `login`, `core_action`, `result_view`처럼 세 단계 이내로 줄여 두는 편이 안전합니다.

데모는 욕심내기보다 세 단계 이내의 핵심 장면으로 줄이는 편이 안전합니다. 길어질수록 실패 가능성도 커집니다.

### 4단계 — Q&A 준비

예상 질문은 `why_this_stack`, `how_we_tested`, `what_we_cut`처럼 미리 목록화해 답변 근거를 준비합니다.

Q&A는 즉흥 대응만 믿지 말고, 예상 질문과 답변을 미리 정리해 두는 편이 좋습니다. 특히 왜 이 기술을 골랐는지, 무엇을 의도적으로 뺐는지는 자주 묻는 질문입니다.

### 5단계 — 시간 분배

시간 배분은 `talk=8`, `demo=5`, `qna=7`처럼 숫자로 적어 둬야 발표 후반이 급해지지 않습니다.

시간 분배가 없으면 발표는 뒤로 갈수록 급해집니다. 발표 시간도 설계 대상이라고 보는 편이 좋습니다.

## 이 코드에서 먼저 볼 점

- 한 슬라이드에는 한 메시지만 둡니다.
- 데모는 세 단계 이내로 줄입니다.
- Q&A는 미리 준비합니다.
- 시간 배분도 문서에 적어 둡니다.

## 자주 하는 실수 5가지

1. 슬라이드에 텍스트를 너무 많이 넣습니다.
2. 기능 나열에 머뭅니다.
3. 데모 실패에 대비한 대체 흐름이 없습니다.
4. Q&A 준비가 없습니다.
5. 시간을 초과합니다.

## 실무에서는 이렇게 이어집니다

실무 발표나 투자자 피치도 문제, 해결, 결과 구조를 자주 사용합니다. 듣는 사람은 모든 기능을 알고 싶어 하기보다, 왜 이 일이 중요하고 어떤 결과가 있었는지를 먼저 이해하고 싶어 합니다. 캡스톤 발표도 같은 원리로 준비하는 편이 훨씬 설득력 있습니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 서사가 기능보다 먼저입니다.
- 슬라이드는 시각적으로 단순해야 합니다.
- 데모는 리허설합니다.
- Q&A도 각본을 씁니다.
- 시간을 반드시 지킵니다.

## 체크리스트

- [ ] 시간 배분 표가 있습니다.

## 정리와 다음 글

발표 자료의 핵심은 많이 보여 주는 것이 아니라 빨리 이해시키는 것입니다. 문제·해결·결과 구조, 데모 각본, 백업 자료, Q&A 근거를 함께 준비하면 발표가 훨씬 단단해집니다. 다음 글에서는 프로젝트를 마친 뒤 학습을 남기는 회고를 다룹니다.

## 실전 확장: 발표 슬라이드 구성 표와 데모 스크립트 예시

발표 자료 품질은 슬라이드 디자인보다 메시지 구조에서 결정됩니다. 특히 캡스톤에서는 제한된 시간 안에 문제, 선택, 결과를 보여줘야 하므로 슬라이드별 목적과 데모 스크립트를 분리해 준비하는 것이 중요합니다.

### 발표 슬라이드 구성 표

| 슬라이드 번호 | 제목 | 핵심 메시지 | 시각 자료 | 발표 시간 |
| --- | --- | --- | --- | --- |
| 1 | 문제 배경 | 왜 이 문제가 실제로 불편한가 | 사용자 상황 이미지 | 1분 |
| 2 | 문제 정의 | 누구의 어떤 불편을 해결하는가 | 문제 문장 박스 | 1분 |
| 3 | 해결 접근 | 왜 이 흐름을 선택했는가 | 시스템 흐름도 | 1분 |
| 4 | MVP 범위 | 무엇을 포함했고 무엇을 제외했는가 | 범위 표 | 1분 |
| 5 | 데모 | 핵심 흐름이 실제로 동작하는가 | 실시간 시연 | 3분 |
| 6 | 결과 지표 | 무엇이 검증되었는가 | 지표 표/그래프 | 1분 |
| 7 | 한계와 개선 | 무엇이 남았고 어떻게 확장할 것인가 | 로드맵 | 1분 |
| 8 | 결론 | 이 프로젝트가 남긴 학습 | 요약 문장 3개 | 1분 |

### 데모 스크립트 예시

```text
0:00-0:20  데모 목표 소개
"지금부터 시간표 충돌 검사 핵심 흐름을 60초 안에 보여드리겠습니다."

0:20-0:50  입력 단계
"샘플 시간표를 붙여 넣고 검사 버튼을 누릅니다."

0:50-1:20  계산/결과 단계
"충돌 과목과 시간대가 즉시 표시됩니다."

1:20-1:40  가치 확인
"수동 계산 대비 확인 시간을 크게 줄일 수 있었습니다."

1:40-2:00  백업 전환 안내
"네트워크 이슈 발생 시 동일 흐름을 녹화 영상으로 이어서 설명하겠습니다."
```

스크립트를 문장 단위로 준비하면 발표 중 긴장으로 인한 설명 누락을 줄일 수 있습니다. 특히 데모의 시작 문장과 종료 문장을 고정해 두면 전달력이 안정됩니다.

### 예상 Q&A 준비 표

| 질문 유형 | 예상 질문 | 답변 핵심 | 근거 문서 |
| --- | --- | --- | --- |
| 기술 선택 | 왜 이 스택을 골랐는가 | 학습 비용과 운영 부담이 낮음 | ADR 문서 |
| 범위 조정 | 왜 특정 기능을 제외했는가 | 데모 기여도 대비 난도 높음 | MVP 범위 표 |
| 테스트 | 신뢰성을 어떻게 확인했는가 | 통합 테스트 시나리오 통과 | 테스트 리포트 |
| 확장 계획 | 다음 단계는 무엇인가 | 추천 기능/모바일 개선 | 로드맵 슬라이드 |

### 리허설 운영 체크리스트

- 발표자 교대 지점을 초 단위로 표시합니다.
- 데모 실패 시 백업 전환 문장을 연습합니다.
- 각 슬라이드의 "한 줄 결론"을 고정합니다.
- 질문 10개를 랜덤으로 뽑아 답변 리허설을 수행합니다.
- 시간 초과 시 줄일 슬라이드를 미리 지정합니다.

### 흔들림을 줄이는 운영 원칙

발표는 완벽한 정보 전달보다 명확한 우선순위 전달이 중요합니다. 따라서 "많이 보여주기"보다 "핵심만 반복하기"가 더 효과적입니다. 슬라이드 구성 표와 데모 스크립트를 함께 운영하면 팀 전체가 같은 메시지를 말하게 되고, 심사위원 입장에서도 프로젝트의 의사결정 품질을 빠르게 이해할 수 있습니다.

## 심화 보강: 의사결정 근거 문서화 패턴

캡스톤 문서의 분량을 늘리는 목적은 글자 수 자체가 아니라 판단의 흔적을 남기는 데 있습니다. 같은 결과물을 만들더라도 판단 근거가 기록된 팀과 기록되지 않은 팀의 학습 밀도는 크게 다릅니다. 아래 패턴은 어떤 장에서든 공통으로 적용할 수 있는 최소 문서화 규칙입니다.

### 의사결정 카드 템플릿

| 항목 | 작성 기준 | 예시 |
| --- | --- | --- |
| 결정 제목 | 동사형으로 짧게 작성 | 입력 형식을 CSV로 고정한다 |
| 배경 | 현재 문제와 제약을 3문장 내 서술 | 발표 전 2주 내 구현 안정화 필요 |
| 선택지 | 최소 2개 이상 비교 | CSV, 엑셀 직접 업로드 |
| 평가 기준 | 시간, 위험, 구현 난도 명시 | 학습 비용, 오류 가능성 |
| 최종 결정 | 한 문장으로 고정 | CSV 입력을 MVP 범위로 채택 |
| 후속 작업 | 실행 가능한 액션으로 변환 | 파서 검증 테스트 5건 추가 |

### 리스크 등록부 예시

| 리스크 ID | 설명 | 발생 가능성 | 영향도 | 대응 전략 | 담당 |
| --- | --- | --- | --- | --- | --- |
| RK-01 | 외부 데이터 접근 지연 | 중간 | 높음 | 샘플 데이터 대체 경로 준비 | 데이터 담당 |
| RK-02 | 데모 환경 네트워크 불안정 | 중간 | 높음 | 오프라인 영상 백업 준비 | 발표 담당 |
| RK-03 | 통합 테스트 누락 | 높음 | 중간 | 주간 회귀 테스트 슬롯 고정 | QA 담당 |
| RK-04 | 요구사항 빈번한 변경 | 중간 | 중간 | 변경 승인 규칙 명시 | 팀 리드 |

리스크 등록부는 불안을 기록하는 문서가 아니라 우선순위를 결정하는 문서입니다. 발생 가능성과 영향도를 함께 보면 어떤 리스크를 먼저 줄여야 하는지 명확해집니다.

### 주간 리뷰 질문

- 이번 주 결정 중 다음 주에 다시 검토할 항목은 무엇인가
- 일정 편차를 만든 원인은 기술 이슈인가, 의사결정 지연인가
- 범위 확장 요청이 들어왔을 때 거절 기준이 문서에 있는가
- 테스트 실패가 특정 구간에 집중되는가
- 발표 시점에 그대로 설명 가능한 근거가 충분한가

### 품질을 높이는 기록 원칙

첫째, 문장은 짧고 검증 가능하게 씁니다. 둘째, 수치가 가능한 항목은 반드시 수치로 남깁니다. 셋째, 결정과 액션을 분리하지 않습니다. 넷째, 회의 직후 24시간 내 문서를 갱신합니다. 다섯째, 변경 이유를 생략하지 않습니다. 이 다섯 가지 원칙만 지켜도 캡스톤 문서는 단순 제출물이 아니라 다음 프로젝트를 위한 운영 매뉴얼로 전환됩니다.

### 적용 체크리스트

- [ ] 결정 카드가 최신 상태로 유지됩니다.
- [ ] 리스크 등록부에 담당자와 대응 전략이 있습니다.
- [ ] 주간 리뷰 질문에 대한 답변이 기록됩니다.
- [ ] 변경 로그가 날짜 순으로 정리됩니다.
- [ ] 발표와 회고 문서가 같은 근거를 참조합니다.

위 보강 블록은 각 장의 주제와 별개로 공통 적용되는 운영 원칙입니다. 캡스톤의 난도는 기술 난도보다 의사결정 난도에서 크게 올라갑니다. 따라서 문서화 패턴을 반복 적용하면 팀의 판단 체계가 안정되고, 일정과 품질, 발표 전달력이 동시에 개선됩니다.

### 추가 보강: 발표 직전 최종 점검 루틴

발표 직전에는 새 기능 추가를 중단하고 안정화 루틴을 고정합니다. 최종 점검 루틴은 기술 완성도를 높이는 절차가 아니라 실패 확률을 낮추는 절차입니다. 핵심 흐름 시연, 오류 메시지 점검, 백업 자료 확인, 예상 질문 재점검, 역할별 멘트 정렬을 반복하면 전달 품질이 크게 향상됩니다.

| 점검 항목 | 확인 내용 | 통과 기준 |
| --- | --- | --- |
| 핵심 시연 | 시작부터 결과 화면까지 흐름 유지 | 3회 연속 성공 |
| 오류 처리 | 비정상 입력 시 안내 문구 노출 | 사용자 행동 유도 가능 |
| 백업 자료 | 네트워크 실패 대비 영상/스크린샷 | 즉시 전환 가능 |
| Q&A 근거 | 기술/범위/테스트 답변 문서화 | 질문 10개 대응 가능 |
| 역할 분담 | 발표자, 데모 조작자, 시간 관리자 확정 | 리허설 동일 운영 |

이 루틴을 문서로 남기면 회고 단계에서 무엇이 실제로 효과가 있었는지 비교할 수 있습니다.

## 실전 앵커: 발표 슬라이드 구조와 데모 대본 템플릿

발표 자료는 디자인보다 구조가 먼저입니다. 슬라이드가 많아도 메시지 순서가 흐리면 전달력이 급격히 떨어집니다. 캡스톤 발표는 문제-해결-검증-회고 순서를 고정하는 것이 가장 안정적입니다.

### 10장 슬라이드 표준 구조

| 장수 | 제목 | 핵심 메시지 |
| --- | --- | --- |
| 1 | 문제 배경 | 왜 이 문제가 중요한가 |
| 2 | 사용자 정의 | 누구를 위한 해결인가 |
| 3 | 기존 방식 한계 | 현재 불편의 근거 |
| 4 | 제안 솔루션 | 핵심 아이디어 |
| 5 | 시스템 흐름 | 동작 원리 요약 |
| 6 | MVP 범위 | 이번 학기 구현 범위 |
| 7 | 검증 결과 | 지표와 테스트 결과 |
| 8 | 데모 장면 | 실제 사용 흐름 |
| 9 | 한계와 리스크 | 남은 과제 |
| 10 | 회고와 다음 단계 | 학습과 확장 계획 |

### 데모 대본 템플릿

```text
오프닝(20초): 사용자 문제를 한 문장으로 설명
장면 1(40초): 입력과 초기 상태
장면 2(60초): 핵심 처리 과정
장면 3(40초): 결과와 지표 확인
클로징(20초): 한계와 다음 실험
```

대본은 발표자 개인 메모가 아니라 팀 문서로 관리해야 합니다. 그래야 발표자가 바뀌어도 메시지 순서가 유지되고, 질문 대응도 동일한 근거로 이어집니다.

## 처음 질문으로 돌아가기

- **왜 기능 목록 중심 발표는 지루해질까요?**
  - 이 글은 발표 자료 만들기을 단순 정의가 아니라, 실무에서 마주하는 구체적 상황과 판단 과정으로 풀어냈습니다. 각 섹션의 예시와 체크리스트를 따라가며 자신의 상황에 적용할 수 있도록 구성했습니다.
- **문제·해결·결과 구조는 왜 전달력이 높을까요?**
  - 본문에서 제시한 예시 코드나 매트릭스를 참고하면, 좋은 판단 기준이 무엇인지 구체적으로 느낄 수 있습니다. 숫자나 기준이 아니라 '어떤 상황에서 이런 기준을 세우는가'를 이해하는 것이 중요합니다.
- **슬라이드는 어떻게 구성해야 한 장당 메시지가 선명할까요?**
  - 이 글의 핵심 메시지는 평가 기준이 무엇이든, 한 번의 판단으로 끝나지 않는다는 점입니다. 초반에 정한 기준을 문서로 남기고, 나중에 돌아봤을 때도 같은 실수를 반복하지 않도록 하는 과정 자체가 프로젝트를 성공으로 이끕니다.
