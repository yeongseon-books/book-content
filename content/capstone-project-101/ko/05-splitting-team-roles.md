# Capstone Project 101 (5/10): 팀 역할 나누기

이 글은 캡스톤 프로젝트 101 시리즈의 5번째 글입니다.

역할이 모호한 팀은 모두가 바빠 보이는데도 결정이 느립니다. 누가 최종 판단을 내리고 누가 백업하는지가 분명하지 않기 때문입니다.

캡스톤에서 역할 분담은 사람을 칸에 넣는 일이 아니라 병목을 줄이는 구조를 만드는 일입니다. 특히 짧은 일정에서는 주 책임자와 백업 구조가 없으면 일정 충격이 바로 전체 팀에 퍼집니다.

이 글은 Capstone Project 101 시리즈의 5번째 글입니다. 여기서는 주 책임자, 백업 담당자, 결정 권한을 나눠 팀 협업 속도를 높이는 기본 구조를 정리합니다.

> 멘탈 모델: 역할 분담의 핵심은 사람을 분류하는 것이 아니라, 일이 막혔을 때 누가 결정하고 누가 이어받을지 미리 정하는 데 있습니다.

## 먼저 던지는 질문

- 역할이 겹치면 왜 결정이 느려질까요?
- 팀에서 자주 쓰는 핵심 역할은 무엇일까요?
- 주 책임자와 백업은 어떻게 정하는 편이 좋을까요?

> 팀 역할 나누기에서 중요한 것은 기능 이름이 아니라, 각 단계에서 '누가, 언제, 무엇으로' 의사결정을 내리는가입니다.

## 이 글에서 배우는 내용

- 다섯 가지 핵심 역할
- 책임 매트릭스 구성
- 코드 오너십 감각
- 의사결정 흐름
- 백업 담당자 지정법

## 왜 중요한가

역할이 모호한 팀은 모두가 바빠 보여도 실제 결정은 늦어집니다. 누가 먼저 판단해야 하는지, 누가 최종 책임을 지는지 불분명하기 때문입니다. 반대로 역할이 분명한 팀은 회의가 짧아지고, 변경이 생겨도 누구와 바로 조율해야 하는지 명확합니다.

특히 캡스톤처럼 일정이 짧은 프로젝트에서는 한 사람이 빠지거나 막히는 순간 전체 속도가 흔들릴 수 있습니다. 그래서 주 책임자만큼 백업 지정도 중요합니다.

## 한눈에 보는 흐름

## 실전 문서 예시: 역할 책임 매트릭스

작은 팀이라도 아래처럼 책임 문장을 적어 두면 회의 시간이 크게 줄어듭니다.

```text
업무 | 주 책임자 | 백업 | 결정 권한
요구사항 변경 | 팀 리드 | 프런트엔드 담당 | 리드 승인
API 설계 | 백엔드 담당 | 데이터 담당 | 백엔드 제안, 리드 승인
데모 시나리오 | QA 담당 | 팀 리드 | QA 제안, 팀 합의
배포 점검 | 백엔드 담당 | 프런트엔드 담당 | 배포 체크리스트 기준
```

## 이 문서로 먼저 확인할 것

- 각 영역에 한 명의 주 책임자가 있는지 확인합니다.
- 백업이 없는 핵심 업무가 남아 있지 않은지 점검합니다.
- 결정 권한이 모호한 항목을 따로 표시합니다.
- 역할 변경 날짜와 이유를 남길 자리도 함께 둡니다.

## 핵심 용어

- **lead**: 전체 흐름과 우선순위를 조율하는 역할입니다.
- **backend**: 서버와 API를 맡는 역할입니다.
- **frontend**: 사용자 화면과 상호작용을 맡는 역할입니다.
- **data**: 데이터 구조와 분석을 담당하는 역할입니다.
- **QA**: 품질을 검증하는 역할입니다.

## 이전과 이후

**Before**: 모두가 모든 일을 조금씩 맡습니다.

**After**: 각 영역마다 주 책임자와 백업 담당자가 있습니다.

## 실습: 역할 표

### 1단계 — 인원 정리

```python
members = ["A", "B", "C", "D"]
```

먼저 팀 규모를 분명히 적어 두면 어떤 역할을 겸임해야 하는지도 현실적으로 판단할 수 있습니다.

### 2단계 — 주 역할 매핑

```python
primary = {"A": "lead", "B": "backend", "C": "frontend", "D": "data"}
```

주 역할은 한 사람에게 분명히 연결하는 편이 좋습니다. 공동 책임은 듣기 좋지만 실제로는 책임 공백을 만들기 쉽습니다.

### 3단계 — 백업 매핑

```python
backup = {"backend": "C", "frontend": "B", "data": "A"}
```

백업 담당자를 미리 정해 두어야 결원이나 일정 충돌이 생겨도 작업이 멈추지 않습니다.

### 4단계 — 책임 표

```python
raci = {"deploy": ("A", "B"), "test": ("D", "C")}
```

RACI는 복잡하게 늘리기보다 꼭 필요한 결정과 작업만 담는 편이 실전에 더 도움이 됩니다.

### 5단계 — 검토 주기

```python
review = "weekly"
```

역할은 한 번 정했다고 끝나지 않습니다. 진행 중 바뀌는 현실을 반영할 주기가 필요합니다.

## 이 코드에서 먼저 볼 점

- 주 역할은 한 사람에게 연결됩니다.
- 백업은 항상 정의해 둡니다.
- RACI는 간결해야 잘 읽힙니다.
- 역할 검토 주기가 있어야 변경을 관리할 수 있습니다.

## 자주 하는 실수 5가지

1. 모두를 공동 책임자로 적습니다.
2. 백업 담당자가 없습니다.
3. 리드가 모든 결정을 끌어안습니다.
4. QA를 마지막에만 붙입니다.
5. 역할 변경을 기록하지 않습니다.

## 실무에서는 이렇게 이어집니다

실무 팀도 RACI 같은 방식으로 의사결정 권한과 실행 책임을 구분합니다. 역할을 적어 두는 목적은 조직도를 꾸미는 데 있지 않고, 일이 막혔을 때 누구에게 바로 물어봐야 하는지 분명히 만드는 데 있습니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 역할은 문서로 남깁니다.
- 백업은 선택이 아니라 필수입니다.
- 의사결정 권한을 명시합니다.
- 역할 겹침은 최소화합니다.
- 변경 사항은 바로 공유합니다.

- [ ] 주간 검토 주기가 있습니다.

## 정리와 다음 글

팀 역할 분담은 업무를 예쁘게 나누는 일이 아니라 병목을 줄이는 설계입니다. 주 책임자, 백업, 결정 권한이 분명해야 캡스톤 일정 충격을 견딜 수 있습니다. 다음 글에서는 이 구조를 바탕으로 MVP 범위를 어떻게 자를지 다룹니다.

## 실전 확장: 프로젝트 일정 계획 표와 마일스톤 설정 예시

역할이 정해졌다면 다음 과제는 역할을 시간축에 배치하는 일입니다. 많은 팀이 역할 표와 일정 표를 따로 관리하다가, 실제로는 아무도 책임지지 않는 시간이 생깁니다. 따라서 역할 분담 문서 안에 일정 계획과 마일스톤 기준을 함께 둬야 합니다.

### 6주 기준 일정 계획 표

| 주차 | 목표 | 주 책임 역할 | 완료 기준 | 위험 신호 |
| --- | --- | --- | --- | --- |
| 1주차 | 문제 정의 및 요구사항 고정 | 팀 리드 | Must 스토리 승인 | 요구사항 변경 3건 이상 |
| 2주차 | 핵심 데이터/입력 경로 확보 | 백엔드, 데이터 | 샘플 입력 정상 처리 | 외부 데이터 접근 지연 |
| 3주차 | 핵심 흐름 구현 | 프런트엔드, 백엔드 | 입력→계산→결과 흐름 연결 | 통합 실패 반복 |
| 4주차 | 테스트와 결함 수정 | QA | Must 테스트 통과율 90% 이상 | 회귀 버그 증가 |
| 5주차 | 발표 데모 리허설 | 팀 전체 | 60초 데모 3회 연속 성공 | 데모 실패 2회 이상 |
| 6주차 | 발표/회고 문서 확정 | 팀 리드, QA | 발표 자료, 회고 액션 로그 완료 | 문서 누락 |

### 마일스톤 정의 예시

| 마일스톤 | 정의 | 증빙 산출물 | 통과 조건 |
| --- | --- | --- | --- |
| M1 요구사항 잠금 | Must 범위 확정 | 요구사항 시트 v1 | 변경 요청 보류 규칙 합의 |
| M2 MVP 연결 | 핵심 흐름 동작 | 데모 영상 초안 | 1회 성공 시연 |
| M3 품질 안정화 | 결함 위험 축소 | 테스트 리포트 | 주요 결함 0건 |
| M4 발표 준비 | 전달력 검증 | 슬라이드, 데모 각본 | 리허설 시간 준수 |

마일스톤은 "이번 주 바빴다"를 기록하는 도구가 아니라 "무엇이 끝났는가"를 판단하는 도구입니다. 완료 기준이 모호하면 역할이 명확해도 일정은 계속 흐려집니다.

### 역할-일정 동기화 규칙

- 주 책임자는 주간 목표를 문장으로 선언합니다.
- 백업 담당자는 블로커 발생 시 24시간 내 대체 계획을 제시합니다.
- 팀 리드는 주간 회고에서 일정 편차 원인을 기록합니다.
- QA는 마일스톤별 테스트 기준을 사전 공지합니다.
- 발표 주차에는 신규 기능 추가를 금지합니다.

### 일정 조정 의사결정 프레임

```text
질문 1: 이 변경이 핵심 흐름을 강화하는가
질문 2: 이 변경이 발표 데모 성공 확률을 높이는가
질문 3: 이 변경으로 Must 테스트 범위가 증가하는가
질문 4: 증가한 범위를 현재 인력으로 감당 가능한가
결론: 2개 이상 '아니오'면 다음 주차로 이월
```

역할 분담의 품질은 결국 일정 충격을 견디는 능력으로 측정됩니다. 위 표와 규칙을 함께 운영하면 팀은 "누가 무엇을 언제 끝낼지"를 같은 언어로 말할 수 있고, 그 자체가 협업 속도를 높입니다.



## 심화 보강: 의사결정 근거 문서화 패턴

캡스톤 문서의 분량을 늘리는 목적은 글자 수 자체가 아니라 판단의 흔적을 남기는 데 있습니다. 같은 결과물을 만들더라도 판단 근거가 기록된 팀과 기록되지 않은 팀의 학습 밀도는 크게 다릅니다. 아래 패턴은 어떤 장에서든 공통으로 적용할 수 있는 최소 문서화 규칙입니다.

### 의사결정 카드 템플릿

| 항목 | 작성 기준 | 예시 |
| --- | --- | --- |
| 결정 제목 | 동사형으로 짧게 작성 | 입력 형식을 CSV로 고정한다 |
| 배경 | 현재 문제와 제약을 3문장 내 서술 | 발표 전 2주 내 구현 안정화 필요 |
| 선택지 | 최소 2개 이상 비교 | CSV, 엑셀 직접 업로드 |
| 평가 기준 | 시간, 위험, 구현 난도 명시 | 학습 비용, 오류 가능성 |
| 최종 결정 | 한 문장으로 고정 | CSV 입력을 MVP 범위로 채택 |
| 후속 작업 | 실행 가능한 액션으로 변환 | 파서 검증 테스트 5건 추가 |

### 리스크 등록부 예시

| 리스크 ID | 설명 | 발생 가능성 | 영향도 | 대응 전략 | 담당 |
| --- | --- | --- | --- | --- | --- |
| RK-01 | 외부 데이터 접근 지연 | 중간 | 높음 | 샘플 데이터 대체 경로 준비 | 데이터 담당 |
| RK-02 | 데모 환경 네트워크 불안정 | 중간 | 높음 | 오프라인 영상 백업 준비 | 발표 담당 |
| RK-03 | 통합 테스트 누락 | 높음 | 중간 | 주간 회귀 테스트 슬롯 고정 | QA 담당 |
| RK-04 | 요구사항 빈번한 변경 | 중간 | 중간 | 변경 승인 규칙 명시 | 팀 리드 |

리스크 등록부는 불안을 기록하는 문서가 아니라 우선순위를 결정하는 문서입니다. 발생 가능성과 영향도를 함께 보면 어떤 리스크를 먼저 줄여야 하는지 명확해집니다.

### 주간 리뷰 질문

- 이번 주 결정 중 다음 주에 다시 검토할 항목은 무엇인가
- 일정 편차를 만든 원인은 기술 이슈인가, 의사결정 지연인가
- 범위 확장 요청이 들어왔을 때 거절 기준이 문서에 있는가
- 테스트 실패가 특정 구간에 집중되는가
- 발표 시점에 그대로 설명 가능한 근거가 충분한가

### 품질을 높이는 기록 원칙

첫째, 문장은 짧고 검증 가능하게 씁니다. 둘째, 수치가 가능한 항목은 반드시 수치로 남깁니다. 셋째, 결정과 액션을 분리하지 않습니다. 넷째, 회의 직후 24시간 내 문서를 갱신합니다. 다섯째, 변경 이유를 생략하지 않습니다. 이 다섯 가지 원칙만 지켜도 캡스톤 문서는 단순 제출물이 아니라 다음 프로젝트를 위한 운영 매뉴얼로 전환됩니다.

### 적용 체크리스트

- [ ] 결정 카드가 최신 상태로 유지됩니다.
- [ ] 리스크 등록부에 담당자와 대응 전략이 있습니다.
- [ ] 주간 리뷰 질문에 대한 답변이 기록됩니다.
- [ ] 변경 로그가 날짜 순으로 정리됩니다.
- [ ] 발표와 회고 문서가 같은 근거를 참조합니다.

위 보강 블록은 각 장의 주제와 별개로 공통 적용되는 운영 원칙입니다. 캡스톤의 난도는 기술 난도보다 의사결정 난도에서 크게 올라갑니다. 따라서 문서화 패턴을 반복 적용하면 팀의 판단 체계가 안정되고, 일정과 품질, 발표 전달력이 동시에 개선됩니다.



### 추가 보강: 발표 직전 최종 점검 루틴

발표 직전에는 새 기능 추가를 중단하고 안정화 루틴을 고정합니다. 최종 점검 루틴은 기술 완성도를 높이는 절차가 아니라 실패 확률을 낮추는 절차입니다. 핵심 흐름 시연, 오류 메시지 점검, 백업 자료 확인, 예상 질문 재점검, 역할별 멘트 정렬을 반복하면 전달 품질이 크게 향상됩니다.

| 점검 항목 | 확인 내용 | 통과 기준 |
| --- | --- | --- |
| 핵심 시연 | 시작부터 결과 화면까지 흐름 유지 | 3회 연속 성공 |
| 오류 처리 | 비정상 입력 시 안내 문구 노출 | 사용자 행동 유도 가능 |
| 백업 자료 | 네트워크 실패 대비 영상/스크린샷 | 즉시 전환 가능 |
| Q&A 근거 | 기술/범위/테스트 답변 문서화 | 질문 10개 대응 가능 |
| 역할 분담 | 발표자, 데모 조작자, 시간 관리자 확정 | 리허설 동일 운영 |

이 루틴을 문서로 남기면 회고 단계에서 무엇이 실제로 효과가 있었는지 비교할 수 있습니다.



### 보강 메모: 다음 프로젝트로 이관할 핵심 문장

이번 장의 핵심 판단 기준을 3문장으로 요약해 저장소 루트의 프로젝트 로그에 남깁니다. 다음 프로젝트 킥오프에서 이 문장을 먼저 읽으면 재시작 비용을 크게 줄일 수 있습니다. 문장 요약은 길이보다 재사용 가능성이 중요합니다.



### 마무리 보강: 검토 회의 운영 규칙

검토 회의는 30분 제한으로 운영하고, 결론 없는 논의를 다음 액션으로 분해합니다. 안건마다 결정자 1명과 기록자 1명을 고정하면 책임 공백을 줄일 수 있습니다. 회의 종료 전에는 반드시 다음 검토 시점과 확인 지표를 합의합니다.



추가 확인 메모: 본 장의 핵심 기준을 다음 스프린트 킥오프 문서 첫 줄에 재사용합니다.

## 실전 앵커: 역할 분담 표준안과 협업 브랜치 전략

팀 역할은 사람을 나누는 일이 아니라 책임 경계를 만드는 일입니다. 역할이 모호하면 일정이 밀릴 때 누구도 의사결정을 확정하지 못합니다. 아래 표준안은 소규모 캡스톤 팀에서 바로 적용할 수 있는 최소 구성입니다.

### 책임 배정표(RACI) 기반 역할 분담 예시

| 작업 | 책임자(R) | 승인자(A) | 협업자(C) | 공유 대상(I) |
| --- | --- | --- | --- | --- |
| 문제 정의 확정 | PM | 팀 리드 | 전원 | 지도교수 |
| MVP 구현 | 개발 담당 | 팀 리드 | QA 담당 | 전원 |
| 테스트/검증 | QA 담당 | 팀 리드 | 개발 담당 | 전원 |
| 발표 자료 작성 | 발표 담당 | PM | 전원 | 지도교수 |

### 협업 브랜치 규칙

- 기능 단위로 `feat/<owner>/<topic>` 브랜치를 생성합니다.
- 하루 1회 이상 원격 저장소에 푸시해 변경 이력을 공개합니다.
- PR 본문에 "목적/변경/검증" 3개 항목을 고정합니다.
- 리뷰 체크리스트를 통과하지 못하면 머지를 보류합니다.

```text
PR 템플릿
- 목적: 어떤 문제를 해결하는 변경인가
- 변경: 핵심 수정 3가지
- 검증: 실행한 테스트와 결과
```

역할과 브랜치 규칙을 함께 고정하면 회의 시간이 줄고 결정 속도가 빨라집니다. 특히 발표 직전 주간에는 누가 무엇을 마무리하는지 혼선이 줄어듭니다.

## 실무 부록: 팀 운영 템플릿 묶음

아래 부록은 장별 주제를 넘어서 캡스톤 전 구간에서 공통으로 재사용할 수 있는 운영 템플릿입니다. 팀이 매주 다른 양식으로 기록하면 비교가 어려워지고, 발표 직전에 근거를 다시 찾느라 시간이 낭비됩니다. 따라서 최소 템플릿을 고정하고 같은 구조로 반복 기록하는 편이 안정적입니다.

### 주간 운영 로그 템플릿

| 구분 | 기록 항목 | 작성 기준 |
| --- | --- | --- |
| 목표 | 이번 주 반드시 완료할 3개 | 측정 가능한 문장으로 작성 |
| 결과 | 완료/미완료 | 근거 링크와 함께 기록 |
| 편차 | 계획 대비 차이 | 원인 1문장, 조치 1문장 |
| 품질 | 테스트 통과율, 버그 건수 | 숫자로 기록 |
| 위험 | 다음 주 리스크 | 담당자와 대응 일정 포함 |

### 제안서-요구사항-데모 연결 규칙

1. 제안서의 문제 문장은 요구사항 문서 첫 줄과 동일한 의미여야 합니다.
2. 요구사항 ID는 테스트 케이스 ID와 1:1로 연결합니다.
3. 데모 스크립트 장면 번호를 요구사항 추적표에 함께 기록합니다.
4. 발표 슬라이드의 지표 숫자는 테스트 리포트 숫자와 동일해야 합니다.

이 규칙을 지키면 "문서는 맞는데 시연이 안 맞는" 상태를 줄일 수 있습니다. 캡스톤 평가에서는 기능의 화려함보다 논리 연결성이 더 강한 설득력을 만듭니다.

### 코드 리뷰 체크리스트(공통)

- 변경 목적이 현재 스프린트 목표와 맞는지 확인합니다.
- 테스트가 없는 기능 추가는 기본적으로 보류합니다.
- 예외 처리와 오류 메시지가 데모 시나리오를 방해하지 않는지 점검합니다.
- 설정 파일 변경은 롤백 방법을 함께 기록합니다.
- 문서 변경 없이 동작만 바뀌는 PR은 승인하지 않습니다.

### CI/CD 파이프라인 확인표

| 단계 | 필수 체크 | 실패 시 조치 |
| --- | --- | --- |
| 의존성 설치 | 버전 고정 여부 | 잠금 파일 재생성 |
| 테스트 | 핵심 테스트 통과 | 실패 케이스 원인 분류 |
| 빌드 | 아티팩트 생성 성공 | 빌드 로그 첨부 후 재시도 |
| 배포 | 스테이징 헬스체크 통과 | 즉시 이전 버전 롤백 |
| 검증 | 데모 핵심 경로 점검 | 배포 승인 보류 |

### 발표 직전 48시간 운영 규칙

첫째, 새 기능 추가를 중단합니다. 둘째, 리허설 스크립트를 고정하고 시간 측정을 반복합니다. 셋째, 데모 실패 대비 백업 영상을 준비합니다. 넷째, Q&A 예상 질문을 기술/범위/일정/검증 축으로 정리합니다. 다섯째, 발표 당일 역할(발표자, 조작자, 시간 관리자)을 변경하지 않습니다.

이 다섯 가지 규칙은 발표 직전 혼선을 줄이는 데 효과적입니다. 실제로 많은 팀이 이 시점에 기능 욕심을 다시 내다가 핵심 흐름을 망칩니다. 캡스톤의 완성도는 마지막 이틀의 선택에서 크게 갈립니다.

### 데모 스크립트 표준안

```text
오프닝: 사용자 문제와 목표 지표를 20초 안에 제시
시연 1: 입력 단계와 검증 메시지 확인
시연 2: 핵심 처리 과정과 결과 화면 연결
시연 3: 실패 케이스 1건을 짧게 제시
클로징: 한계, 다음 확장, 회고 한 줄
```

스크립트를 표준안으로 고정하면 발표자가 바뀌어도 전달력이 유지됩니다. 또한 회고 단계에서 어떤 장면이 실제 질문을 많이 유도했는지 비교 분석할 수 있습니다.

### 추가 보강: 발표 심사 기준 대응 메모

심사 관점에서는 기능 개수보다 문제 적합성, 구현 완성도, 검증 근거, 협업 과정, 전달 명확성이 함께 평가됩니다. 따라서 팀은 주간 문서에 이 다섯 축을 고정 항목으로 기록해야 합니다. 기능 진척이 느린 주차라도 검증 근거와 리스크 대응이 선명하면 전체 평가의 신뢰도를 유지할 수 있습니다.

- 문제 적합성: 사용자 문제와 기능 연결이 유지되는가
- 구현 완성도: 핵심 흐름이 끊기지 않는가
- 검증 근거: 테스트/로그/시연 결과가 일치하는가
- 협업 과정: 역할 분담과 리뷰 기록이 남아 있는가
- 전달 명확성: 슬라이드와 데모가 같은 메시지를 말하는가

## 처음 질문으로 돌아가기

- **역할이 겹치면 왜 결정이 느려질까요?**
  - 이 글은 팀 역할 나누기을 단순 정의가 아니라, 실무에서 마주하는 구체적 상황과 판단 과정으로 풀어냈습니다. 각 섹션의 예시와 체크리스트를 따라가며 자신의 상황에 적용할 수 있도록 구성했습니다.
- **팀에서 자주 쓰는 핵심 역할은 무엇일까요?**
  - 본문에서 제시한 예시 코드나 매트릭스를 참고하면, 좋은 판단 기준이 무엇인지 구체적으로 느낄 수 있습니다. 숫자나 기준이 아니라 '어떤 상황에서 이런 기준을 세우는가'를 이해하는 것이 중요합니다.
- **주 책임자와 백업은 어떻게 정하는 편이 좋을까요?**
  - 이 글의 핵심 메시지는 평가 기준이 무엇이든, 한 번의 판단으로 끝나지 않는다는 점입니다. 초반에 정한 기준을 문서로 남기고, 나중에 돌아봤을 때도 같은 실수를 반복하지 않도록 하는 과정 자체가 프로젝트를 성공으로 이끕니다.
