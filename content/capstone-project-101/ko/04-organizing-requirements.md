# Capstone Project 101 (4/10): 요구사항 정리

이 글은 캡스톤 프로젝트 101 시리즈의 4번째 글입니다.

요구사항 문서가 기능 목록으로만 끝나면 일정이 밀릴 때 무엇을 줄여야 할지 결정하기 어렵습니다. 구현이 끝났는지도 사람마다 다르게 해석합니다.

좋은 요구사항 문서는 기능 이름을 모아 놓은 목록이 아니라 사용자 스토리, 수용 기준, 비기능 조건, 우선순위를 연결한 작은 계약서에 가깝습니다.

## 먼저 던지는 질문

- 왜 기능 목록만으로는 요구사항이 충분하지 않을까요?
- 사용자 스토리와 기능 설명은 무엇이 다를까요?
- 수용 기준은 어느 정도까지 구체적이어야 할까요?

## 이 글에서 배우는 내용

- 사용자 스토리 작성법
- 비기능 요건 분리
- 우선순위 정하는 방법
- 수용 기준 쓰는 법
- 요구사항 추적성 기초

## 왜 중요한가

요구사항은 바뀔 수 있습니다. 중요한 점은 바뀌더라도 무엇이, 왜 바뀌었는지 따라갈 수 있어야 한다는 점입니다. 기능 이름만 적혀 있으면 변경이 생길 때 우선순위를 다시 정하기 어렵고, 테스트 범위도 흐려집니다.

좋은 팀은 요구사항을 스토리, 기준, 우선순위, 비기능 조건으로 분리해 둡니다. 그렇게 해야 일정이 밀릴 때 무엇을 남기고 무엇을 줄일지 빠르게 결정할 수 있습니다.

## 실전 문서 예시: 요구사항 시트

아래처럼 ID를 붙여 문서를 쓰면 구현 항목과 테스트 체크리스트를 나중에 연결하기가 쉬워집니다.

```text
ID: ST-1
사용자 스토리: 학생으로서 시간표 충돌을 바로 확인하고 싶다
수용 기준: 5초 안에 입력, 1초 안에 결과 표시, 오류 메시지 명확
비기능 요구: 모바일 우선, 로그인 없이 체험 가능, 한국어 우선
우선순위: Must
연결 기능: F-1 일정 입력, F-2 충돌 계산, F-3 결과 화면
```

## 이 문서로 먼저 확인할 것

- 수용 기준이 관찰 가능한 문장인지 확인합니다.
- 비기능 요구를 별도 항목으로 분리합니다.
- Must와 Should를 모두 Must로 적지 않았는지 점검합니다.
- 요구사항 ID가 구현 기능이나 테스트 항목과 연결되는지 봅니다.

## 핵심 용어

- **user story**: 사용자 관점에서 쓴 요구입니다.
- **acceptance**: 완료 여부를 판단하는 기준입니다.
- **non-functional**: 속도, 접근성, 가입 절차 같은 비기능 요건입니다.
- **MoSCoW**: Must, Should, Could 같은 우선순위 라벨입니다.
- **traceability**: 요구사항과 기능을 연결해 추적하는 성질입니다.

## 이전과 이후

**Before**: 기능만 나열합니다.

**After**: 스토리, 기준, 우선순위를 함께 기록합니다.

## 실습: 요구사항 표

### 1단계 — 사용자 스토리

```python
story = "as a student I want instant conflict detection"
```

스토리는 기능 이름보다 사용자 행동이 드러나야 합니다. 누가 무엇을 원하고 왜 필요한지 읽혀야 합니다.

### 2단계 — 수용 기준

```python
accept = ["input 5s", "result 1s", "error clear"]
```

수용 기준은 애매한 표현보다 시간, 결과, 오류 처리처럼 확인 가능한 문장으로 적는 편이 좋습니다.

### 3단계 — 비기능

```python
nf = ["mobile", "no_signup", "korean_first"]
```

비기능 요건은 기능 설명에 섞지 말고 따로 분리해야 합니다. 그래야 구현 범위와 품질 기준을 함께 볼 수 있습니다.

### 4단계 — 우선순위

```python
prio = {"core": "Must", "share": "Should", "ai": "Could"}
```

우선순위는 일정이 흔들릴 때 가장 큰 힘을 발휘합니다. 모든 항목이 중요해 보이더라도 반드시 나눠 두어야 합니다.

### 5단계 — 추적

```python
trace = {"ST-1": ["F-1", "F-2"]}
```

요구사항 ID와 구현 항목을 연결해 두면 나중에 변경 영향 범위를 빠르게 파악할 수 있습니다.

## 이 코드에서 먼저 볼 점

- 스토리는 동작 중심으로 씁니다.
- 수용 기준은 측정 가능해야 합니다.
- 비기능 요건은 분리해 둡니다.
- 추적은 ID 기반으로 연결합니다.

## 자주 하는 실수 5가지

1. 사용자 스토리가 기능 설명으로 바뀝니다.
2. 비기능 요건을 빼먹습니다.
3. 모든 항목을 Must로 적습니다.
4. 수용 기준을 주관적으로 씁니다.
5. 추적성을 남기지 않습니다.

## 실무에서는 이렇게 이어집니다

실무에서도 Must, Should, Could 같은 라벨을 자주 사용합니다. 제품이든 내부 도구든, 우선순위를 나눠 두지 않으면 일정이 밀릴 때 무엇을 남겨야 할지 결정하기 어렵습니다. 캡스톤에서 요구사항을 이렇게 정리해 보는 경험은 작은 명세 문서를 다루는 연습이 됩니다.

## 시니어 엔지니어는 이렇게 생각합니다

- 스토리는 짧게 씁니다.
- 기준은 숫자와 가독성으로 적습니다.
- 우선순위는 몇 개의 버킷으로 단순화합니다.
- 비기능 요건은 따로 관리합니다.
- 추적 가능한 문서를 남깁니다.

## 정리와 다음 글

요구사항 정리는 기능 목록을 예쁘게 적는 일이 아니라, 범위와 완료 조건을 추적 가능하게 만드는 작업입니다. 스토리, 수용 기준, 비기능 요구, 우선순위를 함께 적어 두면 이후 설계와 일정 판단이 훨씬 빨라집니다. 다음 글에서는 이 일을 팀 안에서 어떻게 나눌지 다룹니다.

## 실전 확장: 팀 역할 분담 표와 협업 도구 비교

요구사항이 정리되면 다음으로 필요한 것은 역할-도구-의사결정 흐름의 일치입니다. 요구사항 문서만 있고 누가 어떤 항목의 최종 책임자인지 없으면, 변경 요청이 들어올 때 문서가 즉시 낡아집니다. 따라서 요구사항 항목을 팀 역할과 협업 도구에 직접 연결해야 합니다.

### 역할 분담 기준표

| 영역 | 주 책임자 | 백업 담당 | 산출물 | 승인 기준 |
| --- | --- | --- | --- | --- |
| 요구사항 유지보수 | 팀 리드 | QA 담당 | 요구사항 시트, 변경 로그 | 변경 사유와 영향 범위 기록 |
| API 명세 | 백엔드 담당 | 프런트엔드 담당 | OpenAPI 초안, 예시 요청/응답 | 프런트 구현 가능 여부 확인 |
| 화면 흐름 | 프런트엔드 담당 | 팀 리드 | 와이어프레임, 화면 상태표 | 핵심 흐름 60초 시연 가능 |
| 테스트 기준 | QA 담당 | 백엔드 담당 | 테스트 시나리오, 결함 목록 | Must 기능 전부 통과 |
| 일정 연동 | 팀 리드 | 전원 | 주간 계획표, 블로커 로그 | 마일스톤 지연 조기 탐지 |

이 표의 목적은 역할 고정이 아니라 병목 제거입니다. 업무가 막혔을 때 "누가 결정할 사람인가"를 바로 찾을 수 있어야 요구사항 변경이 일정 충격으로 번지지 않습니다.

### 협업 도구 비교표

| 도구 | 장점 | 단점 | 권장 용도 |
| --- | --- | --- | --- |
| Notion | 문서 구조화와 공유가 쉬움 | 버전 추적 세밀도 낮음 | 요구사항 초안, 회의록 |
| GitHub Issues | 변경 이력과 코드 연결이 강함 | 비기술 팀원 진입장벽 | 구현 이슈, 버그, 액션 아이템 |
| Google Sheets | 표 편집이 빠름 | 문맥 연결이 약함 | 우선순위 표, 일정 추적 |
| Figma | 화면 흐름 공유가 쉬움 | 요구사항 텍스트 추적에는 부적합 | UI 흐름, 데모 스토리보드 |
| Slack/Discord | 즉시 소통 가능 | 결정 기록이 흩어짐 | 알림, 빠른 동기화 |

도구 선택 원칙은 "한 도구로 모든 문제를 해결하지 않는다"입니다. 문서 중심 도구, 작업 추적 도구, 소통 도구를 최소 조합으로 구분해야 운영이 단순해집니다.

### 변경 요청 처리 프로토콜

```text
1) 변경 제안 등록: 무엇을, 왜 바꾸는지 3문장 작성
2) 영향 분석: 요구사항 ID, 기능 ID, 테스트 ID 연결
3) 역할 지정: 주 책임자/백업/검토자 지정
4) 승인/보류 결정: Must/Should 기준으로 판단
5) 기록 보존: 변경 로그에 날짜와 이유 추가
```

이 프로토콜을 적용하면 "아이디어는 좋지만 지금은 안 한다"를 근거 있게 설명할 수 있습니다. 팀 협업에서 갈등은 아이디어 자체보다 우선순위 불일치에서 생기므로, 기록 가능한 절차가 필수입니다.

### 실수 예방 체크리스트

- 요구사항마다 담당자와 검토자가 지정되어 있는가
- 구두 합의가 문서에 반영되었는가
- 도구 간 중복 입력이 과도하지 않은가
- Slack 대화의 결정 사항이 이슈나 문서로 승격되었는가
- 일주일에 한 번은 변경 로그를 다시 검토하는가

요구사항 정리 단계에서 역할과 도구를 함께 설계하면, 이후 MVP 구현 단계에서 "문서는 있는데 실행이 느린" 상황을 크게 줄일 수 있습니다.

## 심화 보강: 의사결정 근거 문서화 패턴

캡스톤 문서의 분량을 늘리는 목적은 글자 수 자체가 아니라 판단의 흔적을 남기는 데 있습니다. 같은 결과물을 만들더라도 판단 근거가 기록된 팀과 기록되지 않은 팀의 학습 밀도는 크게 다릅니다. 아래 패턴은 어떤 장에서든 공통으로 적용할 수 있는 최소 문서화 규칙입니다.

### 의사결정 카드 템플릿

| 항목 | 작성 기준 | 예시 |
| --- | --- | --- |
| 결정 제목 | 동사형으로 짧게 작성 | 입력 형식을 CSV로 고정한다 |
| 배경 | 현재 문제와 제약을 3문장 내 서술 | 발표 전 2주 내 구현 안정화 필요 |
| 선택지 | 최소 2개 이상 비교 | CSV, 엑셀 직접 업로드 |
| 평가 기준 | 시간, 위험, 구현 난도 명시 | 학습 비용, 오류 가능성 |
| 최종 결정 | 한 문장으로 고정 | CSV 입력을 MVP 범위로 채택 |
| 후속 작업 | 실행 가능한 액션으로 변환 | 파서 검증 테스트 5건 추가 |

### 리스크 등록부 예시

| 리스크 ID | 설명 | 발생 가능성 | 영향도 | 대응 전략 | 담당 |
| --- | --- | --- | --- | --- | --- |
| RK-01 | 외부 데이터 접근 지연 | 중간 | 높음 | 샘플 데이터 대체 경로 준비 | 데이터 담당 |
| RK-02 | 데모 환경 네트워크 불안정 | 중간 | 높음 | 오프라인 영상 백업 준비 | 발표 담당 |
| RK-03 | 통합 테스트 누락 | 높음 | 중간 | 주간 회귀 테스트 슬롯 고정 | QA 담당 |
| RK-04 | 요구사항 빈번한 변경 | 중간 | 중간 | 변경 승인 규칙 명시 | 팀 리드 |

리스크 등록부는 불안을 기록하는 문서가 아니라 우선순위를 결정하는 문서입니다. 발생 가능성과 영향도를 함께 보면 어떤 리스크를 먼저 줄여야 하는지 명확해집니다.

### 주간 리뷰 질문

- 이번 주 결정 중 다음 주에 다시 검토할 항목은 무엇인가
- 일정 편차를 만든 원인은 기술 이슈인가, 의사결정 지연인가
- 범위 확장 요청이 들어왔을 때 거절 기준이 문서에 있는가
- 테스트 실패가 특정 구간에 집중되는가
- 발표 시점에 그대로 설명 가능한 근거가 충분한가

### 품질을 높이는 기록 원칙

첫째, 문장은 짧고 검증 가능하게 씁니다. 둘째, 수치가 가능한 항목은 반드시 수치로 남깁니다. 셋째, 결정과 액션을 분리하지 않습니다. 넷째, 회의 직후 24시간 내 문서를 갱신합니다. 다섯째, 변경 이유를 생략하지 않습니다. 이 다섯 가지 원칙만 지켜도 캡스톤 문서는 단순 제출물이 아니라 다음 프로젝트를 위한 운영 매뉴얼로 전환됩니다.

### 적용 체크리스트

- [ ] 결정 카드가 최신 상태로 유지됩니다.
- [ ] 리스크 등록부에 담당자와 대응 전략이 있습니다.
- [ ] 주간 리뷰 질문에 대한 답변이 기록됩니다.
- [ ] 변경 로그가 날짜 순으로 정리됩니다.
- [ ] 발표와 회고 문서가 같은 근거를 참조합니다.

위 보강 블록은 각 장의 주제와 별개로 공통 적용되는 운영 원칙입니다. 캡스톤의 난도는 기술 난도보다 의사결정 난도에서 크게 올라갑니다. 따라서 문서화 패턴을 반복 적용하면 팀의 판단 체계가 안정되고, 일정과 품질, 발표 전달력이 동시에 개선됩니다.

### 추가 보강: 발표 직전 최종 점검 루틴

발표 직전에는 새 기능 추가를 중단하고 안정화 루틴을 고정합니다. 최종 점검 루틴은 기술 완성도를 높이는 절차가 아니라 실패 확률을 낮추는 절차입니다. 핵심 흐름 시연, 오류 메시지 점검, 백업 자료 확인, 예상 질문 재점검, 역할별 멘트 정렬을 반복하면 전달 품질이 크게 향상됩니다.

| 점검 항목 | 확인 내용 | 통과 기준 |
| --- | --- | --- |
| 핵심 시연 | 시작부터 결과 화면까지 흐름 유지 | 3회 연속 성공 |
| 오류 처리 | 비정상 입력 시 안내 문구 노출 | 사용자 행동 유도 가능 |
| 백업 자료 | 네트워크 실패 대비 영상/스크린샷 | 즉시 전환 가능 |
| Q&A 근거 | 기술/범위/테스트 답변 문서화 | 질문 10개 대응 가능 |
| 역할 분담 | 발표자, 데모 조작자, 시간 관리자 확정 | 리허설 동일 운영 |

이 루틴을 문서로 남기면 회고 단계에서 무엇이 실제로 효과가 있었는지 비교할 수 있습니다.

### 보강 메모: 다음 프로젝트로 이관할 핵심 문장

이번 장의 핵심 판단 기준을 3문장으로 요약해 저장소 루트의 프로젝트 로그에 남깁니다. 다음 프로젝트 킥오프에서 이 문장을 먼저 읽으면 재시작 비용을 크게 줄일 수 있습니다. 문장 요약은 길이보다 재사용 가능성이 중요합니다.

## 실전 앵커: 요구사항 추적표와 데모 스크립트 템플릿

요구사항 정리 단계에서 가장 자주 생기는 문제는 "문서는 있는데 발표에서 연결이 안 되는" 상태입니다. 이를 막으려면 요구사항-테스트-데모를 한 표에서 추적해야 합니다.

### 요구사항 추적표 예시

| 요구사항 ID | 사용자 스토리 | 수용 기준 | 테스트 케이스 | 데모 장면 |
| --- | --- | --- | --- | --- |
| RQ-01 | 시간표를 입력한다 | 입력 오류 없이 저장 | TC-01, TC-02 | 장면 1 |
| RQ-02 | 충돌 과목을 본다 | 충돌 100% 검출 | TC-03, TC-04 | 장면 2 |
| RQ-03 | 결과를 공유한다 | 공유 링크 생성 | TC-05 | 장면 3 |

### 발표용 데모 스크립트 템플릿

```text
장면 1(30초): 문제 상황 소개
장면 2(60초): 입력과 검증 흐름
장면 3(60초): 결과 확인과 가치 설명
장면 4(30초): 한계와 다음 단계
```

스크립트를 요구사항 문서와 같이 관리하면 발표 직전 급하게 장면을 바꾸는 일이 줄어듭니다. 또한 어떤 요구사항이 실제로 시연되지 않는지도 바로 파악할 수 있어 범위를 다시 정리하기 쉽습니다.

## 처음 질문으로 돌아가기

- **왜 기능 목록만으로는 요구사항이 충분하지 않을까요?**
  - 이 글은 요구사항 정리을 단순 정의가 아니라, 실무에서 마주하는 구체적 상황과 판단 과정으로 풀어냈습니다. 각 섹션의 예시와 체크리스트를 따라가며 자신의 상황에 적용할 수 있도록 구성했습니다.
- **사용자 스토리와 기능 설명은 무엇이 다를까요?**
  - 본문에서 제시한 예시 코드나 매트릭스를 참고하면, 좋은 판단 기준이 무엇인지 구체적으로 느낄 수 있습니다. 숫자나 기준이 아니라 '어떤 상황에서 이런 기준을 세우는가'를 이해하는 것이 중요합니다.
- **수용 기준은 어느 정도까지 구체적이어야 할까요?**
  - 이 글의 핵심 메시지는 평가 기준이 무엇이든, 한 번의 판단으로 끝나지 않는다는 점입니다. 초반에 정한 기준을 문서로 남기고, 나중에 돌아봤을 때도 같은 실수를 반복하지 않도록 하는 과정 자체가 프로젝트를 성공으로 이끕니다.
