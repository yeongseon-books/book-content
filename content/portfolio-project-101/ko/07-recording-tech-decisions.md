# Portfolio Project 101 (7/10): 기술적 의사결정 기록

포트폴리오 프로젝트를 검토할 때 구현 결과만큼 궁금한 것이 있습니다. 왜 그 기술을 골랐는가, 어떤 대안을 비교했는가, 무엇을 얻고 무엇을 포기했는가 같은 질문입니다. 코드만 보면 결과는 보이지만 판단은 잘 보이지 않습니다. 그래서 기술적 의사결정 기록이 필요합니다.

이 글은 Portfolio Project 101 시리즈의 7번째 글입니다. 여기서는 가벼운 ADR 형식으로 기술 선택의 이유를 남기는 방법을 살펴보고, 왜 이 기록이 포트폴리오에서 판단력을 보여 주는 강한 근거가 되는지 정리해 보겠습니다.


![Portfolio Project 101 7장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/portfolio-project-101/07/07-01-diagram.ko.png)
*Portfolio Project 101 7장 흐름 개요*
> 기술 선택의 기록은 '왜 이 도구인가'를 설명합니다. 설명이 없으면 유행만 따라가는 것처럼 보이고, 설명이 있으면 판단 능력을 보여 줍니다.

## 먼저 던지는 질문

- 기술 선택의 이유는 왜 코드만으로 충분히 드러나지 않을까요?
- 대안 비교와 결과 기록은 어떤 설득력을 더할까요?
- ADR은 얼마나 가볍게 시작해도 의미가 있을까요?

## 왜 중요한가

채용에서는 결과만큼 판단을 봅니다. 같은 기능을 만든 두 프로젝트가 있을 때, 왜 그 기술을 골랐고 어떤 한계를 받아들였는지 적어 둔 쪽이 훨씬 더 깊게 읽힙니다. 실무는 늘 제한된 시간, 익숙한 기술, 유지보수 비용 같은 조건 안에서 선택하는 일이기 때문입니다.

기술적 의사결정 기록은 이 과정을 문서로 남깁니다. 나중에 스스로 다시 봐도 도움이 되고, 다른 사람이 저장소를 검토할 때도 프로젝트가 단순 결과물이 아니라 사고 과정이 있는 작업이었다는 점을 보여 줍니다.

## 머릿속에 먼저 그릴 그림

결정 기록은 상황, 대안, 선택, 결과, 다음 변화 가능성으로 이어집니다.

이 흐름이 중요한 이유는 결정이 항상 맥락 안에서 읽혀야 하기 때문입니다. FastAPI를 골랐다는 사실 자체보다, 왜 그 시점에 그 선택이 합리적이었는지가 더 중요합니다. 기록은 바로 그 맥락을 붙여 줍니다.

## 핵심 용어

- **ADR**: 아키텍처 또는 기술 결정을 간단히 남기는 기록 형식입니다.
- 상황: 선택이 일어난 배경 조건입니다.
- 대안: 실제로 검토한 후보 기술이나 접근 방식입니다.
- 결정: 최종적으로 택한 선택입니다.
- 결과: 그 선택으로 얻은 점과 감수한 점입니다.

## 바꾸기 전과 후

**Before**: "그냥 그렇게 만들었습니다"라는 말만 남고, 왜 그런 선택이 나왔는지 보이지 않습니다.

**After**: 어떤 제약과 대안을 거쳐 그런 선택을 했는지 문서로 남아 있습니다.

후자의 프로젝트는 기술 스택 자체보다 판단 과정이 더 잘 보입니다. 포트폴리오에서 강한 부분도 바로 여기입니다. 실무는 늘 이유 있는 선택의 연속이기 때문입니다.

## 단계별로 살펴보기

### 1단계 — 상황

결정은 항상 배경 조건 안에서 읽혀야 합니다.

```python
context = "solo dev, 2-week deadline, Python familiar"
```

혼자 작업하는지, 마감이 촉박한지, 어떤 기술에 익숙한지에 따라 최선의 선택은 달라집니다. 상황을 먼저 적어 두면 결정이 훨씬 현실적으로 읽힙니다.

### 2단계 — 대안

비교가 있어야 선택의 무게가 생깁니다.

```python
options = ["FastAPI", "Flask", "Django"]
```

대안은 많을 필요가 없습니다. 다만 최소 두 개 이상 비교해야 왜 한 기술이 더 적합했는지 설명할 수 있습니다.

### 3단계 — 결정

최종 선택은 짧고 분명하게 적습니다.

```python
decision = "FastAPI"
```

결정 자체는 짧아도 됩니다. 중요한 것은 이 선택이 상황과 대안 비교 뒤에 놓인다는 점입니다. 그래야 결과만 남는 문서가 아니라 판단 이력이 됩니다.

### 4단계 — 근거

선택 이유는 취향보다 기준으로 적는 편이 좋습니다.

```python
why = ["async", "type_hints", "swagger_auto"]
```

이유를 기준으로 적으면 나중에 다시 봐도 도움이 됩니다. 예를 들어 비동기 처리, 타입 힌트, 자동 문서화 같은 기준은 실제 요구와 연결되기 쉽습니다.

### 5단계 — 결과

좋았던 점과 아쉬운 점을 함께 남깁니다.

```python
result = {"build_time": "fast", "trade": "smaller_ecosystem"}
```

결정 기록이 강해지는 지점은 바로 여기입니다. 무슨 이점을 얻었는지뿐 아니라, 무엇을 포기했는지도 적어 둬야 판단이 정직하게 읽힙니다.

## 이 코드에서 먼저 볼 점

- 상황을 맨 앞에 두면 선택의 배경이 먼저 보입니다.
- 대안 비교가 있어야 결정이 설득력을 가집니다.
- 결과를 솔직하게 적을수록 판단 기록이 강해집니다.

## 자주 하는 실수

1. 대안 비교 없이 결론만 적는 경우
2. 이유가 유행이나 취향 수준에서 멈추는 경우
3. 선택 이후 결과를 남기지 않아 평가가 불가능한 경우
4. ADR을 저장소 바깥에 두어 함께 읽기 어려운 경우
5. 번호나 일자를 정리하지 않아 기록 흐름이 흐트러지는 경우

결정 기록은 길게 쓰는 문서가 아닙니다. 중요한 것은 빠짐없이 남기는 것입니다. 상황, 대안, 선택, 이유, 결과만 분명하면 짧아도 충분히 강합니다.

## 실무에서는 이렇게 본다

실무 팀도 `docs/adr/0001-...md` 같은 형식으로 가볍게 결정을 남기는 경우가 많습니다. 이유는 단순합니다. 시간이 지나면 왜 그런 선택을 했는지 다들 잊기 쉽고, 새로 들어온 사람은 결과만 보고 배경을 알기 어렵기 때문입니다.

실제로 많은 오픈소스 프로젝트가 ADR을 공개 저장소에 포함합니다. Next.js, Babel, Kubernetes 같은 프로젝트도 RFC나 ADR 형태로 기술 결정을 공개합니다. 개인 프로젝트에서도 같은 형식을 따르면 "팀 규모와 무관하게 엔지니어링 습관이 갖춰진 사람"이라는 인상을 줍니다.

개인 포트폴리오에서도 이 습관은 크게 도움이 됩니다. 프로젝트가 작아도 선택 이유가 남아 있으면 단순 구현 사례가 아니라 판단 사례로 읽히기 시작합니다.

### ADR과 README의 연결

ADR이 저장소 깊은 곳에 숨어 있으면 리뷰어가 발견하지 못합니다. README에서 연결해 줍니다.

```markdown
## 기술 의사결정

| # | 제목 | 상태 | 날짜 |
| --- | --- | --- | --- |
| 1 | [백엔드 프레임워크](./docs/adr/0001-backend-framework.md) | Accepted | 2026-05-01 |
| 2 | [데이터베이스](./docs/adr/0002-database-choice.md) | Accepted | 2026-05-03 |
| 3 | [캐시 전략](./docs/adr/0003-cache-strategy.md) | Superseded | 2026-05-10 |
| 7 | [DB 인덱스 튜닝](./docs/adr/0007-db-index-tuning.md) | Accepted | 2026-05-18 |
```

이 표를 README의 "프로젝트 구조" 섹션 마지막에 넣으면, 저장소를 처음 보는 사람도 "이 사람은 기술 선택을 구조적으로 하는구나"라는 인상을 받습니다. 특히 칄용 담당자나 시니어 개발자가 리뷰할 때 판단력을 확인하는 첨경이 됩니다.

또한 이 표는 ADR 간 상태 변화를 한눈에 보여 줍니다. 3번 캐시 전략이 Superseded로 표시되어 있으면 "이 프로젝트는 초기 결정을 재검토하고 개선한 이력이 있다"는 메시지를 전달합니다. 이것이 단순 구현자와 설계자의 차이입니다.

## 체크리스트

- [ ] ADR을 둘 폴더를 정했다.
- [ ] 기술 선택 세 가지 이상에 대해 짧게라도 기록했다.
- [ ] 상황, 대안, 결정, 결과를 모두 적었다.
- [ ] 번호나 날짜로 기록 순서를 관리한다.

## 연습 문제

1. 지금 프로젝트에서 남겨야 할 기술 선택 하나를 골라 보세요.
2. 그 선택의 대안 두 가지를 적어 보세요.
3. 선택으로 얻은 점과 감수한 점을 각각 한 줄씩 적어 보세요.

## 정리와 다음 글

기술적 의사결정 기록은 결과 뒤에 숨은 판단을 드러내는 문서입니다. 상황을 적고, 대안을 비교하고, 선택 이유와 결과를 함께 남기면 작은 프로젝트도 훨씬 깊게 읽힙니다. 포트폴리오에서 중요한 것은 정답을 맞히는 일이 아니라, 왜 그런 선택을 했는지 설명할 수 있는 상태입니다.

다음 글에서는 이렇게 정리한 프로젝트를 블로그 글로 확장해 검색과 발견 가능성까지 넓히는 방법을 봅니다.

### 의사결정 흐름도 — 언제 기록하고, 언제 넘어가는가

모든 기술 선택을 ADR로 남길 필요는 없습니다. 아래 흐름도로 판단합니다.

```text
선택이 필요한 시점
  │
  ├─ 대안이 두 개 이상인가?
  │   ├─ 예 → 트레이드오프가 있는가?
  │   │        ├─ 예 → ADR 작성 ✅
  │   │        └─ 아니오 → 간단히 코드 주석으로 기록
  │   └─ 아니오 → 기록 불필요 (유일한 선택지)
  │
  └─ 나중에 바꿠 가능성이 있는가?
      ├─ 예 → ADR 작성 ✅ (미래 참조용)
      └─ 아니오 → 코드 주석으로 충분
```

이 흐름도를 따르면 불필요한 문서를 남기는 일을 피하면서도 중요한 결정은 빠짐없이 기록할 수 있습니다.

### 데이터베이스 선택 ADR 예시

데이터베이스 선택은 프로젝트 초기에 거의 항상 내리는 결정입니다. 아래 예시는 소규모 프로젝트에서 SQLite를 선택한 근거를 기록합니다.

```markdown
# ADR-0002: SQLite 채택

## Status
Accepted

## Context
- 1인 개발, 초기 데이터 규모 1000건 미만
- 별도 DB 서버 운영 부담 최소화 필요
- 테스트 시 외부 의존성 제거 원함

## Options
1) PostgreSQL (Supabase free tier)
2) SQLite (file-based)
3) MongoDB Atlas (free tier)

## Decision
SQLite를 기본 DB로 사용한다.

## Rationale
- 테스트 환경에서 외부 네트워크 불필요
- 로컬 개발 시 설정 제로
- SQLAlchemy를 쓰면 향후 PostgreSQL 전환 용이

## Consequences
- 장점: 배포 단순, 테스트 고립성, 비용 제로
- 단점: 동시 접속 제한, 구모 커지면 전환 필수
- 후속 조건: 사용자 100명 이상 시 PostgreSQL 전환 검토 (ADR-0008 예정)
```

이 ADR의 강점은 "후속 조건"을 명시한 점입니다. 언제 이 결정을 재검토할지 미리 적어 두면, 프로젝트가 성장할 때 판단 근거가 됩니다.

### ADR을 GitHub PR과 연결하기

ADR을 독립 문서로만 남기면 나중에 "이 결정이 언제 적용됐는지" 추적하기 어렵습니다. PR과 연결하면 해결됩니다.

```markdown
## PR 템플릿 예시 (.github/PULL_REQUEST_TEMPLATE.md)

### 변경 요약
- 

### 관련 ADR
- [ ] 새로운 기술 결정이 포함됨 → ADR 추가됨
- [ ] 기존 ADR 변경 → 상태 업데이트됨
- [ ] 해당 없음

### 테스트
- [ ] 단위 테스트 통과
- [ ] 통합 테스트 통과
```

이렇게 하면 PR 리뷰 시 "이 변경의 근거가 뭐지?" 라는 질문에 바로 답할 수 있습니다. 혼자 작업하더라도 이 습관을 들이면 나중에 팀 환경에서 자연스럽게 동작합니다.

### 기술 스택 선정 의사결정 표

기술 선택을 설득력 있게 보이게 하려면 후보와 기준을 같은 테이블에서 비교해야 합니다. 아래 표는 포트폴리오 프로젝트에서 자주 등장하는 백엔드 프레임워크 선택 예시입니다.

| 기준 | FastAPI | Flask | Django |
| --- | --- | --- | --- |
| 학습 곡선 | 중간 | 낮음 | 중간~높음 |
| 문서/타입 지원 | 강함 | 보통 | 강함 |
| 빠른 MVP | 강함 | 강함 | 보통 |
| 내장 기능 | 보통 | 낮음 | 매우 강함 |
| 이번 프로젝트 적합도 | 높음 | 중간 | 낮음 |

이 표는 "정답 프레임워크"를 찾기 위한 것이 아니라 "이번 제약에서 가장 합리적인 선택"을 설명하기 위한 도구입니다. 같은 기술도 마감, 인력, 운영 요구가 달라지면 결론이 달라질 수 있습니다.

### ADR 예시 문서(실전 축약형)

아래는 저장소에 바로 넣을 수 있는 ADR 예시입니다. 파일명은 `docs/adr/0001-backend-framework.md`처럼 번호를 붙이면 관리가 쉽습니다.

```markdown
# ADR-0001: FastAPI 채택

## Status
Accepted

## Context
- 2주 내 데모 완성 필요
- Python 기반 빠른 API 개발 필요
- 자동 문서화와 타입 힌트 활용 필요

## Options
1) Flask
2) FastAPI
3) Django

## Decision
FastAPI를 채택한다.

## Consequences
- 장점: OpenAPI 자동 생성, 비동기 처리, 타입 기반 개발
- 단점: 팀 레퍼런스가 Flask 대비 적음
```

이 정도 분량이면 과하지 않으면서도 판단 근거를 충분히 보여 줄 수 있습니다.

### 의사결정 기록 운영 규칙

ADR은 한 번 쓰고 끝나는 문서가 아니라 업데이트되는 기록입니다. 아래 규칙을 정해 두면 문서 신뢰도를 유지하기 쉽습니다.

1. 결정 직후 24시간 안에 초안 작성
2. "왜"가 없는 결정은 임시 상태로 표시
3. 변경 시 새 ADR 추가(덮어쓰기 금지)
4. 결과 검증 수치(성능/개발속도)를 후속 ADR에 연결

```markdown
## 예시: 상태 변경 기록
- ADR-0003: Redis 캐시 도입 (Accepted)
- ADR-0007: Redis 제거 및 DB 인덱스 튜닝으로 전환 (Superseded)
```

이런 기록이 있으면 프로젝트는 단순 구현 모음이 아니라 설계 이력이 있는 결과물로 읽힙니다. 면접에서도 "선택을 바꾼 경험"을 구조적으로 설명할 수 있습니다.

### ADR 품질 점검 체크리스트

ADR은 작성 자체보다 품질 관리가 중요합니다. 아래 질문에 모두 "예"라고 답할 수 있으면 충분히 강한 기록입니다.

| 점검 질문 | 예/아니오 |
| --- | --- |
| 이 결정을 내린 시점의 제약이 적혀 있는가 |  |
| 최소 두 개 이상의 대안을 비교했는가 |  |
| 선택 기준이 취향이 아닌 요구사항에 연결되는가 |  |
| 선택의 장단점을 모두 기록했는가 |  |
| 추후 변경 시 참조할 수 있는 지표가 있는가 |  |

### ADR 번호 체계 예시

```markdown
docs/adr/
- 0001-backend-framework.md
- 0002-database-choice.md
- 0003-cache-strategy.md
- 0004-deployment-platform.md
```

번호 체계를 고정하면 프로젝트가 커져도 기록 추적이 쉬워집니다. 특히 면접에서 "의사결정 흐름"을 시간순으로 설명할 때 큰 도움이 됩니다.

### ADR 작성 타이밍과 트리거

ADR은 "큰 결정"만 기록하는 것이 아닙니다. 아래 상황 중 하나라도 해당하면 기록할 가치가 있습니다.

| 트리거 | 예시 | ADR 제목 예시 |
| --- | --- | --- |
| 프레임워크/라이브러리 선택 | FastAPI vs Flask | `0001-backend-framework.md` |
| 데이터 저장소 결정 | PostgreSQL vs SQLite | `0002-database-choice.md` |
| 인증 방식 결정 | JWT vs Session | `0003-auth-strategy.md` |
| 배포 플랫폼 선택 | Railway vs Fly.io | `0004-deployment-platform.md` |
| 아키텍처 패턴 선택 | 모놀리스 vs 서비스 분리 | `0005-architecture-style.md` |
| 테스트 전략 결정 | E2E 우선 vs Unit 우선 | `0006-test-strategy.md` |
| 외부 서비스 연동 결정 | Stripe vs 자체 결제 | `0007-payment-integration.md` |

포트폴리오 프로젝트라면 보통 3~5개의 ADR이면 충분합니다. 핵심은 "선택이 있었던 곳"을 빠짐없이 포착하는 것입니다.

### ADR 간 관계 표현

프로젝트가 진행되면 이전 결정을 바꾸는 일이 생깁니다. 이때 기존 ADR을 수정하지 말고 새 ADR을 추가한 뒤 상태를 업데이트합니다.

```markdown
# ADR-0003: Redis 캐시 도입

## Status
Superseded by ADR-0007

## Context
- API 응답 시간이 평균 800ms로 느려짐
- 반복 쿼리 비율이 전체의 60%

## Decision
Redis를 캐시 레이어로 도입한다.

## Consequences
- 장점: 반복 쿼리 응답 시간 120ms로 단축
- 단점: 인프라 복잡도 증가, 무료 티어 용량 제한
---
# ADR-0007: Redis 제거 및 DB 인덱스 튜닝

## Status
Accepted (Supersedes ADR-0003)

## Context
- Redis 무료 티어 만료 후 비용 발생
- 실제 병목은 인덱스 미설정이 원인

## Decision
Redis를 제거하고 PostgreSQL 인덱스 + 쿼리 최적화로 전환한다.

## Consequences
- 장점: 인프라 단순화, 비용 제거, 응답 시간 150ms 유지
- 단점: 캐시 히트율 높은 시나리오에서 Redis보다 느림
```

이처럼 결정의 변천사가 기록에 남으면 "처음부터 완벽한 선택"을 했는지가 아니라 "상황 변화에 따라 합리적으로 적응했는지"를 보여 줄 수 있습니다. 면접에서 가장 좋은 이야기는 "바꾼 경험"입니다.

### 프론트엔드 기술 선택 ADR 예시

백엔드만 ADR을 남기는 것이 아닙니다. 프론트엔드 선택도 동일한 구조로 기록합니다.

```markdown
# ADR-0004: React 채택

## Status
Accepted

## Context
- 대시보드 형태 UI, 컴포넌트 재사용 빈번
- 팀원(본인) React 경험 있음
- SEO 불필요 (관리자 전용 화면)

## Options
1) React + Vite
2) Vue 3 + Vite
3) Next.js (SSR)

## Decision
React + Vite를 채택한다.

## Rationale
- SSR 불필요하므로 Next.js 오버헤드 불필요
- Vue 대비 기존 경험으로 개발 속도 확보
- Vite 번들러로 개발 서버 속도 확보

## Consequences
- 장점: 빠른 개발, HMR 쾌적, 컴포넌트 라이브러리 풍부
- 단점: 상태 관리 라이브러리 추가 선택 필요
```

### 의사결정 기록을 면접에서 활용하는 방법

ADR을 작성하는 것만으로는 부족합니다. 면접에서 이 기록을 어떻게 활용할지 미리 준비해야 합니다.

**STAR 프레임워크로 변환하기:**

| STAR 요소 | ADR 대응 섹션 | 말하기 예시 |
| --- | --- | --- |
| Situation | Context | "2주 마감, 혼자 개발, Python 기반" |
| Task | (암묵적) | "빠르게 API를 만들고 문서화까지 해야 했습니다" |
| Action | Decision + Rationale | "FastAPI를 선택했습니다. 이유는..." |
| Result | Consequences | "자동 문서화로 프론트엔드 협업 시간 50% 단축" |

**면접 답변 템플릿:**

```text
[상황] 이 프로젝트에서 {기술 A}를 선택해야 하는 시점이 있었습니다.
[비교] {기술 B}와 {기술 C}도 검토했는데, 각각 {장단점}이 있었습니다.
[결정] 최종적으로 {기술 A}를 택한 이유는 {기준 1}, {기준 2} 때문입니다.
[결과] 결과적으로 {구체적 수치/효과}를 얻었고, {트레이드오프}는 감수했습니다.
[회고] 다시 선택한다면 {같은/다른 선택}을 할 것입니다. 이유는 {근거}입니다.
```

이 구조로 말하면 기술 깊이와 커뮤니케이션 능력을 동시에 보여 줄 수 있습니다.

### ADR 디렉터리 자동화

ADR 번호를 수동으로 관리하면 실수하기 쉽습니다. 간단한 셸 스크립트로 자동화할 수 있습니다.

```bash
#!/bin/bash
# scripts/new-adr.sh
NEXT=$(ls docs/adr/ | grep -c '.md')
NEXT=$((NEXT + 1))
PADDED=$(printf "%04d" $NEXT)
TITLE=${1:-untitled}
FILE="docs/adr/${PADDED}-${TITLE}.md"

cat > "$FILE" << 'EOF'
# ADR-${PADDED}: ${TITLE}

## Status
Proposed

## Context
- (배경 조건을 적으세요)

## Options
1) 
2) 

## Decision
(결정 내용)

## Consequences
- 장점: 
- 단점: 
EOF

echo "Created: $FILE"
```

사용법: `./scripts/new-adr.sh database-choice`

이 스크립트를 저장소에 포함하면 ADR 작성 허들이 낮아지고, 번호 충돌도 방지됩니다.

### 의사결정 기록이 없는 프로젝트 vs 있는 프로젝트

같은 기술 스택을 사용한 두 포트폴리오를 비교해 봅니다.

| 관점 | ADR 없음 | ADR 있음 |
| --- | --- | --- |
| 리뷰어 첫인상 | "FastAPI 썼네" | "왜 FastAPI인지 알겠다" |
| 기술 깊이 판단 | 코드에서 추측 | 문서에서 확인 |
| 트레이드오프 인식 | 불분명 | 명시적 |
| 변경 이력 | git log만 | 의사결정 흐름 추적 가능 |
| 면접 질문 대응 | 기억에 의존 | 문서 기반 준비 가능 |

ADR이 있는 프로젝트는 "이 사람은 기술 선택을 습관적으로 검토하는 개발자"라는 인상을 줍니다. 주니어 포지션에서 이 습관이 보이면 성장 가능성을 높게 평가받습니다.

## 처음 질문으로 돌아가기

- **기술 선택의 이유는 왜 코드만으로 충분히 드러나지 않을까요?**
  - 코드는 "무엇을 만들었는가"만 보여 줍니다. "왜 이것을 골랐는가"는 결정 기록이 없으면 사라집니다. 리뷰어는 결과보다 판단 과정을 더 궁금해합니다.
- **대안 비교와 결과 기록은 어떤 설득력을 더할까요?**
  - 대안이 있어야 선택에 무게가 생깁니다. 비교 없이 "그냥 골랐다"고 하면 설득력이 없지만, 두세 개 후보와 기준을 함께 보여 주면 판단 능력이 드러납니다.
- **ADR은 얼마나 가볍게 시작해도 의미가 있을까요?**
  - 상황, 대안, 결정, 결과 네 줄만 있어도 됩니다. 분량이 아니라 구조가 중요합니다. 짧아도 트레이드오프를 인식했다는 사실 자체가 강한 증거입니다.
