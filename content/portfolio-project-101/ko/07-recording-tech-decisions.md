# Portfolio Project 101 (7/10): 기술적 의사결정 기록

포트폴리오 프로젝트를 검토할 때 구현 결과만큼 궁금한 것이 있습니다. 왜 그 기술을 골랐는가, 어떤 대안을 비교했는가, 무엇을 얻고 무엇을 포기했는가 같은 질문입니다. 코드만 보면 결과는 보이지만 판단은 잘 보이지 않습니다. 그래서 기술적 의사결정 기록이 필요합니다.

이 글은 Portfolio Project 101 시리즈의 7번째 글입니다. 여기서는 가벼운 ADR 형식으로 기술 선택의 이유를 남기는 방법을 살펴보고, 왜 이 기록이 포트폴리오에서 판단력을 보여 주는 강한 근거가 되는지 정리해 보겠습니다.

## 먼저 던지는 질문

- 기술 선택의 이유는 왜 코드만으로 충분히 드러나지 않을까요?
- 대안 비교와 결과 기록은 어떤 설득력을 더할까요?
- ADR은 얼마나 가볍게 시작해도 의미가 있을까요?

## 큰 그림

![Portfolio Project 101 7장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/portfolio-project-101/07/07-01-diagram.ko.png)

*Portfolio Project 101 7장 흐름 개요*

이 그림에서는 기술 선택이 프로젝트의 어느 지점부터 의사결정으로 기록되어야 하는지 봅니다. 선택의 이유를 남기는 것이 미래 검토의 근거가 됩니다.

> 기술 선택의 기록은 '왜 이 도구인가'를 설명합니다. 설명이 없으면 유툦만 따라가는 것처럼 보이고, 설명이 있으면 판단 능력을 보여 줍니다.

## 왜 중요한가

채용에서는 결과만큼 판단을 봅니다. 같은 기능을 만든 두 프로젝트가 있을 때, 왜 그 기술을 골랐고 어떤 한계를 받아들였는지 적어 둔 쪽이 훨씬 더 깊게 읽힙니다. 실무는 늘 제한된 시간, 익숙한 기술, 유지보수 비용 같은 조건 안에서 선택하는 일이기 때문입니다.

기술적 의사결정 기록은 이 과정을 문서로 남깁니다. 나중에 스스로 다시 봐도 도움이 되고, 다른 사람이 저장소를 검토할 때도 프로젝트가 단순 결과물이 아니라 사고 과정이 있는 작업이었다는 점을 보여 줍니다.

## 머릿속에 먼저 그릴 그림

결정 기록은 상황, 대안, 선택, 결과, 다음 변화 가능성으로 이어집니다.

이 흐름이 중요한 이유는 결정이 항상 맥락 안에서 읽혀야 하기 때문입니다. FastAPI를 골랐다는 사실 자체보다, 왜 그 시점에 그 선택이 합리적이었는지가 더 중요합니다. 기록은 바로 그 맥락을 붙여 줍니다.

## 핵심 용어

- **ADR**: 아키텍처 또는 기술 결정을 간단히 남기는 기록 형식입니다.
- 상황: 선택이 일어난 배경 조건입니다.
- 대안: 실제로 검토한 후보 기술이나 접근 방식입니다.
- 결정: 최종적으로 택한 선택입니다.
- 결과: 그 선택으로 얻은 점과 감수한 점입니다.

## 바꾸기 전과 후

**Before**: "그냥 그렇게 만들었습니다"라는 말만 남고, 왜 그런 선택이 나왔는지 보이지 않습니다.

**After**: 어떤 제약과 대안을 거쳐 그런 선택을 했는지 문서로 남아 있습니다.

후자의 프로젝트는 기술 스택 자체보다 판단 과정이 더 잘 보입니다. 포트폴리오에서 강한 부분도 바로 여기입니다. 실무는 늘 이유 있는 선택의 연속이기 때문입니다.

## 단계별로 살펴보기

### 1단계 — 상황

결정은 항상 배경 조건 안에서 읽혀야 합니다.

```python
context = "solo dev, 2-week deadline, Python familiar"
```

혼자 작업하는지, 마감이 촉박한지, 어떤 기술에 익숙한지에 따라 최선의 선택은 달라집니다. 상황을 먼저 적어 두면 결정이 훨씬 현실적으로 읽힙니다.

### 2단계 — 대안

비교가 있어야 선택의 무게가 생깁니다.

```python
options = ["FastAPI", "Flask", "Django"]
```

대안은 많을 필요가 없습니다. 다만 최소 두 개 이상 비교해야 왜 한 기술이 더 적합했는지 설명할 수 있습니다.

### 3단계 — 결정

최종 선택은 짧고 분명하게 적습니다.

```python
decision = "FastAPI"
```

결정 자체는 짧아도 됩니다. 중요한 것은 이 선택이 상황과 대안 비교 뒤에 놓인다는 점입니다. 그래야 결과만 남는 문서가 아니라 판단 이력이 됩니다.

### 4단계 — 근거

선택 이유는 취향보다 기준으로 적는 편이 좋습니다.

```python
why = ["async", "type_hints", "swagger_auto"]
```

이유를 기준으로 적으면 나중에 다시 봐도 도움이 됩니다. 예를 들어 비동기 처리, 타입 힌트, 자동 문서화 같은 기준은 실제 요구와 연결되기 쉽습니다.

### 5단계 — 결과

좋았던 점과 아쉬운 점을 함께 남깁니다.

```python
result = {"build_time": "fast", "trade": "smaller_ecosystem"}
```

결정 기록이 강해지는 지점은 바로 여기입니다. 무슨 이점을 얻었는지뿐 아니라, 무엇을 포기했는지도 적어 둬야 판단이 정직하게 읽힙니다.

## 이 코드에서 먼저 볼 점

- 상황을 맨 앞에 두면 선택의 배경이 먼저 보입니다.
- 대안 비교가 있어야 결정이 설득력을 가집니다.
- 결과를 솔직하게 적을수록 판단 기록이 강해집니다.

## 자주 하는 실수

1. 대안 비교 없이 결론만 적는 경우
2. 이유가 유행이나 취향 수준에서 멈추는 경우
3. 선택 이후 결과를 남기지 않아 평가가 불가능한 경우
4. ADR을 저장소 바깥에 두어 함께 읽기 어려운 경우
5. 번호나 일자를 정리하지 않아 기록 흐름이 흐트러지는 경우

결정 기록은 길게 쓰는 문서가 아닙니다. 중요한 것은 빠짐없이 남기는 것입니다. 상황, 대안, 선택, 이유, 결과만 분명하면 짧아도 충분히 강합니다.

## 실무에서는 이렇게 본다

실무 팀도 `docs/adr/0001-...md` 같은 형식으로 가볍게 결정을 남기는 경우가 많습니다. 이유는 단순합니다. 시간이 지나면 왜 그런 선택을 했는지 다들 잊기 쉽고, 새로 들어온 사람은 결과만 보고 배경을 알기 어렵기 때문입니다.

개인 포트폴리오에서도 이 습관은 크게 도움이 됩니다. 프로젝트가 작아도 선택 이유가 남아 있으면 단순 구현 사례가 아니라 판단 사례로 읽히기 시작합니다.

## 정리와 다음 글

기술적 의사결정 기록은 결과 뒤에 숨은 판단을 드러내는 문서입니다. 상황을 적고, 대안을 비교하고, 선택 이유와 결과를 함께 남기면 작은 프로젝트도 훨씬 깊게 읽힙니다. 포트폴리오에서 중요한 것은 정답을 맞히는 일이 아니라, 왜 그런 선택을 했는지 설명할 수 있는 상태입니다.

다음 글에서는 이렇게 정리한 프로젝트를 블로그 글로 확장해 검색과 발견 가능성까지 넓히는 방법을 살펴보겠습니다.

### 기술 스택 선정 의사결정 표

기술 선택을 설득력 있게 보이게 하려면 후보와 기준을 같은 테이블에서 비교해야 합니다. 아래 표는 포트폴리오 프로젝트에서 자주 등장하는 백엔드 프레임워크 선택 예시입니다.

| 기준 | FastAPI | Flask | Django |
| --- | --- | --- | --- |
| 학습 곡선 | 중간 | 낮음 | 중간~높음 |
| 문서/타입 지원 | 강함 | 보통 | 강함 |
| 빠른 MVP | 강함 | 강함 | 보통 |
| 내장 기능 | 보통 | 낮음 | 매우 강함 |
| 이번 프로젝트 적합도 | 높음 | 중간 | 낮음 |

이 표는 "정답 프레임워크"를 찾기 위한 것이 아니라 "이번 제약에서 가장 합리적인 선택"을 설명하기 위한 도구입니다. 같은 기술도 마감, 인력, 운영 요구가 달라지면 결론이 달라질 수 있습니다.

### ADR 예시 문서(실전 축약형)

아래는 저장소에 바로 넣을 수 있는 ADR 예시입니다. 파일명은 `docs/adr/0001-backend-framework.md`처럼 번호를 붙이면 관리가 쉽습니다.

```markdown
# ADR-0001: FastAPI 채택

## Status
Accepted

## Context
- 2주 내 데모 완성 필요
- Python 기반 빠른 API 개발 필요
- 자동 문서화와 타입 힌트 활용 필요

## Options
1) Flask
2) FastAPI
3) Django

## Decision
FastAPI를 채택한다.

## Consequences
- 장점: OpenAPI 자동 생성, 비동기 처리, 타입 기반 개발
- 단점: 팀 레퍼런스가 Flask 대비 적음
```

이 정도 분량이면 과하지 않으면서도 판단 근거를 충분히 보여 줄 수 있습니다.

### 의사결정 기록 운영 규칙

ADR은 한 번 쓰고 끝나는 문서가 아니라 업데이트되는 기록입니다. 아래 규칙을 정해 두면 문서 신뢰도를 유지하기 쉽습니다.

1. 결정 직후 24시간 안에 초안 작성
2. "왜"가 없는 결정은 임시 상태로 표시
3. 변경 시 새 ADR 추가(덮어쓰기 금지)
4. 결과 검증 수치(성능/개발속도)를 후속 ADR에 연결

```markdown
## 예시: 상태 변경 기록
- ADR-0003: Redis 캐시 도입 (Accepted)
- ADR-0007: Redis 제거 및 DB 인덱스 튜닝으로 전환 (Superseded)
```

이런 기록이 있으면 프로젝트는 단순 구현 모음이 아니라 설계 이력이 있는 결과물로 읽힙니다. 면접에서도 "선택을 바꾼 경험"을 구조적으로 설명할 수 있습니다.

### ADR 품질 점검 체크리스트

ADR은 작성 자체보다 품질 관리가 중요합니다. 아래 질문에 모두 "예"라고 답할 수 있으면 충분히 강한 기록입니다.

| 점검 질문 | 예/아니오 |
| --- | --- |
| 이 결정을 내린 시점의 제약이 적혀 있는가 |  |
| 최소 두 개 이상의 대안을 비교했는가 |  |
| 선택 기준이 취향이 아닌 요구사항에 연결되는가 |  |
| 선택의 장단점을 모두 기록했는가 |  |
| 추후 변경 시 참조할 수 있는 지표가 있는가 |  |

### ADR 번호 체계 예시

```markdown
docs/adr/
- 0001-backend-framework.md
- 0002-database-choice.md
- 0003-cache-strategy.md
- 0004-deployment-platform.md
```

번호 체계를 고정하면 프로젝트가 커져도 기록 추적이 쉬워집니다. 특히 면접에서 "의사결정 흐름"을 시간순으로 설명할 때 큰 도움이 됩니다.

### 품질 점검 기록 예시(반복 가능한 운영 습관)

포트폴리오 품질은 한 번의 대규모 수정으로 완성되지 않습니다. 짧은 주기로 같은 항목을 반복 점검할 때 품질이 안정됩니다. 아래 기록 방식은 프로젝트 크기와 무관하게 적용할 수 있습니다.

| 점검 항목 | 확인 질문 | 증거 형태 |
| --- | --- | --- |
| 문제 정의 | 문제 문장이 여전히 현재 구현과 일치하는가 | README/블로그 도입 문장 |
| 데모 신뢰성 | 링크가 열리고 핵심 흐름이 실제로 작동하는가 | 라이브 URL, 백업 영상 |
| 검증 체계 | 테스트와 CI 상태가 녹색인가 | CI 배지, 테스트 로그 |
| 문서 정합성 | 실행 방법과 환경 변수 설명이 최신인가 | README, `.env.example` |
| 설명 일관성 | 저장소/블로그/면접 답변이 같은 메시지를 말하는가 | 요약 문장 비교 |

이 표를 기준으로 주기적 점검을 하면 개선 우선순위를 빠르게 정할 수 있습니다. 예를 들어 데모는 정상이지만 문서 정합성이 낮다면 기능 추가보다 문서 보완이 먼저입니다. 반대로 문서는 완벽하지만 CI가 불안정하면 공개 신뢰를 잃기 쉽습니다.

다음 템플릿은 실제 기록에 바로 사용할 수 있습니다.

```markdown
## Weekly Portfolio Health Check
- 날짜: 2026-05-21
- 현재 상태: 데모 정상 / 문서 일부 불일치 / CI 정상
- 발견 이슈:
  1) README 실행 명령이 최신 compose 파일과 다름
  2) 블로그 성과 수치가 최근 측정값과 불일치
- 조치 계획:
  1) README 실행 섹션 업데이트
  2) 성과 표 재계산 및 캡처 갱신
- 완료 기준:
  - 신규 방문자가 5분 안에 문제/데모/실행/결과를 이해할 수 있음
```

핵심은 완벽함이 아니라 반복성입니다. 같은 기준으로 꾸준히 점검하면 프로젝트는 시간이 지나도 낡지 않습니다. 면접에서도 "어떻게 유지했는가"를 설명할 수 있어 결과물의 신뢰도가 더 높아집니다.

### 실행 가능한 개선 TODO 예시

아래 TODO는 "읽고 끝나는 조언"이 아니라 실제로 바로 적용할 수 있는 작업 단위입니다. 각 항목은 30~90분 안에 끝낼 수 있는 크기로 쪼개는 것을 권장합니다.

| 작업 | 예상 시간 | 완료 기준 |
| --- | ---: | --- |
| 핵심 메시지 1문장 재작성 | 30분 | README/발표/이력서 문장 일치 |
| 데모 검증 리허설 1회 | 45분 | 로그인-핵심행동-결과 흐름 무중단 |
| 테스트 명령 정리 | 30분 | 로컬/CI 명령이 문서와 일치 |
| 결과 수치 업데이트 | 60분 | 최신 측정값과 표/그래프 동기화 |

작업을 작은 단위로 나누면 진행률이 눈에 보이고, 품질 개선이 미뤄지지 않습니다. 포트폴리오를 오래 유지하려면 큰 개편보다 이런 짧은 루틴이 더 효과적입니다.

### 마무리 검증 메모

최종 공개 전에는 "내가 이해하는 프로젝트"가 아니라 "처음 보는 사람이 이해하는 프로젝트"인지 확인해야 합니다. 이를 위해 1) 문제를 한 문장으로 말해 보기, 2) 데모를 처음부터 끝까지 클릭해 보기, 3) 실행 명령을 새 환경에서 복사 실행해 보기, 4) 결과 수치를 다시 계산해 보기 네 단계를 반복하는 것이 좋습니다. 이 검증을 통과하면 문서와 데모와 코드의 정합성이 높아지고, 면접 답변도 훨씬 안정됩니다.

### 한 줄 점검

이 글의 실천 포인트를 실제 프로젝트에 바로 반영해, 다음 주기 점검에서 변경 전후 차이를 기록해 두면 품질 개선 효과를 더 명확하게 확인할 수 있습니다.


짧은 주기 점검 결과를 남기면 다음 개선 우선순위를 더 정확히 선택할 수 있습니다.

## 처음 질문으로 돌아가기

- **기술 선택의 이유는 왜 코드만으로 충분히 드러나지 않을까요?**
  - 프로젝트의 핸심 기술 3~5개만 선택하세요. 왜 그 프레임워크를 골라고, 무엇을 포기했으며, 다시 선택한다면 같은 선택을 할 것인지 적습니다.
- **대안 비교와 결과 기록은 어떤 설득력을 더할까요?**
  - ADR(Architecture Decision Record) 형식을 추천합니다. 결정, 선택지, 이유, 트레이드오프, 결과를 간단히 기록하면 명확합니다.
- **ADR은 얼마나 가볍게 시작해도 의미가 있을까요?**
  - 더 있습니다. 어떤 선택에든 트레이드오프가 있고, 그 트레이드오프를 인식했느냐가 중요합니다. 개발자의 판단력을 보여 주는 강력한 즜거입니다.
