# Portfolio Project 101 (6/10): 테스트와 문서화

포트폴리오 프로젝트가 잘 돌아간다고 말하는 것과, 실제로 검증된 상태라고 보여 주는 것은 다릅니다. 저장소를 열어 보았을 때 테스트가 하나도 없고 문서도 README 한 장으로 끝나면, 프로젝트는 구현 연습처럼 보이기 쉽습니다. 반대로 작은 프로젝트라도 기본 테스트와 문서가 있으면 결과물의 신뢰도가 크게 올라갑니다.

이 글은 Portfolio Project 101 시리즈의 6번째 글입니다. 여기서는 포트폴리오 프로젝트에서 어떤 수준의 테스트와 문서화가 있으면 충분히 믿을 만한 프로젝트로 읽히는지, 그리고 그 흔적이 왜 채용과 협업 관점에서 중요한지 살펴보겠습니다.

## 먼저 던지는 질문

- 단위 테스트, 통합 테스트, 전체 흐름 테스트는 각각 무엇을 증명할까요?
- 자동 검증은 왜 작은 포트폴리오에서도 중요한 기준이 될까요?
- API 문서나 변경 기록은 어떤 신뢰를 더해 줄까요?

## 큰 그림

![Portfolio Project 101 6장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/portfolio-project-101/06/06-01-diagram.ko.png)

*Portfolio Project 101 6장 흐름 개요*

이 그림에서는 테스트와 문서가 코드 신뢰도를 어떻게 높이는지 봅니다. 테스트 없는 코드는 기능이고, 테스트 있는 코드는 검증된 사례입니다.

> 테스트는 당신이 해 본 모든 검증을 자동으로 반복 실행하는 기록입니다. 문서는 다음 개발자(또는 당신)이 다시 이해할 수 있길 남기는 지도입니다.

## 왜 중요한가

테스트와 문서는 전문성을 보여 주는 가장 빠른 증거입니다. 특히 포트폴리오에서는 프로젝트가 우연히 한 번 돌아간 코드가 아니라 반복 검증된 결과물이어야 합니다. 테스트는 기능이 깨졌는지 알려 주고, 문서는 다른 사람이 프로젝트에 진입하도록 돕습니다.

채용 관점에서도 이 둘은 강한 신호입니다. 테스트가 있다는 것은 실패를 두려워하지 않고 검증 경로를 남겼다는 뜻이고, 문서가 있다는 것은 혼자만 이해하는 프로젝트로 두지 않았다는 뜻입니다. 두 흔적이 함께 있으면 규모가 작아도 훨씬 성숙한 결과물로 읽힙니다.

## 머릿속에 먼저 그릴 그림

검증은 보통 작은 단위에서 시작해 전체 흐름과 문서, 자동화로 확장됩니다.

이 순서를 기억해 두면 테스트를 무작정 늘리기보다 어디서부터 시작할지 판단하기 쉬워집니다. 빠르게 깨지는 단위 테스트가 있고, 경계면을 확인하는 통합 테스트가 있고, 핵심 사용자 흐름을 점검하는 전체 흐름 테스트가 있고, 그 모든 것을 다시 실행하게 만드는 문서와 자동화가 뒤따릅니다.

## 핵심 용어

- **단위 테스트**: 작은 함수나 로직 단위를 확인하는 테스트입니다.
- **통합 테스트**: API나 저장소 같은 경계면이 함께 동작하는지 확인하는 테스트입니다.
- **전체 흐름 테스트**: 실제 사용자 시나리오를 끝까지 따라가는 테스트입니다.
- **지속 통합**: 코드 변경마다 자동으로 검증을 실행하는 흐름입니다.
- 문서: README, API 문서, 변경 기록처럼 프로젝트 해석을 돕는 기록입니다.

## 바꾸기 전과 후

**Before**: 수동 확인에만 의존해서, 무엇이 깨졌는지 사람 기억에 기대야 합니다.

**After**: 코드 변경이 있을 때마다 자동 검증이 실행되고, 다른 사람도 문서를 따라 프로젝트를 이해할 수 있습니다.

후자의 프로젝트가 강한 이유는 반복 가능성 때문입니다. 한 번 성공한 작업보다 여러 번 같은 방식으로 확인할 수 있는 작업이 훨씬 더 믿을 만합니다.

## 단계별로 살펴보기

### 1단계 — 단위 테스트

가장 작은 단위에서 빠르게 실패를 잡습니다.

```python
def test_add():
    assert 1 + 1 == 2
```

이 예제는 단순하지만 의미는 분명합니다. 단위 테스트는 오류 위치를 빠르게 좁혀 주고, 작은 로직을 바꿀 때 가장 먼저 안전망 역할을 합니다.

### 2단계 — 통합 테스트

경계면은 실제로 함께 붙여 봐야 드러나는 문제가 많습니다.

```python
def test_api(client):
    assert client.get("/health").status_code == 200
```

API, 데이터베이스, 외부 의존성처럼 여러 조각이 만나는 지점은 단위 테스트만으로는 부족합니다. 최소한의 통합 테스트가 있어야 실제 서비스 경로가 어느 정도 검증됩니다.

### 3단계 — 전체 흐름 테스트

사용자 관점의 핵심 시나리오를 끝까지 확인합니다.

```python
e2e_steps = ["login", "create", "delete"]
```

전체 흐름 테스트는 많을 필요가 없습니다. 대신 가장 중요한 사용자 경로 하나라도 있으면 프로젝트가 어디까지 검증됐는지 훨씬 선명하게 보입니다.

### 4단계 — 지속 통합 설정

검증은 사람 기억보다 자동화에 맡기는 편이 낫습니다.

```yaml
on: [push]
jobs:
  test:
    runs-on: ubuntu-latest
```

지속 통합이 있으면 코드 변경이 생길 때마다 같은 기준으로 확인할 수 있습니다. 포트폴리오에서도 이 자동화는 꽤 강한 신호입니다. 반복 검증을 기본값으로 두고 있다는 뜻이기 때문입니다.

### 5단계 — 문서

테스트가 기계의 검증이라면 문서는 사람의 진입 경로입니다.

```python
docs = ["README", "API.md", "CHANGELOG.md"]
```

README는 입구를, API 문서는 사용법을, 변경 기록은 프로젝트가 어떻게 진화했는지를 보여 줍니다. 세 문서가 모두 완벽할 필요는 없지만, 최소한의 흔적은 남겨 두는 편이 좋습니다.

## 이 코드에서 먼저 볼 점

- 단위 테스트는 빠르게 깨지는 안전망입니다.
- 통합 테스트와 전체 흐름 테스트가 있어야 사용자 경로까지 확인됩니다.
- 지속 통합과 문서는 검증을 개인 기억에서 팀의 습관으로 바꿉니다.

## 자주 하는 실수

1. 단위 테스트만 두고 실제 사용자 흐름은 전혀 확인하지 않는 경우
2. 전체 흐름 테스트가 없어 배포 직전 핵심 경로를 검증하지 못하는 경우
3. 지속 통합이 없어 반복 검증이 사람 손에만 의존하는 경우
4. API 문서가 없어 다른 사람이 진입하기 어려운 경우
5. 변경 기록이 없어 프로젝트가 어떻게 달라졌는지 읽기 어려운 경우

테스트와 문서화는 화려한 부가 기능이 아닙니다. 프로젝트를 믿을 수 있게 만드는 기본 장치입니다. 이 둘이 비어 있으면 규모와 무관하게 결과물의 신뢰가 약해집니다.

## 실무에서는 이렇게 본다

오픈소스 프로젝트도 보통 푸시마다 기본 검증을 돌리고, 사용법과 변경 이력을 문서로 남깁니다. 팀 프로젝트에서는 더 중요합니다. 테스트와 문서가 없으면 지식이 사람 머릿속에만 남고, 프로젝트는 금방 다루기 어려워집니다.

개인 포트폴리오에서도 같은 기준이 통합니다. 작은 프로젝트라 해도 테스트 하나, 핵심 흐름 하나, README와 간단한 문서 몇 장이 있으면 훨씬 신뢰할 만한 결과물로 보입니다.

## 정리와 다음 글

포트폴리오 프로젝트에서 테스트와 문서화는 선택 사항이 아닙니다. 단위 테스트는 빠른 확인을, 통합과 전체 흐름 테스트는 실제 경로 검증을, 문서는 사람을 위한 진입 경로를 맡습니다. 여기에 지속 통합까지 더해지면 프로젝트는 우연히 되는 코드가 아니라 반복해서 검증되는 결과물로 읽힙니다.

다음 글에서는 이런 프로젝트에서 왜 그런 선택을 했는지 남기는 기술적 의사결정 기록을 살펴보겠습니다.

### 배포 환경 비교표(학습 프로젝트 기준)

테스트와 문서화 품질은 배포 환경과도 연결됩니다. 환경이 불안정하면 테스트 결과 신뢰도도 흔들리기 때문입니다. 아래 표는 포트폴리오 프로젝트에서 자주 쓰는 환경을 비교한 것입니다.

| 환경 | 장점 | 주의점 | 추천 상황 |
| --- | --- | --- | --- |
| 로컬 Docker Compose | 재현성 높음, 팀 공유 쉬움 | 초기 설정 필요 | 개발/통합 테스트 기본 환경 |
| Render/Railway | 배포 단순, 로그 확인 쉬움 | 무료 티어 슬립 가능 | 공개 데모 운영 |
| Fly.io | 컨테이너 친화, 운영 제어 폭 넓음 | 설정 러닝커브 | API 중심 데모 |
| Vercel(프론트) + API 분리 | 프론트 배포 빠름 | 백엔드 분리 운영 필요 | SPA 데모 프로젝트 |

환경 선택은 정답 문제가 아닙니다. 테스트 재현성과 문서 일관성을 유지할 수 있는지가 핵심 기준입니다.

### Docker 설정 예시(테스트 친화형)

테스트와 문서화를 강화하려면 컨테이너 시작 순서와 헬스체크를 명시해 두는 편이 좋습니다.

```yaml
services:
  db:
    image: postgres:16
    environment:
      POSTGRES_DB: app
      POSTGRES_USER: app
      POSTGRES_PASSWORD: app
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U app -d app"]
      interval: 5s
      timeout: 3s
      retries: 10

  api:
    build: .
    env_file: .env
    depends_on:
      db:
        condition: service_healthy
    command: bash -lc "pytest -q && uvicorn app.main:app --host 0.0.0.0 --port 8000"
```

이 구성의 장점은 "검증 후 실행"을 자동화한다는 점입니다. 테스트가 깨지면 서버가 떠도 배포 성공으로 간주하지 않게 만들 수 있습니다.

### 문서 세트 최소 기준

README 한 장으로는 테스트 전략과 운영 판단을 충분히 전달하기 어렵습니다. 아래 세 문서를 추가하면 프로젝트 신뢰도가 크게 올라갑니다.

1. `docs/testing.md` - 테스트 범위, 실행 명령, 실패 시 해석
2. `docs/runbook.md` - 장애 확인 순서, 롤백 절차, 로그 위치
3. `CHANGELOG.md` - 변경 이력과 사용자 영향

```markdown
## docs/testing.md 예시 섹션
- 테스트 피라미드 요약
- 로컬 실행 방법
- CI에서만 실행하는 테스트
- flaky 테스트 처리 규칙
```

이 문서 세트는 면접에서도 유용합니다. "테스트를 했습니다"를 넘어서 "어떤 실패를 어떻게 관리했는지"까지 설명할 수 있기 때문입니다.

### 테스트 전략 표(릴리스 전 기준)

포트폴리오에서도 릴리스 전 테스트 기준을 정해 두면 품질이 훨씬 안정됩니다.

| 단계 | 필수 테스트 | 통과 조건 |
| --- | --- | --- |
| 로컬 개발 | 단위 테스트 | 핵심 로직 전부 통과 |
| PR 검증 | 단위 + 통합 | CI 녹색 |
| 배포 직전 | 스모크 테스트 | 핵심 API 정상 응답 |
| 배포 직후 | 실사용 시나리오 | 로그인-조회-결과 확인 |

### 문서 업데이트 규칙

문서는 코드보다 늦게 업데이트되기 쉽습니다. 아래 규칙을 두면 문서 부채를 줄일 수 있습니다.

1. 사용자 행동이 바뀌는 변경은 README 동시 수정
2. API 스키마 변경은 `API.md` 동시 수정
3. 배포 방식 변경은 `runbook.md` 동시 수정
4. 릴리스마다 `CHANGELOG.md` 한 줄 추가

이 규칙은 작은 프로젝트에서도 매우 효과적입니다. 문서와 코드의 불일치를 줄이면 리뷰어 신뢰가 빠르게 올라갑니다.

팁 하나를 더 드리면, PR 템플릿에 "문서 업데이트 여부" 체크박스를 추가하면 리뷰어가 문서 누락을 자연스럽게 잡아줍니다. GitHub의 `.github/PULL_REQUEST_TEMPLATE.md`에 한 줄만 넣으면 됩니다.

### 프로젝트 문서 디렉터리 구조

테스트와 문서가 모두 갖춰진 프로젝트의 `docs/` 디렉터리는 다음과 같은 구조를 권장합니다.

```text
docs/
├── architecture.md      # 시스템 구조, 계층 설명, 데이터 흐름
├── testing.md           # 테스트 전략, 실행 방법, 커버리지 기준
├── api.md               # 엔드포인트, 요청/응답, 에러 코드
├── runbook.md           # 장애 대응 순서, 롤백, 로그 위치
└── decisions/           # ADR (기술적 의사결정 기록)
    ├── 001-database-choice.md
    └── 002-auth-strategy.md
```

각 문서의 역할을 정리하면:

| 문서 | 독자 | 핵심 질문 |
| --- | --- | --- |
| architecture.md | 신규 합류자 | "이 시스템은 어떻게 동작하나요?" |
| testing.md | 기여자 / 리뷰어 | "테스트를 어떻게 실행하나요?" |
| api.md | 프론트엔드 개발자 | "어떤 엔드포인트가 있나요?" |
| runbook.md | 운영자 | "장애 시 누가 뭘 하나요?" |
| decisions/ | 미래의 나 | "왜 이렇게 결정했나요?" |

포트폴리오 프로젝트에서 다섯 문서를 모두 작성할 필요는 없습니다. `testing.md`와 `architecture.md` 두 개만 있어도 충분히 차별화됩니다.

### docs/testing.md 전체 템플릿

```markdown
# Testing Strategy

## 테스트 피라미드

| 계층 | 대상 | 도구 | 실행 시점 |
| --- | --- | --- | --- |
| Unit | 순수 함수, 유틸리티 | pytest | 모든 커밋 |
| Integration | API 엔드포인트 | pytest + httpx | PR |
| E2E | 사용자 시나리오 | playwright | 배포 전 |

## 로컬 실행

```bash
# 전체 테스트
pytest

# 커버리지 포함
pytest --cov=src --cov-report=term-missing

# 특정 모듈만
pytest tests/test_scheduler.py -v
\```

## CI 테스트 설정

GitHub Actions에서 `pytest`를 자동 실행합니다.
PR이 녀색이면 merge하지 않습니다.

## Flaky 테스트 처리

- 네트워크 의존 테스트: mock으로 격리
- 시간 의존 테스트: freezegun 사용
- DB 의존 테스트: 트랜잭션 rollback 적용
```

이 템플릿을 그대로 복사해 사용해도 되고, 프로젝트에 맞게 수정해도 됩니다. 중요한 것은 "테스트가 있다"가 아니라 "테스트 전략이 있다"는 것을 보여 주는 것입니다.

### 테스트 보고서 템플릿

테스트 결과를 기록으로 남기면 시간이 지나도 품질 변화를 추적할 수 있습니다.

```markdown
## Test Report - 2024-03-15

| 지표 | 값 | 비고 |
| --- | --- | --- |
| 총 테스트 수 | 47 | +5 from last week |
| 통과 | 45 | |
| 실패 | 0 | |
| 스킵 | 2 | E2E - 외부 API 의존 |
| 커버리지 | 78% | 목표: 80% |
| 실행 시간 | 12s | CI 기준 |

### 실패 분석
- 없음

### 스킵 사유
- test_external_webhook: 외부 API rate limit (mock 전환 예정)
- test_email_send: SMTP 설정 필요 (CI 환경 변수 추가 예정)
```

이 보고서를 주간 단위로 남기면, 프로젝트가 활발히 관리되고 있다는 증거가 됩니다. 면접에서도 "커버리지를 60%에서 78%로 올렸습니다"라고 수치로 말할 수 있습니다.

### CI 배지를 README에 추가하는 방법

CI 배지는 작지만 강한 신호입니다. README 상단에 녹색 배지가 있으면 "이 프로젝트는 자동 검증되고 있다"는 메시지를 전달합니다.

```markdown
<!-- README.md 상단 -->
![CI](https://github.com/username/project/actions/workflows/ci.yml/badge.svg)
![Coverage](https://codecov.io/gh/username/project/branch/main/graph/badge.svg)
```

배지를 추가하려면:

1. GitHub Actions 워크플로 파일을 만듭니다 (`.github/workflows/ci.yml`)
2. GitHub 저장소 → Actions 탭 → 워크플로 선택 → "…" 메뉴 → "Create status badge"
3. 생성된 Markdown을 README 상단에 붙여넣기

커버리지 배지를 함께 넣으려면 Codecov나 Coveralls 같은 서비스를 연동합니다. 둑 다 무료 티어를 제공하므로 포트폴리오에 부담이 없습니다.

### conftest.py 패턴: 테스트 픽스처 구성

테스트 코드의 품질도 포트폴리오에서 평가됩니다. 잘 구성된 conftest.py는 테스트 설계 능력을 보여 줍니다.

```python
# tests/conftest.py
import pytest
from httpx import AsyncClient, ASGITransport
from src.main import app
from src.database import get_test_db, reset_db


@pytest.fixture(scope="session")
def anyio_backend():
    return "asyncio"


@pytest.fixture
async def client():
    """API 테스트용 클라이언트."""
    transport = ASGITransport(app=app)
    async with AsyncClient(transport=transport, base_url="http://test") as ac:
        yield ac


@pytest.fixture(autouse=True)
def reset_database():
    """각 테스트 전후 DB 초기화."""
    reset_db()
    yield
    reset_db()
```

이 픽스처 구성에서 핵심은:

1. **세션 범위 제어**: 전체 테스트에 한 번만 실행하는 설정과 테스트마다 실행하는 설정을 분리합니다.
2. **DB 격리**: 각 테스트가 독립적으로 실행되게 보장합니다.
3. **실제 API 테스트**: httpx의 ASGITransport로 실제 HTTP 요청을 시뮬레이션합니다.

### CHANGELOG 작성법

변경 이력은 프로젝트가 활발히 유지되고 있다는 가장 직관적인 증거입니다. [Keep a Changelog](https://keepachangelog.com/) 형식을 따르면 일관성을 유지할 수 있습니다.

```markdown
# Changelog

## [0.3.0] - 2024-03-15

### Added
- OAuth2 인증 지원 (Google, GitHub)
- 주간 리포트 자동 생성 기능

### Changed
- 데이터베이스 커넥션 풀 최적화 (5 → 10)
- API 응답 포맷을 snake_case로 통일

### Fixed
- 팀원 삭제 시 관련 작업이 미할당되는 버그 수정

## [0.2.0] - 2024-02-28

### Added
- 대시보드 기본 UI
- 시드 데이터 자동 생성 스크립트

### Fixed
- Docker Compose 네트워크 설정 오류
```

CHANGELOG에서 중요한 원칙:

1. **사용자 관점으로 작성**: "코드 리팩터링" 대신 "응답 속도 30% 개선"으로 작성합니다.
2. **버전 번호 사용**: 시맨틱 버전닝으로 변경 규모를 전달합니다.
3. **날짜 마크**: 프로젝트 활동성을 시각적으로 보여 줍니다.

### 테스트 작성 시 자주 하는 실수

| 실수 | 증상 | 해결책 |
| --- | --- | --- |
| 구현을 테스트 | 리팩터링시 테스트가 깨짐 | 행동/결과만 검증 |
| 테스트 간 의존성 | 순서 바꾸면 실패 | fixture로 격리 |
| 외부 API 직접 호출 | 네트워크 없으면 실패 | mock/stub 사용 |
| 매직 넘버 하드코딩 | 값 바꾸면 실패 | 변수/상수 분리 |
| 모든 것을 테스트 | 유지보수 비용 증가 | 핵심 로직만 집중 |

특히 포트폴리오에서는 "모든 것을 테스트"하는 것보다 "가장 중요한 것을 잘 테스트"하는 것이 더 나습니다. 스케줄러의 허피 시간 계산, 인증 토큰 검증, 결제 금액 계산 같은 핵심 로직에 테스트를 집중하면, 적은 수의 테스트로도 높은 신뢰를 줄 수 있습니다.

### 테스트와 문서를 함께 유지하는 워크플로

테스트와 문서는 각각 관리하면 금방 불일치가 생깁니다. PR 체크리스트에 함께 넣으면 자연스럽게 동기화됩니다.

```markdown
<!-- .github/pull_request_template.md -->
## PR Checklist
- [ ] 테스트 추가/수정 (새 기능 또는 버그픽스)
- [ ] 커버리지 감소 없음 (pytest --cov 확인)
- [ ] README 영향 여부 확인
- [ ] API 변경 시 docs/api.md 업데이트
- [ ] CHANGELOG 항목 추가
```

이 체크리스트가 있으면 "코드만 수정하고 문서를 잊는" 실수를 구조적으로 방지할 수 있습니다. 1인 프로젝트라도 PR 템플릿은 유용합니다. 미래의 나에게 보내는 리마인더이기 때문입니다.

### 테스트 없이 시작하는 최소 검증 전략

아직 테스트 코드를 작성하는 것이 부담스럽다면, 다음 순서로 점진적으로 시작할 수 있습니다.

1. **린터 도입** (5분): `ruff check .`만 추가해도 코드 품질 기본선이 생깁니다.
2. **타입 체크** (10분): `mypy src/`로 타입 오류를 잡으면 버그 예방 효과가 있습니다.
3. **스모크 테스트 1개** (15분): 핵심 API에 GET 요청을 보내 200을 확인하는 테스트 1개만 작성합니다.
4. **비즈니스 로직 테스트** (30분): 가장 중요한 계산/검증 함수를 테스트합니다.
5. **CI 연동** (20분): 위 단계를 GitHub Actions에서 자동 실행하게 설정합니다.

이 순서의 핵심은 "완벽한 테스트"가 아니라 "점진적인 검증 문화"입니다. 1단계만 해도 README에 "Lint: ruff"라고 쓸 수 있고, 3단계만 해도 CI 배지를 붙일 수 있습니다. 완벽함을 기다리는 것보다 작은 검증이라도 먼저 시작하는 편이 훨씬 낫습니다.

## 처음 질문으로 돌아가기

- **단위 테스트, 통합 테스트, 전체 흐름 테스트는 각각 무엇을 증명할까요?**
  - 단위 테스트는 함수 하나의 정확성을, 통합 테스트는 모듈 간 연결을, 전체 흐름 테스트는 사용자 시나리오 전체를 증명합니다. 핵심 로직 테스트 30%가 전체 커버리지 90%보다 더 강한 신호입니다.
- **자동 검증은 왜 작은 포트폴리오에서도 중요한 기준이 될까요?**
  - 자동 검증은 프로젝트가 우연히 돌아간 것이 아니라 반복 검증된 결과물임을 증명합니다. CI 파이프라인이 있다는 것만으로도 운영 감각을 보여 줄 수 있습니다.
- **API 문서나 변경 기록은 어떤 신뢰를 더해 줄까요?**
  - API 문서는 프론트엔드와 협업한 흔적을, 변경 기록은 유지보수 능력을 보여 줍니다. 작은 프로젝트라도 CHANGELOG가 있으면 "활발히 개선하는 프로젝트"라는 인상을 줍니다.
