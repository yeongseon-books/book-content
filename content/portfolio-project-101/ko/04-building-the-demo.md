# Portfolio Project 101 (4/10): 데모 만들기

포트폴리오 프로젝트는 README만 좋아도 부족합니다. 결국 다른 사람이 눌러 보고 직접 체감할 수 있어야 프로젝트가 살아 있는 것처럼 보입니다. 데모가 없거나, 있어도 첫 화면이 비어 있거나, 로그인 단계부터 막히면 프로젝트의 가치가 전달되기 전에 방문자가 이탈합니다.

이 글은 Portfolio Project 101 시리즈의 4번째 글입니다. 여기서는 포트폴리오 데모를 기능 전시장처럼 만드는 대신, 처음 들어온 사람이 30초 안에 핵심 가치를 느낄 수 있게 구성하는 방법을 살펴보겠습니다.

## 먼저 던지는 질문

- 데모 첫 화면은 왜 가장 중요한 평가 지점일까요?
- 시드 데이터와 공유 계정은 무엇을 해결해 줄까요?
- 라이브 데모와 백업 영상은 어떤 역할 차이가 있을까요?

## 큰 그림

![Portfolio Project 101 4장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/portfolio-project-101/04/04-01-diagram.ko.png)

*Portfolio Project 101 4장 흐름 개요*

이 그림에서는 데모가 코드와 사용자 사이에서 어떤 역할을 하는지 봅니다. 데모는 설명이 아니라 즜거이며, 방문자가 나라 예기 때 비로소 신뢰가 생겹니다.

> 코드 없이 말로만 설명하면 신뢰하지 않습니다. 실제로 동작하는 데모를 과 사용자는 '낙링도 끼나 바모 사람'이라는 신호를 받습니다.

## 왜 중요한가

데모가 살아 있으면 프로젝트도 살아 있습니다. 반대로 프로젝트 설명이 아무리 좋아도 실제로 열리지 않으면 신뢰가 크게 떨어집니다. 포트폴리오에서 데모는 구현을 과시하는 장치라기보다, 검토자가 스스로 확인할 수 있게 만드는 증거에 가깝습니다.

특히 개인 프로젝트는 제품의 모든 면을 보여 줄 필요가 없습니다. 오히려 핵심 흐름 하나를 짧고 선명하게 보여 주는 편이 더 강합니다. 방문자는 긴 탐색을 원하지 않습니다. 첫 화면, 첫 행동, 첫 결과만으로도 이 프로젝트가 왜 의미 있는지 느낄 수 있어야 합니다.

## 머릿속에 먼저 그릴 그림

데모는 랜딩 화면에서 시작해 샘플 데이터, 핵심 행동, 결과, 다음 행동으로 이어지는 짧은 흐름으로 보는 편이 좋습니다.

이 그림은 데모를 줄이는 기준을 줍니다. 처음 보는 사람은 모든 화면을 보고 싶어 하지 않습니다. 대신 첫 화면에서 무엇을 하는 서비스인지 파악하고, 샘플 데이터로 맥락을 이해하고, 한 번 행동해 보고, 결과를 확인하고, 다음으로 무엇을 눌러야 할지 알면 충분합니다.

## 핵심 용어

- **첫 화면**: 방문자가 가장 먼저 만나는 화면입니다.
- **시드 데이터**: 비어 있지 않은 상태를 보여 주는 예제 데이터입니다.
- **공유 계정**: 누구나 로그인해 볼 수 있도록 준비한 데모용 계정입니다.
- **백업 영상**: 라이브 데모가 막힐 때 대신 보여 줄 짧은 영상입니다.
- **실패 대비 경로**: 데모가 깨졌을 때 상태를 확인하거나 대체 증거를 제시하는 장치입니다.

## 바꾸기 전과 후

**Before**: 로그인 후 빈 화면만 보이거나, 어디를 눌러야 할지 몰라서 가치가 드러나지 않습니다.

**After**: 첫 화면에서 샘플 데이터와 핵심 흐름이 바로 보여서, 처음 보는 사람도 짧은 시간 안에 문제와 해결을 이해할 수 있습니다.

데모의 목적은 전체 제품을 완전히 재현하는 데 있지 않습니다. 프로젝트의 핵심 가치를 빠르게 전달하는 데 있습니다. 이 기준이 분명하면 무엇을 빼야 할지도 자연스럽게 보입니다.

## 단계별로 살펴보기

### 1단계 — 시나리오

먼저 데모의 핵심 흐름을 네 단계 정도로 압축합니다.

```python
flow = ["land", "sample", "action", "result"]
```

시나리오를 짧게 정리해 두면 데모가 기능 투어로 무너지는 일을 막을 수 있습니다. 방문자가 반드시 봐야 할 흐름이 무엇인지 먼저 정하는 편이 좋습니다.

### 2단계 — 시드 데이터

비어 있는 화면은 가치를 설명하기 어렵게 만듭니다.

```python
seed = {"users": 5, "tasks": 12}
```

시드 데이터가 있으면 첫 화면부터 서비스 맥락이 생깁니다. 일정 관리 프로젝트라면 일정이 보여야 하고, 분석 프로젝트라면 예제 결과가 보여야 합니다.

### 3단계 — 데모 계정

직접 써 볼 수 있어야 데모의 설득력이 올라갑니다.

```python
demo = {"id": "guest@demo", "pw": "demo1234"}
```

공유 계정은 포트폴리오 접근성을 높이는 장치입니다. 방문자가 계정 생성 과정에서 시간을 쓰지 않게 해 주기 때문입니다. 다만 민감한 데이터와 운영 계정은 철저히 분리해야 합니다.

### 4단계 — 백업 영상

라이브 데모는 강하지만, 항상 완벽하게 열리리라고 기대할 수는 없습니다.

```python
video_url = "https://youtu.be/example"
```

짧은 백업 영상이 있으면 데모 장애가 곧바로 설명 실패로 이어지지 않습니다. 특히 무료 호스팅이나 슬립 상태가 있는 플랫폼을 쓴다면 이 장치가 매우 유용합니다.

### 5단계 — 헬스체크

데모 상태를 빠르게 확인할 수 있는 경로가 필요합니다.

```python
health = "/healthz"
```

헬스체크는 방문자보다 작성자에게 더 유용합니다. 데모가 안 열릴 때 링크 문제인지, 앱이 죽은 것인지, 초기화가 필요한 것인지 빠르게 판단할 수 있기 때문입니다.

## 이 코드에서 먼저 볼 점

- 첫 화면은 프로젝트 가치를 가장 짧게 증명하는 자리입니다.
- 시드 데이터와 공유 계정은 데모 마찰을 크게 줄여 줍니다.
- 백업 영상과 헬스체크는 데모 품질을 운영 관점으로 끌어올립니다.

## 자주 하는 실수

1. 로그인 단계부터 막혀 핵심 가치를 보기도 전에 방문자가 떠나는 경우
2. 시드 데이터가 없어 빈 상태만 보여 주는 경우
3. 데모 계정을 비공개로 두어 직접 체험이 불가능한 경우
4. 라이브 데모가 깨졌을 때 대체할 영상이나 설명이 없는 경우
5. 상태 확인 경로가 없어 문제 발생 시 원인을 빨리 찾지 못하는 경우

포트폴리오 데모는 기능 수보다 마찰 수가 더 중요합니다. 어디에서 막히는지, 어디에서 가치가 늦게 드러나는지를 줄이는 편이 훨씬 효과적입니다.

## 실무에서는 이렇게 본다

실제 SaaS 제품도 첫 체험 흐름에 많은 공을 들입니다. 게스트 모드, 샘플 데이터, 짧은 온보딩, 즉시 보이는 가치가 중요한 이유는 모두 같습니다. 사용자는 길게 설명을 읽기보다 먼저 체감하기를 원하기 때문입니다.

개인 포트폴리오도 이 원리를 그대로 가져오면 됩니다. 모든 기능을 보여 주려 하기보다, 가장 설득력 있는 흐름 하나를 선명하게 만들어 두는 편이 좋습니다.

## 체크리스트

- [ ] 네 단계 안에 설명할 수 있는 데모 시나리오가 있다.
- [ ] 첫 화면에서 시드 데이터나 예시 결과가 보인다.
- [ ] 공유 계정 또는 우회 체험 경로가 있다.
- [ ] 라이브 데모가 막힐 때를 대비한 백업 영상이 있다.

## 연습 문제

1. 여러분 프로젝트의 데모 핵심 흐름을 네 단계로 줄여 보세요.
2. 첫 화면에서 바로 보여 줄 샘플 데이터 하나를 정해 보세요.
3. 라이브 데모가 막혔을 때 대신 보여 줄 증거가 무엇인지 적어 보세요.

## 정리와 다음 글

좋은 데모는 모든 기능을 설명하지 않아도 됩니다. 첫 화면에서 맥락을 보여 주고, 샘플 데이터로 이해를 돕고, 한 번의 행동으로 결과를 체감하게 만들면 됩니다. 여기에 공유 계정, 백업 영상, 헬스체크까지 더해지면 작은 프로젝트도 훨씬 믿을 만한 결과물로 보입니다.

다음 글에서는 이렇게 만든 프로젝트를 실제로 외부에 공개하고 유지할 수 있게 만드는 배포 원칙을 살펴보겠습니다.

### MVP 정의 체크리스트

데모를 강하게 만들려면 먼저 MVP를 명확히 정의해야 합니다. 많은 프로젝트가 데모 단계에서 흔들리는 이유는 "무엇을 보여 줄지"가 아니라 "무엇을 제외할지"를 결정하지 않았기 때문입니다. 아래 체크리스트는 데모용 MVP 경계를 정하는 기준입니다.

| 항목 | 질문 | 통과 기준 |
| --- | --- | --- |
| 핵심 사용자 | 누구를 위한 데모인가 | 단일 페르소나가 정의됨 |
| 핵심 문제 | 어떤 불편을 줄이는가 | 1문장으로 설명 가능 |
| 핵심 행동 | 사용자가 반드시 해볼 행동은 무엇인가 | 1~2개로 압축됨 |
| 성공 결과 | 행동 후 어떤 변화가 보여야 하는가 | 화면/지표로 즉시 확인 가능 |
| 제외 범위 | 이번 데모에서 빼는 기능은 무엇인가 | TODO로 명시됨 |

체크리스트를 통과한 MVP는 짧은 시간 안에 이해됩니다. 반대로 제외 범위가 없는 MVP는 금방 기능 소개 세션이 되고, 방문자는 핵심 가치를 기억하지 못합니다.

### 사용자 스토리 예시로 데모 흐름 설계

데모 시나리오는 기능 목록보다 사용자 스토리로 작성할 때 훨씬 자연스럽습니다. 아래처럼 "누가, 무엇을, 왜"를 명시하면 첫 화면부터 결과 화면까지 흐름이 안정됩니다.

```markdown
- As a team lead,
  I want to see all team schedules in one calendar,
  so that I can finalize weekly planning in 10 minutes.

Acceptance Criteria:
1) 로그인 후 기본 주간 뷰가 즉시 보인다.
2) 팀원 필터를 선택하면 1초 내 일정이 갱신된다.
3) 공유 링크 생성 후 다른 브라우저에서 동일 화면이 열린다.
```

이 스토리 기반 설계는 데모 준비뿐 아니라 테스트 케이스 작성에도 바로 연결됩니다. 또한 면접에서 "왜 이 흐름을 보여 줬는가"를 설명하기 쉬워집니다.

### 데모 스크립트 템플릿(30초/90초)

실전에서는 데모 시간이 늘 짧습니다. 그래서 시간별 스크립트를 따로 준비해 두는 것이 좋습니다.

| 구간 | 30초 버전 | 90초 버전 |
| --- | --- | --- |
| 도입 | 문제 1문장 | 문제 + 기존 방식 한계 |
| 시연 | 핵심 행동 1개 | 핵심 행동 2개 + 예외 1개 |
| 결과 | 결과 화면 1개 | 결과 + 수치 + 다음 개선 |

```markdown
[30초 스크립트]
1) "팀 일정이 여러 도구에 흩어져 계획 시간이 길어지는 문제를 해결했습니다."
2) "여기서 팀원을 선택하면 주간 일정이 즉시 필터링됩니다."
3) "현재 평균 계획 시간이 40분에서 18분으로 줄었습니다."
```

이렇게 스크립트를 미리 준비하면 라이브 데모 중 긴장으로 흐름이 무너지는 일을 줄일 수 있습니다. 데모는 기능 소개가 아니라 가치 전달이라는 점을 계속 유지할 수 있습니다.

### 데모 실패 포인트 사전 차단표

데모가 약해지는 지점은 대부분 반복됩니다. 아래 표를 기준으로 사전 점검하면 실제 발표나 공유 상황에서 안정성이 크게 올라갑니다.

| 실패 포인트 | 원인 | 사전 조치 |
| --- | --- | --- |
| 첫 화면 공백 | 시드 데이터 없음 | 기본 샘플 자동 로딩 |
| 로그인 막힘 | 계정 준비 누락 | 게스트 계정 문서화 |
| 반응 지연 | 초기 로딩 과다 | 캐시/로딩 메시지 추가 |
| 데모 중 장애 | 배포 상태 불안정 | 백업 영상 + 헬스체크 |

데모는 기술 과시보다 사용자 경험 설계에 가깝습니다. 방문자가 30초 안에 "무엇이 달라졌는가"를 느끼게 하는 데 집중하면 전달력이 훨씬 높아집니다.

### 품질 점검 기록 예시(반복 가능한 운영 습관)

포트폴리오 품질은 한 번의 대규모 수정으로 완성되지 않습니다. 짧은 주기로 같은 항목을 반복 점검할 때 품질이 안정됩니다. 아래 기록 방식은 프로젝트 크기와 무관하게 적용할 수 있습니다.

| 점검 항목 | 확인 질문 | 증거 형태 |
| --- | --- | --- |
| 문제 정의 | 문제 문장이 여전히 현재 구현과 일치하는가 | README/블로그 도입 문장 |
| 데모 신뢰성 | 링크가 열리고 핵심 흐름이 실제로 작동하는가 | 라이브 URL, 백업 영상 |
| 검증 체계 | 테스트와 CI 상태가 녹색인가 | CI 배지, 테스트 로그 |
| 문서 정합성 | 실행 방법과 환경 변수 설명이 최신인가 | README, `.env.example` |
| 설명 일관성 | 저장소/블로그/면접 답변이 같은 메시지를 말하는가 | 요약 문장 비교 |

이 표를 기준으로 주기적 점검을 하면 개선 우선순위를 빠르게 정할 수 있습니다. 예를 들어 데모는 정상이지만 문서 정합성이 낮다면 기능 추가보다 문서 보완이 먼저입니다. 반대로 문서는 완벽하지만 CI가 불안정하면 공개 신뢰를 잃기 쉽습니다.

다음 템플릿은 실제 기록에 바로 사용할 수 있습니다.

```markdown
## Weekly Portfolio Health Check
- 날짜: 2026-05-21
- 현재 상태: 데모 정상 / 문서 일부 불일치 / CI 정상
- 발견 이슈:
  1) README 실행 명령이 최신 compose 파일과 다름
  2) 블로그 성과 수치가 최근 측정값과 불일치
- 조치 계획:
  1) README 실행 섹션 업데이트
  2) 성과 표 재계산 및 캡처 갱신
- 완료 기준:
  - 신규 방문자가 5분 안에 문제/데모/실행/결과를 이해할 수 있음
```

핵심은 완벽함이 아니라 반복성입니다. 같은 기준으로 꾸준히 점검하면 프로젝트는 시간이 지나도 낡지 않습니다. 면접에서도 "어떻게 유지했는가"를 설명할 수 있어 결과물의 신뢰도가 더 높아집니다.

### 실행 가능한 개선 TODO 예시

아래 TODO는 "읽고 끝나는 조언"이 아니라 실제로 바로 적용할 수 있는 작업 단위입니다. 각 항목은 30~90분 안에 끝낼 수 있는 크기로 쪼개는 것을 권장합니다.

| 작업 | 예상 시간 | 완료 기준 |
| --- | ---: | --- |
| 핵심 메시지 1문장 재작성 | 30분 | README/발표/이력서 문장 일치 |
| 데모 검증 리허설 1회 | 45분 | 로그인-핵심행동-결과 흐름 무중단 |
| 테스트 명령 정리 | 30분 | 로컬/CI 명령이 문서와 일치 |
| 결과 수치 업데이트 | 60분 | 최신 측정값과 표/그래프 동기화 |

작업을 작은 단위로 나누면 진행률이 눈에 보이고, 품질 개선이 미뤄지지 않습니다. 포트폴리오를 오래 유지하려면 큰 개편보다 이런 짧은 루틴이 더 효과적입니다.

### 마무리 검증 메모

최종 공개 전에는 "내가 이해하는 프로젝트"가 아니라 "처음 보는 사람이 이해하는 프로젝트"인지 확인해야 합니다. 이를 위해 1) 문제를 한 문장으로 말해 보기, 2) 데모를 처음부터 끝까지 클릭해 보기, 3) 실행 명령을 새 환경에서 복사 실행해 보기, 4) 결과 수치를 다시 계산해 보기 네 단계를 반복하는 것이 좋습니다. 이 검증을 통과하면 문서와 데모와 코드의 정합성이 높아지고, 면접 답변도 훨씬 안정됩니다.

### 한 줄 점검

이 글의 실천 포인트를 실제 프로젝트에 바로 반영해, 다음 주기 점검에서 변경 전후 차이를 기록해 두면 품질 개선 효과를 더 명확하게 확인할 수 있습니다.

## 처음 질문으로 돌아가기

- **데모 첫 화면은 왜 가장 중요한 평가 지점일까요?**
  - 데모는 선택아이니라 즘니니다. 최소 스크린샷나 GIF로 모든 설명 흐름을 보여 주면, 검토자는 나라 실행 명령어를 직접 실행할 수 있습니다.
- **시드 데이터와 공유 계정은 무엇을 해결해 줄까요?**
  - 최소 낙리동은 완전확덜 벽좌 에도 비로는 동제니다. 가장 효과적인 신뢰 신호는 '동입동 실행 가능함'입니다.
- **라이브 데모와 백업 영상은 어떤 역할 차이가 있을까요?**
  - 데모는 성공 쓰낤만 보여 진단기 쇼습니다. 오류 처리, 비는 상태, 대기 중 같은 앵지 케이스도 자연스맙게 보여주면, 생각이 깊은 개발자로 보일니다.
