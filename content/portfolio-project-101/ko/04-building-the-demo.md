# Portfolio Project 101 (4/10): 데모 만들기

포트폴리오 프로젝트는 README만 좋아도 부족합니다. 결국 다른 사람이 눌러 보고 직접 체감할 수 있어야 프로젝트가 살아 있는 것처럼 보입니다. 데모가 없거나, 있어도 첫 화면이 비어 있거나, 로그인 단계부터 막히면 프로젝트의 가치가 전달되기 전에 방문자가 이탈합니다.

이 글은 Portfolio Project 101 시리즈의 4번째 글입니다. 여기서는 포트폴리오 데모를 기능 전시장처럼 만드는 대신, 처음 들어온 사람이 30초 안에 핵심 가치를 느낄 수 있게 구성하는 방법을 살펴보겠습니다.

## 먼저 던지는 질문

- 데모 첫 화면은 왜 가장 중요한 평가 지점일까요?
- 시드 데이터와 공유 계정은 무엇을 해결해 줄까요?
- 라이브 데모와 백업 영상은 어떤 역할 차이가 있을까요?

## 큰 그림

![Portfolio Project 101 4장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/portfolio-project-101/04/04-01-diagram.ko.png)

*Portfolio Project 101 4장 흐름 개요*

이 그림에서는 데모 만들기를 운영 흐름 안에서 어디에 배치해야 하는지 봅니다. 핵심은 개념을 따로 외우는 것이 아니라 입력, 처리, 검증, 운영 신호가 어떤 경계로 이어지는지 확인하는 데 있습니다.

> 데모 만들기의 핵심은 기능 이름이 아니라, 어떤 경계에서 무엇을 검증하고 어떤 신호를 남길지 정하는 데 있습니다.

## 왜 중요한가

데모가 살아 있으면 프로젝트도 살아 있습니다. 반대로 프로젝트 설명이 아무리 좋아도 실제로 열리지 않으면 신뢰가 크게 떨어집니다. 포트폴리오에서 데모는 구현을 과시하는 장치라기보다, 검토자가 스스로 확인할 수 있게 만드는 증거에 가깝습니다.

특히 개인 프로젝트는 제품의 모든 면을 보여 줄 필요가 없습니다. 오히려 핵심 흐름 하나를 짧고 선명하게 보여 주는 편이 더 강합니다. 방문자는 긴 탐색을 원하지 않습니다. 첫 화면, 첫 행동, 첫 결과만으로도 이 프로젝트가 왜 의미 있는지 느낄 수 있어야 합니다.

## 머릿속에 먼저 그릴 그림

데모는 랜딩 화면에서 시작해 샘플 데이터, 핵심 행동, 결과, 다음 행동으로 이어지는 짧은 흐름으로 보는 편이 좋습니다.

이 그림은 데모를 줄이는 기준을 줍니다. 처음 보는 사람은 모든 화면을 보고 싶어 하지 않습니다. 대신 첫 화면에서 무엇을 하는 서비스인지 파악하고, 샘플 데이터로 맥락을 이해하고, 한 번 행동해 보고, 결과를 확인하고, 다음으로 무엇을 눌러야 할지 알면 충분합니다.

## 핵심 용어

- **첫 화면**: 방문자가 가장 먼저 만나는 화면입니다.
- **시드 데이터**: 비어 있지 않은 상태를 보여 주는 예제 데이터입니다.
- **공유 계정**: 누구나 로그인해 볼 수 있도록 준비한 데모용 계정입니다.
- **백업 영상**: 라이브 데모가 막힐 때 대신 보여 줄 짧은 영상입니다.
- **실패 대비 경로**: 데모가 깨졌을 때 상태를 확인하거나 대체 증거를 제시하는 장치입니다.

## 바꾸기 전과 후

**Before**: 로그인 후 빈 화면만 보이거나, 어디를 눌러야 할지 몰라서 가치가 드러나지 않습니다.

**After**: 첫 화면에서 샘플 데이터와 핵심 흐름이 바로 보여서, 처음 보는 사람도 짧은 시간 안에 문제와 해결을 이해할 수 있습니다.

데모의 목적은 전체 제품을 완전히 재현하는 데 있지 않습니다. 프로젝트의 핵심 가치를 빠르게 전달하는 데 있습니다. 이 기준이 분명하면 무엇을 빼야 할지도 자연스럽게 보입니다.

## 단계별로 살펴보기

### 1단계 — 시나리오

먼저 데모의 핵심 흐름을 네 단계 정도로 압축합니다.

```python
flow = ["land", "sample", "action", "result"]
```

시나리오를 짧게 정리해 두면 데모가 기능 투어로 무너지는 일을 막을 수 있습니다. 방문자가 반드시 봐야 할 흐름이 무엇인지 먼저 정하는 편이 좋습니다.

### 2단계 — 시드 데이터

비어 있는 화면은 가치를 설명하기 어렵게 만듭니다.

```python
seed = {"users": 5, "tasks": 12}
```

시드 데이터가 있으면 첫 화면부터 서비스 맥락이 생깁니다. 일정 관리 프로젝트라면 일정이 보여야 하고, 분석 프로젝트라면 예제 결과가 보여야 합니다.

### 3단계 — 데모 계정

직접 써 볼 수 있어야 데모의 설득력이 올라갑니다.

```python
demo = {"id": "guest@demo", "pw": "demo1234"}
```

공유 계정은 포트폴리오 접근성을 높이는 장치입니다. 방문자가 계정 생성 과정에서 시간을 쓰지 않게 해 주기 때문입니다. 다만 민감한 데이터와 운영 계정은 철저히 분리해야 합니다.

### 4단계 — 백업 영상

라이브 데모는 강하지만, 항상 완벽하게 열리리라고 기대할 수는 없습니다.

```python
video_url = "https://youtu.be/example"
```

짧은 백업 영상이 있으면 데모 장애가 곧바로 설명 실패로 이어지지 않습니다. 특히 무료 호스팅이나 슬립 상태가 있는 플랫폼을 쓴다면 이 장치가 매우 유용합니다.

### 5단계 — 헬스체크

데모 상태를 빠르게 확인할 수 있는 경로가 필요합니다.

```python
health = "/healthz"
```

헬스체크는 방문자보다 작성자에게 더 유용합니다. 데모가 안 열릴 때 링크 문제인지, 앱이 죽은 것인지, 초기화가 필요한 것인지 빠르게 판단할 수 있기 때문입니다.

## 이 코드에서 먼저 볼 점

- 첫 화면은 프로젝트 가치를 가장 짧게 증명하는 자리입니다.
- 시드 데이터와 공유 계정은 데모 마찰을 크게 줄여 줍니다.
- 백업 영상과 헬스체크는 데모 품질을 운영 관점으로 끌어올립니다.

## 자주 하는 실수

1. 로그인 단계부터 막혀 핵심 가치를 보기도 전에 방문자가 떠나는 경우
2. 시드 데이터가 없어 빈 상태만 보여 주는 경우
3. 데모 계정을 비공개로 두어 직접 체험이 불가능한 경우
4. 라이브 데모가 깨졌을 때 대체할 영상이나 설명이 없는 경우
5. 상태 확인 경로가 없어 문제 발생 시 원인을 빨리 찾지 못하는 경우

포트폴리오 데모는 기능 수보다 마찰 수가 더 중요합니다. 어디에서 막히는지, 어디에서 가치가 늦게 드러나는지를 줄이는 편이 훨씬 효과적입니다.

## 실무에서는 이렇게 본다

실제 SaaS 제품도 첫 체험 흐름에 많은 공을 들입니다. 게스트 모드, 샘플 데이터, 짧은 온보딩, 즉시 보이는 가치가 중요한 이유는 모두 같습니다. 사용자는 길게 설명을 읽기보다 먼저 체감하기를 원하기 때문입니다.

개인 포트폴리오도 이 원리를 그대로 가져오면 됩니다. 모든 기능을 보여 주려 하기보다, 가장 설득력 있는 흐름 하나를 선명하게 만들어 두는 편이 좋습니다.

## 체크리스트

- [ ] 네 단계 안에 설명할 수 있는 데모 시나리오가 있다.
- [ ] 첫 화면에서 시드 데이터나 예시 결과가 보인다.
- [ ] 공유 계정 또는 우회 체험 경로가 있다.
- [ ] 라이브 데모가 막힐 때를 대비한 백업 영상이 있다.

## 연습 문제

1. 여러분 프로젝트의 데모 핵심 흐름을 네 단계로 줄여 보세요.
2. 첫 화면에서 바로 보여 줄 샘플 데이터 하나를 정해 보세요.
3. 라이브 데모가 막혔을 때 대신 보여 줄 증거가 무엇인지 적어 보세요.

## 정리와 다음 글

좋은 데모는 모든 기능을 설명하지 않아도 됩니다. 첫 화면에서 맥락을 보여 주고, 샘플 데이터로 이해를 돕고, 한 번의 행동으로 결과를 체감하게 만들면 됩니다. 여기에 공유 계정, 백업 영상, 헬스체크까지 더해지면 작은 프로젝트도 훨씬 믿을 만한 결과물로 보입니다.

다음 글에서는 이렇게 만든 프로젝트를 실제로 외부에 공개하고 유지할 수 있게 만드는 배포 원칙을 살펴보겠습니다.

## 처음 질문으로 돌아가기

- **데모 첫 화면은 왜 가장 중요한 평가 지점일까요?**
  - 본문의 기준은 데모 만들기를 한 덩어리 개념으로 보지 않고 입력, 처리, 검증, 운영 신호가 만나는 경계로 나누어 확인하는 것입니다.
- **시드 데이터와 공유 계정은 무엇을 해결해 줄까요?**
  - 예제와 그림에서는 어떤 값이 들어오고, 어느 단계에서 바뀌며, 어떤 기준으로 통과 또는 실패하는지를 먼저 확인해야 합니다.
- **라이브 데모와 백업 영상은 어떤 역할 차이가 있을까요?**
  - 운영에서는 이 판단을 체크리스트, 로그, 테스트로 남겨 다음 변경에서도 같은 실패가 반복되지 않게 막아야 합니다.
