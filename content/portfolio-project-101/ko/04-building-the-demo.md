# Portfolio Project 101 (4/10): 데모 만들기

포트폴리오 프로젝트는 README만 좋아도 부족합니다. 결국 다른 사람이 눌러 보고 직접 체감할 수 있어야 프로젝트가 살아 있는 것처럼 보입니다. 데모가 없거나, 있어도 첫 화면이 비어 있거나, 로그인 단계부터 막히면 프로젝트의 가치가 전달되기 전에 방문자가 이탈합니다.

이 글은 Portfolio Project 101 시리즈의 4번째 글입니다. 여기서는 포트폴리오 데모를 기능 전시장처럼 만드는 대신, 처음 들어온 사람이 30초 안에 핵심 가치를 느낄 수 있게 구성하는 방법을 봅니다.

> 코드 없이 말로만 설명하면 신뢰하지 않습니다. 실제로 동작하는 데모를 본 사용자는 '끝까지 마무리하는 사람'이라는 신호를 받습니다.

## 먼저 던지는 질문

- 데모 첫 화면은 왜 가장 중요한 평가 지점일까요?
- 시드 데이터와 공유 계정은 무엇을 해결해 줄까요?
- 라이브 데모와 백업 영상은 어떤 역할 차이가 있을까요?

## 왜 중요한가

데모가 살아 있으면 프로젝트도 살아 있습니다. 반대로 프로젝트 설명이 아무리 좋아도 실제로 열리지 않으면 신뢰가 크게 떨어집니다. 포트폴리오에서 데모는 구현을 과시하는 장치라기보다, 검토자가 스스로 확인할 수 있게 만드는 증거에 가깝습니다.

특히 개인 프로젝트는 제품의 모든 면을 보여 줄 필요가 없습니다. 오히려 핵심 흐름 하나를 짧고 선명하게 보여 주는 편이 더 강합니다. 방문자는 긴 탐색을 원하지 않습니다. 첫 화면, 첫 행동, 첫 결과만으로도 이 프로젝트가 왜 의미 있는지 느낄 수 있어야 합니다.

## 머릿속에 먼저 그릴 그림

데모는 랜딩 화면에서 시작해 샘플 데이터, 핵심 행동, 결과, 다음 행동으로 이어지는 짧은 흐름으로 보는 편이 좋습니다.

이 그림은 데모를 줄이는 기준을 줍니다. 처음 보는 사람은 모든 화면을 보고 싶어 하지 않습니다. 대신 첫 화면에서 무엇을 하는 서비스인지 파악하고, 샘플 데이터로 맥락을 이해하고, 한 번 행동해 보고, 결과를 확인하고, 다음으로 무엇을 눌러야 할지 알면 충분합니다.

## 핵심 용어

- **첫 화면**: 방문자가 가장 먼저 만나는 화면입니다.
- **시드 데이터**: 비어 있지 않은 상태를 보여 주는 예제 데이터입니다.
- **공유 계정**: 누구나 로그인해 볼 수 있도록 준비한 데모용 계정입니다.
- **백업 영상**: 라이브 데모가 막힐 때 대신 보여 줄 짧은 영상입니다.
- **실패 대비 경로**: 데모가 깨졌을 때 상태를 확인하거나 대체 증거를 제시하는 장치입니다.

## 바꾸기 전과 후

**Before**: 로그인 후 빈 화면만 보이거나, 어디를 눌러야 할지 몰라서 가치가 드러나지 않습니다.

**After**: 첫 화면에서 샘플 데이터와 핵심 흐름이 바로 보여서, 처음 보는 사람도 짧은 시간 안에 문제와 해결을 이해할 수 있습니다.

데모의 목적은 전체 제품을 완전히 재현하는 데 있지 않습니다. 프로젝트의 핵심 가치를 빠르게 전달하는 데 있습니다. 이 기준이 분명하면 무엇을 빼야 할지도 자연스럽게 보입니다.

## 단계별로 살펴보기

### 1단계 — 시나리오

먼저 데모의 핵심 흐름을 네 단계 정도로 압축합니다.

데모 흐름은 보통 아래 네 단계로 압축하면 충분합니다.

- 첫 화면
- 샘플 데이터 확인
- 핵심 행동 수행
- 결과 확인

시나리오를 짧게 정리해 두면 데모가 기능 투어로 무너지는 일을 막을 수 있습니다. 방문자가 반드시 봐야 할 흐름이 무엇인지 먼저 정하는 편이 좋습니다.

### 2단계 — 시드 데이터

비어 있는 화면은 가치를 설명하기 어렵게 만듭니다.

예를 들어 `사용자 5명`, `작업 12개` 정도의 시드 데이터만 있어도 첫 화면이 훨씬 덜 비어 보입니다.

시드 데이터가 있으면 첫 화면부터 서비스 맥락이 생깁니다. 일정 관리 프로젝트라면 일정이 보여야 하고, 분석 프로젝트라면 예제 결과가 보여야 합니다.

### 3단계 — 데모 계정

직접 써 볼 수 있어야 데모의 설득력이 올라갑니다.

예를 들어 `게스트 계정: guest@demo / demo1234`처럼 바로 써 볼 수 있는 계정을 함께 제공하는 방식입니다.

공유 계정은 포트폴리오 접근성을 높이는 장치입니다. 방문자가 계정 생성 과정에서 시간을 쓰지 않게 해 주기 때문입니다. 다만 민감한 데이터와 운영 계정은 철저히 분리해야 합니다.

### 4단계 — 백업 영상

라이브 데모는 강하지만, 항상 완벽하게 열리리라고 기대할 수는 없습니다.

예를 들어 라이브 데모와 별도로 `30초 백업 영상 링크`를 함께 두면 장애 상황에서도 설명이 끊기지 않습니다.

짧은 백업 영상이 있으면 데모 장애가 곧바로 설명 실패로 이어지지 않습니다. 특히 무료 호스팅이나 슬립 상태가 있는 플랫폼을 쓴다면 이 장치가 매우 유용합니다.

### 5단계 — 헬스체크

데모 상태를 빠르게 확인할 수 있는 경로가 필요합니다.

최소한 `/healthz` 같은 상태 확인 경로 하나는 미리 정해 두는 편이 좋습니다.

헬스체크는 방문자보다 작성자에게 더 유용합니다. 데모가 안 열릴 때 링크 문제인지, 앱이 죽은 것인지, 초기화가 필요한 것인지 빠르게 판단할 수 있기 때문입니다.

## 이 코드에서 먼저 볼 점

- 첫 화면은 프로젝트 가치를 가장 짧게 증명하는 자리입니다.
- 시드 데이터와 공유 계정은 데모 마찰을 크게 줄여 줍니다.
- 백업 영상과 헬스체크는 데모 품질을 운영 관점으로 끌어올립니다.

## 자주 하는 실수

1. 로그인 단계부터 막혀 핵심 가치를 보기도 전에 방문자가 떠나는 경우
2. 시드 데이터가 없어 빈 상태만 보여 주는 경우
3. 데모 계정을 비공개로 두어 직접 체험이 불가능한 경우
4. 라이브 데모가 깨졌을 때 대체할 영상이나 설명이 없는 경우
5. 상태 확인 경로가 없어 문제 발생 시 원인을 빨리 찾지 못하는 경우

포트폴리오 데모는 기능 수보다 마찰 수가 더 중요합니다. 어디에서 막히는지, 어디에서 가치가 늦게 드러나는지를 줄이는 편이 훨씬 효과적입니다.

## 실무에서는 이렇게 본다

실제 SaaS 제품도 첫 체험 흐름에 많은 공을 들입니다. 게스트 모드, 샘플 데이터, 짧은 온보딩, 즉시 보이는 가치가 중요한 이유는 모두 같습니다. 사용자는 길게 설명을 읽기보다 먼저 체감하기를 원하기 때문입니다.

개인 포트폴리오도 이 원리를 그대로 가져오면 됩니다. 모든 기능을 보여 주려 하기보다, 가장 설득력 있는 흐름 하나를 선명하게 만들어 두는 편이 좋습니다.

## 체크리스트

- [ ] 네 단계 안에 설명할 수 있는 데모 시나리오가 있다.
- [ ] 첫 화면에서 시드 데이터나 예시 결과가 보인다.
- [ ] 공유 계정 또는 우회 체험 경로가 있다.
- [ ] 라이브 데모가 막힐 때를 대비한 백업 영상이 있다.

## 연습 문제

1. 여러분 프로젝트의 데모 핵심 흐름을 네 단계로 줄여 보세요.
2. 첫 화면에서 바로 보여 줄 샘플 데이터 하나를 정해 보세요.
3. 라이브 데모가 막혔을 때 대신 보여 줄 증거가 무엇인지 적어 보세요.

## 정리와 다음 글

좋은 데모는 모든 기능을 설명하지 않아도 됩니다. 첫 화면에서 맥락을 보여 주고, 샘플 데이터로 이해를 돕고, 한 번의 행동으로 결과를 체감하게 만들면 됩니다. 여기에 공유 계정, 백업 영상, 헬스체크까지 더해지면 작은 프로젝트도 훨씬 믿을 만한 결과물로 보입니다.

다음 글에서는 이렇게 만든 프로젝트를 실제로 외부에 공개하고 유지할 수 있게 만드는 배포 원칙을 다룹니다.

### MVP 정의 체크리스트

데모를 강하게 만들려면 먼저 MVP를 명확히 정의해야 합니다. 많은 프로젝트가 데모 단계에서 흔들리는 이유는 "무엇을 보여 줄지"가 아니라 "무엇을 제외할지"를 결정하지 않았기 때문입니다. 아래 체크리스트는 데모용 MVP 경계를 정하는 기준입니다.

| 항목 | 질문 | 통과 기준 |
| --- | --- | --- |
| 핵심 사용자 | 누구를 위한 데모인가 | 단일 페르소나가 정의됨 |
| 핵심 문제 | 어떤 불편을 줄이는가 | 1문장으로 설명 가능 |
| 핵심 행동 | 사용자가 반드시 해볼 행동은 무엇인가 | 1~2개로 압축됨 |
| 성공 결과 | 행동 후 어떤 변화가 보여야 하는가 | 화면/지표로 즉시 확인 가능 |
| 제외 범위 | 이번 데모에서 빼는 기능은 무엇인가 | TODO로 명시됨 |

체크리스트를 통과한 MVP는 짧은 시간 안에 이해됩니다. 반대로 제외 범위가 없는 MVP는 금방 기능 소개 세션이 되고, 방문자는 핵심 가치를 기억하지 못합니다.

### 사용자 스토리 예시로 데모 흐름 설계

데모 시나리오는 기능 목록보다 사용자 스토리로 작성할 때 훨씬 자연스럽습니다. 아래처럼 "누가, 무엇을, 왜"를 명시하면 첫 화면부터 결과 화면까지 흐름이 안정됩니다.

```markdown
- As a team lead,
  I want to see all team schedules in one calendar,
  so that I can finalize weekly planning in 10 minutes.

Acceptance Criteria:
1) 로그인 후 기본 주간 뷰가 즉시 보인다.
2) 팀원 필터를 선택하면 1초 내 일정이 갱신된다.
3) 공유 링크 생성 후 다른 브라우저에서 동일 화면이 열린다.
```

이 스토리 기반 설계는 데모 준비뿐 아니라 테스트 케이스 작성에도 바로 연결됩니다. 또한 면접에서 "왜 이 흐름을 보여 줬는가"를 설명하기 쉬워집니다.

### 데모 스크립트 템플릿(30초/90초)

실전에서는 데모 시간이 늘 짧습니다. 그래서 시간별 스크립트를 따로 준비해 두는 것이 좋습니다.

| 구간 | 30초 버전 | 90초 버전 |
| --- | --- | --- |
| 도입 | 문제 1문장 | 문제 + 기존 방식 한계 |
| 시연 | 핵심 행동 1개 | 핵심 행동 2개 + 예외 1개 |
| 결과 | 결과 화면 1개 | 결과 + 수치 + 다음 개선 |

```markdown
[30초 스크립트]
1) "팀 일정이 여러 도구에 흩어져 계획 시간이 길어지는 문제를 해결했습니다."
2) "여기서 팀원을 선택하면 주간 일정이 즉시 필터링됩니다."
3) "현재 평균 계획 시간이 40분에서 18분으로 줄었습니다."
```

이렇게 스크립트를 미리 준비하면 라이브 데모 중 긴장으로 흐름이 무너지는 일을 줄일 수 있습니다. 데모는 기능 소개가 아니라 가치 전달이라는 점을 계속 유지할 수 있습니다.

### 데모 실패 포인트 사전 차단표

데모가 약해지는 지점은 대부분 반복됩니다. 아래 표를 기준으로 사전 점검하면 실제 발표나 공유 상황에서 안정성이 크게 올라갑니다.

| 실패 포인트 | 원인 | 사전 조치 |
| --- | --- | --- |
| 첫 화면 공백 | 시드 데이터 없음 | 기본 샘플 자동 로딩 |
| 로그인 막힘 | 계정 준비 누락 | 게스트 계정 문서화 |
| 반응 지연 | 초기 로딩 과다 | 캐시/로딩 메시지 추가 |
| 데모 중 장애 | 배포 상태 불안정 | 백업 영상 + 헬스체크 |

데모는 기술 과시보다 사용자 경험 설계에 가깝습니다. 방문자가 30초 안에 "무엇이 달라졌는가"를 느끼게 하는 데 집중하면 전달력이 훨씬 높아집니다.

### 데모 영상 녹화 스크립트 작성법

라이브 데모가 불안정하거나, 공유 대상이 비동기로 확인하는 경우 백업 영상이 필수입니다. 영상은 '기능을 모두 보여 주는 투어'가 아니라 '핵심 가치를 30초 안에 증명하는 증거'로 만들어야 합니다.

**30초 스크립트 템플릿**

```text
[0:00-0:05] 문제 제시 (자막 또는 나레이션)
  "팀 일정이 3개 도구에 흩어져, 주간 계획에 40분 이상 걸리는 문제를 해결했습니다."

[0:05-0:20] 핵심 행동 1개 시연
  - 브라우저 열기 → 팀원 선택 → 주간 일정 필터링 클릭
  - 화면 전환 시 마우스 커서를 느리게 이동해 시청자가 따라갈 수 있게

[0:20-0:30] 결과 + 마무리
  "계획 시간이 40분 → 18분으로 단축되었습니다. README에서 직접 실행해 보세요."
  - 화면에 GitHub 저장소 URL 자막 표시
```

**90초 스크립트 템플릿**

```text
[0:00-0:10] 문제 + 기존 방식 한계
  "팀 일정을 Notion, Google Calendar, Jira 3곳에서 수동 취합하고 있었습니다."

[0:10-0:40] 핵심 행동 2개 시연
  행동 1: 팀원 선택 → 자동 필터링 (주간 뷰)
  행동 2: 충돌 감지 → 자동 알림 생성

[0:40-0:55] 예외 상황 처리
  - 네트워크 끊김 시 로컬 캐시 동작
  - 데이터 없는 신규 팀원의 빈 상태 처리

[0:55-1:10] 결과 수치 + 아키텍처 한 줄
  "FastAPI + React + PostgreSQL. 계획 시간 55% 단축, 충돌 알림 도입 후 회의 취소 70% 감소."

[1:10-1:30] 다음 단계 + CTA
  "OAuth 연동과 Slack 알림을 추가할 예정입니다. README에서 로컬 실행 방법을 확인하세요."
```

두 스크립트의 차이를 보면, 30초 버전은 문제-행동-결과를 한 번씩만 보여 주고, 90초 버전은 예외 처리와 아키텍처까지 포함합니다. 면접 상황이라면 30초 버전으로 시작한 뒤 "더 보여 드릴까요?"라고 물어 90초로 확장하는 전략이 효과적입니다.

### 녹화 환경 설정 가이드

영상 품질은 내용만큼 중요합니다. 아래 설정을 따르면 무료 도구만으로도 깔끔한 데모 영상을 만들 수 있습니다.

```yaml
# demo-recording-config.yml
recording:
  tool: OBS Studio (무료) 또는 macOS QuickTime
  resolution: 1920x1080
  fps: 30
  format: MP4 (H.264)
  max_file_size: 50MB (GitHub README 임베딩 한계)

browser:
  zoom: 110%  # 작은 텍스트 가독성 향상
  bookmarks_bar: hidden
  extensions: disabled  # 불필요한 아이콘 제거
  tab_count: 1  # 데모 대상 탭만 열기

terminal:
  font_size: 16pt
  theme: dark (고대비)
  prompt: 짧게 설정 ($ 만 남기기)
  history: clear  # 이전 명령 노출 방지

post_processing:
  tool: FFmpeg (무료)
  trim_silence: true
  speed_up_typing: 1.5x  # 타이핑 대기 시간 단축
  add_subtitles: true  # 음소거 환경 대응
```

녹화 전 체크리스트:

```text
[ ] 알림 끄기 (Slack, 메일, OS 알림 모두)
[ ] 바탕화면 정리 (개인 파일 숨김)
[ ] 브라우저 시크릿 모드 또는 새 프로필
[ ] 해상도 1920x1080 고정
[ ] 마이크 테스트 (나레이션 있는 경우)
[ ] 시드 데이터 로드 확인
[ ] 녹화 영역 설정 (전체 화면 or 앱 창만)
```

녹화 후에는 반드시 음소거 상태에서 재생해 봅니다. 자막 없이도 흐름이 이해되는지 확인하면, 소리를 듣지 못하는 환경(사무실, 모바일)에서도 전달력을 유지할 수 있습니다.

### GIF vs 영상: 어떤 형식을 선택할까

README에 데모를 넣을 때 GIF와 영상 링크 중 어느 것이 적합한지는 상황에 따라 다릅니다.

| 기준 | GIF | 영상 (YouTube/Loom) |
| --- | --- | --- |
| 자동 재생 | GitHub README에서 자동 재생 | 클릭 필요 |
| 파일 크기 | 5-15MB (10초 기준) | 제한 없음 |
| 길이 제한 | 10초 이하 권장 | 제한 없음 |
| 음성 설명 | 불가 | 가능 |
| 편집 용이성 | 재촬영 필요 | 구간 편집 가능 |
| 적합한 경우 | 단일 기능 시연 | 전체 흐름 설명 |

실무에서 가장 효과적인 조합은 README 상단에 GIF(5-10초, 핵심 행동 1개)를 넣고, 그 아래에 "전체 데모 영상 보기" 링크를 두는 것입니다. GIF로 관심을 끌고, 영상으로 깊이를 전달하는 2단계 구조입니다.

GIF 생성 명령 예시:

```bash
# FFmpeg로 MP4 → GIF 변환 (10초, 800px 폭)
ffmpeg -i demo.mp4 -t 10 -vf "fps=15,scale=800:-1:flags=lanczos" \
  -gifflags +transdiff demo.gif

# 파일 크기가 10MB 초과 시 프레임 수 줄이기
ffmpeg -i demo.mp4 -t 8 -vf "fps=10,scale=640:-1:flags=lanczos" \
  -gifflags +transdiff demo-small.gif
```

### 데모 시드 데이터 설계

빈 화면은 데모의 가장 큰 적입니다. 시드 데이터를 잘 설계하면 첫 방문자가 즉시 의미 있는 화면을 볼 수 있습니다.

```python
# scripts/seed_demo_data.py
"""데모용 시드 데이터 생성 스크립트.

실행: python scripts/seed_demo_data.py
목적: 첫 방문자가 비어 있지 않은 화면을 볼 수 있게 샘플 데이터 삽입
"""
import json
from pathlib import Path

SEED_DATA = {
    "teams": [
        {"name": "Backend", "members": ["Alice", "Bob", "Charlie"]},
        {"name": "Frontend", "members": ["Diana", "Eve"]},
    ],
    "tasks": [
        {
            "title": "API 응답 캐싱 구현",
            "assignee": "Alice",
            "status": "in_progress",
            "due": "2024-03-20",
        },
        {
            "title": "로그인 페이지 리디자인",
            "assignee": "Diana",
            "status": "done",
            "due": "2024-03-15",
        },
        {
            "title": "E2E 테스트 추가",
            "assignee": "Bob",
            "status": "todo",
            "due": "2024-03-25",
        },
    ],
}


def seed(output_path: Path = Path("data/seed.json")) -> None:
    output_path.parent.mkdir(parents=True, exist_ok=True)
    output_path.write_text(json.dumps(SEED_DATA, ensure_ascii=False, indent=2))
    print(f"Seed data written to {output_path}")


if __name__ == "__main__":
    seed()
```

시드 데이터 설계 원칙:

1. **현실적인 이름과 값 사용**: "test1", "foo" 대신 실제 업무처럼 보이는 데이터를 넣습니다.
2. **다양한 상태 포함**: todo, in_progress, done이 모두 있어야 화면이 풍성합니다.
3. **적절한 양**: 너무 적으면 비어 보이고, 너무 많으면 복잡해 보입니다. 5-15건이 적당합니다.
4. **재현 가능**: 스크립트로 언제든 초기 상태를 복원할 수 있어야 합니다.

### 데모 배포 상태 모니터링

공개된 라이브 데모가 죽어 있으면 역효과입니다. 간단한 헬스체크를 설정해 두면 데모가 다운된 채 방치되는 상황을 방지할 수 있습니다.

```yaml
# .github/workflows/demo-health.yml
name: Demo Health Check
on:
  schedule:
    - cron: '0 9 * * 1'  # 매주 월요일 오전 9시
  workflow_dispatch:

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - name: Check demo is alive
        run: |
          STATUS=$(curl -s -o /dev/null -w "%{http_code}" https://my-demo.vercel.app)
          if [ "$STATUS" != "200" ]; then
            echo "Demo is down! Status: $STATUS"
            exit 1
          fi
          echo "Demo is healthy (HTTP $STATUS)"

      - name: Notify on failure
        if: failure()
        uses: slackapi/slack-github-action@v1
        with:
          payload: |
            {"text": "Portfolio demo is down! Check https://my-demo.vercel.app"}
```

주 1회 체크만으로도 "지난달에 죽었는데 몰랐어요"를 방지할 수 있습니다. 채용 담당자가 링크를 클릭하는 그 순간에 데모가 살아 있어야 합니다.

### 데모 공유 시 맥락 전달 전략

데모 링크를 단독으로 보내면 수신자가 맥락 없이 헤맬 수 있습니다. 공유할 때는 항상 한 줄 안내를 함께 보내는 것이 좋습니다.

**이메일/DM으로 공유할 때 템플릿:**

```text
안녕하세요,

팀 일정 통합 도구를 만든 [이름]입니다.
- 라이브 데모: https://task-tracker-demo.vercel.app
- 게스트 계정: guest@demo.com / demo1234
- 30초 영상: https://youtube.com/watch?v=xxx

팀원 선택 → 주간 일정 필터링이 핵심 기능입니다.
피드백 주시면 감사하겠습니다.
```

**README에서 데모 섹션 구성:**

```markdown
## Demo

| 형식 | 링크 | 비고 |
|------|------|------|
| Live | [task-tracker.vercel.app](url) | Guest: guest@demo.com / demo1234 |
| Video | [30초 데모](youtube-url) | 음소거 자막 포함 |
| GIF | 아래 참조 | 핵심 행동 1개 |

![핵심 기능 시연](./assets/demo.gif)
```

핵심은 수신자가 어떤 채널로 받든 30초 안에 핵심을 체험할 수 있어야 한다는 점입니다. 라이브 링크만 보내면 "뭘 봐야 하지?"가 되고, 영상만 보내면 "직접 만져 보고 싶다"가 됩니다. 둘을 함께 제공하면 수신자의 선호에 맞출 수 있습니다. 데모는 기술적 완성도보다 '얼마나 빨리 가치를 느끼게 하는가'에 초점을 맞춰야 합니다.

## 처음 질문으로 돌아가기

- **데모 첫 화면은 왜 가장 중요한 평가 지점일까요?**
  - 방문자는 첫 화면에서 프로젝트의 가치를 판단합니다. 첫 화면이 비어 있거나 로그인 벽이 있으면, 아무리 내부 로직이 훌륭해도 확인하지 않고 떠납니다. 시드 데이터와 게스트 계정으로 즉시 체험 가능한 상태를 만들어야 합니다.
- **시드 데이터와 공유 계정은 무엇을 해결해 줄까요?**
  - 시드 데이터는 빈 화면 문제를 해결하고, 공유 계정은 로그인 장벽을 제거합니다. 둘 다 '방문자가 아무 준비 없이 바로 핵심 기능을 볼 수 있는 상태'를 만드는 도구입니다.
- **라이브 데모와 백업 영상은 어떤 역할 차이가 있을까요?**
  - 라이브 데모는 실시간 상호작용을 증명하지만 장애 위험이 있습니다. 백업 영상은 안정적이지만 상호작용이 불가합니다. 둘을 병행하면 어떤 상황에서도 데모가 전달됩니다.
