# Portfolio Project 101 (9/10): 면접에서 설명하기

면접에서 프로젝트 설명이 길어질수록 평가가 좋아지는 것은 아닙니다. 오히려 짧은 시간 안에 문제, 판단, 결과를 또렷하게 말하는 편이 훨씬 강합니다. 면접관이 듣고 싶은 것은 기술 이름 목록이 아니라, 어떤 문제를 이해했고 어떤 기준으로 결정을 내렸는지입니다.

이 글은 Portfolio Project 101 시리즈의 9번째 글입니다. 여기서는 포트폴리오 프로젝트를 면접에서 설명할 때 짧은 시간 안에 핵심을 전달하는 구조를 살펴보겠습니다. 목표는 많이 말하는 것이 아니라, 판단력이 들리는 답변을 만드는 데 있습니다.


![Portfolio Project 101 9장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/portfolio-project-101/09/09-01-diagram.ko.png)
*Portfolio Project 101 9장 흐름 개요*
> 같은 프로젝트도 기술 면접에서는 아키텍처 결정을 강조하고, 행동 기반 면접에서는 어려움과 해결 과정을 강조합니다. 청중에 맞춤 설명이 신뢰를 만듭니다.

## 먼저 던지는 질문

- 면접관은 코드 양보다 무엇을 더 듣고 싶어 할까요?
- STAR 구조는 왜 짧은 답변에서 특히 강할까요?
- 숫자와 트레이드오프는 답변 설득력을 어떻게 바꿀까요?

## 왜 중요한가

면접은 짧고, 기억에는 서사가 남습니다. 비슷한 기술 스택을 가진 지원자는 많지만, 문제를 구조적으로 설명하는 지원자는 많지 않습니다. 그래서 같은 프로젝트라도 어떻게 말하느냐에 따라 인상이 크게 달라집니다.

또 면접은 저장소를 함께 읽는 시간이 아닙니다. 대부분은 말로 먼저 설득해야 하고, 그다음에 세부 질문이 이어집니다. 처음 1~2분 설명이 흐리면 뒤 질문도 흩어집니다. 반대로 구조가 분명하면 면접관은 더 깊은 질문을 하기가 쉬워집니다.

## 머릿속에 먼저 그릴 그림

프로젝트 설명은 상황, 과제, 행동, 결과 순서로 정리하면 놓치기 쉽지 않습니다.

이 구조는 답변이 자연스럽게 질문을 예방하게 만듭니다. 왜 이걸 만들었는지, 정확히 무엇을 해야 했는지, 본인이 무엇을 했는지, 결국 어떤 결과가 나왔는지가 한 흐름으로 이어지기 때문입니다.

## 핵심 용어

- **답변 구조**: 상황, 과제, 행동, 결과를 순서대로 말하는 설명 틀입니다.
- **짧은 요약 답변**: 2분 안팎으로 끝내는 프로젝트 소개입니다.
- **트레이드오프**: 어떤 선택을 하면서 함께 감수한 비용이나 포기입니다.
- 지표: 결과를 숫자로 보여 주는 기준입니다.
- **후속 질문**: 첫 답변 뒤에 이어지는 추가 질문입니다.

## 바꾸기 전과 후

**Before**: "Flask로 API를 만들었습니다"처럼 구현 사실만 말합니다.

**After**: "30명이 동시에 쓰는 일정 조회 문제를 풀기 위해 Flask와 Redis를 사용했고 평균 응답 시간을 120ms 수준으로 유지했습니다"처럼 문제와 결과를 함께 말합니다.

둘 다 사실일 수 있지만, 후자의 답변은 훨씬 더 많은 정보를 짧게 담습니다. 면접에서는 바로 그 차이가 중요합니다. 구현을 했다는 사실보다, 어떤 문제를 어떤 기준으로 해결했는지가 더 강하게 남기 때문입니다.

## 단계별로 살펴보기

### 1단계 — 상황

먼저 왜 이 프로젝트가 필요했는지 설명합니다.

```python
situation = "The team schedule kept getting lost across tools"
```

상황 설명은 길게 할 필요가 없습니다. 다만 너무 추상적이면 뒤 설명이 힘을 잃습니다. 실제 불편이 느껴지는 문장으로 적는 편이 좋습니다.

### 2단계 — 과제

그 상황에서 무엇을 해결해야 했는지 분명하게 말합니다.

```python
task = "Show every schedule on a single screen"
```

과제는 프로젝트 범위를 정리해 줍니다. 무엇을 해결하지 않았는지도 함께 암시되기 때문에, 범위 감각이 있는 답변으로 들리게 만듭니다.

### 3단계 — 행동

이제 여러분이 실제로 한 일을 말합니다.

```python
action = ["Flask API", "PostgreSQL", "Deploy to Render"]
```

여기서 중요한 것은 기술 이름 자체보다 선택과 기여입니다. 왜 이 구성을 택했는지, 어떤 대안을 버렸는지를 한 문장 곁들이면 답변이 훨씬 살아납니다.

### 4단계 — 결과

결과는 가능한 한 숫자로 남깁니다.

```python
result = {"users": 30, "latency_ms": 120}
```

숫자는 답변의 증거 역할을 합니다. 사용량, 응답 시간, 배포 횟수, 오류율 감소처럼 무엇이든 좋습니다. 결과가 없으면 설명은 성실해 보여도 설득력은 약해집니다.

### 5단계 — 학습

마지막에는 프로젝트를 통해 얻은 판단을 말합니다.

```python
lesson = "Small MVPs survive"
```

이 한 문장이 답변을 닫습니다. 면접관은 기술 사용 경험뿐 아니라, 그 경험에서 무엇을 배웠는지도 듣고 싶어 합니다. 그래서 마지막 학습 문장은 꽤 중요합니다.

## 이 코드에서 먼저 볼 점

- 답변 구조는 암기 틀이 아니라 설명 순서를 안정시키는 장치입니다.
- 숫자는 결과를 뒷받침하는 증거입니다.
- 학습 문장은 답변의 마침표이자 다음 질문의 연결점이 됩니다.

## 자주 하는 실수

1. 기술 이름만 나열하고 문제와 결과를 말하지 않는 경우
2. 숫자가 하나도 없어 프로젝트 규모나 효과를 짐작하기 어려운 경우
3. 왜 그런 선택을 했는지, 어떤 트레이드오프가 있었는지 설명하지 못하는 경우
4. 팀 프로젝트에서 본인 기여가 흐릿하게 들리는 경우
5. 이 프로젝트를 통해 무엇을 배웠는지 답하지 못하는 경우

이 실수들은 결국 면접관 입장에서 "이 지원자가 정말 이 프로젝트를 이해하고 있나"라는 의문으로 이어집니다. 말의 구조를 세우는 이유도 바로 여기에 있습니다.

## 실무에서는 이렇게 본다

실무에서도 회고나 장애 분석을 할 때 비슷한 구조를 씁니다. 어떤 상황이 있었고, 무엇이 목표였고, 어떤 조치를 했으며, 결과와 교훈이 무엇이었는지 정리해야 다음 판단에 도움이 되기 때문입니다.

면접 답변도 같은 원리입니다. 단지 더 짧고 더 선명해야 할 뿐입니다. 잘 정리된 답변은 프로젝트 설명이자 판단 요약본 역할을 합니다.

### 실무 회고 문서와 면접 답변의 구조 비교

실무에서 작성하는 포스트모템(사후 분석) 문서와 면접 답변의 구조를 비교하면 놀라울 정도로 닮아 있습니다.

| 구분 | 포스트모템 문서 | 면접 답변 |
| --- | --- | --- |
| 상황 | 장애 발생 시점과 영향 범위 | 프로젝트 배경과 문제 |
| 목표 | 복구 목표(MTTR, 영향 최소화) | 해결하려는 과제 |
| 조치 | 타임라인별 행동 기록 | 본인이 한 기술 선택과 구현 |
| 결과 | 복구 시점, 영향 사용자 수 | 지표, 사용자 반응, 배포 결과 |
| 교훈 | 재발 방지 액션 아이템 | 프로젝트에서 배운 점 |

```text
포스트모템 예시 (요약):
- 상황: 2024-03-15 14:00 KST, 결제 API 504 타임아웃 급증
- 목표: 30분 내 정상 응답률 99.9% 복구
- 조치: DB 커넥션 풀 고갈 확인 → 풀 사이즈 20→50 확대 → 슬로우 쿼리 인덱스 추가
- 결과: 14:22 복구, 영향 사용자 약 1,200명, 매출 손실 추정 0원(재시도 성공)
- 교훈: 커넥션 풀 모니터링 알림 추가, 슬로우 쿼리 주간 리뷰 도입
```

이 구조를 면접 답변에 그대로 적용하면:

```text
면접 답변 변환:
"결제 API 타임아웃이 급증하는 장애를 경험했습니다. (Situation)
30분 내 복구가 목표였습니다. (Task)
DB 커넥션 풀 고갈을 확인하고 풀 사이즈 확대와 인덱스 추가로 대응했습니다. (Action)
22분 만에 복구했고, 이후 커넥션 풀 모니터링을 도입했습니다. (Result)
배운 점은 모니터링 없는 리소스는 언제든 고갈된다는 것입니다."
```

실무 경험이 있다면 포스트모템을 STAR로 변환하는 연습이 가장 효과적입니다. 포트폴리오 프로젝트라면 장애 대신 개발 과정의 난관을 같은 구조로 정리하면 됩니다.

### 팀 프로젝트에서 본인 기여 말하기

팀 프로젝트를 설명할 때 가장 흔한 실수는 "우리 팀이 ~했습니다"로 시작하는 것입니다. 면접관은 팀 전체 결과보다 지원자 개인의 기여와 판단을 듣고 싶어 합니다.

**기여를 분명하게 만드는 세 가지 원칙:**

1. **역할 경계를 먼저 말한다**: "4인 팀에서 백엔드 API와 배포 파이프라인을 맡았습니다."
2. **의사결정 주체를 밝힌다**: "제가 제안해서 채택된 것은 ~입니다."
3. **기여 범위를 수치로 보여 준다**: "전체 커밋의 35%, API 엔드포인트 12개 중 8개를 구현했습니다."

```python
# 팀 기여 정리 템플릿
contribution = {
    "team_size": 4,
    "my_role": "Backend API + CI/CD pipeline",
    "key_decisions": [
        "DB 선택: SQLite → PostgreSQL 마이그레이션 제안",
        "테스트 전략: pytest + GitHub Actions CI 도입",
    ],
    "quantified": {
        "commits": "87 / 250 (35%)",
        "endpoints_built": "8 / 12",
        "test_coverage_contribution": "단위 테스트 45개 작성",
    },
}
```

**피해야 할 표현과 대안:**

| 피해야 할 표현 | 대안 |
| --- | --- |
| "우리 팀이 만들었습니다" | "제가 맡은 부분은 ~입니다" |
| "다 같이 했습니다" | "제가 제안하고 구현한 것은 ~입니다" |
| "팀워크가 좋았습니다" | "의견 충돌 시 제가 ~로 중재했습니다" |
| "협업했습니다" | "API 명세를 제가 작성해 프론트엔드와 병렬 개발했습니다" |

팀 프로젝트에서 기여를 명확히 말하는 것은 다른 팀원을 깎아내리는 것이 아닙니다. 면접관이 "이 사람이 팀에서 어떤 역할을 하는지" 판단할 수 있게 도와주는 것입니다.

### 숫자가 없을 때 지표 만드는 방법

포트폴리오 프로젝트는 실 사용자가 적거나 없을 수 있습니다. 그래도 숫자는 만들 수 있습니다.

**측정 가능한 지표 유형:**

| 카테고리 | 지표 예시 | 측정 방법 |
| --- | --- | --- |
| 성능 | 응답시간, 처리량 | k6, locust, time 명령어 |
| 품질 | 테스트 커버리지, 린트 통과율 | pytest --cov, ruff |
| 규모 | 코드 라인 수, 엔드포인트 수 | cloc, 직접 카운트 |
| 개발 효율 | 개발 기간, 커밋 수, PR 수 | git log --oneline \| wc -l |
| 자동화 | CI 빌드 시간, 배포 빈도 | GitHub Actions 로그 |

```bash
# 프로젝트 지표 수집 예시
echo "=== 프로젝트 규모 ==="
cloc src/ --quiet
echo ""
echo "=== 테스트 커버리지 ==="
pytest --cov=src --cov-report=term-missing --quiet 2>/dev/null | tail -3
echo ""
echo "=== 개발 기간 ==="
git log --format='%ai' | tail -1  # 첫 커밋
git log --format='%ai' | head -1  # 마지막 커밋
echo ""
echo "=== 커밋 수 ==="
git log --oneline | wc -l
```

면접에서 이 지표를 활용하는 예시:

- "2주 동안 87개 커밋으로 MVP를 완성했습니다."
- "테스트 커버리지 82%를 유지하면서 엔드포인트 12개를 구현했습니다."
- "k6로 부하 테스트를 돌려 동시 접속 50명까지 응답시간 200ms 이하를 확인했습니다."

숫자가 크지 않아도 괜찮습니다. 중요한 것은 측정하려는 태도와 기준이 있다는 점입니다. "성능이 좋습니다"와 "평균 응답시간 120ms입니다"는 같은 사실이지만 신뢰도가 완전히 다릅니다.

추가로, 면접 당일에 숫자를 기억하지 못할 수도 있습니다. 프로젝트 README나 ADR에 지표를 기록해 두면 면접 준비 시간이 크게 줄어듭니다. 측정 스크립트를 CI에 포함시켜 두면 매 배포마다 최신 수치가 갱신되므로, 면접 직전에 따로 테스트를 돌릴 필요도 없습니다. 이것이 이 시리즈에서 테스트와 CI를 강조하는 이유이기도 합니다.

면접은 결국 신뢰의 게임입니다. 숫자는 그 신뢰를 만드는 가장 빠른 도구입니다.

## 체크리스트

- [ ] 2분 안에 답변을 마칠 수 있다.
- [ ] 최소 한 개 이상의 숫자를 포함했다.
- [ ] 최소 한 개의 트레이드오프를 설명할 수 있다.
- [ ] 마지막에 배운 점을 한 문장으로 정리할 수 있다.

## 연습 문제

1. 여러분 프로젝트를 네 문장 안에 정리해 보세요.
2. 지금 답변에 넣을 수 있는 숫자 하나를 골라 보세요.
3. 기술 선택 하나를 골라, 왜 다른 대안을 버렸는지 한 줄로 적어 보세요.

## 정리와 다음 글

면접에서 포트폴리오를 설명할 때 핵심은 많이 말하는 것이 아니라 구조 있게 말하는 것입니다. 상황, 과제, 행동, 결과, 학습이 차례로 나오면 짧은 답변도 훨씬 선명해집니다. 여기에 숫자와 트레이드오프, 본인 기여가 더해지면 프로젝트는 단순한 구현 사례가 아니라 판단 사례로 읽힙니다.

다음 글에서는 시리즈를 마무리하면서 공개 직전에 무엇을 점검하면 포트폴리오 전체 완성도를 높일 수 있는지 체크리스트 관점에서 정리해 보겠습니다.

### 데모 영상 스크립트 구조(면접 제출용)

면접 전에 포트폴리오 링크와 함께 2~3분 데모 영상을 제출하는 경우가 늘고 있습니다. 이때 영상은 기능 소개보다 문제 해결 서사를 보여 주는 쪽이 효과적입니다. 아래 구조를 기준으로 스크립트를 준비하면 메시지가 선명해집니다.

| 구간 | 시간 | 핵심 메시지 |
| --- | --- | --- |
| 오프닝 | 20초 | 어떤 문제를 해결하는 프로젝트인지 |
| 핵심 시연 | 90초 | 사용자가 실제로 얻는 변화 |
| 기술 선택 | 40초 | 왜 이 구조를 선택했는지 |
| 결과/학습 | 30초 | 수치 결과와 다음 개선 계획 |

```markdown
오프닝 예시:
"팀 일정이 여러 도구에 분산되어 계획 회의 준비에 시간이 오래 걸리는 문제를 해결하기 위해 이 프로젝트를 만들었습니다."
```

이 스크립트는 STAR 답변과도 호환됩니다. 영상에서 말한 구조를 면접에서 그대로 재사용할 수 있어 준비 효율이 높습니다.

### 발표 슬라이드 구성(5장 템플릿)

면접 발표 과제가 있을 때는 슬라이드를 많이 만드는 것보다 메시지 밀도를 높이는 편이 중요합니다. 아래 5장 구조를 추천합니다.

1. 문제 정의: 사용자/상황/불편
2. 해결 접근: 범위와 아키텍처 선택
3. 핵심 데모: 가장 중요한 사용자 흐름
4. 결과 지표: 성능/사용성 변화
5. 회고: 트레이드오프와 다음 단계

| 슬라이드 | 넣어야 할 요소 | 피해야 할 요소 |
| --- | --- | --- |
| 문제 정의 | 실제 사용자 문장 1개 | 기술 스택 나열 |
| 해결 접근 | 선택 기준 3개 | 코드 스크린샷 과다 |
| 결과 지표 | 전/후 비교 수치 | 모호한 "개선됨" 표현 |

슬라이드의 목적은 깊이를 숨기는 것이 아니라, 질문을 잘 유도하는 것입니다. 핵심 판단을 먼저 보여 주면 면접관의 후속 질문도 더 의미 있게 이어집니다.

### 예상 질문 대비 표

면접 대비는 암기보다 질문 분류가 효과적입니다. 아래 표처럼 질문 유형별 기본 답변 축을 준비해 두면 긴장 상황에서도 구조를 유지할 수 있습니다.

| 질문 유형 | 예시 질문 | 답변 축 |
| --- | --- | --- |
| 문제 정의 | 왜 이 문제를 골랐나요? | 사용자 불편 + 영향 |
| 기술 선택 | 왜 FastAPI였나요? | 제약 조건 + 대안 비교 |
| 품질 | 테스트는 어떻게 구성했나요? | 단위/통합/CI 흐름 |
| 운영 | 배포 장애가 나면 어떻게 하나요? | 런북 + 롤백 절차 |
| 확장 | 다음 단계는 무엇인가요? | 우선순위 + 리스크 |

이 표를 기반으로 1분 답변, 3분 답변 두 버전을 각각 준비해 두면 대부분의 인터뷰 형식에 대응할 수 있습니다.

### 2분 답변 템플릿(암기용이 아닌 구조용)

면접에서는 문장 암기보다 구조 암기가 중요합니다. 아래 템플릿을 프로젝트에 맞게 채우면 대부분의 소개 질문에 대응할 수 있습니다.

```markdown
[상황] 어떤 사용자의 어떤 불편이 있었는지 1문장
[과제] 내가 해결해야 했던 목표 1문장
[행동] 핵심 선택 2개 + 이유 1개
[결과] 수치 1~2개
[학습] 다음 프로젝트에 가져갈 기준 1문장
```

### 면접 후속 질문 확장표

| 기본 질문 | 확장 질문 | 준비 포인트 |
| --- | --- | --- |
| 왜 이 기술을 골랐나 | 대안은 무엇이었나 | ADR 근거 |
| 성과가 있었나 | 어떻게 측정했나 | 지표 수집 방식 |
| 어려움이 있었나 | 어떻게 복구했나 | 런북/실패 사례 |

이 표를 기준으로 답변을 준비하면, 예상 밖 질문이 와도 구조를 유지할 수 있습니다. 결국 면접에서 강한 답변은 정보량보다 정리 수준에서 차이가 납니다.

### 모의 면접 스크립트(혼자 연습용)

아래 스크립트를 소리 내어 읽으면서 연습합니다. 스퇱워치로 시간을 재면 더 효과적입니다.

```text
면접관: "이 프로젝트 소개해 주세요."

나: "팀 일정이 여러 도구에 흩어져 계획 회의 준비에 시간이 오래
     걸리는 문제를 해결하기 위해 만들었습니다. (Situation)
     한 화면에서 모든 일정을 보여 주는 것이 목표였습니다. (Task)
     FastAPI와 PostgreSQL로 2주 안에 MVP를 만들었습니다.
     Flask 대신 FastAPI를 선택한 이유는 자동 문서화와 비동기
     처리 때문입니다. (Action)
     결과적으로 30명이 사용하고, 평균 응답시간 120ms를 유지했습니다. (Result)
     배운 점은 작은 MVP가 측정 가능한 결과를 더 잘 만든다는 것입니다."

면접관: "Flask와 비교했다고 했는데, 구체적으로 어떤 기준으로 결정했나요?"

나: "2주 마감이어서 개발 속도가 중요했습니다.
     FastAPI는 타입 힌트와 Swagger 자동 생성 덕분에
     프론트엔드 협업 시간을 50% 줄일 수 있었습니다.
     트레이드오프는 Flask 대비 레퍼런스가 적다는 점이었는데,
     공식 문서가 충분해 문제없었습니다."
```

이 스크립트의 핵심은:
- 각 문장에 STAR 요소를 표시해 구조를 의식하는 것
- 후속 질문에도 동일한 구조(기준-선택-트레이드오프)로 답하는 것
- 구체적 수치를 하나 이상 포함하는 것

이 연습을 3회 이상 반복하면 면접 자리에서도 자연스럽게 나옵니다.

### 기술 면접 vs 행동 면접 답변 전략

같은 프로젝트라도 면접 유형에 따라 강조점이 달라집니다.

| 구분 | 기술 면접 | 행동 면접 |
| --- | --- | --- |
| 중심 메시지 | 아키텍처 결정과 기술 깊이 | 협업, 어려움 극복, 성장 |
| 코드 언급 | 핸들링 로직, 설계 선택 | 최소한(문맥용) |
| 지표 강조 | 성능, 안정성, 확장성 | 사용자 영향, 팀 기여 |
| 트레이드오프 | 기술 대안 비교 | 우선순위 판단 |
| 후속 질문 예상 | "스케일링은?" "장애 대응은?" | "갈등은?" "실패하면?" |

두 유형 모두에 대응하려면 하나의 프로젝트에 대해 두 버전의 답변을 준비해 두는 것이 좋습니다. 기술 버전은 "왜 이 구조인가", 행동 버전은 "어떤 어려움을 어떻게 넘었는가"에 초점을 둡니다.

### "모른다"는 답변의 구조

면접에서 모르는 질문이 나올 때 "모릅니다"로 끝내면 대화가 끊깁니다. 아래 구조로 답하면 배우는 자세를 보여 줄 수 있습니다.

```text
1. 정직하게 인정: "그 부분은 아직 경험이 없습니다."
2. 유사 경험 연결: "다만, 비슷한 상황으로는 ..."
3. 학습 계획 제시: "다음에는 ...을 시도해 보려고 합니다."
```

예시:
- "대규모 트래픽 경험은 없습니다. 다만 30명 동시 접속 시 응답시간을 120ms로 유지한 경험이 있고, 스케일링 필요 시 캐시 레이어 추가를 먼저 검토하겠습니다."

이 구조는 "모른다"를 "아직 경험은 없지만 학습 계획이 있다"로 바꾸어 줍니다.

## 처음 질문으로 돌아가기

- **면접관은 코드 양보다 무엇을 더 듣고 싶어 할까요?**
  - 면접관은 코드를 얼마나 많이 썼는지보다, 어떤 문제를 이해하고 어떤 기준으로 결정했는지를 듣고 싶어 합니다. 판단 과정이 들리는 답변이 가장 강합니다.
- **STAR 구조는 왜 짧은 답변에서 특히 강할까요?**
  - STAR 구조는 상황-과제-행동-결과를 순서대로 배치해 독자(면접관)의 이해 흐름과 일치합니다. 2분 안에 끝내더라도 구조가 있으면 가치가 전달됩니다.
- **숫자와 트레이드오프는 답변 설득력을 어떻게 바꿀까요?**
  - "성능이 좋아졌다"는 주관적 표현이지만, "280ms에서 120ms로 개선"은 증거입니다. 트레이드오프를 함께 말하면 "내가 정말 이해하고 선택했다"는 신호를 줍니다.
