# Portfolio Project 101 (10/10): 포트폴리오 개선 체크리스트

포트폴리오는 만드는 일보다 다듬는 일에서 품질 차이가 크게 납니다. 기능은 얼추 완성됐는데도 어딘가 덜 준비된 느낌이 드는 프로젝트가 있습니다. 대개는 구현이 부족해서가 아니라, 공개 전 마지막 점검이 빠졌기 때문입니다. README가 오래되었거나, 데모 링크가 깨졌거나, 프로젝트 서사가 흩어져 있으면 첫인상에서 바로 손해를 봅니다.

이 글은 Portfolio Project 101 시리즈의 마지막 글입니다. 여기서는 공개 직전에 무엇을 점검하면 포트폴리오 전체 완성도를 끌어올릴 수 있는지, 처음 방문한 사람의 시선으로 다시 보는 체크리스트를 정리하겠습니다.

## 먼저 던지는 질문

- 공개 전 마지막으로 확인해야 할 핵심 영역은 무엇일까요?
- README, 데모, 코드, 서사, 공개 채널은 왜 따로 점검해야 할까요?
- 작성자 시선과 처음 방문한 사람의 시선은 어디에서 가장 크게 어긋날까요?

## 왜 중요한가

첫인상은 짧은 시간 안에 결정됩니다. 방문자는 대부분 README를 읽고, 데모를 눌러 보고, 저장소 구조를 훑고, 프로젝트가 왜 가치 있는지 판단합니다. 이 과정에서 작은 끊김이 몇 번만 생겨도 전체 인상이 크게 떨어집니다.

또 포트폴리오는 한 번만 보여 주고 끝나는 자료가 아닙니다. 링크를 여러 사람에게 보내고, 시간이 지난 뒤 다시 꺼내게 됩니다. 그래서 공개 직전 점검은 그 순간의 품질 관리이면서, 이후 유지 비용을 줄이는 작업이기도 합니다.

## 머릿속에 먼저 그릴 그림

공개 전 점검은 README, 데모, 코드, 서사, 공개 채널 순서로 보면 빠뜨릴 부분이 적습니다.

이 순서는 독자가 프로젝트를 만나는 순서와도 비슷합니다. 설명을 보고, 실제로 눌러 보고, 저장소를 살펴본 뒤, 이 프로젝트를 어떻게 기억할지 정하고, 마지막으로 외부 채널에서 다시 마주칩니다. 그래서 점검도 같은 순서가 잘 맞습니다.

## 핵심 용어

- **기본 동작 점검**: 핵심 기능이 살아 있는지만 빠르게 확인하는 점검입니다.
- **처음 보는 시선**: 프로젝트를 전혀 모르는 방문자의 시선입니다.
- **깨진 링크**: 더 이상 열리지 않는 링크입니다.
- **오래된 정보**: 현재 코드나 배포 상태와 맞지 않는 설명입니다.
- 공개: 프로젝트를 외부에 공유하는 시점입니다.

## 바꾸기 전과 후

**Before**: README는 작성자만 이해하고, 데모는 열리더라도 사용 경로가 모호합니다.

**After**: 처음 보는 사람도 짧은 시간 안에 프로젝트를 이해하고 핵심 흐름을 확인할 수 있습니다.

포트폴리오 품질은 기능 수보다 마찰 개수로 체감됩니다. 어디에서 멈추는지, 어디에서 의문이 생기는지가 줄어들수록 완성도는 높아집니다.

## 단계별로 살펴보기

### 1단계 — README 검수

README에는 프로젝트가 무엇인지, 왜 만들었는지, 어떻게 실행하는지, 데모가 어디 있는지, 라이선스가 무엇인지가 보여야 합니다.

```python
readme = ["What", "Why", "How", "Demo", "License"]
```

이 다섯 요소가 있으면 처음 보는 사람도 저장소를 해석할 기본 재료를 얻습니다. 특히 Why와 Demo가 빠지면 프로젝트 가치와 실제성을 동시에 놓치기 쉽습니다.

### 2단계 — 데모 검수

링크가 열리는지만 보지 말고, 핵심 흐름이 막히지 않는지도 함께 확인합니다.

```python
demo = {"url": "https://demo.example.com", "uptime": 0.99}
```

로그인부터 막히거나 시드 데이터가 비어 있으면 방문자는 금방 이탈합니다. 포트폴리오 데모는 기능 전체보다 핵심 흐름이 살아 있는지가 더 중요합니다.

### 3단계 — 코드 검수

코드 점검은 복잡한 리팩터링이 아니라 공개 가능한 상태인지 보는 확인에 가깝습니다.

```python
code = {"tests": True, "lint": True, "ci": True}
```

테스트가 깨진 채 남아 있거나 기본 실행이 되지 않으면 README가 좋아도 신뢰가 오래 가지 않습니다. 포트폴리오는 코드와 설명이 같은 방향을 봐야 합니다.

### 4단계 — 서사 검수

프로젝트가 어떤 문제를 풀었고, 어떤 해결을 시도했고, 결과와 학습이 무엇인지 한 번에 말할 수 있어야 합니다.

```python
story = ["Problem", "Solution", "Result", "Lesson"]
```

서사가 정리되지 않으면 저장소, 블로그 글, 면접 답변이 서로 따로 놀기 쉽습니다. 포트폴리오 힘은 결과물 개수보다 일관된 설명에서 나옵니다.

### 5단계 — 공개 채널 점검

마지막에는 어디에 어떻게 공유할지 확인합니다.

```python
launch = ["GitHub", "Blog", "LinkedIn"]
```

공개 채널마다 보여 주는 정보는 조금씩 다릅니다. GitHub는 원본, 블로그는 과정, 외부 소개 글은 요약에 가깝습니다. 그래서 링크가 서로 자연스럽게 이어지도록 정리해 두는 편이 좋습니다.

## 이 코드에서 먼저 볼 점

- README는 프로젝트 입구라서 여기서 막히면 뒤 내용도 읽히기 어렵습니다.
- 데모는 말보다 강한 증거입니다.
- 서사는 프로젝트를 기억하게 만드는 장치입니다.
- 공개 채널 간 링크가 자연스럽게 연결되면 전체 신뢰도가 올라갑니다.
- 품질 점검은 일회성 이벤트가 아니라 반복 가능한 루틴일 때 효과가 있습니다.

### 스크린샷과 시각 자료 점검

README와 블로그에 스크린샷이 없으면 방문자는 결과를 직접 돌려보기 전에는 프로젝트가 어떻게 생겼는지 알 수 없습니다. 스크린샷은 데모의 축약판입니다.

**스크린샷 체크리스트:**

| 항목 | 확인 질문 |
| --- | --- |
| 존재 여부 | 핵심 화면 스크린샷이 1장 이상 있는가 |
| 최신성 | 스크린샷이 현재 UI와 일치하는가 |
| 화질 | 글자가 읽히고, 주요 요소가 보이는가 |
| 맥락 | 스크린샷 앞뒤에 설명이 있는가 |
| 경로 | 이미지 링크가 깨지지 않았는가 |

스크린샷은 프로젝트가 "실제로 동작한다"는 가장 직관적인 증거입니다. 특히 백엔드 API 프로젝트는 UI가 없어도 Swagger 화면이나 테스트 결과 스크린샷을 대신 넣을 수 있습니다.

## 자주 하는 실수

1. README가 현재 코드와 배포 상태를 반영하지 못하는 경우
2. 데모 링크가 깨졌거나, 열려도 핵심 흐름이 작동하지 않는 경우
3. 테스트나 기본 검증 상태를 확인하지 않고 공개하는 경우
4. 라이선스나 사용 조건을 남기지 않아 저장소 성격이 모호한 경우
5. 스크린샷이나 설명이 부족해 방문자가 결과를 빠르게 파악하지 못하는 경우

이 다섯 가지는 모두 처음 보는 사람의 시점을 놓쳤을 때 생깁니다. 작성자는 익숙해서 지나치지만, 방문자는 그 자리에서 바로 멈춥니다. 특히 채용 담당자는 수십 개 포트폴리오를 비교하므로, 작은 마찰 하나가 탈락의 기준이 될 수 있습니다.

## 실무에서는 이렇게 본다

오픈소스 프로젝트나 제품 릴리스도 공개 전에 비슷한 체크리스트를 반복합니다. 문서, 링크, 테스트, 릴리스 노트, 배포 상태를 확인하는 이유는 단순합니다. 공개 이후에 발견되는 사소한 실수는 생각보다 크게 보이기 때문입니다.

개인 포트폴리오도 다르지 않습니다. 규모는 작아도 공개 순간에는 제품처럼 읽힙니다. 그래서 마지막 점검은 미적 다듬기가 아니라 신뢰를 높이는 운영 작업에 가깝습니다.

실제로 오픈소스 프로젝트는 릴리스 전에 아래와 같은 체크리스트를 쓴니다:

```markdown
## Release Checklist (v1.2.0)
- [ ] CHANGELOG 업데이트
- [ ] README 실행 명령 정합성 확인
- [ ] 데모 환경 시드 데이터 갱신
- [ ] CI 전체 녹색 확인
- [ ] 링크 전수 점검 (linkchecker)
- [ ] 스크린샷 최신화
- [ ] 마이그레이션 스크립트 테스트
- [ ] 성능 벤치마크 재측정
```

이 패턴을 개인 포트폴리오에 그대로 적용할 수 있습니다. `RELEASE_CHECKLIST.md`를 저장소에 넣어 두면, 공개 전마다 똑같은 기준으로 점검할 수 있어 실수를 줄입니다.

여기서 한 걸음 더 나가면, 공개 전 점검을 워크플로우로 자동화하는 것도 가능합니다:

```yaml
# .github/workflows/pre-release-check.yml
name: Pre-Release Check
on:
  pull_request:
    branches: [main]
    types: [opened, synchronize]

jobs:
  validate:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Validate README links
        run: |
          pip install linkchecker
          linkchecker README.md --check-extern
      - name: Run full test suite
        run: |
          pip install -r requirements.txt
          pytest --tb=short --strict-markers
      - name: Check demo health
        run: |
          STATUS=$(curl -s -o /dev/null -w '%{http_code}' https://demo.example.com/health)
          if [ "$STATUS" != "200" ]; then
            echo "::error::Demo health check failed (HTTP $STATUS)"
            exit 1
          fi
```

이렇게 하면 main 브랜치에 머지하기 전에 항상 기본 품질이 검증됩니다. 포트폴리오를 "운영"하는 자세는 면접에서도 긍정적으로 읽힙니다. 단순히 만들어 놓은 것이 아니라 유지하고 있다는 신호를 주기 때문입니다.

## 체크리스트

- [ ] README에 What, Why, How, Demo, License가 있다.
- [ ] 데모 링크가 실제로 작동하고 핵심 흐름을 확인할 수 있다.
- [ ] 테스트나 기본 검증 상태를 설명할 수 있다.
- [ ] 문제, 해결, 결과, 학습의 서사를 한 번에 말할 수 있다.
- [ ] GitHub와 블로그, 외부 공유 채널이 자연스럽게 연결된다.

## 정리와 다음 글

포트폴리오를 공개하기 전 마지막 점검은 품질을 마무리하는 단계입니다. README, 데모, 코드, 서사, 공개 채널을 차례로 확인하면 구현보다 전달에서 생기는 문제를 크게 줄일 수 있습니다. 특히 처음 보는 사람의 시선으로 다시 읽고 눌러 보고 실행해 보는 습관이 프로젝트 완성도를 눈에 띄게 높여 줍니다.

이 글로 Portfolio Project 101 시리즈를 마칩니다. 다음 프로젝트를 만들 때도 같은 점검 루틴을 반복하면, 결과물 수보다 완성도에서 더 큰 차이가 쌓이게 됩니다.

### 이력서 연동 방법

포트폴리오를 개선할 때 가장 자주 놓치는 부분은 "이력서와의 연결"입니다. 프로젝트가 좋아도 이력서에서 찾기 어렵거나, 이력서 문장이 저장소 내용과 맞지 않으면 신뢰가 떨어집니다. 연동의 핵심은 링크 추가가 아니라 동일한 메시지를 유지하는 것입니다.

| 이력서 섹션 | 연결 요소 | 작성 원칙 |
| --- | --- | --- |
| 프로젝트 요약 | 문제 한 줄 + 데모 링크 | 기술명보다 결과 중심 |
| 기술 스택 | 핵심 3~5개만 | 프로젝트 본문과 일치 |
| 성과 | 수치 1~2개 | 측정 방법 설명 가능 |
| 역할 | 본인 기여 범위 | 팀/개인 구분 명확 |

좋은 연동 예시는 "일정 조회 시간을 40분에서 18분으로 단축한 프로젝트(데모 링크)"처럼 문제-행동-결과를 짧게 담습니다. 반대로 "FastAPI 프로젝트 구현"처럼 결과 없는 문장은 강도가 약합니다.

### 면접 질문 대비 표(최종 점검형)

시리즈 마지막 단계에서는 프로젝트 완성도뿐 아니라 설명 준비도를 함께 점검해야 합니다. 아래 표를 이용해 예상 질문에 대한 답변 상태를 확인할 수 있습니다.

| 질문 | 준비 수준 점검 | 보강 방법 |
| --- | --- | --- |
| 왜 이 문제를 골랐나요? | 문제 배경을 30초 안에 설명 가능 | 사용자 사례 1개 추가 |
| 왜 이 기술을 선택했나요? | 대안 2개와 트레이드오프 설명 가능 | ADR 링크 연결 |
| 어떤 성과가 있었나요? | 수치 1개 이상 제시 가능 | 전/후 비교표 작성 |
| 실패 경험은 있나요? | 장애/실패 사례 1개 보유 | 런북/회고 문서 추가 |
| 다음 개선은 무엇인가요? | 우선순위 3개 제시 가능 | Roadmap 최신화 |

이 표는 단순 면접 대비가 아니라 프로젝트 설명 일관성을 검증하는 도구입니다. 답변이 막히는 질문은 대개 프로젝트 문서에서도 비어 있는 부분입니다.

### 최종 공개 전 7일 루틴

완성도를 높이려면 마지막 주에 무리한 기능 추가를 멈추고 품질 루틴을 반복하는 편이 좋습니다.

1일차: README 사실관계 검증, 링크 전수 점검
2일차: 데모 시나리오 리허설, 백업 영상 갱신
3일차: 테스트 안정화, flaky 제거
4일차: ADR/CHANGELOG 최신화
5일차: 이력서 문장과 프로젝트 문서 정합성 점검
6일차: 예상 질문 10개 답변 리허설
7일차: 외부 시선 리뷰(동료 1명 이상)

```markdown
## Final Week Goal
- 기능 추가 금지
- 품질/설명/검증만 개선
- 공개 후 1주 유지보수 계획 수립
```

이 루틴의 목적은 "더 많이 만들기"가 아니라 "이미 만든 것을 정확히 전달하기"입니다. 포트폴리오는 결국 전달 품질에서 차이가 납니다.

### 최종 점검 대시보드 템플릿

공개 직전에는 산발적 메모보다 한 장짜리 대시보드가 유용합니다. 아래 항목을 주간으로 갱신하면 상태를 빠르게 파악할 수 있습니다.

| 영역 | 상태 | 근거 링크 | 다음 액션 |
| --- | --- | --- | --- |
| README 최신화 | Pass/Fail | README 링크 | 실행 섹션 갱신 |
| 데모 동작 | Pass/Fail | 데모 URL | 시드 데이터 보강 |
| 테스트 안정성 | Pass/Fail | CI 링크 | flaky 제거 |
| 서사 일관성 | Pass/Fail | 블로그/이력서 | 수치 표현 통일 |
| 면접 준비 | Pass/Fail | Q&A 문서 | 2분 답변 리허설 |

### 공개 이후 유지보수 체크

포트폴리오는 공개 순간보다 공개 이후가 더 중요합니다. 아래 주기로 유지하면 링크 신뢰도를 오래 지킬 수 있습니다.

1. 주 1회 데모 링크 헬스체크
2. 월 1회 README 사실관계 점검
3. 분기 1회 의존성 업데이트
4. 반기 1회 이력서-프로젝트 정합성 재검토

이 유지보수 루틴은 화려하지 않지만 매우 실용적입니다. 오래 살아 있는 포트폴리오가 결국 면접과 네트워킹에서 더 큰 힘을 발휘합니다.

### GitHub 프로필 README 최적화

GitHub 프로필 README는 저장소를 방문하기 전에 보는 첫 화면입니다. 포트폴리오 프로젝트가 좋아도 프로필이 비어 있으면 방문자는 어떤 프로젝트를 먼저 봐야 할지 모릅니다.

```markdown
# 👋 안녕하세요, [이름]입니다

## 관심 분야
- 백엔드 API 설계와 배포 자동화
- Python, FastAPI, PostgreSQL, Docker

## 대표 프로젝트
| 프로젝트 | 한 줄 설명 | 핵심 기술 | 데모 |
| --- | --- | --- | --- |
| schedule-hub | 팀 일정 통합 조회 | FastAPI, Redis | [Live](https://...) |
| log-analyzer | 서버 로그 이상 탐지 | Python, pandas | [Demo](https://...) |

## 최근 블로그 글
- [면접에서 프로젝트 설명하는 법](https://...)
- [CI/CD 파이프라인 구축기](https://...)
```

프로필 README 작성 원칙:

1. **대표 프로젝트는 3개 이하로 좁힌다**: 많으면 오히려 초점이 흐려집니다.
2. **각 프로젝트에 데모 링크를 넣는다**: 클릭 한 번으로 결과를 확인할 수 있어야 합니다.
3. **기술 스택은 실제로 쓴 것만 적는다**: 이력서와 일치하지 않으면 신뢰가 떨어집니다.
4. **주기적으로 갱신한다**: 6개월 이상 방치된 프로필은 오히려 역효과입니다.

### 포트폴리오 웹사이트 구조

별도 포트폴리오 사이트를 운영한다면 아래 구조를 기준으로 시작하는 것을 추천합니다.

```text
portfolio-site/
├── index.html          # 소개 + 대표 프로젝트 3개
├── projects/
│   ├── schedule-hub.html   # 프로젝트별 상세 페이지
│   └── log-analyzer.html
├── blog/               # 기술 블로그 링크 또는 내장 글
├── resume/             # PDF 다운로드 + 온라인 버전
└── assets/
    ├── screenshots/    # 프로젝트 스크린샷
    └── diagrams/       # 아키텍처 다이어그램
```

| 페이지 | 필수 요소 | 흔한 실수 |
| --- | --- | --- |
| 메인 | 한 줄 소개, 대표 프로젝트 카드 | 프로젝트 10개 이상 나열 |
| 프로젝트 상세 | 문제-해결-결과, 데모 링크, 스크린샷 | 기술 스택만 나열 |
| 이력서 | PDF 다운로드, 온라인 버전 병행 | 링크만 있고 미리보기 없음 |

포트폴리오 사이트의 핵심은 화려함이 아니라 탐색 효율입니다. 방문자가 30초 안에 "이 사람이 뭘 잘하는지"를 파악할 수 있으면 성공입니다.

### 유지보수 자동화 설정

포트폴리오는 공개 이후에도 관리가 필요합니다. 수동 점검을 최소화하려면 자동화를 미리 설정해 두는 것이 좋습니다.

```yaml
# .github/workflows/portfolio-health.yml
name: Portfolio Health Check
on:
  schedule:
    - cron: '0 9 * * 1'  # 매주 월요일 09:00 UTC

jobs:
  check:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Check demo URL
        run: |
          curl -sf https://demo.example.com/health || echo '::warning::Demo is down'
      - name: Run tests
        run: |
          pip install -r requirements.txt
          pytest --tb=short
      - name: Check links in README
        run: |
          pip install linkchecker
          linkchecker README.md --check-extern
```

이 워크플로우가 실패하면 GitHub 알림으로 바로 알 수 있습니다. 주간 헬스체크만 설정해도 데모 다운, 테스트 실패, 링크 깨짐을 자동으로 감지할 수 있습니다.

추가로 의존성 업데이트도 자동화할 수 있습니다:

```yaml
# .github/dependabot.yml
version: 2
updates:
  - package-ecosystem: pip
    directory: /
    schedule:
      interval: monthly
    open-pull-requests-limit: 3
```

Dependabot이 매월 PR을 열면, 테스트가 통과하는지 확인하고 머지하면 됩니다. 이 과정 자체가 "프로젝트를 유지보수하고 있다"는 증거가 되므로, 면접에서도 활용할 수 있습니다.

### 공개 채널별 메시지 정렬표

같은 프로젝트라도 공개 채널마다 강조점이 다릅니다. 아래 표를 기준으로 채널별 메시지를 정렬하면 일관성을 유지하면서도 각 플랫폼에 맞는 소개를 할 수 있습니다.

| 채널 | 강조점 | 분량 | 링크 방향 |
| --- | --- | --- | --- |
| GitHub README | 문제, 실행 방법, 데모 | 스크롤 1~2회 | 블로그, 데모 |
| 블로그 | 과정, 판단, 학습 | 2,000~4,000자 | GitHub, 데모 |
| LinkedIn | 성과, 역할, 수치 | 3~5줄 | GitHub, 블로그 |
| 이력서 | 문제-결과-기술 | 2~3줄 | GitHub |
| 포트폴리오 사이트 | 시각적 요약, 스크린샷 | 한 페이지 | 전체 연결 |

채널 간 메시지가 어긋나면 방문자는 혼란을 느낍니다. 예를 들어 이력서에는 "성능 50% 개선"이라고 적었는데 GitHub README에 관련 수치가 없으면 신뢰가 떨어집니다. 한 곳에서 수치를 바꾸면 다른 채널도 함께 갱신하는 습관이 필요합니다.

## 처음 질문으로 돌아가기

- **공개 전 마지막으로 확인해야 할 핵심 영역은 무엇일까요?**
  - README와 데모부터 시작하세요. 코드 리팩토링도 중요하지만, 방문자가 프로젝트를 이해하지 못하면 코드를 읽을 동기가 없습니다.
- **README, 데모, 코드, 서사, 공개 채널은 왜 따로 점검해야 할까요?**
  - 각 영역은 서로 다른 품질 기준을 갖고 있기 때문입니다. 데모는 작동 여부, 코드는 테스트 상태, 서사는 일관성이 핵심입니다. 한꺼번에 보면 놓치기 쉬운 부분이 영역별로 분리하면 드러납니다.
- **작성자 시선과 처음 방문한 사람의 시선은 어디에서 가장 크게 어긋날까요?**
  - 작성자는 맥락을 알고 있어 빈 부분을 머릿속으로 채우지만, 방문자는 보이는 것만 봅니다. README의 오래된 실행 명령, 깨진 데모 링크, 설명 없는 스크린샷이 대표적입니다. 동료 한 명에게 3분만 보여 달라고 하면 바로 드러납니다.
