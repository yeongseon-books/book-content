# Portfolio Project 101 (3/10): README 작성

좋은 프로젝트도 README가 흐리면 입구에서 바로 힘을 잃습니다. 저장소에 처음 들어온 사람은 코드를 읽기 전에 README부터 봅니다. 그 몇십 초 안에 이 프로젝트가 어떤 문제를 다루는지, 실제로 열어 볼 데모가 있는지, 직접 실행할 수 있는지가 보이지 않으면 방문자는 금방 다른 저장소로 이동합니다.

이 글은 Portfolio Project 101 시리즈의 3번째 글입니다. 여기서는 포트폴리오 README를 단순한 안내문이 아니라 첫 번째 데모로 보고, 리뷰어가 60초 안에 프로젝트를 이해할 수 있게 만드는 구성과 문장 방식을 봅니다.

## 먼저 던지는 질문

- 포트폴리오 README에서 가장 먼저 보여 줘야 할 정보는 무엇일까요?
- 한 줄 소개, 데모, 스택, 실행 방법, 다음 작업은 왜 이 순서가 읽기 쉬울까요?
- 스크린샷이 많아도 README가 약하게 느껴지는 이유는 무엇일까요?

## 왜 중요한가

README는 프로젝트의 입구입니다. 같은 프로젝트라도 README가 단단하면 정리된 결과물처럼 보이고, README가 비어 있으면 미완성 연습 저장소처럼 보입니다. 리뷰어는 README를 통해 문제 감각, 커뮤니케이션 수준, 온보딩 배려까지 함께 읽습니다.

포트폴리오에서는 특히 이 차이가 큽니다. 저장소를 검토하는 사람은 여러분 프로젝트에 이미 익숙한 동료가 아닙니다. 낯선 사람이 짧은 시간 안에 핵심을 파악해야 합니다. README가 이 역할을 못 하면 구현 품질을 보여 줄 기회 자체가 줄어듭니다.

## 머릿속에 먼저 그릴 그림

README를 설계할 때는 독자의 질문 순서를 따라가면 좋습니다.

처음 보는 사람은 대개 이렇게 생각합니다. "이게 뭔가요?" 다음에는 "정말 돌아가나요?"가 오고, 그다음에야 "무엇으로 만들었나요?"와 "내가 직접 실행할 수 있나요?"가 이어집니다. README는 작성자 머릿속 순서가 아니라 방문자 머릿속 순서를 따라야 읽기 쉽습니다.

## 핵심 용어

- **한 줄 소개**: 프로젝트의 문제와 성격을 한 문장으로 압축한 설명입니다.
- 데모: 실제 동작을 보여 주는 링크나 스크린샷입니다.
- **기술 스택**: 구현에 사용한 핵심 기술 목록입니다.
- **실행 방법**: 저장소를 내려받은 사람이 바로 따라 할 수 있는 명령입니다.
- **다음 작업**: 아직 남아 있는 개선 계획이나 미완성 항목입니다.

## 바꾸기 전과 후

**Before**: 제목과 설치 명령만 있고, 왜 이 프로젝트가 존재하는지 알기 어렵습니다.

**After**: 한 줄 소개, 데모, 스택, 실행 방법, 다음 작업이 모두 있어 처음 보는 사람도 1분 안에 핵심을 파악할 수 있습니다.

전자의 README는 작성자 중심입니다. 후자의 README는 방문자 중심입니다. 포트폴리오에서는 후자 쪽이 훨씬 강합니다. 프로젝트가 무엇인지 설명하는 일을 독자에게 떠넘기지 않기 때문입니다.

## 단계별로 살펴보기

### 1단계 — 한 줄 소개

기술 이름보다 해결하려는 문제를 앞에 둡니다.

```markdown
> A mini SaaS that fixes lost team schedules
```

한 줄 소개는 제목을 다시 설명하는 문장이 아니라, 프로젝트의 가치를 짧게 압축한 문장이어야 합니다. 이 한 줄이 선명하면 이후 섹션도 자연스럽게 읽힙니다.

### 2단계 — 데모 링크

직접 확인할 수 있는 경로를 빠르게 보여 줍니다.

```markdown
[Live Demo](https://demo.example.com)
```

데모 링크는 신뢰를 가장 빨리 올리는 요소입니다. 글로 길게 설명하는 것보다, 실제로 열리는 링크 하나가 더 많은 판단을 대신합니다.

### 3단계 — 스택

기술 스택은 핵심만 간단히 적습니다.

```markdown
- FastAPI, PostgreSQL, Docker
```

여기서 목표는 기술 자랑이 아닙니다. 검토자가 프로젝트의 구성 요소를 빠르게 파악하게 만드는 일입니다. 선택 이유는 별도 문서나 본문에서 풀어도 충분합니다.

### 4단계 — 실행

실행 명령은 복사해서 바로 붙여 넣을 수 있어야 합니다.

```bash
docker compose up
```

복잡한 전제조건이 많으면 README의 신뢰가 떨어집니다. 실행 방법은 설치 문서가 아니라 첫 온보딩 경로라는 감각으로 정리하는 편이 좋습니다.

### 5단계 — 다음 작업

남은 작업을 숨기지 말고 체크박스로 남겨 둡니다.

```markdown
- [ ] add notifications
```

이 섹션은 미완성을 드러내는 약점이 아니라 범위 감각을 보여 주는 장치입니다. 무엇을 끝냈고 무엇이 아직 남았는지 솔직하게 적을수록 프로젝트는 오히려 더 믿을 만하게 보입니다.

## 이 코드에서 먼저 볼 점

- 한 줄 소개는 문제를 말해야 하고, 제목 반복으로 끝나면 약합니다.
- 데모는 글보다 먼저 신뢰를 만듭니다.
- 실행 명령은 짧고 재현 가능할수록 좋습니다.

## 자주 하는 실수

1. 머리말이 길어서 정작 프로젝트 핵심이 아래로 밀리는 경우
2. 스크린샷은 많지만 실제 데모 링크나 실행 방법이 없는 경우
3. 실행 명령이 복잡해서 처음 보는 사람이 바로 따라 하지 못하는 경우
4. 왜 이런 구조를 택했는지 전혀 드러나지 않는 경우
5. 남은 작업이 없어 현재 상태를 오히려 판단하기 어려운 경우

README는 많이 쓰는 문서가 아니라, 필요한 정보를 먼저 배치하는 문서에 가깝습니다. 문장이 화려할 필요는 없지만, 독자가 다음 행동을 바로 정할 수 있어야 합니다.

## 실무에서는 이렇게 본다

잘 관리되는 오픈소스 프로젝트를 보면 README 구조가 크게 다르지 않습니다. 짧은 소개, 빠른 시작, 데모나 예시, 핵심 문서 링크, 다음 단계가 반복됩니다. 이유는 처음 들어온 사람이 찾는 정보가 대부분 비슷하기 때문입니다.

개인 포트폴리오도 마찬가지입니다. 프로젝트 규모가 작아도 README가 안정적이면 검토자는 구현 품질뿐 아니라 커뮤니케이션 습관과 배려 수준까지 함께 읽을 수 있습니다.

## 체크리스트

- [ ] 문제를 설명하는 한 줄 소개가 있다.
- [ ] 데모 링크 또는 스크린샷이 바로 보인다.
- [ ] 실행 명령을 복사해서 바로 실행할 수 있다.
- [ ] 핵심 스택이 과하지 않게 정리되어 있다.
- [ ] 다음 작업이 체크박스로 남아 있다.

## 연습 문제

1. 여러분 프로젝트의 한 줄 소개를 기술 이름 없이 다시 써 보세요.
2. README를 처음 보는 사람이 30초 안에 찾아야 할 정보 세 가지를 적어 보세요.
3. 실행 섹션에서 지금 가장 불친절한 부분을 한 줄로 적어 보세요.

## 정리와 다음 글

README는 프로젝트의 요약본이 아니라 첫 번째 사용자 경험입니다. 문제를 먼저 말하고, 데모로 증명하고, 실행 경로를 단순하게 만들고, 남은 작업을 솔직하게 남겨 두면 작은 프로젝트도 훨씬 선명하게 읽힙니다.

다음 글에서는 이 프로젝트를 실제로 보여 주는 데모를 어떻게 구성해야 처음 보는 사람도 가치와 흐름을 금방 이해할 수 있는지 다룹니다.

### README 구조 템플릿(복붙 가능한 기본형)

README는 방문자가 프로젝트를 평가하는 첫 화면입니다. 따라서 문서 길이보다 정보 배치가 중요합니다. 아래 템플릿은 포트폴리오 프로젝트에 필요한 최소 구조를 담고 있으며, 실제 저장소에 바로 적용할 수 있습니다.

```markdown
# Project Name

한 줄 소개: 어떤 사용자의 어떤 문제를 해결하는 프로젝트인지 한 문장으로 설명합니다.

## Demo
- Live: https://...
- Video: https://...
- Test Account: guest@example.com / demo1234

## Problem
- 사용자 상황
- 기존 방식의 불편
- 해결 목표

## Solution
- 핵심 기능 3개
- 기술 선택 이유 요약

## Tech Stack
- Backend: FastAPI
- DB: PostgreSQL
- Infra: Docker, Render

## Run Locally
```bash
cp .env.example .env
docker compose up --build
```

## Test
```bash
pytest -q
```

## Roadmap
- [x] MVP
- [ ] Notification
- [ ] Audit log
```

이 템플릿의 포인트는 "독자의 질문 순서"를 따르는 것입니다. 이게 무엇인지, 동작하는지, 왜 필요한지, 어떻게 실행하는지, 지금 어디까지 왔는지를 순서대로 답하면 README가 훨씬 강해집니다.

### 좋은 README와 약한 README 비교

아래 비교는 같은 프로젝트라도 문서 구조가 평가를 어떻게 바꾸는지 보여 줍니다.

| 항목 | 약한 README | 좋은 README |
| --- | --- | --- |
| 첫 문장 | 기술 소개 중심 | 문제와 사용자 중심 |
| 데모 정보 | 없음 또는 하단 배치 | 상단에 즉시 노출 |
| 실행 방법 | 누락/긴 문단 | 복붙 가능한 명령 블록 |
| 현재 상태 | 완성처럼 보이게 포장 | 완료/미완료를 구분해 명시 |
| 검증 경로 | 테스트 정보 없음 | 테스트 명령과 상태 포함 |

좋은 README는 화려한 문장을 쓰지 않아도 됩니다. 대신 방문자가 "다음으로 무엇을 해야 하는지"를 즉시 알 수 있어야 합니다. 포트폴리오는 코드 품질뿐 아니라 커뮤니케이션 품질도 함께 평가되기 때문입니다.

### README 품질 점검 체크포인트

README를 작성한 뒤에는 아래 항목으로 스스로 검수하면 좋습니다.

1. 제목 아래 5줄 안에 문제와 데모 링크가 있는가
2. 로컬 실행이 3단계 안에 끝나는가
3. 환경 변수 설명이 `.env.example`과 일치하는가
4. 테스트/배포 상태를 확인할 링크가 있는가
5. TODO가 현실적인 순서로 관리되는가

이 다섯 가지가 충족되면 README는 단순 안내문을 넘어 "검토 가능한 프로젝트 입구"가 됩니다. 특히 면접 전에 README만 다시 다듬어도 프로젝트 인상이 크게 좋아지는 경우가 많습니다.

### README 리팩터링 절차(1시간 버전)

이미 작성된 README를 빠르게 개선하려면 아래 순서로 진행하면 효율적입니다.

1. 상단 20줄만 읽고 문제/데모/실행 경로가 보이는지 확인합니다.
2. 보이지 않으면 제목 아래에 한 줄 문제 정의와 데모 링크를 추가합니다.
3. 실행 방법을 문단에서 명령 블록으로 바꿉니다.
4. 현재 상태를 체크박스로 명시합니다.
5. 마지막으로 링크와 명령을 실제 실행해 검증합니다.

| 점검 항목 | 수정 전 | 수정 후 |
| --- | --- | --- |
| 첫 진입 가독성 | 낮음 | 높음 |
| 실행 재현성 | 불명확 | 명확 |
| 프로젝트 상태 투명성 | 낮음 | 높음 |

이 절차는 짧지만 효과가 큽니다. README가 정리되면 코드 검토 진입률이 올라가고, 프로젝트 신뢰도도 함께 개선됩니다.

### README 템플릿 변형: 프로젝트 유형별 구성

모든 프로젝트에 같은 README 구조를 쓰면 오히려 어색할 수 있습니다. 프로젝트 유형에 따라 강조할 섹션이 달라집니다.

**API 서버 프로젝트**

```markdown
# task-tracker-api

> 팀 일정이 여러 도구에 흩어져 조회 비용이 높은 문제를 해결합니다.

## 데모
- Live: https://demo.example.com
- API Docs: https://demo.example.com/docs (Swagger UI)

## 주요 엔드포인트

| Method | Path | 설명 |
|--------|------|------|
| GET | /schedules | 주간 일정 조회 |
| POST | /schedules | 일정 생성 |
| GET | /schedules/shared/{token} | 공유 링크 조회 |

## 실행
```bash
cp .env.example .env
docker compose up -d
open http://localhost:8000/docs
```

## 테스트
```bash
pytest --cov=src tests/
```

## 기술 선택 이유
- FastAPI: 자동 OpenAPI 문서 생성, async 지원
- PostgreSQL: 복합 조회 성능, JSON 필드 지원

## 다음 단계
- [ ] 캘린더 동기화 (Google Calendar API)
- [ ] 이메일 알림
```

**데이터 분석 프로젝트**

```markdown
# sales-funnel-analysis

> B2B SaaS 퍼널 전환율 하락 구간을 식별해 마케팅 예산 배분을 독습니다.

## 결과 요약
- 퍼널 3단계 전환율: 12% → 18% (개선 후)
- 예산 재배분으로 CAC 23% 감소

## 데이터
- 샘플 데이터: `data/sample_funnel.csv` (10,000건, 익명화)
- 원본: 비공개 (NDA)

## 실행
```bash
pip install -r requirements.txt
jupyter notebook notebooks/01-eda.ipynb
```

## 분석 흐름
1. EDA → 2. 퍼널 단계별 전환율 계산 → 3. 병목 식별 → 4. 개선 제안 → 5. A/B 테스트 설계
```

두 템플릿의 차이를 보면, API 프로젝트는 엔드포인트와 실행 명령이 핵심이고, 데이터 분석 프로젝트는 결과 요약과 분석 흐름이 핵심입니다. 프로젝트 유형에 맞는 섹션을 강조해야 방문자가 핵심 정보를 빠르게 찾을 수 있습니다.

### 포트폴리오 웹사이트 구조 설계

포트폴리오 웹사이트는 여러 프로젝트를 하나의 입구에서 보여 주는 상위 문서입니다. GitHub 프로필 README와 별도로, 정적 사이트나 간단한 웹 페이지로 만들면 프로젝트 간 맥락을 연결할 수 있습니다.

```text
portfolio-site/
├── index.html              # 방문자 진입점: 소개 + 프로젝트 목록
├── projects/
│   ├── task-tracker.html   # 프로젝트 상세: 문제, 데모, 스택, 결과
│   └── sales-analysis.html
├── blog/                   # 기술 글 링크 (외부 블로그로 연결 가능)
├── about.html              # 자기소개, 경력, 관심 분야
└── assets/
    ├── screenshots/        # 프로젝트 스크린샷
    └── resume.pdf          # PDF 이력서 다운로드
```

포트폴리오 웹사이트에서 가장 중요한 페이지는 `index.html`입니다. 여기에 한 줄 소개, 대표 프로젝트 2~3개 카드, 연락 수단이 모두 보여야 합니다. 프로젝트 상세 페이지는 README의 웹 버전이라고 생각하면 됩니다. 문제, 데모, 스택, 결과를 같은 구조로 보여 주되, 스크린샷과 GIF를 더 자유롭게 배치할 수 있습니다.

웹사이트를 만들 때 자주 보이는 실수는 디자인에 너무 많은 시간을 쓰는 것입니다. 포트폴리오 웹사이트의 목적은 디자인 능력 증명이 아니라 프로젝트 접근성 향상입니다. GitHub Pages나 Vercel로 30분 안에 배포할 수 있는 미니멀 사이트면 충분합니다.

### README 작성 시 흔한 실수와 해결책

README를 작성할 때 자주 발생하는 실수를 정리하면 다음과 같습니다.

| 실수 | 증상 | 해결책 |
| --- | --- | --- |
| 기술 나열로 시작 | "이 프로젝트는 React + Node.js + ..."로 시작 | 문제 문장으로 시작하고 스택은 하단에 배치 |
| 실행 방법 누락 | "실행은 당연히 되겠지" 심리 | 3단계 명령 블록으로 명시 |
| 데모 링크 없음 | 스크린샷만 있음 | 라이브 URL 또는 GIF를 상단에 배치 |
| 상태 숨김 | 미완성 부분을 언급하지 않음 | TODO 섹션으로 투명하게 공개 |
| 너무 긴 README | 스크롤이 끝없이 내려감 | 핵심만 남기고 상세는 docs/로 분리 |

이 표를 두고 작성 후 자가 점검하면 대부분의 흔한 문제를 미리 잡을 수 있습니다.

### README 셀프 리뷰 체크리스트

README를 작성한 뒤에는 반드시 셀프 리뷰를 거쳐야 합니다. 아래 체크리스트를 순서대로 확인하면 빠뜨리기 쉬운 항목을 잡을 수 있습니다.

```text
[셀프 리뷰 체크리스트]

1단계: 첫인상 (5초 테스트)
  □ 프로젝트 이름만 보고 무엇을 하는지 짐작할 수 있는가?
  □ 한 줄 설명이 기술이 아닌 문제/가치를 말하는가?
  □ 데모 GIF 또는 라이브 URL이 스크롤 없이 보이는가?

2단계: 실행 가능성 (3분 테스트)
  □ 새 터미널에서 README 명령만으로 실행할 수 있는가?
  □ 환경 변수나 API 키 설정 방법이 명시되어 있는가?
  □ Python/Node 버전 요구사항이 있는가?
  □ docker-compose up 또는 이에 준하는 원커맨드 실행이 가능한가?

3단계: 맥락 전달
  □ 왜 이 프로젝트를 만들었는지 한 문단으로 설명하는가?
  □ 기술 스택을 선택한 이유가 한 줄이라도 있는가?
  □ 현재 상태(완성/진행 중/실험)가 명시되어 있는가?

4단계: 구조 품질
  □ 목차가 있고, 클릭하면 해당 섹션으로 이동하는가?
  □ 코드 블록에 언어 태그가 붙어 있는가?
  □ 이미지 alt 텍스트가 의미 있는가?
  □ 링크가 모두 유효한가? (404 없음)
```

이 체크리스트를 PR 템플릿에 넣어 두면, 리뷰어가 README 품질을 별도로 지적할 필요가 줄어듭니다. 특히 '5초 테스트'는 처음 방문하는 채용 담당자의 시선을 시뮬레이션하는 것이므로, 동료에게 5초만 보여 주고 "이 프로젝트가 뭐 하는 거야?"라고 물어보는 것이 가장 효과적인 검증입니다.

### README 버전 관리 전략

README는 코드와 함께 변해야 합니다. 코드는 업데이트했는데 README가 과거 버전을 설명하고 있으면 오히려 혼란을 줍니다. README를 코드와 동기화하는 실용적인 방법을 정리합니다.

**방법 1: PR 체크리스트에 README 항목 추가**

```markdown
<!-- .github/pull_request_template.md -->
## PR Checklist
- [ ] 코드 변경에 따라 README를 업데이트했습니다
- [ ] 새 환경 변수가 있으면 README에 추가했습니다
- [ ] API 엔드포인트가 변경되면 README 예시를 수정했습니다
```

**방법 2: CI에서 README 유효성 검증**

```yaml
# .github/workflows/readme-check.yml
name: README Validation
on: [pull_request]
jobs:
  check-readme:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - name: Check README links
        run: |
          pip install linkchecker
          linkchecker README.md --check-extern
      - name: Verify install commands work
        run: |
          # README의 설치 명령 블록을 추출해 실행
          grep -A 3 '```bash' README.md | grep -v '```' | bash
```

**방법 3: 변경 로그 섹션 유지**

README 하단에 최근 변경 3건을 유지하면, 방문자가 프로젝트가 활발한지 바로 판단할 수 있습니다.

```markdown
## Recent Changes
- 2024-03-15: OAuth2 인증 추가, 환경 변수 2개 신규
- 2024-03-01: Python 3.12 지원, 최소 버전 3.10으로 상향
- 2024-02-20: Docker Compose 설정 추가, 원커맨드 실행 지원
```

README 버전 관리의 핵심 원칙은 "README가 거짓말하지 않게 하라"입니다. 오래된 설치 명령, 삭제된 엔드포인트, 변경된 환경 변수가 README에 남아 있으면 신뢰를 잃습니다.

### 프로젝트 유형별 README 섹션 우선순위

모든 프로젝트에 동일한 README 구조를 적용하면 핵심 정보가 묻힐 수 있습니다. 프로젝트 유형에 따라 어떤 섹션을 상단에 배치할지 우선순위가 달라집니다.

| 섹션 | 웹 앱 | CLI 도구 | 라이브러리 | 데이터 분석 | 인프라/DevOps |
| --- | --- | --- | --- | --- | --- |
| 데모/스크린샷 | 1순위 | 3순위 | 4순위 | 2순위 | 5순위 |
| 설치/실행 방법 | 2순위 | 1순위 | 1순위 | 3순위 | 2순위 |
| API/사용 예시 | 3순위 | 2순위 | 2순위 | 5순위 | 4순위 |
| 결과/성과 지표 | 4순위 | 5순위 | 5순위 | 1순위 | 3순위 |
| 아키텍처 다이어그램 | 5순위 | 4순위 | 3순위 | 4순위 | 1순위 |

이 표에서 핵심은 "방문자가 이 유형의 프로젝트에서 가장 먼저 확인하고 싶은 것"을 최상단에 놓는 것입니다. 웹 앱이라면 동작하는 화면을 보고 싶고, CLI 도구라면 설치 후 바로 실행해 보고 싶고, 데이터 분석이라면 결과부터 보고 싶습니다.

실제로 적용하는 방법은 간단합니다. README를 작성할 때 위 표에서 자신의 프로젝트 유형 열을 확인하고, 1순위 섹션을 H1 바로 아래에, 2순위를 그 다음에 배치합니다. 4~5순위 섹션은 README 하단이나 별도 문서(`docs/architecture.md` 등)로 옮겨도 됩니다.

### GitHub 프로필 README와 프로젝트 README 연결

GitHub 프로필 README(`username/username` 저장소)는 포트폴리오의 진입점입니다. 여기서 대표 프로젝트 README로의 동선을 설계해야 합니다.

```markdown
<!-- GitHub Profile README 예시 -->
## Featured Projects

| Project | Description | Stack | Status |
|---------|-------------|-------|--------|
| [task-tracker](link) | 팀 업무 진행률 실시간 대시보드 | FastAPI, React, PostgreSQL | ✅ Production |
| [sales-funnel](link) | B2B SaaS 퍼널 전환율 분석 | Python, Pandas, Plotly | ✅ Complete |
| [deploy-bot](link) | Slack 챗봇으로 배포 자동화 | Python, GitHub Actions | 🔧 In Progress |
```

프로필 README에서 프로젝트 README로 넘어갈 때 방문자가 겪는 인지 부하를 줄이려면, 프로필의 한 줄 설명과 프로젝트 README의 한 줄 설명이 동일한 문장을 사용해야 합니다. 표현이 다르면 "같은 프로젝트 맞나?"라는 의문이 생깁니다.

또한 프로필 README에 나열하는 프로젝트는 3개를 넘기지 않는 것이 좋습니다. 5개 이상 나열하면 어느 것도 대표작으로 인식되지 않습니다. 가장 완성도 높은 프로젝트 2~3개를 pinned repository로 설정하고, 프로필 README의 표와 일치시키면 일관된 인상을 줄 수 있습니다.

## 처음 질문으로 돌아가기

- **포트폴리오 README에서 가장 먼저 보여 줘야 할 정보는 무엇일까요?**
  - 한 문장 문제 정의입니다. '무엇을 해결하는가'를 명확히 하면, 방문자가 이 저장소가 자신의 관심사인지 판단할 수 있습니다.
- **한 줄 소개, 데모, 스택, 실행 방법, 다음 작업은 왜 이 순서가 읽기 쉬울까요?**
  - 한 줄 설명, 데모 또는 GIF, 단계별 실행 방법이 가장 중요합니다. README는 진입점이지, 모든 내용을 담는 공간이 아닙니다.
- **스크린샷이 많아도 README가 약하게 느껴지는 이유는 무엇일까요?**
  - 데모는 성공 케이스만 보여주기 쉽습니다. 오류 처리, 빈 상태, 대기 상태 같은 엣지 케이스도 자연스럽게 보여주면, 생각이 깊은 개발자로 보입니다.
