# Portfolio Project 101 (2/10): 좋은 프로젝트의 조건

포트폴리오 프로젝트를 만들다 보면 기능을 많이 넣을수록 더 좋아 보일 것 같다는 생각이 듭니다. 하지만 실제로는 반대인 경우가 많습니다. 범위가 커질수록 마무리 품질은 떨어지고, 문제 정의는 흐려지고, 결과는 설명하기 어려워집니다. 포트폴리오에서 강하게 남는 프로젝트는 큰 프로젝트가 아니라 끝까지 다듬은 프로젝트입니다.

이 글은 Portfolio Project 101 시리즈의 2번째 글입니다. 여기서는 채용 관점에서 좋은 프로젝트를 가르는 조건이 무엇인지, 왜 완성도와 문제 선명도가 기능 개수보다 더 중요하게 읽히는지 살펴보겠습니다.

## 먼저 던지는 질문

- 왜 큰 프로젝트가 항상 더 좋은 포트폴리오가 되지는 않을까요?
- 문제 정의, 결과 지표, 유지보수성, 재현성은 각각 어떻게 드러날까요?
- 기능이 적어도 강한 프로젝트로 읽히게 만드는 기준은 무엇일까요?

## 큰 그림

![Portfolio Project 101 2장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/portfolio-project-101/02/02-01-diagram.ko.png)

*Portfolio Project 101 2장 흐름 개요*

이 그림에서는 좋은 프로젝트를 만드는 조건들이 어떤 계층에 속하는지 봅니다. 기능의 화려함 대신 문제 정의의 명확함, 완성도의 성실함, 설명의 정돈 정도가 훨씬 더 중요합니다.

> 좋은 포트폴리오 프로젝트는 스택의 최신성으로 만들어지지 않습니다. 문제의 명확함과 설명의 정돈 정도로 만들어집니다.

## 왜 중요한가

좋은 프로젝트의 조건을 알면 어디에 시간을 써야 하는지가 분명해집니다. 초보 개발자는 종종 기능 추가에 대부분의 시간을 쓰고, README나 테스트, 배포 안정성은 뒤로 미룹니다. 하지만 리뷰어는 기능 개수보다 문제의 선명함, 마무리 수준, 반복 가능성을 더 빨리 확인합니다.

포트폴리오는 제품 홍보물이 아니라 역량 증명 자료입니다. 그래서 "무엇을 얼마나 많이 만들었는가"보다 "무엇을 왜 만들었고 어디까지 완성했는가"가 더 중요합니다. 범위를 줄이더라도 끝까지 밀어 붙인 흔적이 있으면 프로젝트는 훨씬 믿을 만하게 보입니다.

## 머릿속에 먼저 그릴 그림

좋은 프로젝트는 초점, 문제 선명도, 결과, 유지보수성, 재현성이 한 흐름으로 이어집니다.

이 흐름은 단순한 평가표가 아닙니다. 범위가 작아야 문제를 또렷하게 설명할 수 있고, 문제가 또렷해야 결과를 수치로 보여 주기 쉽고, 그 결과를 믿게 하려면 유지보수성과 재현성이 뒤따라야 합니다. 좋은 프로젝트는 이 연결이 매끄럽습니다.

## 핵심 용어

- **집중된 범위**: 기능을 많이 넣기보다 핵심 흐름 몇 개에 초점을 맞춘 상태입니다.
- **분명한 문제**: 누구의 어떤 불편을 줄이려는지 한 문장으로 설명할 수 있는 상태입니다.
- **측정 가능한 결과**: 숫자나 지표로 확인할 수 있는 변화입니다.
- **유지보수성**: 테스트와 문서가 있어 나중에 다시 다루기 쉬운 상태입니다.
- 재현성: 다른 사람이 같은 환경을 다시 띄울 수 있는 상태입니다.

## 바꾸기 전과 후

**Before**: 기능은 많지만 끝까지 다듬지 못했고, 무엇이 핵심인지 설명하기 어렵습니다.

**After**: 기능은 적어도 문제와 결과가 분명하고, 실행 경로와 검증 방법이 정리되어 있습니다.

후자의 프로젝트가 더 강한 이유는 평가 포인트가 명확하기 때문입니다. 리뷰어는 이 프로젝트가 무엇을 겨냥했고, 어디까지 끝냈고, 얼마나 믿을 수 있는지를 빠르게 읽을 수 있습니다.

## 단계별로 살펴보기

### 1단계 — 범위 점수

먼저 프로젝트 범위가 과하지 않은지 봅니다.

```python
focus = 5
```

이 숫자는 절대 기준이 아니라 범위를 스스로 평가하는 출발점입니다. 핵심 기능이 몇 개인지, 그 기능들이 하나의 문제를 향하고 있는지 점검하면 프로젝트 초점이 훨씬 또렷해집니다.

### 2단계 — 문제 명확도

문제는 추상적일수록 약해집니다.

```python
problem_score = 4
```

"일정 관리 앱"보다 "팀 일정이 여러 도구에 흩어져 한 번에 보기 어려운 문제"가 훨씬 강합니다. 문제를 더 구체적으로 말할수록 구현 선택도 자연스럽게 설명됩니다.

### 3단계 — 결과 측정

좋은 프로젝트는 결과를 감각이 아니라 수치로 말합니다.

```python
result = {"latency_ms": 120, "users": 30}
```

숫자가 있으면 결과를 비교하고 질문하기가 쉬워집니다. 사용자 수, 응답 시간, 작업 시간 단축처럼 작은 지표라도 남겨 두면 프로젝트가 훨씬 현실적으로 읽힙니다.

### 4단계 — 유지보수성

프로젝트는 돌아가는 것만으로 끝나지 않습니다.

```python
maintainable = {"tests": True, "docs": True}
```

테스트와 문서는 "지금은 되지만 나중에는 모른다"는 상태를 줄여 줍니다. 포트폴리오에서 이 흔적은 구현 역량뿐 아니라 협업 감각까지 보여 줍니다.

### 5단계 — 재현성

다른 사람이 다시 띄울 수 있어야 프로젝트가 오래 살아남습니다.

```python
reproducible = {"docker": True, "seed": True}
```

Docker, 시드 데이터, 실행 명령은 모두 재현성을 높이는 장치입니다. 내 노트북에서만 되는 프로젝트는 설명이 길어질수록 약해집니다.

## 이 코드에서 먼저 볼 점

- 작은 범위가 오히려 완성도와 설명력을 높입니다.
- 결과는 숫자로 남겨야 비교와 질문이 쉬워집니다.
- 유지보수성과 재현성은 마무리 능력을 보여 주는 흔적입니다.

## 자주 하는 실수

1. 기능을 계속 추가하면서 핵심 문제가 무엇인지 잃어버리는 경우
2. 해결 결과를 숫자로 남기지 않아 성과가 흐릿한 경우
3. 테스트와 문서를 나중으로 미뤄 신뢰를 떨어뜨리는 경우
4. Docker나 시드 데이터가 없어 다른 사람이 재현하지 못하는 경우
5. 범위를 줄이지 못해 결국 완성도가 낮아지는 경우

좋은 프로젝트를 만드는 일은 더 많이 만드는 일이 아니라, 무엇을 끝까지 가져갈지 정하는 일에 가깝습니다. 범위를 줄이는 결정이 곧 품질을 올리는 결정인 경우가 많습니다.

## 실무에서는 이렇게 본다

오픈소스 프로젝트나 초기 제품도 대부분 작고 분명한 목표에서 출발합니다. 핵심 흐름이 정리되기 전에는 기능을 넓히지 않고, 결과와 운영 가능성을 먼저 다듬습니다. 개인 포트폴리오도 같은 기준으로 읽힙니다.

작은 프로젝트라도 문제 정의가 분명하고, 결과를 보여 줄 수 있고, 다시 실행할 수 있으면 충분히 강합니다. 반대로 기능이 많아도 방향이 흐리면 오래 기억되기 어렵습니다.

## 체크리스트

- [ ] 핵심 기능 수를 의식적으로 줄였다.
- [ ] 문제를 한 문장으로 설명할 수 있다.
- [ ] 결과를 보여 줄 숫자나 지표가 있다.
- [ ] 테스트, 문서, 재현 경로를 준비했다.

## 연습 문제

1. 여러분 프로젝트에서 지금 빼도 되는 기능 하나를 골라 보세요.
2. 현재 문제 정의를 더 구체적인 한 문장으로 다시 써 보세요.
3. 결과로 남길 수 있는 숫자 두 가지를 적어 보세요.

## 정리와 다음 글

좋은 포트폴리오 프로젝트는 크지 않아도 됩니다. 대신 문제를 분명히 말할 수 있어야 하고, 결과를 확인할 수 있어야 하며, 테스트와 문서, 재현 경로까지 갖춘 상태여야 합니다. 이런 조건이 모이면 작은 프로젝트도 충분히 강한 증거가 됩니다.

다음 글에서는 이렇게 다듬은 프로젝트를 저장소 입구인 README에서 어떻게 읽히게 만들지 이어서 살펴보겠습니다.

### 프로젝트 스코프 평가 루브릭

좋은 프로젝트를 만드는 핵심은 "더 많이 구현"이 아니라 "더 정확하게 완주"입니다. 이를 위해서는 범위를 감으로 조절하지 말고, 루브릭으로 점수화해 판단하는 편이 안전합니다. 아래 루브릭은 포트폴리오 프로젝트를 시작하거나 중간 점검할 때 사용할 수 있습니다.

| 항목 | 1점 | 3점 | 5점 |
| --- | --- | --- | --- |
| 문제 명확도 | 문제 대상이 불명확함 | 대상은 있으나 상황이 넓음 | 사용자/상황/불편이 구체적임 |
| 핵심 흐름 완결성 | 기능 파편만 존재 | 기본 흐름은 있으나 예외 약함 | 시작-행동-결과가 끝까지 연결됨 |
| 검증 가능성 | 지표 없음 | 일부 로그/수치 존재 | 핵심 지표가 정의되고 재측정 가능 |
| 유지보수성 | 테스트/문서 없음 | README 중심 최소 문서 | 테스트 + 문서 + 운영 메모 존재 |
| 재현성 | 로컬 의존 큼 | 실행법은 있으나 불안정 | 환경 분리 + 일관 실행 경로 존재 |

총점이 낮다고 프로젝트를 버릴 필요는 없습니다. 중요한 것은 점수가 낮은 축을 정확히 찾아 개선 순서를 정하는 일입니다. 예를 들어 문제 명확도와 검증 가능성이 낮다면 기능 추가를 멈추고 문제 문장과 지표부터 고쳐야 합니다.

### 기능 과잉을 막는 컷라인 규칙

많은 프로젝트가 중간에 무너지는 이유는 "있으면 좋을 기능"이 "지금 필요한 기능"을 계속 밀어내기 때문입니다. 아래 컷라인을 미리 선언해 두면 과잉 개발을 줄일 수 있습니다.

1. 핵심 사용자 흐름과 무관한 기능은 1차 릴리스에서 제외합니다.
2. 결과 지표를 바꾸지 않는 기능은 TODO로 이동합니다.
3. 설명하기 어려운 기능은 구현 전에 목적을 문장으로 먼저 씁니다.
4. 테스트 없이 추가되는 기능은 병합하지 않습니다.

컷라인은 창의성을 막는 규칙이 아니라 완성도를 지키는 안전장치입니다. 포트폴리오에서는 "10개 기능 중 3개가 불안정한 상태"보다 "핵심 3개 기능이 안정적인 상태"가 훨씬 강하게 읽힙니다.

### 품질 판단을 위한 미니 리포트 템플릿

프로젝트를 좋은 상태로 유지하려면 주기적으로 미니 리포트를 작성하는 습관이 유용합니다. 아래 템플릿을 주 1회 정도 기록하면 개선 우선순위가 선명해집니다.

```markdown
## Weekly Quality Report
- 핵심 문제 문장: 팀 일정 조회 시간을 줄인다.
- 이번 주 완료: 주간 뷰 필터, 공유 링크, README 실행 섹션 개선
- 지표 변화: 평균 조회 시간 42초 -> 19초
- 품질 이슈: 통합 테스트 flaky 2건
- 다음 주 우선순위: flaky 제거, 데모 시드 데이터 보강
```

이 문서의 장점은 화려함이 아니라 일관성입니다. 프로젝트의 건강 상태를 반복해서 확인하면, 범위가 커지기 전에 문제를 잡고 설명 가능한 결과로 마무리할 수 있습니다.

### 실전 사례: 작은 범위가 만든 높은 완성도

예를 들어 일정 관리 프로젝트에서 "캘린더 동기화, 알림, 권한, 분석"을 한 번에 넣으면 일정이 크게 밀립니다. 반대로 1차 범위를 "주간 뷰 조회 + 팀원 필터 + 공유 링크"로 줄이면 다음 이점이 생깁니다. 첫째, 핵심 흐름 테스트를 끝까지 작성할 수 있습니다. 둘째, README와 데모가 실제 구현 상태와 정확히 일치합니다. 셋째, 결과 지표를 수집할 시간이 생깁니다.

| 범위 전략 | 기능 수 | 테스트 완성도 | 데모 안정성 | 설명 난이도 |
| --- | ---: | ---: | ---: | ---: |
| 넓은 범위 | 10 | 낮음 | 낮음 | 높음 |
| 집중 범위 | 3 | 높음 | 높음 | 낮음 |

결국 좋은 프로젝트의 핵심은 기능 수가 아니라 설명 가능한 완성도입니다. 작은 범위를 깊게 끝낸 프로젝트가 면접에서도 질문을 더 잘 견딥니다.

### 프로젝트 아키텍처 다이어그램 설명 방법

좋은 프로젝트는 구조를 시각적으로 보여 주는 다이어그램이 있습니다. 다이어그램은 코드를 열기 전에 전체 설계 의도를 전달하는 역할을 합니다. 포트폴리오 수준에서는 복잡한 UML보다 간결한 컴포넌트 다이어그램이 더 효과적입니다.

```text
┌─────────────┐      ┌─────────────┐      ┌─────────────┐
│  클라이언트  │─────▶│  API 서버   │─────▶│  데이터베이스 │
│  (React)     │      │  (FastAPI)   │      │  (PostgreSQL)│
└─────────────┘      └──────┬──────┘      └─────────────┘
                              │
                       ┌─────┴───────┐
                       │  외부 API     │
                       │  (Google Cal) │
                       └─────────────┘
```

이 다이어그램이 보여 주는 정보는 세 가지입니다.

1. **컴포넌트 경계**: 프론트엔드, 백엔드, 데이터베이스가 어떻게 나뉘어 있는지 즉시 보입니다.
2. **데이터 흐름**: 화살표 방향으로 요청이 어디서 어디로 가는지 알 수 있습니다.
3. **외부 의존성**: Google Calendar 같은 외부 서비스와의 연동 지점이 드러납니다.

포트폴리오 프로젝트에 이런 다이어그램을 README나 `docs/architecture.md`에 넣으면, 리뷰어는 코드를 한 줄도 읽지 않고 전체 구조를 파악할 수 있습니다. 다이어그램을 그릴 때는 Mermaid나 ASCII 아트처럼 텍스트 기반 도구를 쓰면 버전 관리에도 편합니다.

### 스코프 평가 매트릭스

프로젝트를 시작하기 전, 또는 중간에 범위를 조정할 때 아래 매트릭스를 채우면 감으로 범위를 정하는 실수를 줄일 수 있습니다.

```markdown
# 스코프 평가 매트릭스

## 프로젝트: [task-tracker]
## 평가일: 2026-05-21

| 기능 | 핵심 흐름 기여도 | 구현 난이도 | 데모 효과 | 1차 포함 |
| --- | --- | --- | --- | --- |
| 주간 뷰 조회 | 높음 | 중간 | 높음 | O |
| 팀원 필터 | 높음 | 낮음 | 중간 | O |
| 공유 링크 | 중간 | 낮음 | 높음 | O |
| 캘린더 동기화 | 낮음 | 높음 | 중간 | X (2차) |
| 알림 | 낮음 | 중간 | 낮음 | X (보류) |
| 권한 관리 | 낮음 | 높음 | 낮음 | X (보류) |

## 판단 기준
- 핵심 흐름 기여도 높음 + 데모 효과 중간 이상 → 1차 포함
- 구현 난이도 높음 + 핵심 흐름 기여도 낮음 → 보류
- 1차 기능은 최대 3~5개로 제한
```

이 매트릭스를 작성하면 두 가지 이점이 있습니다. 첫째, 감정적으로 끌리는 기능을 객관적으로 평가할 수 있습니다. "있으면 멋있을 것 같다"는 기능도 매트릭스를 채우면 대부분 핵심 흐름 기여도가 낮거나 구현 난이도가 높아서 자연스럽게 보류됩니다. 둘째, 면접에서 "왜 이 기능을 넣지 않았나요?"라는 질문에 매트릭스를 보여 주며 답할 수 있습니다. 범위를 줄인 것이 포기가 아니라 전략이었다는 점을 증명할 수 있습니다.

### 범위 조정 의사결정 기록 예시

프로젝트 중간에 범위를 줄이는 결정을 했다면, 그 판단 과정을 기록으로 남기는 것이 좋습니다.

```markdown
# 범위 조정 결정

## 날짜: 2026-05-15
## 결정: 캘린더 동기화 기능을 2차로 연기

### 배경
- 1차 마감일까지 2주 남음
- OAuth 연동 난이도로 핵심 흐름 완성도가 위험받음
- 테스트 커버리지가 40%로 낮아짐

### 판단 기준
- 핵심 흐름 완성 > 부가 기능 추가
- 테스트 커버리지 80% 이상 확보가 1차 목표

### 대안
- A: 캘린더 동기화를 stub으로 구현 → 데모 효과 낮음
- B: 2차로 연기 + 핵심 흐름 품질 강화 → 채택

### 결과
- 핵심 흐름 테스트 85% 도달
- 데모 안정성 확보
- README에 "다음 단계" 섹션에 캘린더 동기화 명시
```

이 기록이 있으면 면접에서 "왜 이 기능이 없나요?"라는 질문에 판단 근거를 즉시 보여 줄 수 있습니다. 범위 조정은 포기가 아니라 우선순위 판단입니다. 이를 기록으로 남기는 습관이 있으면 프로젝트의 성숙도가 크게 올라갑니다.

### 결과 지표 설계 가이드

좋은 프로젝트의 조건 중 "측정 가능한 결과"를 실제로 만들려면, 어떤 지표를 어떻게 측정할지 미리 설계해야 합니다. 아래는 포트폴리오 프로젝트에서 자주 사용하는 지표 유형입니다.

| 지표 유형 | 예시 | 측정 방법 | README 표기 |
| --- | --- | --- | --- |
| 성능 | API 응답 시간 | 벤치마크 스크립트 | 평균 120ms (p95: 200ms) |
| 효율 | 작업 시간 단축 | Before/After 비교 | 42초 → 19초 (55% 감소) |
| 안정성 | 테스트 커버리지 | pytest --cov | 85% line coverage |
| 사용성 | 핵심 흐름 완주율 | 데모 시나리오 성공률 | 10회 시도 중 10회 성공 |

지표를 설계할 때 핵심은 "재측정 가능성"입니다. 한 번 측정하고 끝나는 수치는 약합니다. 같은 명령어로 다시 측정할 수 있는 지표를 만들어야 개선 전후를 비교할 수 있고, 면접에서도 신뢰를 얻습니다.

실제 코드로 지표를 측정하는 예시입니다.

```python
import time
import httpx


def benchmark_api(url: str, iterations: int = 100) -> dict:
    """API 응답 시간을 측정합니다."""
    latencies = []
    for _ in range(iterations):
        start = time.perf_counter()
        response = httpx.get(url)
        elapsed = (time.perf_counter() - start) * 1000
        latencies.append(elapsed)

    latencies.sort()
    return {
        "mean_ms": sum(latencies) / len(latencies),
        "p95_ms": latencies[int(len(latencies) * 0.95)],
        "p99_ms": latencies[int(len(latencies) * 0.99)],
        "success_rate": sum(1 for l in latencies if l < 500) / len(latencies),
    }


if __name__ == "__main__":
    result = benchmark_api("https://demo.example.com/api/schedules")
    print(f"Mean: {result['mean_ms']:.1f}ms, P95: {result['p95_ms']:.1f}ms")
    print(f"Success rate: {result['success_rate']:.1%}")
```

이 스크립트를 `scripts/benchmark.py`에 넣어 두면 언제든 다시 측정할 수 있습니다. README에 측정 명령과 결과를 함께 적어 두면 리뷰어가 직접 확인할 수도 있습니다.

### 유지보수성을 높이는 최소 설정

유지보수성이 높은 프로젝트는 보통 다음 세 가지를 갖추고 있습니다.

1. **자동화된 테스트**: `pytest`나 `unittest`로 핵심 흐름을 검증하는 테스트가 있으면, 코드를 수정하더라도 기존 기능이 깨지지 않았다는 것을 증명할 수 있습니다.
2. **CI 파이프라인**: GitHub Actions로 push마다 테스트와 린트를 자동 실행하면, 프로젝트가 항상 검증된 상태를 유지합니다.
3. **환경 분리**: `.env.example`로 환경 변수를 문서화하고, `docker-compose.yml`로 로컬 환경을 일관되게 띄울 수 있으면, 새로운 맥에서도 5분 안에 시작할 수 있습니다.

이 세 가지가 갖춰져 있으면, 프로젝트는 시간이 지나도 낙지 않습니다. 6개월 후에 다시 돌아와도 테스트가 녹색이면 안심하고 수정할 수 있고, 데모가 살아 있으면 면접 준비를 바로 시작할 수 있습니다. 유지보수성은 미래의 나를 위한 투자입니다.

### 재현성 체크리스트

프로젝트가 다른 사람의 환경에서도 정상 동작하는지 확인하는 체크리스트입니다.

- [ ] `git clone` 후 README의 실행 명령만으로 동작하는가
- [ ] `.env.example`을 복사해 `.env`를 만들면 추가 설정 없이 시작되는가
- [ ] Docker 환경에서 `docker compose up`으로 전체 스택이 뜨는가
- [ ] 시드 데이터가 있어서 데모를 바로 확인할 수 있는가
- [ ] Python 버전, OS 제약이 README에 명시되어 있는가
- [ ] 외부 API 키가 없어도 기본 흐름은 동작하는가 (mock/stub)

이 체크리스트를 통과하는 프로젝트는 채용 담당자가 직접 띄어 볼 수 있다는 뜻입니다. "내 노트북에서는 되는데"라는 변명이 필요 없는 상태가 가장 이상적입니다.

재현성이 높은 프로젝트는 시간이 지나도 데모가 깨지지 않습니다. Docker 이미지를 고정 버전으로 관리하고, 시드 데이터를 저장소에 포함하며, 외부 API에 의존하는 부분은 mock으로 대체할 수 있게 만들어 두면 6개월 후에도 동일한 결과를 보여 줄 수 있습니다. 면접 시점에 데모가 깨져 있으면 아무리 코드가 좋아도 인상이 크게 나빠집니다. 재현성은 미래의 나를 위한 보험이기도 합니다.

### 좋은 프로젝트를 시작할 때 순서 정리

지금까지 살펴본 조건들을 종합하면, 좋은 프로젝트를 만드는 순서는 다음과 같습니다.

1. 문제 문장을 한 줄로 작성합니다.
2. 스코프 매트릭스를 채워 1차 범위를 3~5개 기능으로 제한합니다.
3. 아키텍처 다이어그램을 그려 컴포넌트 경계를 정합니다.
4. README 골격을 먼저 작성합니다.
5. 핵심 흐름부터 구현하고, 흐름마다 테스트를 작성합니다.
6. 결과 지표를 측정하는 스크립트를 만들고 수치를 기록합니다.
7. Docker와 시드 데이터로 재현 경로를 확보합니다.
8. 주간 품질 리포트를 작성하며 상태를 유지합니다.

이 순서를 따르면 기능이 3개뿐이어도 문제 정의, 구현, 검증, 문서, 재현이 모두 갖춰진 완성도 높은 프로젝트가 나옵니다. 순서를 뒤집어 기능부터 쌓으면 나중에 문서와 테스트를 보강할 시간이 부족해지는 경우가 대부분입니다.
## 처음 질문으로 돌아가기

- **왜 큰 프로젝트가 항상 더 좋은 포트폴리오가 되지는 않을까요?**
  - 범위가 커지면 마무리 품질이 떨어지고 문제 정의가 흐려집니다. 작은 범위를 끝까지 다듬은 프로젝트가 더 선명하게 읽히기 때문입니다.
- **문제 정의, 결과 지표, 유지보수성, 재현성은 각각 어떻게 드러날까요?**
  - 문제 정의는 README 첫 문단에, 결과 지표는 성과 표에, 유지보수성은 테스트와 CI 배지에, 재현성은 Docker와 실행 가이드에 드러납니다.
- **기능이 적어도 강한 프로젝트로 읽히게 만드는 기준은 무엇일까요?**
  - 규모보다는 '왜'를 설명할 수 있는지가 중요합니다. 작은 프로젝트도 판단 기준이 명확하면 실무 감각을 보여 줍니다.
