# Portfolio Project 101 (1/10): 포트폴리오 프로젝트란 무엇인가

많은 초보 개발자가 GitHub 저장소 하나를 만든 뒤 곧바로 포트폴리오라고 부릅니다. 하지만 채용 관점에서 코드를 올려 두는 일과 경험을 증명하는 일은 다릅니다. 저장소에는 구현이 담기지만, 포트폴리오에는 문제를 어떻게 이해했고 어떤 판단으로 결과를 만들었는지가 함께 담겨야 합니다.

이 글은 Portfolio Project 101 시리즈의 첫 번째 글입니다. 여기서는 포트폴리오 프로젝트를 숙제나 토이 저장소와 구분하는 기준을 먼저 잡고, 무엇이 있어야 검토 가능한 결과물로 읽히는지 살펴보겠습니다.

> 포트폴리오 프로젝트는 코드 한 덩어리가 아니라, 문제 정의에서 배포와 설명까지 모든 단계를 보여 주는 사례 연구입니다.

## 먼저 던지는 질문

- 포트폴리오 프로젝트와 단순한 연습 저장소는 무엇이 다를까요?
- 채용 담당자나 리뷰어는 코드보다 먼저 무엇을 확인할까요?
- 작은 프로젝트라도 포트폴리오로 읽히게 만드는 최소 구성은 무엇일까요?

## 왜 중요한가

포트폴리오는 경력을 대신해 주지 않지만, 경험을 빠르게 보여 주는 도구는 됩니다. 특히 실무 경험이 아직 많지 않은 개발자라면, 내가 어떤 문제를 골랐고 어디까지 완성했고 어떤 기준으로 선택했는지를 남겨 두는 자료가 필요합니다. 이때 가장 강한 증거가 포트폴리오 프로젝트입니다.

채용 담당자나 면접관은 저장소를 오래 읽지 않습니다. 보통 짧은 시간 안에 프로젝트의 문제, 해결 방식, 결과, 완성도를 훑습니다. 이때 코드만 보이면 구현 연습처럼 읽히고, 문제와 데모와 설명이 함께 보이면 실제로 끝까지 밀어 본 프로젝트처럼 읽힙니다. 차이는 기능 개수보다 맥락의 선명함에서 생깁니다.

## 머릿속에 먼저 그릴 그림

포트폴리오 프로젝트를 설명할 때는 문제에서 시작해 해결, 코드, 배포, 설명 문서로 이어지는 흐름으로 보면 이해가 쉽습니다.

이 그림이 말하는 핵심은 간단합니다. 포트폴리오는 코드 한 덩어리로 끝나지 않습니다. 문제 정의가 있어야 해결이 왜 필요한지 보이고, 배포와 README가 있어야 다른 사람이 실제로 확인할 수 있습니다. 결국 포트폴리오 프로젝트는 구현 결과물을 세상에 읽히는 형태로 정리한 패키지에 가깝습니다.

## 핵심 용어

- **포트폴리오**: 다른 사람이 검토할 수 있게 공개한 작업 결과물입니다.
- **사례 정리**: 문제, 해결, 결과를 묶어서 보여 주는 설명 방식입니다.
- **README**: 저장소에 처음 들어온 사람이 가장 먼저 읽는 입구 문서입니다.
- 데모: 프로젝트가 실제로 동작한다는 증거입니다.
- **결정 기록**: 왜 그런 선택을 했는지 남기는 판단 흔적입니다.

## 바꾸기 전과 후

**Before**: GitHub에 코드만 올려 두고, 무엇을 해결했는지 설명하지 않습니다.

**After**: 문제 설명, 데모, README, 선택 이유가 함께 있어 처음 보는 사람도 프로젝트의 가치를 짧은 시간 안에 파악할 수 있습니다.

전자는 작성자만 맥락을 알고 있는 저장소입니다. 후자는 방문자가 스스로 이해할 수 있는 포트폴리오입니다. 둘의 차이는 화려한 기술 스택이 아니라, 독자가 길을 잃지 않게 만드는 정리 수준에 있습니다.

## 단계별로 살펴보기

### 1단계 — 프로젝트 정의

포트폴리오 프로젝트는 기술 이름보다 문제에서 출발해야 합니다.

```python
project = {"name": "task-tracker", "problem": "lost team schedules"}
```

이 한 줄이 중요한 이유는 프로젝트의 목적을 정하기 때문입니다. 리뷰어는 "무엇을 만들었나"보다 "무슨 문제를 해결하려 했나"를 먼저 이해해야 이후 선택을 해석할 수 있습니다.

### 2단계 — 데모 URL

설명만으로는 충분하지 않습니다. 직접 열어 볼 수 있는 경로가 있어야 합니다.

```python
demo_url = "https://demo.example.com"
```

데모 URL은 프로젝트를 주장 단계에서 증명 단계로 옮깁니다. 링크 하나가 살아 있으면 구현 품질뿐 아니라 마무리 능력도 함께 드러납니다.

### 3단계 — README 골격

README는 방문자가 프로젝트를 읽는 순서를 설계하는 문서입니다.

```python
sections = ["problem", "demo", "stack", "run", "next"]
```

이 다섯 섹션만 있어도 저장소의 입구가 훨씬 선명해집니다. 문제, 데모, 스택, 실행 방법, 다음 작업이 보이면 프로젝트의 현재 상태를 빠르게 해석할 수 있습니다.

### 4단계 — 결정 기록

좋은 포트폴리오는 결과만이 아니라 판단도 남깁니다.

```python
decisions = [{"why": "FastAPI", "trade": "less_admin"}]
```

같은 기능을 만든 두 저장소가 있을 때, 왜 FastAPI를 골랐고 무엇을 포기했는지 적어 둔 쪽이 훨씬 강하게 읽힙니다. 채용에서는 정답보다 판단 기준을 더 궁금해하는 경우가 많습니다.

### 5단계 — 한 줄 소개

프로젝트를 한 문장으로 설명할 수 있어야 합니다.

```python
pitch = "A mini SaaS that fixes lost team schedules"
```

이 문장은 README, 블로그 글, 면접 답변까지 모두 이어지는 중심축이 됩니다. 설명이 길어질수록 프로젝트가 흐려지는 경우가 많기 때문에, 짧고 정확한 소개 문장을 먼저 다듬는 편이 좋습니다.

## 이 코드에서 먼저 볼 점

- 프로젝트는 기능 목록이 아니라 문제 정의에서 시작합니다.
- 데모는 선택 사항이 아니라 신뢰를 만드는 증거에 가깝습니다.
- README와 결정 기록이 있어야 구현이 판단 사례로 읽힙니다.

## 자주 하는 실수

1. 스크린샷만 올리고 실제 사용 흐름이나 데모 URL은 남기지 않는 경우
2. README가 한두 줄로 끝나서 문제와 실행 방법이 보이지 않는 경우
3. 왜 이런 기술을 골랐는지 전혀 설명하지 않는 경우
4. 기능 자랑만 있고 해결하려는 문제가 흐릿한 경우
5. 완성되지 않은 항목을 숨겨서 현재 상태를 오히려 더 모호하게 만드는 경우

이 실수들은 모두 같은 결과로 이어집니다. 방문자가 프로젝트를 해석할 재료를 얻지 못한다는 점입니다. 포트폴리오는 많이 보여 주는 것보다, 필요한 것을 빠르게 보여 주는 편이 더 강합니다.

## 실무에서는 이렇게 본다

실무에서도 프로젝트를 평가할 때 코드를 보기 전에 문제와 결과부터 확인하는 경우가 많습니다. 오픈소스 저장소를 봐도 README, 데모, 빠른 시작 문서, 선택 이유가 먼저 보이는 경우가 많습니다. 이유는 단순합니다. 처음 들어온 사람이 전체 코드를 읽을 시간은 거의 없기 때문입니다.

개인 포트폴리오도 같은 원리로 읽힙니다. 규모가 작아도 문제 정의가 분명하고, 데모가 살아 있고, 문서가 정돈되어 있으면 훨씬 성숙한 결과물처럼 보입니다.

## 체크리스트

- [ ] 한 줄로 설명할 수 있는 문제가 있다.
- [ ] 직접 열어 볼 수 있는 데모가 있다.
- [ ] README에 문제, 데모, 실행 방법이 정리되어 있다.
- [ ] 기술 선택 이유를 한두 개라도 남겨 두었다.

## 정리와 다음 글

포트폴리오 프로젝트는 코드를 공개하는 일보다 한 단계 더 나아간 결과물입니다. 문제를 정의하고, 구현을 보여 주고, 데모로 증명하고, README와 결정 기록으로 맥락을 남겨야 비로소 다른 사람이 검토 가능한 프로젝트가 됩니다.

다음 글에서는 이런 프로젝트가 실제로 좋은 포트폴리오로 읽히기 위해 어떤 조건을 갖춰야 하는지 더 구체적으로 살펴보겠습니다.

### 프로젝트 주제 선정 기준표

포트폴리오 프로젝트의 시작점은 기술이 아니라 문제입니다. 많은 초보 개발자가 "요즘 많이 쓰는 기술"을 먼저 고르고 주제를 나중에 맞추지만, 이 순서는 면접과 리뷰에서 약하게 읽힙니다. 주제 선정 단계에서는 "누가 어떤 문제를 겪고 있는가", "내가 그 문제를 어느 범위까지 해결할 수 있는가", "결과를 측정할 수 있는가"를 먼저 판단해야 합니다. 아래 기준표는 실제로 후보를 걸러내기 위한 최소 기준입니다.

| 기준 | 질문 | 통과 기준 | 실패 신호 |
| --- | --- | --- | --- |
| 문제 선명도 | 문제를 한 문장으로 말할 수 있는가 | 사용자/상황/불편이 한 문장에 들어감 | "편한 앱 만들기"처럼 추상적임 |
| 접근 가능성 | 내가 2~4주 안에 끝낼 수 있는가 | 범위 축소 후 핵심 흐름 완주 가능 | 기능 목록이 끝없이 늘어남 |
| 검증 가능성 | 결과를 숫자로 보여줄 수 있는가 | 응답시간, 전환율, 처리시간 등 지표 존재 | "좋아졌다" 수준의 주관적 표현 |
| 공개 가능성 | 민감 정보 없이 공개 가능한가 | 샘플/익명 데이터로 재현 가능 | 회사 데이터 없이는 데모 불가 |
| 확장 가능성 | 다음 개선 포인트가 보이는가 | TODO로 확장 계획 정리 가능 | 첫 버전이 곧 종착점 |

이 표를 쓰는 이유는 좋은 주제를 찾기 위해서가 아니라, 나쁜 시작을 빨리 버리기 위해서입니다. 포트폴리오에서 가장 흔한 실패는 어려운 주제를 고른 뒤 핵심 흐름 완주 전에 지치는 경우입니다. 반대로 적절한 난이도의 주제는 결과뿐 아니라 학습 경로까지 깔끔하게 남길 수 있습니다.

### 포트폴리오 프로젝트 vs 과제 프로젝트 비교

과제 프로젝트는 정답 근처를 재현하는 데 초점이 있고, 포트폴리오 프로젝트는 판단 과정을 증명하는 데 초점이 있습니다. 두 프로젝트 모두 코드가 나오지만, 읽히는 방식은 다릅니다.

| 항목 | 과제 프로젝트 | 포트폴리오 프로젝트 |
| --- | --- | --- |
| 출발점 | 주어진 요구사항 | 스스로 정의한 문제 |
| 성공 기준 | 제출/통과 | 검토자 이해 + 실제 동작 증명 |
| 범위 전략 | 요구사항 충족 | 핵심 흐름 완주 + 품질 강화 |
| 문서 역할 | 최소 안내 | 문제/선택/결과를 연결하는 입구 |
| 면접 활용도 | 구현 설명 중심 | 문제정의/트레이드오프/결과 설명 중심 |

이 차이는 기술 스택보다 중요합니다. React를 썼는지 FastAPI를 썼는지는 두 번째 질문이고, 왜 이 문제를 골랐고 왜 이 범위에서 멈췄는지는 첫 번째 질문입니다. 포트폴리오를 준비할 때는 "나는 무엇을 만들었나"보다 "나는 어떤 판단을 했나"가 더 강한 증거라는 점을 계속 기억해야 합니다.

### 주제 선정 의사결정 로그 예시

아래 예시는 실제 주제 선정 문서를 어떻게 남기면 좋은지 보여 줍니다. 복잡한 문서를 만들 필요는 없고, 기준과 결론이 드러나면 충분합니다.

```markdown
# Topic Decision Log

## Candidate A: 팀 일정 통합 대시보드
- 문제: 일정이 메신저/캘린더/문서에 분산되어 조회 비용이 큼
- 범위: 조회 + 필터 + 주간 뷰
- 리스크: OAuth 연동 복잡도
- 결론: 1차 포트폴리오 주제로 채택

## Candidate B: 실시간 채팅
- 문제: 명확하나 차별화 포인트 약함
- 범위: 인증/방/메시지/읽음 처리
- 리스크: WebSocket 운영 복잡도
- 결론: 보류 (2차 확장 주제)
```

이 기록이 있으면 나중에 면접에서 "왜 이 주제를 골랐나요?"라는 질문에 즉시 답할 수 있습니다. 또한 프로젝트가 진행되면서 범위를 줄이는 결정이 필요할 때도 출발 기준으로 돌아가 판단할 수 있습니다. 주제 선정은 시작 단계 같지만, 실제로는 끝까지 품질을 좌우하는 핵심 설계입니다.

### 실전 워크시트: 주제 후보 3개 평가

아래 워크시트는 실제로 프로젝트 시작 전에 30분 안에 작성할 수 있는 형태입니다. 각 후보를 같은 질문으로 비교하면 감정적 선택을 줄일 수 있습니다.

| 질문 | 후보 A | 후보 B | 후보 C |
| --- | --- | --- | --- |
| 해결하려는 사용자 문제는 무엇인가 |  |  |  |
| 2주 안에 MVP 완주 가능한가 |  |  |  |
| 데모로 증명 가능한 핵심 흐름은 무엇인가 |  |  |  |
| 결과를 숫자로 측정할 방법이 있는가 |  |  |  |
| 면접에서 트레이드오프 설명이 가능한가 |  |  |  |

작성 팁은 단순합니다. 빈칸이 많은 후보는 좋은 아이디어처럼 보여도 아직 포트폴리오로 옮길 준비가 안 된 주제일 가능성이 큽니다. 이 단계에서 보류하는 것이 실패가 아니라 품질 관리입니다.

### GitHub 프로필 README 템플릿

GitHub 프로필 README는 저장소 목록보다 먼저 보이는 자기소개 문서입니다. 프로필 README가 정돈되어 있으면 방문자는 이 사람이 어떤 문제에 관심이 있고, 어떤 결과물을 만들었는지 빠르게 파악할 수 있습니다. 반대로 비어 있거나 자동 생성된 프로필은 저장소를 하나씩 열어 봐야 전체 그림을 알 수 있어서 탐색 비용이 높아집니다.

아래 템플릿은 포트폴리오 목적에 맞게 구성한 최소 구조입니다.

```markdown
# 안녕하세요, [이름]입니다 👋

## 관심 분야
- 백엔드 API 설계와 운영 자동화
- 데이터 파이프라인과 관측 시스템

## 대표 프로젝트

| 프로젝트 | 한 줄 설명 | 스택 | 데모 |
|---|---|---|---|
| [task-tracker](링크) | 팀 일정 분산 문제 해결 | FastAPI, PostgreSQL | [demo](링크) |
| [log-viewer](링크) | 로그 검색 시간 80% 단축 | Python, Elasticsearch | [demo](링크) |

## 기술 스택
- Language: Python, SQL
- Framework: FastAPI, Flask
- Infra: Docker, GitHub Actions, AWS EC2

## 최근 활동
- 📝 [블로그 제목](링크) — 최근 작성한 기술 글
- 🔧 [PR 제목](링크) — 오픈소스 기여
```

이 템플릿에서 핵심은 대표 프로젝트 테이블입니다. 프로젝트 이름, 한 줄 설명, 스택, 데모를 한 행에 모아 두면 방문자가 3초 안에 관심 프로젝트를 골라 클릭할 수 있습니다. 기술 스택 목록은 너무 길면 오히려 초점이 흐려지므로, 실제로 프로젝트에 사용한 것만 적는 편이 좋습니다.

프로필 README를 작성할 때 자주 보이는 실수는 기술 배지를 수십 개 나열하는 경우입니다. 배지가 많으면 화려해 보이지만, 실제로 어떤 기술을 깊이 쓸 수 있는지는 프로젝트를 통해서만 증명됩니다. 배지는 핵심 3~5개로 줄이고, 대신 프로젝트 링크를 눈에 띄게 배치하는 편이 더 강합니다.

### 포트폴리오 프로젝트 디렉터리 구조

프로젝트 디렉터리 구조는 코드를 열기 전에 전체 설계를 보여 주는 지도 역할을 합니다. 구조가 정돈되어 있으면 리뷰어는 어디에 핵심 로직이 있고, 어디에 설정이 있고, 어디에 문서가 있는지 빠르게 찾을 수 있습니다.

```text
task-tracker/
├── README.md                  # 입구 문서: 문제, 데모, 실행 방법
├── docs/
│   ├── adr/                   # 기술 선택 기록 (Architecture Decision Records)
│   │   └── 001-why-fastapi.md
│   ├── architecture.md        # 전체 구조 설명
│   └── api-spec.md            # API 엔드포인트 명세
├── src/
│   ├── main.py                # 앱 진입점
│   ├── routes/                # HTTP 라우트
│   ├── services/              # 비즈니스 로직
│   ├── models/                # 데이터 모델
│   └── utils/                 # 공통 유틸리티
├── tests/
│   ├── unit/                  # 단위 테스트
│   ├── integration/           # 통합 테스트
│   └── conftest.py            # pytest fixture
├── .github/
│   └── workflows/
│       ├── ci.yml             # 테스트 + 린트 자동화
│       └── deploy.yml         # 배포 파이프라인
├── Dockerfile                 # 컨테이너 이미지 정의
├── docker-compose.yml         # 로컬 개발 환경
├── pyproject.toml             # 의존성 및 빌드 설정
├── .env.example               # 환경 변수 샘플
└── CHANGELOG.md               # 버전별 변경 이력
```

이 구조에서 눈여겨볼 점은 세 가지입니다.

첫째, `docs/` 디렉터리가 별도로 존재합니다. README만으로는 설명이 부족한 설계 판단이나 API 명세를 여기에 분리하면 README가 짧고 선명해집니다.

둘째, `tests/` 디렉터리가 `unit`과 `integration`으로 나뉘어 있습니다. 이 구분은 CI에서 빠른 피드백(unit)과 깊은 검증(integration)을 분리 실행할 수 있게 해 줍니다.

셋째, `.github/workflows/`에 CI와 배포가 모두 들어 있습니다. 이 파일들이 있으면 프로젝트가 자동화된 검증 체계를 갖추고 있다는 증거가 됩니다. 채용 담당자가 저장소를 열었을 때 Actions 탭에 녹색 배지가 보이면 완성도 인상이 크게 달라집니다.

디렉터리 구조를 설계할 때 흔한 실수는 모든 파일을 루트에 평탄하게 두는 경우입니다. 파일이 10개를 넘어가면 루트가 혼잡해지고, 처음 보는 사람은 어디서부터 읽어야 할지 판단하기 어렵습니다. 반대로 폴더를 너무 깊게 중첩하면 탐색 비용이 올라갑니다. 포트폴리오 수준에서는 2~3단계 깊이가 적절합니다.

### 채용 담당자의 저장소 탐색 순서

채용 담당자나 기술 면접관이 GitHub 저장소를 열었을 때 실제로 보는 순서를 알면, 어디에 힘을 써야 하는지 판단할 수 있습니다. 아래는 여러 채용 관련 글과 실제 리뷰어 피드백을 종합한 일반적인 탐색 경로입니다.

| 순서 | 확인 항목 | 소요 시간 | 판단 기준 |
| --- | --- | --- | --- |
| 1 | 저장소 설명(About) | 3초 | 한 줄로 문제가 보이는가 |
| 2 | README 첫 화면 | 15초 | 문제/데모/스택이 즉시 보이는가 |
| 3 | 데모 링크 클릭 | 30초 | 실제로 동작하는가 |
| 4 | 커밋 히스토리 | 10초 | 꾸준히 작업했는가, 메시지가 읽히는가 |
| 5 | 디렉터리 구조 | 10초 | 정돈되어 있는가 |
| 6 | CI/CD 상태 | 5초 | 녹색 배지가 있는가 |
| 7 | 코드 일부 | 60초 | 읽기 좋은가, 테스트가 있는가 |

총 탐색 시간은 보통 2~3분입니다. 이 시간 안에 프로젝트의 가치가 전달되지 않으면 다음 후보로 넘어갑니다. 이 표에서 알 수 있는 핵심은, 코드 품질을 확인하는 단계가 마지막이라는 점입니다. 그 전에 문제 정의, 데모, 구조가 먼저 필터 역할을 합니다.

이 순서를 알면 우선순위가 명확해집니다. 코드를 더 예쁘게 만드는 것보다 README 첫 화면과 데모 링크를 먼저 다듬는 편이 투자 대비 효과가 큽니다.

### 포트폴리오를 시작할 때 가장 먼저 할 일

지금까지 살펴본 내용을 종합하면, 포트폴리오 프로젝트를 시작할 때 가장 먼저 해야 할 일은 기술 선택이 아니라 문제 정의 문장 작성입니다. "누가, 어떤 상황에서, 어떤 불편을 겪는다"를 한 문장으로 쓸 수 있어야 이후 모든 선택이 그 문장을 기준으로 정렬됩니다.

문제 문장이 명확해지면 자연스럽게 범위가 줄어듭니다. "팀원이 일정을 한 곳에서 볼 수 없어서 매번 메신저를 뒤진다"는 문장이 있으면, 캘린더 통합 대시보드라는 범위가 바로 나옵니다. 반대로 "편한 협업 도구를 만들겠다"는 문장은 범위가 무한히 넓어져서 핵심 흐름 완주가 어려워집니다.

두 번째로 할 일은 README 골격을 먼저 작성하는 것입니다. 코드를 한 줄도 쓰지 않은 상태에서 README를 먼저 만들면, 프로젝트의 완성 형태를 미리 설계하는 효과가 있습니다. 문제, 예상 데모, 기술 스택, 실행 방법, 다음 단계를 빈칸으로라도 적어 두면 방향이 흔들릴 때 돌아올 기준점이 됩니다.

세 번째는 디렉터리 구조를 잡는 것입니다. 앞서 본 구조 템플릿을 복사해서 빈 폴더와 빈 파일을 만들어 두면, 코드를 작성할 때 자연스럽게 역할별로 분리됩니다. 나중에 구조를 바꾸는 것보다 처음부터 잡아 두는 편이 훨씬 비용이 적습니다.

## 처음 질문으로 돌아가기

- **포트폴리오 프로젝트와 단순한 연습 저장소는 무엇이 다를까요?**
  - 포트폴리오 프로젝트는 문제에서 시작해 데모, README, 선택 이유까지 함께 보여 줍니다. 연습 저장소는 구현만 있지만, 포트폴리오는 판단도 함께 남김니다.
- **채용 담당자나 리뷰어는 코드보다 먼저 무엇을 확인할까요?**
  - 가장 먼저 문제 설명과 README를 확인합니다. 그 다음 실제로 동작하는 데모 링크를 찾고, 마지막으로 왜 그런 선택을 했는지 설명한 부분을 봅니다.
- **작은 프로젝트라도 포트폴리오로 읽히게 만드는 최소 구성은 무엇일까요?**
  - 한 줄 문제 정의, 작동 중인 데모 또는 스크린샷, 단계별 설명이 있는 README, 기술 선택 이유 네 가지면 충분합니다.
