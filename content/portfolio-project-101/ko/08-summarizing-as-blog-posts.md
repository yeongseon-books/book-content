# Portfolio Project 101 (8/10): 블로그 글로 정리하기

GitHub 저장소는 결과물을 보여 주는 장소이지만, 검색으로 사람을 데려오는 장소는 아닙니다. 프로젝트를 잘 만들었는데도 아무도 보지 않는다면 구현이 부족해서가 아니라 발견될 수 있는 설명이 없어서일 때가 많습니다. 포트폴리오가 저장소에서 끝나면 결과는 남지만 과정과 판단은 널리 읽히기 어렵습니다.

이 글은 Portfolio Project 101 시리즈의 8번째 글입니다. 여기서는 프로젝트를 코드 나열이 아니라 문제, 접근, 결과, 학습이 있는 블로그 글로 바꾸는 방법을 살펴보겠습니다. 목표는 프로젝트를 더 많이 포장하는 것이 아니라, 더 많은 사람이 제대로 이해할 수 있게 만드는 데 있습니다.

## 먼저 던지는 질문

- 프로젝트를 블로그 글로 바꿀 때 왜 문제 중심 구조가 필요할까요?
- 코드 발췌와 저장소 링크는 어떻게 역할을 나누면 좋을까요?
- 결과를 숫자로 남기는 일이 왜 글의 설득력을 키울까요?

## 큰 그림

![Portfolio Project 101 8장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/portfolio-project-101/08/08-01-diagram.ko.png)

*Portfolio Project 101 8장 흐름 개요*

이 그림에서는 포트폴리오 프로젝트의 경험을 블로그 글로 재정의하는 과정을 봅니다. 블로그 글은 프로젝트의 배경이 된 학습을 다른 개발자와 나누는 채널입니다.

> 프로젝트를 만든 경험은 개인의 자산입니다. 그 경험을 글로 쓸 때, 비로소 다른 개발자에게 공유 가능한 자산이 됩니다.

## 왜 중요한가

블로그 글은 프로젝트를 발견 가능하게 만듭니다. 저장소를 바로 찾는 사람은 많지 않지만, 문제 해결 과정을 검색하는 사람은 많습니다. 그래서 "무엇을 만들었는가"보다 "어떤 문제를 어떻게 풀었는가"를 설명하는 글이 훨씬 오래 살아남습니다.

또 글은 코드보다 맥락을 잘 남깁니다. 왜 이 문제를 골랐는지, 어떤 시행착오가 있었는지, 어떤 기준으로 범위를 줄였는지는 저장소만으로는 잘 보이지 않습니다. 블로그 글은 바로 그 빈칸을 메워 줍니다.

## 머릿속에 먼저 그릴 그림

좋은 기술 글은 문제에서 출발해 접근, 핵심 코드, 결과, 학습으로 이어집니다.

이 흐름이 강한 이유는 독자의 관심 이동과 맞아떨어지기 때문입니다. 문제를 이해하면 접근이 궁금해지고, 접근을 이해하면 구현이 궁금해지고, 구현을 보면 결과가 궁금해집니다. 마지막 학습은 그 경험을 다시 일반화해 줍니다.

## 핵심 용어

- 글: 하나의 주제를 다루는 단일 아티클입니다.
- 발췌: 전체 코드가 아니라 설명에 꼭 필요한 작은 조각입니다.
- **검색 최적화**: 검색 결과에서 글이 잘 발견되도록 돕는 제목과 구조입니다.
- 시리즈: 여러 글이 이어지는 연결 구조입니다.
- **기준 URL**: 원본 글이나 저장소로 이어지는 대표 경로입니다.

## 바꾸기 전과 후

**Before**: 코드 블록만 길게 이어져 있어서 읽는 사람이 맥락을 잡기 어렵습니다.

**After**: 문제, 접근, 결과가 분리되어 있어 프로젝트의 가치가 짧은 시간 안에 드러납니다.

전자의 글은 작성자 메모에 가깝고, 후자의 글은 독자 경험을 고려한 설명에 가깝습니다. 포트폴리오 글은 구현 일지이면서 동시에 독자를 안내하는 문서여야 합니다.

## 단계별로 살펴보기

### 1단계 — 문제를 한 줄로 쓰기

먼저 독자가 공감할 문제를 한 문장으로 적습니다.

```markdown
> How we fixed the lost team schedule problem
```

이 한 줄이 글 전체의 중심축입니다. 제목과 도입, 결과가 모두 이 문장으로 다시 연결되어야 글이 흔들리지 않습니다.

### 2단계 — 접근 정리

문제를 보고 어떤 순서로 판단했는지 짧게 정리합니다.

```python
approach = ["observe", "hypothesis", "MVP", "deploy"]
```

이 부분은 기술 자랑이 아니라 사고 흐름 설명입니다. 독자는 정답보다 판단 과정을 보고 싶어 합니다. 그래서 접근 섹션은 기술 목록보다 의사결정 순서가 더 잘 보여야 합니다.

### 3단계 — 코드 발췌

코드는 설명에 꼭 필요한 조각만 남깁니다.

```python
def normalize(date_str):
    return date_str.replace(".", "-")
```

발췌의 목적은 전체 구현을 복제하는 데 있지 않습니다. 핵심 아이디어가 어디에 있었는지 보여 주는 데 있습니다. 나머지 구현은 저장소 링크로 보내는 편이 더 읽기 좋습니다.

### 4단계 — 결과 제시

결과는 모호한 만족감보다 숫자로 남기는 편이 좋습니다.

```python
result = {"users": 30, "latency_ms": 120}
```

숫자가 있으면 결과가 훨씬 구체적으로 읽힙니다. 사용자가 몇 명이었는지, 응답 시간이 어느 정도였는지, 무엇이 얼마나 개선되었는지가 보여야 글이 사례로 남습니다.

### 5단계 — 학습으로 마무리

마지막에는 다시 적용할 수 있는 판단을 한 문장으로 남깁니다.

```python
lesson = "MVP only survives when small"
```

이 문장은 단순한 소감이 아닙니다. 앞으로 비슷한 문제를 만났을 때 어떤 기준을 다시 쓸 수 있는지 보여 주는 부분입니다.

## 이 코드에서 먼저 볼 점

- 문제는 한 줄이어야 글 전체 초점이 흐려지지 않습니다.
- 코드는 설명을 돕는 정도만 남기는 편이 더 읽기 좋습니다.
- 결과는 숫자로 남겨야 글이 기억에 남는 사례가 됩니다.

## 자주 하는 실수

1. 문제 설명 없이 코드만 길게 붙여 넣는 경우
2. 결과나 사용 지표가 없어 성과가 흐릿한 경우
3. 제목이 모호해서 검색 의도와 맞지 않는 경우
4. 스크린샷이나 데모 링크가 없어 증거가 부족한 경우
5. 다음 글이나 관련 글 연결이 없어 시리즈 흐름이 끊기는 경우

블로그 글은 정보가 많다고 좋은 것이 아닙니다. 독자가 어디에서 들어왔든 문제를 이해하고, 접근을 따라가고, 결과를 가져갈 수 있어야 합니다. 그 경로를 막는 군더더기를 줄이는 편이 더 중요합니다.

## 실무에서는 이렇게 본다

좋은 엔지니어링 블로그는 대체로 문제, 해결, 결과 구조를 유지합니다. 회사 기술 블로그도 마찬가지입니다. 독자가 궁금한 것은 "이 기술을 썼다"가 아니라 "왜 이 선택이 필요했고 어떤 결과가 났는가"이기 때문입니다.

개인 포트폴리오 글도 이 원리를 따르면 됩니다. 코드 전체를 다 보여 주기보다, 문제 해결의 핵심 장면을 잘라 보여 주는 편이 훨씬 설득력 있습니다.

## 체크리스트

- [ ] 문제를 한 줄로 설명했다.
- [ ] 코드 블록 수를 꼭 필요한 수준으로 제한했다.
- [ ] 결과를 숫자나 지표로 남겼다.
- [ ] 마지막에 학습을 한 문장으로 적었다.
- [ ] 다음 글이나 관련 글로 이어지는 경로를 넣었다.

## 연습 문제

1. 여러분 프로젝트를 검색 유입형 제목으로 바꿔 한 줄 써 보세요.
2. 전체 코드 대신 글에 남길 핵심 코드 조각 하나를 골라 보세요.
3. 결과를 숫자로 적기 어렵다면 무엇을 측정하면 좋을지 적어 보세요.

## 정리와 다음 글

포트폴리오 프로젝트를 글로 바꿀 때 중요한 것은 코드 양이 아니라 맥락의 선명함입니다. 문제를 먼저 제시하고, 접근을 설명하고, 핵심 코드만 보여 주고, 결과를 숫자로 남기고, 마지막에 학습을 정리하면 저장소보다 더 넓은 독자에게 도달할 수 있습니다.

다음 글에서는 이렇게 정리한 프로젝트를 면접 자리에서 짧고 설득력 있게 설명하는 방법을 살펴보겠습니다.

### 성과 측정 지표 표 설계

블로그 글의 설득력을 높이려면 결과를 감상문이 아니라 지표로 제시해야 합니다. 지표는 복잡할 필요가 없지만, 문제와 직접 연결되어야 합니다. 아래 표는 포트폴리오 글에 자주 쓰는 기본 지표 예시입니다.

| 영역 | 지표 | 측정 방법 | 보고 형식 |
| --- | --- | --- | --- |
| 성능 | 평균 응답시간(ms) | APM/로그 집계 | 이전 대비 변화율 |
| 사용성 | 핵심 작업 완료 시간 | 사용자 시나리오 측정 | Before/After 비교 |
| 안정성 | 오류율(%) | 에러 로그 집계 | 주간 추이 |
| 도입 | 주간 활성 사용자 | 분석 도구 집계 | 누적/증감 |

지표를 고를 때 중요한 기준은 "면접에서 설명 가능한가"입니다. 수치를 제시했는데 측정 방법을 설명하지 못하면 오히려 신뢰를 잃을 수 있습니다.

### 트래픽/성능 데이터 시각화 방법

글에 데이터가 들어갈 때는 숫자 나열보다 구조화가 중요합니다. 최소한 아래 세 가지 방식 중 하나를 쓰면 가독성이 올라갑니다.

1. 기간 비교 표: 도입 전/후를 한 번에 비교
2. 핵심 그래프 1개: 응답시간 또는 오류율 추이
3. 요약 박스: "가장 큰 변화 3개"를 텍스트로 압축

```markdown
| 지표 | 도입 전 | 도입 후 | 변화 |
| --- | ---: | ---: | ---: |
| 평균 응답시간 | 280ms | 120ms | -57% |
| 오류율 | 3.2% | 0.8% | -75% |
| 주간 활성 사용자 | 12 | 31 | +158% |
```

이 형식은 독자가 한눈에 결과를 해석하게 돕습니다. 블로그 글에서 가장 중요한 것은 "읽는 사람이 빠르게 이해하는 경험"입니다.

### 사례 글 본문 구조(문제-해결-결과)

성과를 담은 글은 아래 구조를 따를 때 전달력이 높습니다.

```markdown
## 문제
팀 일정 조회가 여러 도구에 흩어져 계획 시간이 길어짐

## 접근
MVP 범위 축소, FastAPI + PostgreSQL, 주간 뷰 중심 UX

## 결과
응답시간 57% 개선, 계획 시간 40분 -> 18분, 주간 활성 사용자 31명

## 교훈
범위를 줄인 MVP가 오히려 측정 가능한 개선을 만든다.
```

이 구조의 장점은 면접 답변과도 연결된다는 점입니다. 블로그 글과 면접 설명이 같은 서사를 공유하면 프로젝트 메시지가 훨씬 일관되게 남습니다.

### 글의 신뢰도를 높이는 증거 묶음

블로그 글에서 신뢰를 높이려면 주장-증거-해석을 함께 제시해야 합니다. 아래 묶음을 기준으로 작성하면 내용이 단단해집니다.

| 주장 | 증거 | 해석 |
| --- | --- | --- |
| 응답속도가 개선되었다 | 평균 280ms -> 120ms | 캐시 전략이 병목 구간에 효과 |
| 사용성이 좋아졌다 | 작업 완료 시간 40분 -> 18분 | 단일 화면 통합이 탐색 비용 감소 |
| 운영 안정성이 높아졌다 | 오류율 3.2% -> 0.8% | 배포 전 테스트 자동화 효과 |

증거 없는 주장은 기억되지 않고, 해석 없는 숫자는 의미가 흐립니다. 두 가지를 함께 써야 글이 사례 연구로 남습니다.

### 시각 자료 배치 원칙

1. 그래프는 본문 주장 바로 아래에 배치합니다.
2. 표 제목은 무엇을 비교하는지 명확히 씁니다.
3. 한 섹션에 그래프를 과도하게 넣지 않습니다.
4. 원본 데이터 출처를 함께 남깁니다.

이 원칙을 지키면 긴 글에서도 독자가 핵심 메시지를 놓치지 않습니다.

### 품질 점검 기록 예시(반복 가능한 운영 습관)

포트폴리오 품질은 한 번의 대규모 수정으로 완성되지 않습니다. 짧은 주기로 같은 항목을 반복 점검할 때 품질이 안정됩니다. 아래 기록 방식은 프로젝트 크기와 무관하게 적용할 수 있습니다.

| 점검 항목 | 확인 질문 | 증거 형태 |
| --- | --- | --- |
| 문제 정의 | 문제 문장이 여전히 현재 구현과 일치하는가 | README/블로그 도입 문장 |
| 데모 신뢰성 | 링크가 열리고 핵심 흐름이 실제로 작동하는가 | 라이브 URL, 백업 영상 |
| 검증 체계 | 테스트와 CI 상태가 녹색인가 | CI 배지, 테스트 로그 |
| 문서 정합성 | 실행 방법과 환경 변수 설명이 최신인가 | README, `.env.example` |
| 설명 일관성 | 저장소/블로그/면접 답변이 같은 메시지를 말하는가 | 요약 문장 비교 |

이 표를 기준으로 주기적 점검을 하면 개선 우선순위를 빠르게 정할 수 있습니다. 예를 들어 데모는 정상이지만 문서 정합성이 낮다면 기능 추가보다 문서 보완이 먼저입니다. 반대로 문서는 완벽하지만 CI가 불안정하면 공개 신뢰를 잃기 쉽습니다.

다음 템플릿은 실제 기록에 바로 사용할 수 있습니다.

```markdown
## Weekly Portfolio Health Check
- 날짜: 2026-05-21
- 현재 상태: 데모 정상 / 문서 일부 불일치 / CI 정상
- 발견 이슈:
  1) README 실행 명령이 최신 compose 파일과 다름
  2) 블로그 성과 수치가 최근 측정값과 불일치
- 조치 계획:
  1) README 실행 섹션 업데이트
  2) 성과 표 재계산 및 캡처 갱신
- 완료 기준:
  - 신규 방문자가 5분 안에 문제/데모/실행/결과를 이해할 수 있음
```

핵심은 완벽함이 아니라 반복성입니다. 같은 기준으로 꾸준히 점검하면 프로젝트는 시간이 지나도 낡지 않습니다. 면접에서도 "어떻게 유지했는가"를 설명할 수 있어 결과물의 신뢰도가 더 높아집니다.

### 실행 가능한 개선 TODO 예시

아래 TODO는 "읽고 끝나는 조언"이 아니라 실제로 바로 적용할 수 있는 작업 단위입니다. 각 항목은 30~90분 안에 끝낼 수 있는 크기로 쪼개는 것을 권장합니다.

| 작업 | 예상 시간 | 완료 기준 |
| --- | ---: | --- |
| 핵심 메시지 1문장 재작성 | 30분 | README/발표/이력서 문장 일치 |
| 데모 검증 리허설 1회 | 45분 | 로그인-핵심행동-결과 흐름 무중단 |
| 테스트 명령 정리 | 30분 | 로컬/CI 명령이 문서와 일치 |
| 결과 수치 업데이트 | 60분 | 최신 측정값과 표/그래프 동기화 |

작업을 작은 단위로 나누면 진행률이 눈에 보이고, 품질 개선이 미뤄지지 않습니다. 포트폴리오를 오래 유지하려면 큰 개편보다 이런 짧은 루틴이 더 효과적입니다.

### 마무리 검증 메모

최종 공개 전에는 "내가 이해하는 프로젝트"가 아니라 "처음 보는 사람이 이해하는 프로젝트"인지 확인해야 합니다. 이를 위해 1) 문제를 한 문장으로 말해 보기, 2) 데모를 처음부터 끝까지 클릭해 보기, 3) 실행 명령을 새 환경에서 복사 실행해 보기, 4) 결과 수치를 다시 계산해 보기 네 단계를 반복하는 것이 좋습니다. 이 검증을 통과하면 문서와 데모와 코드의 정합성이 높아지고, 면접 답변도 훨씬 안정됩니다.

## 처음 질문으로 돌아가기

- **프로젝트를 블로그 글로 바꿀 때 왜 문제 중심 구조가 필요할까요?**
  - 기술 선택의 이유, 해결한 문제의 어려움, 실패과 다시 시도, 배운 교훈 순서가 자연스럽니다. 코드를 설명하지 말고, 왜 그 코드를 썼는지 설명하세요.
- **코드 발췌와 저장소 링크는 어떻게 역할을 나누면 좋을까요?**
  - 프로젝트 문서는 '사용 설명서'이고, 블로그 글은 '해결 사례'입니다. 문서는 WHAT, 글은 WHY를 설명합니다.
- **결과를 숫자로 남기는 일이 왜 글의 설득력을 키울까요?**
  - 모든 프로젝트가 글감이 되는 것은 아닙니다. 다른 개발자들도 격을 만한 문제를 해결했거나, 다른 개발자들이 배울 만한 어려움을 기록했을 때 글을 쓰는 편이 효과적습니다.
