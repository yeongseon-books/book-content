# 블로그 글로 정리하기

> 포트폴리오 프로젝트 101 시리즈 (8/10)

GitHub 저장소는 결과물을 보여 주는 장소이지만, 검색으로 사람을 데려오는 장소는 아닙니다. 많은 개발자가 프로젝트를 만들고도 “왜 아무도 보지 않지?”라는 질문을 합니다. 대개 이유는 단순합니다. 구현은 있었지만, 발견될 수 있는 설명이 없었기 때문입니다.

블로그 글은 포트폴리오의 확장판입니다. 저장소가 결과를 보여 준다면, 글은 과정과 판단을 보여 줍니다. 특히 검색 유입이 중요한 개인 브랜딩에서는 “무엇을 만들었는가”보다 “어떤 문제를 어떻게 풀었는가”를 설명하는 글이 훨씬 오래 살아남습니다.

## 이 글에서 다룰 문제

- 프로젝트를 블로그 글로 바꿀 때 코드 나열이 아니라 문제 중심 글이 되어야 하는 이유는 무엇일까요?
- 한 편의 글에서 문제, 접근, 코드 발췌, 결과, 학습을 어떤 순서로 배치해야 독자가 끝까지 읽을까요?
- SEO 제목과 스크린샷, 시리즈 연결은 단순한 장식이 아니라 어떤 역할을 할까요?
- 프로젝트를 읽는 사람과 검색으로 들어오는 사람은 무엇을 다르게 궁금해하는지 어떻게 구분할 수 있을까요?

## 왜 중요한가

블로그 글은 프로젝트를 발견 가능하게 만듭니다. 저장소를 바로 찾는 사람은 많지 않지만, 문제 해결 과정을 검색하는 사람은 많습니다. “Flask로 일정 관리 앱 만들기”보다 “팀 일정이 자꾸 흩어질 때 어떻게 정리했는가”가 더 자연스러운 검색 질문에 가깝습니다.

또 글은 코드보다 문맥을 잘 남깁니다. 왜 이 문제를 골랐는지, 어떤 시행착오가 있었는지, 어떤 기준으로 MVP 범위를 줄였는지는 저장소만으로 잘 보이지 않습니다. 블로그는 그 빈칸을 채워 주는 채널입니다.

## 한눈에 보는 흐름

좋은 기술 글은 대개 문제에서 시작해 접근을 설명하고, 필요한 만큼의 코드만 보여 주고, 결과를 숫자로 남기고, 마지막에 배운 점으로 닫습니다.

```mermaid
flowchart LR
    P[문제] --> A[접근]
    A --> C[코드]
    C --> R[결과]
    R --> L[학습]
```

이 흐름이 강한 이유는 독자의 관심이 자연스럽게 이동하기 때문입니다. 문제를 이해하면 접근이 궁금해지고, 접근을 이해하면 구현이 궁금해지고, 구현을 보면 결과가 궁금해집니다. 마지막 학습은 그 경험을 다시 일반화해 줍니다.

## 핵심 용어

- **글(post)**: 하나의 주제를 다루는 단일 아티클입니다.
- **발췌(excerpt)**: 전체 코드가 아니라 설명에 꼭 필요한 작은 조각입니다.
- **SEO**: 검색 결과에서 글이 잘 발견되도록 돕는 제목과 구조입니다.
- **시리즈(series)**: 여러 글이 이어지는 연결 구조입니다.
- **원문 URL(canonical)**: 기준이 되는 원본 글의 주소입니다.

## 바꾸기 전 / 후

**Before**: 코드 블록만 길게 붙어 있는 글이라 읽는 사람이 맥락을 잡기 어렵습니다.

**After**: 문제, 해결 접근, 결과가 분리되어 있어 프로젝트의 가치가 짧은 시간 안에 드러납니다.

전자의 글은 작성자 회고에 가깝고, 후자의 글은 독자 경험을 고려한 설명에 가깝습니다. 블로그 글은 구현 일지이면서 동시에 독자를 안내하는 문서여야 합니다.

## 실습: 글 골격 만들기

### 1단계 — 문제를 한 줄로 쓰기

가장 먼저 독자가 공감할 문제를 한 문장으로 적습니다.

```markdown
> 팀 일정 분실 문제를 어떻게 해결했나
```

이 한 줄이 글의 나침반 역할을 합니다. 제목과 도입, 결과가 모두 여기로 다시 연결되어야 글이 흔들리지 않습니다.

### 2단계 — 접근 정리

문제를 보고 어떤 순서로 판단했는지 짧게 정리합니다.

```python
approach = ["관찰", "가설", "MVP", "배포"]
```

이 부분은 기술 선택 자랑이 아니라 사고 흐름 설명입니다. 독자는 정답보다 판단 과정을 보고 싶어 합니다.

### 3단계 — 코드 발췌

코드는 전부 옮기지 말고 설명에 꼭 필요한 조각만 남깁니다.

```python
def normalize(date_str):
    return date_str.replace(".", "-")
```

발췌의 목적은 전체 구현을 복제하는 데 있지 않습니다. 핵심 아이디어가 어디에 있었는지 보여 주는 데 있습니다.

### 4단계 — 결과 제시

결과는 모호한 만족감보다 숫자로 남기는 편이 좋습니다.

```python
result = {"users": 30, "latency_ms": 120}
```

숫자가 있으면 결과가 훨씬 구체적으로 읽힙니다. 사용자가 몇 명이었는지, 응답 시간이 어느 정도였는지, 무엇이 개선되었는지가 보여야 글이 사례로 남습니다.

### 5단계 — 학습으로 마무리

마지막에는 배운 점을 한 문장으로 적습니다.

```python
lesson = "MVP 는 작아야 산다"
```

이 문장은 단순한 소감이 아닙니다. 앞으로 비슷한 문제를 만났을 때 다시 적용할 기준을 남기는 부분입니다.

## 이 코드에서 봐야 할 점

- 문제는 한 줄이어야 합니다. 길어지면 글 전체의 초점이 흐려집니다.
- 코드는 설명을 돕는 정도만 남겨야 합니다. 전체 구현은 저장소 링크로 넘기는 편이 낫습니다.
- 결과는 숫자로 말해야 합니다. 독자가 변화를 판단할 재료가 있어야 합니다.

## 자주 하는 실수 5가지

1. 문제 설명 없이 코드만 길게 붙여 넣는 경우
2. 결과나 사용 지표가 없어 글의 성과가 흐릿한 경우
3. 제목이 모호해서 검색 의도와 맞지 않는 경우
4. 스크린샷이나 데모 링크가 없어 시각적 증거가 부족한 경우
5. 다음 글이나 관련 글 연결이 없어 시리즈 맥락이 끊기는 경우

이런 실수가 생기면 글은 정보는 많아도 기억에는 남지 않습니다. 블로그 글은 기술 메모이면서 동시에 독자의 이동 경로를 설계하는 작업이기도 합니다.

## 실무에서는 이렇게 보입니다

좋은 엔지니어링 블로그는 대체로 문제, 해결, 결과 구조를 유지합니다. 회사 기술 블로그도 마찬가지입니다. 독자가 궁금한 것은 “이 기술을 썼다”가 아니라 “왜 이 선택이 필요했고 어떤 결과가 났는가”이기 때문입니다.

개인 포트폴리오 글도 같은 원리를 따르면 됩니다. 코드 전체를 다 보여 주기보다, 문제 해결의 핵심 장면을 잘라 보여 주는 편이 더 설득력 있습니다.

## 시니어 엔지니어는 이렇게 판단합니다

- 문제 정의가 공감대를 만드는지 먼저 봅니다.
- 접근은 서사처럼 읽혀야 합니다. 시행착오가 있어도 좋지만 방향은 보여야 합니다.
- 코드는 최소량만 남기고 나머지는 링크로 보내는 편이 읽기 좋습니다.
- 결과는 숫자로 남겨야 나중에 다시 인용하기 쉽습니다.
- 학습은 솔직할수록 좋습니다. 과장보다 판단 기준이 남는 편이 낫습니다.

블로그 글은 프로젝트를 포장하는 수단이 아니라, 프로젝트를 더 잘 읽히게 만드는 해설서라고 보면 됩니다.

## 체크리스트

- [ ] 문제를 한 줄로 설명했다.
- [ ] 코드 블록 수를 꼭 필요한 수준으로 제한했다.
- [ ] 결과를 숫자나 지표로 남겼다.
- [ ] 마지막에 배운 점을 한 문장으로 적었다.
- [ ] 다음 글이나 관련 글로 이어지는 경로를 넣었다.

## 연습 문제

1. 여러분 프로젝트를 검색 유입형 제목으로 바꿔 한 줄 써 보세요.
2. 전체 코드 대신 글에 남길 핵심 코드 조각 하나를 골라 보세요.
3. 결과를 숫자로 적기 어렵다면 무엇을 측정하면 좋을지 세 가지 적어 보세요.

## 정리 및 다음 글

포트폴리오 프로젝트를 글로 바꿀 때 중요한 것은 코드 양이 아니라 맥락의 선명함입니다. 문제를 먼저 제시하고, 접근을 설명하고, 핵심 코드만 보여 주고, 결과를 숫자로 남기고, 마지막에 학습을 정리하면 저장소보다 더 넓은 독자에게 도달할 수 있습니다.

다음 글에서는 이렇게 정리한 프로젝트를 면접 자리에서 짧고 설득력 있게 설명하는 방법을 이어서 살펴보겠습니다.
