# Portfolio Project 101 (8/10): 블로그 글로 정리하기

GitHub 저장소는 결과물을 보여 주는 장소이지만, 검색으로 사람을 데려오는 장소는 아닙니다. 프로젝트를 잘 만들었는데도 아무도 보지 않는다면 구현이 부족해서가 아니라 발견될 수 있는 설명이 없어서일 때가 많습니다. 포트폴리오가 저장소에서 끝나면 결과는 남지만 과정과 판단은 널리 읽히기 어렵습니다.

이 글은 Portfolio Project 101 시리즈의 8번째 글입니다. 여기서는 프로젝트를 코드 나열이 아니라 문제, 접근, 결과, 학습이 있는 블로그 글로 바꾸는 방법을 봅니다. 목표는 프로젝트를 더 많이 포장하는 것이 아니라, 더 많은 사람이 제대로 이해할 수 있게 만드는 데 있습니다.

*Portfolio Project 101 8장 흐름 개요*
> 프로젝트를 만든 경험은 개인의 자산입니다. 그 경험을 글로 쓸 때, 비로소 다른 개발자에게 공유 가능한 자산이 됩니다.

## 먼저 던지는 질문

- 프로젝트를 블로그 글로 바꿀 때 왜 문제 중심 구조가 필요할까요?
- 코드 발췌와 저장소 링크는 어떻게 역할을 나누면 좋을까요?
- 결과를 숫자로 남기는 일이 왜 글의 설득력을 키울까요?

## 왜 중요한가

블로그 글은 프로젝트를 발견 가능하게 만듭니다. 저장소를 바로 찾는 사람은 많지 않지만, 문제 해결 과정을 검색하는 사람은 많습니다. 그래서 "무엇을 만들었는가"보다 "어떤 문제를 어떻게 풀었는가"를 설명하는 글이 훨씬 오래 살아남습니다.

또 글은 코드보다 맥락을 잘 남깁니다. 왜 이 문제를 골랐는지, 어떤 시행착오가 있었는지, 어떤 기준으로 범위를 줄였는지는 저장소만으로는 잘 보이지 않습니다. 블로그 글은 바로 그 빈칸을 메워 줍니다.

추가로, 블로그 글은 "시간이 지나도 살아 있는 포트폴리오"를 만들어 줍니다. 저장소는 마지막 커밋 날짜로 판단되지만, 블로그 글은 발행 날짜와 무관하게 검색됩니다. 2년 전에 쓴 글이라도 문제와 해결이 명확하면 여전히 새 독자를 데려옵니다.

## 머릿속에 먼저 그릴 그림

좋은 기술 글은 문제에서 출발해 접근, 핵심 코드, 결과, 학습으로 이어집니다.


이 흐름이 강한 이유는 독자의 관심 이동과 맞아떨어지기 때문입니다. 문제를 이해하면 접근이 궁금해지고, 접근을 이해하면 구현이 궁금해지고, 구현을 보면 결과가 궁금해집니다. 마지막 학습은 그 경험을 다시 일반화해 줍니다. 이 순서를 바꾸면 독자는 글을 이탈합니다.

## 핵심 용어

- 글: 하나의 주제를 다루는 단일 아티클입니다.
- 발췌: 전체 코드가 아니라 설명에 꼭 필요한 작은 조각입니다.
- **검색 최적화**: 검색 결과에서 글이 잘 발견되도록 돕는 제목과 구조입니다.
- 시리즈: 여러 글이 이어지는 연결 구조입니다.
- **기준 URL**: 원본 글이나 저장소로 이어지는 대표 경로입니다.

## 바꾸기 전과 후

**Before**: 코드 블록만 길게 이어져 있어서 읽는 사람이 맥락을 잡기 어렵습니다.

**After**: 문제, 접근, 결과가 분리되어 있어 프로젝트의 가치가 짧은 시간 안에 드러납니다.

전자의 글은 작성자 메모에 가깝고, 후자의 글은 독자 경험을 고려한 설명에 가깝습니다. 포트폴리오 글은 구현 일지이면서 동시에 독자를 안내하는 문서여야 합니다. 이 츨이를 만드는 것이 "문제 중심 구조"의 핵심입니다.

## 단계별로 살펴보기

### 1단계 — 문제를 한 줄로 쓰기

먼저 독자가 공감할 문제를 한 문장으로 적습니다.

```markdown
> How we fixed the lost team schedule problem
```

이 한 줄이 글 전체의 중심축입니다. 제목과 도입, 결과가 모두 이 문장으로 다시 연결되어야 글이 흔들리지 않습니다.

### 2단계 — 접근 정리

문제를 보고 어떤 순서로 판단했는지 짧게 정리합니다.

접근은 보통 아래 순서처럼 짧게 정리하면 충분합니다.

- 관찰
- 가설 설정
- MVP 범위 확정
- 배포

이 부분은 기술 자랑이 아니라 사고 흐름 설명입니다. 독자는 정답보다 판단 과정을 보고 싶어 합니다. 그래서 접근 섹션은 기술 목록보다 의사결정 순서가 더 잘 보여야 합니다.

### 3단계 — 코드 발췌

코드는 설명에 꼭 필요한 조각만 남깁니다.

```python
def normalize(date_str):
    return date_str.replace(".", "-")
```

발췌의 목적은 전체 구현을 복제하는 데 있지 않습니다. 핵심 아이디어가 어디에 있었는지 보여 주는 데 있습니다. 나머지 구현은 저장소 링크로 보내는 편이 더 읽기 좋습니다.

### 4단계 — 결과 제시

결과는 모호한 만족감보다 숫자로 남기는 편이 좋습니다.

예를 들어 `사용자 30명`, `응답 시간 120ms`처럼 결과를 숫자로 적어 두면 글의 설득력이 올라갑니다.

숫자가 있으면 결과가 훨씬 구체적으로 읽힙니다. 사용자가 몇 명이었는지, 응답 시간이 어느 정도였는지, 무엇이 얼마나 개선되었는지가 보여야 글이 사례로 남습니다.

### 5단계 — 학습으로 마무리

마지막에는 다시 적용할 수 있는 판단을 한 문장으로 남깁니다.

예를 들어 `MVP는 작을수록 끝까지 살아남기 쉽다` 같은 한 문장으로 학습을 정리할 수 있습니다.

이 문장은 단순한 소감이 아닙니다. 앞으로 비슷한 문제를 만났을 때 어떤 기준을 다시 쓸 수 있는지 보여 주는 부분입니다. 독자가 글을 다 읽은 뒤 "나도 이 기준을 써 봐야지"라고 느끼면 성공입니다.

## 이 코드에서 먼저 볼 점

- 문제는 한 줄이어야 글 전체 초점이 흐려지지 않습니다.
- 코드는 설명을 돕는 정도만 남기는 편이 더 읽기 좋습니다.
- 결과는 숫자로 남겨야 글이 기억에 남는 사례가 됩니다.

## 자주 하는 실수

1. 문제 설명 없이 코드만 길게 붙여 넣는 경우
2. 결과나 사용 지표가 없어 성과가 흐릿한 경우
3. 제목이 모호해서 검색 의도와 맞지 않는 경우
4. 스크린샷이나 데모 링크가 없어 증거가 부족한 경우
5. 다음 글이나 관련 글 연결이 없어 시리즈 흐름이 끊기는 경우

블로그 글은 정보가 많다고 좋은 것이 아닙니다. 독자가 어디에서 들어왔든 문제를 이해하고, 접근을 따라가고, 결과를 가져갈 수 있어야 합니다. 그 경로를 막는 군더더기를 줄이는 편이 더 중요합니다.

특히 주의할 점은 "모든 것을 설명하려는 충동"입니다. 프로젝트를 만들면서 배운 것이 많을수록 글에 다 담고 싶어집니다. 하지만 한 글에 담을 주제는 하나여야 합니다. 다루고 싶은 주제가 세 개라면 글도 세 편으로 나누는 편이 각 글의 집중도와 검색 유입 모두에 유리합니다.

## 실무에서는 이렇게 본다

좋은 엔지니어링 블로그는 대체로 문제, 해결, 결과 구조를 유지합니다. 회사 기술 블로그도 마찬가지입니다. 독자가 궁금한 것은 "이 기술을 썼다"가 아니라 "왜 이 선택이 필요했고 어떤 결과가 났는가"이기 때문입니다.

개인 포트폴리오 글도 이 원리를 따르면 됩니다. 코드 전체를 다 보여 주기보다, 문제 해결의 핵심 장면을 잘라 보여 주는 편이 훨씬 설득력 있습니다.

회사 기술 블로그에서 배울 점을 구체적으로 보면:

- **토스(Toss) 기술 블로그**: 문제 상황을 독자가 공감할 수 있게 보여 주고, 해결 과정의 시행착오를 순서대로 나열합니다.
- **우아한형제들 기술 블로그**: 변경 전후 수치를 표로 제시하고, 실제 코드와 배포 결과를 스크린샷으로 보여 줍니다.
- **라인(LINE) 기술 블로그**: 대규모 시스템의 제약 조건을 먼저 설명하고, 그 안에서 최선의 선택이 뭘였는지 보여 줍니다.

이 명의 블로그 모두 "뭔가 신기한 것을 썼다"가 아니라 "어려운 문제를 어떻게 풀었다"라는 구조입니다. 개인 포트폴리오 글도 동일한 구조를 따를 수 있습니다. 규모는 작아도 구조는 같을 수 있기 때문입니다.

## 체크리스트

- [ ] 문제를 한 줄로 설명했다.
- [ ] 코드 블록 수를 꼭 필요한 수준으로 제한했다.
- [ ] 결과를 숫자나 지표로 남겼다.
- [ ] 마지막에 학습을 한 문장으로 적었다.
- [ ] 다음 글이나 관련 글로 이어지는 경로를 넣었다.

## 연습 문제

1. 여러분 프로젝트를 검색 유입형 제목으로 바꿔 한 줄 써 보세요.
2. 전체 코드 대신 글에 남길 핵심 코드 조각 하나를 골라 보세요.
3. 결과를 숫자로 적기 어렵다면 무엇을 측정하면 좋을지 적어 보세요.

## 정리와 다음 글

포트폴리오 프로젝트를 글로 바꿀 때 중요한 것은 코드 양이 아니라 맥락의 선명함입니다. 문제를 먼저 제시하고, 접근을 설명하고, 핵심 코드만 보여 주고, 결과를 숫자로 남기고, 마지막에 학습을 정리하면 저장소보다 더 넓은 독자에게 도달할 수 있습니다.

다음 글에서는 이렇게 정리한 프로젝트를 면접 자리에서 짧고 설득력 있게 설명하는 방법을 다룹니다.

### 글 본문 전체 템플릿

아래 템플릿은 프로젝트 블로그 글을 처음 쓸 때 빈 화면에서 시작하지 않도록 돕습니다. 각 섹션을 채워 넣기만 하면 구조가 잡힌 글이 완성됩니다.

```markdown
# [기술 키워드]로 [문제 동사]: [결과 수치] 달성기

## 문제
- 어떤 상황에서 어떤 불편/병목이 있었는가?
- 이 문제를 해결하지 않으면 어떤 일이 벌어지는가?

## 접근
1. 첫 번째 시도: (무엇을 해 봤고, 왜 부족했는가)
2. 방향 전환: (어떤 기준으로 다른 접근을 선택했는가)
3. 최종 설계: (핵심 아키텍처/기술 선택 한 줄 요약)

## 핵심 코드
- 전체 구현이 아닌, 문제 해결의 핵심 20줄 이내
- 코드 위/아래에 "이 코드가 하는 일" 설명 한 줄씩

## 결과
| 지표 | Before | After | 변화 |
| --- | ---: | ---: | ---: |
| (지표1) | | | |
| (지표2) | | | |

## 교훈
- 다시 적용할 수 있는 판단 1~2문장
- "다음에는 이렇게 하겠다"까지 포함하면 더 좋음

## 참고
- [저장소 링크]()
- [라이브 데모]()
```

이 템플릿의 강점은 "빈칸을 채우면 글이 된다"는 점입니다. 구조가 미리 잡혀 있으면 글쓰기 허들이 크게 낮아집니다. 첫 글에서 완벽함을 추구하기보다 이 틀을 채우는 데 집중하는 편이 더 좋은 결과를 냅니다.

### 기술 블로그 글과 개발 일지의 차이

프로젝트를 하면서 남긴 메모와 블로그 글은 목적이 다릅니다. 혼동하면 독자에게 맞지 않는 글이 됩니다.

| 구분 | 개발 일지 | 블로그 글 |
| --- | --- | --- |
| 독자 | 미래의 나 | 같은 문제를 겪을 다른 개발자 |
| 구조 | 시간순 | 문제-접근-결과 순 |
| 코드 | 전체 포함 가능 | 핵심만 발췌 |
| 맥락 | 생략 가능 (나는 안다) | 반드시 제시 (독자는 모른다) |
| 길이 | 제한 없음 | 5~10분 읽기 분량 권장 |
| 성과 | 선택적 | 반드시 수치로 제시 |

개발 일지를 블로그 글로 전환할 때는 "나만 아는 맥락"을 "독자도 이해하는 배경 설명"으로 바꾸는 작업이 핵심입니다. 일지에서 시간순으로 적은 내용을 문제-접근-결과 순으로 재배치하면 절반은 완성됩니다.

### 글 발행 전 리허설 체크

글을 발행하기 전 아래 세 단계를 거치면 품질이 눈에 띄게 올라갑니다.

1. **3분 읽기 테스트**: 글 전체를 소리 내어 읽습니다. 3분 안에 핵심 메시지가 전달되지 않으면 도입부를 줄입니다.
2. **코드 복사 테스트**: 글의 코드 블록을 복사해 실제로 실행합니다. 실행되지 않으면 독자 신뢰를 잃습니다.
3. **링크 검증**: 저장소 URL, 데모 URL, 참고 자료 링크를 모두 클릭합니다. 깨진 링크 하나가 글 전체의 인상을 떨어뜨립니다.

이 세 단계는 5분이면 끝나지만, 글의 전문성을 크게 높여 줍니다. 특히 코드 블록이 실행 가능하다는 것은 기술 블로그에서 가장 강한 신뢰 신호입니다.

### 성과 측정 지표 표 설계

블로그 글의 설득력을 높이려면 결과를 감상문이 아니라 지표로 제시해야 합니다. 지표는 복잡할 필요가 없지만, 문제와 직접 연결되어야 합니다. 아래 표는 포트폴리오 글에 자주 쓰는 기본 지표 예시입니다.

| 영역 | 지표 | 측정 방법 | 보고 형식 |
| --- | --- | --- | --- |
| 성능 | 평균 응답시간(ms) | APM/로그 집계 | 이전 대비 변화율 |
| 사용성 | 핵심 작업 완료 시간 | 사용자 시나리오 측정 | Before/After 비교 |
| 안정성 | 오류율(%) | 에러 로그 집계 | 주간 추이 |
| 도입 | 주간 활성 사용자 | 분석 도구 집계 | 누적/증감 |

지표를 고를 때 중요한 기준은 "면접에서 설명 가능한가"입니다. 수치를 제시했는데 측정 방법을 설명하지 못하면 오히려 신뢰를 잃을 수 있습니다.

### 트래픽/성능 데이터 시각화 방법

글에 데이터가 들어갈 때는 숫자 나열보다 구조화가 중요합니다. 최소한 아래 세 가지 방식 중 하나를 쓰면 가독성이 올라갑니다.

1. 기간 비교 표: 도입 전/후를 한 번에 비교
2. 핵심 그래프 1개: 응답시간 또는 오류율 추이
3. 요약 박스: "가장 큰 변화 3개"를 텍스트로 압축

```markdown
| 지표 | 도입 전 | 도입 후 | 변화 |
| --- | ---: | ---: | ---: |
| 평균 응답시간 | 280ms | 120ms | -57% |
| 오류율 | 3.2% | 0.8% | -75% |
| 주간 활성 사용자 | 12 | 31 | +158% |
```

이 형식은 독자가 한눈에 결과를 해석하게 돕습니다. 블로그 글에서 가장 중요한 것은 "읽는 사람이 빠르게 이해하는 경험"입니다.

### 사례 글 본문 구조(문제-해결-결과)

성과를 담은 글은 아래 구조를 따를 때 전달력이 높습니다.

```markdown
## 문제
팀 일정 조회가 여러 도구에 흩어져 계획 시간이 길어짐

## 접근
MVP 범위 축소, FastAPI + PostgreSQL, 주간 뷰 중심 UX

## 결과
응답시간 57% 개선, 계획 시간 40분 -> 18분, 주간 활성 사용자 31명

## 교훈
범위를 줄인 MVP가 오히려 측정 가능한 개선을 만든다.
```

이 구조의 장점은 면접 답변과도 연결된다는 사실입니다. 블로그 글과 면접 설명이 같은 서사를 공유하면 프로젝트 메시지가 훨씬 일관되게 남습니다.

### 글의 신뢰도를 높이는 증거 묶음

블로그 글에서 신뢰를 높이려면 주장-증거-해석을 함께 제시해야 합니다. 아래 묶음을 기준으로 작성하면 내용이 단단해집니다.

| 주장 | 증거 | 해석 |
| --- | --- | --- |
| 응답속도가 개선되었다 | 평균 280ms -> 120ms | 캐시 전략이 병목 구간에 효과 |
| 사용성이 좋아졌다 | 작업 완료 시간 40분 -> 18분 | 단일 화면 통합이 탐색 비용 감소 |
| 운영 안정성이 높아졌다 | 오류율 3.2% -> 0.8% | 배포 전 테스트 자동화 효과 |

증거 없는 주장은 기억되지 않고, 해석 없는 숫자는 의미가 흐립니다. 두 가지를 함께 써야 글이 사례 연구로 남습니다.

### 시각 자료 배치 원칙

1. 그래프는 본문 주장 바로 아래에 배치합니다.
2. 표 제목은 무엇을 비교하는지 명확히 씁니다.
3. 한 섹션에 그래프를 과도하게 넣지 않습니다.
4. 원본 데이터 출처를 함께 남깁니다.

이 원칙을 지키면 긴 글에서도 독자가 핵심 메시지를 놓치지 않습니다.

### 블로그 글 SEO 체크리스트

글을 발행하기 전 아래 항목을 점검하면 검색 유입을 높일 수 있습니다.

| 항목 | 확인 질문 | 예시 |
| --- | --- | --- |
| 제목 | 검색 의도와 일치하는가? | "FastAPI 인증 구현기" vs "내 프로젝트 소개" |
| 도입부 | 3문장 안에 문제와 결과가 보이는가? | "응답시간 280ms → 120ms로 개선한 방법" |
| H2 구조 | 독자가 목차만 보고 내용을 예측할 수 있는가? | "문제 / 접근 / 결과 / 교훈" |
| 링크 | 저장소와 데모 URL이 포함되었는가? | 하단 참고자료 또는 본문 중 |
| 이미지 | 스크린샷/다이어그램에 alt 텍스트가 있는가? | `![API 응답시간 추이 그래프](...)` |
| 태그 | 플랫폼 태그를 3~5개 붙였는가? | Python, FastAPI, PostgreSQL, 성능개선 |

이 체크리스트를 발행 전 1분만 확인하면 검색 노출 확률이 눈에 띄게 높아집니다.

### 콘텐츠 캐린더 템플릿

프로젝트 하나에서 여러 편의 글을 만들 수 있습니다. 아래 캘린더는 하나의 프로젝트를 4주에 걸쳐 4편의 글로 펼치는 예시입니다.

```text
Week 1: "왜 이 문제를 골랐는가" — 문제 정의와 동기
Week 2: "FastAPI + PostgreSQL로 MVP 만들기" — 기술 선택과 구현 핵심
Week 3: "응답시간 57% 줄이기" — 성능 개선 과정과 결과
Week 4: "2주 프로젝트에서 배운 것" — 회고와 다음 방향
```

이렇게 하면 하나의 프로젝트로 여러 번 노출되고, 각 글이 서로 다른 검색 의도를 잡습니다. 첫 번째 글은 문제 정의를 검색하는 독자를, 세 번째 글은 성능 개선 방법을 찾는 독자를 데려옵니다.

### 플랫폼별 발행 전략

블로그 플랫폼마다 강점이 다릅니다. 아래 표를 참고해 플랫폼 특성에 맞게 글을 조정합니다.

| 플랫폼 | 강점 | 글 조정 포인트 |
| --- | --- | --- |
| Tistory | 한국어 SEO, 네이버 유입 | 한국어 제목, 키워드 중심 |
| Hashnode | 개발자 커뮤니티, 영어권 | 영어 제목, 코드 비중 높음 |
| Medium | 일반 독자 포함, 넘은 도달 | 설명 비중, 코드 최소화 |
| dev.to | 개발자 전용, 태그 중심 발견 | 태그 3~5개, 짧은 글 |

모든 플랫폼에 똑같은 글을 올리는 것보다, 핵심 메시지는 유지하되 길이와 코드 비중을 플랫폼에 맞게 조정하는 편이 효과적입니다.

### 글 제목 작성 공식

제목은 검색 유입의 첫 관문입니다. 아래 공식을 적용하면 독자의 클릭 확률이 높아집니다.

```text
공식: [기술/도구] + [문제/동사] + [결과/수치]

예시:
- "FastAPI로 인증 API 만들기: 응답시간 120ms 달성기"
- "PostgreSQL 인덱스 튜닝으로 쿼리 속도 3배 개선한 방법"
- "GitHub Actions로 CI/CD 구축: 배포 시간 15분 → 3분"

나쁜 예시:
- "내 프로젝트 소개" (검색 의도 없음)
- "개발일지 #3" (독자 가치 불명)
- "TIL" (주제 예측 불가)
```

제목에 기술 키워드와 성과 수치가 함께 들어가면 검색 결과에서 눈에 띄고, 클릭률도 올라갑니다.

### 글 마무리 CTA(Call to Action) 템플릿

글의 마지막에는 독자가 다음에 할 수 있는 행동을 제시합니다.

```markdown
---
## 마무리

이 글이 도움이 됐다면:
- 저장소에서 전체 코드를 확인해 보세요: [GitHub](https://github.com/...)
- 다음 글에서는 "면접에서 이 프로젝트 설명하기"를 다룹니다.
- 질문이나 피드백은 [이슈 택](https://github.com/.../issues)으로 남겨 주세요.
```

CTA가 있으면 글이 일방적 정보 전달이 아니라 대화의 시작점이 됩니다. 독자가 저장소를 방문하거나 이슈를 남기면, 그 자체가 포트폴리오의 활동성을 보여 주는 증거가 됩니다.

### 글 발행 후 추적 지표

글을 발행한 뒤에는 최소한 다음 세 가지를 2주 간 추적합니다.

| 지표 | 측정 방법 | 목표 기준 |
| --- | --- | --- |
| 조회수 | 플랫폼 통계 | 발행 2주 내 100회 이상 |
| 저장소 유입 | GitHub referrer 통계 | 글 발행 후 저장소 방문 증가 |
| 독자 반응 | 댓글/좋아요/공유 | 질문이나 피드백 1건 이상 |

이 지표를 보면 "제목이 문제인지, 내용이 문제인지, 플랫폼이 문제인지" 판단할 수 있습니다. 조회수는 높은데 저장소 유입이 없으면 CTA가 약한 것이고, 조회수 자체가 낮으면 제목이나 발행 타이밍을 재검토합니다.

## 처음 질문으로 돌아가기

- **프로젝트를 블로그 글로 바꿀 때 왜 문제 중심 구조가 필요할까요?**
  - 코드 나열은 독자에게 맥락을 주지 못합니다. "어떤 문제를 왜 이렇게 풀었는가"를 먼저 제시해야 독자가 코드를 보는 이유가 생깁니다.
- **코드 발췌와 저장소 링크는 어떻게 역할을 나누면 좋을까요?**
  - 글 본문에는 설명에 꼭 필요한 핵심 코드만 남기고, 전체 구현은 저장소 링크로 내보냅니다. 이렇게 하면 글의 흐름이 끊기지 않으면서도 독자가 전체 코드를 확인할 수 있습니다.
- **결과를 숫자로 남기는 일이 왜 글의 설득력을 키울까요?**
  - "성능이 좋아졌다"는 주관적 표현은 기억에 남지 않습니다. "280ms에서 120ms로 개선"처럼 수치가 있으면 결과가 구체적으로 전달되고, 글의 신뢰도가 높아집니다.
