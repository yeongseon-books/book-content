# 프롬프트 엔지니어링 기초 — AI에게 원하는 답을 얻는 기술

같은 모델을 써도 누구는 실무에 바로 쓰는 결과를 얻고, 누구는 몇 번 묻다가 창을 닫습니다. 차이는 대개 모델 자체보다 요청을 어떻게 구조화했는지에서 갈립니다. AI는 마음을 읽지 않으므로, 맥락과 출력 계약을 개발자가 먼저 정리해 줘야 합니다.

이 글은 AI 웹 개발 입문 시리즈의 2번째 글입니다.

여기서는 AI에게 원하는 답을 더 안정적으로 얻기 위한 프롬프트 설계 원칙을 실무 감각으로 정리합니다.

## 이 글에서 다룰 문제

- 프롬프트는 단순한 질문과 무엇이 다를까요?
- `system`과 `user` 역할은 각각 어떤 책임을 가질까요?
- 좋은 프롬프트는 어떤 정보를 빠뜨리지 않을까요?
- `temperature`, `max tokens` 같은 파라미터는 언제 조절해야 할까요?
- 원하는 답이 안 나올 때는 무엇부터 고쳐야 할까요?

> 프롬프트는 모델을 설득하는 문장이 아니라, 모델이 따라야 할 작업 계약입니다. 그래서 좋은 프롬프트는 멋진 표현보다 역할, 맥락, 출력 형식, 제약 조건이 먼저 분명합니다.

## 프롬프트를 왜 따로 배워야 할까

지난 글에서는 API를 호출하는 법을 살펴봤습니다. 이제부터는 “어떻게 호출할 것인가”가 중요해집니다. 같은 모델이라도 요청이 모호하면 평균적인 답을 내놓고, 요청이 구체적이면 훨씬 목적에 맞는 답을 돌려줍니다.

프롬프트를 어렵게 생각할 필요는 없습니다. 새 팀원에게 업무를 맡긴다고 생각하면 됩니다. “보고서 좀 써줘”라고만 말하면 주제, 분량, 독자, 형식이 비어 있습니다. AI도 마찬가지입니다. 배경 지식과 기대 결과를 주지 않으면 가장 무난한 쪽으로 대답할 뿐입니다.

프롬프트 엔지니어링은 결국 모델이 읽어야 할 문맥을 설계하는 작업입니다. 모델 성능을 바꾸지 못하더라도, 입력 구조를 바꾸면 결과 품질은 꽤 크게 달라집니다.

## System Prompt와 User Prompt

OpenAI의 Chat Completions API에서는 `role` 필드로 메시지 성격을 구분합니다. 입문 단계에서 가장 중요한 것은 `system`과 `user` 두 가지입니다.

- System Prompt: 모델의 기본 역할, 태도, 전문성, 금지 사항을 정합니다.
- User Prompt: 지금 수행해야 할 구체적인 작업과 입력값을 전달합니다.

예를 들어 `system`에는 “너는 보안과 성능을 중시하는 시니어 개발자다” 같은 기본 규칙을 두고, `user`에는 “이 함수를 리뷰해 달라”는 현재 작업을 넣습니다. 이렇게 나누면 모델의 기본 자세와 개별 요청을 분리해서 관리할 수 있습니다.

사람에게도 “전문가 관점에서 답해 달라”는 요청과 “이 코드의 문제점을 짚어 달라”는 요청은 역할이 다릅니다. 모델도 비슷합니다. `system`이 장기 규칙이라면 `user`는 단기 작업 지시라고 이해하면 됩니다.

## 좋은 프롬프트의 네 가지 원칙

입문 단계에서는 아래 네 가지를 먼저 지키는 것만으로도 결과가 꽤 안정됩니다.

### 1. 구체적으로 말하기

“블로그 글 써줘”보다 “초보 개발자를 위한 리액트 상태 관리 글을 3문단 내외로 써줘”가 낫습니다. 주제, 대상, 길이, 목적이 분명할수록 모델이 흔들릴 여지가 줄어듭니다.

### 2. 예시를 주기

말로 설명하기 어려운 출력 형태는 예시를 보여주는 편이 더 빠릅니다. 이를 Few-shot 프롬프팅이라고 부릅니다. 예시가 없으면 Zero-shot, 하나면 One-shot입니다. 특히 문체나 JSON 형태처럼 형식이 중요한 작업에서 효과가 큽니다.

### 3. 출력 형식을 지정하기

“JSON으로 달라”, “마크다운 표로 정리해 달라”, “불릿 포인트 3개로 달라” 같은 제약은 후속 처리 비용을 크게 줄여 줍니다. 서비스 코드가 결과를 다시 파싱해야 한다면 이 부분은 선택이 아니라 계약에 가깝습니다.

### 4. 제약 사항을 명시하기

“한국어로만 답해 달라”, “비속어를 쓰지 말라”, “전문 용어는 최소화하라” 같은 제한을 먼저 주면 엉뚱한 방향으로 벗어나는 일을 줄일 수 있습니다.

## 생성 성격을 바꾸는 파라미터

프롬프트 문장만큼 중요한 것이 생성 파라미터입니다. 특히 `temperature`와 `max tokens`는 결과의 성격을 크게 바꿉니다.

- Temperature: 0에 가까울수록 더 일관되고 보수적인 답을 내고, 값이 올라갈수록 더 다양한 답을 시도합니다.
  - 코드 생성이나 정보 추출: 0.1 ~ 0.3 권장
  - 소설 쓰기나 아이디어 브레인스토밍: 0.7 ~ 1.0 권장
- Max Tokens: 출력 길이의 상한입니다. 너무 짧게 잡으면 답이 중간에 끊길 수 있습니다.

이 설정은 “정답”이 정해져 있다기보다 작업 성격에 맞게 조절하는 값입니다. 코드 리뷰처럼 안정성이 중요한 작업과, 카피 초안처럼 다양성이 필요한 작업을 같은 설정으로 다룰 필요는 없습니다.

## 실습 1: 상품 설명 자동 생성기

마케팅 문구 생성처럼 톤과 형식이 중요한 작업은 프롬프트 차이가 특히 크게 드러납니다.

System Prompt:
> 너는 유능한 이커머스 카피라이터야. 상품의 이름과 특징을 주면, 구매 욕구를 자극하는 감성적인 문구 3개를 만들어줘. 각 문구는 2문장 이내로 짧고 강렬하게 작성해.

User Prompt:
> 상품명: 무소음 기계식 키보드
> 특징: 밤샘 작업 가능, 부드러운 타건감, 파스텔 블루 컬러

이 예제에서 배울 점은 단순합니다. 모델에게 “카피 써줘”라고만 말하지 않고, 역할·입력 필드·출력 개수·문장 길이까지 함께 전달했다는 점입니다.

## 실습 2: 코드 리뷰어 만들기

개발 생산성과 더 가까운 예제로 바꿔 보겠습니다. 이번에는 모델에게 리뷰어 역할과 평가 기준을 함께 줍니다.

System Prompt:
> 너는 보안과 성능을 중시하는 구글 출신 시니어 개발자야. 전달받은 코드를 분석해서 1) 잠재적 버그, 2) 성능 개선점, 3) 가독성 향상 방안을 불렛 포인트로 정리해줘. 칭찬보다는 냉철한 비판 위주로 답변해.

User Prompt:
```javascript
function login(user, pass) {
  if (user == 'admin' && pass == '1234') {
    return true;
  }
  return false;
}
```

이 프롬프트가 유용한 이유는 출력 평가 기준이 이미 포함되어 있기 때문입니다. 잠재적 버그, 성능, 가독성이라는 세 축을 먼저 정해 두면 모델 응답도 그 틀 안에서 조직되기 쉽습니다.

## 프롬프트 디버깅 체크리스트

원하는 답이 나오지 않을 때는 모델을 탓하기 전에 입력을 먼저 점검하는 편이 좋습니다.

1. 배경 지식이 부족한가? AI가 모를 법한 고유 명사나 상황 설명을 더 추가합니다.
2. 부정문만 잔뜩 넣지 않았는가? “하지 마”만 반복하기보다 “이렇게 해 달라”는 형태가 더 안정적일 때가 많습니다.
3. 단계별 사고가 필요한 작업인가? 복잡한 문제라면 순서를 나눠 생각하게 만드는 편이 낫습니다.
4. 예시가 실제 기대 출력과 닮아 있는가? Few-shot의 품질은 예시의 적절성에 크게 좌우됩니다.

실전에서는 프롬프트를 한 번에 완성하기보다, 실패 사례를 보고 반복해서 다듬는 과정이 더 일반적입니다. 그래서 프롬프트 엔지니어링은 글쓰기 감각만이 아니라 디버깅 감각에 더 가깝습니다.

## 체크리스트

- [ ] `system`과 `user` 역할 차이를 설명할 수 있다.
- [ ] 출력 형식과 제약 조건을 프롬프트에 명시했다.
- [ ] 작업 성격에 맞게 `temperature`를 조절할 수 있다.
- [ ] 실패 사례를 보고 프롬프트를 다시 고칠 기준이 있다.

## 정리

프롬프트 엔지니어링은 AI에게 예쁘게 말하는 기술이 아니라, 모델이 따라야 할 작업 계약을 설계하는 기술입니다.

- `system`은 장기 규칙, `user`는 현재 작업 지시라는 구분을 먼저 잡습니다.
- 좋은 프롬프트는 구체성, 예시, 출력 형식, 제약 조건을 빠뜨리지 않습니다.
- `temperature`와 `max tokens`는 작업 성격에 맞게 조절해야 합니다.
- 결과가 흔들릴수록 프롬프트를 디버깅 대상으로 보고 반복해서 개선해야 합니다.

다음 글에서는 이 원칙을 브라우저 UI와 연결해, 사용자가 실제로 대화할 수 있는 챗봇 형태로 구현해 보겠습니다.
