# Serverless란 무엇인가?

## 이 글에서 다룰 문제

- Serverless는 정말 서버가 없다는 뜻일까요?
- FaaS와 Serverless는 같은 말일까요, 아니면 다른 층위의 개념일까요?
- 사용량 기반 과금은 왜 비용 절감으로 바로 이어지지 않을 때가 있을까요?
- 어떤 워크로드는 Serverless에 잘 맞고, 어떤 워크로드는 처음부터 다른 선택지가 더 나을까요?

> Serverless 101 시리즈 (1/10)

처음 Serverless를 접하면 대개 이렇게 이해합니다. “서버를 안 만져도 되고, 호출이 있을 때만 돈을 내면 되겠네.” 방향은 맞습니다. 다만 여기서 멈추면 실무에서 금방 헷갈립니다. 서버를 직접 운영하지 않는다고 해서 서버가 사라지는 것은 아니고, 함수 하나만 올리면 모든 시스템 문제가 단순해지는 것도 아닙니다.

Serverless를 제대로 이해하려면 단어보다 책임 이동을 먼저 봐야 합니다. 누가 서버를 만들고, 누가 패치하고, 누가 스케일링하고, 누가 장애를 흡수하는지가 바뀌는 모델입니다. 이 관점을 잡아 두면 비용, 성능, 콜드 스타트, 상태 관리 같은 뒤의 주제도 한 줄로 연결됩니다.

이 글은 Serverless 101 시리즈의 출발점입니다. 여기서는 정의를 외우는 대신, Serverless를 어떤 멘탈 모델로 이해해야 하는지, 왜 작은 팀에 특히 유리한지, 그리고 어디까지 기대해야 하는지를 차근차근 정리하겠습니다.

## 이 글에서 배울 것

- Serverless의 정확한 뜻과 FaaS와의 관계
- 사용량 기반 과금이 동작하는 방식
- 플랫폼으로 넘어가는 운영 책임의 범위
- 잘 맞는 워크로드와 잘 맞지 않는 워크로드를 구분하는 기준

## 왜 중요한가

Serverless의 가장 큰 장점은 생산성입니다. 팀이 서버 패치, 용량 예측, 기본 스케일링 작업에 쓰던 시간을 줄이고 제품 기능과 사용자 경험에 더 집중할 수 있기 때문입니다. 특히 인프라 전담 인력이 없는 작은 팀에서는 이 차이가 훨씬 크게 느껴집니다.

하지만 장점만 보고 접근하면 곤란합니다. 호출 수만 적으면 무조건 싸다고 생각하거나, 함수만 쪼개면 아키텍처가 자동으로 좋아진다고 생각하면 실제 운영에서 바로 벽을 만납니다. 비용은 호출 수만이 아니라 실행 시간과 리소스 크기에 따라 달라지고, 함수가 짧다고 해서 분산 시스템의 복잡성이 사라지지도 않습니다.

Serverless를 올바르게 이해하는 일은 기술 선택의 기준을 세우는 일과 같습니다. 무엇을 플랫폼에 맡기고, 무엇은 여전히 우리 설계 책임으로 남는지 구분할 수 있어야 이후의 모든 판단이 흔들리지 않습니다.

## 한눈에 보는 흐름

```mermaid
flowchart LR
    Event["이벤트"] --> Platform["서버리스 플랫폼"]
    Platform --> Func["함수"]
    Func --> Result["응답 / 부수 효과"]
```

> Serverless는 서버가 없는 모델이 아니라, 서버 운영 책임을 플랫폼이 대신 가져가는 실행 모델입니다.

이 그림을 머릿속에 두면 출발이 쉬워집니다. 어떤 신호가 들어오면 플랫폼이 함수를 깨우고, 함수는 짧은 시간 동안 코드를 실행한 뒤 응답이나 부수 효과를 남깁니다. 여기서 중요한 주체는 함수가 아니라 플랫폼입니다. 함수가 언제 깨어나고 얼마나 오래 살아 있고 어떤 자원을 받는지는 대부분 플랫폼 규칙이 결정합니다.

## 핵심 용어

- **Serverless**: 서버 운영 책임을 플랫폼에 위임하는 모델입니다.
- **FaaS**: 함수를 실행 단위로 다루는 환경입니다. Serverless의 대표 구현 방식 가운데 하나입니다.
- **이벤트 소스**: 함수를 깨우는 신호입니다. HTTP 요청, 큐 메시지, 파일 업로드, 스케줄 등이 여기에 들어갑니다.
- **수명(lifetime)**: 함수 인스턴스가 살아 있는 짧은 실행 구간입니다.
- **사용량 기반 과금**: 프로비저닝된 서버 시간 대신 호출, 실행 시간, 자원 사용량 기준으로 비용을 계산하는 방식입니다.

여기서 가장 자주 생기는 오해는 Serverless와 FaaS를 완전히 같은 말로 쓰는 경우입니다. 실무에서는 거의 겹쳐 보일 수 있지만, 엄밀하게는 Serverless가 더 큰 개념이고 FaaS는 그 안의 실행 모델이라고 보는 편이 정확합니다. 관리형 데이터베이스, 이벤트 버스, 인증 서비스처럼 서버를 의식하지 않는 다른 관리형 서비스도 넓게 보면 Serverless 생태계 안에 들어갑니다.

## Before / After

**Before**: 24시간 켜 둔 서버를 기준으로 용량을 잡고, 트래픽이 0이어도 기본 비용과 운영 부담이 계속 남습니다.

**After**: 호출이 들어올 때만 함수를 실행하고, 플랫폼이 기본 스케일링과 서버 운영을 맡습니다. 개발자는 프로비저닝보다 이벤트와 함수 경계에 집중하게 됩니다.

이 변화는 단순히 비용 모델만 바꾸지 않습니다. 서버를 늘릴지 줄일지 고민하는 방식이 이벤트를 어떻게 나누고, 함수를 얼마나 짧게 만들고, 상태를 어디에 둘지 고민하는 방식으로 바뀝니다. 즉, 인프라 계획 중심 사고가 실행 경계 중심 사고로 이동합니다.

## 실습: 가장 작은 함수부터 보기

### 1단계 — Python 함수 작성

```python
def handler(event, context):
    name = event.get("name", "world")
    return {"message": f"hello, {name}"}
```

가장 작은 형태의 함수입니다. 입력은 `event`, 실행 환경 정보는 `context`, 결과는 직렬화 가능한 값으로 돌려줍니다. 플랫폼이 달라도 이 패턴은 꽤 비슷하게 반복됩니다.

### 2단계 — 로컬에서 호출 시뮬레이션

```python
def invoke_local(handler, event):
    return handler(event, context=None)

print(invoke_local(handler, {"name": "Alice"}))
```

Serverless를 처음 배울 때 중요한 습관 하나는 함수 본체를 플랫폼과 분리해서 생각하는 것입니다. 함수는 결국 입력을 받아 결과를 내는 코드이므로, 로컬에서도 비슷한 방식으로 충분히 감각을 익힐 수 있습니다.

### 3단계 — 이벤트 형태 익히기

```python
http_event = {"path": "/hello", "method": "GET", "name": "Bob"}
queue_event = {"records": [{"body": "msg-1"}, {"body": "msg-2"}]}
```

Serverless 함수는 함수 이름보다 이벤트 모양에 더 크게 좌우됩니다. HTTP 요청을 받을 때와 큐 메시지를 받을 때의 입력 구조가 다르므로, 실무에서는 코드보다 먼저 이벤트 스키마를 이해해야 합니다.

### 4단계 — 타임아웃 인지하기

```python
import time

def slow_handler(event, context):
    time.sleep(0.1)
    return {"ok": True}
```

함수는 보통 짧게 살아 있습니다. 그래서 오래 걸리는 작업, 외부 API 대기, 대용량 변환 같은 일은 타임아웃과 바로 연결됩니다. Serverless에서는 “언젠가 끝나겠지”라는 태도가 가장 위험합니다.

### 5단계 — 결과 형태 표준화

```python
def http_response(status, body):
    return {"statusCode": status, "body": body}
```

입문 단계에서는 작은 헬퍼가 별것 아닌 것처럼 보이지만, 응답 형식을 표준화해 두면 함수가 늘어날수록 효과가 커집니다. 특히 API 성격의 함수는 예외 처리와 응답 구조를 일찍 통일하는 편이 운영에 유리합니다.

## 이 코드에서 주목할 점

- `event`와 `context`라는 공통 서명이 반복됩니다.
- 함수는 짧고 결정적으로 작성할수록 운영이 쉬워집니다.
- 상태는 함수 안에 붙잡아 두지 말고 외부 저장소로 빼야 합니다.

Serverless 함수는 길게 끌고 가는 프로세스보다 한 번 호출되어 한 가지 일을 끝내는 작업에 가깝습니다. 이 특성을 받아들이면 코드 구조도 자연스럽게 바뀝니다. 복잡한 일을 한 함수에 몰아넣기보다 입력 검증, 처리, 저장, 후속 이벤트 발행처럼 책임을 나누는 편이 더 잘 맞습니다.

## 자주 하는 실수 5가지

1. 장시간 실행 작업을 함수 하나로 만들기
2. 로컬 파일이나 메모리에 상태를 저장하기
3. 콜드 스타트를 무시한 채 응답 시간을 설계하기
4. 비용을 호출 수만 보고 추정하기
5. 로그와 메트릭 없이 운영을 시작하기

이 다섯 가지는 모두 같은 오해에서 출발합니다. Serverless를 단순히 “싸고 편한 서버”로 생각하는 오해입니다. 실제로는 실행 시간 제한, 상태 외부화, 재시도, 관측성, 비용 모델을 함께 봐야 하는 별도 운영 모델입니다.

## 실무에서는 이렇게 보입니다

Serverless는 이벤트 처리, ETL, 가벼운 API 백엔드, 스케줄 작업처럼 짧고 독립적인 단위를 빠르게 처리하는 데 잘 맞습니다. 반대로 긴 연결을 오래 유지해야 하는 작업, 메모리를 오래 붙잡는 계산, 세밀한 런타임 제어가 필요한 시스템은 맞지 않을 수 있습니다.

예를 들어 사용자 업로드를 받아 썸네일을 만들고 저장한 뒤 알림을 보내는 흐름은 Serverless와 잘 맞습니다. 반면 수십 분 이상 CPU를 점유하는 대형 배치나 매우 긴 세션을 유지해야 하는 워크로드는 처음부터 다른 실행 환경이 더 단순할 수 있습니다.

## 실무에서는 이렇게 생각합니다

- Serverless는 기본값이 아니라 선택지입니다.
- 비용은 호출 수 + 실행 시간 + 자원 크기를 함께 봐야 합니다.
- 상태는 외부 저장소에 두는 것이 출발점입니다.
- 콜드 스타트는 피할 대상이 아니라 설계 변수입니다.
- 디버깅의 절반은 관측성에서 시작합니다.

숙련된 엔지니어는 “함수를 몇 개 만들까”보다 “어떤 경계에서 깨우고, 어디까지를 한 번의 호출로 볼까”를 먼저 고민합니다. Serverless의 품질은 함수 개수보다 경계 설계에서 더 크게 갈립니다.

## 체크리스트

- [ ] 함수가 짧고 결정적으로 끝나는가
- [ ] 상태를 외부 저장소로 분리했는가
- [ ] 비용 모델을 호출 수 외 요소까지 검토했는가
- [ ] 로그와 메트릭을 준비했는가

## 정리 및 다음 글

Serverless의 핵심은 서버 삭제가 아니라 책임 이동입니다. 플랫폼이 서버 운영을 대신 맡는 대신, 우리는 함수 경계, 이벤트 구조, 상태 위치, 관측성을 더 분명하게 설계해야 합니다. 이 관점을 잡고 나면 이후 주제인 FaaS, Trigger, Cold Start, 상태 관리가 하나의 흐름으로 이어집니다.

다음 글에서는 FaaS의 구조와 사용 패턴을 더 구체적으로 살펴보겠습니다.
