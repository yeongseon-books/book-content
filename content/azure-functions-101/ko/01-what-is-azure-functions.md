# Azure Functions 101 (1/7): Azure Functions란? — 이벤트가 함수를 호출하는 세상

서버리스라는 말을 처음 들었을 때 가장 흔한 반응은 둘 중 하나입니다. “서버가 없다는 게 무슨 뜻이야?” 아니면 “결국 클라우드가 서버를 돌리는 건데 그게 그거 아닌가?” 둘 다 절반은 맞고 절반은 틀립니다. 서버는 분명히 있습니다. 다만 개발자와 운영자가 매번 그 서버를 직접 의식하지 않도록 추상화한 실행 모델이 서버리스이고, Azure Functions는 그 모델을 Azure에서 가장 직접적으로 보여 주는 서비스입니다.

입문 단계에서 많은 분이 Azure Functions를 “HTTP 함수 몇 개 빠르게 띄우는 서비스” 정도로 이해합니다. 그렇게 시작해도 처음에는 큰 문제가 없습니다. 하지만 조금만 운영 쪽으로 들어가면 질문이 곧 달라집니다. 왜 어떤 함수는 호출이 없을 때 완전히 사라졌다가 다시 뜨는지, 왜 첫 호출이 느릴 수 있는지, 왜 트리거와 바인딩이라는 개념이 일반 웹 앱보다 더 중요해지는지 같은 질문이 따라옵니다.

이 글은 Azure Functions 101 시리즈의 첫 번째 글입니다. 여기서는 Azure Functions를 단순한 제품명이 아니라 **이벤트가 코드를 깨우고, 일이 끝나면 실행 환경이 다시 회수될 수 있는 플랫폼 모델**로 이해해 보겠습니다. 뒤의 트리거, 바인딩, Host/Worker, 플랜, 스케일링 이야기가 모두 이 출발점 위에 올라가기 때문입니다.

특히 이 장에서 잡아야 할 핵심 감각은 하나입니다. **Azure Functions의 주인공은 함수 코드만이 아니라, 그 코드를 깨우는 이벤트와 그 실행을 둘러싼 플랫폼 계약 전체**라는 점입니다. 이 관점을 먼저 잡아두면 뒤에서 나오는 운영 판단이 훨씬 쉬워집니다.

이제 “서버리스니까 그냥 편하다”는 설명 대신, 어떤 사건이 어떤 식으로 코드를 호출하고 비용과 성능까지 바꾸는지를 차근차근 보겠습니다.

*Azure Functions 101 1장 흐름 개요*
> Azure Functions란? — 이벤트가 함수를 호출하는 세상의 핵심은 기능 이름이 아니라, 어떤 경계에서 무엇을 검증하고 어떤 신호를 남길지 정하는 데 있습니다.

## 먼저 던지는 질문

- Azure Functions는 정확히 어떤 서버리스 컴퓨트 모델로 이해해야 할까요?
- 실행 횟수와 실행 시간 기준 과금은 언제 유리하고 언제 오히려 불리해질까요?
- Azure Functions는 이벤트 기반 아키텍처에서 어떤 위치를 차지할까요?

## 왜 이 글이 중요한가

Azure Functions를 처음 배우는 단계에서는 코드 몇 줄로 HTTP 엔드포인트가 생긴다는 사실이 가장 먼저 눈에 들어옵니다. 하지만 실무에서는 그 편리함보다 **실행 모델을 잘못 이해했을 때 생기는 판단 오류**가 더 크게 돌아옵니다. 상시 실행 서버처럼 가정하고 설계했다가 콜드 스타트와 인스턴스 재활용을 뒤늦게 문제로 만나는 경우가 대표적입니다.

또한 Azure Functions는 단순히 “싼 API 호스팅”이 아닙니다. 큐 메시지, Blob 업로드, 타이머, 변경 피드처럼 다양한 이벤트를 기준으로 코드를 실행하는 서비스이기 때문에, 입출력 모델과 운영 모델이 일반 웹 앱과 다릅니다. 이 차이를 이해하지 못하면 플랜 선택, 재시도 설계, 멱등성, 비용 예측 같은 후속 주제가 전부 흐릿해집니다.

무엇보다 이 글은 시리즈 전체의 기준점을 만듭니다. 뒤에서 트리거와 바인딩을 배울 때도, Host와 Worker의 분리를 볼 때도, 스케일링과 모니터링을 다룰 때도 결국 다시 돌아오는 질문은 같습니다. **“어떤 이벤트가 코드를 깨우고, 그 실행을 플랫폼이 어디까지 대신해 주는가?”** 이 질문을 선명하게 만드는 것이 첫 글의 역할입니다.

## 핵심 관점

Azure Functions를 가장 짧고 정확하게 설명하면 이렇습니다. **어떤 이벤트가 발생하면, 그 이벤트에 연결된 작은 함수가 호출되어 실행되고, 작업이 끝나면 실행 환경은 다시 회수되거나 유휴 상태로 돌아갈 수 있습니다.** 이 한 문장 안에 이벤트, 함수, 실행 환경, 비용 모델, 콜드 스타트의 출발점이 모두 들어 있습니다.

전통적인 웹 앱이 “항상 문을 열고 손님을 기다리는 식당”에 가깝다면, Azure Functions는 “호출이 들어왔을 때만 움직이는 출장 요리사”에 더 가깝습니다. 호출이 없을 때 계속 가게를 밝히고 인력을 상주시킬 필요가 없는 워크로드라면 이 모델은 매우 경제적입니다. 반대로 손님이 24시간 끊임없이 몰리는 구조라면, 이 모델의 장점이 줄어들고 다른 컴퓨트가 더 단순해질 수 있습니다.

이 멘탈 모델이 중요한 이유는 Azure Functions를 “코드 배포 대상”이 아니라 “이벤트와 실행 수명주기를 다루는 플랫폼”으로 보게 만들기 때문입니다. 그래야 트리거를 올바르게 고르고, 바인딩이 줄여 주는 코드와 숨기는 책임을 구분하고, 스케일링과 콜드 스타트를 자연스럽게 이해할 수 있습니다.

> Azure Functions의 본질은 “서버가 없다”가 아니라 “서버를 계속 의식하지 않아도 되는 이벤트 호출형 실행 모델”입니다. 운영 판단은 항상 이 문장에서 출발하는 편이 안전합니다.

## 핵심 개념

### 한 문장 정의부터 정확히 잡습니다

Azure Functions를 한 줄로 요약하면 다음과 같습니다.

> 어떤 이벤트가 발생하면, 그 이벤트에 연결된 작은 함수가 호출되어 실행되고, 작업이 끝나면 인스턴스는 회수됩니다.

이 문장을 세 단어로 다시 풀면 아래와 같습니다.

- 이벤트라는 축 — HTTP 요청, 큐 메시지, 파일 업로드, 타이머, 데이터베이스 변경 등 함수를 깨우는 사건입니다.
- 함수라는 축 — 여러분이 작성하는 코드입니다. 보통 작고 단일 책임으로 유지하는 편이 좋습니다.
- **실행 환경 회수** — 인스턴스가 유휴 시에는 아예 떠 있지 않을 수 있고, 트래픽이 늘면 인스턴스 수도 함께 늘거나 줄 수 있습니다.

이 관점을 먼저 잡아두면 Azure Functions가 “항상 켜져 있는 웹 서버”가 아니라는 사실이 분명해집니다. 이후의 비용, 플랜, 콜드 스타트 설명도 모두 여기서 자연스럽게 이어집니다.

### 가장 작은 예시부터 봅니다

짧은 코드가 긴 설명보다 빠를 때가 많습니다. 아래는 Azure Functions Python v2 프로그래밍 모델에서 HTTP 요청을 받아 인사말을 돌려주는 가장 단순한 함수입니다.

```python
import azure.functions as func

app = func.FunctionApp(http_auth_level=func.AuthLevel.ANONYMOUS)

@app.function_name(name="hello")
@app.route(route="hello", methods=["GET"])
def hello(request: func.HttpRequest) -> func.HttpResponse:
    name = request.params.get("name", "world")
    return func.HttpResponse(f"Hello, {name}!")
```

이 코드가 중요한 이유는 서버를 직접 시작하는 코드, 포트를 여는 코드, 라우팅 서버를 구성하는 코드가 보이지 않기 때문입니다. 여러분은 **무슨 이벤트가 이 함수를 깨우는지**와 **깨어났을 때 무엇을 할지**만 적습니다. 나머지는 Functions Host와 플랫폼이 맡습니다.

이 그림은 시리즈 전체의 출발점입니다. 뒤로 갈수록 여기에 HTTP 외의 트리거, Host와 Worker 경계, 스케일아웃, 콜드 스타트 같은 설명이 덧붙습니다.

### 기존 웹 앱과의 차이는 실행 단위와 과금 단위에 있습니다

App Service나 VM 기반 웹 앱과 비교하면 Azure Functions의 차이가 훨씬 또렷해집니다.

| 항목 | 상시 실행형 (App Service / VM) | 이벤트 호출형 (Functions) |
|---|---|---|
| 배포 단위 | 애플리케이션 (전체) | 함수 단위 (Function App에 모음) |
| 실행 시점 | 항상 켜져 있음 | 이벤트가 들어올 때만 |
| 스케일 단위 | 인스턴스(VM) 수 | 함수 실행 동시성 |
| 과금 모델 | 시간 단위 (인스턴스 띄워 둔 시간) | 실행 시간 + 실행 횟수 (또는 혼합) |

여기서 특히 중요한 차이는 **무엇을 비용의 단위로 삼는가**입니다. Azure Functions의 Consumption 계열은 호출이 적고 불규칙할수록 유리한 경우가 많습니다. 반대로 호출이 24시간 꾸준히 높다면, “켜 둔 시간”에 과금되는 App Service나 컨테이너가 더 단순하고 더 저렴할 수도 있습니다.

![상시 실행형과 이벤트 호출형의 차이](https://yeongseon-books.github.io/book-public-assets/assets/azure-functions-101/01/01-02-how-is-this-different-from-a-traditional.ko.png)

*상시 실행형과 이벤트 호출형의 차이*

오른쪽의 “유휴 → 깨우기 → 실행 → 다시 유휴” 순환이 서버리스의 핵심입니다. 그리고 바로 이 “깨우기” 구간이 나중에 콜드 스타트라는 운영 주제로 이어집니다.

이 차이를 비용으로만 읽으면 절반만 이해한 셈입니다. 더 중요한 차이는 시스템이 기본적으로 무엇을 기다리는가에 있습니다. 상시 실행형 앱은 프로세스가 이미 떠 있다는 전제를 활용하고, Functions는 요청이 왔을 때 실행 환경을 붙이거나 기존 warm 인스턴스를 재사용합니다. 같은 API 제공이라도 운영 감각이 달라지는 이유가 여기에 있습니다.

또한 App Service나 VM에서는 애플리케이션 전체를 하나의 런타임 덩어리로 보고 배포와 스케일을 생각하는 경우가 많습니다. Azure Functions에서는 Function App 단위로 배포하되, 실제 실행은 함수 호출과 인스턴스 수명주기를 중심으로 읽게 됩니다. 이 차이는 뒤의 배포, 스케일링, 모니터링 장에서 계속 다시 나타납니다.

### Azure Functions의 네 가지 핵심 단어를 먼저 익혀 둡니다

이 시리즈 전체를 관통하는 단어는 네 개입니다.

- **Trigger (트리거)** — 함수를 깨우는 이벤트의 종류입니다. 함수 하나에는 정확히 하나의 트리거가 묶입니다.
- **Binding (바인딩)** — 함수의 입력과 출력을 외부 리소스에 선언적으로 연결하는 장치입니다.
- **Host** — 트리거를 감지하고 함수를 로드하고 호출하는 런타임 프로세스입니다.
- **Function App** — 배포, 과금, 스케일링의 단위입니다. 함수는 이 단위로 묶여 배포됩니다.

아래 그림은 이 관계를 한눈에 보여 줍니다.

![트리거와 바인딩, Host, 앱의 관계](https://yeongseon-books.github.io/book-public-assets/assets/azure-functions-101/01/01-03-the-four-core-concepts-of-azure-function.ko.png)

*트리거와 바인딩, Host, 앱의 관계*

이 그림에서 먼저 가져가야 할 포인트는 두 가지입니다. 첫째, Host는 함수를 호출하는 공통 런타임이라는 점입니다. 둘째, 트리거와 바인딩은 함수와 바깥세계를 연결하는 표면이라는 점입니다. 나중에 Python, Node.js, Java 함수가 실제로는 별도 Worker 프로세스에서 실행된다는 설명도 바로 여기에서 이어집니다.

### HTTP만 보면 반만 본 셈입니다

앞의 Hello 예제만 보면 Azure Functions를 “작은 웹 API 프레임워크”로 오해하기 쉽습니다. 하지만 같은 프로그래밍 모델은 HTTP가 아닌 이벤트에도 거의 같은 표면으로 적용됩니다. 아래 예제는 주문 메시지가 Queue에 들어왔을 때 동작하는 가장 작은 형태의 큐 트리거 함수입니다.

```python
import logging
import azure.functions as func

app = func.FunctionApp()

@app.function_name(name="order_received")
@app.queue_trigger(
    arg_name="msg",
    queue_name="orders-incoming",
    connection="StorageConnection",
)
def order_received(msg: func.QueueMessage) -> None:
    payload = msg.get_json()
    logging.info("order_id=%s total=%s", payload["order_id"], payload["total"])
```

```json
{
  "order_id": "ord-20260515-001",
  "total": 12900,
  "currency": "KRW"
}
```

같은 함수 앱 안에서도 “무엇이 이 함수를 깨우는가”만 HTTP 요청에서 큐 메시지로 바뀌었을 뿐, 코드 표면은 크게 달라지지 않습니다. 이 점이 Azure Functions를 단순한 웹 API 호스팅이 아니라 **이벤트 처리 플랫폼**으로 봐야 하는 이유입니다. 나중에 트리거 장에서 더 자세히 보겠지만, 첫 장에서도 이 정도 예시는 직접 보고 넘어가는 편이 훨씬 이해가 빠릅니다.

### 어떤 워크로드에 특히 잘 맞을까요

Azure Functions가 빛나는 장면은 비교적 분명합니다.

- **이벤트 처리 파이프라인** — Event Hub 메시지를 받아 가공 후 적재하는 경우입니다.
- **예약 작업** — 매일 새벽 실행되는 정산 배치처럼 간헐적인 작업입니다.
- **파일 후처리** — Blob 업로드 후 썸네일 생성, 메타데이터 추출 같은 흐름입니다.
- **가벼운 HTTP API / Webhook** — 외부 SaaS의 Webhook 수신이나 작은 마이크로서비스입니다.
- **비동기 큐 소비자** — 큐에 메시지가 쌓일 때만 깨어나 처리하는 워커입니다.

공통점은 “일이 있을 때는 빠르게 처리해야 하지만, 평소에는 계속 떠 있을 이유가 약하다”는 점입니다. 이런 워크로드에서는 이벤트 기반 실행 모델이 비용과 운영 양쪽에서 모두 합리적입니다.

여기에 한 가지 운영 감각을 더 붙이면 이해가 더 쉬워집니다. Azure Functions는 “항상 붙어 있어야 하는 대화형 시스템”보다 “사건이 생겼을 때 반응해야 하는 시스템”에 더 자연스럽습니다. 그래서 설계 문서에서도 함수 이름보다 이벤트 이름과 외부 시스템 이름이 먼저 나오는 경우가 많습니다.

예를 들어 `orders-created` 메시지가 오면 송장을 만든다, Blob 업로드가 발생하면 이미지를 가공한다, 매시 정각이면 정산 작업을 돌린다 같은 문장 구조가 Functions와 잘 맞습니다. 이 문장 자체가 이미 이벤트 기반 설계의 힌트입니다.

### 반대로 안 맞는 경우도 분명히 있습니다

Azure Functions가 만능은 아닙니다. 아래 조건은 도입 전에 반드시 다시 봐야 합니다.

- **장기 실행 작업** — 한 번 실행이 매우 길다면 Durable Functions나 다른 컴퓨트를 검토해야 합니다.
- **매우 일관된 고부하** — 트래픽이 24시간 높은 서비스라면 상시 실행형이 더 단순할 수 있습니다.
- **콜드 스타트가 치명적인 초저지연 API** — 첫 요청 지연이 비즈니스적으로 허용되지 않으면 Premium이나 Always Ready 같은 다른 선택이 필요합니다.

즉 Azure Functions는 “서버리스라서 멋진 도구”가 아니라, **워크로드 특성과 비용 모델이 맞을 때 강한 도구**입니다. 이 판단 기준을 초기에 갖고 가는 편이 중요합니다.

여기서 특히 조심해야 할 함정은 “작은 함수로 쪼개면 무조건 Functions에 잘 맞겠다”는 생각입니다. 코드 크기와 워크로드 적합성은 같은 문제가 아닙니다. 작은 코드라도 stateful 세션 유지가 중요하거나, 초저지연이 절대 조건이거나, 긴 실행 시간이 기본값이라면 다른 컴퓨트가 더 자연스러울 수 있습니다.

반대로 코드가 길더라도 이벤트 중심으로 분해할 수 있고, 작업이 멱등적으로 나뉘며, 호출 패턴이 불규칙하다면 Azure Functions가 오히려 운영을 단순하게 만들 수 있습니다. 결국 중요한 것은 코드 줄 수가 아니라 실행 수명주기와 이벤트 경계입니다.

### Logic Apps와 Durable Functions와의 경계도 초기에 알아둘 만합니다

Azure Functions를 처음 볼 때 Logic Apps나 Durable Functions와의 차이도 자주 함께 묻습니다. 입문 단계에서는 아래 정도로 구분해 두면 충분합니다.

- **Azure Functions** — 코드 중심의 이벤트 처리입니다.
- **Logic Apps** — 시스템 간 연결과 워크플로 오케스트레이션에 강합니다.
- **Durable Functions** — Functions 모델 안에서 장기 실행과 상태 있는 오케스트레이션을 구현할 때 씁니다.

즉 Azure Functions는 가장 낮은 레벨의 코드 실행 단위에 가깝고, Durable Functions는 그 위에 상태 있는 흐름 제어를 얹는 방식이며, Logic Apps는 더 높은 수준의 워크플로 자동화에 가깝습니다. 이 구분을 알고 있으면 “왜 장기 실행은 기본 Functions에 그대로 밀어 넣지 말라고 하는가”도 더 쉽게 이해됩니다.

### 이 장의 한 줄 멘탈 모델을 다시 고정합니다

여기까지의 설명을 다시 한 줄로 접으면 다음과 같습니다.

> Azure Functions는 이벤트가 코드를 깨우고, 작업이 끝나면 실행 환경이 다시 유휴 상태로 돌아가거나 회수될 수 있는 플랫폼입니다.

이 문장을 기억하면 뒤의 트리거와 바인딩, Host와 Worker, 플랜 선택, 스케일링, 모니터링이 모두 한 축 위에 정렬됩니다. 첫 글의 역할은 바로 그 축을 만드는 데 있습니다.

### 실제 아키텍처 문장으로 바꾸면 더 또렷해집니다

실무에서 Azure Functions를 설명할 때는 제품 이름보다 사건 중심 문장으로 바꾸는 편이 좋습니다. 예를 들어 “사용자가 이미지를 업로드하면 썸네일을 만든다”, “주문 메시지가 들어오면 송장을 생성한다”, “매일 새벽 3시에 정산을 수행한다” 같은 문장은 이미 Functions에 잘 맞는 실행 모델을 담고 있습니다.

반대로 “항상 로그인 세션을 메모리에 유지한다”, “사용자와 장시간 연결을 붙잡고 초저지연 응답을 준다”, “긴 작업을 하나의 프로세스에서 끝까지 붙들고 간다” 같은 문장은 기본 Functions보다 다른 선택지를 더 먼저 떠올리게 만듭니다. 이처럼 요구사항을 사건 중심 문장으로 다시 써 보는 것만으로도 적합성 판단이 쉬워집니다.

이 감각은 이후의 기술 선택에도 그대로 이어집니다. 트리거를 고를 때도, 바인딩을 설계할 때도, 플랜을 선택할 때도, 결국은 “이 시스템은 무엇을 기다리고, 어떤 사건이 오면 어떻게 반응하는가”라는 문장을 더 정확하게 만드는 작업이기 때문입니다.

## 흔히 헷갈리는 지점

- **“서버리스면 서버가 없다”는 뜻이 아닙니다.** 서버는 있지만, 사용자가 직접 서버 수명주기를 관리하지 않는다는 뜻에 가깝습니다.
- **“Functions는 HTTP API 서비스다”라고만 보면 절반만 이해한 셈입니다.** 실제 강점은 다양한 이벤트를 기준으로 코드를 실행하는 모델에 있습니다.
- **“호출이 없을 때 비용이 0이면 항상 가장 싸다”는 단정은 위험합니다.** 꾸준한 고부하에서는 다른 플랜이나 다른 컴퓨트가 더 유리할 수 있습니다.
- **“함수는 작게 쓰면 된다”와 “아무 책임이나 잘게 쪼개면 된다”는 같은 말이 아닙니다.** 이벤트 경계와 운영 경계를 같이 봐야 합니다.
- **“Functions를 쓰면 운영을 몰라도 된다”는 생각도 틀립니다.** 콜드 스타트, 재시도, 멱등성, 연결 재사용 같은 운영 감각은 오히려 더 중요해집니다.

실무에서 이 오해들이 위험한 이유는 대부분이 설계 초기에 문장 하나로 굳어지기 때문입니다. “서버리스니까 싸다”, “함수니까 작게만 만들면 된다”, “플랫폼이 다 알아서 한다” 같은 문장은 짧지만, 이후의 스토리지 설계와 권한 설계, 재시도 설계, 비용 예측을 모두 비틀 수 있습니다. 그래서 첫 글에서는 제품 기능보다 멘탈 모델을 더 엄격하게 잡는 편이 중요합니다.

## 운영 체크리스트

- [ ] 예상 트리거 종류와 호출 빈도를 과금 단위와 함께 정리했습니다.
- [ ] 콜드 스타트 허용 한도를 비기능 요구사항에 명시했습니다.
- [ ] 같은 Function App에 묶일 함수들의 배포·권한 라이프사이클을 정리했습니다.
- [ ] 스토리지, DB 같은 outbound 의존성의 연결 재사용 정책을 결정했습니다.
- [ ] Azure Functions를 쓰지 말아야 할 워크로드를 ADR이나 설계 문서에 남겼습니다.

## 정리

Azure Functions를 이해하는 첫걸음은 제품 기능을 외우는 데 있지 않습니다. **이벤트가 코드를 깨우고, 일이 끝나면 실행 환경이 다시 회수될 수 있는 모델**이라는 사실을 받아들이는 데 있습니다. 이 관점이 잡혀야 왜 트리거와 바인딩이 중요한지, 왜 Host와 Worker의 구조를 알아야 하는지, 왜 플랜 선택과 콜드 스타트가 뒤따르는지 자연스럽게 연결됩니다.

이 글에서 기억할 핵심만 다시 압축하면 이렇습니다. Azure Functions는 이벤트 기반 실행 모델이고, 비용과 성능은 그 실행 모델 위에서 결정됩니다. 상시 실행 서버처럼 가정하면 설계와 운영이 쉽게 어긋납니다. 반대로 이벤트, 함수, 실행 수명주기라는 세 축으로 보면 Functions가 잘 맞는 장면과 안 맞는 장면이 빠르게 보입니다.

다음 글에서는 이 모델의 바깥 표면인 **트리거와 바인딩**으로 들어갑니다. 어떤 이벤트가 함수를 깨우는지, 함수의 입력과 출력을 선언적으로 연결한다는 말이 정확히 무엇인지 분명해지면 Azure Functions를 다루는 감각이 한 단계 더 구체화됩니다.

그리고 더 중요한 점은, 지금 만든 멘탈 모델이 뒤의 모든 장에서 계속 재사용된다는 사실입니다. Azure Functions를 배운다는 것은 기능 목록을 외우는 일이 아니라, 이벤트 기반 플랫폼을 같은 눈으로 계속 읽는 연습에 가깝습니다. 첫 장의 문장이 단단할수록 이후의 기술 세부도 훨씬 덜 흔들립니다.

## 처음 질문으로 돌아가기

- **Azure Functions는 정확히 어떤 서버리스 컴퓨트 모델로 이해해야 할까요?**
  - Azure Functions는 `@app.route(route="hello")` 같은 HTTP 요청이나 `@app.queue_trigger(queue_name="orders-incoming")` 같은 이벤트가 들어올 때만 함수를 깨우는 실행 모델입니다. 호출이 끝나면 인스턴스가 유휴로 돌아가거나 회수될 수 있으므로, 상시 실행 서버보다 이벤트와 실행 수명주기를 먼저 보는 플랫폼으로 이해하는 편이 맞습니다.
- **실행 횟수와 실행 시간 기준 과금은 언제 유리하고 언제 오히려 불리해질까요?**
  - `hello` 같은 가벼운 HTTP 함수나 `order_received`처럼 큐 메시지가 들어올 때만 도는 워크로드는 실행 기반 과금이 잘 맞습니다. 반대로 호출이 24시간 꾸준히 높거나 첫 요청 지연을 거의 허용하지 않는 서비스라면, Functions의 장점보다 warm 인스턴스를 계속 유지하는 비용과 운영 단순성이 더 중요해질 수 있습니다.
- **Azure Functions는 이벤트 기반 아키텍처에서 어떤 위치를 차지할까요?**
  - 이 시리즈의 예시처럼 Azure Functions는 `orders-incoming` 큐, Blob 업로드, 타이머 같은 사건이 발생했을 때 코드를 실행하는 반응 계층입니다. 그래서 “항상 요청을 받는 웹 서버”보다 “이벤트가 오면 작은 함수를 호출해 후속 처리를 만든다”는 구조에서 가장 자연스럽게 자리 잡습니다.
