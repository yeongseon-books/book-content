# Azure Functions 101 (2/7): 트리거와 바인딩 — 함수 입출력의 모든 것

앞 글에서 Azure Functions를 “이벤트가 코드를 깨우는 플랫폼”으로 설명했습니다. 그 모델이 실제 코드 표면에서 어떻게 드러나는지 보면, 거의 모든 길은 결국 **트리거와 바인딩**으로 모입니다. 함수가 왜 이 시점에 실행됐는지, 함수가 어떤 입력을 받는지, 함수 결과가 왜 별도 클라이언트 코드 없이 저장소나 메시징 시스템으로 나가는지를 이해하려면 이 둘부터 정확히 잡아야 합니다.

입문 단계에서는 이 경계가 자주 흐려집니다. HTTP 요청이 들어와 함수가 실행되면 그 요청 객체는 트리거의 일부인지, 입력 바인딩인지, 그냥 런타임이 넣어 준 값인지 헷갈리기 쉽습니다. 출력 바인딩도 비슷합니다. “코드에 DB 클라이언트가 없는데 왜 저장되지?”라는 질문이 자연스럽게 따라옵니다.

이 글은 Azure Functions 101 시리즈의 두 번째 글입니다. 여기서는 트리거와 바인딩을 단순한 문법 요소가 아니라 **함수의 입출력 계약을 정의하는 선언적 인터페이스**로 이해해 보겠습니다. 이 계약이 선명해져야 새 함수를 설계할 때도, 재시도와 멱등성을 다룰 때도, 운영 문제를 좁힐 때도 흔들리지 않습니다.

특히 이번 장에서는 “어디까지가 플랫폼이 대신하는 일이고, 어디부터가 여전히 내 책임인가”를 분명히 하겠습니다. 이 경계를 모르면 바인딩을 과신하거나, 반대로 불필요한 보일러플레이트를 다시 직접 쓰게 되기 쉽습니다.

이제 함수가 깨어나는 원인과, 함수가 바깥세상과 데이터를 주고받는 선이 어떻게 나뉘는지 차례로 보겠습니다.

> 트리거와 바인딩 — 함수 입출력의 모든 것의 핵심은 기능 이름이 아니라, 어떤 경계에서 무엇을 검증하고 어떤 신호를 남길지 정하는 데 있습니다.

## 먼저 던지는 질문

- 트리거와 바인딩은 근본적으로 무엇이 다르고 왜 분리되어 있을까요?
- 입력 바인딩과 출력 바인딩은 코드를 얼마나 줄여 주고, 대신 어떤 제약을 가져올까요?
- 함수 하나에 여러 트리거를 붙일 수 없다면 그 이유는 무엇일까요?

## 왜 이 글이 중요한가

Azure Functions를 쓰면서 처음 체감하는 편리함은 대개 코드 길이에서 옵니다. HTTP 라우터를 직접 열지 않아도 되고, 큐 메시지를 읽는 루프를 손으로 짜지 않아도 되고, 어떤 경우에는 데이터 저장 코드도 몇 줄이면 끝납니다. 그런데 이 편리함은 모두 **트리거와 바인딩이라는 추상화** 위에 서 있습니다. 추상화를 이해하지 못하면 편리함이 곧 불투명함으로 바뀝니다.

실무에서는 이 경계가 특히 중요합니다. 예를 들어 “함수 본체는 성공했는데 왜 전체 호출은 실패로 잡혔지?”라는 질문은 출력 바인딩 실패를 모르고는 답하기 어렵습니다. “왜 같은 메시지가 두 번 처리됐지?”라는 질문도 트리거의 재시도 모델과 멱등성 설계를 함께 봐야 풀립니다. 즉 입출력 계약을 모르면 장애를 봐도 어디서부터 읽어야 할지 애매해집니다.

또한 다음 글의 Host/Worker 구조를 이해하려면 먼저 함수 바깥 인터페이스가 정리돼 있어야 합니다. Host는 결국 트리거를 감지하고 바인딩을 해석한 뒤 Worker에게 호출을 넘기는 런타임이기 때문입니다. 그래서 이번 장은 문법 소개가 아니라, **Functions가 외부 세계와 맺는 계약을 해석하는 장**이라고 보는 편이 맞습니다.

## 핵심 관점

Azure Functions에서 트리거는 “무슨 사건이 이 함수를 깨우는가”를 정의하고, 바인딩은 “깨어난 함수가 바깥세상과 어떤 방식으로 데이터를 주고받는가”를 정의합니다. 즉 둘 다 함수의 인터페이스에 속하지만, 역할은 분명히 다릅니다. **트리거는 실행의 원인이고, 바인딩은 데이터 통로**입니다.

이렇게 보면 많은 혼동이 한 번에 정리됩니다. HTTP 요청이 들어왔다는 사실은 트리거이고, 함수가 실행 중에 Cosmos DB에서 추가 데이터를 읽는 일은 입력 바인딩입니다. 함수가 결과를 Service Bus나 Blob Storage로 내보내는 일은 출력 바인딩입니다. 하나는 실행을 일으키고, 나머지는 실행된 함수의 입출력을 선언적으로 줄여 주는 장치입니다.

따라서 새 함수를 설계할 때는 먼저 “무엇이 이 함수를 깨워야 하는가?”를 묻고, 그다음 “함수가 깨어난 뒤 어떤 외부 자원을 읽거나 써야 하는가?”를 묻는 편이 좋습니다. 이 순서를 지키면 트리거와 바인딩의 책임이 섞이지 않습니다.

> Azure Functions에서 트리거와 바인딩을 잘 고른다는 말은 곧 함수의 실행 원인과 데이터 통로를 명확히 분리해 설계한다는 뜻입니다. 여기서부터 코드 길이, 재시도, 운영 가시성이 모두 달라집니다.

## 핵심 개념

### 트리거는 함수를 깨우는 원인입니다

먼저 정의를 분명히 두겠습니다. **트리거(trigger)는 함수가 실행되도록 만드는 외부 사건**입니다. HTTP 요청, 큐 메시지, Blob 업로드, 타이머 스케줄, 데이터베이스 변경 피드 같은 것들이 모두 여기에 들어갑니다.

규칙은 단순합니다.

- 함수 하나에는 **정확히 하나의 트리거**가 붙습니다.
- 트리거는 실행 시점만 정하는 것이 아니라 **함수의 기본 입력 페이로드**도 함께 정합니다.

즉 HTTP 트리거는 요청 객체를, Queue 트리거는 메시지 본문을, Timer 트리거는 스케줄 정보를 함수의 출발 입력으로 정해 줍니다. 그래서 트리거는 “언제”와 “무엇을 가지고”를 동시에 결정합니다.

이 특성 때문에 트리거 선택은 단순한 문법 선택이 아닙니다. 함수가 언제 실행되는지뿐 아니라, 어떤 재시도 모델을 따르고, 어떤 관측 신호가 남고, 어떤 멱등성 전략이 필요한지까지 트리거 종류에 따라 달라집니다. 트리거는 실행 원인이면서 운영 모델의 일부입니다.

### 자주 쓰는 트리거는 생각보다 정해져 있습니다

공식 문서에는 훨씬 더 많은 종류가 있지만, 실무 입문 단계에서 가장 자주 만나는 것은 아래 범주 안에 거의 들어옵니다.

| 카테고리 | 트리거 | 대표 사용 사례 |
|---|---|---|
| **HTTP / Webhook** | HTTP, Event Grid | REST API, 외부 SaaS Webhook 수신 |
| **Schedule** | Timer | 정산 배치, 주기적 정리 작업 |
| **Storage** | Blob, Queue | 파일 업로드 후처리, 비동기 작업 큐 |
| **Messaging** | Service Bus, Event Hub, Kafka | 마이크로서비스 간 비동기 통신, 텔레메트리 수집 |
| **Database** | Cosmos DB Change Feed, SQL | 데이터 변경 감지 후처리 (CDC 패턴) |

새 함수를 설계할 때 가장 먼저 던질 질문은 늘 같습니다. **“이 함수는 무엇 때문에 깨어나야 하는가?”** 그 답이 곧 트리거 선택입니다. 이 질문이 흐리면 설계도 금방 흐려집니다.

실무에서는 반대로 질문을 잘못 던지는 경우도 많습니다. “어떤 SDK가 더 익숙한가?”, “예제에서 무엇을 많이 봤는가?” 같은 이유로 트리거를 고르면, 나중에 재시도와 실패 처리 모델이 예상과 달라져 구조를 다시 바꾸기 쉽습니다. Functions에서는 구현보다 먼저 실행 원인을 정하는 편이 안전합니다.

### 바인딩은 함수 입출력의 선언적 연결선입니다

Azure Functions가 특히 편하게 느껴지는 이유는 바인딩 덕분입니다. 일반 웹 앱에서 외부 시스템과 연결하려면 클라이언트를 만들고 인증을 붙이고 직렬화를 처리하고 저장 호출까지 직접 써야 합니다. 바인딩은 이 보일러플레이트를 “이 함수의 입력은 여기서 읽고, 출력은 여기로 보낸다”는 선언으로 줄여 줍니다.

Azure Functions Python v2 모델의 대표 예시는 아래와 같습니다.

```python
import json
import azure.functions as func

app = func.FunctionApp()

def build_invoice(payload):
    return {"id": payload["order_id"], "amount": payload["total"]}

@app.function_name(name="process_order")
@app.queue_trigger(arg_name="msg", queue_name="orders-incoming", connection="StorageConnection")
@app.cosmos_db_output(
    arg_name="invoice_out",
    database_name="orders",
    container_name="invoices",
    connection="CosmosConnection",
)
def process_order(msg: func.QueueMessage, invoice_out: func.Out[func.Document]) -> None:
    queue_item = msg.get_json()
    invoice = build_invoice(queue_item)
    invoice_out.set(func.Document.from_json(json.dumps(invoice)))
```

이 예제는 평어로 풀면 “큐 메시지가 들어오면 주문 데이터를 invoice로 바꾸고, 그 결과를 Cosmos DB에 저장한다”입니다. 여기서 DB 연결 관리, 인증, 재시도 같은 공통 작업은 Functions Host와 바인딩 확장이 맡습니다.

이 지점이 Azure Functions 코드가 짧아 보이는 이유이기도 합니다. 보일러플레이트가 적다는 것은 생산성이 좋아진다는 뜻이지만, 동시에 플랫폼이 숨겨 주는 동작이 늘어난다는 뜻이기도 합니다. 따라서 코드를 짧게 쓰는 것과 플랫폼 동작을 이해하는 것은 오히려 같이 가야 합니다.

### 입력 바인딩과 출력 바인딩은 방향이 다릅니다

바인딩은 크게 두 방향으로 나뉩니다.

- **입력 바인딩(input binding)** — 함수가 실행되는 시점에 외부 데이터를 읽어옵니다.
- **출력 바인딩(output binding)** — 함수가 만든 결과를 외부 자원으로 보냅니다.

여기서 가장 흔한 오해가 하나 있습니다. **트리거와 입력 바인딩은 같은 것이 아닙니다.** 트리거는 함수를 깨우는 사건 그 자체이고, 입력 바인딩은 이미 깨어난 함수가 추가로 읽는 데이터입니다.

예를 들어 “HTTP 요청이 들어오면, 요청의 user_id를 가지고 Cosmos DB에서 사용자 정보를 읽는다”라는 함수를 생각해 보면 이해가 쉽습니다. HTTP가 트리거이고, Cosmos DB 조회가 입력 바인딩입니다.

![트리거와 입출력 바인딩의 역할 차이](https://yeongseon-books.github.io/book-public-assets/assets/azure-functions-101/02/02-03-input-bindings-vs-output-bindings.ko.png)

*트리거와 입출력 바인딩의 역할 차이*

함수는 가운데 있고, 외부와는 트리거·입력·출력 세 갈래로 연결됩니다. 바인딩은 이 연결선을 코드에서 선언적으로 표현하는 방법입니다.

이 그림을 실무 문장으로 바꾸면 더 쉽습니다. “무엇이 나를 깨웠는가”는 트리거, “깨어난 뒤 무엇을 더 읽는가”는 입력 바인딩, “끝난 뒤 결과를 어디로 보내는가”는 출력 바인딩입니다. 새 함수를 설계할 때 이 세 문장을 따로 적어 보면 책임 경계가 훨씬 선명해집니다.

### 바인딩이 마법처럼 보이는 이유와 실제 정체

처음 보면 “DB 클라이언트도 없는데 왜 저장되지?”라는 생각이 드는 것이 자연스럽습니다. 하지만 실제로는 마법이 아니라 **확장(extension)** 과 Host가 대신 써 둔 코드입니다.

예를 들어 Cosmos DB 출력 바인딩은 관련 확장이 Host에 “이 함수 결과를 Cosmos DB로 보내야 한다”는 메타데이터를 등록하고, 함수 실행이 끝나면 그 확장 핸들러가 실제 클라이언트를 통해 저장을 수행합니다. 즉 여러분이 직접 작성했을 연결·인증·직렬화 코드를 플랫폼이 공통화해서 제공하는 것입니다.

그래서 경계를 이렇게 기억하는 편이 좋습니다.

- 바인딩이 자동으로 처리하는 것: 연결, 인증, 직렬화, 기본 재시도
- 바인딩이 자동으로 처리하지 않는 것: 비즈니스 검증, 부분 실패 정책, 트랜잭션 경계, 도메인 로직

복잡한 트랜잭션과 정교한 예외 처리가 필요하면 출력 바인딩 대신 직접 SDK를 쓰는 편이 더 명확할 때도 있습니다. 바인딩은 보일러플레이트를 줄여 주는 도구이지, 도메인 로직을 대신 판단해 주는 도구는 아닙니다.

현업에서는 이 판단이 꽤 중요합니다. 입문 단계에서는 바인딩이 거의 항상 더 좋아 보이지만, 실제로는 복잡한 부분 실패 처리나 세밀한 제어가 필요한 워크로드에서 직접 SDK가 더 투명하고 운영 친화적일 수 있습니다. 바인딩은 강한 기본값이지만 언제나 최종 해답은 아닙니다.

### 자주 반복되는 조합을 익혀 두면 설계가 빨라집니다

입문 단계에서 자주 보는 패턴은 생각보다 반복적입니다.

| 패턴 | 트리거 | 입력 바인딩 | 출력 바인딩 | 흔한 사용 사례 |
|---|---|---|---|---|
| **API + DB 조회** | HTTP | Cosmos DB | — | 단순 조회 API |
| **API + DB 저장** | HTTP | — | Cosmos DB | Webhook 수신 후 적재 |
| **큐 → DB 적재** | Queue | — | Cosmos DB | 비동기 주문 처리 |
| **파일 → 썸네일** | Blob | — | Blob | 이미지 후처리 |
| **DB 변경 → 알림** | Cosmos Change Feed | — | Service Bus | CDC 기반 이벤트 발행 |

이 패턴을 표로 외우려는 것보다, “실행 원인 하나 + 추가 입력 + 결과 출력”이라는 구조로 읽는 편이 더 오래 갑니다.

이 구조로 생각하면 패턴이 바뀌어도 적응이 쉽습니다. HTTP + Cosmos DB 대신 Event Grid + Blob Storage로 바뀌어도, 결국 실행 원인과 추가 입출력 선을 다시 고르는 문제로 환원되기 때문입니다. Azure Functions를 오래 쓸수록 개별 바인딩 이름보다 이 추상화가 더 중요해집니다.

### 설정 주입 경계는 보안과 운영을 동시에 좌우합니다

바인딩을 사용할 때 connection 관련 값은 코드보다 설정 계층에서 더 중요합니다. 같은 코드를 로컬, 스테이징, 프로덕션에 그대로 올리되 실제 연결 대상만 환경별로 바꿔야 하기 때문입니다.

이 원칙을 지키면 비밀값이 코드 저장소에 남지 않고, 환경 전환이 단순해지고, Key Vault 참조나 Managed Identity 같은 운영 표준을 적용하기 쉬워집니다. 반대로 이 경계를 흐리면 바인딩이 편리할수록 비밀값 관리가 더 위험해집니다.

### 재시도와 멱등성은 트리거 선택과 함께 봐야 합니다

Azure Functions에서 재시도는 함수 본체만의 문제가 아닙니다. 트리거 종류에 따라 재처리 방식이 달라지고, 출력 바인딩 실패가 전체 호출 실패로 기록될 수도 있기 때문입니다. 그래서 입문 단계에서도 “같은 메시지가 다시 와도 결과가 같을 수 있는가”를 먼저 생각하는 편이 좋습니다.

특히 큐 기반 처리에서는 멱등성이 설계에 녹아 있어야 합니다. 이미 처리한 주문 메시지가 다시 들어왔을 때 중복 송장을 만들지 않는지, 동일 Blob 이벤트가 한 번 더 들어와도 결과가 안전한지 같은 질문이 여기에 속합니다. 트리거와 바인딩은 코드를 줄여 주지만, 재처리 가능성 자체를 없애 주지는 않습니다.

이 감각이 있으면 바인딩을 더 현실적으로 쓸 수 있습니다. 즉 “코드가 짧아졌다”에서 멈추지 않고, “실패했을 때 같은 입력이 다시 들어오면 내 함수는 어떤 결과를 만들까”까지 함께 보게 됩니다. 실제 운영 품질은 대개 이 질문에서 갈립니다.

### 운영에서 특히 조심해야 할 함정 두 가지

첫째, **출력 바인딩 실패는 함수 본체 성공과 별도로 문제를 만들 수 있습니다.** 함수 코드가 끝까지 잘 실행됐더라도 출력 바인딩이 외부 저장에 실패하면 전체 호출은 실패로 잡히고, 트리거 정책에 따라 다시 처리될 수 있습니다. 그래서 같은 메시지가 다시 들어와도 결과가 같아야 하는 멱등성이 중요합니다.

둘째, 바인딩의 `connection` 값은 **연결 문자열 자체가 아니라 설정 키 이름**입니다. 예를 들어 `connection="StorageConnection"`은 “환경 변수나 앱 설정에 있는 `StorageConnection` 값을 사용하라”는 뜻입니다. 코드에 비밀값을 직접 넣는 구조로 이해하면 안 됩니다.

## 흔히 헷갈리는 지점

- **트리거와 입력 바인딩은 다릅니다.** 트리거는 실행 원인이고, 입력 바인딩은 깨어난 함수가 추가로 읽는 데이터입니다.
- **함수 하나에 트리거 하나라는 제약은 불편한 문법이 아니라 실행 원인을 명확하게 하려는 설계입니다.**
- **바인딩은 외부 SDK를 없애 주는 마법이 아닙니다.** 공통 연결 작업을 Host와 확장이 대신할 뿐입니다.
- **출력 바인딩이 있으면 실패 처리가 쉬워지는 것이 아니라, 오히려 재시도와 멱등성을 더 의식해야 할 수 있습니다.**
- **`connection`은 연결 문자열이 아니라 설정 키 이름입니다.** 실제 비밀값은 코드 밖에 둬야 합니다.

이 오해들이 자주 반복되는 이유는 Functions가 코드를 너무 쉽게 줄여 주기 때문입니다. 눈앞 코드가 짧아지면 입문자는 그만큼 시스템도 단순해졌다고 느끼기 쉽습니다. 하지만 실제로는 연결, 직렬화, 재시도, 실패 기록 같은 일이 여전히 일어나고 있고, 단지 그 위치가 여러분 코드에서 플랫폼으로 옮겨 갔을 뿐입니다. 운영 감각은 그 이동한 책임까지 함께 따라가야 합니다.

## 운영 체크리스트

- [ ] 트리거와 바인딩을 코드와 설정 관점에서 분리해 문서화했습니다.
- [ ] 바인딩이 자동으로 처리하는 동작과 직접 책임져야 할 동작을 구분했습니다.
- [ ] connection string은 Key Vault나 Managed Identity 기반 경로로 주입하도록 설계했습니다.
- [ ] 재시도 정책과 poison queue 또는 DLQ 경로를 트리거 종류별로 정리했습니다.
- [ ] 바인딩 변경이 생겨도 깨지지 않도록 통합 테스트를 준비했습니다.

## 정리

이번 글의 핵심은 단순합니다. **트리거는 함수를 깨우는 원인이고, 바인딩은 함수의 입출력을 선언적으로 연결하는 장치**입니다. 이 둘을 분리해서 볼 수 있어야 Azure Functions의 짧은 코드가 어떻게 가능해지는지, 그리고 그 짧음 뒤에 어떤 플랫폼 계약이 숨어 있는지 이해할 수 있습니다.

또한 바인딩은 보일러플레이트를 크게 줄여 주지만, 비즈니스 검증과 실패 정책까지 대신해 주지는 않습니다. 이 경계를 분명히 알아야 바인딩을 과신하지 않고, 필요한 곳에서는 직접 SDK를 쓰는 판단도 할 수 있습니다. 멱등성, 재시도, 설정 주입 같은 운영 주제가 여기와 바로 연결된다는 점도 함께 기억할 만합니다.

다음 글에서는 한 단계 더 아래로 내려가, 이렇게 정의된 함수가 실제로 **누구에 의해 어떤 프로세스 구조 안에서 실행되는지**를 봅니다. 즉 트리거와 바인딩이라는 바깥 표면 아래에 있는 Host와 Worker의 실행 경계로 들어가겠습니다.

트리거와 바인딩을 선명하게 이해하고 나면 Functions 코드를 읽는 시야도 달라집니다. 이제는 “이 함수가 무슨 일을 하느냐”뿐 아니라 “무엇이 이 함수를 깨우고, 어떤 데이터 선이 자동으로 연결되어 있느냐”를 함께 읽게 됩니다. 그 감각이 잡히면 Host와 Worker 장도 훨씬 자연스럽게 이어집니다.

## 처음 질문으로 돌아가기

- **트리거와 바인딩은 근본적으로 무엇이 다르고 왜 분리되어 있을까요?**
  - `@app.queue_trigger(arg_name="msg", queue_name="orders-incoming")`는 함수를 깨우는 실행 원인이고, `@app.cosmos_db_output(...)`은 실행된 함수의 결과를 어디로 보낼지 정하는 데이터 통로입니다. 실행 원인과 입출력 통로를 분리해 두어야 재시도, 실패 기록, 멱등성 판단도 어느 층의 문제인지 선명해집니다.
- **입력 바인딩과 출력 바인딩은 코드를 얼마나 줄여 주고, 대신 어떤 제약을 가져올까요?**
  - `invoice_out.set(func.Document.from_json(json.dumps(invoice)))`처럼 출력 바인딩을 쓰면 직접 Cosmos DB 클라이언트를 만들지 않아도 저장 경로를 짧게 적을 수 있습니다. 대신 바인딩은 연결과 직렬화를 줄여 줄 뿐이고, 출력 실패 시 재처리, 부분 실패 정책, 멱등성 보장은 여전히 함수 설계와 운영 책임으로 남습니다.
- **함수 하나에 여러 트리거를 붙일 수 없다면 그 이유는 무엇일까요?**
  - 함수 하나에 트리거 하나만 두는 이유는 “무엇이 이 함수를 깨웠는가”를 모호하게 만들지 않기 위해서입니다. HTTP인지 `orders-incoming` 큐인지 실행 원인이 하나로 고정돼야 입력 페이로드, 재시도 모델, poison queue나 DLQ 같은 후속 운영 경로도 같이 명확해집니다.
