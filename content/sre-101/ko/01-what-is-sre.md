# SRE 101 (1/10): SRE란 무엇인가?

서비스가 자주 흔들릴 때 팀은 보통 사람을 더 붙이거나, 야간 대응을 더 강하게 하거나, 출시 속도를 낮추는 쪽으로 먼저 움직입니다. 당장은 버틸 수 있어도 시간이 갈수록 운영은 사람의 기억과 희생에 기대게 됩니다.

SRE를 이해해야 하는 이유는 바로 여기 있습니다. 신뢰성을 더 많은 근성으로 버티는 대신, 측정 가능한 목표와 자동화 가능한 절차로 다루자는 관점이기 때문입니다.

이 글은 SRE 101 시리즈의 첫 번째 글입니다. 여기서는 SRE가 무엇인지, 왜 운영을 소프트웨어 문제로 본다고 말하는지, 그리고 입문자가 어디서부터 출발하면 좋은지 설명합니다.

## 먼저 던지는 질문

- SRE는 단순히 운영팀의 다른 이름이 아니라 무엇을 뜻할까요?
- SRE와 DevOps는 왜 자주 함께 언급되지만 같은 말은 아닐까요?
- 신뢰성을 제품의 일부로 본다는 말은 실무에서 어떤 차이를 만들까요?

## 큰 그림

![SRE 101 1장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/sre-101/01/01-01-concept-at-a-glance.ko.png)

*SRE 101 1장 흐름 개요*

SRE 개념이 어떻게 입력(요구사항), 처리(설계), 검증(측정), 운영 신호(결정)로 연결되는지 보여줍니다.

> SRE는 운영을 소프트웨어 엔지니어링 원칙으로 다루는 방식입니다.

## 왜 이 주제가 중요한가

기능을 빠르게 배포하는 팀도 서비스가 자주 멈추면 신뢰를 잃습니다. 반대로 안정성만 붙잡고 모든 변경을 두려워하면 제품은 느려집니다. SRE는 이 둘 중 하나를 포기하자는 접근이 아니라, 속도와 안정성을 같은 프레임 안에서 다루자는 접근입니다.

현업에서는 이 차이가 꽤 큽니다. 장애가 날 때마다 누가 잘못했는지부터 찾는 팀과, 어떤 지표를 봐야 하고 어떤 반복 작업을 자동화해야 하는지부터 따지는 팀은 시간이 지날수록 완전히 다른 운영 문화를 갖게 됩니다.

## 한 문장으로 잡는 멘탈 모델

> SRE는 운영을 사람의 숙련도에 맡기지 않고, 측정과 자동화로 다루는 엔지니어링 방식입니다.

## 한눈에 보는 구조

이 그림은 SRE를 가장 단순하게 보여 줍니다. 코드를 배포하고 끝나는 것이 아니라, 운영에서 관찰한 결과를 다시 다음 개발 판단으로 연결합니다. 신뢰성은 개발 뒤에 붙는 부록이 아니라, 개발과 운영을 한 바퀴로 묶는 피드백 구조 안에 들어 있습니다.

## 핵심 용어 먼저 정리

| 용어 | 뜻 | 실무에서 왜 중요한가 |
| --- | --- | --- |
| SRE | 신뢰성을 공학적으로 다루는 운영 방식 | 감으로 운영하던 일을 수치와 코드로 바꾸게 합니다 |
| reliability | 시스템이 기대한 방식으로 동작하는 정도 | 사용자가 실제로 느끼는 품질의 바닥선을 만듭니다 |
| SLO | 팀이 내부적으로 합의한 서비스 수준 목표 | 출시 속도와 안정성 사이의 기준점이 됩니다 |
| error budget | 목표가 허용하는 실패 여유분 | 어느 정도 위험까지 감수할지 정하게 해 줍니다 |
| toil | 반복적이고 자동화 가능한 수작업 | 사람 시간을 갉아먹는 운영 부채를 드러냅니다 |
| postmortem | 장애 뒤에 남기는 학습 문서 | 같은 장애를 다시 처음부터 겪지 않게 돕습니다 |

## 핵심 관점

SRE를 역할 이름으로만 보면 자주 헷갈립니다. 온콜을 하고, 장애를 처리하고, 대시보드를 보는 사람을 통칭하는 것처럼 보이기 때문입니다. 그런데 그 설명만으로는 왜 어떤 팀은 SRE를 두고도 계속 지치고, 어떤 팀은 같은 인원으로 더 큰 시스템을 운영하는지 설명하기 어렵습니다.

더 실용적인 설명은 이렇습니다. **SRE는 운영을 소프트웨어 문제로 다루는 방식**입니다. 반복되는 대응은 자동화하고, 신뢰성은 숫자로 정의하고, 장애 이후에는 학습을 시스템 변경으로 연결합니다. 사람이 계속 더 열심히 해야만 유지되는 구조라면, SRE 관점에서는 이미 개선 대상입니다.

이 시각을 잡아 두면 DevOps와의 관계도 정리됩니다. DevOps가 개발과 운영의 협업 문화를 강조한다면, SRE는 그 협업을 실제 수치와 절차로 운영하는 방법론에 가깝습니다. 서로 경쟁하는 개념이라기보다, SRE가 DevOps의 목표를 더 구체적인 운영 규칙으로 바꾸는 경우가 많습니다.

## 운영팀과 SRE의 차이는 어디서 드러날까

전통적인 운영 모델에서는 장애가 나면 숙련된 담당자가 경험으로 대응하고, 같은 문제가 다시 생기면 문서를 조금 보강하거나 체크리스트를 늘리는 식으로 버팁니다. 이런 방식은 작은 규모에서는 통할 수 있지만, 서비스가 커질수록 사람 의존성이 급격히 커집니다.

반대로 SRE는 반복되는 일을 그냥 익숙한 일로 두지 않습니다. 왜 이런 호출이 반복되는지, 어떤 지표를 먼저 보고 판단해야 하는지, 이 절차를 코드로 바꿀 수 없는지, 목표 수치가 현실적인지 같은 질문을 던집니다. 운영의 중심이 영웅적 대응에서 재현 가능한 시스템으로 옮겨가는 셈입니다.

이 차이는 장애 이후에도 드러납니다. 단순 운영 문화는 복구가 끝나면 사건을 닫아 버리기 쉽습니다. SRE 문화는 복구 이후에 무엇을 바꿔야 같은 장애를 줄일 수 있는지까지를 일의 일부로 봅니다.

## 단계별로 첫 SLO를 만들어 보기

### 1단계 — 지표 선택

```python
# Example: HTTP success ratio
indicator = "http_2xx / http_total"
```

SRE의 출발점은 측정입니다. 여기서는 HTTP 성공 비율을 지표로 잡았습니다. 좋은 지표는 서버 내부 상태보다 사용자 경험에 더 가깝습니다. CPU 사용률이 높아도 사용자는 문제를 못 느낄 수 있지만, 요청 성공률이 떨어지면 바로 영향을 받습니다.

### 2단계 — 목표 설정

```python
slo = {"indicator": indicator, "target": 0.999, "window": "30d"}
```

지표만 있으면 관찰은 되지만 판단은 어렵습니다. 그래서 목표가 필요합니다. 목표에는 수치와 기간이 함께 있어야 합니다. 99.9%라는 숫자만 적어 두는 것보다 30일 동안 99.9%를 지킨다고 써야 운영 판단으로 이어집니다.

### 3단계 — 측정

```python
def availability(success, total):
    return success / total
```

이 함수는 단순하지만 중요한 태도를 보여 줍니다. 신뢰성은 느낌이 아니라 계산 가능한 값이어야 합니다. 팀이 같은 숫자를 보고 이야기할 수 있어야 논쟁이 줄고 우선순위가 선명해집니다.

### 4단계 — 에러 버짓 계산

```python
def error_budget(target, total):
    return (1 - target) * total
```

목표를 세우면 자연스럽게 허용 가능한 실패 범위도 계산할 수 있습니다. SRE는 실패를 없애겠다고 말하지 않습니다. 어느 정도 실패까지는 감수할지 명시하고, 그 범위 안에서 더 빠르게 움직이게 만듭니다.

### 5단계 — 의사결정에 연결

```python
def can_release(spent, budget):
    return spent < budget
```

여기까지 와야 비로소 SRE가 문서가 아니라 운영 체계가 됩니다. 에러 버짓이 남아 있으면 더 빠르게 실험하고, 빠르게 소모되고 있으면 안정화 작업을 우선하는 식으로 숫자를 행동으로 바꿔야 합니다.

## 이 코드에서 먼저 봐야 할 점

- SRE는 지표를 고르는 일에서 시작하지만, 목표와 행동 규칙까지 이어져야 의미가 있습니다.
- 지표는 고객 경험과 연결될수록 팀 대화가 단순해집니다.
- 에러 버짓은 실패를 정당화하는 장치가 아니라, 위험 허용 범위를 명시하는 장치입니다.
- 수치가 있어야 출시 속도와 안정성을 한 문맥에서 이야기할 수 있습니다.

## 여기서 자주 헷갈립니다

첫 번째 혼동은 SRE를 운영 인력의 다른 이름으로 보는 것입니다. 운영을 담당한다고 해서 모두 SRE인 것은 아닙니다. 반복되는 운영 일을 측정하고 줄이는 구조가 있어야 SRE라고 부를 근거가 생깁니다.

두 번째 혼동은 DevOps와 SRE를 대체 관계로 보는 것입니다. 현업에서는 DevOps가 협업 문화와 흐름을 말하고, SRE가 그 흐름을 실제 운영 지표와 정책으로 정리하는 경우가 많습니다.

세 번째 혼동은 SLO를 만든 뒤 문서만 남기는 경우입니다. SLO는 보고서가 아니라 의사결정 도구입니다. 출시, 알림, 안정화 우선순위와 연결되지 않으면 금방 장식품이 됩니다.

## 운영 체크리스트

- [ ] 핵심 사용자 경로 하나에 대한 지표를 고를 수 있다.
- [ ] 그 지표의 목표 수치와 측정 기간을 함께 적었다.
- [ ] 에러 버짓이 어떤 결정을 바꿀지 설명할 수 있다.
- [ ] 반복 운영 업무 가운데 자동화 후보를 하나 이상 찾았다.
- [ ] 장애 이후 학습을 남기는 절차가 필요하다는 데 팀이 동의한다.

## 실무에서는 이렇게 생각합니다

시니어 엔지니어는 SRE를 더 많은 운영 업무를 맡는 역할로 보지 않습니다. 오히려 사람이 직접 해야 하는 운영 업무를 줄이는 방향으로 봅니다. 반복 호출이 많아질수록 인력을 더 넣기보다, 왜 그 호출이 발생하는지와 어떤 부분을 코드로 옮길 수 있는지를 먼저 따집니다.

또한 신뢰성은 인프라 팀만의 관심사가 아닙니다. 제품 기능이 사용자를 만나는 방식 자체가 신뢰성에 포함되기 때문입니다. 그래서 강한 팀일수록 제품팀과 플랫폼팀이 같은 SLO를 기준으로 대화합니다.

## 정리

SRE는 운영을 더 열심히 하자는 말이 아닙니다. 운영을 측정 가능하고 자동화 가능한 공학 문제로 다루자는 말입니다. 이 관점이 있어야 SLO, 에러 버짓, 모니터링, 인시던트 대응, 포스트모템, Toil 감소가 서로 따로 노는 개념이 아니라 하나의 운영 체계로 연결됩니다.

다음 글에서는 reliability를 더 구체적으로 다룹니다. 신뢰성을 막연한 안정감이 아니라 어떤 차원으로 나눠 측정해야 하는지 이어서 정리하겠습니다.

## 처음 질문으로 돌아가기

- **SRE는 단순히 운영팀의 다른 이름이 아니라 무엇을 뜻할까요?**
  - SRE를 운영의 주요 경계(입력→검증→신호)로 구분하고, 각 경계에서 무엇을 측정하고 판단할지 정하는 과정을 보여줍니다.
- **SRE와 DevOps는 왜 자주 함께 언급되지만 같은 말은 아닐까요?**
  - DevOps는 협업 문화를 강조하고, SRE는 그 협업을 측정 가능한 목표와 자동화로 구체화한 방법론입니다. 본문의 단계별 SLO 구축이 이 차이를 드러냅니다.
- **신뢰성을 제품의 일부로 본다는 말은 실무에서 어떤 차이를 만들까요?**
  - 에러 버짓으로 계산된 실패 여유를 출시 속도와 안정화 작업의 기준점으로 삼아, 관계없이 보이던 두 활동을 같은 의사결정 프레임 안에 놓는 것이 핵심입니다.
