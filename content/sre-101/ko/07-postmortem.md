# SRE 101 (7/10): Postmortem

장애가 끝나면 팀은 대개 안도감부터 느낍니다. 복구는 끝났고, 당장 불이 꺼졌기 때문입니다. 그런데 같은 유형의 장애가 반복된다면 실제로 끝난 것은 복구뿐이고, 학습은 아직 시작되지 않은 셈입니다.

포스트모템은 그 학습을 조직 자산으로 바꾸는 장치입니다. 누가 실수했는지 적는 문서가 아니라, 시스템과 절차의 약점이 무엇이었고 무엇을 바꿔야 하는지 남기는 문서여야 합니다.

이 글은 SRE 101 시리즈의 7번째 글입니다. 여기서는 postmortem을 학습 시스템으로 보고, blameless 원칙, 문서 템플릿, 액션 아이템 추적, 조직 차원의 공유 방식까지 정리합니다.

> 포스트모템은 장애 설명이 아니라, 같은 장애를 덜 겪게 만드는 학습 시스템입니다.

## 먼저 던지는 질문

- 포스트모템은 왜 보고서보다 학습 장치에 가깝다고 할까요?
- blameless 원칙이 없으면 왜 중요한 정보가 빠질까요?
- 좋은 포스트모템 문서에는 어떤 항목이 반드시 들어가야 할까요?

## 왜 이 주제가 중요한가

반복 장애는 종종 학습의 실패에서 나옵니다. 기록이 없거나, 기록은 있어도 후속 작업이 추적되지 않거나, 개인 비난 때문에 중요한 맥락이 드러나지 않으면 팀은 같은 함정에 다시 빠집니다.

반대로 좋은 포스트모템은 특정 담당자가 떠나도 남습니다. 당시 무엇이 보였고 무엇이 보이지 않았는지, 어떤 판단이 왜 내려졌는지, 무엇을 바꿔야 하는지가 문서와 액션으로 남기 때문입니다. 포스트모템은 조직 기억을 만드는 작업입니다.

## 한 문장으로 잡는 멘탈 모델

> 포스트모템은 장애를 설명하는 문서가 아니라, 같은 장애를 덜 겪게 만드는 학습 시스템입니다.

## 한눈에 보는 구조

장애가 문서로 끝나면 의미가 약합니다. 문서가 액션으로 연결되고, 그 액션이 추적과 리뷰로 이어질 때 비로소 재발 방지 효과가 생깁니다.

## 핵심 용어 먼저 정리

| 용어 | 뜻 | 실무에서 하는 역할 |
| --- | --- | --- |
| postmortem | 장애 이후 분석 문서 | 사실과 학습을 구조화합니다 |
| blameless | 개인 비난보다 시스템 개선에 초점을 두는 원칙 | 숨겨진 맥락을 더 많이 드러내게 합니다 |
| timeline | 사건 진행 순서 기록 | 사실 관계를 정리합니다 |
| root cause | 증상 아래의 구조적 원인 | 재발 방지 방향을 찾게 합니다 |
| action item | 후속 개선 작업 | 학습을 실제 변경으로 연결합니다 |

## 왜 비난 없는 문화가 먼저일까

개인 비난이 강한 팀에서는 사람들은 자신을 방어하는 방식으로 말하게 됩니다. 그러면 장애 당시 정보 부족, 경고 신호 부재, 절차 빈틈, 도구 한계 같은 진짜 원인이 잘 드러나지 않습니다.

blameless는 책임이 없다는 뜻이 아닙니다. 무엇이 시스템적으로 잘못 설계되었는지, 어떤 정보가 부족했는지, 어떤 안전장치가 없었는지를 먼저 보자는 원칙입니다. 진실한 설명이 나와야 다음 변경도 의미가 생깁니다.

## 비난 없는 문화가 포스트모템 품질에 미치는 영향

blameless 원칙은 단순히 좋은 팀 문화가 아니라, 포스트모템 품질을 직접 좌우합니다. 개인 비난이 강한 조직에서는 사람들이 중요한 사실을 숨기거나, 자신을 방어하는 방식으로 말하게 됩니다. 그러면 실제 원인이 잘 드러나지 않고, 같은 문제가 반복됩니다.

비난 없는 문화가 있으면:

- 장애 당시 정보가 부족했던 이유를 솔직히 말할 수 있습니다.
- 경고 신호가 있었는데 무시한 배경을 설명할 수 있습니다.
- 절차가 불명확했거나 도구가 부족했던 사실을 드러낼 수 있습니다.
- 판단 실수보다 시스템 설계 빈틈을 먼저 볼 수 있습니다.

이런 정보가 포스트모템에 들어가야 후속 개선 방향도 현실적으로 정해집니다. 비난 문화는 장기적으로 신뢰성을 낮춥니다.

### 비난 없는 질문 vs 비난적 질문

| 비난적 질문 | 비난 없는 질문 |
| --- | --- |
| 왜 백업을 확인하지 않았나요? | 백업 확인 절차가 문서화되어 있었나요? |
| 왜 알림을 놓쳤나요? | 알림이 평소에도 자주 발화했나요? |
| 왜 롤백을 더 빨리 하지 않았나요? | 롤백 버튼이 바로 보이는 위치에 있었나요? |
| 왜 장애를 예측하지 못했나요? | 장애 전에 어떤 신호가 있었는지 찾아볼까요? |

같은 사실을 묻더라도 질문의 방향을 바꾸면 답의 품질이 달라집니다. 시스템과 절차를 개선하는 정보가 더 많이 나옵니다.

## 템플릿이 필요한 이유

포스트모템을 매번 자유 형식으로 쓰면 비교와 재사용이 어렵습니다. 어떤 문서는 타임라인이 자세하고, 어떤 문서는 영향 범위가 빠지고, 어떤 문서는 액션 항목이 모호해질 수 있습니다. 공통 템플릿은 기억에 기대던 부분을 구조로 바꿉니다.

강한 팀은 문장력을 겨루지 않습니다. 누구나 같은 뼈대로 사실, 영향, 원인, 후속 조치를 남기도록 구조를 다듬는 데 더 신경 씁니다.

## 포스트모템 템플릿 항목 상세

포스트모템 문서는 항목별로 명확한 구조가 있어야 합니다. 각 섹션이 어떤 질문에 답하는지, 어떤 수준의 상세함이 필요한지 미리 정해 두면 누가 쓰든 비슷한 품질의 문서가 나옵니다.

| 항목 | 답하는 질문 | 작성 팁 |
| --- | --- | --- |
| 요약 | 장애를 한 문장으로 표현하면? | 일시, 증상, 영향 범위를 압축합니다. 2-3문장 이내로 제한합니다. |
| 타임라인 | 사건이 어떻게 전개됐나? | 시각(HH:MM), 이벤트, 행동, 결과를 시간 순서로 나열합니다. Slack 채널 메시지를 그대로 복사해도 좋습니다. |
| 근본 원인 | 증상의 진짜 원인은? | "느렸다" 대신 "타임아웃 설정 부재"처럼 시스템 구조적 원인을 찾습니다. 5-why 기법을 사용할 수 있습니다. |
| 행동 항목 | 무엇을 바꿀 것인가? | 각 항목에 오너, 기한, 우선순위(P0/P1/P2)를 붙입니다. 막연한 "개선 검토" 대신 "메모리 사용률 80% 알림 추가" 처럼 구체적으로 적습니다. |
| 배운 교훈 | 이 장애에서 얻은 통찰은? | 시스템 설계, 모니터링, 절차, 도구 가각에 대한 교훈을 2-3개 남깁니다. 다른 팀이 비슷한 실수를 피할 수 있게 작성합니다. |

각 섹션이 단단하게 채워져 있어야 나중에 비슷한 장애가 다시 오거나, 다른 팀에서 참고할 때 도움이 됩니다.

## 좋은 포스트모템 vs 나쁨 포스트모템

포스트모템의 품질은 문서의 길이가 아니라, 얼마나 재발 방지에 도움이 되는지로 판단합니다. 다음은 좋은 포스트모템과 나쁨 포스트모템의 비교입니다.

### 나쁨 포스트모템 예시

```
**제목:** DB 장애

**요약:** DB가 느려졌습니다.

**원인:** DB 서버에 문제가 있었습니다.

**해결:** DB를 재시작했습니다.

**후속 조치:** DB 모니터링 강화
```

**문제점:**

- 타임라인 없음 (언제 발생했는지, 얼마나 지속됐는지 불명)
- 근본 원인 모호함 ("DB 서버에 문제"는 증상일 뿐 원인 아님)
- 행동 항목 막연함 (오너, 기한, 구체적 내용 없음)
- 영향 범위 누락 (볇 명의 사용자가 영향받았는지 불명)

### 좋은 포스트모템 예시

```
**제목:** 2026-05-19 DB 커넥션 풀 고갈로 인한 API 지연

**요약:** 18:45~19:20 사이 API latency가 3초 → 15초로 증가, 12,000명 사용자 영향, 지속 시간 35분

**타임라인:**
- 18:45 - latency 급증 알림 발화 (p99 15초 초과)
- 18:47 - IC 지정, #inc-0519-db-pool 채널 생성
- 18:52 - DB 커넥션 수 max(100) 도달 확인
- 18:58 - 임시 조치: pool size 100 → 200 증설
- 19:10 - latency 3초 이하 복구 확인
- 19:20 - incident 종료

**근본 원인:** 배치 작업이 커넥션을 반환하지 않고 계속 점유. 커넥션 타임아웃 설정 부재.

**행동 항목:**
1. [긴급] 배치 작업에 커넥션 타임아웃 10분 추가 (P0, @backend-team, 2026-05-22)
2. [예방] 커넥션 풀 사용률 80% 초과 시 알림 추가 (P1, @sre-team, 2026-05-25)
3. [분석] 배치 작업 커넥션 사용 패턴 분석 (P2, @backend-team, 2026-05-30)

**배운 교훈:**
- 데이터베이스 커넥션 풀은 무한히 기다리는 자원이 아님. 타임아웃이 필수.
- 커넥션 풀 모니터링은 커넥션 수뿐 아니라 대기 시간도 함께 봐야 함.
- 배치 작업은 API 서버와 동일한 커넥션 풀을 공유하면 안 됨.
```

**왜 좋은가:**

- 타임라인이 상세하며 사건 흐름을 재구성할 수 있음
- 근본 원인이 구체적이고 시스템 구조적 문제를 지적
- 행동 항목에 오너, 기한, 우선순위가 명확
- 다른 팀이 배울 수 있는 교훈 포함
- 영향 범위와 지속 시간 정량화

포스트모템 문서는 항목별로 명확한 구조가 있어야 합니다. 각 섹션이 어떤 질문에 답하는지, 어떤 수준의 상세함이 필요한지 미리 정해 두면 누가 쓰든 비슷한 품질의 문서가 나옵니다.

| 섹션 | 답하는 질문 | 작성 가이드 |
| --- | --- | --- |
| 타임라인 | 언제 무엇이 일어났는가? | 타임스탬프, 이벤트, 누가 한 행동을 시간 순으로 나열합니다 |
| 영향 | 고객과 비즈니스에 어떤 결과를 불러왔는가? | 사용자 수, 지속 시간, 금액 손실, 평판 영향 등을 정량화합니다 |
| 근본 원인 | 증상 아래의 구조적 원인은 무엇인가? | "DB가 느리다" 대신 "쿼리 타임아웃 부재"처럼 깊게 파고듭니다 |
| 개선 항목 | 같은 문제를 예방하려면 무엇을 바꿔야 하는가? | 오너, 기한, 우선순위를 함께 적습니다 |

각 섹션이 단단하게 채워져 있어야 나중에 비슷한 장애가 다시 오거나, 다른 팀에서 참고할 때 도움이 됩니다.
## 단계별로 사후 분석 문서 작성하기

### 1단계 — 템플릿 정의

```python
template = {
    "title": "",
    "summary": "",
    "impact": "",
    "timeline": [],
    "root_cause": "",
    "actions": [],
    "lessons": [],
}
```

공통 템플릿이 있으면 어떤 사건이든 최소한 같은 프레임으로 비교할 수 있습니다. 기록의 일관성이 생기면 조직 학습 속도도 빨라집니다.

### 2단계 — 영향 요약

```python
def impact_line(users, minutes):
    return f"{users} users affected for {minutes} min"
```

영향 요약은 장애의 무게를 보여 주는 출발점입니다. 사용자 수, 지속 시간, 영향 기능을 짧게 적어 두면 후속 작업 우선순위도 분명해집니다.

### 3단계 — 타임라인 기록

```python
def event(t, msg):
    return {"time": t, "event": msg}
```

타임라인은 기억의 혼선을 줄입니다. 누가 언제 무엇을 보고 어떤 조치를 했는지 순서대로 적으면, 증상과 대응 사이의 관계가 훨씬 잘 드러납니다.

### 4단계 — 액션 아이템 작성

```python
def action(desc, owner, due):
    return {"desc": desc, "owner": owner, "due": due}
```

좋은 액션 아이템은 오너와 기한이 있습니다. 막연한 다짐은 쉽게 사라지지만, 담당자와 일정이 있는 작업은 추적이 가능합니다.

### 5단계 — 열린 작업 추적

```python
def open_actions(items):
    return [a for a in items if not a.get("done")]
```

포스트모템 품질은 문서가 아니라 후속 작업 완료율에서 드러납니다. 열린 액션을 계속 보지 않으면 포스트모템은 기록 보관함으로 밀려납니다.

## 포스트모템 예시 — 캐시 서버 다운으로 인한 홈 피드 느림

```python
postmortem = {
    "title": "2026-05-19 Redis 클러스터 다운으로 인한 홈 피드 latency 증가",
    "date": "2026-05-19",
    "severity": "SEV2",
    "impact": {
        "affected_users": 8500,
        "duration_min": 25,
        "service": "홈 피드 API"
    },
    "timeline": [
        "14:12 - Redis 클러스터 primary 노드 OOM kill",
        "14:13 - 홈 피드 latency 경보 (p99 5초 초과)",
```

## 포스트모템 작성 타이밍

포스트모템은 장애가 끝난 직후가 가장 좋습니다. 시간이 지나면 기억이 희미해지고, 중요한 맥락이 빠질 수 있습니다.

| 타이밍 | 적합도 | 이유 |
| --- | --- | --- |
| 장애 복구 후 24시간 내 | 매우 좋음 | 기억이 선명하고 채널 기록이 살아 있음 |
| 복구 후 1주 내 | 좋음 | 주요 타임라인 복원 가능, 일부 디테일 누락 가능성 |
| 복구 후 1개월 후 | 부적합 | 대부분의 맥락을 쟃고, 기록에만 의존하게 됨 |

포스트모템은 장애 대응의 마지막 단계가 아니라 필수 부분으로 보고, 일정에 미리 포함하는 편이 좋습니다.

## 액션 아이템 추적 방법

포스트모템의 가치는 문서 작성에서 끝나지 않고, 후속 조치가 실제로 완료되는지에서 나타납니다. 액션 아이템을 GitHub Issues, Jira, Linear 같은 도구에 티켓으로 만들고, 주간 리뷰에서 진행 상태를 확인해야 합니다.

액션 아이템에는 다음을 포함해야 합니다:

- **명확한 설명**: 무엇을 해야 하는지 구체적으로 서술
- **오너**: 누가 이 작업을 책임지는지 명시
- **기한**: 언제까지 완료할지 날짜 지정
- **우선순위**: P0 (즉시), P1 (1주 내), P2 (1개월 내) 등 분류
- **연결된 포스트모템**: 어떤 장애에서 나온 액션인지 링크

주간 SRE 미팅에서 열린 액션 아이템을 리뷰하고, 지연되는 항목은 이유를 기록하며, 완료된 항목은 실제 효과를 확인하는 프로세스가 필요합니다.

이 예시는 타임라인과 원인을 구조화하고, 액션 아이템에 우선순위(P0/P1/P2)까지 붙였습니다. 배운 교훈(lessons)도 함께 남겨서 다른 팀이 비슷한 실수를 피할 수 있게 합니다.

## 비난 없는 문화 정착시키기

blameless 원칙은 선언만으로는 정착되지 않습니다. 포스트모템 미팅에서 매번 실천해야 합니다. 퍼실리테이터가 개인 비난을 차단하고, 시스템과 절차 개선으로 대화를 이끌어야 합니다.

비난 없는 문화 체크리스트:

- [ ] 포스트모템 미팅에서 "누구 잘못" 대신 "무엇이 모자라"를 묻습니다
- [ ] 개인 이름보다 역할과 시스템 구성 요소로 서술합니다
- [ ] "실수"보다 "정보 부족" "안전 장치 부재" 같은 구조적 표현을 씩니다
- [ ] 후속 조치가 사람이 아니라 시스템을 바꾸는 방향으로 작성됩니다

이런 원칙을 반복하면 팀은 점차 실패를 숨기려 하지 않고, 투명하게 공유하는 방향으로 바뀝니다.

## 포스트모템 공유 전략

포스트모템은 작성 팀 내부에만 남기기에는 가치가 너무 큽니다. 비슷한 문제를 겪을 수 있는 다른 팀, 같은 인프라를 공유하는 조직, 외부 고객까지 포스트모템을 공유하는 방식은 조직마다 다르지만 원칙은 비슷합니다.

**내부 공유:**

- 전사 위키 또는 Confluence에 포스트모템 전체 공개
- 주간 엔지니어링 미팅에서 주요 장애 리뷰
- 공통 인프라 팀에 관련 포스트모템 링크 전달

**제한적 외부 공유:**

- 고객 대상으로는 기술 상세를 제외한 요약본 제공
- 상태 페이지(Status Page)에 간단한 타임라인과 영향 범위 게시
- 보안 취약점이 포함된 경우 패치 완료 후 공개

**완전 공개:**

- 블로그 형식으로 전체 포스트모템 공개 (예: Cloudflare, GitHub, GitLab)
- 기술 커뮤니티와 학습 공유, 신뢰 구축
- 경쟁사도 비슷한 문제를 예방할 수 있게 함

어느 방식을 택하든, 포스트모템을 숨기기보다는 공유하는 쪽이 장기적으로 조직 신뢰와 학습 속도를 높입니다.

## 포스트모템 메트릭 추적

포스트모템의 효과를 측정하려면 메트릭이 필요합니다. 다음 지표를 추적하면 포스트모템 프로세스의 건강도를 확인할 수 있습니다.

```python
def postmortem_health_metrics(postmortems):
    """포스트모템 프로세스 건강도 메트릭"""
    total = len(postmortems)
    if total == 0:
        return {}
    
    # 작성 소요 시간 (장애 복구 후 ~ 문서 완료)
    avg_write_time_hours = sum(
        p["write_time_hours"] for p in postmortems
    ) / total
    
    # 액션 아이템 완료율
    total_actions = sum(len(p["actions"]) for p in postmortems)
    completed_actions = sum(
        sum(1 for a in p["actions"] if a.get("done"))
        for p in postmortems
    )
    action_completion_rate = completed_actions / total_actions if total_actions > 0 else 0
    
    # 반복 장애 비율 (같은 root cause 재발)
    root_causes = [p["root_cause"] for p in postmortems]
    unique_causes = len(set(root_causes))
    repeat_rate = 1 - (unique_causes / total)
    
    return {
        "total_postmortems": total,
        "avg_write_time_hours": f"{avg_write_time_hours:.1f}h",
        "action_completion_rate": f"{action_completion_rate*100:.0f}%",
        "repeat_incident_rate": f"{repeat_rate*100:.0f}%",
    }
```

| 지표 | 목표 | 의미 |
| --- | --- | --- |
| 작성 소요 시간 | 24시간 이내 | 빠를수록 기억이 정확함 |
| 액션 완료율 | 90% 이상 | 학습이 실제 변경으로 이어지는지 |
| 반복 장애 비율 | 10% 미만 | 포스트모템이 실제로 재발을 줄이는지 |
| 공유 범위 | 관련 팀 100% | 다른 팀도 같은 실수를 피할 수 있는지 |

이 지표를 분기마다 리뷰하면 포스트모템 문화가 실질적으로 개선되고 있는지 확인할 수 있습니다. 액션 완료율이 낮다면 우선순위 설정이나 오너십에 문제가 있다는 신호입니다.

## 이 코드에서 먼저 봐야 할 점

- 템플릿은 기록 품질의 편차를 줄여 줍니다.
- 영향, 타임라인, 원인, 액션이 함께 있어야 학습이 완성됩니다.
- 오너와 기한은 액션 추적의 기본 축입니다.
- 포스트모템은 쓰는 순간보다 이후 관리에서 진짜 가치가 드러납니다.

## 여기서 자주 헷갈립니다

첫 번째 실수는 증상을 root cause로 착각하는 것입니다. 데이터베이스 과부하는 결과일 수 있고, 원인은 쿼리 폭주를 막지 못한 보호 장치 부재일 수 있습니다. 원인 층위를 더 깊게 봐야 합니다.

두 번째 실수는 문서를 쓰는 것으로 일을 끝냈다고 생각하는 것입니다. 액션 아이템이 추적되지 않으면 학습은 조직에 남지 않습니다.

세 번째 실수는 결과를 공유하지 않는 것입니다. 같은 유형의 문제가 다른 팀에서도 반복될 수 있기 때문에, 포스트모템은 가능한 범위에서 널리 열려 있을수록 좋습니다.

## 운영 체크리스트

- [ ] 공통 포스트모템 템플릿이 있다.
- [ ] blameless 원칙을 팀이 공유한다.
- [ ] 액션 아이템에 오너와 기한이 있다.
- [ ] 열린 후속 작업을 정기적으로 리뷰한다.
- [ ] 포스트모템 결과를 관련 팀과 공유한다.

## 실무에서는 이렇게 생각합니다

시니어 엔지니어는 포스트모템을 문서 작업으로 보지 않습니다. 시스템을 바꾸기 위한 입력 장치로 봅니다. 같은 장애가 다시 오지 않도록 만드는 가장 현실적인 수단이기 때문입니다.

또한 Jira나 Linear 같은 추적 도구와 포스트모템이 분리되지 않게 운영합니다. 문서 속 액션이 실제 작업 티켓으로 이어져야 학습이 조직의 일정과 우선순위 안으로 들어옵니다.

## 정리

postmortem은 장애 뒤에 남기는 학습 시스템입니다. 비난보다 구조를 보고, 문서 작성보다 후속 액션 완료를 중시할 때 같은 유형의 장애를 덜 겪는 조직으로 바뀔 수 있습니다.

다음 글에서는 Toil 줄이기를 다룹니다. 반복 수작업을 어떻게 측정하고, 어떤 기준으로 자동화 우선순위를 정해야 하는지 이어서 정리하겠습니다.

## 처음 질문으로 돌아가기

- **포스트모템은 왜 보고서보다 학습 장치에 가깝다고 할까요?**
  - 포스트모템은 Postmortem를 타임라인과 액션 항목으로 나누어 보면서, 개별 기술 문제가 아니라 시스템 설계 결함으로 이해하는 프로세스입니다.
- **blameless 원칙이 없으면 왜 중요한 정보가 빠질까요?**
  - 비난 없는 문화에서는 사람들이 방어적으로 말하지 않으며, 정보 부족, 경고 신호 부재, 절차 빈틈 같은 진짜 원인이 드러납니다.
- **좋은 포스트모템 문서에는 어떤 항목이 반드시 들어가야 할까요?**
  - 공통 템플릿은 누구나 같은 뼈대로 사실, 영향, 원인, 후속 조치를 남기게 하여, 문장력보다 구조에 의존합니다.
