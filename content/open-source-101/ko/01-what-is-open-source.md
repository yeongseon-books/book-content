# 오픈소스란 무엇인가

처음 오픈소스를 접하면 대개 가격부터 떠올립니다. 무료로 내려받아 쓸 수 있는 코드라는 감각은 출발점으로는 맞습니다. 하지만 그 설명만으로는 왜 라이선스를 읽어야 하는지, 왜 이슈와 PR이 중요한지, 왜 메인테이너와 커뮤니티 문화가 프로젝트 품질에 직접 영향을 주는지 설명되지 않습니다.

이 글은 Open Source 101 시리즈의 첫 번째 글입니다.

여기서는 오픈소스를 단순한 코드 공개가 아니라, 읽고 고치고 다시 나눌 수 있는 권리와 그 권리를 실제 협업으로 굴리는 문화까지 포함한 개념으로 정리하겠습니다.

## 이 글에서 다룰 문제

- 오픈소스를 공짜 코드라고만 보면 왜 계속 오해가 생길까요?
- free software, upstream, fork, contributor 같은 기본 용어는 어떻게 구분해야 할까요?
- 코드 작성 외에도 왜 문서, 번역, 재현 절차 정리가 모두 기여가 될까요?
- 처음 저장소를 볼 때 무엇부터 읽어야 프로젝트 성격이 보일까요?
- 작은 첫 기여가 이후 참여 난도를 왜 크게 낮출까요?

> 오픈소스는 공개 저장소가 아니라, 권리와 책임과 협업 규칙이 함께 움직이는 작업장입니다.

## 왜 중요한가

현대 소프트웨어는 오픈소스 위에 서 있습니다. 웹 프레임워크, 데이터 처리 도구, 인프라 자동화 도구, 테스트 라이브러리까지 거의 모든 층에서 오픈소스를 만납니다. 그래서 오픈소스를 이해한다는 것은 남이 만든 코드를 가져다 쓰는 법을 아는 수준을 넘어, 지금의 소프트웨어가 어떤 방식으로 함께 만들어지고 유지되는지 이해하는 일에 가깝습니다.

실무에서도 이 감각은 바로 필요합니다. 어떤 라이브러리를 도입할지 볼 때는 기능만 보지 않습니다. 라이선스가 무엇인지, 릴리스가 꾸준한지, 이슈 응답이 빠른지, 기여 문서가 정리돼 있는지, 한 사람에게 운영 부담이 몰려 있지 않은지까지 함께 봅니다. 결국 오픈소스를 읽는 일은 코드와 커뮤니티를 함께 읽는 일입니다.

## 한 번에 잡는 개념 지도

```mermaid
flowchart LR
    U[사용자] --> C[기여자]
    C --> M[메인테이너]
    M --> P[프로젝트]
```

이 그림을 위계도로 읽으면 반만 이해한 셈입니다. 실제로는 순환 구조에 더 가깝습니다. 사용자는 버그를 발견해 이슈를 남길 수 있고, 문서 오타를 고치면서 기여자가 될 수 있습니다. 기여자는 경험이 쌓이면 리뷰를 맡고, 장기적으로는 메인테이너 역할까지 이어질 수 있습니다. 오픈소스의 핵심은 모두가 같은 수준이라는 말이 아니라, 참여 경로가 열려 있다는 점입니다.

그래서 오픈소스는 파일이 공개돼 있다는 사실만으로 정의되지 않습니다. 코드를 읽을 수 있어야 하고, 수정할 수 있어야 하며, 다시 배포할 수 있어야 합니다. 여기에 실제로 사람을 받아들이는 운영 문화가 더해져야 비로소 살아 있는 프로젝트가 됩니다.

## 기본 용어를 먼저 바로잡기

오픈소스를 처음 읽을 때는 많은 용어를 외우기보다, 저장소를 이해하는 데 꼭 필요한 최소 단어를 정확히 구분하는 편이 낫습니다.

오픈소스는 소스 코드를 볼 수 있다는 사실만이 아니라, 수정과 재배포를 허용하는 라이선스 조건까지 포함한 개념입니다. free software는 가격보다 자유에 무게를 둡니다. 비용이 0원이라는 뜻이 아니라 사용자의 권리를 강조하는 표현입니다.

upstream은 원래 프로젝트 저장소를 가리킵니다. 내가 포크한 저장소가 아니라 기준이 되는 본류입니다. fork는 원본 저장소를 내 계정 아래 복제한 작업 사본입니다. 기여 흐름에서는 안전하게 실험하는 개인 작업 공간 역할을 합니다. contributor는 코드만 고치는 사람이 아닙니다. 문서, 번역, 디자인, 재현 절차 정리, 사용자 지원도 모두 기여에 들어갑니다.

이 다섯 단어만 정확히 잡아도 README, LICENSE, Issues, Pull Requests가 왜 서로 연결되는지 훨씬 또렷하게 보입니다.

## 생각이 어떻게 바뀌어야 할까

처음에는 오픈소스를 무료 코드라고만 보기 쉽습니다. 하지만 조금만 더 들여다보면, 더 정확한 정의는 따로 있습니다. 오픈소스는 코드를 읽고 고치고 공유할 수 있는 권리와, 그 권리를 실제 협업으로 바꾸는 프로젝트 운영 방식까지 포함한 생태계입니다.

이 차이를 이해하면 이후 글들도 한 줄로 이어집니다. 라이선스는 권리를 문서로 적어 둔 장치이고, 이슈는 문제를 함께 정의하는 공간이며, PR은 변경을 공동 검토하는 절차이고, 메인테이너는 이 흐름을 오래 유지하게 만드는 사람입니다.

## 직접 따라해 보기: 오픈소스 저장소 읽기

### 1단계 — 저장소 찾기

처음에는 아무 프로젝트나 보기보다 언어와 주제가 분명한 저장소를 고르는 편이 좋습니다. 그래야 이후에 라이선스와 기여 흐름을 읽을 때 맥락이 흐려지지 않습니다.

```bash
gh search repos --language python --topic open-source
```

### 2단계 — 라이선스 확인하기

프로젝트 소개보다 먼저 라이선스를 확인해 보세요. 어떤 권리가 허용되고, 어떤 고지를 남겨야 하는지 읽는 습관이 여기서 시작됩니다.

```bash
gh repo view fastapi/fastapi --json licenseInfo
```

### 3단계 — 기여자 흐름 보기

저장소가 혼자 유지되는지, 여러 사람이 함께 돌보는지 보는 가장 빠른 단서입니다. 사람 수보다도 기여가 분산돼 있는지가 더 중요할 때가 많습니다.

```bash
gh api repos/fastapi/fastapi/contributors --jq '.[].login' | head
```

### 4단계 — 입구가 열려 있는지 확인하기

초보자에게 가장 좋은 시작점은 범위가 잘 잘린 작은 문제입니다. `good first issue` 라벨은 프로젝트가 신규 기여자를 받아들일 준비가 되어 있다는 신호이기도 합니다.

```bash
gh issue list --repo fastapi/fastapi --label "good first issue"
```

### 5단계 — 관심 표시하기

별표 하나가 대단한 기여처럼 보이지 않을 수 있습니다. 그래도 프로젝트 입장에서는 관심의 크기와 사용자 분포를 읽는 신호가 됩니다.

```bash
gh repo star fastapi/fastapi
```

## 이 예시에서 먼저 읽어야 할 점

이 다섯 단계는 명령을 익히는 훈련이 아니라, 저장소를 보는 순서를 익히는 훈련입니다. 라이선스는 기능 설명보다 먼저 읽어야 하는 문서입니다. 기여자 목록은 프로젝트가 개인 취미 수준인지, 지속 가능한 협업 구조를 갖췄는지 가늠하게 해 줍니다. `good first issue` 라벨은 단순 태그가 아니라 진입 경로입니다.

같은 저장소라도 무엇을 보기 위해 열었는지에 따라 완전히 다르게 읽힙니다. 오픈소스를 제대로 이해한 사람은 단순히 "쓸 만한 코드인가"만 묻지 않고, "이 프로젝트는 함께 일할 준비가 되어 있는가"까지 묻습니다.

## 자주 하는 실수 다섯 가지

1. 저장소 소개만 읽고 라이선스를 확인하지 않습니다.
2. 포크한 저장소와 원본 저장소를 같은 것으로 생각합니다.
3. 이슈 탭을 질문 게시판처럼만 사용합니다.
4. 코드 기여만 진짜 기여라고 여깁니다.
5. 첫 기여부터 큰 기능 추가를 시도합니다.

## 실무에서는 이렇게 생각한다

시니어 엔지니어는 오픈소스를 기능 카탈로그처럼만 보지 않습니다. 릴리스 주기, 이슈 응답 속도, 문서 품질, 라이선스 호환성, 메인테이너 수까지 함께 봅니다. 이유는 간단합니다. 회사 서비스는 그 프로젝트의 건강 상태 위에 올라가기 때문입니다.

또 작은 기여 경험은 생각보다 큰 신뢰로 이어집니다. 문서 오타 하나를 고치는 일이라도 저장소 구조를 읽고, 규칙을 따르고, 리뷰를 받고, 변경 이력을 남기는 과정을 한 번 통과하면 이후 기여 난도가 크게 낮아집니다. 첫 기여가 작아도 의미가 큰 이유가 여기 있습니다.

## 체크리스트

- [ ] 관심 있는 저장소 하나를 찾았습니다.
- [ ] LICENSE 또는 `licenseInfo`를 확인했습니다.
- [ ] `good first issue` 라벨이 붙은 이슈를 하나 찾았습니다.
- [ ] `CONTRIBUTING` 또는 README에서 기여 흐름을 읽었습니다.

## 연습 문제

1. 오픈소스를 한 문장으로 정의해 보세요.
2. upstream과 fork의 차이를 한 문장으로 설명해 보세요.
3. 코드 작성 외의 기여 예시를 세 가지 적어 보세요.

## 정리

이번 글에서는 오픈소스를 무료 코드가 아니라 권리와 협업 구조를 가진 생태계로 보는 기본 관점을 잡았습니다. 이 관점이 있어야 이후에 나오는 라이선스, 이슈, PR, 커뮤니티, 메인테이너 역할이 따로 놀지 않고 하나의 흐름으로 이어집니다.

다음 글에서는 라이선스를 읽는 법을 봅니다. 오픈소스를 쓰는 순간 기술 선택만 하는 것이 아니라, 법적 조건도 함께 받아들인다는 점을 더 구체적으로 살펴보겠습니다.
