# Open Source 101 (1/10): 오픈소스란 무엇인가

처음 오픈소스를 접하면 대개 가격부터 떠올립니다. 무료로 내려받아 쓸 수 있는 코드라는 감각은 출발점으로는 맞습니다. 하지만 그 설명만으로는 왜 라이선스를 읽어야 하는지, 왜 이슈와 PR이 중요한지, 왜 메인테이너와 커뮤니티 문화가 프로젝트 품질에 직접 영향을 주는지 설명되지 않습니다.

이 글은 Open Source 101 시리즈의 첫 번째 글입니다.

여기서는 오픈소스를 단순한 코드 공개가 아니라, 읽고 고치고 다시 나눌 수 있는 권리와 그 권리를 실제 협업으로 굴리는 문화까지 포함한 개념으로 정리하겠습니다.

## 먼저 던지는 질문

- 오픈소스를 공짜 코드라고만 보면 왜 계속 오해가 생길까요?
- free software, upstream, fork, contributor 같은 기본 용어는 어떻게 구분해야 할까요?
- 코드 작성 외에도 왜 문서, 번역, 재현 절차 정리가 모두 기여가 될까요?

이 그림은 오픈소스의 생태계를 구성하는 네 가지 축—권리, 기여자, 라이선스, 커뮤니티—을 보여줍니다. 순환 구조이며, 이 글에서 우리가 가장 중점적으로 지켜볼 부분입니다.

> 오픈소스를 이해한다는 것은 기술 선택을 넘어 법적 조건, 참여 경로, 커뮤니티 문화까지 함께 읽는 능력입니다.

## 오픈소스 사용자와 기여자의 차이

오픈소스에는 여러 역할이 있습니다. 각 역할은 권한과 책임이 다릅니다.

**사용자(User)** — 코드를 내려받아 사용하는 사람입니다. 라이선스 조건을 지키면 자유롭게 사용할 수 있습니다. 하지만 프로젝트에 직접 영향을 주지는 못합니다.

**기여자(Contributor)** — 버그 리포트, 코드 수정, 문서 개선, 번역 등을 통해 프로젝트를 더 나은 방향으로 만드는 사람입니다. 한 번의 기여만 해도 기여자가 됩니다.

**메인테이너(Maintainer)** — 프로젝트의 방향을 결정하고, PR을 머지하고, 이슈를 분류하고, 릴리스를 관리하는 사람입니다. 장기적으로 프로젝트의 건강을 책임집니다.

**커뮤니티 리더(Community Leader)** — 기술보다 사람을 관리합니다. 행동 강령(Code of Conduct) 유지, 갈등 조정, 신규 기여자 온보딩, 문화 형성 등을 맡습니다.

역할은 고정되지 않습니다. 사용자가 버그를 발견해 이슈를 열면 기여자가 되고, 지속적으로 기여하면 메인테이너가 될 수 있습니다. 중요한 점은 모든 역할이 가치 있다는 것입니다.

## 왜 중요한가

현대 소프트웨어는 오픈소스 위에 서 있습니다. 웹 프레임워크, 데이터 처리 도구, 인프라 자동화 도구, 테스트 라이브러리까지 거의 모든 층에서 오픈소스를 만납니다. 그래서 오픈소스를 이해한다는 것은 남이 만든 코드를 가져다 쓰는 법을 아는 수준을 넘어, 지금의 소프트웨어가 어떤 방식으로 함께 만들어지고 유지되는지 이해하는 일에 가깝습니다.

실무에서도 이 감각은 바로 필요합니다. 어떤 라이브러리를 도입할지 볼 때는 기능만 보지 않습니다. 라이선스가 무엇인지, 릴리스가 꾸준한지, 이슈 응답이 빠른지, 기여 문서가 정리돼 있는지, 한 사람에게 운영 부담이 몰려 있지 않은지까지 함께 봅니다. 결국 오픈소스를 읽는 일은 코드와 커뮤니티를 함께 읽는 일입니다.

## 한 번에 잡는 개념 지도

이 그림을 위계도로 읽으면 반만 이해한 셈입니다. 실제로는 순환 구조에 더 가깝습니다. 사용자는 버그를 발견해 이슈를 남길 수 있고, 문서 오타를 고치면서 기여자가 될 수 있습니다. 기여자는 경험이 쌓이면 리뷰를 맡고, 장기적으로는 메인테이너 역할까지 이어질 수 있습니다. 오픈소스의 핵심은 모두가 같은 수준이라는 말이 아니라, 참여 경로가 열려 있다는 점입니다.

그래서 오픈소스는 파일이 공개돼 있다는 사실만으로 정의되지 않습니다. 코드를 읽을 수 있어야 하고, 수정할 수 있어야 하며, 다시 배포할 수 있어야 합니다. 여기에 실제로 사람을 받아들이는 운영 문화가 더해져야 비로소 살아 있는 프로젝트가 됩니다.

## 기본 용어를 먼저 바로잡기

오픈소스를 처음 읽을 때는 많은 용어를 외우기보다, 저장소를 이해하는 데 꼭 필요한 최소 단어를 정확히 구분하는 편이 낫습니다.

오픈소스는 소스 코드를 볼 수 있다는 사실만이 아니라, 수정과 재배포를 허용하는 라이선스 조건까지 포함한 개념입니다. free software는 가격보다 자유에 무게를 둡니다. 비용이 0원이라는 뜻이 아니라 사용자의 권리를 강조하는 표현입니다.

upstream은 원래 프로젝트 저장소를 가리킵니다. 내가 포크한 저장소가 아니라 기준이 되는 본류입니다. fork는 원본 저장소를 내 계정 아래 복제한 작업 사본입니다. 기여 흐름에서는 안전하게 실험하는 개인 작업 공간 역할을 합니다. contributor는 코드만 고치는 사람이 아닙니다. 문서, 번역, 디자인, 재현 절차 정리, 사용자 지원도 모두 기여에 들어갑니다.

이 다섯 단어만 정확히 잡아도 README, LICENSE, Issues, Pull Requests가 왜 서로 연결되는지 훨씬 또렷하게 보입니다.

## 오픈소스 라이선스 분류

오픈소스 라이선스를 처음 읽을 때는 종류가 너무 많아 혼란스럽습니다. 하지만 실무에서는 네 가지 기준으로 대부분의 상황을 정리할 수 있습니다.

| 라이선스 | 허용 범위 | 주요 제약 | 적합한 경우 |
|---|---|---|---|
| MIT | 사용, 수정, 배포, 판매 | 저작권 고지 유지 | 작은 라이브러리, 최소 제약 선호 |
| Apache 2.0 | 동일 | 저작권 고지 + 특허 보호 | 기업 환경, 특허 리스크 회피 |
| GPL v3 | 동일 | 파생물 소스 공개 의무 | 공유 의무 강제, 커뮤니티 환원 |
| LGPL | 동일 | 라이브러리로 사용 시 공개 의무 제외 | 상용 제품과 함께 쓰이는 라이브러리 |
| BSD (3-Clause) | 동일 | 저작권 고지 + 이름 사용 제한 | 학술 · 연구 배경 프로젝트 |

이 표가 중요한 이유는 라이선스 선택이 단순한 문서 작업이 아니라 프로젝트의 배포 전략과 협업 문화를 결정하기 때문입니다. 같은 코드라도 어떤 라이선스를 붙이느냐에 따라 상용 제품에 포함 가능 여부, 파생물 공개 의무, 특허 보호 수준이 모두 달라집니다.

초보자에게 가장 흔한 실수는 "오픈소스는 다 무료니까 아무거나 써도 되겠지"라고 생각하는 것입니다. 하지만 GPL 코드를 상용 제품에 넣고 소스를 공개하지 않으면 라이선스 위반이 되고, Apache 2.0의 특허 조항을 무시하면 나중에 특허 소송 리스크가 커질 수 있습니다.

## 오픈소스 참여의 실무적 이점

오픈소스 기여는 단순히 코드를 나눠 쓰는 일을 넘어 실무 역량을 키우는 가장 직접적인 경로입니다. 회사 프로젝트만으로는 얻기 어려운 경험이 여기 있습니다.

**이력서와 포트폴리오 효과** — 깃허브 프로필은 이력서를 보완하는 증거입니다. 면접에서 "Django를 잘 다룹니다"라고 말하는 것과 "Django 공식 저장소에 버그 수정 PR 3개를 머지했습니다"라고 말하는 것은 신뢰도가 다릅니다. 특히 신입 개발자에게는 오픈소스 기여 이력이 경력 공백을 메워 주는 가장 강한 무기가 됩니다.

**네트워크와 평판** — 오픈소스 커뮤니티에서 꾸준히 활동하면 자연스럽게 같은 기술 스택을 쓰는 개발자와 연결됩니다. 좋은 PR 하나가 메인테이너의 눈에 띄면 채용 제안으로 이어지기도 합니다. 실제로 많은 기업이 오픈소스 기여자를 우선 채용 풀로 봅니다.

**기술력과 코드 리뷰 감각** — 회사 안에서는 비슷한 사람들과 일하기 쉽지만, 오픈소스에서는 전 세계 개발자와 코드 리뷰를 주고받습니다. 다른 사람의 코드 스타일, 문서 작성법, 테스트 전략을 보면서 자연스럽게 수준이 올라갑니다. 특히 코드 리뷰 대응 능력은 실무에서 바로 통하는 협업 감각입니다.

**도구와 생태계 이해** — 프로젝트를 직접 쓰는 것과 기여하는 것은 깊이가 다릅니다. 문서 오타를 고치더라도 저장소 구조, 빌드 도구, CI 설정, 배포 흐름을 한 번 통과하면 그 도구에 대한 이해가 훨씬 단단해집니다.

**일상 도구로 쓰는 확신** — 오픈소스 기여 경험은 단순히 경력 항목을 넘어십니다. 내가 실제로 사용하는 도구를 개선하고, 그 과정에서 내부 동작까지 이해하게 되면 일상 업무에서도 훨씬 더 효율적으로 문제를 해결할 수 있습니다. 나만 쓰는 도구가 아니라 전 세계가 함께 쓰는 도구로 만들어 가는 경험은 업무 관점을 크게 바꿉니다.

**실패의 비용이 난은** — 오픈소스 기여는 실패해도 크게 손해가 없습니다. PR이 거부되거나 이슈가 닫힘 수 있지만, 그 과정에서 코드 리뷰 경험, Git 흐름 연습, 커뮤니티 규칙 이해는 모두 남습니다. 회사 프로젝트에서는 실패하면 비용이 크지만, 오픈소스는 학습의 장이기도 합니다.

## 첫 기여 체크리스트

오픈소스 기여를 처음 시도할 때는 무엇부터 준비해야 할지 막막합니다. 이 체크리스트는 첫 PR을 열기 전에 놓치기 쉬운 단계를 정리한 것입니다.

**프로젝트 선택 단계**

- [ ] 내가 실제로 사용해 본 프로젝트인가
- [ ] 최근 3개월 내 활동 이력이 있는가
- [ ] `good first issue` 또는 `help wanted` 라벨이 있는가
- [ ] 기여 가이드 (`CONTRIBUTING.md`)가 있는가

**이슈 선택 단계**

- [ ] 이슈 본문에 재현 방법이나 구체적 요구사항이 있는가
- [ ] 담당자가 비어 있거나 명시적으로 도움을 요청했는가
- [ ] 내가 이해할 수 있는 범위의 문제인가
- [ ] 해결 방향에 대해 메인테이너와 합의가 있는가

**작업 준비 단계**

- [ ] 저장소를 포크하고 로컬에 클론했는가
- [ ] 개발 환경 설정 문서를 따라 빌드에 성공했는가
- [ ] 작업용 브랜치를 새로 만들었는가
- [ ] 코드 스타일 가이드를 확인했는가

**PR 제출 단계**

- [ ] 커밋 메시지가 프로젝트 규칙을 따르는가
- [ ] 테스트를 추가하거나 기존 테스트가 통과하는가
- [ ] PR 템플릿이 있다면 모든 항목을 채웠는가
- [ ] 관련 이슈를 `Closes #N` 형식으로 연결했는가

이 체크리스트를 한 번에 다 지키려고 하면 부담스럽습니다. 하지만 첫 PR에서 절반만 지켜도 메인테이너는 "이 사람은 규칙을 읽고 따라하려고 노력했구나"라는 인상을 받습니다. 그 인상이 리뷰 속도와 머지 가능성을 크게 바꿉니다.

## 생각이 어떻게 바뀌어야 할까

처음에는 오픈소스를 무료 코드라고만 보기 쉽습니다. 하지만 조금만 더 들여다보면, 더 정확한 정의는 따로 있습니다. 오픈소스는 코드를 읽고 고치고 공유할 수 있는 권리와, 그 권리를 실제 협업으로 바꾸는 프로젝트 운영 방식까지 포함한 생태계입니다.

이 차이를 이해하면 이후 글들도 한 줄로 이어집니다. 라이선스는 권리를 문서로 적어 둔 장치이고, 이슈는 문제를 함께 정의하는 공간이며, PR은 변경을 공동 검토하는 절차이고, 메인테이너는 이 흐름을 오래 유지하게 만드는 사람입니다.

## 직접 따라해 보기: 오픈소스 저장소 읽기

### 1단계 — 저장소 찾기

처음에는 아무 프로젝트나 보기보다 언어와 주제가 분명한 저장소를 고르는 편이 좋습니다. 그래야 이후에 라이선스와 기여 흐름을 읽을 때 맥락이 흐려지지 않습니다.

```bash
gh search repos --language python --topic open-source
```

### 2단계 — 라이선스 확인하기

프로젝트 소개보다 먼저 라이선스를 확인해 보세요. 어떤 권리가 허용되고, 어떤 고지를 남겨야 하는지 읽는 습관이 여기서 시작됩니다.

```bash
gh repo view fastapi/fastapi --json licenseInfo
```

### 3단계 — 기여자 흐름 보기

저장소가 혼자 유지되는지, 여러 사람이 함께 돌보는지 보는 가장 빠른 단서입니다. 사람 수보다도 기여가 분산돼 있는지가 더 중요할 때가 많습니다.

```bash
gh api repos/fastapi/fastapi/contributors --jq '.[].login' | head
```

### 4단계 — 입구가 열려 있는지 확인하기

초보자에게 가장 좋은 시작점은 범위가 잘 잘린 작은 문제입니다. `good first issue` 라벨은 프로젝트가 신규 기여자를 받아들일 준비가 되어 있다는 신호이기도 합니다.

```bash
gh issue list --repo fastapi/fastapi --label "good first issue"
```

### 5단계 — 관심 표시하기

별표 하나가 대단한 기여처럼 보이지 않을 수 있습니다. 그래도 프로젝트 입장에서는 관심의 크기와 사용자 분포를 읽는 신호가 됩니다.

```bash
gh repo star fastapi/fastapi
```

## 이 예시에서 먼저 읽어야 할 점

이 다섯 단계는 명령을 익히는 훈련이 아니라, 저장소를 보는 순서를 익히는 훈련입니다. 라이선스는 기능 설명보다 먼저 읽어야 하는 문서입니다. 기여자 목록은 프로젝트가 개인 취미 수준인지, 지속 가능한 협업 구조를 갖췄는지 가늠하게 해 줍니다. `good first issue` 라벨은 단순 태그가 아니라 진입 경로입니다.

같은 저장소라도 무엇을 보기 위해 열었는지에 따라 완전히 다르게 읽힙니다. 오픈소스를 제대로 이해한 사람은 단순히 "쓸 만한 코드인가"만 묻지 않고, "이 프로젝트는 함께 일할 준비가 되어 있는가"까지 묻습니다.

## 자주 하는 실수 다섯 가지

1. 저장소 소개만 읽고 라이선스를 확인하지 않습니다.
2. 포크한 저장소와 원본 저장소를 같은 것으로 생각합니다.
3. 이슈 탭을 질문 게시판처럼만 사용합니다.
4. 코드 기여만 진짜 기여라고 여깁니다.
5. 첫 기여부터 큰 기능 추가를 시도합니다.

## 실무에서는 이렇게 생각한다

시니어 엔지니어는 오픈소스를 기능 카탈로그처럼만 보지 않습니다. 릴리스 주기, 이슈 응답 속도, 문서 품질, 라이선스 호환성, 메인테이너 수까지 함께 봅니다. 이유는 간단합니다. 회사 서비스는 그 프로젝트의 건강 상태 위에 올라가기 때문입니다.

또 작은 기여 경험은 생각보다 큰 신뢰로 이어집니다. 문서 오타 하나를 고치는 일이라도 저장소 구조를 읽고, 규칙을 따르고, 리뷰를 받고, 변경 이력을 남기는 과정을 한 번 통과하면 이후 기여 난도가 크게 낮아집니다. 첫 기여가 작아도 의미가 큰 이유가 여기 있습니다.

**오픈소스 건강도 평가 지표** — 프로젝트를 선택할 때 다음 지표를 확인하면 좋습니다: (1) 최근 커밋 날짜 (활발히 유지보수 중인가) (2) 이슈 응답 속도 (커뮤니티가 살아 있는가) (3) 메인테이너 수 (버스 팩터가 높은가) (4) 릴리스 주기 (안정적인 개발 사이클이 있는가) (5) 기여자 수 증가 추세 (커뮤니티가 성장하는가). 이 지표들은 GitHub API나 라이브러리 평가 도구로 자동 수집할 수 있습니다.

## 정리

이번 글에서는 오픈소스를 무료 코드가 아니라 권리와 협업 구조를 가진 생태계로 보는 기본 관점을 잡았습니다. 이 관점이 있어야 이후에 나오는 라이선스, 이슈, PR, 커뮤니티, 메인테이너 역할이 따로 놀지 않고 하나의 흐름으로 이어집니다.

다음 글에서는 라이선스를 읽는 법을 봅니다. 오픈소스를 쓰는 순간 기술 선택만 하는 것이 아니라, 법적 조건도 함께 받아들인다는 점을 더 구체적으로 살펴보겠습니다.

## 처음 질문으로 돌아가기

- **오픈소스를 공짜 코드라고만 보면 왜 계속 오해가 생길까요?** 오픈소스는 가격이 아니라 **권리와 책임**을 정의하는 생태계입니다. 기술적 편의와 법적 조건을 함께 읽지 않으면 라이선스, 기여 과정, 메인테이너의 역할이 따로따로 보입니다.

- **free software, upstream, fork, contributor 같은 기본 용어는 어떻게 구분해야 할까요?** 용어는 혼자 외우기보다 **실제 저장소에서 코드가 움직이는 흐름 안에서** 만나면 자연스럽게 구분됩니다. 라이선스, 포크, 기여자 목록을 함께 읽으면 각 용어의 위치가 명확해집니다.

- **코드 작성 외에도 왜 문서, 번역, 재현 절차 정리가 모두 기여가 될까요?** 오픈소스 생태계를 유지하려면 코드 변경 외에 **접근성, 안정성, 지속성**이 필요합니다. 문서와 번역은 프로젝트의 입구를 넓히고, 재현 절차는 다음 기여자가 더 빨리 시작하게 합니다.
