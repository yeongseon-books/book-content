# Open Source 101 (6/10): 릴리스와 버전 관리

프로젝트를 처음 공개할 때는 코드가 돌아가기만 하면 된다고 생각하기 쉽습니다. 그런데 사용자가 생기기 시작하면 다른 문제가 바로 따라옵니다. 이번 배포가 호환성을 깨는지, 버그 수정만 들어갔는지, 지금 업데이트해도 되는지 사용자에게 어떻게 알려 줄 것인지가 중요해집니다.

이 글은 Open Source 101 시리즈의 여섯 번째 글입니다.

여기서는 시맨틱 버전, 변경 기록, 태그, 릴리스 노트를 함께 묶어 예측 가능한 릴리스를 만드는 기본 절차를 정리하겠습니다.

![Open Source 101 6장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/open-source-101/06/06-01-the-basic-release-map.ko.png)
*Open Source 101 6장 흐름 개요*
> 버전 관리는 기술 정보일 뿐만 아니라 **사용자와의 신뢰 계약**입니다. 버전 이름이 의미하는 바를 명확히 해야 사용자는 안심하고 업그레이드합니다.

## 먼저 던지는 질문

- 시맨틱 버전의 세 숫자는 각각 어떤 위험과 변화를 전달할까요?
- 사전 릴리스 태그는 언제 써야 하고 왜 생략하면 안 될까요?
- 변경 기록은 릴리스 노트와 어떻게 연결될까요?

## 왜 중요한가

버전 번호가 일관되지 않으면 사용자는 업데이트를 두려워하게 됩니다. 호환성을 깨는 변경이 PATCH로 나가거나, 기능 추가가 아무 설명 없이 배포되면 프로젝트를 믿기 어렵습니다. 생태계는 기능보다 예측 가능성을 더 오래 기억합니다.

오픈소스뿐 아니라 내부 라이브러리도 마찬가지입니다. 의존성이 늘어날수록 버전 표기와 변경 기록은 협업 비용을 줄이는 핵심 수단이 됩니다. 프로젝트가 커질수록 버전은 숫자 장식이 아니라 사용자와의 계약이 됩니다.

## 커뮤니티 건강 지표

릴리스와 버전 관리는 프로젝트 내부의 기술 문제처럼 보이지만, 실제로는 커뮤니티 건강 상태를 가장 직접적으로 드러내는 신호입니다. 버전이 예측 가능하면 사용자는 안심하고 업그레이드하고, 변경 기록이 명확하면 문의가 줄어듭니다.

| 지표 | 측정 방법 | 건강 기준 |
|---|---|---|
| 평균 응답 시간 | 이슈·PR 생성부터 첫 댓글까지 | 48시간 이내 |
| 활성 기여자 수 | 최근 90일 커밋 작성자 | 3명 이상 |
| 이슈 해결률 | 닫힌 이슈 / 전체 이슈 | 70% 이상 |
| 문서 품질 | README, CHANGELOG, API docs 존재 | 모두 있음 |

커뮤니티가 건강하면 릴리스 주기도 안정됩니다. 반대로 릴리스가 불규칙하거나 변경 기록이 비어 있으면 기여자는 다음 버전을 믿기 어렵습니다. 이 지표들은 숫자만 보는 것이 아니라, 프로젝트가 사람과 신뢰를 함께 유지하고 있는지 확인하는 수단입니다.

## 릴리스를 읽는 기본 지도

시맨틱 버전을 암기 규칙처럼만 외우면 금방 잊습니다. 대신 사용자 입장에서 이해하면 쉽습니다. MAJOR는 조심하라는 신호이고, MINOR는 새로운 기능이 들어왔다는 신호이며, PATCH는 큰 사용법 변화 없이 안심하고 올릴 가능성이 높다는 신호입니다.

여기에 변경 기록이 붙으면 숫자만으로는 모호했던 부분이 구체화됩니다. 사용자는 버전을 보고 영향 범위를 짐작하고, 변경 기록을 보고 실제 내용을 확인합니다. 두 문서가 함께 있어야 릴리스가 읽히는 상태가 됩니다.

## 꼭 알아야 할 다섯 가지 개념

시맨틱 버전은 `MAJOR.MINOR.PATCH` 규칙으로 변화를 표현합니다. 호환성 깨짐 변경은 이전 사용법과 맞지 않는 수정입니다. 변경 기록은 버전별 변경 이력을 기록하는 문서입니다. 태그는 특정 커밋을 고정하는 불변 참조입니다. 사전 릴리스는 정식 배포 전 시험판을 뜻합니다.

버전만 있고 변경 기록이 없으면 이유를 모르고, 변경 기록만 있고 태그가 없으면 정확히 무엇을 배포했는지 재현하기 어렵습니다. 릴리스는 이 다섯 가지가 함께 움직여야 비로소 신뢰를 얻습니다.

## 행동 강령 핵심 조항

릴리스는 기술 작업이지만, 커뮤니티가 없으면 지속되지 않습니다. 그래서 행동 강령은 릴리스 문서만큼 중요합니다. 다음은 Contributor Covenant 2.1 기준 핵심 조항입니다.

```markdown
## 우리의 약속

우리는 나이, 신체 크기, 장애, 민족성, 성별 정체성과 표현, 경험 수준,
교육, 사회경제적 지위, 국적, 외모, 인종, 종교 또는 성적 정체성과
지향에 관계없이 모든 사람에게 괴롭힘 없는 경험을 제공할 것을 약속합니다.

## 우리의 기준

긍정적인 환경을 만드는 데 기여하는 행동:
- 환영하고 포용적인 언어 사용
- 다른 관점과 경험 존중
- 건설적 비판을 우아하게 받아들임
- 커뮤니티에 가장 좋은 것에 집중

허용되지 않는 행동:
- 성적 언어나 이미지 사용
- 트롤링, 모욕적/경멸적 댓글, 개인/정치적 공격
- 공개적 또는 사적 괴롭힘
- 명시적 허가 없이 물리적/전자적 주소 게시
```

이 조항은 릴리스 노트만큼 자주 업데이트되지 않지만, 프로젝트 분위기의 기준점 역할을 합니다. 릴리스가 반복될수록 커뮤니티는 커지고, 경계가 명확하지 않으면 갈등 비용이 급격히 늘어납니다.

버전 관리는 지속적인 일상 작업이지만, 프로젝트의 경로를 바꾸는 결정적 순간들이 있습니다. `1.0.0` 첫 정식 버전은 소프트웨어를 public API로 공개하는 신호입니다. `2.0.0` 메이저 버전 업그레이드는 기존 사용자에게 마이그레이션 부담을 준다는 각오를 요구합니다. `3.0.0` 연속 메이저 버전은 프로젝트 방향 재정리를 의미하기도 합니다. 각 메이저 버전에서 메인테이너는 사용자의 고통을 이해하고, 마이그레이션 가이드를 제공하고, 충분히 공지하는 책임을 집니다. 이 결정이 신중하지 않으면 커뮤니티는 분열되거나 버전 업그레이드를 회피합니다.
## 생각이 어떻게 바뀌어야 할까

처음에는 날짜나 느낌으로 버전을 붙여도 된다고 생각하기 쉽습니다. 하지만 사용자가 생기는 순간부터 버전은 프로젝트 내부 메모가 아니라 대외 신호가 됩니다.

그래서 더 정확한 관점은 이렇습니다. 버전은 변경의 크기를 말해 주는 약속이고, 변경 기록은 그 약속의 실제 내용을 설명하는 기록입니다. 이 원칙이 흔들리면 코드보다 신뢰가 먼저 흔들립니다.

## 직접 따라해 보기: 릴리스 절차

### 1단계 — 버전 결정하기

이번 변경이 새 기능인지, 버그 수정인지, 호환성을 깨는지부터 판단해야 합니다. 이 판단이 버전 번호의 출발점입니다.

```text
1.2.3 → 1.3.0 (new feature)
1.2.3 → 2.0.0 (breaking change)
1.2.3 → 1.2.4 (bug fix)
```

### 2단계 — 변경 기록 갱신하기

배포 후에 기억을 더듬어 쓰지 말고, 릴리스 직전에 변경점을 정리하는 습관이 좋습니다. 누가 봐도 추가, 수정, 제거가 구분되게 쓰면 더 좋습니다.

````markdown
```markdown
## [1.3.0] - 2026-05-04
### Added
- new --json flag
```
````

### 3단계 — 태그 만들기

태그는 배포된 상태를 고정합니다. 나중에 버그가 나도 정확히 어느 커밋이 릴리스였는지 다시 가리킬 수 있습니다.

```bash
git tag -a v1.3.0 -m "v1.3.0"
git push origin v1.3.0
```

### 4단계 — 릴리스 노트 발행하기

깃허브 릴리스는 변경 기록을 사용자에게 전달하는 공개 창구입니다. 태그만 푸시하고 끝내지 말고, 읽을 수 있는 형태로 남기는 편이 좋습니다.

```bash
gh release create v1.3.0 --notes-file CHANGELOG.md
```

### 5단계 — 자동화 연결하기

릴리스가 반복되면 사람 손으로 빠뜨릴 가능성이 커집니다. 태그를 기준으로 배포가 자동으로 열리게 두면 재현성과 일관성이 좋아집니다.

```yaml
on:
  push:
    tags: ['v*']
jobs:
  release:
    runs-on: ubuntu-latest
```

## 이 예시에서 먼저 읽어야 할 점

MAJOR는 사용자에게 경고를 보내는 숫자입니다. MINOR는 기존 사용 흐름을 유지한 채 기능을 넓힙니다. PATCH는 자주, 작게, 안전하게 나가는 편이 좋습니다. 변경 기록과 태그가 연결되어야 릴리스를 신뢰할 수 있습니다.

결국 릴리스는 배포 자체보다 소통에 가깝습니다. 사용자가 무엇을 기대해도 되는지, 어디서부터 조심해야 하는지, 문제가 생기면 어느 상태로 돌아가야 하는지를 버전과 태그가 말해 줍니다.

## 사전 릴리스 전략

정식 릴리스 전에 사전 릴리스 버전을 공개하면 사용자 피드백을 받을 수 있고, 호환성 문제를 조기에 발견할 수 있습니다. `alpha`, `beta`, `rc`(release candidate)가 일반적인 사전 릴리스 태그입니다.

```text
1.0.0-alpha.1 → 초기 시험판, 기능 불완전
1.0.0-beta.1 → 기능 완성, 버그 수정 단계
1.0.0-rc.1 → 최종 후보, 치명적 버그만 수정
1.0.0 → 정식 릴리스
```

사전 릴리스를 건너뛰고 바로 정식 버전을 내면 사용자가 예상치 못한 버그를 만날 가능성이 높습니다. 특히 메이저 버전(2.0.0)은 사전 릴리스를 여러 번 거치는 편이 안전합니다.

```yaml
# .github/workflows/pre-release.yml
on:
  push:
    tags:
      - 'v*-alpha*'
      - 'v*-beta*'
      - 'v*-rc*'
jobs:
  pre-release:
    runs-on: ubuntu-latest
    steps:
    - uses: actions/checkout@v4
    - run: gh release create ${{ github.ref_name }} --prerelease
      env:
        GITHUB_TOKEN: ${{ secrets.GITHUB_TOKEN }}
```

사전 릴리스는 GitHub에서 자동으로 "pre-release" 배지를 붙여 표시됩니다. 사용자는 이 배지를 보고 안정성을 판단할 수 있습니다.

## 독성 참여자 대응

릴리스 주기가 안정되고 사용자가 늘어나면, 때로는 공격적이거나 반복적으로 규칙을 어기는 참여자가 나타납니다. 이때 메인테이너는 코드 리뷰어인 동시에 커뮤니티 경계 관리자 역할도 해야 합니다.

**1단계: 경고와 명확한 근거**

첫 위반 시에는 행동 강령 특정 조항을 인용하며 경고합니다. 감정적 표현 대신 객관적 기준을 제시하는 편이 좋습니다.

```markdown
> @username, 이 댓글은 우리의 Code of Conduct 2.1절 "건설적 비판"
> 기준에 어긋납니다. 수정하거나 삭제해 주시기 바랍니다.
```

**2단계: 임시 차단**

경고 후에도 반복되면 일시적 참여 제한을 고려합니다. GitHub Org 설정에서 일정 기간 interaction limit을 걸 수 있습니다.

```bash
# Organization settings → Moderation → Interaction limits
# 7 days / 30 days / indefinite
```

**3단계: 영구 차단 및 문서화**

반복적이고 심각한 위반은 영구 차단 사유가 됩니다. 이때는 결정 이유를 공개 문서로 남기는 편이 투명성 측면에서 좋습니다.

```markdown
## Moderation Log

- 2026-05-20: @username permanently banned for repeated harassment
  after two warnings (issue #123, PR #456).
```

독성 참여자를 방치하면 건강한 기여자가 떠납니다. 릴리스 품질만큼 커뮤니티 안전도 메인테이너의 책임입니다.

## 버전 관리 도구 비교

프로젝트 규모가 커지면 버전 태깅과 릴리스 노트 생성을 자동화하는 도구를 고려할 수 있습니다. 다음은 대표적인 도구들입니다.

| 도구 | 방식 | 장점 | 단점 |
|------|------|------|------|
| **semantic-release** | commit message 규칙 기반 자동 릴리스 | CI/CD 완전 자동화, GitHub/GitLab 지원 | 초기 설정 복잡, commit 규칙 엄격 |
| **bump2version** | 설정 파일 기반 버전 증가 | Python 프로젝트 통합 쉬움, 단순 | 릴리스 노트 자동 생성 없음 |
| **release-drafter** | PR 라벨 기반 초안 작성 | 릴리스 노트 템플릿 자동 생성 | 버전 태깅은 수동 |
| **changie** | 개별 changelog 조각 병합 | 팀 협업 시 conflict 없음 | CLI 사용법 익혀야 함 |

소규모 프로젝트는 수동 태깅과 release-drafter 조합으로 시작하고, 릴리스 빈도가 높아지면 semantic-release를 고려합니다.
## 릴리스 노트 작성 모범 사례

릴리스 노트는 CHANGELOG와 달리 사용자 관점에서 쓴 설명입니다. 기술 변경보다 사용자 경험 변화를 먼저 설명하는 편이 좋습니다.

````markdown
```markdown
# Release v2.0.0

## 변경 사항

- **Async API**: All network calls now return Promises
- **Type safety**: Full TypeScript support with strict mode
- **Better errors**: Error messages now include fix suggestions

## Breaking Changes

⚠️ If you're upgrading from v1.x:

1. `connect()` is now async — add `await`
2. Config format changed — see migration guide below
3. Dropped Python 2 support

## Migration Guide

# Before (v1.x)
client = Client(api_key)
result = client.fetch()

# After (v2.0)
client = await Client(api_key)
result = await client.fetch()

Full migration guide: https://docs.example.com/v2-migration

## Contributors
````

## 자주 하는 실수 다섯 가지

1. 호환성을 깨는 변경을 MINOR로 배포합니다.
2. 변경 기록 없이 릴리스를 밀어 넣습니다.
3. 버전 문자열과 태그가 서로 다릅니다.
4. 시험판 구분을 생략합니다.
5. 릴리스 노트를 비워 둡니다.

## 실무에서는 이렇게 생각한다

시니어 엔지니어는 버전 번호를 기술 장식이 아니라 소통 수단으로 봅니다. 버전 표기가 정확하면 업그레이드 판단이 빨라지고, 변경 기록이 좋으면 지원 비용이 줄어듭니다. 반대로 릴리스가 불투명하면 사용자는 오래된 버전에 머무르기 쉽습니다.

회사 내부 패키지에서도 원리는 같습니다. 배포 대상이 외부 커뮤니티냐 내부 팀이냐만 다를 뿐, 버전 계약과 변경 이력의 중요성은 그대로입니다.

- [ ] 태그를 만들고 푸시할 준비를 했습니다.
- [ ] 릴리스 노트를 공개할 방법을 정했습니다.

3. 태그와 브랜치의 차이를 한 문장으로 적어 보세요.

## 시맨틱 버전을 운영 규칙으로 고정하는 방법

시맨틱 버전은 숫자 규칙이 아니라 팀의 릴리스 합의입니다. 이 합의를 문서로 고정해 두지 않으면 같은 변경을 두고도 사람마다 MAJOR/MINOR/PATCH 판단이 달라집니다.

추천 방식은 `RELEASING.md` 문서 한 장을 두는 것입니다. 어떤 변경이 어떤 버전을 올리는지, 릴리스 전 체크 항목이 무엇인지, 태그와 릴리스 노트를 누가 만드는지 역할을 명확히 적습니다.

```markdown
## 버전 상승 규칙
- MAJOR: 공개 API 호환성 깨짐
- MINOR: 하위 호환 기능 추가
- PATCH: 버그 수정, 문서/내부 개선
```

체크리스트에는 LICENSE/NOTICE 영향도 포함해야 합니다. 기능은 같아도 라이선스 조건이 바뀌면 사용자에게는 큰 변경이기 때문입니다. 또한 `CONTRIBUTING.md`에서 "브레이킹 체인지 PR은 반드시 마이그레이션 가이드 포함"을 강제하면 릴리스 품질이 안정됩니다.

PR 리뷰 시에는 다음 항목을 추가로 봅니다. 변경 로그 초안 작성 여부, SemVer 판단 근거, 롤백 태그 준비 여부, 배포 후 모니터링 지표. 이 네 가지가 빠지면 릴리스는 되었어도 운영은 불안정해집니다.

CI에서는 태그 패턴을 기준으로 릴리스 작업을 분리합니다. `v*`는 정식 릴리스, `v*-rc*`는 사전 릴리스로 나누면 배포 실수를 줄일 수 있습니다. 결국 중요한 것은 "버전 숫자"가 아니라 "버전 판단 과정을 재현 가능하게 남기는 것"입니다.

## 정리

이번 글에서는 릴리스와 버전 관리를 코드 배포 절차가 아니라 사용자와의 약속 관리로 보는 관점을 정리했습니다. 시맨틱 버전과 변경 기록이 함께 움직이면 프로젝트는 훨씬 예측 가능해집니다.

다음 글에서는 커뮤니티 운영을 다룹니다. 코드를 공개하고 릴리스까지 했다면, 이제는 사람들이 오래 머물 수 있는 환경도 함께 만들어야 합니다.

## 처음 질문으로 돌아가기

- **Semantic Versioning(1.2.3)이 뭘 의미하고 왜 중요할까요?** MAJOR.MINOR.PATCH는 변경의 **호환성 수준**을 명시합니다. 1.2.0으로 업그레이드했는데 코드가 깨지면 신뢰가 깨집니다.

- **개발 중에는 버전을 어떻게 관리해야 할까요?** Pre-release(0.1.0-alpha, 0.1.0-beta)를 통해 **변경 사항을 미리 공유**하고, 피드백을 받은 후 정식 릴리스를 합니다.

- **릴리스 노트가 중요한 이유는?** 릴리스 노트는 업그레이드 **필수 정보, 주의사항, 마이그레이션 가이드**를 담습니다. 명확한 노트는 개발자의 번거로운 추측을 줄입니다.
