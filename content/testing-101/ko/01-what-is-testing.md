# Testing 101 (1/10): 테스트란 무엇인가?

처음 테스트를 배울 때는 코드보다 절차부터 떠올리기 쉽습니다. 서버를 띄우고, 브라우저를 열고, 회원가입 버튼을 눌러 보고, 로그인이 되는지 확인하는 식입니다. 이 방식은 처음 한두 번은 통합니다. 그런데 기능이 늘고 사람이 늘면 곧 같은 질문이 따라옵니다. 어제 확인한 기능을 오늘도 다시 손으로 눌러 봐야 할까요?

변경이 잦은 코드베이스에서는 수작업 확인만으로 품질을 지키기 어렵습니다. 확인 범위가 넓어질수록 빠뜨리는 항목이 생기고, 그 공백은 배포 뒤에 사고로 돌아옵니다.

이 글은 Testing 101 시리즈의 첫 번째 글입니다. 여기서는 테스트가 무엇인지, 자동 테스트가 왜 필요한지, 그리고 뒤이어 나올 단위 테스트·통합 테스트·E2E 테스트가 어떤 자리에 놓이는지부터 정리하겠습니다.

## 먼저 던지는 질문

- 테스트는 정확히 무엇을 검증하는 코드일까요?
- 수동 확인과 자동 테스트는 무엇이 다를까요?
- 단위 테스트, 통합 테스트, E2E 테스트는 어떻게 구분할까요?

## 큰 그림

![Testing 101 1장 흐름 개요](https://yeongseon-books.github.io/book-public-assets/assets/testing-101/01/01-01-diagram.ko.png)

*Testing 101 1장 흐름 개요*

이 그림에서는 테스트의 역사와 현실 사이의 괴리, 특히 수동 확인만으로는 커버할 수 없는 위험을 보여줍니다. 반복 불가능한 수동 검사와 자동화된 검증의 차이는 팀 규모가 커질수록 극명하게 드러납니다.

> 테스트의 본질은 한 번의 확인이 아니라, 같은 검증을 여러 번 재현할 수 있는 자동화된 절차입니다.

## 왜 중요한가

테스트가 없으면 모든 수정이 추측에 가까워집니다. 회원가입을 고쳤는데 결제가 깨지고, 결제를 고쳤는데 로그인이 깨지는 일이 반복됩니다. 문제는 버그가 생긴다는 사실만이 아닙니다. 팀이 변경을 두려워하게 된다는 점이 더 큽니다.

자동 테스트는 변경을 멈추지 않으면서 위험을 줄이는 장치입니다. 코드를 고친 뒤 바로 다시 돌려 볼 수 있고, 다른 사람이 만든 변경까지 같은 기준으로 확인할 수 있습니다. 그래서 테스트는 기능 개발의 반대편에 있는 부담이 아니라, 계속 바꿀 수 있게 해 주는 안전장치에 가깝습니다.

## 한눈에 보는 구조

가장 단순한 흐름은 이렇습니다. 프로덕션 코드가 있고, 그 코드를 검증하는 테스트 코드가 있습니다. 테스트 러너는 테스트를 모아서 실행하고, 통과 여부와 실패 위치를 알려 줍니다.

```text
┌─────────────────┐     ┌──────────────────┐     ┌────────────────┐
│ Production Code │ ←── │   Test Code      │ ←── │  Test Runner   │
│  (src/)         │     │  (tests/)        │     │  (pytest)      │
└─────────────────┘     └──────────────────┘     └────────────────┘
                                                         │
                                                         ▼
                                                  ┌────────────────┐
                                                  │  결과 리포트    │
                                                  │  Pass / Fail   │
                                                  └────────────────┘
```

중요한 점은 테스트도 코드라는 사실입니다. 기대하는 동작을 사람이 말로 설명하는 대신 실행 가능한 형태로 남겨 둔 것입니다. 테스트가 코드이기 때문에 버전 관리가 됩니다. PR에서 리뷰할 수 있고, CI에서 자동 실행할 수 있습니다.

## 핵심 용어

| 용어 | 의미 | 예시 |
|------|------|------|
| 테스트 | 기대하는 동작을 코드로 표현한 검증 절차 | `assert add(2, 3) == 5` |
| 단언문(assertion) | 실제 값이 기대와 맞는지 확인하는 표현 | `assert result == expected` |
| 테스트 러너 | 테스트를 수집하고 실행하는 도구 | pytest, unittest, Jest |
| 픽스처(fixture) | 테스트에서 재사용할 데이터나 상태 | DB 연결, 임시 파일, 테스트 사용자 |
| 커버리지(coverage) | 테스트가 프로덕션 코드를 어디까지 실행했는지 보여 주는 지표 | line coverage 85% |
| SUT(System Under Test) | 현재 테스트 대상이 되는 코드 단위 | 함수, 클래스, 모듈 |
| 테스트 스위트(suite) | 관련 테스트를 묶은 집합 | `tests/test_auth.py` 파일 전체 |

## 바꾸기 전과 후

**바꾸기 전 — 수동 확인 중심**

```text
1. 로컬에서 서버를 띄운다
2. 브라우저에서 회원가입 → 로그인 → 결제를 눌러 본다
3. "잘 되네"라고 판단하고 PR을 머지한다
4. 사흘 뒤 다른 변경 때문에 같은 흐름이 다시 깨진다
```

**바꾼 뒤 — 자동 테스트 중심**

```bash
$ pytest
collected 142 items
.................................... 142 passed in 3.4s
```

차이는 단순합니다. 수동 확인은 그 순간에만 유효합니다. 자동 테스트는 다음 주에도, 반년 뒤에도, 다른 사람이 같은 코드를 건드렸을 때도 다시 실행됩니다. 팀 단위로 작업할수록 이 차이가 크게 벌어집니다.

| 비교 항목 | 수동 확인 | 자동 테스트 |
|-----------|-----------|-------------|
| 반복 비용 | 매번 사람이 시간 투입 | 한 번 작성, 무한 반복 |
| 범위 | 기억에 의존, 빠뜨림 발생 | 정의된 시나리오 전부 실행 |
| 피드백 속도 | 분~시간 | 초~분 |
| 신규 인원 활용 | 매뉴얼 필요 | `pytest` 한 줄 |
| 회귀 감지 | 우연히 발견 | 즉시 실패 |

## 다섯 단계로 첫 자동 테스트 만들기

### 1단계 — 검증할 함수 준비

```python
# src/calc.py
def add(a: int, b: int) -> int:
    return a + b
```

### 2단계 — 테스트 파일 작성

```python
# tests/test_calc.py
from src.calc import add

def test_add_positive_numbers():
    assert add(2, 3) == 5

def test_add_with_zero():
    assert add(0, 7) == 7
```

### 3단계 — 실행

```bash
pip install pytest
pytest -v
```

### 4단계 — 일부러 깨뜨려 보기

```python
def add(a: int, b: int) -> int:
    return a - b   # 버그
```

```bash
pytest -v
# FAILED tests/test_calc.py::test_add_positive_numbers - assert -1 == 5
```

### 5단계 — 다시 고치고 신뢰 확인

`add` 함수를 원래대로 되돌린 뒤 `pytest`를 다시 실행합니다. 모든 테스트가 초록색으로 돌아오면, 테스트가 실제로 버그를 잡아냈다는 경험을 한 번 얻게 됩니다.

## 이 코드에서 먼저 볼 점

- 테스트는 생각보다 작습니다. 거대한 프레임워크를 먼저 배울 필요가 없습니다.
- 코드를 일부러 깨뜨렸다가 다시 초록색으로 만드는 과정이 테스트의 가치를 가장 빨리 보여 줍니다.
- 테스트는 문서 역할도 합니다. 함수가 어떻게 쓰이는지 예제로 남기기 때문입니다.

테스트 입문에서 중요한 감각은 복잡한 예제를 만드는 일이 아닙니다. 실패를 읽고, 원인을 고치고, 다시 통과시키는 반복이 어떻게 신뢰를 쌓는지 몸으로 익히는 일입니다.

실무에서 테스트 코드를 처음 작성하는 순간은 보통 두 가지입니다. 첫째는 버그를 고친 뒤 같은 버그가 다시 나오지 않는다는 확신이 필요할 때이고, 둘째는 리팩토링 전에 기존 동작을 보존해야 할 때입니다. 두 경우 모두 테스트가 없으면 눈으로 확인하는 수밖에 없고, 확인 비용은 코드가 커질수록 기하급수적으로 늘어납니다.

## 테스트의 세 가지 레벨

자동 테스트는 검증 범위에 따라 세 레벨로 나뉩니다. 이 시리즈 뒤편에서 각각을 깊이 다루지만, 여기서 큰 그림을 먼저 잡아 두면 이후 글의 위치가 선명해집니다.

```text
                ┌───────────┐
               /  E2E Test   \         ← 적고, 느리고, 비싸다
              /───────────────\
             / Integration Test\       ← 중간
            /───────────────────\
           /     Unit Test       \     ← 많고, 빠르고, 싸다
          └─────────────────────────┘
```

| 레벨 | 범위 | 속도 | 주요 검증 대상 |
|------|------|------|----------------|
| Unit | 함수 하나, 클래스 하나 | ms 단위 | 로직 정확성 |
| Integration | 모듈 간 경계 (DB, API, 큐) | 초 단위 | 계약 준수 |
| E2E | 사용자 시나리오 전체 | 분 단위 | 비즈니스 흐름 완결성 |

이 비율은 고정값이 아닙니다. 프로젝트 성격에 따라 integration이 두꺼울 수도, E2E가 핵심일 수도 있습니다. 중요한 것은 각 레벨이 **다른 종류의 위험**을 잡는다는 점입니다.

## 좋은 테스트의 조건

모든 테스트가 가치 있지는 않습니다. 유지보수 비용이 가치보다 크면 테스트가 오히려 짐이 됩니다. 좋은 테스트는 다음 네 가지 성질을 갖습니다.

1. **빠릅니다.** 느린 테스트는 실행을 피하게 됩니다. 단위 테스트는 전체 스위트가 수 초 안에 끝나야 합니다.
2. **결정적(deterministic)입니다.** 같은 코드에서 항상 같은 결과가 나와야 합니다. 시간, 네트워크, 랜덤에 의존하면 flaky test가 됩니다.
3. **독립적입니다.** 테스트 A의 결과가 테스트 B에 영향을 주면 안 됩니다. 실행 순서에 따라 결과가 바뀌면 디버깅이 극도로 어려워집니다.
4. **실패 메시지가 명확합니다.** 실패 시 어떤 기대가 어떻게 깨졌는지 한 줄로 읽을 수 있어야 합니다.

```python
# 나쁜 예: 실패 시 "AssertionError" 한 줄만 나옴
def test_bad():
    assert process(data)

# 좋은 예: 실패 시 기대값과 실제값이 모두 보임
def test_good():
    result = process(data)
    assert result.status == "completed", f"Expected completed, got {result.status}"
```

## 테스트를 처음 도입할 때의 현실적 조언

이미 테스트 없이 운영 중인 프로젝트에 테스트를 넣으려면 전체를 한 번에 커버하려 들지 마세요. 다음 순서가 현실적입니다.

1. **새로 작성하는 코드부터** 테스트를 붙입니다. 기존 코드는 일단 놔둡니다.
2. **버그가 나올 때마다** 해당 버그를 재현하는 테스트를 먼저 작성한 뒤 수정합니다.
3. **리팩토링 전에** 현재 동작을 검증하는 테스트를 추가합니다.
4. **CI에 연결합니다.** 로컬에서만 돌리면 잊힙니다. PR마다 자동 실행되게 하면 빠뜨림이 줄어듭니다.

완벽한 커버리지가 목표가 아닙니다. 팀이 "테스트를 돌리는 습관"을 갖는 것이 첫 번째 목표입니다.

## 어디서 자주 헷갈릴까요?

첫 번째 오해는 브라우저에서 한 번 눌러 봤다면 테스트를 한 것이라고 생각하는 경우입니다. 확인 자체는 맞지만, 그 확인을 반복 가능하게 남기지 못하면 팀의 자산이 되지 않습니다.

두 번째 오해는 테스트가 많으면 자동으로 안전하다고 믿는 경우입니다. 테스트는 작성한 시나리오만 검증합니다. 빠진 경로가 있다면 초록색 결과가 떠도 그 경로는 여전히 비어 있습니다.

세 번째 오해는 테스트를 나중에 몰아서 써도 된다고 생각하는 경우입니다. 실제로는 기능을 구현한 맥락이 머릿속에 남아 있을 때 같이 적는 편이 훨씬 저렴합니다. 미루면 테스트는 자주 빠지고, 빠진 테스트는 다시 채워지지 않는 경우가 많습니다.

## 직접 검증해 볼 것

1. `add` 함수를 일부러 `return a - b`로 바꾼 뒤 `pytest -v`를 다시 실행해 봅니다. 실패가 바로 보이지 않으면 테스트가 실제 버그를 감시하지 못하고 있다는 뜻입니다.
2. 두 테스트 이름만 읽고도 함수 계약이 떠오르는지 확인합니다. 이름이 동작을 설명하지 못하면 테스트는 문서 역할도 못 합니다.
3. `pytest -q` 실행 시간을 적어 둡니다. 입문 단계의 테스트가 이미 오래 걸린다면, 이후 시리즈에서 더 많은 테스트를 추가할수록 습관으로 굳기 어렵습니다.

**예상 결과:** 의도적으로 넣은 버그에서는 최소 1개 테스트가 즉시 실패하고, 원래 구현으로 되돌리면 다시 모두 통과해야 합니다.

## 실패 신호와 첫 점검

- 코드를 깨뜨렸는데도 테스트가 모두 통과하면 단언문이 약하거나 검증 경로가 빠졌을 가능성이 큽니다.
- 테스트 이름이 `test_case_1`처럼 모호하면 실패 메시지에서 바로 원인을 읽기 어렵습니다.
- 입문 예제인데도 실행 시간이 길다면 파일 I/O, 네트워크, 전역 상태 같은 외부 의존이 섞였는지 먼저 봅니다.

## 실무에서는 이렇게 생각합니다

현업에서는 테스트를 단순한 검사 도구보다 변경 비용을 낮추는 장치로 봅니다. 팀이 코드를 자주 배포하려면, 사람의 기억과 수동 체크리스트 대신 자동 검증이 먼저 갖춰져야 합니다.

그래서 강한 팀일수록 PR에서 먼저 묻는 질문이 있습니다. 무엇을 바꿨는가보다 무엇을 검증했는가입니다. 테스트가 없으면 리뷰어는 구현 의도와 실패 조건을 파악하기 어려워지고, 머지 뒤의 위험도 크게 올라갑니다.

## 체크리스트

- [ ] `pytest`를 한 번 설치하고 실행해 봤습니다.
- [ ] 코드를 일부러 깨뜨린 뒤 실패 메시지를 읽어 봤습니다.
- [ ] 테스트 하나가 한 가지 동작만 확인하도록 작성했습니다.
- [ ] 테스트 실행 시간이 몇 초 안에 끝나는지 확인했습니다.

## 연습 문제

3. 동료에게 테스트가 왜 필요한지 한 문단으로 설명해 보세요.

## 정리

테스트는 코드를 믿게 만드는 장치가 아니라, 변경을 겁내지 않게 만드는 장치입니다. 수동 확인을 자동 검증으로 바꾸는 순간부터 팀은 같은 기능을 더 자주, 더 안전하게 고칠 수 있습니다. 다음 글에서는 이 전체 그림에서 가장 작은 단위인 단위 테스트를 자세히 보겠습니다.

## 처음 질문으로 돌아가기

- **테스트는 정확히 무엇을 검증하는 코드일까요?**
  - 테스트는 기대하는 동작을 실행 가능한 형태로 남기는 것입니다. 수작업 확인과 달리, 한 번 작성하면 계속 재사용할 수 있으므로 팀의 변경 속도와 신뢰를 동시에 높입니다.
- **수동 확인과 자동 테스트는 무엇이 다를까요?**
  - 수동 확인은 그 순간에만 유효하지만, 자동 테스트는 다음 주에도, 다른 사람이 코드를 건드렸을 때도 같은 기준으로 다시 실행됩니다. 팀이 커질수록 이 차이가 중요해집니다.
- **단위 테스트, 통합 테스트, E2E 테스트는 어떻게 구분할까요?**
  - 세 가지 테스트는 검증 범위와 속도가 다닙니다. 가장 빠른 단위 테스트로 핵심 로직을 다지고, 통합 테스트로 부품 간 연결을 확인하고, E2E 테스트로 사용자 흐름을 검증합니다.
